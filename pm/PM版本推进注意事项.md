# PM版本推进注意事项

## 1. 文档定位
- 本文件记录 `PM` 在版本推进中已经暴露出的稳定遗漏点、当前补洞口径和后续版本细化要求。
- 本文件不是每日执行流水，也不是当前 active 版本快照。
- 本文件只保留高信号、可长期复用的推进注意事项；单轮现场继续写 `pm/versions/<active_version>/history/...`。

## 2. 当前已识别的遗漏点
1. 需求文档与版本计划没有形成稳定的一一映射。
   - 当前 `docs/workflow/overview/需求概述.md` 索引了 `24` 份需求详情文档，但 `V1 ~ V5` 仍主要停留在粗粒度主题级需求点。
   - 结果是：部分有效需求还停留在 `docs/workflow/requirements/*.md`，没有被明确归属到某个版本需求点或 backlog。
2. 测试资产存在，但“需求点 -> 用例 -> 验收证据”映射还没有工程化。
   - `workflow_testmate` 当前项目级用例目录已有 `24` 条正式登记用例。
   - 但当前 `V1` 只在少数需求点上形成了明确强覆盖，仍缺稳定的覆盖率矩阵和盲区清单。
3. 单次 checkpoint / smoke 交付件不能替代项目级用例目录。
   - 类似 `workflow_testmate handoff smoke checkpoint` 这类交付件只代表某一轮命中的最小回归口径。
   - PM 不能把单次 checkpoint 误当成“测试体系已经完整覆盖”的证据。
4. 后续版本计划细化程度不足。
   - `V2 ~ V5` 已有主题级规划，但不少条目仍然过粗，离“可以激活并真正执行”的粒度还有距离。
   - 若不提前细化，版本切换时会再次出现“标题有了，切片不够，测试和验收无绑定”的老问题。
5. `workflow_ucdmate` 的创建时机曾长期悬空。
   - 这条已在 `2026-04-13` 收口为：创建与首批职责接线前移到 `V2`，深度 UCD 改造继续留在 `V4`。
6. “最小闭环已完成”和“完整产品落地仍未完成”曾被混写进同一条版本需求。
   - 典型风险是：底座/模型/最小运行能力已经闭环，但一级模块界面、完整工作面或首期 UI 入口仍未落地；如果继续共用同一条 `completed` 需求，会让版本读面错误显示为“整块已经做完”。
   - 结果是：真实未完成部分会在排期、汇报和切版判断里被提前隐藏，直到用户追问才暴露出来。
7. 版本最低配置泳道容易缺位。
   - 一些版本虽然有主功能线，但没有显式挂工程质量优化、下一迭代需求排期、UCD 优化、小伙伴提升计划、接口/性能优化这些切片。
   - 结果是：版本看起来在推进，实际上只有单条功能线在拉，下一版仍然要从空白起稿，用户也会直接感到“版本太小”。
8. 标准迭代前置环节长期缺位。
   - 现状里常见的是：需求文档存在、版本条目存在、实现和 gate 也在推进，但中间缺 `需求澄清 / 方案设计评审 / 测试设计评审 / 功能演示` 这几道门。
   - 结果是：版本看起来很忙，但你很难直接判断“这条需求当前到底卡在方案、开发还是测试”。
9. 版本进度长期缺少阶段看板和甘特图。
   - 当前进度大多散在 `版本计划.md`、`history/` 和 `PM当前版本计划.md` 的叙述里。
   - 结果是：用户能看到很多推进文字，但很难从同一页直接读到阶段、依赖、里程碑和剩余窗口。
10. 外部项目现场与 `workflow` 平台需求边界容易混写。
   - 当前真实使用里，PM 可能会临时去处理外部工作区、外部运营项目的创建、点火或排障现场。
   - 如果不主动做边界隔离，就容易把“外部项目本身的运营内容”误写进 `workflow` 的版本需求、next version 规划或 7x24 主线。
   - 结果是：`workflow` 的版本计划会被外部项目业务语义污染，平台基础能力问题和外部项目治理问题混成一团，后续切版、验收和范围控制都会失真。

## 3. 当前指定收口版本
1. `V2`
   - 负责把当前有效需求从“文档存在”收口到“明确属于哪个版本需求点”。
   - 负责建立 `需求点 -> 用例 -> 验收证据` 的映射矩阵。
   - 负责把 `V3 ~ V5` 从主题级计划继续细化到激活前可执行的切片。
   - 负责完成 `workflow_ucdmate` 的创建、工作区、记忆骨架和首批职责接线。
2. `V3`
   - 负责把测试资产责任边界、覆盖率维护和小伙伴专业责任一起制度化。
   - 负责把“谁维护哪组用例、谁负责覆盖率盘点、谁负责回归口径”从临时协作变成稳定机制。
3. `V4`
   - 继续承接 `workflow_ucdmate` 接线后的深度 UCD 改造、体验验收与设计回归。
4. `V5`
   - 继续承接当前版本体系之外的长期扩展项，不用于兜底当前版本遗漏的基础治理工作。
   - 同时负责把“最小闭环”和“完整界面/完整产品落地”拆成独立需求维护，避免再次把已完成底座和未完成工作面混写到同一条需求里。

## 4. PM推进注意事项
1. 任何当前有效需求，不能只停在 `docs/workflow/requirements/*.md`。
   - 必须明确写成：`当前 active 版本 / 后续具体版本 / backlog` 三者之一。
2. 任何 planned 版本，不能长期只保留主题标题。
   - 在接近激活前，至少要细化到：需求点、责任人、协作方、目标、依赖、进入前提、退出门槛。
3. active 版本和 next 版本，都必须维护一份可读的映射真相。
   - 最低要求：`需求文档 -> 版本需求点 -> 测试用例 -> 验收证据`。
4. 报告测试覆盖率时，必须显式区分三层口径：
   - 项目级正式用例目录有多少；
   - 本轮命中并执行了多少；
   - 对当前 active 版本形成强覆盖的需求点有多少。
5. 单次 smoke/checkpoint 只代表局部回归，不代表整版覆盖。
   - 需要把这类交付件挂回版本需求点和正式用例目录，才能算稳定资产。
6. 版本切换前，不能只看“当前版本是不是快做完了”。
   - 还要同时确认：后续版本是否已经细化到足够可执行，否则切版只会把混乱往后滚。
7. 后续若再出现“版本太小”或“需求很多没进计划”的反馈，优先补版本映射和切片，而不是先扩大 active 版本体积。
8. 同一份需求文档如果同时覆盖“最小运行底座”和“完整产品工作面”，默认必须拆成两条需求。
   - 例如：模型/链路/后端 contract/最小 live 闭环是一条；
   - 一级导航、默认页面、页签结构、完整 UI 工作面是另一条。
9. 任何需求只要出现“前半段已经完成，但后半段还没做”的状态，默认不能继续用一个 `completed` 覆盖整条需求。
   - 必须显式拆成：
     - `最小完成/底座闭环`
     - `完整落地/界面落地/增强工作面`
   - 两条需求之间要写清依赖关系和各自完成定义。
10. 版本计划里的“完成定义”必须对应单一完成面。
   - 如果一条完成定义同时包含“后端模型闭环”“运行链路闭环”“界面工作面落地”三层，而实际只完成了前两层，就说明这条需求切片过粗，必须立即拆分，不能靠补一句说明糊过去。
11. active 版本里一旦发生需求拆分，必须同步更新 4 处真相源。
   - `pm/versions/<version>/版本计划.md`
   - `pm/PM当前版本计划.md`
   - `pm/versions/<version>/需求映射与覆盖矩阵.md`
   - `pm/versions/<version>/history/YYYY-MM/YYYY-MM-DD.md`
12. 若后续再出现“这个需求明明写 completed，但我看到还有明显没做完的工作面”的反馈，默认先检查是否混写了多个完成面。
   - 优先动作不是争辩口径，而是先拆需求、补完成定义、再重排优先级。
13. 若 active 版本只剩两三条过粗需求、而 next 版本还是空白或纯占位，必须同轮做两件事。
   - 先把 active 版本补成“需求点 + 执行切片 + 与下一版的 handoff 边界”。
   - 再把 next 版本至少补成正式 `planned` 版本，写清主题、需求点、blocking_items 和 go/no-go。
   - 不允许继续用“下一版以后再规划”来掩盖当前切片过粗的问题。
13. 每个版本都要显式回答“最低配置泳道是否已安排”。
   - 最低检查项至少包括：`工程质量优化 / 下一迭代需求排期 / UCD 优化 / 小伙伴提升计划 / 接口或性能优化`。
   - 如果某一类当前版本不做，必须在版本正文里写清 `waiver_reason / 补偿版本 / 替代切片`，不能直接留白。
14. 若当前 active 版本只剩 `2~3` 条大需求，但每条内部还包多类工作面，默认先判定为“粒度不足”。
   - 优先动作是继续拆分成更清楚的需求点或 dedicated 子切片，而不是继续用大需求承接新增工作。
15. 下一版规划不能只写标题和主题词。
   - 至少要在当前 active 版本仍进行时，就提前把 next 版本的最低配置泳道、go/no-go、blocking_items 和需求映射骨架补出来，避免切版时再次从空白起草。
16. active 版本里的每条需求，都要显式带 `current_stage / next_gate / plan_start / plan_end`。
   - 没有这四项，默认就还不够“可管理”，只能算在推进文字里存在。
17. 标准需求不得跳过前置评审直接进入开发。
   - 最低要求是：需求先被拆到足够细粒度，再经过需求澄清、方案设计评审和测试设计评审。
   - 若当前现场必须压缩，也只能走快车道，并且要补合并记录，不能口头略过。
18. 没有功能演示通过，不得把需求默认记成“进入测试阶段”。
   - 若当前版本已经先做了 gate 或回归，再补流程时要显式写成“历史证据回填 / demo 待补录”，不能假装原本就走完了企业流程。
19. 甘特图不能独立发明事实。
   - 甘特图只能来自需求台账和阶段看板；若阶段、依赖或 ETA 变化，先改事实源，再改图。
20. 当前版本若处在“流程补课期”，要显式区分两类状态。
   - `真实推进状态`：开发、验证、发布边界是否在前进。
   - `流程治理状态`：澄清、评审、演示资产是否已经补齐。
   - 不允许再把两类状态揉成一句“in_progress/90%”。
21. 临时处理外部项目现场时，必须先判断“这是平台问题，还是外部项目业务内容”。
   - 只有平台级项目运营基础能力问题，才允许进入 `workflow` 的版本需求、版本 history、后续 probe 或 7x24 治理动作。
   - 外部项目本身的题材、经营策略、项目计划、内容排期、盈利动作等，都不得直接写入 `workflow` 的版本需求池。
22. 若外部项目现场触发了平台问题，版本文案必须用“平台通用能力”表述，不能把外部项目名直接写成需求本体。
   - 允许写：`项目创建后点火桥接缺失`、`外部项目主控最小治理模板支撑`、`任务中文 goal 乱码`。
   - 不允许写：`AI小说项目该怎么运营`、`某外部项目的盈利排期`、`某项目的 7x24 主线治理`。
23. 即使用户明确要求我临时处理外部工作区或外部项目，也不等于这些内容自动进入 `workflow` 的长期治理。
   - 默认口径仍然是：临时现场按一次性处理；只有从中踩出来的平台基础能力问题，才回灌到 `workflow` 的后续版本。
24. `7x24`、看门狗与“假运行/假待派发”治理不能只靠 live 现场肉眼盯。
   - 只要已经出现过 `ghost_running_detected=true`、`ready_dispatch_recovery_requested`、`starting/provider_pid=0`、node/run/project 读面分叉这类现场，后续版本就必须补单测、定向 probe 和必要的压力/耐久测试。
   - 单靠一次 live 修复或一次 status 回读，不足以证明这类问题已经稳定收口；必须让 watchdog/运行真相链有可重复验证资产。
25. 涉及 `7x24` 稳定性的一类问题，默认优先级高于大多数体验优化和外部场景扩展。
   - 角色创建、项目点火、任务派发、ghost running、假待派发、升级门禁分叉这几条都属于“系统能不能稳定运行”的基础层。
   - 若当前已命中这类问题，后续版本优先动作默认是补真相修复和测试护栏，而不是继续往上叠新工作面。
26. 接口性能一旦开始直接影响实际使用体验或 `7x24` 连续推进，不能再长期停留在 `planned / draft` 文字层。
   - 后续必须把“页面可见的接口性能读面”和“真实调优动作”拆成明确执行切片，而不是只保留 `metrics/*.json` 或文档占位。
   - 若 live 现场已经出现分钟级卡顿、超时、慢链路锁阻塞等现象，应优先把它们显式归类为接口/性能问题，而不是继续被其他叙事吞掉。
27. 业务逻辑若已经被拆得过散，不能把“文件变小了”误当成工程治理完成。
   - 后续治理的默认目标应是职责收口、边界清晰和调用链缩短，而不是继续把同一条逻辑切散到更多文件里。
   - 命中这类问题时，优先做“模块职责回归 + 入口收口 + 依赖减法”，不接受只靠机械 split 继续平移复杂度。
28. 所有运营项目创建都必须先落“API 能力合同 + 7x24 运行合同 + 启动准入合同”。
   - 创建项目不能只登记 `project_id / controller_role_id / runtime_policy`；必须同时说明当前可调用 API、项目主控角色何时被派发、派发前置条件是什么。
   - `startup_readiness=ready` 只能说明具备启动条件，不等于已经 7x24 运行；没有 enabled schedule、ready 节点或明确下一次 handoff，就不能说项目已经持续运行。
   - 后续项目创建需求、验收和页面表达都要显式区分：`已登记 / 可启动 / 已点火 / ready 待派发 / running 真执行 / blocked / failed / false-running`。
   - 外部项目样本只能用于验证这套平台通用能力，不能把外部项目业务内容本身写进 `workflow` 版本需求。
29. 版本 AAR 不能只由“需求超时”触发。
   - 只要版本关闭后暴露出排期误判、scope 漂移、生产事故、用户追问遗漏项或 completed 需求仍有实质工作面后移，就要补版本 AAR。
   - AAR 必须回答“后续排期怎么改”，否则只是一份事故说明，不算完成。
   - AAR 产出的调整要同步写回当前或后续版本计划；如果不承接，必须写清豁免理由、补偿版本和风险接受条件。
30. UCD/界面优化最低配置泳道必须有可验收页面范围。
   - 不能只写“优化体验”“前端 view model 减法”这类抽象目标。
   - 至少要写清本版覆盖哪些页面、哪些视觉/交互标准、哪些 probe 或 focused smoke 作为证据。
   - 若本版只完成运行态展示或交互反馈，而没有覆盖配色/信息层级/扫读效率，必须把未覆盖部分写入 AAR 或后续版本硬门槛。
31. 代码坏味道治理优先走质量流水线 harness。
   - PM 根仓默认入口是 `python .repository/<developer_id>/scripts/quality/run_code_quality_pipeline.py --root .repository/<developer_id>`；进入对应代码工作区后可用 `python scripts/quality/run_code_quality_pipeline.py --root .`。
   - 流水线至少产出 `status / checks / debt_items / pm_schedule_signal`，红灯时退出非零，供 7x24 与 PM 排期直接消费。
   - 函数参数个数与圈复杂度也属于坏味道检查项；圈复杂度默认用较大阈值，只在函数明显复杂时给出债务信号。
   - 本地代码工作区可安装 `post-commit` hook：`python scripts/quality/install_code_quality_post_commit_hook.py --root .`。该 hook 提交后自动刷新质量报告，但按 harness 口径不阻断提交。
   - `status=fail` 不等于机械打断所有生产止血，但必须形成当前版本或后续版本的明确整改排期；若不承接，写清 waiver 与风险接受条件。
   - 质量流水线输出的首批债务默认优先交给 `workflow_devmate`，必要时由 `workflow_reviewmate` 做边界复审、`workflow_testmate` 做回归验证。

## 5. 更新规则
1. 只有以下内容才写进本文件：
   - 已稳定识别出的规划遗漏点
   - 已明确决定归属哪个版本收口的补洞项
   - 对 PM 长期有效的版本推进注意事项
2. 以下内容不写进本文件：
   - 当天某一轮 live 运行状态
   - 某个 run / node / ticket 的即时现场
   - 单次 smoke 结果全文
3. 若某条遗漏点已经被对应版本稳定收口，可在后续版本总结时从本文件移除，避免注意事项无限膨胀。
