## 1. Brief 目标
- 先为“多项目运行”的产品层工作面定出信息架构，不急着先做高保真视觉。
- 当前目标不是继续微调任务中心，而是确认 `项目运营` 是否应成为与 `任务中心` 同级的一级模块，以及其首期工作面边界。

## 2. 已冻结方向
1. `任务中心` 负责承接 agent 任务调用，是通用执行面。
2. `项目运营` 负责以团队协作为核心的运营项目，是后续持续创造收益的项目层工作面。
3. `workflow(pm)` 作为项目主控角色，需要像 `workflow` 一样承接 7x24 小时持续运行。
4. `项目运营` 应与 `任务中心` 同级，不再继续埋在任务中心内部。
5. 项目列表首屏保持极简，只展示：
   - 项目名称
   - 项目运行状态
   - 少量高信号状态摘要
6. 单个项目内部以多页签展示不同维度信息，首期至少考虑：
   - 组织架构
   - 版本信息
   - 项目产出
   - 运维参数
7. 项目内部只看任务摘要；任务详情默认回到 `任务中心` 查看。
8. 项目级配置优先放入 `运维参数` 页签。

## 3. 希望原型回答的问题
1. 一级导航上，`项目运营` 与 `任务中心` 的相对位置和命名是什么。
2. 项目列表页如何做到“极简但不失方向感”。
3. 项目详情页的默认落点应该是哪一个页签。
4. 组织架构页签里，如何表现：
   - 项目主控角色
   - 领域成员
   - 成员之间的协作关系
   - 各自承载的制作职责
5. 版本信息 / 项目产出 / 运维参数 这几类页签，哪些信息应该首期就出现，哪些留到后续增强。
6. 项目中的任务摘要与任务中心详细信息之间，跳转关系怎么最顺。

## 4. 原型范围建议
1. 先做低到中保真，不做重视觉。
2. 至少给出以下 4 个画面或等价工作面：
   - 一级导航进入 `项目运营` 后的项目列表页
   - 单个项目的默认首页
   - 组织架构页签
   - 运维参数页签
3. 每个画面附一句说明：
   - 这个画面的主要目标
   - 为什么放这些信息
   - 为什么不在这一层放更多信息

## 5. 当前不要做的事
1. 不要把项目运营继续做成任务中心里的一个大区块。
2. 不要把项目页直接做成“另一个任务中心”。
3. 不要一开始就把所有未来可能的信息都塞满。
4. 不要先纠结视觉皮肤，先把 IA 和工作面边界定住。
