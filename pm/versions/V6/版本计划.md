## 1. 版本定位
- `V6` 承接了 `V5` 退出后的平台发现面建设，把“接口目录能力”从源码与少数人记忆里拉到了稳定的产品与证据读面。
- 这条线最终收口的是“首批接口目录基础闭环”：统一接口目录 API、interface-center 页面、搜索/详情 contract、quality freeze、host/prod live regression，以及 `metrics/evidence` 的最小退出闭环。
- `V6` 的完成口径不等于“全站所有接口都已经补齐元数据、指标与证据”；那条全量覆盖仍留给后续版本继续收口。
- `V6` 完成后，消费型工作不再继续回塞到这一版；项目/角色消费接线、证据聚合与消费 smoke 正式交给 `V7`。

## 2. 进入条件
- `V5` 六项需求已全部完成，`V5` 退出门槛满足，`V6` 在 `2026-04-21` 以 `next_activation_candidate=V6 / next_activation_ready=true` 的 activation gate 进入 active。
- 当时 `workflow` 已具备第二项目最小启动能力、项目运营一级模块和角色质量读面，下一条高价值主线已经从 release 收口转向平台能力发现面建设。

## 3. 版本目标
- 让 `workflow` 提供统一接口目录与 API 帮助发现能力，降低后续运营项目接入成本。
- 把接口级性能、压测与回归证据接进同一工作面，为后续治理、优化和版本比较提供稳定真相源。
- 把 `V6` 的实现切片收成可并行推进的边界，而不是把“接口目录能力”继续保留在单条大需求描述里。

## 4.3 与 V7 的边界
- `V6` 只负责把“接口目录可发现、可搜索、可展开、可回归”收成最小闭环，不再继续扩成通用消费平台。
- 以下事项已经正式移交给 `V7`：
  - 项目运营页、角色工作台、任务合同对接口目录的直接消费接线
  - 接口质量证据的跨版本比较、跨项目比较与消费导览
  - 面向第二项目或 helper 日常工作的接口消费 smoke 套件
- 若后续新想法同时涉及“继续修 `V6` contract”和“把接口能力嵌进更多工作面”，默认先问一句：这条是 defect，还是已经属于 `V7-R1 / V7-R2 / V7-R3`。

## 5. 退出门槛
1. 统一接口目录查询 API 的 contract、字段和来源边界已经落成真实实现。
2. 接口目录页面具备列表、搜索/筛选和详情展开的最小工作面。
3. 接口级性能指标与压测/回归证据至少有一条稳定聚合链，不再只是需求说明。
4. `V6-R1 / V6-R2` 都已经从基线推进到可验证实现，不再停在规划层。
