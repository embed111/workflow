## 5.2 当前阶段并行提效规则
1. 触发条件
   只要同时满足以下条件，本轮就不得继续默认单线程：
   - 主线连续性已经成立
   - 发布边界已 `clean_synced`
   - 至少一个 helper developer workspace 为 `ready`
   - 当前最高价值事项可拆成两个及以上边界清楚、互不阻塞的切片
2. 最小并行目标
   - 命中触发条件后，本轮至少并行派发 `1` 个 helper
   - 若同时存在“探测/复现”和“开发/回归”两类切片，本轮默认目标提升为并行 `2` 个 helper
3. 当前阶段默认并行矩阵
   - `workflow_bugmate`
     - 优先承接 `V1-R2 / V1-R7` 的历史失败节点复现、根因收口、稳定性缺陷定位
   - `workflow_qualitymate`
     - 优先承接 `V1-R2 / V1-R4` 的现网巡检、证据采集、质量异常冻结
   - `workflow_testmate`
     - 优先承接 `V1-R4 / V1-R8` 的 smoke、回归、升级后验证
   - `workflow_devmate`
     - 优先承接 `V1-R3 / V1-R5 / V1-R8` 的工程治理切片、发布链整改、可独立验证的开发实现
4. 允许继续单线程的例外
   - 主线连续性本身未恢复
   - 发布边界未收口
   - helper workspace 不可用
   - 当前任务强耦合，切开会显著降低效率
   - 当前存在高优先生产止损，必须先由 `workflow(pm)` 亲自收口
5. 失效判定
   - 若连续 `2` 轮出现“存在明确可并行切片，但没有派发任何 helper”，默认视为低效轮次
   - 命中上述情况后，下一轮必须先完成 helper 派发，再继续扩大 `workflow(pm)` 自己的工作面
6. 每轮留痕要求
   - 每轮版本记录里都要显式写出：
     - `parallel_candidate_count`
     - `parallel_dispatched_count`
     - `active_helper_tasks`
     - `parallel_block_reason`

## 6. 每轮 PM 主线必查项
1. 当前版本最高优先级需求点是否发生变化。
2. 当前轮是否需要派发、续挂、恢复或调整小伙伴。
3. 当前轮是否满足并行触发条件；若满足，是否已经达到最小并行目标。
4. 当前轮若没有派发 helper，`parallel_block_reason` 是否足够具体且可信。
5. 当前轮是否识别出新的高杠杆功能，应排入后续版本。
6. 当前轮是否识别出新的低维护价值功能，应排入后续重构版本。
7. 当前轮是否需要把某个内容从 `V1` 后移到 `V2 / V3 / V4 / backlog`。
8. 当前轮是否需要更新责任人、协作方或需求拆分方式。
9. 当前轮今日日记是否已追加。
10. 当前轮若识别到风险，是否已经先执行了至少 `1` 项受支持治理动作，而不是只观察风险。
