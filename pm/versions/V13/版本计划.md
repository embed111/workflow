## 1. 版本定位
- `V13` 专门解决当前代码仓“业务逻辑太散”和“代码量/冗余实现偏重”的结构性问题。
- 这一版不把目标定义为继续拆文件，而是把主链、真相源、状态机、投影和验收支撑收口到更少、更清楚、可 review 的领域边界。
- `V13` 允许在 `V11/V12` 完成当前 blocker 后切入；在它正式 active 前，已先建立审计基线、合入 review 角色和硬门槛，避免继续把问题留在口头层面。

## 2. 进入条件
- `V11` 当前 baseline blocker 已完成最小修复或明确移交，不再阻塞工程治理主线。
- `V12-R1 / V12-R5` 至少完成方案冻结，避免角色创建 start 卡住、假运行和长请求反馈问题继续污染 `V13` 的治理证据。
- `workflow_reviewmate` 已通过生产环境创建角色链路建立，并能作为合入前 review 门禁参与工作。
- `.repository/pm-main` 与 `../workflow_code/main` 已处于可同步、可验证、可小批提交状态。

## 3. 版本目标
- 根治业务逻辑横向散落：每条主链只有一个 application facade，每个状态真相只有一个 owner projection。
- 根治冗余实现堆叠：优先删除重复 read model、旧 fallback、一次性 acceptance helper 和无 owner wrapper。
- 降低核心维护成本：逐步退出 `exec_local_parts` 和 manifest globals，让核心域改成显式 import、显式 contract、显式 probe。
- 建立高质量合入流程：所有正式代码合入前默认先过 `workflow_reviewmate` review，不通过就打回。
- 让代码一看就能读出设计：主链入口、领域规则、存储边界、投影边界和验收证据要能被快速画出来。

## 5. 退出门槛
1. 核心业务模块不再新增 `exec_local_parts`；已存在的核心域要么清零，要么有明确 legacy adapter 退出计划。
2. `assignment_service_parts` 从 `37` 个 part 收口到不超过 `12` 个有明确 owner 的 domain/application/repository/projection 模块。
3. `src/workflow_app/server/services` 行数至少减少 `25%`，且关键主链调用深度下降。
4. `scripts/acceptance` 行数至少减少 `20%`，重复支撑函数进入 shared support。
5. 核心业务文件默认不超过 `650` 行；例外必须有 review 通过的 owner 和退出条件。
6. 每批正式代码合入前都有 `workflow_reviewmate` review 记录；命中 `request_changes/block` 不得合入。
7. V13 结束时必须能用一张主链图说明：用户动作、application use case、domain、repository、projection、API/UI response 的边界。
