# V14 治理成果产品化、性能收口与协作效率二期

- version: `V14`
- status: `backlog`
- owner: `workflow(pm)`
- history_root: `pm/versions/V14/history/`
- depends_on: `V13`

## 1. 版本定位
- `V14` 是 `V13` 全仓逻辑边界治理后的承接版本。
- 这一版不在 `V13` 尚未完成前抢跑实现；它先作为 planned 骨架，承接 `V13` 之后需要产品化、性能化和协作效率化的结果。
- 当前目标是保证 `V13` active 期间已经有明确下一版入口，同时避免下一版计划悬空或反向扩大 `V13` scope。

## 2. 进入条件
- `V13` 至少完成 `V13-R3/R4/R5` 的首批代码、review 和 focused gate。
- `V13-R2` reviewmate 合入门禁在真实代码批中至少通过一轮。
- `V13-R7` 对 legacy/fallback 债务给出清晰到期删除或保留条件。
- `.repository/pm-main` 与产生代码的 helper workspace 相对本机 `../workflow_code/main` 均可同步。

## 3. 版本目标
- 把 `V13` 形成的 truth owner、projection、facade 和 review gate 从工程治理资产转成可复用产品/平台能力。
- 降低任务中心与项目工作面关键读接口的延迟和历史扫描成本。
- 让 helper 派发、review、test、release 的并行节奏更可控、更少依赖 PM 手工记忆。
- 继续改善高密度工作面 UCD，让运行真相、项目状态和证据链更易扫读。
- 把项目创建 / 项目初始化 API 的输出从“登记项目与角色”升级为“同时给项目主控 PM 注入可执行的平台 API 调用方法”，让项目 PM 知道如何委派成员执行任务。

## 5. 退出门槛
1. `V13` 产出的核心 truth owner 至少有一条产品级 readback contract。
2. 任务中心/项目工作面的关键读接口有可回读性能基线与回归护栏。
3. helper 实现 -> review -> test 的后继编排不再完全依赖 PM 手动创建每条节点。
4. UCD 工作面至少有一组真实浏览器截图或布局 probe 证明改进。
5. 每个参与 helper 至少留下一份与真实任务绑定的方法资产或学习报告。
6. 新运营项目创建 / bootstrap 后，controller workspace 自动包含成员委派 API runbook，并能回读 `project_id / member_role_ids / assignment endpoints / evidence paths`。
