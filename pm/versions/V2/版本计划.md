## 4.1 当前窗口小伙伴派发与专家化硬约束
1. 当前 `V2` 窗口里，`pm` 默认只保留方向判断、任务切片、优先级、验收、发布边界和跨角色收口；只要任务边界清楚且与 helper 专业匹配，就优先派给对应小伙伴。
2. 本版本里的“有效派发”不按节点数判断，而按“对应领域主任务是否由对应 helper 真实承接并形成 run / 交付”判断；纯学习任务、纯 `ready/blocked` 节点、角色创建链节点，默认不单独算达标。
3. 当前默认领域分工：
   - `workflow_devmate`：工程实现、重构、工程治理执行
   - `workflow_testmate`：smoke、current-version 回归、gate、验收证据
   - `workflow_qualitymate`：质量巡检、证据冻结、风险提报
   - `workflow_bugmate`：缺陷复现、根因定位、修复建议与补丁链
   - `workflow_ucdmate`：UCD 诊断、brief、信息架构与体验验收口径
4. 每轮 `7x24` 必查：是否存在至少 `1` 条可切给 helper 的高价值领域任务；命中则先派发，不允许 `pm` 先亲自做完，再让 helper 只补 smoke、学习报告或总结。
5. 若本轮未派发 helper，必须在 `pm/versions/V2/history/YYYY-MM/YYYY-MM-DD.md` 明确写出：
   - `helper_dispatch_decision`
   - `helper_dispatch_effect`
   - `non_dispatch_reason`
   - `next_helper_batch`
6. 若连续 `2` 轮仍由 `pm` 自己吃掉本可切分的 helper 领域任务，本版本默认视为推进方式失真；下一轮先纠偏派发，再继续扩大 `pm` 自己的工作面。
