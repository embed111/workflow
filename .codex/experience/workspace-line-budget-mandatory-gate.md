# 工作区行数门禁强制化经验

## 适用范围
- `workspace line budget`
- `workflow gate`
- `runtime release gate`
- 把历史 advisory 收口成真正 fail-closed 的工程治理场景

## 稳定经验
- 当维护中代码文件的通用超大文件阈值（当前 `>1000` 行）已经被团队明确要求“必须硬拦”时，不要继续只让它挂在 `refactor_trigger_gate` 这种隐含语义上。更稳的默认是把这条阈值显式并入 `hard_gate`，同时保留 `refactor_trigger_gate` 作为治理视图/兼容字段，并让 `workflow gate / runtime release gate` 一起回写 `oversize_hard_gate_pass / oversize_hard_gate_offender_count`；这样报告、门禁和版本推进都能直接读出“这是超大文件硬门禁”，不会再靠口头解释。
- 不要只改 `check_workspace_line_budget.py` 的退出码就以为强制门禁生效了。`workflow gate` 和 `runtime release gate` 都会自己解析 JSON 结果；如果它们仍只看 `hard_gate_pass`，实际发布链路还是会放行。
- 更稳的默认是给 line-budget 报告新增独立的 `mandatory_gate` 聚合层，把 `Hard Gate / Refactor Trigger Gate / Guideline Gate` 的联合结论单独写出来，再让两个 gate helper 都显式读取 `mandatory_gate_pass`。
- 当 `Mandatory Gate` 已经正式成为当前 active 版本 blocker 后，不要再把“首批冻结对象”只写在 version history 或口头下一步里。更稳的默认是让 line-budget payload / markdown report / gate detail 一起稳定输出 `first_batch_targets`，否则 PM 巡检、发布边界和后续治理批次会反复手工重复同一组冻结判断。
- 当 `schedule_service` 已经把 rule/status/query 拆到独立 runtime 模块后，不要把 list/detail/calendar 这一整层读面投影继续留在 `schedule_query_runtime.py` 里。更稳的默认是再抽一层 `schedule_projection_runtime.py` 承接 `_build_schedule_*` 和公开读 API，否则新 query 模块自己会立刻跨过 Mandatory Gate 1000 行，等于只是把 blocker 平移到新文件。
- 当 `schedule_trigger_runtime.py` 因 trigger progress、supersede bookkeeping 和 processing lock / assignment ref helper 混在一起而命中 Mandatory Gate 时，不要继续只在主 runtime 里删空行或硬压局部表达式。更稳的默认是把这组 trigger bookkeeping helper 抽到独立 `schedule_trigger_progress_runtime.py`，并补 dedicated split probe，再用 `verify_schedule_trigger_recovery_worker.py`、`verify_schedule_pending_trigger_replaced_by_newer.py` 和 `verify_self_iteration_backup_schedule_on_smoke_block.py` 一起兜住恢复、覆盖和 smoke-block 行为回归；这样才能把 trigger runtime 真正压回 1000 行以内，而不是把风险藏在表面行数优化里。
- 当当前 dirty 批次里的功能首刀把 `graph_model_and_payloads.py` 这类核心后端文件继续顶过 guideline gate 时，不要再在原文件上叠第二刀。更稳的默认是先把这一批 feature-specific node payload / project-binding 归一化抽成独立 runtime 模块，让当前批次先从“新增 guideline blocker”退回到“只剩历史 refactor trigger”，再继续 clean-slice 收口。
- 当 `graph_model_and_payloads.py` 已经把 feature-specific node payload / project-binding 抽出去，却仍留在 Mandatory Gate 首批冻结对象里时，不要继续在原文件上补第三刀。更稳的默认是再把节点显示修复、workflow 主线/保底识别和 live schedule goal refresh 抽到独立 `assignment_node_surface_runtime.py`，并补 dedicated split probe 锁住“graph_model 不再定义这批 surface helper + 行数压到首批冻结对象之外”；这样才能把第三个 blocker 真正让位给下一个历史大文件，而不是让图模型继续半拆半留。
- 当 `graph_model_and_payloads.py` 已经把 node payload / node surface 两层先后抽走，却仍因执行提示词、结构化结果归一化和 run process registry 混在同一文件里卡在 Mandatory Gate 首批冻结对象时，不要继续把这些 execution helper 留在“graph model”文件名下。更稳的默认是再抽一层 `assignment_execution_contract_runtime.py`，并补 dedicated split probe 锁住“graph_model < 1000 行 + support 已进 manifest + moved execution helper 全部迁移”；同时用 `verify_assignment_self_iteration_plan_reference.py` 和 `verify_assignment_runtime_metrics_node_fallback.py` 兜住 prompt/runtime-metrics 回归。
- 当 `assignment_execution_contract_runtime.py` 又因为 role-quality contract、评分回写解析和 prompt section 一起长到 split probe 上限时，不要继续把 rubric parser、role-assets lookup 和 prompt fragment 硬塞回 support file 里。更稳的默认是再抽一层独立 `assignment_role_quality_runtime.py`，并把 `_assignment_build_node_record` 的新增合同字段保持向后兼容默认值；否则 `workflow gate` 会先炸在旧 fixture/bootstrap 的签名不兼容，再炸在 support line budget，发布边界会被两层假红灯连续卡住。
- 当 `V5-R3` 的 live helper artifact 已经带上 `## 角色质量评分回写`，而 `status-detail` 也能从 `result_ref + node contract` 自愈出非空评分对象时，不要继续把 `result.json.role_quality_assessment={}`` 误判成“解析器整体失效”。更稳的默认是先检查 worker 在 `_normalize_assignment_execution_result(...)` 时有没有把完整 node record 带进来；如果只传了 `{\"node_id\": ...}`，role-quality normalize 会因为缺少 `problem_type / method_card_id / return_contract` 上下文而把 completed `result.json` 写成空对象，最终形成“result.json 为空、status-detail 非空”的读面分叉。修复这类问题时，优先补一个共享 helper，让 worker 先回读 node record 再归一化结果，并用 dedicated probe 锁住同一路径。
- 当 `defect_service_task_commands.py` 这类缺陷中心 command 文件同时混着全局主图绑定、task name base、task ref upsert 和 legacy defect graph repair runtime 时，不要继续把 create/review 入口和运行时细节硬塞在同一个文件里。更稳的默认是把这组 assignment/runtime helper 抽到独立 `defect_service_assignment_runtime.py`，并补 dedicated split probe 锁住“command file < 500 行 + support file < 1000 行 + moved symbol/constant 全部迁移”；同时用 `run_acceptance_defect_task_naming_and_global_graph_check.py` 兜住 defect task naming/global graph 行为回归。若 schedule scan 现场先回 `queued_for_processing`，验收还要短轮询 `related_task_refs`，不要把正常异步回写误报成假红灯。
- 当前端 runtime 已经按 `bundle_manifest.json` 拆成多个 bundle part 后，不要继续让 layout/workboard probe 只读旧的单文件 `assignment_center_render_runtime.js`。更稳的默认是按 `bundle_manifest.json` 组装整包 JS 再断言 jump target / section marker / helper lane contract，否则拆片本身会把 probe 打成假红灯。
- 当 `assignment_center_probe_runtime.js` 已经承接 workboard layout / surface compact 这类探针收集器，而 `assignment_center_workboard_group_runtime.js / assignment_center_workboard_sections.js` 又承接了 mainline/signal/project/version/collab helper 时，不要继续让 node acceptance probe 只 `vm.runInThisContext('assignment_center_render_runtime.js')` 或只读 `assignment_center_events.js`。更稳的默认是按 `bundle_manifest.json` 组装 `assignment_center_state_helpers.js -> ... -> assignment_center_probe_runtime.js` 这一段 bundle，再在整包上断言/执行；否则 `assignment_center_render_runtime.js` 早就不再自带这些 helper，probe 会稳定误报 `is not a function` 或“collector missing”。
- 当 `assignment_center_render_runtime.js` 已经完成 workboard group/sections 拆片后，不要把 detail/drawer surface 继续留在主 runtime 里。更稳的默认是再抽一个独立 bundle part 承接 detail 与 drawer helper，并补 dedicated split probe 锁住 manifest 装配顺序与函数迁移；这样可以把主文件继续压出 Mandatory Gate 首批冻结对象，而不是让前一轮 split 收益又被详情/表单 UI 吃回去。
- 当 `index.html` 这类超长单页骨架进入 Mandatory Gate 首批冻结对象、但当前页面又还没准备整体迁到新模板引擎时，不要继续把 startup overlay、settings pane、policy confirm modal 全硬挤在主模板里。更稳的默认是给主模板加 fail-closed 的 marker 占位，并由 `pages.py + index_html_manifest.json` 装配 HTML partial，再补 dedicated split probe 锁住“main template < 1000 行 + marker 全部装配 + 关键 DOM id 仍保真”；这样能把主模板压出首批冻结对象，又不会把现有页面直接改造成另一套渲染系统。
- 当 `training_center_role_creation.js` 已经先把 stage-flow/task-preview 运行时拆出去、但 `index_training_center_role_creation.css` 仍把 detail/stage/profile/hover-float surface 全堆在一起命中 Mandatory Gate 时，不要只盯着 JS 继续打转。更稳的默认是再补一个 `index_training_center_role_creation_detail.css` 这类 dedicated CSS part，并用 `index_css_manifest.json + dedicated split probe` 锁住“main CSS < 1000 行 + moved selectors 不再留在主文件 + bundle 仍完整装配”；这样才能让 role-creation workbench 的样式边界跟 JS 边界一起真正对齐。
- 当 `index_training_center_layout.css` 因 training center agent skill/detail surface 命中 Mandatory Gate 时，不要继续把 agent skill card、状态徽标和 detail summary 样式压在 layout 主文件里。更稳的默认是把这组 selector 抽到现有 `index_training_center_cards.css` dedicated part，并用 `verify_training_center_layout_css_split.py` 锁住“main CSS < 1000 行 + moved selectors 不再留在主文件 + bundle 仍完整装配”；这样能让 layout 主文件退出首批冻结对象，又不用把 training center 再拆出一层新的模板壳。
- 当 `index_training_loop_overview.css` 因 training center overview 的 create/workbench surface 过厚而命中 Mandatory Gate 首批冻结对象时，不要继续把 create composer、history/run panel 和 create-only responsive 规则混在主 overview CSS 里。更稳的默认是补一个 `index_training_loop_overview_create.css` 这类 dedicated manifest part，并用 `verify_training_loop_overview_css_split.py` 锁住“main CSS < 1000 行 + moved selectors 不再留在主文件 + bundle 仍完整装配”；这样可以把 overview 主文件压出首批冻结对象，同时不去碰 live scheduler / runtime 逻辑。
- 当 `pm_daily_governance_service.py` 这类 PM 治理服务同时混着 daily history 状态判断、学习报告 delivery/result_ref 发现、optional agent formalization 和 projected markdown 写回 runtime 时，不要继续把“daily governance 壳”和“learning-report discovery/projection helper”硬塞在同一个 service 里。更稳的默认是把这组学习报告发现/投影 runtime 抽到独立 `pm_daily_governance_learning_reports.py`，并补 dedicated split probe 锁住“main service < 900 行 + support file < 1000 行 + moved learning-report helper/constant 全部迁移”；同时用 `verify_pm_daily_governance_tc_pm_001.py` 和 `verify_pm_daily_governance_tc_pm_002.py` 兜住 daily history 与 learning projection 行为回归。
- 当 `training_loop_service_parts/loop_round_runtime.py` 同时背着 loop read-model sync 和 status-detail 的 capability/tasks-evolution/baseline projection 时，不要继续把这两层混在同一个 part 里。更稳的默认是把 `get_training_queue_status_detail` 及其 capability/task-evolution/baseline helper 抽到独立 `loop_round_status_detail_runtime.py`，并补 dedicated split probe 锁住“main part < 900 行 + support part < 1000 行 + manifest 已挂载 + `enter-next-round / rollback-round-increment` 后的 status-detail 仍返回能力、任务演进和能力基线三块真相”；这样能把 `loop_round_runtime.py` 退出 Mandatory Gate 首批冻结对象，又不会把 training loop 的状态详情 contract 拆坏。
- 当 `assignment_core.py` 这类 assignment 基础 part 同时混着 assignment system settings、execution settings 和 support table/schema bootstrap 时，不要继续把“基础 normalize/ID/runtime 入口”和“系统设置/SQLite schema”硬塞在同一个 part 里。更稳的默认是把这层抽到独立 `assignment_settings_runtime.py`，并补 `verify_assignment_settings_runtime_split.py` 锁住“main part < 900 行 + support part < 1000 行 + manifest 已挂载 + settings get/set 行为仍通”；这样才能让 `assignment_core.py` 真正退出 Mandatory Gate 首批冻结对象，而不是只做文本搬家。
- 当 `policy_confirm_and_interactions.js` 这类 web client runtime 已经开始拆片时，不要把 `interruptCurrentTask`、`renderWorkflowPlan` 这类函数体截断到相邻 bundle part 里靠纯文本串接续命。更稳的默认是把 session task runtime / workflow queue plan runtime 抽成显式 support file，并补 dedicated split probe 同时锁住“main file < 900 行 + support file 单文件 `node --check` 通过 + bundle 仍能整包通过”；否则整包语法可能表面没坏，单文件维护面却会持续处于不可 parse 的假稳定状态。
- 当 `run_acceptance_role_creation_browser.py` 这类超长 browser acceptance runner 进入 Mandatory Gate 首批冻结对象时，不要只在同一个脚本里继续堆 setup / API / assertion / screenshot / summary。更稳的默认是把通用 helper 和收尾 evidence builder 抽到独立 support 模块，再补 dedicated split probe 锁住“runner < 1000 行 + support import + 关键 helper 已迁移”；这样既能让 runner 退出首批冻结对象，也不会把原有 browser acceptance contract 打散到难以回归。
- 当 `run_acceptance_schedule_center_browser.py` 这类超长 schedule browser acceptance runner 进入 Mandatory Gate 首批冻结对象时，不要继续把 HTTP helper、Edge probe、runtime bootstrap、SQLite/event 取证和 markdown report 全堆在同一个 runner 里。更稳的默认是抽出 `schedule_center_browser_acceptance_support.py` 这类 support 模块，并补 dedicated split probe 同时锁住“runner < 1000 行 + support import + 关键 helper 已迁移 + 依赖它的 `run_acceptance_assignment_self_iteration_schedule.py / run_acceptance_schedule_trigger_idempotency.py` 仍可导入”；这样既能把 runner 压出 blocker，又不会把同目录下复用这些 helper 的 schedule acceptance 一起拆坏。
- 当 `run_acceptance_workflow_gate.py` 这类总 gate runner 自己命中 Mandatory Gate 首批冻结对象时，不要继续把 API 请求、session fallback、line-budget gate helper 和 script-probe bridge 全堆在主 runner 里。更稳的默认是抽出 `workflow_gate_acceptance_support.py` 这类 support 模块，并补 dedicated split probe 同时锁住“runner < 1000 行 + support import + 关键 helper 已迁移 + `verify_workspace_line_budget_mandatory_gate.py` 仍能透过 runner 绑定调用 line-budget helper”；这样既能把 gate runner 压出 blocker，也不会把 checker 契约或 gate 自验收链拆断。
- 当 `developer_workspace_service.py` 这类 developer workspace 边界服务只因为 bootstrap/registry helper 累积而命中 Mandatory Gate 时，不要继续把 developer_id 清洗、workspace target 解析、tracking branch 默认值和 registry payload writeback 留在主 service 里。更稳的默认是抽出 `developer_workspace_bootstrap_support.py` 这类 support 模块，并补 dedicated split probe + `verify_developer_workspace_registry_reconcile.py` 一起锁住“main service < 1000 行 + support 已接进 workflow gate probe registry + registry/live boundary reconcile 仍成立”；这样能把主 service 压回门禁线以内，又不会把 developer workspace 的 live 真相写坏。
- 当 `project_registry_service.py` 这类后端 registry/service 因新增项目创建合同、startup readiness 和 runtime policy summary 而命中 Mandatory Gate 时，不要继续把 contract 计算、agent_registry readiness 和 bootstrap summary projection 全塞回主 service 里。更稳的默认是把这组 contract/startup-readiness/runtime-policy helper 抽到独立 support runtime（当前是 `project_registry_contract_runtime.py`），再用 `check_workspace_line_budget.py + verify_project_creation_contract_startup_readiness.py + workflow gate` 一起复跑；这样既能把主 service 压回 1000 行以内，也能保证 `bootstrap/list/task_summary` 合同在 root sync 前就被同一条门禁链锁住。
- 当 `workflow_env_common.ps1` 在 `workflow_env_json_and_paths.ps1` 首刀之后，仍因 runtime config / device-path writeback 和 runtime-upgrade/snapshot helper 混在一起而命中 Mandatory Gate 时，不要继续在主脚本里硬压表达式或只删空行。更稳的默认是再拆成 `workflow_env_runtime_config.ps1` 与 `workflow_env_runtime_upgrade.ps1` 两个 helper，并把 `verify_workflow_env_common_split.py` 升级成“多 helper + dot-source + exported function”合同，同时让 `verify_powershell_script_parse.py` 和 `workflow_gate_probe_registry.py` 一起纳管新脚本；这样才能既把 `workflow_env_common.ps1` 压回 1000 行以内，又不把 deploy/runtime-upgrade 的 PowerShell 绑定拆断。
- 当 `role_creation_service_parts/session_commands.py` 这类 module-parts runner 同时背着 create/message/start/task 主命令和 delete/cleanup/complete 生命周期时，不要继续把生命周期命令塞在同一个 runner 里硬压行数。更稳的默认是把 cleanup/complete 整簇抽到独立 support file，并补 `verify_role_creation_session_commands_split.py + verify_role_creation_session_live_recovery.py` 这一组 probe：前者锁 manifest 装配、函数迁移和 runner 行数，后者兜住 `role_creation_service` 的真实导入与 live recovery runtime，不让“文本拆分成功但 module-parts 绑定悄悄坏掉”混进 clean head。
- 当 `role_creation_service_parts/session_queries_and_internal_tasks.py` 这类 module-parts runner 同时背着会话 list/detail query surface、delete-state projection 和内部任务 helper 时，不要继续把 public query surface 留在同一个 runner 里硬压行数。更稳的默认是把 `list_role_creation_sessions / get_role_creation_session_detail / _role_creation_delete_state` 抽到独立 `session_query_surface.py`，并补 `verify_role_creation_session_query_surface_split.py + verify_role_creation_session_live_recovery.py`：前者锁 manifest 装配、runner/support 行数和 `__code__.co_filename` 绑定，后者兜住 live recovery 这条真正依赖 query surface 的 runtime，不让“拆分后导入面悄悄回退到旧 runner”混进 clean head。
- 当 `assignment_center_state_helpers.js` 这类前端 state helper 同时背着表单/格式化 helper 和整套任务图排序、泳道计算、graph meta / render runtime 时，不要继续把 graph surface 挤在 state helper 里硬压行数。更稳的默认是把这组 compare/lane/render helper 抽到独立 `assignment_center_graph_runtime.js`，并补 `verify_assignment_center_graph_runtime_split.py + verify_assignment_center_view_tabs_assets.py`：前者锁 `assignment_center_state_helpers.js < 1000 行 + graph runtime < 900 行 + bundle_manifest` 装配顺序与函数迁移，后者兜住 tabs / bundle 语法回归；这样才能让当前 blocker 真正退出首批冻结对象，又不把任务中心主入口拆坏。
- 当 `task_artifact_store_queries.py` 这类大文件同时背着内部 stale-recovery helper、却又直接暴露公共 query surface 时，不要继续把对外入口留在同一个 runner 里硬长。更稳的默认是把 `list_assignments / get_assignment_overview / get_assignment_graph / get_assignment_scheduler_state / get_assignment_status_detail` 抽到独立 `task_artifact_store_query_surface.py`，由原 runner 只做 alias，并补 dedicated split probe 同时锁住 `runner` 行数、support 行数和 `assignment_service` 暴露出的函数绑定模块；这样才能既把当前文件压出首批冻结对象，又避免只做文本搬家后导入面悄悄失效。
- 当 `task_artifact_store_run_runtime.py` 这类已经贴着 line-budget 上限的 `*_parts` 文件，需要再补 request object/compat runtime 来推出当前质量债时，不要把 dataclass/coerce/helper 全继续内联回原文件。更稳的默认是把这组 request-object 支撑抽到独立 support runtime（当前是 `assignment_codex_failure_runtime.py`），让主 part 只保留用例入口和显式 import，再用 targeted contract probe + line-budget report 一起锁住“旧债出队且主文件不反弹过线”；否则你会先把质量队列前进一步，再立刻被 Mandatory Gate 反咬成新 blocker。
- 当 `platform_interfaces_center.js` 因追加 live regression probe runtime 被顶过 `1000` 行硬门禁和 `1200` 行前端 guideline 时，不要继续把 probe helper 堆回主模块里。更稳的默认是把 `ensureInterfaceCenterProbeOutputNode / runInterfaceCenterProbe` 这一整簇 probe runtime 抽到独立 `platform_interfaces_probe_runtime.js`，并把新文件接进 `bundle_manifest.json`；这样既能让 `platform_interfaces_center.js` 立刻退回 Mandatory Gate 以内，也不会把 `verify_api_catalog_live_regression.py` 的 DOM contract 打断。
- 当某条 active-version targeted regression 已经写成独立 probe、却还没正式接进 `workflow gate` 时，不要把“脚本存在”误当成“门禁已经覆盖”。更稳的默认是让这条 probe 自己同时断言 `workflow_gate_probe_registry.py` 里存在对应 `key / relative_path`，再把 probe 正式接进 registry；这样可以先稳定打出“功能在、但 gate 还没绑定”的红灯，再在注册后用同一条 probe 转绿，避免版本切片长期停在“有脚本但没有持续执行”的假完成。
- 当 `project_ops_live_regression` 这类默认指向真实 host 的 live probe 需要正式接进 `workflow gate` 时，不要只补 registry。更稳的默认是让 probe 自己同时断言 registry 绑定，并优先从 `WORKFLOW_GATE_PROBE_HOST / PORT` 读取 isolated listener、在 gate 场景下允许 fixture bootstrap；否则脚本会继续误打 live `8090` 或在 isolated gate 里找不到 quiet project，把“已接 gate”做成假绿。
- 当 `project_ops_center.js` 这类前端核心工作面已经卡在 `1000` 行硬门禁边上时，不要继续把 create wizard、产出图表达和事件处理整包硬塞回原文件。更稳的默认是先抽一个 dedicated `project_ops_surface_runtime.js` 承接详情/创建 surface，再让 `bundle_manifest.json` 明确把 helper 排在 shell 前面，并把 `verify_project_ops_module_ui.js` 升级成按 manifest 顺序同时加载 surface runtime + center shell；否则你会先被 Mandatory Gate 卡死，再被 UI probe 因 helper 未加载而打成假红灯。
- 当 `measure_assignment_dashboard_latency.py` 这类原本只适合手工点测的脚本，需要正式进入 `workflow gate` 时，不要继续把 `runtime-root / agent-search-root` 留成纯人工参数，也不要只打印一次 point sample 就算完成。更稳的默认是让脚本优先从 `WORKFLOW_GATE_RUNTIME_ROOT` 与 runtime-config 解析运行上下文、把多次采样结果稳定写进 `TEST_ARTIFACTS_DIR/assignment-dashboard-latency/summary.json + summary.md`，再用单独的 verify probe 去断言 registry 绑定和 `p50 / p95 / p99`；否则 gate 只能复用一段临时命令，`V8-R5` 这类性能切片会长期停在“脚本存在但没有持续门禁”的假完成。
- 当 `project_task_summary` 这类项目 contract 读面明明已经被一级模块和 live dashboard 消费时，不要继续只靠 UI 手看或单次 live proof 兜底。更稳的默认是把 dedicated contract probe（例如 `verify_project_task_summary.py`）正式接进 `workflow gate`，并在 probe 里同时断言关键字段和 registry 绑定；否则这类 `project_type / project_goal / project_board_ref / runtime_policy_ref` 漏字段问题会在版本 freeze 之后继续潜伏成“页面能开，但 contract 其实不完整”的假稳定。
- 把 advisory 升级成强制门禁后，验证不能只看 fixture。除了定向 fixture probe，还要在真实 workspace 上直接跑一遍 `check_workspace_line_budget.py`、`run_acceptance_workflow_gate.py` 和 `run_acceptance_runtime_release_gate.py`；真实返回 `1` 本身就是“门禁已经真的开始拦”的证据。
- 当真实 workspace 早就有大量历史 offender 时，强制门禁落地后的首轮验证大概率应该失败。这不是回归，而是治理口径从“只报不拦”切到“正式阻塞发布”的预期结果；版本计划、发布约束和运行报告都要同步改口径。

## 已踩过的坑
- 坑 1：只把 checker 改成 `return 1`，却忘了 `run_acceptance_workflow_gate.py` 和 `run_acceptance_runtime_release_gate.py` 仍用 `hard_gate_pass` 判定，结果“看起来更严了”，实际发布链路还是没变。
  - 避免方式：强制门禁必须同时改三处：checker、workflow gate helper、runtime release gate helper。
- 坑 2：把真实 workspace 上的预期失败当成“测试没过”，从而又退回 advisory 口径。
  - 避免方式：先在版本计划里把这批治理排成当前 active requirement，再把真实 fail 解释成“强制门禁已生效”的发布阻塞证据，而不是产品回归。
- 坑 3：`Mandatory Gate` 已经开始拦，但“这轮到底先拆哪几个文件”只存在 PM 口头判断里，导致后续巡检、版本 history 和 gate detail 会重复冻结同一组对象，甚至彼此漂移。
  - 避免方式：把当前冻结首批稳定写回 `first_batch_targets`，并让 line-budget detail 和版本执行约束都从同一份真相源读取。
- 坑 4：把 `schedule_service` 的 row/trigger/projection 真逻辑迁进 `schedule_query_runtime.py` 后，如果不继续把读面投影层拆出来，新模块自己就会从 `800+` 很快长到 `1200+`，Mandatory Gate blocker 只会从旧文件平移到新文件。
  - 避免方式：把“底层查询/enrich”和“列表/详情/日历 projection”视为两层职责，前者留在 `schedule_query_runtime.py`，后者另建 `schedule_projection_runtime.py`，并用现成的 schedule preview/detail/text-repair probes 锁住。
