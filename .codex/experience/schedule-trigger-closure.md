# 定时触发闭环经验

## 适用范围
- `定时任务 -> 任务中心` 命中建单
- pm / `workflow` 周期性唤醒与自迭代
- schedule trigger 的失败恢复与回写

## 稳定经验
- 定时 worker 命中后不要同步等待“建节点 + 派发 + 结果回写”整条链完成。应先把 `trigger_hit` 落盘，再后台继续做建单/派发/回写；否则一次慢建单就会把 `scan` 请求和 worker 一起拖死。
- schedule 真相源不能只记“命中过了”，必须尽快补齐：
  - `assignment_ticket_id`
  - `assignment_node_id`
  - `last_trigger_at`
  - `last_result_status`
  否则计划明明已经在任务图里建出节点，schedule detail 仍会显示“待触发”，用户无法继续追踪。
- 半成功现场要允许恢复。只要任务图节点已经带有 `source_schedule_id + trigger_instance_id`，后续扫描或 detail 读取就应能反查这些字段，把 schedule trigger 回写补齐，而不是一直卡在 `trigger_hit`。
- 只靠 `scan` 或手工打开 detail 做恢复不够稳。worker 本身也要周期扫最近的 `trigger_hit/queued/running/dispatch_failed` 非终态 trigger，把丢失的后台处理线程重新挂起来；否则计划会长期卡在“已建单待调度/运行中”，只有人工点开页面时才继续推进。
- 当同一张 assignment graph 里已经有别的 `running` 节点时，recovery worker 不该反复重挂另一个只是 `queued/ready` 的 trigger。更稳的做法是让它安静等待当前运行槽释放，再由成功收尾或下一轮恢复去继续 dispatch；否则 `dispatch_requested -> trigger_resume_requested -> recover_assignment_node` 会反复刷爆 `schedules.jsonl`，还会把“其实只是排队等待”伪装成“恢复失败”。
- schedule trigger 恢复不能无差别地把所有 `dispatch_failed` 一次性重放。像 `[持续迭代] workflow` 这种被 `smoke baseline guard` 挡下来的 trigger，如果每轮都立刻重试，会把 `schedules.jsonl` 刷爆，还会把真正需要优先恢复的 `queued` trigger 挤掉。恢复顺序应该优先 `trigger_hit/queued/running`，`dispatch_failed` 至少要加冷却窗口。
- 如果把 schedule trigger / recovery worker 从 `schedule_service.py` 拆到独立 runtime 模块，恢复类 probe 仍可能继续 monkeypatch `schedule_service._start_schedule_trigger_processing`、`_request_assignment_dispatch` 这类 service 入口。更稳的默认是：split 模块内部对这些 hook 走 runtime-bound service symbol，而不是硬调模块内私有函数；同时像 `_find_schedule_assignment_ref` 这类 assignment bridge wrapper 不要再回指 split 模块自身，否则 probe 会拦不到，或直接绕成递归。
- schedule trigger 恢复在同一轮里不能并发起太多线程，尤其是这些 trigger 都指向同一张全局任务图时。一次性恢复多条会把任务图快照、run 记录回写和 work-record index 一起挤到同一把 SQLite 锁上，最终表现成 `database is locked`、`/api/status` 超时和恢复线程互相放大。更稳的默认是每轮只启动极小批次恢复，再由下一轮继续补。
- 对同一条 `[持续迭代] workflow` 来说，旧的 backlog trigger 没必要按历史顺序全部补跑。worker 恢复更稳的口径是“每个 schedule 只恢复最新的一条非终态 trigger”，否则昨天下午的旧唤醒、旧自迭代会重新塞满 ready 队列，把真正最新的一轮挤到后面。
- 对同一条 schedule 来说，如果新的命中到来时旧 trigger 还停在“待执行/待派发”，默认不应该继续把 backlog 越堆越多。更稳的口径是：同一 schedule 只保留最新的一条 pending trigger；旧的 pending 节点直接删掉，并把旧 trigger 标成 `superseded`，但已经 `running` 的旧节点保持不动。
- schedule-trigger 节点创建不能再为了返回一个 created node，把整张全局主图重新完整序列化一遍。像 `2026-04-25` 这轮现场里，`[持续迭代] workflow` 的 `trigger_hit -> create_assignment_node` 在大图上会被这条重路径拖成分钟级，后续 trigger 只能在同一把 schedule processing lock 后面排队，表面看起来就像“主线命中了但不继续建单”。更稳的默认是：schedule-trigger 节点创建只构造 node-only fast path，图级 overview 只复用 metrics/node_map，不再为这一条 ready 节点强制 materialize 全量 `serialized_nodes`；同时用 dedicated probe 锁住“不再请求 full response snapshot serialization”。
- schedule future 的 priority update 不会 retroactively 改掉已经创建好的 `ready` 节点。若 live 现场已经出现历史 `mainline=P2 / patrol=P1` 这类错序，光改 schedule 主表还不够；更稳的默认是收口 live 队列，删除错误顺位的旧 patrol ready，并确保只剩 `[持续迭代] workflow` 或 durable handoff 作为下一棒出口。
- `workflow` 主线当前只保留 `[持续迭代] workflow=P1` 这一条用户可见推进主线；旧的 `pm持续唤醒 - workflow 主线巡检` 不再作为 live 保底计划存在。后续不要为了兼容历史字段再恢复 `patrol` 读面、P2 巡检优先级或 `workflow-pm-wake-summary` 产物。
- 即使已经把主线默认 priority 修正到代码里，历史 `schedule_plans` 行也可能继续带着旧值跨版本存在；如果桥接层继续直接照抄 `schedule.priority`，升级后仍可能再次建出错序 ready 节点。更稳的默认是：
  - schedule create/update 时对 `[持续迭代] workflow` 强制 priority 归一
  - `schedule -> assignment` bridge 建节点时再补一层 canonical priority 兜底
  - live 现场若已经 materialize 出错误 patrol ready，要同步删除这条旧节点并退出旧 patrol schedule
- 如果用户要求“先把 7x24 停住，等下一台设备再恢复”，不能只把 `[持续迭代] workflow` schedule 设成 `enabled=false`。已经命中过并落成 `ready` 的节点仍会继续执行；更稳的默认是先确认当前没有真实 running，再同时删除未开始的 ready 节点，并把 `assignment_mainline_handoffs` 中对应接棒保持为暂停态，最终以 `running_task_count=0`、`queued_task_count=0`、assignment graph `active.total_count=0` 三处一起收口。对 live API 的节点删除不要把超时卡太短，否则会留下“审计已写、节点文件还没收完”的假完成。
- 当 `workflow` 自迭代节点不是业务失败，而是被系统按“运行句柄缺失 / workflow 已重启”收口时，也要像正常失败一样续挂下一轮 `[持续迭代] workflow`。否则站点虽然能被 watchdog 拉回，但 7x24 连续推进会在这类重启现场悄悄断链。
- `create_node` 和 `dispatch` 也必须对 `trigger_instance_id` 做幂等保护。否则同一 trigger 会生成多个 assignment node，甚至把 dispatch 打到与计划详情回写不同的节点上，最终留下“计划详情还指着 ready 节点，真实执行却跑在另一个节点”的错位现场。
- 生产 smoke 验收时，要把用户可见的 `GET /api/schedules/{id}` 详情当成 done_definition 的真相源，而不是只看本地 raw detail。现网里 raw detail 可能还停在 `queued`，但 HTTP detail 已经把 assignment/result 富化到 `running/failed`；同时接口本身也可能有瞬时抖动，所以验收脚本应采用较长超时和重试，并把 raw/http 偏差记成警告而不是直接误判失败。
- 手工核任务图时，不能只对 `tasks/.../nodes/*.json` 直接 grep `status=ready/running`。节点文件即使已经 `record_state=deleted`，也会保留原来的 `status` 文本；更稳的默认是先按 `record_state=active` 过滤，再看 `status`，否则像历史孤儿 `ready` 节点也会被误当成 live 待派发节点。
- schedule calendar 的验收不要把“今天已经过去的一次性触发时间”直接断言成 `plans` 里仍然可见。`get_schedule_calendar` 对当天只展示 `_now_bj()` 之后的 future candidates；如果测试要同时断言 `plans` 和 `results`，应冻结 `_now_bj()` 或把样例触发时间放在未来，否则会把已经修好的 list/detail 误报成 calendar 失败。
- `schedule -> assignment` 桥接层要给同一 `trigger_instance_id` 固定一个稳定 `node_id`，并优先在已知全局主图 ticket 内做恢复扫描；不要每次都全局扫所有任务目录，更不要把“随机新建节点”当成默认恢复策略。
- `run_schedule_smoke_baseline` 这类自测入口不能直接内联调用 `_process_schedule_trigger_async` 并换一把 thread key。它必须复用正常的 `_start_schedule_trigger_processing` 门闩，否则同一 trigger 会绕过进程内锁，重新触发建单/派发竞争。
- 读取任务中心全局主图时，若数据库里已经绑定了 canonical ticket，就先直接验证并复用该 ticket；不要每次 list/dashboard 都重新全量扫描所有 workflow-ui 图和全部节点，否则面板很容易从“数据存在”退化成“接口超时后显示 0”。
- 如果要做 `workflow` 的长期自迭代，不要在 assignment 完成后立即递归再派发一条同类节点；更稳的做法是把“下一轮”写成一个滚动的一次性 schedule，让系统在几分钟后再唤醒，既不断链，也避免瞬时热循环。
- 对 pm/`workflow` 这类自迭代计划，默认要走低频、分钟粒度、短上下文、低风险目标；不要一上来就做高频自治。
- 运营项目的 controller starter route 成功，并不等于 controller cadence closure 已成立。像 `project-comics-smoke / controller_role_id=workflow_devmate` 这类现场里，如果 self-iteration 只对白名单 `workflow` 生效，starter route 虽已 `succeeded`，也不会续挂 `[持续迭代] workflow_devmate`，`next_handoff_interval_effective_after_run` 也永远空着。更稳的默认是：self-iteration eligibility 同时接受项目 `controller_role_id`，controller 续挂使用项目主控专用 prompt，不复用 PM prompt，并用 dedicated probe 锁住“项目 cadence -> future schedule -> effective_after_run 回写”这一整条闭环。
- 低频不等于要等到半小时级。对当前 `workflow` 的 7x24 主线，更稳的默认是健康态十几分钟级、失败态约 `10` 分钟；异常恢复直接重排 `[持续迭代] workflow`，不再另挂低频用户可见巡检线。这样改规则后的新口径能在短时间内被下一棒重新读取，同时避开秒级热循环。
- 如果 `PM版本推进计划.md` 已经把 `4.6.1 当前现场更新` 收成“最新快照 + 外部总览引用”，自迭代/保底提示里不能只笼统写“更新版本计划”。更稳的默认是把约束写死成：只允许覆盖 `4.6.1` 的单份最新快照，详细时序现场外提到月度现场总览；否则 7x24 很容易又把主计划正文写回成长流水。
- 已经终态且 `once` 规则也耗尽的旧计划，不能继续留在 active schedule 视角里。否则 detail 可能已经显示 `succeeded/failed`，但 dashboard / preview / schedule_total 仍把它算作启用中的活跃计划。更稳的做法是读取最新 trigger 已终态且 `next_trigger_at` 为空时，同步修正 `schedule_plans.enabled` 与 `last_result_*`，让 detail、list、preview 三处共用同一份真相。
- `[持续迭代] workflow` 这类自迭代 schedule 如果挂在 `smoke baseline guard` 后面，可能在计划命中后直接被拦成 `dispatch_failed`，即使版本计划和唤醒文案都已经写对。要保持 7x24 连续推进，不能只依赖这一条滚动 schedule，还要保留一条更低频的 pm 保底唤醒，必要时允许 PM 直接往任务中心补一条当前版本任务。
- 当治理口径已经决定不再把 `pm持续唤醒 - workflow 主线巡检` 展示成用户可见的长期保底任务时，失败恢复不能继续创建/续挂这条 legacy patrol schedule。更稳的默认是：smoke block / dispatch_failed / stale recovery 直接 update 原 `[持续迭代] workflow` 一次性计划，按失败延迟重排主线，priority 固定 `P1`，并用 dedicated probe 断言不再产出 `workflow-pm-wake-summary` 或 `主线巡检` schedule。
- 禁用 legacy patrol schedule 本身还不够。像 `2026-04-25 15:40 +08:00` 这轮现场里，`pm持续唤醒 - workflow 主线巡检` 已经命中并留下 `ready` 节点；如果只把 schedule 设为 disabled，这个 ready 节点仍会在当前任务结束后被派发，继续抢主线。当前稳定做法是：删除旧巡检创建与读面兼容字段，清掉遗留 ready/queued 节点，后续接力只看 `[持续迭代] workflow` 与 `assignment_mainline_handoffs`。
- 调整 `[持续迭代] workflow` 的 assignment prompt 时，要同步检查 durable handoff 和失败重排路径是否仍引用同一套推进口径；不要再去维护 `pm持续唤醒` 的旧 prompt 分支。
- 自迭代补链不能只在 assignment 正常完成路径里续挂。像 `smoke guard`、`dispatch_failed`、`stale running/运行句柄缺失` 这类失败收口，也必须写入 durable handoff 或直接重排 `[持续迭代] workflow`；不能恢复成另补一条 `pm持续唤醒 - workflow 主线巡检`。
- 旧巡检计划曾因 `once`、`node_goal too long` 和 prompt 分叉制造过静默断链；当前结论不是把它升级成 daily，而是彻底退出用户可见主线，统一交给 durable handoff + 原主线失败重排承担连续性。
- `schedule -> assignment` 桥接层不能把 `workflow` 主线的整套 prompt 直接塞进 `node_goal` 再硬截到 `3800`。像 `2026-04-19 05:14 +08:00` 这轮现场里，旧实现会把 `V4-R5` 的 judgment-first、`version_transition` 和 `done_definition` 关键约束直接截掉；更稳的默认是：bridge 先把 schedule text repair 成最新合同，再生成 compact assignment goal，并且 execution prompt 对 workflow schedule node 再补一层 schedule-goal 重建。这样即使 `ready` 节点是在旧版本里建出来的，只要它在新版本里真正启动，也能补吃到新的 prompt 合同。
- 只修 `node_goal` 和旧 `ready` 节点的 execution-time rebuild 还不够。像 `2026-04-19 06:27 +08:00` 这轮现场里，mainline 的 `run.prompt.txt` 仍可能沿用节点创建时的 `prod=20260419-052046 / ahead_dirty` 旧快照，因为 dispatch 阶段组 prompt 还在直接抄 `node_goal`。更稳的默认是：workflow schedule 节点在 dispatch materialize prompt 时，也要优先回读 live schedule detail，再落 `prompt.txt`；同时用定向 probe 锁住“旧 node_goal 被新 baseline / release-boundary 覆盖”的合同。
- 如果 workflow prompt rebuild 依赖 live schedule detail，不能把 `get_schedule_detail()` 当成平铺 payload 直接取 `schedule_name / launch_summary / execution_checklist / done_definition`。真实返回结构是 `{schedule: {...}, future_triggers: [...], recent_triggers: [...]}`；若忘了先解包 `schedule`，dispatch / execution-time rebuild 会静默回退到旧 `node_goal`，即使 schedule detail 已经把 baseline / release boundary 刷到新真相。更稳的默认是：读取 live schedule detail 时先兼容嵌套 `schedule` 结构，并用 probe 同时锁住 flat payload 与 detail payload 两条路径。
- 只让 workflow schedule 节点的 provider prompt 吃到 refreshed schedule goal 还不够。像 `2026-04-19 07:25 +08:00` 这轮现场里，`run.prompt.txt` 已经读到新的 `prod=20260419-073148 / clean_synced`，但 `status-detail.selected_node.node_goal` 仍停在旧的 `prod=20260419-065255 / ahead_dirty`，会继续误导 PM 和人工排障。更稳的默认是：dispatch 准备 execution run 时，把同一份 refreshed workflow schedule goal 同步回写到 node snapshot，再生成 `prompt.txt`；并用 probe 锁住 refreshed node payload 的 `node_goal / updated_at` 合同，避免只修 prompt 不修节点快照。
- 自迭代 `previous-result` 的脱敏也不能只替换真实 version/head。像 `2026-04-19 09:19 +08:00` 这轮现场里，上一轮 `result_summary` 已经把旧口径写成字面量 ``@<head> / <version>``，还顺带带着 `memory_ref=.codex/memory/...` 长路径；如果 compact summary 只认真实版本号和 commit，新一轮 schedule launch_summary / node_goal 就会把这些模板壳和截断路径原样带回 live prompt。更稳的默认是：compact summary 同时替换 literal `<head>/<version>` 和 `memory_ref/daycut_ref` 路径；即使上游摘要已经先把 `.codex/memory/...` 截断到缺少 `.md`，也要继续把这类半截路径收成 `今日日记`，再用 probe 把 launch_summary 与 schedule node goal 一起锁住。
- `smoke baseline guard` 不能只按 `assigned_agent_id=workflow` 粗暴套到所有计划上。真正需要被 smoke 约束的是 `[持续迭代] workflow` 这类滚动自迭代计划；失败恢复应回到原主线重排或 durable handoff，不再依赖 `pm持续唤醒` 这种用户可见保底入口。
- 如果 live prod 上 `POST /api/assignments/<ticket>/dispatch-next` 在 ready 节点恢复场景里长时间不返回，不要连续重试把同一节点重复派发。先核 `audit.jsonl` 和 `run.json` 是否完全没有新的 `dispatch`/`run_id`；若只是 HTTP 壳卡住、但当前源码工作区已经验证过同一逻辑，可临时用本地 `workflow_app.server.bootstrap.web_server_runtime.dispatch_assignment_next(root, ...)` 直连当前 prod runtime 做一次止血派发，然后立刻回核 `node status / run_id / future trigger` 三处真相。
- 如果为了现场止血，必须从一次性本地 Python 进程里直连 `workflow_app.server.bootstrap.web_server_runtime.dispatch_assignment_next(root, ...)`，不能继续让真正的 execution worker 线程保持 `daemon=True`。否则宿主进程一退出，就会稳定留下 `run.json=starting`、`provider_pid=0`、`events.log` 只有 `dispatch` 的假运行，后续 schedule 还会被这条脏 `starting/running` 误挡住并发槽。更稳的默认是：`pm-manual-recovery` 这类一次性恢复入口改成非 daemon，或者进一步演进成独立进程执行；同时 worker 顶层必须有 fail-closed guard，把 `provider_start` 前异常显式收口。
- schedule -> assignment 桥接层如果遇到的是“graph 并发槽满 / 同 agent 仍在运行”，不能继续把返回文案压成统一的 `dispatch_requested` 或 `resume_scheduler_requested`。要把真实等待原因透传到 schedule detail，这样现场才能区分“需要恢复的半断裂 trigger”和“其实只是正常排队中的 queued 节点”。
- file-based stale recovery 不能在没有串行门闩和终态预落盘的前提下，直接挂在 `/api/status` / `/api/schedules` 这类高频读链上做副作用。否则多个读请求会同时看到同一份旧的 `node.json=status=running`，反复写出 `recover_stale_running` 与 `schedule_self_iteration`，直到某一次终于把 node 文件改成终态。更稳的默认是：按 `ticket_id + node_id` 串行 stale recovery，锁内先重读最新 node 文件；一旦确认真 stale，就先把 node terminal 状态落盘，再写 audit 和续挂 schedule。
- assignment finalize 的“收尾后继续派发下一条 ready”不能在 stale recovery / thread-start failure 这类已经持有同 ticket dispatch 门闩的路径里递归执行。否则 `_finalize_assignment_execution_run()` 尾段再次调用 `dispatch_assignment_next()` 时，会把同 ticket 的非重入锁直接卡死，现场表现成 schedule 还在命中、ready 节点持续堆积、真实 workflow run 却停在上一轮。更稳的默认是：恢复态 finalize 只负责收口，follow-up dispatch 改到锁外或外层调度点执行，并把 blocked/failed follow-up dispatch 明确写审计。
- 看门狗不能只看“还有 future 入口”或“recent trigger 里写着 assigned agent already has running node”就判健康。对 `workflow` 这类 7x24 主线，更稳的口径是：只要出现 `running_task_count=0`、ready 节点已到时堆积，且找不到对应 live `run.json/events.log`，就要直接判定为假健康并把假运行收成失败/重派发原主线；不要再产出第二条巡检主线。
- `workflow` 主线 7x24 的核心接力不能继续分散在 schedule trigger、assignment finalize、stale recovery、ready dispatch recovery、prod upgrade watcher 和 smoke guard 多段隐式副作用里。像 `2026-04-25 20:03 +08:00` 这轮现场，主线 run 已终态，但正常 finalize 未完成，`recover_terminal_run_truth` 又因 `ticket_lock_unavailable_bypassed` 没能续排；后续手工 schedule 虽到点，却卡在 `trigger_hit -> assignment node` 桥接，再到 `dispatch -> starting/provider_pid=0`。更稳的默认是把“任意终态必须生成下一棒”收成一条 durable handoff outbox：终态写回同事务落 `handoff_pending`，独立幂等 worker 只负责 `ensure next node/run or explicit pause gate`，schedule 只做外部定时入口，不再承载主线连续性唯一真相。
- durable handoff 已把下一棒写成 rolling once schedule 后，还不能让通用“已终态且 once 耗尽则退休”的读模型直接套到 `[持续迭代] workflow` 上。像 `2026-04-26 06:47 -> 11:30` 这类现场里，handoff 行已经是 `scheduled`，但 worker 如果错过 `next_trigger_at` 那一分钟，普通 once repair 会把主线投影成 `enabled=false / next=''`，后续就不再补命中。更稳的默认是：自迭代 schedule 不参与通用 once retirement；schedule scan 对已过期但未落 trigger 的自迭代 `next_trigger_at` 做 catch-up hit，并用 handoff outbox probe 同时断言 detail/list/preview 仍保持主线 active。
- schedule worker 恢复 terminal trigger 时，不能只把 `schedule_triggers` 和 `schedule_plans` 修成 `succeeded/failed`。像 `2026-04-28` 这轮现场里，`[持续迭代] workflow` 的节点和 run 已经成功终态，但缺少 `assignment_mainline_handoffs`，导致 plan 显示成功却没有下一棒。更稳的默认是：终态 trigger recovery 必须从同节点最新 terminal run 回填 durable handoff，并立即 drain 到下一次 rolling once schedule；用 `verify_schedule_terminal_trigger_progress_recovery.py` 断言 trigger 终态、handoff scheduled、plan next_trigger_at 三处一起成立。
- ready-dispatch recovery 不能在 schedule worker 主循环里同步调用 `dispatch-next`。如果 dispatch API/锁路径卡住，现象会是 `/healthz` 仍正常，但 `schedules.jsonl` 长时间没有新事件、ready 节点长期堆积，手工 `dispatch-next` 也会超时。更稳的默认是：ready-dispatch 恢复只在主循环里完成老化判断和 `ready_dispatch_recovery_requested` 留痕，然后按 `root + ticket_id` 启一个带 in-flight guard 的后台线程执行真实 dispatch，并补写 `completed/failed` 事件；验收要用慢 dispatch stub 断言 `_resume_pending_schedule_triggers` 能快速返回。
- `workflow` 主线的 durable handoff 不能再只回落到“15 分钟后的 rolling once schedule”。当用户已经明确要求“两个任务之间最短间隔不应大于 30 秒”时，更稳的默认是：`workflow_pm` 的 success/failure terminal handoff 先确保真实 `[持续迭代] workflow` schedule 行存在，再立刻写一条当前分钟的 trigger，直接 materialize 下一棒 `ready/running` 节点，同时让 `/api/schedules` 继续保留 `[持续迭代] workflow` 这条真相。这样既满足快速接棒，也不打破 schedule list/detail、terminal trigger recovery 和 self-iteration acceptance 的既有合同。
- 任何涉及 canonical workflow-ui 全局主图或 self-iteration 的 acceptance，如果不显式隔离 `artifact_root/task_artifact_root`，都可能误复用工作区外已有的 canonical graph，表现成“probe 里 handoff next_trigger_at、node_id、ticket_id 看起来随机串台”。更稳的默认是：这类 probe 一律在 `state/runtime-config.json` 写入独立 `artifact_root + task_artifact_root`，再创建自己的 fixture graph；否则回归结果不可信。
- 如果现场只看到 `create_schedule -> trigger_hit -> create_assignment_node -> dispatch_requested`，但 `node` 仍停在 `ready` 且没有 `run.json / provider_start / token` 迹象，不要把它判断成“执行很慢”。更稳的默认是先判定为调度层只完成了建单和派发请求，执行层根本没起跑，再去查 `resume_scheduler_requested`、provider 启动门闩和 ready-dispatch 恢复链。

## 已踩过的坑
- 坑 1：schedule worker 在 `trigger_hit` 之后同步调用慢建单/慢派发，HTTP `scan` 超时了，但 trigger/plan 状态还没来得及更新，现场就会留下“节点已建出来，schedule detail 却还是 pending”的半断裂状态。
  - 避免方式：把建单/派发改成后台线程；trigger 命中只负责落最小必要留痕。
- 坑 2：prod 重启后，已经跑起来的自迭代节点可能被任务中心按“运行句柄缺失”回收成失败；如果 schedule 没有恢复逻辑，就会继续显示空 assignment refs。
  - 避免方式：后续扫描对 `trigger_hit/queued/running` 且 assignment refs 为空的触发实例做恢复检查，优先从任务图节点里的 `source_schedule_id/trigger_instance_id` 反查补齐。
- 坑 3：异步补偿或重复调度如果没有按 `trigger_instance_id` 去重，同一条一次性 smoke trigger 会被连续建出两个节点，计划表头和 trigger 明细又只保留最后一次回写，结果就是页面上显示的 assignment node 仍是 `ready`，而真正被 dispatch 的已经是另一条节点。
  - 避免方式：`create_node` 前先反查同 trigger 的既有节点；`dispatch` 前确认计划/trigger 指向的节点和目标节点一致，否则先修正回写再派发。
- 坑 4：smoke baseline 为了“补一脚恢复”而直接内联调用 `_process_schedule_trigger_async`，还故意拼了一个 `::smoke-inline` 的新 thread key，结果同一 trigger 的后台线程门闩完全失效，重复建节点和重复 dispatch 都会被放大。
  - 避免方式：smoke 和正式 scan 一律只走 `_start_schedule_trigger_processing`；如果需要补偿恢复，也要复用同一个 `trigger_instance_id` 对应的 thread key。
- 坑 5：dashboard / assignments 每次都为了解析 canonical workflow-ui 全局主图而重新全量扫描所有候选图和全部节点，在历史任务堆起来后会把接口拖到 20 秒级，前端最终表现成“活跃小伙伴 0、运行中任务 0、定时任务像是没有”。
  - 避免方式：先走已绑定 ticket 的轻量验证快路径；只有绑定缺失或失效时，才退回全量候选扫描。
- 坑 6：prod 恢复后，[持续迭代] workflow 虽然能成功命中，但如果当前 smoke baseline 保护条件没过，trigger 会直接写成 `dispatch_failed / smoke baseline last run not passed`，结果是计划看起来在跑、实际上没有给 PM 建出下一轮任务。
  - 避免方式：给 `workflow` 额外保留一条低频 pm 保底唤醒；同时在需要时允许 PM 直接往任务中心补一条带版本计划上下文的 `P0 ready` 任务，而不是死等滚动 schedule 自己穿过 smoke guard。
- 坑 7：如果 guard 只看 `assigned_agent_id=workflow`，历史 `pm持续唤醒 - workflow 主线巡检` 也会被拦成 `dispatch_failed`，表面上 future trigger 还在，实际上能真正补链的入口已经失效。
  - 避免方式：把 guard 范围收窄到真正的 `[持续迭代]` / `continuous-improvement-report.md` 计划；当前版本不再恢复保底唤醒，而是用 durable handoff 与原主线失败重排补链。
- 坑 8：如果只在 `run_schedule_scan` 里补偿 trigger，worker 一旦在 `trigger_hit -> create_node -> dispatch_requested` 之间重启，旧 trigger 会一直停在 `queued/running` 的中间态；后续虽然能在任务图里看到 node，但计划详情和下一轮接力都不会自动往前走。
  - 避免方式：让 worker 每轮都额外扫一遍最近的非终态 trigger，并基于 `trigger_instance_id` 重挂 `_start_schedule_trigger_processing`；终态 `succeeded/failed` 的 trigger 则直接跳过，避免无意义重试。
- 坑 9：如果只在 `[持续迭代] workflow` 的“正常完成”路径里接棒，一旦现场落到 `smoke guard`、`dispatch_failed` 或“运行句柄缺失”这种失败收口，计划表面上可能只剩一个已结束节点，真正的 future trigger 却已经空了。
  - 避免方式：把失败收口也写入 durable handoff 或直接重排 `[持续迭代] workflow`；验收至少覆盖“smoke blocked”和“stale running recovered”两条场景，不再通过旧巡检计划补链。
- 坑 10：当 live prod 的 `dispatch-next` HTTP 壳卡住时，直接用一次性本地 Python 进程调用 `dispatch_assignment_next(root, operator='pm-manual-recovery')` 虽然会立刻写下 `dispatch` 审计和 `run_id`，但如果 execution worker 线程仍是 daemon，宿主进程退出后线程会被直接带死；现场就会留下 `starting/provider_pid=0/events 只有 dispatch`，并在几十分钟后被 stale recovery 回收成“运行句柄缺失”。
  - 避免方式：`pm-manual-recovery` 默认改成非 daemon 线程，至少保证一次性宿主不会在 worker 入口前提前退出；同时给 worker 顶层补 `provider_start_failed -> finalize failed` 的显式收口，避免再出现静默 limbo。
- 坑 11：如果只修 trigger 行而不修 `schedule_plans` 主表，历史 `once` 计划即使已经 `succeeded/failed` 且没有未来触发，也会继续以 `enabled=1` 混进 dashboard / schedule preview，导致 `schedule_total` 和 active 列表持续偏大。
  - 避免方式：读取最新 trigger 已终态且 `next_trigger_at=''` 时，同步把该计划从 active 视角退场，并新增定向验收覆盖 detail / preview / 数据库行三处一起收口。
- 坑 12：当 `task_artifact_store_queries._assignment_reconcile_stale_task_state_internal()` 直接在高频 GET 读链里做 stale recovery，而 recovery 前又没有节点级串行锁和“先落 `node.json=failed`”的动作时，同一 stale node 会被多个读请求重复回收，现场表现成 `run.json` 早已 `cancelled`，但 audit 里持续刷 `recover_stale_running`，还会重复续挂同一条自迭代 schedule。
  - 避免方式：按 `ticket_id + node_id` 做 file-based stale recovery 串行化；锁内先重读 node 真相，若仍是 `running` 才执行 recovery，并把 node terminal 状态先落盘，再放出 audit / schedule side effects。
- 坑 13：如果只更新 assignment finalize 路径上的 `[持续迭代] workflow` 提示，而旧 `pm持续唤醒` 创建/读面兼容壳还留在代码里，主链一旦走到历史分支，PM 就会重新收到旧口径的巡检任务，结果 7x24 只维持活跃不推进版本。
  - 避免方式：凡是调整 PM 的 7x24 版本推进职责，都同步检查并删除旧巡检创建、读面字段和前端展示兼容壳；gate 要覆盖不再产出 `workflow-pm-wake-summary`、`主线巡检`、`is_workflow_patrol` 等历史信号。
- 坑 14：如果 stale recovery 在 `dispatch_assignment_next()` 已持锁的上下文里直接调用 `_finalize_assignment_execution_run()`，而 finalize 尾段又无条件递归再调一次 `dispatch_assignment_next()`，同 ticket 的 dispatch 非重入锁会被自己卡住。日志上常见的假象是：最后一条真实 run 已在几十分钟前结束，后续 schedule 仍持续 `trigger_hit -> create_assignment_node`，但再也没有新的 `dispatch_requested/run_id`。
  - 避免方式：给 finalize 增加 `suppress_followup_dispatch` 这类显式开关；恢复态和启动失败态 finalize 默认禁止递归 follow-up dispatch，真正的下一条 ready 派发改在锁外 helper 执行，并把 blocked/failed follow-up dispatch 落成审计，避免静默卡死。
- 坑 15：如果看门狗仍把“future 入口还在”当成主要健康信号，就会漏掉一种非常危险的半断链：`running_task_count=0`、ready 队列已堆积、recent trigger/message 还残留 `assigned agent already has running node`，但真实运行记录已经没了。
  - 避免方式：看门狗必须把“ready pileup + no live run”明确列成假健康特征；验收至少覆盖 stale running recovered 后 ready 可继续派发，以及 ready 堆积时系统能补链/重派发原主线。
- 坑 16：如果 assignment finalize 继续把 `workspace memory / pm daily governance / prod upgrade request / follow-up dispatch` 一整包压在同一把全局 ticket mutation lock 里，只要某条 helper finalize 在锁内慢住，后续 schedule `create_assignment_node` 和已经存在的 `ready` 节点派发都会一起假停。
  - 避免方式：锁内只保留 run/node 终态与必要 schedule 回写，慢副作用移到锁外；同时 worker 额外监控 `0 running + ready 已老化` 并主动补一次 ready dispatch recovery。
- 坑 17：把“任务终态后生成下一轮”拆成很多 best-effort 副作用，会让任何一段锁竞争、线程退出或恢复路径分叉都能掉棒。现场表现通常不是服务挂掉，而是 `run.json` 已终态、后续 audit/recovery 还在写，但没有真实 provider token 消耗；人工补一次 schedule 后又可能停在裸 `trigger_hit` 或 `starting/provider_pid=0`。
  - 避免方式：把主线接力改成单一 durable handoff 协议，并用红灯 probe 覆盖 `success / failed / cancelled / recover_terminal_run_truth / process restart / ticket lock busy / provider_start_failed` 六类路径，断言每类路径最终都只能产生一个下一棒或一个明确 pause gate。
- 坑 18：durable handoff 表里写了 `status=scheduled / next_trigger_at=...` 不等于 7x24 已经安全。只要 schedule worker 错过那个 once 分钟，且读面继续按普通 once 计划修复，就会把 `[持续迭代] workflow` 关掉，表面是“服务在线、无 queued/running”，本质是主线掉棒。
  - 避免方式：handoff probe 必须覆盖“scheduled handoff -> missed once minute -> detail/list/preview 仍 active -> 下一次 scan catch-up trigger”这条红灯链；自迭代 once 计划不得被通用 exhausted-once repair 自动退休。
- 坑 19：`0 running + ready 已老化` 的补派发逻辑如果同步调用 `_request_assignment_dispatch()`，一次卡住就会把整个 schedule worker 循环卡死。现场不会出现 `worker_error`，但 `schedules.jsonl` 会停更，主线计划继续显示 old `last_result_status=queued`。
  - 避免方式：ready-dispatch recovery 必须异步化，并且同一 `ticket_id` 只允许一个后台 dispatch 线程在飞；新增非阻塞 probe 时要让 fake dispatch 持续阻塞，断言 worker 恢复函数小于 1 秒返回，同时确认 requested/completed 事件会落盘。
- 坑 20：把 terminal trigger 恢复成 `succeeded` 后，如果没有同时回填 durable handoff，页面会呈现“上一轮已成功”，但 `[持续迭代] workflow` 没有 `next_trigger_at`，7x24 会在成功状态下安静停止。
  - 避免方式：终态 trigger recovery 必须调用 handoff backfill；红灯用例要先证明缺少 handoff，再绿灯断言 `assignment_mainline_handoffs.status=scheduled` 与 schedule `next_trigger_at` 同步更新。
