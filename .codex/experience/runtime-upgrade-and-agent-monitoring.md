# 正式升级与长任务监控经验

## 适用范围
- `prod` 正式升级链路
- 可能超过 3 分钟的 agent 调用
- 部署副本启动与运行时配置同步
- 工作区升级 banner 与训练优化 create 态前端回归

## 稳定经验
- 长任务 agent 调用不要使用固定总超时。只要子进程仍在正常运行并持续有状态，就继续等待；只有进程异常或无结果且无运行迹象时才判失败。
- 当 assignment run 的 `stdout/stderr` 或 provider 事件只出现 `stream disconnected before completion` 这类瞬时 `Codex` 流断开、且还没有最终结构化结果时，默认不要立即判成终态失败；应先映射成可识别的瞬时故障并走一次自动重试，否则 7x24 主线会把可恢复网络抖动误打成真实失败。
- Windows 上通过 npm 全局 shim 启动 Codex 时，不要把 `codex.cmd / codex.ps1` wrapper 当成真实 provider。更稳的默认是直接落到 `node(.exe) + node_modules/@openai/codex/bin/codex.js`，这样 run record 里的 `provider_pid`、stale 判活和进程树清理才会对准真实执行树，而不是对着 `cmd.exe` 误判。
- 当前 Codex 的 Windows PowerShell shell 里，不要写 bash 风格 heredoc（例如 `python - <<'PY'`）；PowerShell 会把 `<<` 当成非法重定向。更稳的默认是用 here-string 管道：`@'... '@ | python -`。
- 在 PowerShell 里，不要把 `rg scripts/*.ps1` 这类带通配符的路径直接交给 `rg`；ripgrep 会把它当成字面路径并报 `os error 123`。更稳的默认是先 `rg --files scripts` 再二次过滤，或直接用 `Get-ChildItem -Recurse -Filter *.ps1`。
- 如果 `test-session-manager` 包装层被并行触发，可能出现同一批真实验证已通过，但包装会话的 `report.md` 被后一个会话元数据覆盖的情况。更稳的默认是：并行场景下优先信任 `result.json`、真实 acceptance run JSON 和生成时序更靠后的产物目录，再把 wrapper 报告只当辅助索引；同时尽量让 wrapper 保持串行，不把它作为并行 fan-out 的外层壳。
- 在 Windows PowerShell 里通过 here-string 管道把内嵌 Python 源码送给 `python -` 时，手写的中文字符串字面量可能在 live 文本写入里落成 `?`。更稳的默认是：中文正文优先复用仓内 UTF-8 模块常量或文件模板，或先放进环境变量 / UTF-8 文件再让 Python 读取；如果只是临时摘要，占位时宁可先用 ASCII，也不要把手敲中文直接埋进管道源码。
- PowerShell smoke / 巡检脚本如果消费 `Invoke-RestMethod` 返回的 JSON，不要直接把疑似标量字段喂给 `[int]` / `[bool]` 或内联拼进复杂对象；live 现场里这些值可能以单元素 `Object[]` 形式出现。更稳的默认是先做“取首个标量”的归一化，再转换类型、拼装 summary 或 markdown。
- PowerShell 侧把 `ConvertFrom-Json` 结果收成 plain data 时，不能先按 `PSObject.Properties` 展开再处理 `IEnumerable`；像 `changed_files` 这种 JSON 数组会被误折成属性字典，helper 最终就会回传空列表。更稳的默认是：保留 `IDictionary` 优先级后，先处理 `IEnumerable`，最后才回退到 `PSObject.Properties`。
- `prod` 启动链也不能把 `runtime-config.json` 的启动期写回当成“必须成功才能开站”的硬前提。像 `2026-04-23 19:45 +08:00` 这轮现场里，部署副本启动时在 `save_runtime_config -> write_runtime_config_payload -> tmp.replace(path)` 命中 `PermissionError: [WinError 5]`，整个 launcher 会直接 `exit 1`，supervisor 随后无限 watchdog restart，表面像“schedule 又停了”，实际是 web server 根本没起来。更稳的默认是：启动期 runtime-config 持久化先做 no-op 跳过，并在命中权限拒绝时 fail-open 继续开站，同时用 dedicated probe 锁住“启动写配置失败不能拖死 prod”。
- 评估长任务或 live run“running 但 token 不增长”时，不要把整段都当成单点执行慢。更稳的默认是先拆成三段：上一轮 finalize 尾巴、ready 节点 dispatch 老化、provider_start 空窗；只有 provider_start 后仍持续产出模型活动，才算真实执行在耗 token。
- 只要触碰了 `prod` 停机、重启、直发部署或 supervisor 恢复，就不能把“deploy success / manifest 已更新 / candidate 已刷新”误当成 `7x24` 已恢复。像 `2026-04-23 20:25 +08:00` 这轮现场里，我把修复发布到了 `prod=20260423-202504`，却没有继续核到 `8090` listener、`/healthz`、`/api/status` 和主线/保底下一棒出口都恢复，结果现网仍是停的。更稳的默认是：凡是触碰 `prod` 运行态，收尾条件必须同时满足 `healthz 可读 + status 可读 + 至少一个 mainline/patrol 出口存在`；没满足前不能算完成。
- `run_acceptance_workflow_gate.py` 不能默默复用一个已经被 live `workflow` listener 占用的端口；更稳的默认是先做端口隔离检查，再启动隔离验收环境，并在 boundary 读链不指向 fixture runtime 时直接 fail-closed。否则一次误传 `8090` 就会把 live 现场误当成 gate 夹具，报告里只留下迷惑性的 fixture-boundary 失败。
- `run_acceptance_runtime_release_gate.py` 里的 `/api/assignments/test-data/bootstrap` 不能只要旧 prototype 图存在就直接复用。像 `2026-04-17 22:01 +08:00` 这轮现场里，旧测试图已经停在 `running=0`，release gate 会稳定炸在 `test assignment graph should surface running nodes`。更稳的默认是：bootstrap 命中 `total_nodes!=20` 或 `running<1` 时，先在同 ticket 上重种 `20` 节点测试图，再继续 gate；部署后若 `8092` 还残留 `T9` test ghost，再用受支持的 `repair-ghost-running` 把它清到 `running_task_count=0 / ghost_running_detected=false`。
- 原型测试图里的占位代理节点不能继续保持“无依赖 + 待派发”。像 `2026-04-18 11:13 +08:00` 这轮现场里，`T17/T18/T19` 每次 bootstrap 后都会从占位节点漂成可派发节点，随后反复打出 `assignment_agent_workspace_missing` 噪音。更稳的默认是：让这类纯展示节点显式挂在稳定 failed upstream（当前是 `T20`）后面，保持 `blocked` 而不是误派发到不存在的 workspace，并用 `verify_assignment_test_graph_placeholder_nodes.py` + `workflow gate` 锁住。
- 不要假设 `test` 上已经 repair 过一次的 `T9` ghost ref 会在后续部署里自然消失。像 `2026-04-18 10:40 +08:00` 这轮现场里，`deploy_test_workflow_env.ps1` 成功把 `test` 切到 `20260418-104033` 后，`/api/runtime-upgrade/status` 又把历史 `asg-20260417-202951-ec981b / T9` 投影回 `running_task_count=1 / ghost_running_detected=true`。更稳的默认是：不要再把这条动作留给值班者手工补刀，而是把它正式沉到 `deploy_workflow_env.ps1` 的 `test gate` 后置合同里。当前稳定口径是：`test gate` 通过后立即读取 `8092` 的 `/api/runtime-upgrade/status`，命中 `ghost_running_detected=true` 就自动调用受支持的 `repair-ghost-running`，并把 before/after 真相落到 `deploy-<version>.json` 的 `post_deploy_ghost_running` 字段；若修不干净，就 fail-closed，不再把“干净 test 基线”建立在人工补 repair 上。
- `deploy_test_workflow_env.ps1` 的 post-deploy ghost repair 也不能把 repair POST 的客户端超时直接当成候选刷新失败。像 `2026-04-30 13:08 +08:00` 这轮现场里，repair 请求侧报 timeout，但随后 `/api/runtime-upgrade/status` 已经显示 `ghost_running_detected=false / ghost_running_count=0`；如果脚本不做 readback settle，就会误阻断 `prod candidate=20260430-130822` 的生成。更稳的默认是：repair 请求失败或超时后立刻回读 status；只要 ghost 已清空，就以 `ghost_running_repaired_after_request_error` 继续，并把 request error 与 before/after 状态一起写进 `deploy-<version>.json`。
- `request_prod_auto_upgrade_single_check()` 当前是拉起 detached watcher 子进程后立刻返回；验证这条链时，不能在收到 `requested=true` 后马上同步断言 watcher 日志已经写完。更稳的默认是等待日志里出现 `等待升级完成中 / prod 已切到目标 candidate / 快照刷新结果` 这些终态片段，再做断言，否则 probe 会把“日志还没刷完”误判成产品回退。
- `apply_prod_candidate_when_idle.py` / `Invoke-WorkflowProdAutoUpgradeSingleCheck` 的默认 `apply_wait_seconds` 不能继续停在 `420s`。像 `2026-04-18 15:14 +08:00` 这轮现场里，`prod` 从 `20260418-141234` 切到 `20260418-145016` 实际耗到了 `8` 分多钟，watcher 会在升级已经成功前约 `1` 分钟先写出“超过等待窗口仍未确认切换”，误导值班判断。更稳的默认是：Python watcher、`workflow_env_common.ps1` 与 `start_workflow_env.ps1` 三处默认一起抬到至少 `600s`，并用 `verify_apply_prod_candidate_when_idle.py` 和 `verify_prod_auto_upgrade_single_check_helper.py` 同时锁住脚本入口与 PowerShell 入口。
- `prod-idle-upgrade-watchdog` 也不能假设外层 caller 传进来的 `--apply-wait-seconds` 一定已经跟磁盘新默认对齐。像 `2026-04-18 17:15 +08:00` 这轮现场里，`prod` supervisor 进程 `20500` 仍是 `2026-04-17 09:14:37 +08:00` 启动的旧进程，会继续显式传入历史较小 wait 值；结果 `165853` 虽在 `2026-04-18 17:23:02 +08:00` 升级成功，watcher 仍在 `2026-04-18 17:22:04 +08:00` 先写超时。更稳的默认是：Python watcher 对 `operator=prod-idle-upgrade-watchdog` 传入的过小 wait 值按当前最小安全值 `600s` 兜底，并把校正日志落盘；同时用 `verify_apply_prod_candidate_when_idle.py` 的 `legacy_watchdog_apply_wait_clamp` probe 锁住，不把效果继续绑在 supervisor 何时重启上。
- `apply_prod_candidate_when_idle.py` 也不能把 `repair-ghost-running` 的慢响应直接记成最终失败。像 `2026-04-18 15:50 +08:00` 这轮现场里，watcher 命中 `running_finalize_stall` 后，`POST /api/runtime-upgrade/repair-ghost-running` 会先在客户端 `20s` 超时，但 `/api/runtime-upgrade/status` 很快已经翻成 `ghost_running_detected=false / can_upgrade=true`；如果 watcher 直接把 `request_failed` 当终态，就会错过同轮可升级空窗。更稳的默认是：repair 请求一旦没拿到确定响应，立刻进入短状态复核窗口，继续回读 `/api/runtime-upgrade/status`；只要 ghost 已清空，就按修复成功继续后续 apply 判断，并用 `verify_apply_prod_candidate_when_idle.py` 的 slow-repair settle probe 锁住。
- 当 `prod` 已存在更高 `candidate_version` 时，不能再把“candidate 更高”本身当成硬 drain 去冻结下一棒任务；否则只要 idle watcher 或 prod 启动链掉一环，主线就会整段卡在“ready 但不派发”。更稳的默认是：只有 `upgrade_request_pending` 才是真硬 drain；更高 candidate 只触发 single-check 尝试，若升级没真正挂起，就继续派发，连续性优先。
- `apply_prod_candidate_when_idle.py` 也不要在升级重连窗口里把同一条 `request_failed` 连续刷满日志。像 `2026-04-18 16:43 +08:00` 这轮现场里，`prod` 从 `153421` 切到 `161644` 时会连续出现多拍相同的 `10061/request_failed`；如果 watcher 每轮都原样落日志，值班视角会被同一条 unavailable 噪声淹没。更稳的默认是：对 `等待升级完成时状态接口不可用` 这类连续同错，先保留首条原始错误；状态恢复或错误变化时，再补一条累计摘要，并用 `verify_apply_prod_candidate_when_idle.py` 的 `upgrade_wait_unavailable_log_compaction` probe 锁住。
- `apply_prod_candidate_when_idle.py` 也不要在 ghost-repair settle 窗口里把同一组 `继续复核状态 / 假 running 修复后状态 / 修复后仍存在假 running` 连续刷满日志。像 `2026-04-18 17:56 +08:00` 这轮现场里，watcher 在同一条 `running_finalize_stall` 上会重复刷出完全相同的状态块；如果不压缩，值班视角仍然会被同一组 ghost 状态淹没。更稳的默认是：连续相同的 ghost-repair settle 状态只保留首组原始块；状态变化或退出时，再补一条累计摘要，并用 `verify_apply_prod_candidate_when_idle.py` 的 `repair_settle_log_compaction` probe 锁住。
- `prod` supervisor 同步调用 idle watcher 时，不能让 watcher 在已经发起 `/api/runtime-upgrade/apply` 后继续长期等待 `8090` 恢复，从而堵住 supervisor 自己处理已落盘的 `prod-upgrade-request.json`。像 `2026-04-26 12:28 +08:00` 现场里，watcher 已申请 `20260426-120441` 升级，生产进程退出并写入 upgrade request，但 watcher 子进程仍在等待状态接口恢复，父级 supervisor 没进入切候选重启分支，导致 `8090` 暂停到手工结束 watcher 后才恢复。更稳的默认是：现场恢复时先确认 watcher 是否仍是 supervisor 子进程并阻塞切换；必要时只结束该 watcher 子进程，让 supervisor 继续处理已存在的升级请求，再复核 `/healthz /api/status /api/runtime-upgrade/status /api/schedules`。
- `final_result` 落盘不等于 assignment 已完成终态提交。像 `2026-04-29 13:54 +08:00` 现场里，`task_artifact_store_run_runtime.py` 在写完 `result.json/final_result` 后、进入 `_assignment_finalize_execution_run_fail_closed()` 前就触发 `assignment-executor-final-result` watcher；watcher 把正常 finalizing 窗口看成 `running_finalize_stall`，随后又在 ghost 未清干净时 apply `candidate=20260429-133742`，重启打断了后半段 finalize，最终拖到 `14:49` 靠 `recover_terminal_run_truth` 补终态。更稳的默认是：prod auto-upgrade 只能在 node/run 终态已经提交后触发，或者 terminal recovery 明确成功后触发；不要把“final_result 文件存在”当成安全升级空窗，已有 `verify_assignment_final_result_single_check.py` 这类用例需要反向调整。
- `prod` 升级后不会自动给旧版本已经终态的 project-controller 轮次补 durable handoff。像 `2026-04-27 19:35 +08:00` 现场里，`novel_project_pm` 的 13:48 轮次已经 `succeeded`，但因为完成时仍在旧 `prod`，`[持续迭代] novel_project_pm` 保持 `next_trigger_at=""`；升级到新版本后必须先确认终态 node/run，再用当前 handoff runtime 回补 `assignment_mainline_handoffs` 并 drain，才能恢复项目下一棒。
- `refresh_pm_current_version_snapshot.py` 不能只匹配旧的“当前 `prod` 已升到 …”句式。只要 `V2` 版本文件里的快照文案已经演进成“当前 `prod` 仍是/已是 …”，idle watcher 升级成功后的快照回写就会直接失败，现场表现成 `PM当前版本计划.md` 已追平、`pm/versions/V2/版本计划.md` 却停在旧 baseline。更稳的默认是让快照刷新脚本兼容 `已升到 / 已是 / 仍是` 三种 live 文案，并用真实现网句式 fixture 锁住。
- `refresh_pm_current_version_snapshot.py` 也不能漏掉“当前 `prod` 仍是旧版本，但新的 `candidate_version` 已生成”这种 live 过渡句式。像 `2026-04-17 00:50 +08:00` 这轮现场里，idle watcher 已经把 `prod` 切到 `20260417-003801`，但 PM 文档还停在“旧 `prod` + 新 candidate 已生成”；如果脚本只认标准 `当前 prod 已是 ...，candidate_version=...` 结构，升级完成后的自动回写就会稳定报 `missing or ambiguous pattern: plan_prod_status_current_shape`，`document_baseline` 也会一直卡旧。更稳的默认是把这类“candidate 已生成”过渡文案也纳入兼容正则，并用真实 fixture 锁住。
- `refresh_pm_current_version_snapshot.py` 也不能只兼容“但新的 `candidate_version=...`”这一种字样。像 `2026-04-17 10:20 +08:00` 这轮 live 文案里，`PM当前版本计划.md / pm/versions/V3/版本计划.md` 实际写成的是“当前 `prod` 仍是 `...`，但 `candidate_version=...`：...”；如果正则只认“但新的”，supported refresh 就会继续炸在 `missing or ambiguous pattern: plan_prod_status_current_shape`，`document_baseline` 会稳定停旧。更稳的默认是把“但 / 但新的 / 已生成”三类 candidate 过渡句式一起纳入兼容 fixture。
- `refresh_pm_current_version_snapshot.py` 也不能漏掉“当前 `prod` 仍是 `...`，新的 `candidate_version=...`：...”这种 live 句式。像 `2026-04-18 18:49 +08:00` 这轮现场里，idle watcher 已把 `prod` 切到 `20260418-182357`，但 `PM当前版本计划.md / pm/versions/V4/版本计划.md` 还停在“旧 `prod` + 新 candidate”等待升级的文案，脚本只因不认单独的“新的”前缀就直接炸在 `missing or ambiguous pattern: plan_prod_status_current_shape`，`document_baseline` 也继续卡旧。更稳的默认是把“新的 / 但 / 但新的 / 已生成”四类 candidate 过渡句式一起纳入正则和真实 `V4` fixture，并在回刷失败后优先用受支持脚本把文档 baseline 追平 live。
- `refresh_pm_current_version_snapshot.py` 也不能漏掉“当前 `prod` 仍是 `...`，但 `candidate_version=...` 已更高：...”这种 live 句式。像 `2026-04-19 00:27 +08:00` 这轮现场里，主线节点已经因为这条文案差异直接炸在 `missing or ambiguous pattern: plan_prod_status_current_shape`；更稳的默认是把 `已更高` 也纳入 candidate 过渡句式兼容，并用真实 `V4` fixture 锁住，再跑一遍完整 `workflow gate` 后再收口 release boundary。
- `refresh_pm_current_version_snapshot.py` 也不能只兼容 `已更高`，而漏掉“`candidate_version=...` 已更新为更高版本”这种 live 文案。像 `2026-04-19 02:46 +08:00` 这轮现场里，`prod` 已经升到 `20260419-021858`，但 supported refresh 仍会因为旧文案停在“candidate 已更新为更高版本”而炸在 `plan_prod_status_current_shape`。更稳的默认是把 `已更新为更高版本` 一并纳入 current plan/version live-status 正则，并用真实 `V4` fixture 锁住“旧文案 -> 升级完成后回刷”的路径。
- 部署副本里的 current-version snapshot refresh 不能把 `SourceRoot` 机械当成治理根。像 `.running/prod`、`.test/runtime-*` 这类部署副本或 gate 夹具里，真正的治理根通常要先按 `.workflow-deployment.json / .workflow-local-deployment.json / .running/control / AGENTS.md` 回落；否则一条路径会直接 `pm_plan_missing`，另一条路径会穿透到真实工作区，把 gate 夹具和现网治理根串起来。更稳的默认是：deployment metadata 优先，其次识别本地 `.running/control` fixture 根，最后才向上找 `pm/` 或 `AGENTS.md`。
- `refresh_pm_current_version_snapshot.py` 也不能继续强依赖 `PM当前版本计划.md` 里存在独立 `document_baseline` 行，或把版本计划的 baseline 句式固定写成 `baseline 已对齐为 live`。当前 `V3` 文案已经演进成“只保留 baseline 行 + current prod live status”，如果脚本仍按旧结构精确匹配，idle watcher 升级成功后 PM 文档就会停在旧 baseline。更稳的默认是：`document_baseline` 行改成可选、current plan live-status 允许 `running_task_count` 缺省、version baseline 同时兼容带/不带 `live`，并用当前 `V3` fixture 锁住。
- `refresh_pm_current_version_snapshot.py` 还不能假设当前 `PM当前版本计划.md / pm/versions/V3/版本计划.md` 一定已经存在显式 `baseline` 行。像 `2026-04-16 18:00 +08:00` 这类 live 现场里，文件可能只剩“`\`prod\` 已切到 ...；当前 \`current_version=...\``”句式；如果脚本仍把 `plan_baseline / version_baseline` 当成必有模式，巡检或部署调用会直接炸在 `missing or ambiguous pattern`。更稳的默认是：命中这类 live 句式时，先把那一行回写成显式 `baseline 已对齐为 \`prod=...\``，再继续刷新 live-status，并用真实 fixture 锁住。
- `pm_version_status_service.load_pm_version_status()` 也不能只识别 `baseline 已追到 live`。只要当前 active 版本文件把快照文案写成 `baseline 仍是 live`，`/api/status.pm_version_status.document_baseline` 就会继续退成空字符串，即使 watcher 已经把版本文件写对。更稳的默认是让 baseline 解析同时兼容 `已追到 live / 已对齐为 / 仍是 live`，并用 truth-source probe 锁住。
- `pm_version_status_service.load_pm_version_status()` 也不能把版本计划正文里的历史 `baseline=` 备注误吸成当前 baseline。像 `2026-04-17 15:37 +08:00` 这轮现场里，`V4` 快照已经写成 `baseline 继续对齐为 prod=20260417-145421`，但正文里还保留着旧 `baseline=20260417-134734`；如果解析器既不兼容“继续对齐为”，fallback 又继续扫描正文任意 `baseline=`，`/api/status.pm_version_status.baseline/document_baseline` 就会稳定回退到旧值。更稳的默认是：显式兼容 `继续对齐为` 句式，并把 fallback baseline 匹配收紧到真实 snapshot baseline 行，不再横扫正文里的历史备注。
- `refresh_pm_current_version_snapshot.py` 也不能只让 `pm_version_status_service` 兼容 `继续对齐为`，自己却还停在旧句式。像 `2026-04-17 16:26 +08:00` 这轮现场里，idle watcher 已经把 `prod` 切到 `20260417-160828`，但快照刷新脚本的 `PLAN_BASELINE_RE / PLAN_DOCUMENT_BASELINE_RE / VERSION_BASELINE_RE` 仍不认 `V4` 当前 `baseline/document_baseline 继续对齐为`，结果 watcher 成功升级后直接炸在 `missing or ambiguous pattern: plan_baseline_fallback`，`/api/status.pm_version_status.document_baseline` 会继续卡在旧 `145421`。更稳的默认是：刷新脚本和解析器都同步兼容 `继续对齐为`，并用真实 `V4` fixture 锁住 idle watcher 升级成功后的 snapshot refresh。
- `refresh_pm_current_version_snapshot.py` 也不能假设 numbered snapshot 的 live 行会直接以“当前 `prod` ...”起头。像 `2026-04-21 10:16 +08:00` 这轮现场里，`PM当前版本计划.md` 已经演进成“同一编号行前半句先写 `/healthz / /api/status / /api/schedules / /api/runtime-upgrade/status` 可读，后半句才写 `当前 \`prod=current=... / candidate=...\``”；如果 current-shape 正则仍强依赖行首就是 `当前 \`prod...`，supported refresh 会继续炸在 `plan_prod_status_current_shape`，`document_baseline` 也会卡在旧值。更稳的默认是：允许 numbered snapshot line 在 live `prod` 片段前带前缀健康摘要，并用真实 `V5` fixture 锁住这条 current `prod=current=... / candidate=...` 形态。
- `apply_prod_candidate_when_idle.py` 在升级完成后也不能继续沿用发起升级的旧部署根去调用 `refresh_pm_current_version_snapshot.py`。像 `2026-04-21 14:37 +08:00` 这轮现场里，旧 watcher 虽然已经把 `prod` 切到 `20260421-142239`，但如果它还拿旧部署里的 refresh 脚本回写，新的 lane/stale-pending-upgrade 修复根本不会被用上，PM 快照就会继续停在“待升级”旧文案。更稳的默认是：升级完成后的 snapshot refresh 优先使用当前活跃 `prod` 副本里的脚本，并用 `verify_apply_prod_candidate_when_idle.py` 的 `snapshot_refresh_script_resolution` probe 锁住。
- `refresh_pm_current_version_snapshot.py` 也不能把 `待空窗升级` 漏出待升级 hint 集合，更不能让 prod restart drift repair 只盯 `document_baseline`。像 `2026-04-21 14:52 +08:00` 这轮现场里，live `prod=current=candidate=20260421-142239` 已经追平，但因为 lane 文案写成 `待空窗升级`、PowerShell drift repair 又只看 `document_baseline`，supported refresh 会把快照误留在“待升级”状态。更稳的默认是：待升级 hint 统一兼容 `待升级 / 待空窗升级 / idle window`，prod restart drift repair 也要在 `document_baseline` 已对齐但 lane 仍旧待升级时主动 refresh，并用 `V5` fixture + PowerShell drift-state probe 一起锁住。
- `refresh_pm_current_version_snapshot.py` 也不能假设 `PM当前版本计划.md` 的快照头还停在旧的“最新有效快照截至 …”，更不能默认版本正文里的当前状态快照一定也带时间戳。像 `2026-04-22 11:36 +08:00` 这轮现场里，idle watcher 已经把 `prod` 升到 `20260422-111915`，但 `PM当前版本计划.md` 已改成 `snapshot_updated_at:` 紧凑写法，`pm/versions/V8/版本计划.md` 又没有独立快照时间；如果 refresh 脚本和 `pm_version_status_service` 仍只认旧句式、且不从 reference snapshot 回退，watcher 会稳定炸在 `plan_snapshot_at`，`document_baseline` 也会继续卡在旧 `103706`。更稳的默认是：refresh 脚本兼容 `snapshot_updated_at:` 与紧凑 `当前 live 判断` 形态，版本正文 live line 缺席时 fail-open 只更新 baseline；同时 `pm_version_status_service` 在版本正文缺时间戳时回退读取 `PM当前版本计划.md` 的 reference snapshot，并用真实 `V8` fixture + truth-source probe 一起锁住。
- `refresh_pm_current_version_snapshot.py` 也不能漏掉 `当前 live 发布判断` 这种 PM 当前计划紧凑 live 行。像 `2026-04-27 07:00 +08:00` 这轮现场里，`prod` 已经切到 `20260427-060104`，但 `PM当前版本计划.md` 的 `document_baseline` 仍停在 `033442`；受支持 refresh 因不认 `当前 live 发布判断：\`prod current=... / candidate=...\`` 直接炸在 `plan_prod_status_current_shape`。更稳的默认是：compact live status 兼容 `发布判断 / 判断 / 真相` 三类标签，并用 `live_release_decision_shape` fixture 锁住真实 V12 文案。
- self-iteration / patrol 的 `上一轮结果 / 最近上下文` 也不能继续把旧 `prod=<version>`、`candidate=<version>`、`workspace_head/code_root_head` 或 `@<git head>` 原样带进当前 node_goal。像 `2026-04-17 17:15 +08:00` 这轮现场里，`164304` 的 API truth 已经对齐，但 smoke 仍把 current mainline 合同判成“未全量 materialize”，根因就是上一轮结果里还残留 `160828` 等旧快照。更稳的默认是：当前 baseline/head 只保留在动态 prompt 行里，上一轮结果/最近上下文里的旧 version/head 一律脱敏成占位符，并用定向 probe 锁住。
- 通过 `test-session-manager` 的 `run-test-in-session.ps1` 跑本地验证时，不要并行起多个 wrapper。当前脚本的 session 目录仍按秒级时间戳落盘，并发时容易争抢同一个 `session.json/report.md`，现场会表现成 `Set-Content : Stream was not readable.`、多个命令撞到同一 session id，或后一个命令把前一个测试的 `latest-report` 覆掉。更稳的默认是串行跑这些 wrapper；如果确实需要并行，先显式建出各自独立 session 再分发。
- PowerShell 脚本里的 `param(...)` 必须保持在文件首个可执行结构；一旦前面先写了变量赋值，即使后续逻辑本身没问题，parse 验收也会把整个参数块判坏。
- assignment run 的 stale grace 不能和 heartbeat keepalive 设成同一条紧边界；只要 run 仍有 live provider pid，就应该给至少 `3 x keepalive` 的新鲜度缓冲，否则像 `workflow` 这种长上下文自迭代很容易在静默几十秒时被误回收成“运行句柄缺失”。
- 当 provider 已经退出、但 assignment worker 还在解析 stdout/structured result 并准备 finalize 时，不能继续沿用“死进程仅 `15s`”的 stale 判活。更稳的默认是先把 run 标成“正在整理结果”、清空 `provider_pid` 并刷新 `latest_event_at/updated_at`，给 finalize 一个长于 cold-start 的缓冲窗口；否则真实 `result.json/final_result` 会被 stale recovery 抢先打成 `cancelled`。
- 当某条 helper run 已经写出了 `result.json`，且 `run.json` 长时间停在 `status=running / provider_pid=0 / latest_event=Provider 已退出，正在整理结果。`，同时 audit 迟迟没有 `deliver_artifact / execution_succeeded / execution_failed` 时，不要再把它当成真实运行。更稳的默认是先把它判成 `finalize stall / ghost running`：token 通常会在 `latest_event_at` 对应的 provider 退出时刻停止，后续 mainline / patrol 只会继续堆 `ready`，`/api/runtime-upgrade/status.running_task_count` 也会虚高。
- `final_result` 已经落盘后，不能再让 `_finalize_assignment_execution_run()` 的异常把 run/node 留在 `running` 壳里。更稳的默认是：worker 在写完 `result.json + final_result event` 后，调用 finalize 时必须包一层 fail-closed 恢复；一旦 finalize 抛错，立刻回读结果文件执行 `recover_terminal_run_truth`，至少把 run/node 同步到终态并写下 `finalize_exception(_recovered)` 事件，而不是等后面的 watchdog 或升级门禁再发现 ghost running。
- `repair_ghost_running()` 不能只在“当前节点文件仍是 running”时生效。像 `2026-04-16` 这轮现场里，历史 ghost 还会落在两种更隐蔽的真相上：一是节点已经 `succeeded/failed`，但对应 `run.json` 还停在 `running`；二是节点已经被标成 `record_state=deleted`，但旧 run 仍未收口。更稳的默认是：修复逻辑先看 live snapshot 里的节点；拿不到时，再回退读原始 node 文件（包括 `deleted` 节点）；只要已经有 terminal node truth，就直接把 stale `run.json` 同步到同一个 terminal status，而不是返回 `already_terminal` 空动作。否则 watchdog 会一直重复报同一条 `stale_run_with_terminal_evidence`，永远清不掉历史旧账。
- 当 read-path / stale recovery 已经拿到 `result.json`、`final_result` 或其他 terminal truth 时，不要再重走完整 `_finalize_assignment_execution_run()` 成功链去“补收尾”；那条链里夹着 artifact delivery、memory append、follow-up dispatch 等重副作用，现场一旦就是卡在 finalize，本次重试很可能再次卡死。更稳的默认是：先做 lightweight terminal commit，只把 `run/node` 拉到 terminal、保留 `result_ref`、补必要 schedule/follow-up，再把 artifact delivery / memory 视为 best-effort 副作用。
- `prod-idle-upgrade-watchdog` 不应只记录 `running_task_count`。更稳的默认是让 `/api/runtime-upgrade/status` 同时暴露 `ghost_running_detected / ghost_running_count / ghost_running_refs`，并让 watchdog 日志在“当前仍未到可升级空窗”之前先打印“检测到假 running”与首批 refs；否则值班视角只会看到“running_task_count=1”，却不知道这 1 条其实已经是 terminal evidence 存在的假 running。
- `watchdog` 的轮询频率不能大于 finalize-stall 的可见窗口。当前 live 里 `_assignment_mark_run_result_finalizing()` 会把 `provider_pid=0 + latest_event=Provider 已退出，正在整理结果。` 写回 run record，而 `_assignment_run_row_is_live()` 又会对 `provider_pid=0` 额外给 `DEFAULT_ASSIGNMENT_PROVIDER_START_GRACE_SECONDS=180` 秒缓冲；如果 watchdog 还是按整小时单次检查，它就会错过这 `~3` 分钟的 `running_finalize_stall` 窗口，只在下一次巡检时看到更晚的 `stale_run_with_terminal_evidence`。更稳的默认是：要么把 watchdog 频率压到短于这段 grace，要么在 finalize 标记写回后补一轮近实时单次检查，不要把“02:15 才进入 finalize-stall、03:14 才第一次巡检”误解成“watchdog 完全看不出假 running”。
- `repair-ghost-running` 也不能在 `recover_terminal_run_truth` 前无条件等待 finalize 锁。像 `2026-04-19 16:40 +08:00` 这轮现场里，`prod-idle-upgrade-watchdog` 已经从 `/api/runtime-upgrade/status` 识别出 `running_node_projected_terminal / stale_run_with_terminal_evidence`，但 repair API 会被同一条 run 的 finalize 锁拖到客户端 `20s` 超时，表面看起来像“watchdog 看见了也修不掉”。更稳的默认是：watchdog repair 走 run-file terminal recovery 时，先给 finalize 锁一个很短的等待窗口；如果锁还被卡住，就允许在 ticket mutation 锁内旁路恢复 terminal truth，让 watchdog 先把 `run/node` 从 ghost 壳里拉回终态，再让后续 finalize 线程看到终态后自行早退，并用 `verify_runtime_upgrade_ghost_running_repair.py` 的 held-finalize-lock probe 锁住。
- `runtime_upgrade` 的 ghost-running repair 也不能在“只读快照”这一步重新拿一遍同 ticket 的 mutation lock。像 `2026-04-20 23:40 +08:00` 这轮现场里，held-ticket-lock probe 不是卡在真正的 repair，而是先卡在 `_assignment_snapshot_from_files(persist_changes=True)`；更稳的默认是：runtime-upgrade repair 读节点快照时显式走 `persist_changes=False` 的只读视图，并优先修带终态证据的 `stale_run_with_terminal_evidence / running_finalize_stall` ref，再去碰纯 `running_node_projected_terminal`，否则同 ticket 上更老的 raw ghost 会把当前最值钱的 repair 挤掉，还会把 held-ticket-lock probe放大成“repair hung”。
- `repair-ghost-running` 命中 `running_finalize_stall` 时，客户端超时或“没有立即返回 repair 结果”也不能被直接当成修复失败。像 `2026-05-05 01:00 +08:00` 这轮现场里，`novel_project_pm` 的 `arun-20260505-003543-6a36ba` 已先写出 `execution_succeeded`，随后 watchdog 才补上 `repair_ghost_running`、`mainline_handoff_recorded` 与 `mainline_handoff_scheduled`；更稳的默认是：一旦 repair 请求表现成慢响应，就立刻回读 `status-detail + run.json + audit.jsonl + /api/runtime-upgrade/status` 四层真相。只要 `status-detail` 已是 terminal、`run.json` 已收敛、audit 补齐 repair/handoff，且 runtime-upgrade 继续 `ghost=false/count=0`，就应把这次 repair 视为已成功收口，而不是继续误判成 live ghost。
- `stop_workflow_env.ps1` / `deploy_workflow_env.ps1` 不能把 `instances/<env>.json` 当成绝对真相源。像 `2026-04-18` 这轮现场里，`prod` 明明由 `workflow_web_server.py --root ...runtime\\prod --port 8090` 在跑，但 `instances/prod.json` 却残留成 `environment=test / port=8168`；如果脚本先被 stale instance 拦住、还没来得及看进程命令行，就会把真正的 `prod` 误判成“none matched descriptor”，导致停服/部署双双 fail-closed。更稳的默认是：实例文件只作辅证；只要进程命令行里的 `workflow_web_server.py` 和 `runtime_root` 明确匹配当前 descriptor，就应信进程并清 stale instance，不让坏台账反过来挡住正式维护。
- 直升 `prod` 的维护路径也不能只写 `deploy report/manifest`，却漏掉 `prod-last-action.json`。像 `2026-04-18` 这轮现场里，`prod` 已经切到 `20260418-090320`，但 `prod-last-action.json` 还停在旧的 `watchdog_restart`，会继续误导后续值班判断。更稳的默认是：凡是显式 `AllowDirectProdDeploy` 成功落地，都同步把 `prod-last-action.json` 写成当前版本的成功直升记录。
- `test/prod` 的人用运维入口不应继续共用同一条脚本名再靠 `-Environment` 区分。更稳的默认是：保留共享底层 helper，但对人工与 agent 的直接入口显式拆成 `deploy_test_workflow_env.ps1 / deploy_prod_workflow_env.ps1 / stop_test_workflow_env.ps1 / stop_prod_workflow_env.ps1`。这样就算执行者状态起伏、上下文混乱或命令补全误触，也更难把测试操作误打到生产环境。
- 长耗时验收脚本不能只依赖底层默认 HTTP 超时或静默轮询。只要单步可能超过几十秒，就要补阶段 checkpoint、heartbeat 日志和 partial summary 落盘；否则用户体感会变成“Codex 直接退了”，即使真实问题只是脚本无输出或请求超时。
- `collect_v4_r1_r4_current_version_smoke.py` 这类直连 live `/api/assignments/{ticket}/status-detail` 的 current-version smoke，不能继续把单次 `60s` 超时当成最终失败。像 `2026-04-18 14:00 +08:00` 这轮现场里，`status-detail` 本身会偶发慢到把 smoke 直接打成 `TimeoutError`，但手工复核同一路径几秒后又恢复正常。更稳的默认是：对这类 live detail 读链加少量超时重试，只在连续超时后再 fail-closed，同时把这条 contract 保留在正式 smoke 资产里，而不是回退成一次性人工判断。
- `run_acceptance_assignment_center_browser.py` 这类 assignment browser acceptance 不能在 `assignment_probe=1` 时继续开启 detail realtime `EventSource / poller`，也不要让 probe 的 `afterWait` 把页面切到一种最终态、断言却还要求另一种最终态。更稳的默认是：assignment probe 模式下先停掉 realtime stream/poller，并额外禁止 bootstrap 自动启动 workflow poller、runtime-upgrade poller 和 policy auto-analyze 这类后台轮询；browser harness 统一用完整 Edge flags、`shot/dom` 独立 profile、按 probe delay 推导 `budget/timeout`，并在 `view_tabs / surface_compact` 这类 case 里把页面恢复到与断言一致的最终态；否则现场会表现成首张 `task-center` 截图 `msedge --headless` 卡 `45~180s`，或截图恢复后马上又被 probe 自身状态漂移打红。
- `api_catalog_live_regression` 这类接口目录回归不能把 Edge `--screenshot` 或 `--dump-dom` 超时直接等同于接口合同失败。只要 host root 壳体、`/api/platform/interfaces` list/detail 和 runtime version 都已验证通过，浏览器 DOM/截图超时应降级为 browser evidence warning，并落 `api_contract_fallback` probe；否则 Edge headless 偶发卡死会把已经恢复的 API/readback 合同误打红。
- `assignment_probe=1` 的 browser smoke 也不能继续沿用完整工作台启动链。像 `2026-04-19 03:00 +08:00` 这轮现场里，如果 probe 还会先恢复 chat/session/workflow 列表和最近会话，再切到 task-center，`msedge --headless` 的首张截图会稳定卡在 `45s` 超时。更稳的默认是：assignment probe 直接从 `task-center` 起步，跳过 `refreshSessions / refreshWorkflows / loadSessionMessages` 这类无关预热，只保留任务中心需要的 `agents + dashboard + runtime upgrade`。
- `assignmentWorkboardGroups()` 也不能只在 `graphData.nodes` 为空时才回退 `dashboardMetrics.assignment_workboard_agents`。像 `2026-04-19 03:08 +08:00` 这轮现场里，live 全局图仍有节点，但 helper 的 historical residue 只存在于 dashboard workboard 聚合真相里；如果渲染逻辑继续优先吃图节点，browser probe 会看到 `helper_lane_total=0`，而 API smoke 已经能看到 `5` 条 helper lane。更稳的默认是：当 dashboard workboard 聚合已可用时，helper lane 分区优先复用那条真相，图节点只作为 fallback。
- 串行覆盖多个慢 phase 的 acceptance 不适合作为“每次小改默认门禁”。像角色创建 async-delete 这种 `draft + async + cleanup + completed` 的全量链路，应拆成可单独执行的 phase；开发期默认跑命中改动面的最小 phase，`all` 只留给发布前或专项回归。
- 当某项关键能力已经通过真实单项测试后，同批次的全流程串联回归应优先改用 mock/stub，避免把同一条慢链路重复跑两遍；真实全流程只留给发布前、专项回归或 mock 无法覆盖的差异场景。
- assignment acceptance 如果继续复用 codex provider stub，stub 不能用 `cmd.exe /c rem` 这类瞬时退出命令。更稳的默认是让 stub 先吞掉 `stdin`、再回结构化结果，并且外层验收要显式断言“原节点真正 `succeeded`”；否则 Windows 会在 prompt 写入阶段抛 `OSError: [Errno 22] Invalid argument`，wrapper 还可能把假通过吞过去。
- 开发环境验证 agent 调用时，优先通过环境变量覆写注入 mock，并在验收脚本或开发环境内隔离，禁止让 mock 配置进入 `test/prod` 正式链路。
- 正式升级优先走受支持接口 `POST /api/runtime-upgrade/apply`，由 `scripts/start_workflow_env.ps1` 托管进程接管切换、健康检查与回滚，不要手工拷贝覆盖运行目录。
- 当 live `prod` 因当前 `running` 节点卡住、而更高 `candidate` 已通过门禁时，比起人工守着空窗，更稳的办法是挂一个外部 idle-upgrade watcher：只轮询 `/api/runtime-upgrade/status`，严格等到 `running_task_count=0 && can_upgrade=true` 后再调用 `/api/runtime-upgrade/apply`，并把等待/发起/apply 完成全写进独立日志。不要对当前执行中的节点做 exclusion，否则容易在任务还没正常收尾时就把当前进程切走。
- 若要把 idle-upgrade watcher 接进现网 `prod`，更稳的默认不是让它长轮询常驻在当前主线节点里，而是把它挂到 `start_workflow_env.ps1` 的 supervisor 旁边做“周期性单次检查”。当前默认保底间隔应控制在 `1` 小时级别，而不是 `2` 小时以上；没有 candidate、candidate 不更新，或当前还有 running task 时都直接跳过；只有命中 `0 running && can_upgrade=true` 才发起 `apply`。这样检查脚本自身失败也只是 fail-open，不会把现网主链一起拖下去。
- assignment finalize 不该把升级尝试和下一棒派发做成“二选一”。更稳的默认是：finalize 先补一次节流后的 `--single-check`，但不因为 `candidate_is_newer` 就硬冻结 follow-up dispatch；只有升级请求真正挂起后，才进入硬 drain。
- 如果当前 `[持续迭代] workflow` 还没收尾，下一条 future 就已经再次变成 `ready/queued`，现网会长期表现成 `1 running + 1 ready`，idle watcher 继续只会看到 `running_task_count=1`。这时重复做 manual single-check 基本不会带来升级空窗；更稳的默认是把问题提升成 `drain / dispatch 分流` 任务包，优先给升级链创造真正的 idle window，而不是把 watcher 误判成失效。
- 隔离 acceptance / dispatch probe 如果只是验证本地 runtime 行为，却没有显式把 `WORKFLOW_RUNTIME_ENV` 压到 `test`（或清空），会把 live `prod` 的 upgrade drain 直接串进夹具，现场表现成 `upgrade_drain_active:candidate_newer_pending_idle_window` 抢走 dispatch。更稳的默认是：这类 probe 在进入 dispatch 前先显式设置 `WORKFLOW_RUNTIME_ENV=test`，收尾再恢复原环境变量，不让 live `prod` candidate 窗口污染本地结论。
- 当当前工作区已经把 release boundary 修成“只以本机 `../workflow_code` clean_synced 为准、`origin/main ahead` 仅作上游参考”，但 live `prod` 还停在旧版本时，active schedule future 文本仍可能重新漂回 `root_sync_state=ahead_clean / push_block_reason=unpushed_commits_present`，`/api/status.active_version` 也可能继续落后。排查时先同时比对 `/api/runtime-upgrade/status` 里的 `current_version/candidate_version`、live `.running/prod/.../release_boundary_service.py` 与当前工作区同文件，再决定这是工作区边界失控，还是单纯等待 idle watcher 把候选切进现网。
- 当 AB 功能默认关闭时，`/api/status`、`/api/dashboard` 这类 PM version truth 读链不能把 runtime 侧占位值 `disabled` 直接拿去和 `pm plan active_version` 对比；更稳的默认是把 `disabled/ab_disabled` 视为“runtime version unavailable”，回退到 `active_slot=pm-plan / active_version_source=pm_version_plan`，否则会制造假阳性的 `truth_mismatch_count=1`。
- 当通过受支持的 `deploy_workflow_env.ps1 -Environment prod` 直接重发 `prod` 后，`.running/control/envs/prod.json`、`.running/control/instances/prod.json` 和 `/api/runtime-upgrade/status.current_version` 会切到新版本，但 `last_action` 仍可能停在更早一次 `watchdog_restart` 或旧升级记录。排查 live 当前版本时，不要只看 `last_action`，要同时核对 `current_version`、`envs/prod.json`、`instances/prod.json` 和最新 `deploy-*.json`。
- `deploy_workflow_env.ps1 -Environment prod` 不能只部署运行目录；如果这条直发路径不顺手调用 `Invoke-WorkflowRefreshPmCurrentVersionSnapshot`，`pm/PM当前版本计划.md`、`pm/versions/V3/版本计划.md` 和 `/api/status.pm_version_status.document_baseline` 会继续停在旧 baseline，即使 `runtime-upgrade/status.current_version` 已经切到新版本。更稳的默认是：prod 直发和 idle watcher/正式升级成功路径都统一刷新 PM current-version snapshot，并用 `verify_pm_current_version_snapshot_refresh.py` 锁住。
- `prod-idle-upgrade-watchdog` 的 `apply_wait_seconds` 不能继续停在 `240s` 这种紧窗口。像 `2026-04-15 21:29 +08:00 -> 21:34:09 +08:00` 这次现场里，watcher 会在 supervisor 真正切到 `20260415-205106` 前约 `40s` 先写超时，结果升级其实成功了，但后续的 PM current-version snapshot refresh 没跑，`document_baseline` 就会继续卡在旧版本。更稳的默认是把 watcher 等待窗口放到 `6~7` 分钟级，并让 `start_workflow_env.ps1 / workflow_env_common.ps1` 显式透传这条参数。
- 当 `prod` 已经重启并重新监听、`runtime-upgrade/status.current_version` 也已切到新版本，但 `/api/status.pm_version_status.document_baseline` 仍停在旧值时，不要只等下一轮主线/巡检顺手补文档。更稳的默认是把这类 document baseline drift repair 接进 `start_workflow_env.ps1 / workflow_env_common.ps1`：prod restart 后主动对照 live `/api/status` 与 `/api/runtime-upgrade/status`，发现 `document_baseline != prod=<current_version>` 就立刻调用受支持的 snapshot refresh helper 收口，避免 PM 文档长期落后一拍。
- `runtime_upgrade_service.runtime_snapshot()` 里的 `current_version` 不能优先信 `WORKFLOW_RUNTIME_VERSION` 这类进程环境变量；watchdog restart 或历史 launch 环境可能让它停在上一个版本。更稳的默认是优先读当前 runtime manifest / instance，再把环境变量只当兜底，并用 poisoned version 场景的定向验收锁住这条读链。
- `runtime-upgrade/status` 的运行门禁不能只看原始 assignment runtime metrics；它应该和 dashboard 共用 workboard fallback，把任务图/文件真相里的 `running` 节点补回 `running_task_count`，否则现网会出现 workboard 明明还有运行中任务、升级接口却给出 `can_upgrade=true` 的假空窗。
- 正式升级验收不能只停在“`current_version` 已切换”。至少还要继续核对一次任务中心 live 真相：是否已有旧 `ready` 节点被新版本自动接回到 `dispatch -> provider_start`，以及当前全局主图里是否还保留下一条 `ready/future` 入口；否则版本虽然已切换，7x24 连续性仍可能只是表面恢复。
- 当当前 `workflow(pm)` 主线或保底节点自己触发 `POST /api/runtime-upgrade/apply` 并成功切版时，要警惕“旧 run 已被升级切走，但节点仍长时间停在 `running`”的假运行。像 `2026-04-11 03:43` 这轮就是在 `2026-04-11 04:06:57 +08:00` 升到 `20260411-040421` 后，旧 run 直到 `2026-04-11 08:38:19 +08:00` 才被 stale recovery 回收；期间同 agent 的 `03:46` 保底节点只创建未派发，所以 token 会在升级时刻后停止。排查时先对照 `.running/control/prod-last-action.json`、旧 run `run.json/latest_event_at`，以及 audit / schedules 里的 `recover_stale_running`、`trigger_resume_requested`。
- 因此当前更稳的默认是：主线/保底节点只负责巡检升级门禁与续挂出口，不再靠 `exclude_assignment_*` 自己触发 `apply`；真正的自动升级申请统一交给独立 watcher/supervisor sidecar 发起。
- 当 `prod` 已被 idle watcher 自动切到新版本，但 PM 版本快照和 future schedule 文本还停在旧 baseline 时，不要只改 `pm/*.md` 就算收口；如果下一条主线节点已经 `ready`，还要同步用受支持的 `schedule update + schedule scan` 把旧 `ready` 覆成新的 trigger，否则下一棒仍会沿用旧 snapshot。
- 如果 workflow 主线/保底 schedule 的 `launch_summary` 仍只依赖 `schedule_plans` 里那份持久化快照，`prod` 自动升级后 `/api/schedules` 和后续新 trigger 仍会继续吃旧 baseline。更稳的默认是：schedule 的 `list/detail/calendar/scan` 读链在返回前动态覆盖 live PM version 与发布边界，把持久化 schedule row 只当静态骨架，不再要求每次升级后手工 refresh 才能把新 baseline 送进下一棒。
- `/api/schedules` 列表不能继续对每一条 schedule 都走完整 `_assignment_runtime_status()` 慢路径。像 `2026-04-16` 这轮现场里，live `prod=20260416-194053` 上 current-version smoke 直接打出了 `schedule_list_response_too_slow`：`/api/schedules=44926ms > 15000ms`。更稳的默认是：列表级读链优先走 task-file runtime preview，只把 `last_result_status / last_result_summary` 从文件真相投影出来；同时把 `/api/schedules` 的响应预算显式写进 current-version smoke，避免“列表能用但已经慢到看板不可接受”长期漏报。
- `collect_v4_r1_r4_current_version_smoke.py` 不能继续硬编码 `V4` 的 `lane`，也不能只看 browser regression。像 `2026-04-18 11:44 +08:00` 这轮现场里，live current-version smoke 会因为默认 lane 仍写成 `测试探测` 而假红，同时还会漏掉 workboard `recent failure pool` 的 live 排序合同。更稳的默认是：smoke 先从 PM 版本真相解析 `active_version / lane / lifecycle_stage`，再显式断言 `workflow` 的 `recent failure pool` 保持 recent-first 且只露前四条。
- 即使 `/api/schedules` 已经把运行态摘要切成 task-file preview，也不能在 `_row_to_schedule()` 里对每一条 workflow 主线/巡检 schedule 重复跑 `collect_release_boundary_snapshot()`。像 `2026-04-16` 这轮现场里，live runtime root 上只剩 `4` 条历史 workflow 主线/巡检 schedule，这条重复 release-boundary 读取仍会把列表拖回 `32s+`。更稳的默认是：把 workflow schedule 的动态 prompt 上下文收成“当前请求级缓存”，让 `list_schedules()`、`run_schedule_scan()` 和恢复循环共享同一份 PM version / release-boundary 快照，必要时只在 detail 里单次刷新。
- 顶层便捷启动入口如果承接 `prod`，不能直连部署副本里的 `launch_workflow.ps1`。必须走 supervisor 入口；否则 `POST /api/runtime-upgrade/apply` 只会把当前 Web 进程停掉，升级请求会一直卡在 `request_pending=true`，没有外层进程负责真正切换 candidate。
- 顶层 `run_workflow.bat` 如果需要承接 `prod`，默认应优先调用当前默认开发工作区里的 `scripts/start_workflow_env.ps1`，只在工作区 supervisor 缺失时才回退到部署副本；这样入口总是跟着最新主线脚本走，不会再被旧 deploy copy 里的 `source_root`/`.runtime` 漂移拖回历史坑。
- 顶层便捷启动入口如果只是为了把现有 `prod` 站点拉起来，默认不要再额外跑一次 `backfill`；对于已有较大 runtime 数据的 `prod`，这一步会显著拖慢启动，体感上像“入口没反应”。更稳的做法是顶层 wrapper 在 prod 模式下默认传 `-SkipBackfill`，把回填留给单独维护动作。
- 部署副本启动前，要用 manifest/runtime descriptor 回写 `.runtime` 内的关键配置，尤其是 `agent_search_root`、`artifact_root`、`task_artifact_root`，否则部署副本容易沿用旧路径。
- `test/dev` 部署在解析 `agent_search_root` 时，不能盲信自己 runtime 目录里遗留的旧配置；更稳的默认是优先继承当前 `prod` 已验证可用的 `agent_search_root`，否则很容易在 test gate 里出现 `available_agents=0`、`assignment test-data bootstrap failed` 这种假性发布阻塞。
- 设备路径配置在 Windows 运行态里不要用 `Path.resolve()` 做高频绝对路径归一化；这类读链路更适合用纯字符串级 `abspath/normpath`，否则 assignment/dashboard 这类只读接口也可能卡进 `realpath` 慢路径。
- `work_record_store` 的 SQLite 索引读链不能在每次 GET 上都重复跑全量 `_ensure_schema` DDL。更稳的默认是先看 `index_meta.schema_version`，命中当前版本就直接走快路径；否则 `CREATE TABLE/INDEX IF NOT EXISTS` 在 live 写入竞争下也会把 `/api/status` 这类只读接口拖到 10-20 秒。
- `work_record_store` 的 SQLite 只读查询（例如 `/api/status -> pending_counts_from_index`）不要在 `ensure_sqlite_index()` 里每次都重写 `artifact_root` 这类元数据。更稳的默认是：只有元数据真的变化时才 `_set_meta + commit`；正常读链保持只读，否则 live 并发下很容易直接打成 `sqlite3.OperationalError: database is locked`。
- runtime-config 只有在已经启用 `device_path_configs` 时，保存路径 patch 才应该写回当前设备槽位；如果当前 payload 还只有顶层 `artifact_root/agent_search_root`，强行自动创建设备槽位会把 gate/test 这类临时 runtime 的产物路径覆写到当前设备默认 `.output`，随后 `task_agent_runner` 会到错误目录里读 session，表现成 `agent_policy_extract_failed:session_not_found`。
- 即使 `deploy_workflow_env.ps1` 已经显式传了 `-AgentSearchRoot / -ArtifactRoot`，当前环境的 `runtime-config.json` 也必须至少保持“本机可解析”。如果里面还残留一个当前机器不存在的盘符，`Get-WorkflowRuntimeConfig` 会先在路径派生阶段崩掉，导致覆盖参数来不及接管；这种现场要先把 runtime-config 修回可解析路径，再重跑部署。
- 当 `deploy_workflow_env.ps1 -Environment test` 因健康实例而 fail-closed 时，不要再手工 `Stop-Process` 或靠删 `pid_file/instance_file` 硬闯。更稳的默认是先走受支持的 `scripts/stop_workflow_env.ps1 -Environment test`：只在识别到 `workflow_web_server.py + runtime_root` 命中的受信实例时才停服，`prod` 仍要求显式 `-AllowProdStop`，然后再重跑部署刷新新的 `test / prod candidate`。
- `runtime_upgrade_service.current_runtime_base_url()` 不能盲信当前 `instances/prod.json` 里的 `host/port`。像 `2026-04-15 06:17 +08:00` 这次现场里，`followup_dispatch_blocked -> request_prod_auto_upgrade_single_check()` 明明在 `prod` 进程里触发，却因为 `instances/prod.json` 被污染成 `environment=test / port=8168`，把 drain-hit single-check 打到了关闭端口，日志直接落成 `WinError 10061`，结果主线长期停在 `candidate_newer_pending_idle_window`。更稳的默认是：`current_runtime_instance()` 先校验 `environment / control_root / manifest_path / deploy_root` 是否与当前 runtime 真相一致；`current_runtime_base_url()` 再优先用 manifest 的 `host/port`，只把 instance 当补充来源。
- `dashboard /api/status` 里的 `schedule_workboard_preview` 不能复用完整 `list_schedule_preview` 详情构造。更稳的默认是给 workboard 单独走 compact preview：只保留 `schedule_name / next_trigger / last_result_*` 等工作面必需字段，并仅用 `_assignment_runtime_status_from_task_files()` 这类文件快路径补运行态；否则整段 `launch_summary / execution_checklist / done_definition` 和慢补算会把首屏状态请求拖到几十秒。
- 如果用户明确要求“强制升级/强制部署 prod”，不要只停掉当前 `workflow_web_server.py` 进程；`start_workflow_env.ps1` supervisor 和 `launch_workflow.ps1` 子进程可能会立刻把旧 prod listener 重新拉起，导致 `deploy_workflow_env.ps1 -Environment prod -AllowDirectProdDeploy` 持续 fail-closed。更稳的默认是：先显式停掉 `prod` 的 supervisor / launch / web 三层进程，再清理 `prod.pid / instances/prod.json`，随后执行一次带 `-AllowDirectProdDeploy -StartAfterDeploy` 的正式部署，并在部署后立刻核 `healthz`、`envs/prod.json`、`instances/prod.json`。
- `deploy_workflow_env.ps1` 判断 `dev/test/prod` 是否“当前正在运行”时，不能只信 `pid_file` 里那个进程还活着；更稳的默认是同时核对“预期端口是否真的在监听”。如果 pid 还活着、实例文件也对得上，但目标端口根本没有 listener，这应直接按 stale instance state 清掉并继续部署；反过来，只要预期端口仍被占着，就继续 fail-closed，避免把活环境误当死环境覆盖掉。
- `start_workflow_env.ps1` 如果允许在 `.running/<env>` 部署副本里直接执行，必须先从 `.workflow-deployment.json` 或 `.workflow-local-deployment.json` 反解真实 `source_root`；否则它会把部署副本本身误当源码根，再递归部署出 `.running/<env>/.running/<env>` 这种嵌套运行树。
- 如果已经误拉起了“嵌套 `.running` 的假 prod”，不要继续在那份错误实例上叠补丁；先停掉错误 listener，再用部署副本的 `launch_workflow.ps1` 显式传入正确的 `RuntimeRoot / RuntimeManifestPath / RuntimeDeployRoot / RuntimeInstanceFile / RuntimePidFile`，把正式入口拉回到正确的 runtime。
- 当用户反馈“页面显示运行中 1 条，但 token 没增长”时，不要只看 `/api/status`；先直接核 `runs/<run_id>/run.json`、`events.log` 和对应 `node.json`。真实现场很可能已经在几秒前 `exit=1` 或被系统以“运行句柄缺失”回收，页面只是还没来得及刷新。
- 升级 banner 这类顶部浮层不能只看接口返回或 DOM 节点是否存在；必须补真实截图。字段赋值前一旦有前端异常，页面会留下“标题还在、正文全空”的假渲染。
- 训练优化 create 态不要直接复用 status 态的完整能力卡。create 阶段信息密度更高，中部主列又更窄，应该用更紧凑的草案卡和更短的对话文案。
- 训练优化 create 态要判断“是否真正铺满父容器”，不能只看 `composer_bottom_gap/right_bottom_gap=0`。必须同时对照“创建角色”界面的壳体模型，确认 `detail_body` 与 `chat_shell` 顶底完全对齐、底部 composer 用 `border-top` 直接贴底、右侧由外层 panel 承担壳体而不是内层再套一张大卡片。
- 拼接式前端 CSS 一旦上游文件少一个 `}`，后续样式文件会被级联污染，表面症状往往落在下游页面。训练优化样式回归时，要同时检查合并后的 `/static/workflow-web.css` 和 probe 里的计算样式，不要只盯当前 CSS 文件。
- `scripts/start_workflow_env.ps1` 这种长驻 supervisor 脚本一旦已经启动，就会把当时的逻辑常驻在内存里。后续只改磁盘文件不会影响当前正在跑的 supervisor；如果本轮修的是升级/健康检查逻辑，必须先重启 `prod` 的 supervisor，再重新发起正式升级。
- 在当前 Codex/终端环境里，如果只是从一次性 shell 会话里 `Start-Process powershell ...start_workflow_env.ps1`，外层会话结束后 supervisor 可能不会稳稳留住，结果只剩被拉起的 web 进程短暂存活、watchdog 本体却已经消失。要把 `prod` 真正交给 watchdog 托管，至少要确认 `start_workflow_env.ps1` 和它拉起的 `launch_workflow.ps1` 进程都还在；必要时用更脱离当前 shell 作业树的创建方式，比如 `Win32_Process.Create`。
- Windows 机器从睡眠恢复后，`prod` 还会出现另一种近似分叉现场：`8090` 一度不可达、`prod.pid` 缺失、`.running/control/instances/prod.json` 已写成 `stopped`，随后又可能残留一个孤立 `python workflow_web_server.py` 或 stale listener。更稳的默认是：先用受支持的 `stop_prod_workflow_env.ps1` 清掉当前 `prod` listener 和 `pid/instance`，确认不存在重复的 `start_workflow_env.ps1`，再只启动一份 `start_workflow_env.ps1 -Environment prod -SkipBackfill`；恢复完成要同时看到 `healthz=true`、`/api/runtime-upgrade/status.supervisor_attached=true` 和 `can_upgrade/running_task_count` 回到一致真相，不要在已有孤立 server 上继续叠多份 supervisor。
- 验证 `pm-*` helper 派发是否真的修好时，不能只看 HTTP 请求有没有在超时前返回；更该直接核 assignment audit、`runs/<run_id>/run.json` 和 `events.log`。只要新的 `pm-*` dispatch 已经落审计、并且 run 在请求结束后还能继续推进到 `provider_start/turn.started`，就说明“派发留存”成立；后续如果任务仍失败，应单独按 worker 执行问题继续查，不要再把两类故障混成“没派发起来”。
- 临时 PowerShell 取证脚本如果需要按可选标识过滤 audit lines，不要把 mandatory 的 `[string[]]` 参数直接喂进可能为空的元素；应先在调用侧过滤空值，或让函数本身接收空数组而不是空字符串。否则像 `Read-AuditLines -Patterns @('', ...)` 这类调用会在参数绑定阶段直接失败，让 helper 明明已经拿到主要证据，却卡死在脚本自身。
- 当 `/api/status` 或任务图里出现一大串 `failed_task_count` 时，不要先把它等同于“当前有这么多异常正在发生”。更稳的默认是先拆三层：哪些只是全局主图里保留下来的历史失败节点，哪些是升级切换导致的可接受 `cancelled/failed`，哪些才是最近一两轮里的新异常。像 `Get-Content` 读不存在的旧 run 路径、或在 Windows 下对 `rg` 传 `scripts/*.ps1` 这种 glob 失配，应归为当前异常；而“正式环境已升级，当前批次自动结束”更接近合理失败/切换副作用。
- `agent_registry.runtime_status` 不能当 live execution 真相源。像 `workflow` 这种场景里，即使任务图、`status-detail`、`run.json/events.log` 已经证明同一个 agent 正在真实运行，`agent_registry` 也可能还停在 `idle` 或落后多个分钟；值班巡检、升级门禁排查和单 agent 并发判断都应优先以 `status-detail + run.json + events.log + node.json` 为准，把 `agent_registry` 只当辅助口径。
- assignment workspace 解析不能只信 `agent_registry` 或 `discover_agents` 缓存。像 `2026-04-15 18:39 +08:00` 这类现场里，只要 registry 短暂缺行或 `discover_agents` 绑定失效，`workflow` 主线就会直接打成 `assignment_agent_workspace_missing`。更稳的默认是：在 `agent_search_root/<agent_id>/AGENTS.md` 存在时，assignment 直接回落到这个确定性路径，并用专项 probe 锁住。
- execution worker thread 不要在 dispatch lock 还没释放时就启动。更稳的默认是：先把 dispatch snapshot / audit / run record 全写稳，再启动 worker；否则一旦 worker 立即进入 finalize/recovery，就可能和同 ticket 的半写入 dispatch 互相踩脚。
- 角色创建会话也不能只信 `role_creation_sessions` 这一张表。像 `workflow_ucdmate` 这类现场里，即使 `session.status` 还停在 `draft/workspace_init`，只要 `agent_registry`、角色工作区和全局主图 starter 链已经证明该角色真实启动，`training center detail/list` 就应该自动从 live 真相回填 `creating / workspace_init_ref / assignment_ticket_id / created_agent_* / task_refs`，而不是继续把用户困在假草稿态。
- 运行态角色工作区如果本地 `AGENTS.md` 启动顺序要求读取当日日记，而 `.codex/memory/YYYY-MM/YYYY-MM-DD.md` 当天文件缺失，helper run 会在真正开始任务前就直接因读文件失败退出。重跑 helper 前要先补齐角色工作区当天日记骨架，避免把“工作区上下文缺失”误判成派发链路问题。
- 更稳的默认不是等 helper 第一次失败后再人工补文件，而是在 assignment 解析目标工作区时就做一次轻量 memory scaffold：只要该工作区声明了 `.codex/memory/` 读链，就自动补齐 `全局记忆总览.md`、当月 `记忆总览.md` 和当日日记骨架。这样小伙伴任务第一次派发就能直接进入真实执行，不会再死在 `AGENTS.md` 启动阶段。
- 当多个 helper 第一次在当前机器上并发启动、且其中某个运行态工作区还没写进 `C:\\Users\\<user>\\.codex\\config.toml` 的 trusted project 记录时，这条 node 可能会直接失败在 `failed to persist trusted project state: failed to persist config.toml`。更稳的默认不是把它误报成产品 smoke 失败，而是先确认这属于 helper 启动瞬时故障，再通过受支持的 `POST /api/assignments/{ticket_id}/nodes/{node_id}/rerun` 把 failed 节点接回；如果其他 helper 已经把配置写稳，重跑通常就能继续进入真实执行。
- 生产环境创建出来的角色，不能在任务执行 prompt 里被降成 generic `worker` 壳。prompt 必须明确“它现在是以自己身份工作”，并提醒先遵守该工作区的 `AGENTS.md / SOUL / USER / MEMORY` 读链；如果工作区使用 `.codex/memory`，还要明确要求以第一人称、带一点日记感地更新当日日记，否则角色很容易只交付产物、不留下自己的工作记忆。
- 对带 `.codex/memory` 的角色工作区，不能只靠 agent 自觉写记忆。更稳的默认是在 assignment finalize 阶段做后端兜底：只要节点成功或失败收尾，就自动把 `ticket/node/run/result` 补记进当天日记，并保留 audit。这样即使角色中途忘写、或交付链只返回 JSON 产物，也不会出现“任务做了但今日日记完全空白”的现场。
- assignment finalize 给 `.codex/memory` 补写今日日记时，不能继续把 `result_summary / warning / failure` 直接按严格 `_normalize_text(max_len=...)` fail-closed 校验。像 `2026-04-21 06:51 +08:00` 这类现场里，真实 run 已经完成，但单条摘要过长会把 `append_workspace_memory_failed` 留成长期治理债务。更稳的默认是：先做 whitespace compact，再按日记字段上限截断成 fail-open 文本，并用 `verify_assignment_workspace_memory_writeback.py` 锁住“长 summary / 长 warning / 长 failure 仍能正常落日记且不再写失败 audit”。
- 当 node 文件还停在 `running`，但对应 `run.json` 已经是 `failed/succeeded/cancelled`，而且磁盘上已经有 `result.json` 或 `final_result` 证据时，读链不能再把它统一回收成“运行句柄缺失”。更稳的默认是先投影 terminal truth，再决定是否需要真正的 stale recovery；否则会把真实终态抹成另一种失败真相。
- 当 assignment node 新增了 `interface_catalog_entry` 这类消费型字段时，不能只靠“创建新节点时顺手写入”来修 readback。像 `2026-04-22 00:44 +08:00` 这轮现场里，历史 `V7` 节点文件里根本没有这个字段，`status-detail.selected_node` 就会继续回空对象。更稳的默认是：只要节点已经带了 `problem_type / method_card_id / return_contract` 这类合同字段，read-model 序列化时就要按当前 runtime 的接口目录能力动态回补 `interface_catalog_entry`，并用 legacy node probe 锁住，不把历史节点升级成本继续推给人工迁移。
- `exit=1` 不等于“没有结果”。只要这轮已经写出了真正的结构化结果正文、附件或 warnings，失败收尾也应保留 `result_ref` 并把失败摘要和结果上下文一起写清；但纯 timeout / 纯空壳失败不要误标成“已保留结构化结果”，否则会继续误导排障。
- failed/cancelled 节点若为了诊断继续保留 `result_ref`，`status-detail` 就不能再把同一 failed run 的 `role_quality_assessment=pass` 直接投影成 node-level pass。更稳的默认是：诊断 `result_ref` 可以保留，但 node-level quality 只对 `succeeded` 节点生效；若确实要暴露 failed run 的评分，也必须显式区分成 `diagnostic_*` 或 `result_ref_semantics=diagnostic_failed_run`，否则会稳定制造 `selected_node.status=failed` 但 `role_quality_assessment=pass` 的 mixed truth。
- 在 Windows PowerShell 里手工调用本地 `workflow` API 创建任务节点时，如果 `POST /api/assignments/{ticket_id}/nodes` 的 JSON body 直接用 `ConvertTo-Json` 输出的普通字符串发送，中文 `node_name / node_goal` 可能会被 live API 写成 `?`；更稳的做法是显式把 JSON 编成 `UTF-8` bytes 再发。这个坑不会阻止节点真实 `dispatch`，但会把任务图和交付目录里的中文标题污染掉。
- 核对 assignment 节点 live 状态时，`status-detail` 的稳定接口是 `/api/assignments/{ticket_id}/status-detail?node_id=<node_id>`，不是 `/api/assignments/{ticket_id}/nodes/{node_id}/status-detail`。后者会返回 `not found`，容易让人误以为读面坏了；更稳的默认是先查路由或接口目录，再用 status-detail 返回的 `run_id/result_ref/stdout_ref/stderr_ref/execution_truth` 决定是否需要直读 run 文件。
- 在 Windows PowerShell 里手工构造本地 `workflow` API 的 JSON body 时，不要在双引号 here-string 里夹带反引号包裹的字段名或示例（例如 ``ticket_id``、``node_id``）。PowerShell 会把 `` `a ``、`` `n `` 这类序列解释成控制字符，导致 `node_goal` 被污染。更稳的默认是：API payload 里的示例字段不用反引号包裹，或改用单引号 here-string，再配合 `UTF-8` bytes body 发送。
- 当前本地 `workflow` API 的 hand-crafted `create_assignment_node` 也不要继续假设“只要客户端发的是 UTF-8 bytes，中文 `node_name / node_goal` 就一定能保真”。像 `2026-04-18 23:06 +08:00` 这轮现场里，即使改用 Python `urllib` + `utf-8` body，live 节点文件里的中文正文仍会落成 `?`。更稳的默认是：手工 live 派单时优先用 ASCII-safe `node_name/node_goal`，把中文细节放进可回读的 ASCII 路径产物（例如 `result.json` / markdown artifact）里；如果必须保中文，就优先走产品内现成创建入口或先修 server 侧 decode。
- 在 live `prod` 上手工调用 `POST /api/projects/bootstrap`、`POST /api/assignments/<ticket>/nodes` 或 `POST /api/assignments/<ticket>/dispatch-next` 时，客户端 `Invoke-RestMethod` 超时不等于动作失败。像 `2026-04-19 15:23 +08:00` 这轮现场里，客户端对 `create_assignment_node / dispatch-next` 都报了 timeout，但 `project-registry.json`、`node.json`、`run.json` 和 `status-detail` 随后都证明第二项目对象、starter node 和真实 run 已经创建成功。更稳的默认是：命中这类超时后，先查 `project-registry.json / node.json / run.json / status-detail` 的文件与 API 真相，再决定是否重试；否则很容易因为“以为没成功”而重复建项目或重复派发节点。
- `POST /api/assignments/<ticket>/nodes` 客户端超时后也不要立刻补发同义 create。像 `2026-04-22 04:51 +08:00` 这轮现场里，第一条 `workflow_devmate` 的 `V7-R4` implementation node 实际已经落盘；如果按“超时=失败”继续补发，就会多造一条 duplicate ready node。更稳的默认是：先回读 `audit.jsonl / node.json / status-detail` 确认是否已经创建成功；若已经重复建出节点，就保留带完整 upstream/downstream 的 canonical chain，再用受支持的 `DELETE /api/assignments/{ticket}/nodes/{node_id}` 清掉多余 ready node。删除或后续 `dispatch-next` 若继续超时，也仍然先以 `audit / node / run` 真相为准，不盲重试。
- 共享同一份 `prod` runtime root 的多 listener 不能继续只靠进程内 `_assignment_ticket_dispatch_lock()` 保命。像 `2026-04-22 10:15 +08:00` 这轮现场里，authoritative `8090` 和脱管 `8098` 会同时 dispatch 同一条 `ready` 主线 node，留下两条真实 run。更稳的默认是：在 assignment dispatch 路径加跨进程可见的原子 claim，并在 `finalize / stale recovery / cancel / rerun / delete` 统一 release；同时一旦识别出对准同一 `prod` runtime root 的 stray listener，要优先清掉旧 listener 树，而不是只盯着 in-memory lock 继续观察。
- `verify_assignment_cross_process_dispatch_claim.py` 这类 isolated dispatch probe 不能直接继承 gate fixture 的 `WORKFLOW_RUNTIME_ENV=prod`。像 `2026-04-22 16:12 +08:00` 这轮现场里，只要 gate runtime 恰好挂着更高 `candidate_version`，两个 worker 就会在 claim 竞争前先一起吃到 `upgrade_drain_active`，把不相关 probe 短路成假失败。更稳的默认是：这类只验证 isolated runtime 锁竞争/派发行为的 probe，在调用 `dispatch_assignment_next()` 前显式切到 `WORKFLOW_RUNTIME_ENV=test`；prod drain 则单独交给 `verify_runtime_upgrade_dispatch_drain.py` 覆盖。
- 页面刷新长期停在“正在恢复数据”时，不能只按浏览器启动慢处理。像 `2026-04-26 13:00 +08:00` 这轮现场里，`/healthz` 和 `/api/projects` 很快，但 `/api/assignments/<ticket>/graph` 与 `status-detail` 超过 75 秒仍不返回；直接读 SQLite 只要几十毫秒，真正慢点是 assignment 读面按 `task_artifact_root=C:/work/J-Agents/.output` 扫 1500+ node / 800+ run 的历史主图，并且读接口会进入 `_assignment_snapshot_from_files(... persist_changes=True)` 的同 ticket mutation lock 与 stale recovery。更稳的默认是：先用 `healthz/projects/status/graph/status-detail` 分层点测，再对照 `state/runtime-config.json` 的 `task_artifact_root` 和实际 `tasks/<ticket>/nodes|runs` 数量；修复上要把页面恢复读面改成轻量只读 / fail-open，不在 startup、graph 或 status-detail 里持 ticket mutation lock 做慢恢复。
- 把 assignment 页面读面改成只读后，旧 stale recovery 用例不能再靠 `get_assignment_graph/status-detail` 触发写回；这会把性能修复又退回读时持锁。更稳的默认是：页面读模型用 `persist_changes=false / reconcile_running=false` 验证不抢 mutation lock，stale recovery/terminal truth 用例改走显式维护入口 `_assignment_snapshot_from_files(... reconcile_running=True, persist_changes=True)`，分别锁住“读快”和“维护仍能修”两条合同。

## 已踩过的坑
- 坑 1：把长任务 agent 调用按固定 180 秒超时处理，首轮策略分析会因为真实执行时间不稳定而大量误判失败。
  - 避免方式：改成“进程存活监控 + 结果收敛”，并增加单独验收用例覆盖长调用。
- 坑 2：页面提示“正式升级正在切换中”后卡住，前端重连态没有根据后端终态及时消退。
  - 避免方式：重连与成功态统一以后端 `/api/runtime-upgrade/status` 的 `request_pending`、`last_action`、当前版本为准，不使用前端本地粘滞态当真相源。
- 坑 20：`/api/status` / workboard 已经能看到 `running_task_count=1`，但 `/api/runtime-upgrade/status` 仍然只读到原始 assignment runtime metrics 的 `0`，结果正式升级入口被错误放开。
  - 避免方式：升级门禁和 dashboard 共用同一条 workboard fallback；如果任务图文件真相里已经存在 `running` 节点，就把它回补到升级门禁的 `running_task_count / agent_call_count`，避免出现“假空闲”的升级窗口。
- 坑 24：AB 功能已经关闭，但 `/api/status` 仍把 `runtime_active_version=disabled` 直接和 `pm plan active_version=V1` 对比，导致看板持续误报 `truth_mismatch_count=1`，把“未升级到新候选”与“运行真相分叉”混成同一个阻塞。
  - 避免方式：把 `disabled/ab_disabled` 归一成 runtime truth unavailable，不再参与 mismatch 计算；live 是否落后改由 `current_version / candidate_version / candidate_is_newer` 单独表达。
- 坑 3：部署副本切换后仍沿用旧 runtime 配置，导致工作区路径或产物路径漂移。
  - 避免方式：启动部署副本前显式同步 runtime-config，并在正式升级后核对 manifest、instance、healthz 三处状态是否一致。
- 坑 4：升级 banner 只渲染出标题和空白骨架，误以为是后端没返回数据，实际是前端 refs 漏登记后在赋值前抛异常。
  - 避免方式：给 banner 结构节点做完整 refs 登记；修这类浮层时同时检查“字段赋值”和“display/aria-hidden”是否被硬编码覆盖。
- 坑 5：训练优化 create 态复用 status 卡片后，中部能力卡文本堆叠、右侧预览又挤占宽度，真实截图看起来像“样式没生效”。
  - 避免方式：create 态单独收口卡片结构，优先保证主阅读区单列可读；右侧演进图在 create 态取消大 `min-width` 横向挤压。
- 坑 6：上游 CSS 文件语法断裂后，`workflow-web.css` 里后续训练优化规则虽然还能看到文本片段，但浏览器计算样式已经回退成默认块级布局，导致中部输入区和右侧铺满同时失效。
  - 避免方式：训练优化布局异常时，优先检查拼接后的 `/static/workflow-web.css` 是否在当前文件前一段出现未闭合规则；再用 `tc_probe=1` 输出 `display/clientHeight/scrollHeight` 验证真实生效情况。
- 坑 7：布局探针已经显示 gap=0，但真实截图仍然像“没有铺满”，根因是中部和右侧仍在使用“卡片套卡片”的嵌套壳，视觉上和创建角色单壳体完全不是一个模型。
  - 避免方式：出现这类反馈时，不要继续微调 `padding/gap/max-height`；直接把训练优化 create 态改成创建角色同款的“头部固定 + 中间滚动 + 底部贴底 composer / 右侧 outer panel + inner scroll”结构，再重新截图验证。
- 坑 8：已经把 `start_workflow_env.ps1` 的升级健康检查超时修到磁盘了，但正式环境仍然继续按旧的 60 秒逻辑回滚，原因不是补丁没生效，而是当前 `prod` supervisor 进程早就把旧脚本加载进内存了。
  - 避免方式：凡是修正式升级 supervisor 逻辑，补丁落盘后都要先重启 `prod` 的 `start_workflow_env.ps1` 进程，再重试 `/api/runtime-upgrade/apply`；不要直接假设现网 supervisor 会热加载脚本改动。
- 坑 9：顶层 `run_workflow.bat` 直接调用部署副本的 `launch_workflow.ps1`，看起来能正常开站，但一旦点“升级正式环境”，请求只会写进 `prod-upgrade-request.json` 并让当前进程以 `73` 退出，随后没有任何 supervisor 接手切换，页面就永久卡在“正式升级正在切换中”。
  - 避免方式：顶层入口统一改走 supervisor；如果历史现场已经留下悬挂的 `prod-upgrade-request.json`，先清掉挂起请求，再用受 supervisor 托管的入口重新拉起服务。
- 坑 14：`test` 运行目录里曾经留下旧的 `agent_search_root=C:\\work\\J-Agents`，后续 `deploy_workflow_env.ps1 -Environment test` 又优先沿用了这份残留配置，结果 test gate 明明是新代码，仍然报 `available_agents=0` 和 `assignment test-data bootstrap failed`。
  - 避免方式：非 prod 部署默认优先继承当前 `prod` runtime 已验证可用的 `agent_search_root`；只有用户显式传参或 prod 也没有有效值时，才退回本环境旧配置或默认路径。
- 坑 15：prod 已停机后，直接执行部署副本里的 `start_workflow_env.ps1` 可能又把部署副本自身当成源码根，拉起一个 `deploy_root/.running/control/runtime/prod` 的错误实例；表面上 `8090` 会起来，但 `agent_search_root=''`、`available_agents=0`、`schedule_total=0`，本质上是假的空壳 prod。
  - 避免方式：出现这种现象时先看监听进程命令行里有没有 `.running\\prod\\.running\\control\\runtime\\prod`；若命中，立刻停掉并改用 `launch_workflow.ps1` 显式传正确 runtime 参数恢复。
- 坑 19：顶层 `run_workflow.bat` 走 `.running/prod/scripts/start_workflow_env.ps1` 时，如果部署副本里的 `.runtime/state/runtime-config.json` 还残留旧机器的 `D:` 路径，会先在 `workflow_env_common.ps1` 的运行态路径派生阶段直接炸掉；就算把这处修完，若部署副本里的 `start_workflow_env.ps1` 还没带上 `Resolve-WorkflowStartSourceRoot` 修复，又会继续把 `.running/prod` 自己当成源码根，递归生成 `.running/prod/.running/...` 假实例。
  - 避免方式：先把 `.running/prod/.runtime/state/runtime-config.json` 修回当前机器可解析路径，再确保部署副本里的 `start_workflow_env.ps1` 含有 `Resolve-WorkflowStartSourceRoot` 逻辑；顶层 wrapper 还应在 prod 模式下默认加 `-SkipBackfill`，避免启动长时间卡在回填。
- 坑 23：`deploy_workflow_env.ps1` 只要看到 `pid_file` 对应的进程还活着，就把环境认成“正在运行”；如果当时留下的是一个不再监听目标端口的陈旧 pid，这条部署链就会被永久卡在“请先停止后再部署”，只能靠人工删实例文件或杀错进程解锁。
  - 避免方式：部署前同时核对预期端口的真实 listener。若 pid 还活着但目标端口没有 listener，直接按 `expected_port_not_listening` 清理 stale `pid_file/instance_file`；若端口仍被占用，则继续 fail-closed，不要误清活环境。
- 坑 16：任务中心里 `running_task_count=1` 并不等于模型还在持续消耗 token。像 `arun-20260406-011407-ffb18d` 这种 run，可以在 `provider_start -> turn.started -> final_result(exit=1)` 后几秒内就结束，再被系统自动标记为 `cancelled/failed`；如果只看顶栏数字，会误以为“还在跑但就是不吃 token”。
  - 避免方式：先核 `run.json.status / latest_event_at / finished_at` 和 `node.json.status`，确认是不是已经自动收敛或回收；再判断是否需要重排任务。
- 坑 17：`test` 运行态里如果还留着当前机器不存在的盘符（例如历史 `D:`）并且该值落在 `agent_search_root / development_workspace_root / agent_runtime_root` 这些会参与派生的字段里，`deploy_workflow_env.ps1` 会在读取旧 `runtime-config` 时直接因为 `Join-Path drive not found` 中断，即使本次命令已经显式传了新的 `C:` 覆盖参数。
  - 避免方式：先把对应环境的 `.running/control/runtime/<env>/state/runtime-config.json` 修成当前机器可解析的路径，再执行部署；必要时补一轮健康检查，确认新的 runtime-config 已被 manifest 和实例文件接管。
- 坑 18：给 runtime-config 保存逻辑补“当前设备槽位回写”时，如果没有区分“已启用 device_path_configs”和“只有顶层路径字段”，test gate 这种临时 runtime 会在第一次保存路径 patch 时被塞进一个当前设备专属槽位，且该槽位往往继承默认 `.output`；后续 chat/task 读 session 就会去错目录，统一表现成 `session_not_found`。
  - 避免方式：只有在 payload 已经存在 `device_path_configs` 时才更新/新增设备槽位；否则继续写顶层字段，保持 gate/test/prod 现有 runtime 行为不变。
- 坑 10：角色创建 creating 清理删除链路里，脚本先补做 starter task 的交付/完成，再等待 delete 开放；如果这段没有 heartbeat，十分钟内都可能没有任何输出，外部观察者会误判成“整体退出”。
  - 避免方式：验收脚本对“本地慢步骤 + 慢 HTTP 请求”都加 heartbeat；同时不要复用底层固定 180 秒 HTTP 超时，改用可配置长超时。
- 坑 25：角色工作区、agent_registry 和 starter 节点都已经在 live 里启动了，但 `training center` 会话仍停在 `draft/workspace_init`，结果 PM 版本面和角色中心会同时出现“角色已经在跑 / 会话还没开始”的分叉真相。
  - 避免方式：`role_creation_session detail/list` 读链必须加一层 live recovery。命中“draft 会话 + 可启动 role_spec + live agent_registry + 全局主图 starter 链”时，自动回填 `creating / workspace_init_ref / assignment_ticket_id / created_agent_* / role_creation_task_refs`，并用专项 probe 锁住这条修复。
- 坑 11：验收脚本继续断言“cleanup 后 assignment graph 必须 404”，但产品真实语义已经变成“图可能为空而非物理删除”，会制造假失败。
  - 避免方式：每次优化产品清理语义后，同步回看 acceptance 断言；只断言用户可感知/契约级结果，不把历史内部实现细节当成稳定门禁。
- 坑 12：把 `all` 全量 acceptance 当成每次提交前的默认检查，会让单次验证时间膨胀到三四十分钟甚至更久；一旦外层执行器再叠加超时或人工中止，就容易被感知成“一个小时还没测完”。
  - 避免方式：把长链路验收拆成 phase，并按“开发期最小覆盖、发布前全量回归”的节奏执行；同时在报告里明确每个 phase 的真实耗时。
- 坑 13：单项真实测试已经证明关键能力没问题后，又把同一条慢链路放进真实全流程再跑一遍，本质是在重复付费，既拖慢节奏，也放大长任务偶发波动。
  - 避免方式：把验证拆成“真实单项 + mock 全流程”；只有真实依赖本身就是本轮变更对象时，才补真实全流程。
- 坑 21：给 `workflow_testmate / workflow_devmate / workflow_bugmate / workflow_qualitymate` 这类运行态角色派单时，如果工作区跨了新的一天、但 `.codex/memory/YYYY-MM/YYYY-MM-DD.md` 还没创建，任务会在真正执行前就因为启动读取链缺文件而退出，表面像“helper 又失败了”，其实不是执行逻辑本身出了新问题。
  - 避免方式：不要只靠人工记得去这些工作区补日记；在 assignment workspace 解析阶段就自动补齐 `全局记忆总览.md`、当月 `记忆总览.md` 和今日日记骨架，并用定向验收覆盖这条 bootstrap。
- 坑 22：在 Windows PowerShell 里手工用普通字符串 body 调 `POST /api/assignments/{ticket_id}/nodes`，中文 `node_name / node_goal` 会被 live 节点记录成一串 `?`，结果任务虽然真实开始了，但看板和交付目录标题被污染。
  - 避免方式：这类手工节点创建统一改成 `UTF-8` bytes body；如果已经误建出乱码节点，不要因为标题脏了就重置任务图，优先保留现有 node_id/run 证据并从下一条节点开始修正调用方式。
- 坑 26：把 current-version smoke 只写成 helper 运行态里的临时 PowerShell 脚本，且脚本内部直接对 `Invoke-RestMethod` 结果做 `[int]/[bool]` 强转或在 `param(...)` 前先写赋值，容易在 live 里连续卡成 “Argument types do not match” 或 parse 失败；结果真正的产品回归还没开始，helper 自己先在脚本壳上空转。
  - 避免方式：把这类 smoke 尽快沉到对应 developer workspace 的正式测试资产里，并在脚本入口先做“取首个标量”的统一归一化，同时把脚本接进现有 `verify_powershell_script_parse.py`，不要继续依赖运行态里一次性生成的临时版本。
- 坑 27：helper 交付文件已经落盘，但 run/node 仍停在 `running`，页面和升级门禁继续把它算成活跃任务，于是看起来“17 点后一直有 running，但 token 不再增长”。
  - 避免方式：先对照 `run.json/result.json/audit/node.json` 四层真相。若 `result.json` 已存在、`run.json` 停在 `provider_pid=0 + 正在整理结果`、且 audit 没有对应终态写回，就按 finalize stall 处理，而不是继续把它当作真实执行中的 provider。

## 最小检查清单
1. 升级前确认 `/api/runtime-upgrade/status` 返回 `can_upgrade=true` 且 `running_task_count=0`。
2. 升级后确认 `/healthz` 恢复、`current_version` 已切到候选版本、`request_pending=false`。
3. 再核对 `.running/control/envs/prod.json`、`.running/control/prod-last-action.json`、`.running/control/instances/prod.json` 三处版本一致。
4. 涉及升级 banner 或训练优化布局时，补一组真实截图，不接受只看 DOM/接口。
