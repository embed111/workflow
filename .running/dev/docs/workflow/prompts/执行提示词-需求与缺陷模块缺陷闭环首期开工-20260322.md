## 2. 本轮目标
1. 在 `../workflow` 中正式落地一个与 `任务中心` 同级的 `需求与缺陷` 模块。
2. 本轮只做 `缺陷` 闭环首期；`需求` 视图只保留空态占位。
3. 当前实现目标不是静态展示，而是把“提交缺陷 -> agent 预判定 -> DTS 记录 -> 任务中心真实任务 -> 版本回写 -> 用户关闭/复评”做成真实可用闭环。

## 3. 你必须按以下口径实现
1. 左侧导航当前有效口径固定为：
   1. `对话`
   2. `工作记录与训练`
   3. `训练中心`
   4. `任务中心`
   5. `需求与缺陷`
   6. `设置`
2. `需求与缺陷` 模块内部必须通过同一内容区顶部选项卡切换 `需求 / 缺陷`。
3. `需求` 当前只保留预留空态：
   1. 不创建正式记录；
   2. 不进入状态机；
   3. 不联动任务中心。
4. `缺陷` 视图必须支持同一条上报内提交：
   1. 文字描述；
   2. 图片证据；
   3. 图片与文字必须属于同一条问题记录，而不是拆成两套入口。
5. 用户提交后必须先由 `workflow agent` 判定是否构成真实缺陷：
   1. 不是缺陷：不创建正式 DTS；
   2. 是缺陷：创建正式 DTS 并进入闭环。
6. 首轮判定“不构成缺陷”后，用户必须可在当前缺陷详情继续补充证据与说明，并将记录转为 `有分歧`。
7. 正式缺陷状态固定只有四种：
   1. `未解决`
   2. `已解决`
   3. `已关闭`
   4. `有分歧`
8. `discovered_iteration` 只记录缺陷被确认时所在的 workflow 迭代/版本；`resolved_version` 只记录修复真实发布到的 workflow 版本。
9. `分析缺陷 / 复核争议 / 修复缺陷 / 推送到目标版本` 四类动作必须全部通过 `任务中心` 创建真实任务推进。
10. `缺陷` 视图只负责：
   1. 展示当前缺陷记录；
   2. 展示判定/复评结论；
   3. 展示关联任务引用；
   4. 展示发现迭代与解决版本；
   5. 承接用户确认关闭或继续复评。
11. DTS 详情默认必须具备：
   1. 状态标签；
   2. 发现迭代；
   3. 解决版本；
   4. 原始上报；
   5. 当前判定或复评结论；
   6. 任务引用；
   7. 状态变更历史；
   8. 稳定可见的纵向滚动条。
12. 列表交互固定收口为：
   1. 单一下拉状态筛选；
   2. 单一关键词搜索；
   3. 左侧记录列表与右侧详情联动。
13. `已解决` 态默认展示固定复评输入区：
   1. 不再保留单独的 `触发复评` 前置按钮；
   2. 用户直接在该输入区补充图片和说明；
   3. 通过 `提交复评` 触发真实复核链路。
14. `首轮不成立后的补充` 与 `已解决后的复评` 共用同一种详情输入区语法，只是按钮文案和上下文语义不同。
15. 后端必须预留可供自动化测试直接提交缺陷与追加材料的接口，至少覆盖：
   1. 提交缺陷；
   2. 获取缺陷列表与详情；
   3. 更新状态；
   4. 补充图片证据；
   5. 补充文字描述；
   6. 标记有分歧；
   7. 触发处理或复核；
   8. 查询状态变更历史。

## 4. 明确禁止
1. 不要把 `需求` 视图提前做成正式需求收集模块。
2. 不要在 `缺陷` 视图内直接执行真实分析、修复或推版逻辑。
3. 不要为 `已解决` 态再保留额外的 `触发复评` 前置按钮。
4. 不要把状态筛选重新做回平铺按钮组。
5. 不要把 DTS 详情页做成无独立滚动的长整页。
6. 不要让 `需求与缺陷` 和 `任务中心` 维护两套互相冲突的执行状态。

## 5. 推荐实现顺序
1. 先补 `需求与缺陷` 模块路由、页签与基础数据结构。
2. 再补缺陷提交、agent 预判定、DTS 记录与状态机。
3. 再补 `任务中心` 真实任务引用、处理入口和版本回写。
4. 最后补 `有分歧`、复评输入区、状态变更历史与自动化测试接口。

## 6. 最低验收回传
1. 页面截图：
   1. `需求` 空态；
   2. `缺陷` 主视图；
   3. `有分歧` 视图；
   4. `有分歧补充` 视图；
   5. `复评输入` 视图；
   6. `筛选搜索` 视图。
2. 接口与行为证据：
   1. 提交缺陷请求/响应；
   2. agent 预判定结果；
   3. DTS 列表/详情读取；
   4. `处理缺陷` 后在任务中心创建真实任务的请求/响应；
   5. 解决版本写回证据；
   6. `提交复评` 后的复核任务创建证据；
   7. 自动化测试上下文透传证据。
3. 代码落点：
   1. `需求与缺陷` 页面入口；
   2. 缺陷列表与详情实现位置；
   3. DTS 状态历史与滚动容器实现位置；
   4. 缺陷接口与任务中心联动实现位置。
4. 风险与未完成项：
   1. 若首版自动化测试提缺陷接口只完成部分字段，也必须逐项列出；
   2. 若 `需求` 视图仍仅为空态，也必须在回传中明确说明当前边界。
