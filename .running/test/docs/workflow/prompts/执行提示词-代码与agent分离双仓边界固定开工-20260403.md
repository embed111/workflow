8. `scripts/launch_workflow.ps1`

## 2. 用户已固定的最新口径
1. `../workflow` 继续作为 PM/控制工作区。
2. `../workflow_code` 固定作为 `workflow` 的代码根仓。
3. 当前阶段不要求先成立新的 coder/tester 角色。
4. PM 当前可以继续兼任代码开发。
5. 平时每个开发主体都各自在自己的开发工作区中，从 `../workflow_code` 拉取代码进行开发。
6. 各开发工作区本地完成 Git 提交后，再推送到指向 `../workflow_code` 的远程分支。
7. 后续如果要新增角色，默认仍走当前 `../workflow` 在 `8090` 端口的“创建角色”能力；这条链路本轮不要改坏，也不要重做。

## 3. 本轮只做第一批，范围严格收口
1. 本轮目标不是一次做完整个“代码与 agent 分离”大改。
2. 本轮只做第一批：
   1. 固定 `../workflow` 和 `../workflow_code` 的双仓边界；
   2. 为“开发工作区从代码根仓拉取、提交、推送”补最小可用基础设施；
   3. 为后续运行态外置和集成门禁打地基。
3. 本轮先不要尝试把全部 `Codex` 运行链路、全部锁模型、全部自动集成一次做完。

## 4. 本轮必须完成的内容
1. 固定双仓语义
   1. 在后端形成稳定的路径契约：
      1. `pm_root = ../workflow`
      2. `code_root = ../workflow_code`
   2. 代码中凡是把当前 `../workflow` 误当成 `workflow` 正式代码仓的地方，必须收口。
   3. 若 `../workflow_code` 缺失、路径无效或不可访问，系统必须 fail-closed，并返回明确错误。
2. 补“代码根仓”配置与可见性
   1. 至少在后端形成统一读取入口，不允许后续继续到处手写 `../workflow_code`。
   2. 若当前工程代价可控，可在设置页或等效 API 中增加只读可见性，让用户能看到：
      1. `PM 工作区路径`
      2. `代码根仓路径`
      3. `开发工作区根路径`
   3. 当前阶段不要求做复杂 UI，只要可验证、可见、可追溯。
3. 补开发工作区 bootstrap / refresh 能力
   1. 支持给定 `developer_id` 与目标 `workspace_path` 后：
      1. 从 `../workflow_code` 初始化开发工作区；
      2. 若目录已存在，则执行 `fetch/pull` 或等效同步；
      3. 把 Git 远程指向 `../workflow_code`；
      4. 为当前开发主体切到独立开发分支或等效安全分支。
   2. 这部分可以通过：
      1. 脚本
      2. CLI
      3. 服务层 + API
      任一合理组合落地。
   3. 代码落点请按当前工程化目录治理选择，不要再塞回单个巨型文件。
4. 补开发工作区状态留痕
   1. 至少可追溯：
      1. `developer_id`
      2. `workspace_path`
      3. `source_repo_path=../workflow_code`
      4. `tracking_branch`
      5. `last_synced_commit`
   2. 留痕方式可用当前已有 `state/`、DB 或等效结构化存储，但必须稳定可回读。
5. 守住边界
   1. 不要让普通执行链路直接把 `../workflow_code` 当成 `Codex` 写入目录。
   2. 不要让 `../workflow` 自己继续承担 `workflow` 正式代码开发仓职责。
   3. `.running/dev|test|prod` 当前部署语义不要回退。

## 5. 本轮明确不做
1. 不创建新的 coder/tester 角色。
2. 不重做 `8090` 的“创建角色”产品链路。
3. 不在本轮完成整套运行态外置。
4. 不在本轮完成整套集成锁、发布锁与自动合并。
5. 不把活动开发工作区直接接成部署源。
6. 不额外引入一套新的角色创建系统或新的 PM 系统。

## 6. 关键实现建议
1. 优先做一个统一路径契约模块，别让 `../workflow_code` 到处散落硬编码。
2. 开发工作区 bootstrap 优先走 Git 原生命令，减少自造同步逻辑。
3. Git 远程当前默认仍直接指向 `../workflow_code`。
4. 如果你确认“向非 bare 本地仓直接推分支”在当前 Windows 本地形态下存在安全限制：
   1. 可以做最小兜底；
   2. 但不要擅自把架构改成另一套三方代理仓；
   3. 必须在回传中明确写清原因、限制和当前选择。
5. 当前批次优先补后端与脚本，不要先做重 UI。

## 7. 最低验收回传
1. 代码与配置证据
   1. 贴出你新增或修改的关键文件路径。
   2. 说明 `pm_root / code_root / development_workspace_root` 分别在哪里定义、怎么读取。
2. 开发工作区 bootstrap 证据
   1. 给出一次实际创建或刷新开发工作区的命令。
   2. 给出该工作区中的：
      1. `git remote -v`
      2. 当前分支
      3. 最近同步到的 commit
3. 行为边界证据
   1. 证明当前 `../workflow` 没再被当成 `workflow` 正式代码仓。
   2. 证明 `../workflow_code` 被识别为正式代码根仓。
4. 回归证据
   1. 至少保证服务仍可启动或关键 `healthz` / 设置 API 可用。
   2. 至少说明当前没有改坏 `8090` 现有“创建角色”链路。
5. 若阻塞
   1. 明确写清阻塞点。
   2. 明确写清是 Git 语义问题、路径权限问题，还是当前工程结构问题。
   3. 不要把“部分完成”写成“整套分离已完成”。

## 8. 一句话判断标准
1. 这轮真正要过的不是“已经完全支持多 agent 并行开发”，而是：
   1. `../workflow` 和 `../workflow_code` 的职责边界已经钉死；
   2. 开发工作区已经能从代码根仓拉取、提交、推送；
   3. 后续运行态外置和集成门禁已有稳定落点。
