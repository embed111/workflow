## 2. 2026-03-31 live 已核实结论
1. 当前 `角色中心 -> 创建角色` 的对话分析师已经能在聊天里理解“能力包型 agent”诉求：
   1. 能回答能力模块怎么拆；
   2. 能回答要沉淀哪些知识文件；
   3. 能继续追问缺少的关键问题。
2. 但底层仍然没有真正过验收：
   1. 结构化落点仍主要只有六要素画像；
   2. 能力模块、默认交付策略、格式边界、知识资产、首批任务建议没有独立真相源；
   3. 高密度输入仍会被错位挤进 `边界 / 协作方式 / 适用场景`。
3. 当前 create role live 里还存在明显体验问题：
   1. 新建草稿后输入区激活状态不够直观；
   2. 分析执行中长期只显示 `分析中...`；
   3. 页面会给出“草稿信息已齐，可直接开始创建”，但下方仍是 `能力对象 0 项`；
   4. `创建角色` 页面底部还混着与当前角色无关的 `训练优化` 空态工作台。

## 3. 本轮目标
1. 不再把 `创建角色` 只做成“角色画像收集器”。
2. 让它具备“能力包驱动 + 知识沉淀 + 首批任务化”的真实能力。
3. 把我在 live 实测里踩到的体验问题一起打掉，而不是只修字段抽取。

## 4. 本轮必须整改到位的点
1. 在 `创建角色` 链路中新增多层真相源：
   1. `role_profile_spec`
   2. `capability_package_spec`
   3. `knowledge_asset_plan`
   4. `seed_delivery_plan`
2. 高密度输入不得继续被压扁到六要素画像：
   1. 能力模块不能再只落成一句 `核心能力`
   2. 默认交付策略不能再混进 `协作方式`
   3. 格式边界不能再混进 `边界`
   4. 模板 / 示例 / 反例不能再消失在聊天正文里
3. `创建角色` 页面必须新增清晰的结构化落点：
   1. `角色画像`
   2. `能力包`
   3. `知识沉淀`
   4. `首批任务`
4. 当用户完成最小收口后，页面必须出现真实首批能力对象或首批任务建议：
   1. 不允许继续停留在 `能力对象 0 项`
   2. 不允许只有说明文案，没有真实对象
5. `可开始创建` 的判定必须升级：
   1. 不能只看六要素字段是否齐
   2. 必须同时看能力包和知识沉淀最小计划是否已形成
6. `创建角色` 页面不得默认混入与当前角色无关的 `训练优化` 空壳：
   1. 若复用同类工作台组件，必须替换成当前创建链路自己的对象语义
   2. 不要继续出现 `目标角色 / 能力目标 / 本轮能力` 这类当前场景不成立的空态字段
7. 分析阶段必须具备更细粒度进度：
   1. 不要只显示 `分析中...`
   2. 至少显示当前在做：
      1. 抽取角色画像
      2. 拆能力模块
      3. 整理知识资产
      4. 生成首批任务建议
8. 创建角色分析失败必须接入统一失败原因与显式重试：
   1. 显示失败阶段
   2. 显示中文失败原因
   3. 显示尝试次数
   4. 提供显式重试动作
   5. 保留当前草稿，不得失败即丢

## 5. 我实测踩到的体验差点，也必须一起修
1. `新建` 后草稿已创建，但输入框什么时候可输入并不直观，容易误判为页面卡住。
2. 分析阶段等待时间长时，页面没有告诉我“到底在抽字段、拆模块还是生成任务建议”。
3. 页面已经说“草稿信息已齐，可直接开始创建”，但下面仍显示 `能力对象 0 项`，前后信号冲突。
4. `创建角色` 会话里还混着 `训练优化` 的空态结构，阅读重心被打散。
5. 用户无法直接看到“哪些内容被识别为角色画像，哪些被识别为能力包，哪些仍待确认”。

## 6. 推荐实现顺序
1. 先补多层真相源与抽取映射，不要先堆更多说明文案。
2. 再把页面结构改成 `角色画像 / 能力包 / 知识沉淀 / 首批任务` 四块。
3. 再打通“最小收口 -> 首批能力对象 / 首批任务建议出现”。
4. 最后补分析进度、失败原因与显式重试。

## 7. 明确禁止
1. 不要只让聊天回复更像“懂能力包”，但结构层仍然只有六要素画像。
2. 不要把能力模块、知识资产继续偷塞进 `边界 / 协作方式 / 适用场景`。
3. 不要继续让 `创建角色` 页面默认混入无关的 `训练优化` 空壳。
4. 不要用“能力对象说明文案”替代真实能力对象。
5. 不要把失败重试继续做成隐式行为，用户看不见失败原因。
