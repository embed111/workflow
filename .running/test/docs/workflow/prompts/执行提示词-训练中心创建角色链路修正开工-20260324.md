## 2. 本轮目标
1. 本轮目标只有一个：把 `训练中心 -> 创建角色` 的草稿链路修正为“可独立改字段、状态语义真实、附件不过载、开始创建前不伪装执行态”的正式能力。
2. 这是一轮增量修正，不重做 `创建角色` 的六阶段主流程，不回退到旧表单方案，也不推翻已经落下的聊天壳与右侧辅助区结构。
3. 当前实现必须以最新需求和已确认原型为准，重点修“草稿对象、字段编辑、状态推进、附件注入、右侧呈现”，不要从旧的脏数据抽取逻辑和旧的“阶段 4 创建中”假状态继续演化。

## 3. 你必须按以下口径实现
1. `role_spec` 必须升级为持久化的一等草稿对象：
   1. 至少包含 `role_name / role_goal / core_capabilities / boundaries / applicable_scenarios / collaboration_style / example_assets`；
   2. 每个字段至少保留 `当前值 / 最近修改来源 / 当前确认状态`；
   3. 聊天原文是证据，不是最终字段值本身。
2. 创建角色必须支持字段级独立编辑：
   1. 用户说“改成 workflow_tester”时，只更新 `role_name`；
   2. 用户说“角色目标写得不对”时，只进入 `role_goal` 差异对齐；
   3. 不得因为改单个字段而要求用户重填整份画像。
3. 聊天抽取必须有脏数据拦截：
   1. `你好`、链路抱怨、系统能力质疑、元对话说明不得直接写入 `role_goal / boundaries / applicable_scenarios / collaboration_style`；
   2. 无法归类的文本先记为待确认线索，不要硬塞进结构化字段。
4. 用户表达“不对 / 不满意 / 帮我调整”时，系统必须进入字段差异对齐回路：
   1. 明确展示当前值；
   2. 给出建议修正值或修正方向；
   3. 给出最小待确认问题；
   4. 在用户确认前，不得把草稿标成“已收口”。
5. 必须拆开以下状态语义，不得继续混写：
   1. `消息已处理`
   2. `草稿待对齐 / 已对齐`
   3. `可开始未开始`
   4. `已真正开始创建`
6. 当最小字段满足但用户还没点击 `开始创建` 时：
   1. 页面状态必须是“可开始未开始”；
   2. `started_at` 仍为空；
   3. 不得展示真实或伪造的任务图；
   4. 不得把会话写成 `creating`。
7. 只有当用户明确点击 `开始创建` 且后台真实受理成功后，才能推进到执行态：
   1. 写入 `started_at`；
   2. 创建真实 `assignment_ticket_id` 或等效任务引用；
   3. 展示可回查的 `task_id / task_refs`；
   4. 再继续推进阶段 1 初始化与后续阶段任务。
8. 附件注入必须改为轻量引用：
   1. analyst prompt 中只保留附件元数据和可读取引用；
   2. 不得再把 `data:image/...;base64,...` 直接拼进 prompt；
   3. 单图 + 单轮字段修正场景下，优先把 prompt 体积压到需求文档要求的量级。
9. 前端右侧辅助区必须对齐这版确认原型：
   1. 顶部状态采用“待对齐 / 可开始未开始 / 当前无任务图”；
   2. `阶段与任务演进` 在开始前只呈现“阶段 2 草稿对齐中”，不伪造阶段 4 任务流；
   3. `角色画像` 页保持单列阅读，不要双列摘要，不要并排 diff；
   4. 不要再出现“字段 01 / 字段 02”这类编号标题；
   5. 标题直接使用 `角色名 / 角色目标 / 核心能力 / 边界 / 协作方式 / 附件与引用`。
10. 当前实现必须保留现有系统风格和信息架构：
   1. 继续沿用当前聊天工作台与右侧详情区壳；
   2. 不改左侧全局导航；
   3. 不重新发明新的视觉语言；
   4. 只修真实状态、字段编辑和草稿质量。

## 4. 明确禁止
1. 不要把当前问题做成“重画一版创建角色页面”。
2. 不要回退成独立表单创建页。
3. 不要把“消息已处理”继续当成“草稿已对齐”。
4. 不要在未开始前伪造 `创建中`、`阶段 4 / 6`、任务图或废案口袋。
5. 不要继续把聊天原文直接覆盖写入结构化字段。
6. 不要把附件 base64 直接拼进 analyst prompt。
7. 不要改动与本轮需求无关的训练中心其他模块语义。

## 5. 推荐实现顺序
1. 先梳理后端草稿数据结构与状态字段：
   1. `role_spec` 字段结构；
   2. 字段确认状态；
   3. 草稿对齐状态；
   4. `ready_to_start` 与 `started` 分离。
2. 再补聊天消息到字段编辑动作的解析：
   1. 改名；
   2. 改目标；
   3. 改协作方式；
   4. 改边界；
   5. 脏数据拦截。
3. 再补差异对齐回路和可见状态回写。
4. 再补附件轻量引用链路，去掉 prompt 内联 base64。
5. 最后补前端右侧辅助区与开始创建门禁联调。

## 6. 最低验收回传
1. 页面截图：
   1. 草稿待对齐视图；
   2. 单独改名后的视图；
   3. 角色目标进入差异对齐后的视图；
   4. 右侧 `角色画像` 单列视图；
   5. 用户点击 `开始创建` 后，出现真实开始态与真实任务引用的视图。
2. 接口与数据证据：
   1. `role_spec` 读取结果；
   2. 单字段编辑请求/响应；
   3. 草稿对齐状态与 `ready_to_start` 分离后的数据样例；
   4. 开始创建后的 `started_at / assignment_ticket_id / task_refs` 数据样例。
3. prompt 与附件证据：
   1. analyst prompt 文件或组装日志；
   2. 证明其中不再包含内联 base64；
   3. 附件改为元数据 + 引用的证据。
4. 代码落点：
   1. 创建角色页面入口；
   2. 草稿对象与字段状态模型；
   3. 字段编辑动作解析；
   4. 开始创建门禁与状态推进；
   5. 附件注入链路。
5. 风险与未完成项：
   1. 若首版仍有字段未支持独立编辑，逐项列出；
   2. 若附件当前只完成本地引用而未完成最终产物链路，也必须明确说明；
   3. 若 legacy 假状态仍保留在某些分支或旧组件中，也必须逐项列出。
