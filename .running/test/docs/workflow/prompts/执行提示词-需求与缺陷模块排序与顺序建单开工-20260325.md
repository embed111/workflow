## 2. 本轮目标
1. 这是一轮在既有 `需求与缺陷 -> 缺陷` 闭环基础上的增量实现，不重做整个模块。
2. 本轮只实现两件事：
   1. 缺陷列表默认按任务优先级从高到低排序；
   2. 缺陷界面新增“按顺序创建任务”总开关，并按缺陷顺序向 `任务中心` 串行建单。
3. 当前实现目标不是只补一个前端开关，而是把“排序规则、主动处理位、自动建单门禁、任务中心优先级继承”做成一套真实可用的闭环。

## 3. 你必须按以下口径实现
1. 正式缺陷记录必须补入并稳定维护以下字段：
   1. `task_priority`
   2. `reported_at`
2. 缺陷列表默认排序固定为：
   1. `task_priority` 高到低；
   2. 当前页面口径固定为 `P0 -> P1 -> P2 -> P3`；
   3. 若优先级相同，则按 `reported_at` 升序，越早上报越靠前。
3. 状态筛选与关键词搜索只能缩小结果集合，不得打乱上述默认排序。
4. `缺陷` 视图页头必须新增一个总开关：
   1. 文案口径固定为“按顺序创建任务”；
   2. 开关附近必须直接可见：
      1. 当前主动处理缺陷；
      2. 下一条待建单缺陷。
5. 当总开关关闭时：
   1. 系统不得自动为任何缺陷创建新的 `任务中心` 任务；
   2. 仍保留人工点击 `处理缺陷` / `提交复核` 的手动建单路径。
6. 当总开关开启时：
   1. 若当前没有占用主动处理位的缺陷，系统必须自动按当前排序规则，为队首可处理缺陷创建下一步 `任务中心` 任务；
   2. 同一时刻最多只允许 1 条缺陷占用主动处理位；
   3. 自动创建的任务必须继承该缺陷的 `task_priority`。
7. 主动处理位的释放规则固定为：
   1. 当前主动处理缺陷只有在进入 `已解决` 或 `已关闭` 后，系统才允许继续为下一条缺陷建单；
   2. 若当前主动处理缺陷仍处于 `未解决`、`有分歧` 或其处理链未结束，则系统不得自动跳过它去处理下一条。
8. `有分歧` 记录也属于可处理缺陷：
   1. 若它当前排在队首，且主动处理位空闲，则总开关开启后应为其创建复核类任务；
   2. 不要把 `有分歧` 从顺序建单队列里排除。
9. 缺陷详情中至少要把以下信息显式展示出来：
   1. `任务优先级`
   2. `上报时间`
   3. 当前是否处于主动处理位 / 排队中 / 手动建单模式
10. 当前实现必须与已有 `任务中心` 真相源保持一致：
   1. `缺陷` 视图自己不维护第二套执行状态；
   2. 自动建单后展示的任务引用必须能在 `任务中心` 中一一对应；
   3. 若任务中心当前已有该缺陷的活动处理链，不得重复建单。
11. 当前原型 HTML 是本轮主预览真相源：
   1. 优先按 `docs/workflow/prototypes/需求与缺陷模块-缺陷闭环首期/需求与缺陷模块参考图.html` 对齐；
   2. 旧 PNG 仅作历史参考，不要求按旧图面继续实现。

## 4. 明确禁止
1. 不要把这次实现做成“列表排序控件可切换多种模式”。
2. 不要允许多条缺陷在总开关开启后并行自动建单。
3. 不要在总开关开启后仍然跳过前一条未解决缺陷，直接处理下一条。
4. 不要让筛选、搜索或刷新导致缺陷列表顺序抖动。
5. 不要只改前端展示，不补真实字段、真实排序和真实自动建单门禁。
6. 不要让自动创建出的任务丢失缺陷优先级信息。

## 5. 推荐实现顺序
1. 先补缺陷数据结构与接口字段：
   1. `task_priority`
   2. `reported_at`
   3. 主动处理位判断所需状态
2. 再补缺陷列表默认排序与筛选/搜索后的稳定排序。
3. 再补页头总开关、当前主动处理/下一条待建单摘要。
4. 再补自动顺序建单门禁：
   1. 空闲时为队首建单；
   2. 已占位时禁止后续缺陷抢占；
   3. 上一条进入 `已解决/已关闭` 后自动推进下一条。
5. 最后联调 `缺陷 -> 任务中心` 的任务引用与优先级继承。

## 6. 最低验收回传
1. 页面截图：
   1. 总开关关闭态；
   2. 总开关开启态；
   3. 当前存在主动处理缺陷时的列表与详情；
   4. 上一条进入 `已解决` 后下一条进入待建单/已建单的视图。
2. 接口与行为证据：
   1. 缺陷列表返回 `task_priority / reported_at` 的证据；
   2. 缺陷列表默认排序正确的证据；
   3. 总开关关闭时不自动建单的证据；
   4. 总开关开启且主动处理位空闲时自动建单的请求/响应；
   5. 上一条未解决时不会为下一条自动建单的证据；
   6. 自动创建任务后，任务引用与 `任务中心` 一致的证据；
   7. 自动创建任务继承缺陷优先级的证据。
3. 代码落点：
   1. 缺陷数据模型与接口位置；
   2. 缺陷列表排序实现位置；
   3. 总开关与队列摘要实现位置；
   4. 顺序建单门禁实现位置；
   5. 与 `任务中心` 的建单桥接实现位置。
4. 风险与未完成项：
   1. 若历史缺陷数据缺少 `task_priority / reported_at`，必须说明回填策略；
   2. 若首版仍存在“同一缺陷重复建单”的风险，也必须明确列出当前防重边界。
