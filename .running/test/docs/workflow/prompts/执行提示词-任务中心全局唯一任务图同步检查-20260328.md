## 2. 本轮检查的核心口径
1. `任务中心` 页面当前只允许展示 1 张全局唯一任务图。
2. 若后端仍保留 `ticket_id` 概念，该标识必须稳定指向这张全局唯一任务图，而不是每次建单新生成一张图。
3. 其他模块与外部 workflow 的接入语义统一改为：
   1. 向全局唯一任务图追加任务节点；
   2. 向全局唯一任务图追加依赖关系；
   3. 读取同一张图的状态、详情、执行链路与审计。
4. `缺陷`、`定时任务`、手工创建任务都不得再各自产生新的独立活动任务图。

## 3. 你这次只做检查，不直接开改
1. 本轮目标是“同步当前需求口径并做一次差异检查”。
2. 先不要直接进入大规模实现修改。
3. 你需要输出的是：
   1. 当前实现与新口径是否一致；
   2. 哪些位置已满足；
   3. 哪些位置仍是旧口径；
   4. 若要改，最小改动面在哪里。

## 4. 你必须重点检查的对象
1. `任务中心` 前端：
   1. 是否仍存在多图切换器、任务图列表、或创建第二张图的入口；
   2. 页面是否默认只围绕同一张图展示；
   3. `清空任务` 是否作用于这张全局唯一任务图。
2. `任务中心` 后端：
   1. `POST /api/assignments` 当前到底是“创建新图”还是“确保全局图存在”；
   2. `GET /api/assignments` 是否会返回多张活动任务图；
   3. `ticket_id` 是否稳定；
   4. `POST /api/assignments/{ticket_id}/nodes` 是否只向同一张图追加节点。
3. `缺陷` 联动：
   1. `process-task` / `review-task` 是否只向同一张全局图挂链；
   2. 是否仍可能生成新的独立活动任务图；
   3. 缺陷详情任务引用与全局图节点链是否一一对应。
4. `定时任务` 联动：
   1. 当前实现是“创建新的单任务实例图”，还是“向全局图追加单任务节点”；
   2. 命中后是否会额外生成新的图级对象；
   3. 计划详情中的任务引用是否指向全局图中的节点。
5. 历史数据与恢复逻辑：
   1. 现有数据目录或数据库中是否仍保留多张并存活动图；
   2. 服务重启或恢复时是否会重新扫出多张活动图；
   3. 如果存在历史多图，当前实现是否已有收口策略。

## 5. 你必须按以下格式回传检查结果
1. 先给结论：
   1. `已符合`
   2. `部分符合`
   3. `不符合`
2. 再按模块分组列差异：
   1. `任务中心前端`
   2. `任务中心后端`
   3. `缺陷联动`
   4. `定时任务联动`
   5. `历史数据/恢复逻辑`
3. 每个差异点至少包含：
   1. 现状
   2. 目标口径
   3. 风险
   4. 最小修正建议
4. 若当前已经符合，也要给出证据，而不是只写“已支持”。

## 6. 最低检查证据
1. 代码证据：
   1. 任务中心主页面渲染入口；
   2. `/api/assignments` 相关接口实现；
   3. 缺陷 `process-task / review-task` 挂链实现；
   4. 定时任务命中后向任务中心写入的实现位置。
2. 运行态证据：
   1. `/api/assignments` 返回；
   2. 至少 1 个 `ticket_id` 的 `/graph` 返回；
   3. 若当前仍有多张活动图，列出它们的标识和来源。
3. 页面证据：
   1. 任务中心当前页面截图或等效说明；
   2. 若存在多图入口，明确指出位置；
   3. 若不存在，说明当前页面如何体现“全局唯一任务图”。

## 7. 明确禁止
1. 不要把“缺陷已经统一挂全局主图”误判成“任务中心整体已经全局唯一”。
2. 不要只看前端页面当前只展示 1 张图，就忽略后端是否仍实际生成多张活动图。
3. 不要只凭口头结论，不给代码路径、接口返回或运行态证据。
4. 不要在未说明风险的情况下直接改历史数据。

## 8. 本轮交付边界
1. 本轮默认目标是“同步需求并完成一次差异检查回传”。
2. 若检查后确认需要实现修正，再基于检查结果单独进入开工阶段，不要把“检查”和“实现改造”混成同一轮未说明动作。
