# 需求详情-需求与缺陷模块（缺陷闭环首期）

- 状态：正式需求（当前有效口径）
- 当前边界：本轮收口 `需求与缺陷` 顶层模块，其中 `需求` 视图先预留为空态；正式交付范围仍只覆盖 `缺陷` 视图的闭环能力。

## 1. 主题目标与价值说明
1. 主题目标：
   1. 在 `../workflow` 左侧导航中新增一个与 `任务中心` 同级的 `需求与缺陷` 模块，其中 `缺陷` 视图用于上报和闭环 `workflow` 自身的缺陷。
   2. `需求 / 缺陷` 必须通过同一内容区顶部选项卡切换显示，而不是分散为多个入口。
   3. 支持用户通过“图片证据 + 缺陷描述”提交问题，由 `workflow agent` 先判断这是否属于真实缺陷。
   4. 仅当 agent 判定其为真实缺陷，或用户对“不成立”结论提出异议并进入 `有分歧` 时，系统才生成可追溯的 DTS 展示记录并进入后续闭环。
   5. `分析缺陷 / 复核争议 / 修复缺陷 / 推送到目标版本` 四类动作必须全部走 `任务中心` 的真实任务，而不是在 `缺陷` 视图内直接隐式执行。
   6. 对每个缺陷记录“发现于哪个 workflow 迭代”和“修复落在哪个 workflow 版本”，形成完整版本追溯链。
2. 业务价值：
   1. 把“问题上报 -> 真伪判定 -> 任务执行 -> 版本回写 -> 用户确认”收口为单一链路，避免缺陷处理继续散落在聊天与人工口头判断中。
   2. 避免 `缺陷` 视图、`任务中心` 和版本发布链分别维护三套不一致的状态。
   3. 让 workflow 的缺陷治理具备可留痕、可复盘、可版本回查的闭环，而不是只在修复后口头说明“已经好了”。
   4. 为未来补齐需求收集能力预留同模块入口，但当前仍不把缺陷与需求混成同一个状态机。

## 2. 用户画像与使用场景
1. 用户画像 A：问题上报者
   1. 关注点：能快速提交截图和描述，不需要先自己判断这到底是缺陷、需求还是操作问题。
2. 用户画像 B：workflow 处理者
   1. 关注点：系统能先帮忙判定“是不是真缺陷”，并把后续处理自动接到任务中心，而不是自己手工拆任务。
3. 用户画像 C：版本治理者
   1. 关注点：每条缺陷都要能追溯到发现迭代、修复版本、关联任务和最终关闭结论。
4. 核心场景：
   1. 用户进入 `需求与缺陷` 模块，并在 `缺陷` 视图提交图片和缺陷描述。
   2. `workflow agent` 判断该问题是否属于真实的 `workflow` 缺陷。
   3. 若首轮分析认为不是缺陷：
      1. 用户可直接接受结论并返回补充描述；
      2. 用户也可选择标记为 `有分歧`，继续补充证据和描述。
   4. 若判定为缺陷，系统生成 DTS 记录，并记录其发现迭代。
   5. 用户点击 `处理缺陷` 后，系统在 `任务中心` 中创建分析、修复、推送版本等真实任务。
   6. 修复发布到某个 workflow 版本后，缺陷记录写入解决版本并变为 `已解决`。
   7. 用户确认后，缺陷变为 `已关闭`；若用户仍认为未解决，则触发一次 `workflow agent` 复评。
   8. 若复评结论为“当前问题已修复/不再构成缺陷”，则该记录转为 `有分歧`。

## 3. 用户旅程或关键流程
1. 用户进入 `需求与缺陷` 模块。
2. 用户提交一条问题上报，输入内容至少包含：
   1. 缺陷描述；
   2. 一张或多张图片证据。
3. `workflow agent` 基于当前描述与图片证据分析该问题是否属于真实的 `workflow` 缺陷。
4. 若判定为“不是缺陷”：
   1. 系统不给当前问题创建正式 DTS 记录；
   2. 系统明确说明当前判断；
   3. 用户可直接在当前模块中重新输入或补充描述；
   4. 若用户不同意该结论，系统允许将当前记录转为 `有分歧`，并继续补充证据与描述。
5. 若判定为“是缺陷”：
   1. 系统创建正式 DTS 记录；
   2. 记录该缺陷 `发现于哪个 workflow 迭代`；
   3. 初始状态设为 `未解决`。
6. 用户点击 `处理缺陷` 后：
   1. 系统在 `任务中心` 创建真实任务，而不是在 `缺陷` 视图里直接开始执行；
   2. 至少覆盖以下三类动作：
      1. 分析缺陷；
      2. 修复缺陷；
      3. 推送到目标版本。
7. 当修复内容已发布到某个 workflow 版本：
   1. 系统将该版本写入 `解决版本`；
   2. 缺陷状态切为 `已解决`。
8. 用户查看修复结果后：
   1. 若确认问题已解决，则状态切为 `已关闭`；
   2. 若用户认为问题仍未解决，则系统触发一次 `workflow agent` 复评。
9. 争议复核后：
   1. 若复核结论仍认为“不构成缺陷”或“原问题已不再构成缺陷”，状态保持 `有分歧`；
   2. 若复核结论认为“当前问题确实构成缺陷”，则转入正式缺陷处理链，并允许继续在 `任务中心` 创建下一轮分析/修复任务。

## 4. 功能需求清单
1. FR-BM-01 顶层模块新增
   1. `workflow` 左侧导航当前有效口径至少包含以下 6 个顶层模块：
      1. `对话`
      2. `工作记录与训练`
      3. `训练中心`
      4. `任务中心`
      5. `需求与缺陷`
      6. `设置`
   2. `需求与缺陷` 模块内至少包含两个内部视图：`需求`、`缺陷`。
   3. 两个内部视图必须通过同一内容区顶部选项卡切换显示。
   4. `需求` 视图当前只保留空态占位，不创建正式记录、不进入状态机，也不创建任务中心任务。
   5. 本轮正式交付范围只覆盖 `缺陷` 视图，不展开 `需求` 视图的收集交互与状态机。
2. FR-BM-02 缺陷提交通道
   1. `需求与缺陷` 模块内的 `缺陷` 视图必须支持用户提交：
      1. 缺陷描述；
      2. 图片证据。
   2. 图片与文字必须属于同一条问题上报语义，而不是分裂成互不关联的两套输入。
   3. 用户在被判定“不是缺陷”后，必须可继续重新输入或补充描述，而不是被迫重新开一个完全无关的新入口。
3. FR-BM-03 缺陷判定边界
   1. 当前模块中的 `缺陷` 视图只处理 `workflow` 自身的缺陷。
   2. 与角色能力设计、角色训练目标偏差、需求新增建议有关的问题，不纳入当前缺陷状态机。
   3. 当前轮不实现 `需求` 视图的正式承接逻辑；若问题不属于缺陷，仅要求允许用户重新描述，不强制自动转需求池。
4. FR-BM-04 workflow agent 预判定
   1. 用户提交问题后，必须由 `workflow agent` 先判断是否属于真实缺陷。
   2. 系统不得在未经过 agent 判定前，直接把所有上报都落成正式缺陷记录。
   3. 判定结论至少区分：
      1. 是缺陷；
      2. 不是缺陷。
5. FR-BM-05 首轮不成立与异议路径
   1. 若 agent 判定“不是缺陷”，系统默认不得直接进入正式缺陷处理链。
   2. 系统必须明确告知用户当前判定，并允许用户继续补充或重写描述后再次提交。
   3. 若用户对首轮“不成立”结论提出异议，系统必须允许将当前记录转为 `有分歧`。
   4. 进入 `有分歧` 后，用户必须可继续补充图片证据与文字描述。
6. FR-BM-06 DTS 记录创建
   1. 若 agent 判定“是缺陷”，或用户对“不成立”结论提出异议并进入 `有分歧`，系统必须创建可追溯的 DTS 展示记录。
   2. DTS 记录至少包含：
      1. `dts_id`
      2. `defect_summary`
      3. `report_text_ref`
      4. `evidence_image_refs`
      5. `discovered_iteration`
      6. `status`
      7. `resolved_version`（可空）
      8. `linked_task_refs`
      9. `review_history`
      10. `status_history`
      11. `report_source`
      12. `automation_context`（可空）
   3. 当前缺陷的 `discovered_iteration` 固定记录“该问题首次进入可追溯记录时所在的 workflow 迭代/版本”。
7. FR-BM-07 缺陷状态机
   1. 正式缺陷记录固定只有以下 4 种状态：
      1. `未解决`
      2. `已解决`
      3. `已关闭`
      4. `有分歧`
   2. 新创建并被确认的缺陷记录，初始状态必须为 `未解决`。
   3. `已解决` 表示该缺陷的修复内容已经发布到某个 workflow 版本，且解决版本已写回记录。
   4. `已关闭` 表示用户已明确确认该问题处理完成。
   5. `有分歧` 表示用户与 agent 当前结论不一致；既可出现在“修复后用户仍不认可”的场景，也可出现在“首轮分析认为不成立，但用户坚持继续追诉”的场景。
8. FR-BM-08 处理缺陷必须走任务中心
   1. `处理缺陷` 动作不得在 `缺陷` 视图内直接执行真实处理逻辑。
   2. 点击 `处理缺陷` 后，系统必须在 `任务中心` 创建真实任务图或等效真实任务集合。
   3. 至少以下四类动作必须通过 `任务中心` 任务推进：
      1. 分析缺陷；
      2. 复核有分歧记录；
      3. 修复缺陷；
      4. 推送到目标版本。
   4. `缺陷` 视图只负责：
      1. 展示缺陷记录；
      2. 展示判定结果；
      3. 展示关联任务引用；
      4. 展示发现迭代与解决版本；
      5. 承接用户确认与复评触发。
9. FR-BM-09 缺陷视图与任务中心的真相源关系
   1. 一旦进入处理阶段，缺陷的执行状态、执行证据和处理进度真相源必须来自 `任务中心`。
   2. `缺陷` 视图不得私自维护第二套独立执行状态。
   3. `缺陷` 视图必须能展示关联的 `task_id` 或等效任务引用标识。
   4. 用户必须可从缺陷记录跳转或联动查看 `任务中心` 中的对应任务。
10. FR-BM-10 发现迭代与解决版本
   1. `discovered_iteration` 固定记录缺陷被确认时的 `workflow` 迭代/版本。
   2. `resolved_version` 固定记录“修复内容实际发布到了哪个 workflow 版本”。
   3. 未完成版本推送前，不得提前写入 `resolved_version`。
   4. 任务完成但尚未进入目标版本发布时，缺陷不得提前标记为 `已解决`。
11. FR-BM-11 用户确认关闭
   1. 缺陷进入 `已解决` 后，系统必须允许用户明确确认是否关闭。
   2. 仅当用户明确确认后，状态才允许切为 `已关闭`。
   3. 未收到用户确认前，`已解决` 不得自动转为 `已关闭`。
12. FR-BM-12 复评与有分歧
   1. 当缺陷已处于 `已解决`，但用户认为仍未解决时，系统必须触发一次 `workflow agent` 复评。
   2. 当首轮分析认为“当前问题不构成缺陷”，但用户明确提出异议时，系统也必须允许该记录进入 `有分歧`。
   3. `有分歧` 状态必须保留结论冲突、复评结论、补充证据与补充描述，不得只保留最终状态字样。
   4. `首轮不成立后的补充` 与 `已解决后的复评` 必须共用当前缺陷详情页中的同一输入区，不另开第二套独立页面或独立入口。
   5. 用户在当前详情页补充文字说明、图片证据后，系统才可继续调用后端复核接口，并在 `任务中心` 创建对应复核任务。
13. FR-BM-13 缺陷视图的默认展示信息
   1. `缺陷` 视图默认至少展示：
      1. 缺陷摘要；
      2. 当前状态；
      3. 发现迭代；
      4. 解决版本（若有）；
      5. 关联任务引用；
      6. 最近一次判定/复评结论。
   2. 对当前使用无帮助的长说明和冗余解释，默认不占据主视图。
   3. 默认主视图应优先收口为四类对象：
      1. 提交入口；
      2. 缺陷列表；
      3. 当前记录；
      4. 任务引用与过程记录。
   4. `需求` 视图在未开放前应只保留单一空态，不附带多余说明卡片或伪交互。
   5. 缺陷列表的状态筛选应优先使用单一下拉控件，而不是平铺多个筛选按钮。
   6. 缺陷列表应提供关键词搜索，用于按 `dts_id`、摘要或相关文本快速定位记录。
   7. DTS 展示页在内容超出可视区时必须提供稳定可见的纵向滚动条。
   8. DTS 展示页默认应包含“状态变更历史”区块，用于回看结论变化、补充材料和状态切换过程。
   9. 当缺陷处于 `已解决` 且用户尚未确认关闭时，当前缺陷详情页内必须默认展示复评输入区，至少承载补充图片证据、补充文字说明与提交复评动作。
   10. `已解决` 态不再单独提供一个仅用于前置跳转或展开的 `触发复评` 按钮，复评应由该输入区内的 `提交复评` 动作直接触发。
14. FR-BM-14 后端缺陷操作接口
   1. 后端必须提供面向 UI 与自动化测试统一复用的缺陷操作接口。
   2. 接口至少覆盖：
      1. 提交缺陷；
      2. 获取缺陷列表与详情；
      3. 更新状态；
      4. 补充图片证据；
      5. 补充文字描述；
      6. 标记有分歧；
      7. 触发处理或复核；
      8. 查询状态变更历史。
   3. 接口必须允许携带自动化测试上下文，如 `suite_id`、`case_id`、`run_id`、环境、版本号等信息，用于后续自动化测试直接提交缺陷。
15. FR-BM-15 本轮边界
   1. 本轮只正式处理 `缺陷` 视图中的缺陷上报、缺陷判定、缺陷状态机、与任务中心/版本链路的关系。
   2. `需求` 视图当前只保留空态占位，后续再独立收口其交互和状态机。

## 5. 非功能需求
1. 可追溯性：
   1. 每条正式缺陷记录都必须可追溯到原始描述、图片证据、发现迭代、关联任务和解决版本。
   2. `有分歧` 状态必须可追溯到用户异议与复评结论。
2. 一致性：
   1. `缺陷` 视图中的处理进度必须与 `任务中心` 的真实任务状态一致。
   2. `已解决` 与 `resolved_version` 的写入时机必须与版本发布事实一致，不允许“任务完成了但版本还没发”时提前显示已解决。
   3. `status_history`、当前状态、补充材料和任务引用必须能相互对齐，不允许出现历史已变更但当前展示未同步的情况。
3. 可理解性：
   1. 用户进入缺陷记录详情后，应在 3 秒内理解：
      1. 这是不是一个正式缺陷；
      2. 当前状态是什么；
      3. 发现于哪个迭代；
      4. 是否已有解决版本；
      5. 当前关联哪些任务。
4. 治理边界：
   1. 当前轮 `需求` 视图不承接正式需求收集。
   2. 当前轮 `缺陷` 视图不直接执行分析、修复或推版动作，只负责创建和引用任务中心任务。

## 6. 验收标准
1. AC-BM-01 顶层模块可见
   1. Given 用户打开 `workflow`
   2. When 左侧导航加载完成
   3. Then 当前有效口径下可看到 `对话 / 工作记录与训练 / 训练中心 / 任务中心 / 需求与缺陷 / 设置`
   4. And 用户进入该模块后可切换 `需求 / 缺陷` 两个内部视图，其中 `需求` 当前显示预留空态
2. AC-BM-02 图片与描述可同条提交
   1. Given 用户进入 `需求与缺陷` 模块的 `缺陷` 视图
   2. When 用户提交缺陷描述与图片证据
   3. Then 系统将其作为同一条问题上报处理，而不是拆成互不关联的两条记录
3. AC-BM-03 非缺陷不生成正式记录
   1. Given 用户提交了一条问题上报
   2. When `workflow agent` 判定其“不属于 workflow 缺陷”
   3. Then 系统不生成正式 DTS 记录
   4. And 页面允许用户继续补充或重新输入描述
4. AC-BM-04 首轮不成立但进入有分歧
   1. Given 用户提交的问题被首轮分析判定为“不构成 workflow 缺陷”
   2. When 用户选择标记为 `有分歧` 并补充证据或描述
   3. Then 系统为当前记录创建可追溯的 DTS 展示记录
   4. And 其当前 `status=有分歧`
   5. And 可继续提交补充图片与补充描述
5. AC-BM-05 真实缺陷生成 DTS
   1. Given 用户提交的问题被判定为真实 workflow 缺陷
   2. When 系统创建正式缺陷记录
   3. Then 该记录具有 `dts_id`
   4. And 其 `status=未解决`
   5. And 已写入 `discovered_iteration`
6. AC-BM-06 处理缺陷走任务中心
   1. Given 某条缺陷记录已被确认
   2. When 用户点击 `处理缺陷`
   3. Then 系统在 `任务中心` 创建真实任务，而不是在 `缺陷` 视图内直接执行
   4. And 至少可追溯到分析、修复、推送版本三类任务或等效真实任务
7. AC-BM-07 解决版本写回
   1. Given 某缺陷的修复内容已发布到某个 workflow 版本
   2. When 系统完成版本回写
   3. Then 该缺陷的 `resolved_version` 被写入
   4. And 其状态切为 `已解决`
8. AC-BM-08 用户确认关闭
   1. Given 某缺陷当前状态为 `已解决`
   2. When 用户明确确认问题已处理完成
   3. Then 系统将该缺陷状态切为 `已关闭`
9. AC-BM-09 异议复评与有分歧
   1. Given 某缺陷当前状态为 `已解决`
   2. When 用户明确表示“仍未解决”
   3. Then 当前缺陷详情页默认展示复评输入区，允许继续补充图片证据与文字说明
   4. And 用户通过 `提交复评` 动作触发一次 `workflow agent` 复评
   5. And 若复评结论为“当前问题已修复/不再构成缺陷”，则状态切为 `有分歧`
10. AC-BM-10 任务引用一致
   1. Given 某缺陷已进入处理阶段
   2. When 用户在 `缺陷` 视图查看该记录
   3. Then 可看到关联的真实任务引用
   4. And 这些引用可与 `任务中心` 中的对应任务一一对应
11. AC-BM-11 证据完备性
   1. Given 任一缺陷记录进入验收或复核
   2. When 提交验收材料
   3. Then 至少包含：
      1. 缺陷上报截图；
      2. agent 判定结果；
      3. 关联任务引用；
      4. 解决版本或分歧复评证据；
      5. 对应代码路径与日志/记录路径。
12. AC-BM-12 列表筛选与搜索
   1. Given 用户进入 `缺陷` 视图
   2. When 用户通过状态下拉选择某一状态，或输入关键词搜索
   3. Then 左侧缺陷列表按当前筛选条件收窄
   4. And 若当前选中记录不再命中筛选条件，详情区自动切换到当前筛选结果中的首条记录
13. AC-BM-13 DTS 页面滚动与历史
   1. Given 当前 DTS 展示内容超过可视区高度
   2. When 用户浏览详情页
   3. Then DTS 展示页提供稳定可见的纵向滚动条
   4. And 页面内可查看状态变更历史
14. AC-BM-14 缺陷操作接口
   1. Given 自动化测试或外部调用方需要直接提交缺陷
   2. When 调用后端缺陷操作接口提交记录、补充材料或查询状态
   3. Then 系统可在不依赖手工 UI 操作的前提下完成对应动作
   4. And 请求中可带 `suite_id / case_id / run_id / env / version` 等自动化上下文

## 7. 边界条件与异常处理
1. 若用户只提交了图片但没有可理解描述：
   1. 推断/假设（置信度: 中）：系统应提示补充描述，再进入正式判定。
2. 若 agent 判定“不是缺陷”：
   1. 不创建正式 DTS 记录；
   2. 不创建处理任务。
3. 若缺陷已创建但用户尚未点击 `处理缺陷`：
   1. 状态保持 `未解决`；
   2. 不得提前创建真实处理任务。
4. 若修复任务完成但目标版本尚未发布：
   1. 不得提前写入 `resolved_version`；
   2. 状态不得切为 `已解决`。
5. 若用户在 `已解决` 后提出异议：
   1. 系统必须先在当前详情页承接复评输入，并由 `提交复评` 动作直接触发复评，再根据结果决定是否进入 `有分歧` 或重新回到处理链。
6. 若复评结论与用户结论冲突：
   1. 系统不得静默覆盖用户意见；
   2. 必须保留“有分歧”状态与复评证据。
