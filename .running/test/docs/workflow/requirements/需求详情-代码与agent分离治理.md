# 需求详情-代码与agent分离治理

- 状态：正式需求（当前有效口径）
- 现状依据：基于 2026-04-02 对 `../workflow/state/runtime-config.json`、`../workflow/src/workflow_app/runtime/task_agent_runner.py`、`../workflow/.running/` 与任务产物外置链路的盘点

## 1. 主题目标与价值说明
1. 主题目标：
   1. 将 `workflow` 从“代码主仓同时承担 agent 搜索根、执行 cwd 与写入范围”的形态，升级为 `PM/控制层 / 代码根仓层 / 开发工作区层 / agent 运行态层` 四层分离方案。
   2. 让同一 agent 可并行分析与执行，但不再并发改写同一份源码目录。
   3. 让 `Codex` 提示词、stdout/stderr、结果、`.codex` 记忆、锁与租约从业务源码主仓中拆出，避免把代码仓继续当作运行态承载层。
   4. 固定 `../workflow` 继续作为 PM/控制平面/任务编排入口与发布治理工作区。
   5. 固定 `../workflow_code` 作为 `workflow` 的代码根仓（canonical code repo）。
2. 业务价值：
   1. 把“可以并行跑”和“可以安全集成”拆成两套边界，降低并发覆盖与误发布风险。
   2. 让 `训练优化`、`创建角色`、`任务中心真实执行`、`发布评审` 共用同一套隔离与集成门禁，而不是各自维护私有写回规则。
   3. 让 agent 运行失败、会话中断、租约泄漏后的恢复路径更清晰，不必手工清理主代码仓中的临时文件与半成品改动。
   4. 为后续多 agent 并行开发、差异预览、自动发布评审与正式发布治理提供稳定底座。

## 2. 用户画像与使用场景
1. 用户画像 A：workflow 平台治理者
   1. 关注点：多 agent 并行时不能互相覆盖源码，集成和发布必须可控、可追溯、可恢复。
2. 用户画像 B：workflow 操作者 / PM
   1. 关注点：仍然通过现有 `角色中心 / 任务中心 / 发布评审` 使用系统；当前阶段即便 PM 兼做代码开发，也不希望再把 `../workflow` 本身当作正式代码开发仓。
3. 用户画像 C：agent 维护者
   1. 关注点：希望每轮运行都能看到自己到底改的是“正式代码仓”“临时工作副本”还是“运行态目录”，避免路径语义混乱。
4. 核心场景：
   1. `../workflow` 负责管理项目、编排任务、验收与发布，不作为日常 `workflow` 代码开发目录。
   2. `../workflow_code` 作为唯一正式代码根仓，承载 `workflow` 产品源码与正式 Git 历史。
   3. 每个开发者或开发 agent 都在各自独立开发工作区中，从 `../workflow_code` 拉取代码进行开发。
   4. 各开发工作区在本地完成提交后，向指向 `../workflow_code` 的 Git 远程分支提交。
   5. 任务中心真实执行启动 `Codex` 时，只写当前开发工作区或受控工作副本，并把运行痕迹落到运行态根目录。
   6. 优化任务完成后，系统先出差异与证据，再由统一集成链路把结果收口到 `../workflow_code`。
   4. 发布和部署只能消费已经集成通过的版本，不直接消费活动工作副本。

## 3. 用户旅程或关键流程
1. `../workflow` 作为 PM/控制工作区，负责识别目标项目、分发开发任务、追踪进度与触发验收/发布。
2. `../workflow_code` 作为 `workflow` 代码根仓，保存正式代码、正式分支与正式集成结果。
3. 当 PM 或某个开发 agent 需要改动 `workflow` 代码时，系统先在对应开发工作区中执行 `clone/fetch/pull`，确保该工作区以 `../workflow_code` 为代码来源。
4. 每个开发工作区至少记录：
   1. `developer_id`
   2. `workspace_path`
   3. `source_repo_path=../workflow_code`
   4. `tracking_branch`
   5. `last_synced_commit`
5. 系统同时在 `agent_runtime_root` 下为该执行单元创建运行态包，承载：
   1. 提示词；
   2. stdout/stderr；
   3. 结果文件；
   4. `.codex` 当轮记忆与运行附属文件；
   5. 锁与租约；
   6. 执行审计证据。
6. 任务执行器启动 `Codex` 时：
   1. `cwd` 指向当前开发工作区或受控工作副本；
   2. 写入范围指向当前开发工作区或受控工作副本；
   3. 运行痕迹落到当前运行态包。
7. 运行完成后，系统先生成当前开发工作区相对于 `../workflow_code` 的差异、证据与门禁结论。
8. 若门禁未通过：
   1. 当前会话停留在阻塞或待处理状态；
   2. `../workflow_code` 保持不变。
9. 若门禁通过：
   1. 开发者或开发 agent 先把本地提交写入当前开发分支；
   2. 再向指向 `../workflow_code` 的 Git 远程分支推送；
   3. 系统创建 `integration_job`；
   4. 由统一集成链路把当前开发分支差异收口到 `../workflow_code` 的正式集成分支或候选分支；
   3. 集成完成后再进入发布评审与部署链路。
10. 部署到 `.running/dev|test|prod` 时，只允许消费已经集成通过的 `../workflow_code` 代码版本，不允许直接从活动开发工作区部署。
11. 会话结束、服务重启或异常中断后，系统可基于租约与运行态记录恢复现场、标记陈旧开发工作区或临时工作副本并执行清理或归档。

## 4. 功能需求清单（编号）
1. FR-CA-01 四层真相源分离
   1. `PM/控制层` 固定为 `../workflow`，只承载项目管理、任务编排、验收、发布治理与角色运营功能。
   2. `代码根仓层` 固定为 `../workflow_code`，只承载可发布源码、`AGENTS.md`、长期知识/技能资产与正式版本基线。
   3. `开发工作区层` 承载各开发者或开发 agent 各自拉取下来的开发副本，不作为正式发布真相源。
   4. `agent 运行态层` 只承载提示词、stdout/stderr、结果、`.codex` 记忆、锁、租约与审计证据。
2. FR-CA-02 代码主仓禁止被普通运行链路直接写入
   1. `任务中心真实执行`、`训练优化`、`创建角色`、`发布评审生成报告` 等普通 agent 执行链路，不得直接把 `Codex` 的 `cwd` 或写入范围指向 `../workflow_code`。
   2. `../workflow_code` 的任何变更都必须经统一 `集成门禁` 收口。
3. FR-CA-03 代码根仓路径固定
   1. 当前阶段 `workflow` 项目的正式代码根仓固定命名为 `../workflow_code`。
   2. `../workflow` 不再承担 `workflow` 产品的正式代码根仓职责。
   3. 后续如需调整路径，必须视为架构级变更，单独评审。
4. FR-CA-04 发现根与写入根解耦
   1. 当前对用户暴露的 `agent_search_root` 首期继续作为 agent 发现根与 PM 工作区定位根。
   2. 系统内部必须新增或派生：
      1. `agent_workspace_root`
      2. `agent_runtime_root`
   3. 首期允许默认从现有 `artifact_root` 派生上述根路径，以减少设置项膨胀。
5. FR-CA-05 开发工作区分配
   1. 当前阶段允许 PM 兼做开发者，不要求先创建新的专职 coder/tester 角色。
   2. 每个开发者或开发 agent 必须拥有各自独立的开发工作区。
   3. 各开发工作区必须从 `../workflow_code` 拉取代码，不允许直接把 `../workflow` 当作开发工作区。
   4. 每个开发工作区至少记录：
      1. `developer_id`
      2. `workspace_path`
      3. `source_repo_path=../workflow_code`
      4. `tracking_branch`
      5. `last_synced_commit`
      6. `created_at`
      7. `last_used_at`
6. FR-CA-06 Git 提交与远程约束
   1. 各开发工作区必须在本地完成 Git 提交。
   2. 各开发工作区的 Git 远程必须指向 `../workflow_code` 所代表的代码根仓。
   3. 各开发工作区默认向各自独立开发分支推送，不得直接把未审查代码写入正式集成分支。
   4. 分支命名规则、合并规则与保护策略留待实现阶段细化，但“每个开发独立分支 -> 推送到代码根仓”是当前固定口径。
7. FR-CA-07 Codex 执行写入边界
   1. `task_agent_runner.py` 同类执行器的 `-C`、`--add-dir` 与声明写入目标归一化，都必须以当前开发工作区或受控工作副本为准。
   2. 相对路径、默认写入路径与自动发现写入目标，都不得越过当前开发工作区或受控工作副本边界。
   3. 若用户请求写当前 agent 以外的 `../workflow_code`、其他开发工作区或其他受控工作副本，系统必须 fail-closed，并返回明确阻塞原因。
8. FR-CA-08 运行态外置
   1. 提示词、stdout/stderr、result、事件流、执行证据、`.codex` 当轮记忆与租约文件，不得再以主代码仓为长期落点。
   2. 若执行链路需要兼容现有 `.codex` 目录契约，必须由运行态层提供等价结构，而不是继续让 `../workflow_code` 滚动增长运行态记忆。
9. FR-CA-09 会话冻结与追溯字段升级
   1. 会话冻结信息至少新增：
      1. `source_repo_path`
      2. `source_commit`
      3. `workspace_lease_id`
      4. `workspace_path`
      5. `runtime_bundle_path`
   2. `AGENTS.md` 快照仍按会话冻结，但必须同时可追溯到源仓与工作副本。
10. FR-CA-10 锁与租约模型
   1. 系统至少区分以下三类控制对象：
      1. `workspace_lease`
      2. `agent_integration_lock`
      3. `publish_or_deploy_lock`
   2. 不同工作副本上的分析与执行允许并行。
   3. 同一 `agent_id` 的集成与发布必须串行。
   4. 对冲突会话，系统必须外显：
      1. 当前持锁者；
      2. 锁开始时间；
      3. 阻塞原因；
      4. 下一步恢复动作。
11. FR-CA-11 集成门禁与差异预览
   1. 开发工作区结果进入 `../workflow_code` 正式集成分支之前，必须先生成：
      1. 差异摘要；
      2. 相关运行证据；
      3. 门禁结论；
      4. 对应 `run_id / workspace_lease_id` 追溯关系。
   2. 集成门禁至少校验：
      1. 是否存在回写冲突；
      2. `base_commit` 是否漂移；
      3. 相关验收/测试证据是否齐备；
      4. 差异是否已成功实体化。
   3. 只有统一集成链路可以把变更应用到 `../workflow_code`；底层采用 merge、cherry-pick 还是等效策略，当前留待实现阶段细化。
12. FR-CA-12 部署与发布边界
   1. `.running/dev|test|prod` 只能消费已集成版本或其候选快照。
   2. 活动开发工作区或活动工作副本不得直接作为部署目录、发布版本真相源或正式环境升级输入。
13. FR-CA-13 崩溃恢复与清理
   1. 服务重启后必须可恢复：
      1. 活跃租约；
      2. 已阻塞的集成任务；
      3. 运行态证据索引。
   2. 对陈旧工作副本，系统必须支持：
      1. 标记为 `stale`
      2. 复用
      3. 归档
      4. 清理
   3. 即便工作副本已清理，运行证据仍必须保留可追溯入口。
14. FR-CA-14 新角色创建边界
   1. 当前阶段不要求为了代码分离先创建新的 coder/tester 角色。
   2. 后续若需要新增角色，默认通过当前 `../workflow` 中现有“创建角色”功能在 `8090` 端口链路中创建。
   3. “代码与 agent 分离”方案不得阻塞后续使用该角色创建能力。
15. FR-CA-15 渐进迁移
   1. 第一阶段不得要求整站 UI 同步大改。
   2. 现有 `agent_search_root + artifact_root` 配置口径必须仍可启动系统。
   3. 落地顺序固定为：
      1. 固定 `../workflow` 与 `../workflow_code` 的双仓边界；
      2. 运行态外置；
      3. 开发工作区隔离；
      4. 集成与发布串行化。
16. FR-CA-16 现有模块接入边界
   1. `任务中心真实执行`、`训练优化`、`创建角色`、`发布评审` 必须共用同一套“工作副本 + 运行态 + 集成门禁”基础设施，不得形成四套私有实现。
   2. `角色中心` 中现有 `Gate-D 回写冲突门禁` 必须升级为基于 `workspace_lease + integration_lock` 的真实门禁，而不是只停留在逻辑占位。
17. FR-CA-17 审计与可见性
   1. 每个 `run_id` 至少可追溯到：
      1. `agent_id`
      2. `source_repo_path`
      3. `source_commit`
      4. `workspace_path`
      5. `runtime_bundle_path`
      6. `integration_job_id`
      7. `publish_record_id`
   2. 页面或接口至少需要可见：
      1. 当前是否占用工作副本；
      2. 当前是否被集成锁阻塞；
      3. 当前变更尚处于“工作副本态”还是“已集成态”。

## 5. 非功能需求（性能、安全、可用性、合规等）
1. 隔离性：
   1. 普通 agent 执行链路直接写 `../workflow_code` 的次数必须为 `0`。
2. 并发性：
   1. 同一 `agent_id` 可存在多个活跃工作副本；
   2. 同一 `agent_id` 同时处于 `集成中` 或 `发布中` 的任务数最大为 `1`。
3. 可追溯性：
   1. 100% 真实执行批次都必须能关联 `source_repo -> workspace -> runtime -> integration -> publish` 全链路引用。
4. 恢复能力：
   1. 服务启动后 `60 秒` 内必须完成一次租约与锁恢复扫描，并把异常状态外显。
5. 兼容性：
   1. 首期不得强制要求用户理解多根路径配置；
   2. 允许先沿用当前 `agent_search_root` 的用户心智，并由系统内部完成分层派生。
6. 组织约束：
   1. 当前阶段允许 PM 同时承担项目管理与代码开发，不把“新增角色”作为分离方案前置条件。
6. 可维护性：
   1. 工作副本、运行态、集成链路必须是独立模块，不得继续散落到多个业务控制器中临时拼接。

## 6. 验收标准（Given/When/Then）
1. AC-CA-01 同 agent 并行执行隔离
   1. Given 同一 `agent_id` 存在两个并行优化或执行会话
   2. When 两个会话都进入真实执行阶段
   3. Then 系统为它们分配不同的 `workspace_lease_id` 与 `workspace_path`
   4. And 两个会话都不会直接写 `../workflow_code`
2. AC-CA-02 双仓职责生效
   1. Given `../workflow` 与 `../workflow_code` 已按方案分离
   2. When 审核两者职责
   3. Then `../workflow` 只承担 PM/控制/编排职责
   4. And `../workflow_code` 只承担正式代码根仓职责
3. AC-CA-03 开发工作区从代码根仓拉取
   1. Given 某个开发者或开发 agent 首次进入 `workflow` 代码开发
   2. When 系统为其准备开发环境
   3. Then 该开发工作区的代码来源为 `../workflow_code`
   4. And 不直接以 `../workflow` 作为开发代码源
4. AC-CA-04 本地提交并推送到根仓远程分支
   1. Given 某个开发工作区已完成本地代码修改
   2. When 开发者执行 Git 提交与推送
   3. Then 提交先落在本地开发分支
   4. And 推送目标为指向 `../workflow_code` 的远程分支
5. AC-CA-05 运行态不再落到代码仓
   1. Given 某次真实执行产生提示词、stdout/stderr、结果与 `.codex` 运行附属文件
   2. When 检查路径落点
   3. Then 这些文件位于 `agent_runtime_root` 或其派生目录
   4. And `../workflow_code` 中不新增对应运行态明文副本
6. AC-CA-06 任务执行器写入边界正确
   1. Given 系统启动一次 `Codex` 真实执行
   2. When 审核该次执行的命令元数据
   3. Then `-C`、`--add-dir` 与默认写入边界均指向当前 `workspace_path`
   4. And 不再直接指向 `agent_search_root` 或 `../workflow_code` 根
7. AC-CA-07 门禁未通过时主仓不变
   1. Given 某次工作副本执行完成但集成门禁未通过
   2. When 检查 `../workflow_code`
   3. Then 代码主仓保持不变
   4. And 当前会话明确停留在阻塞或待处理状态
8. AC-CA-08 门禁通过后统一集成
   1. Given 某次工作副本执行已通过集成门禁
   2. When 系统执行集成链路
   3. Then 变更由统一 `integration_job` 收口到 `../workflow_code`
   4. And 该次集成可追溯到对应 `run_id` 与 `workspace_lease_id`
9. AC-CA-09 同 agent 集成串行
   1. Given 同一 `agent_id` 存在两个待集成结果
   2. When 第一个结果已持有 `agent_integration_lock`
   3. Then 第二个结果不得进入正式集成
   4. And 页面或接口可见当前持锁者与阻塞原因
10. AC-CA-10 重启后可恢复锁与租约
   1. Given 系统在存在活跃工作副本与锁的情况下异常重启
   2. When 服务恢复完成
   3. Then 系统可识别活跃租约、陈旧租约与阻塞中的集成任务
   4. And 不会因重复恢复而触发两次集成
11. AC-CA-11 部署只消费已集成版本
   1. Given 某个工作副本仍处于活动状态但尚未集成
   2. When 触发 `dev/test/prod` 部署或升级候选生成
   3. Then 部署输入必须来自已集成版本
   4. And 不得直接来自活动工作副本
12. AC-CA-12 旧配置仍可启动
   1. Given 当前配置仍只有 `agent_search_root` 与 `artifact_root`
   2. When 系统启动并执行一次真实任务
   3. Then 系统可自动派生 `agent_workspace_root` 与 `agent_runtime_root`
   4. And 现有核心模块仍可正常进入执行链路
13. AC-CA-13 Gate-D 升级为真实冲突门禁
   1. Given `角色中心 -> 训练优化` 中存在两个针对同一 `agent_id` 的并行会话
   2. When 其中一个会话已经进入集成或发布持锁阶段
   3. Then 另一个会话的 `Gate-D 回写冲突门禁` 必须明确阻塞
   4. And 阻塞理由直接指向真实 `workspace_lease` 或 `integration_lock`
14. AC-CA-14 当前阶段不要求先创建新角色
   1. Given 当前正在推进代码与 agent 分离
   2. When 审核前置条件
   3. Then 不要求先创建新的 coder/tester 角色
   4. And PM 可同时承担代码开发
15. AC-CA-15 后续新角色仍走 8090 创建链路
   1. Given 后续需要新增角色
   2. When 进入角色创建流程
   3. Then 默认通过当前 `../workflow` 的 `8090` 端口“创建角色”能力完成
   4. And 本方案不要求额外建立独立创建角色系统
16. AC-CA-16 非 Git agent 的退化模式可见
   1. Given 目标环境不具备标准 Git 拉取/推送条件
   2. When 系统为其分配工作副本
   3. Then 系统进入 `copy mode`
   4. And 明确提示其集成与回收能力边界弱于 Git 模式
17. AC-CA-17 运行态根失效时 fail-closed
   1. Given `agent_runtime_root` 不可写或不可达
   2. When 系统尝试启动真实执行
   3. Then 当前任务不得进入伪 `running`
   4. And 系统必须返回明确失败原因

## 7. 边界条件与异常处理
1. 若目标 agent 不具备 Git 能力：
   1. 允许退化到 `copy mode`；
   2. 但必须限制其自动集成能力，并保留显式风险提示。
2. 若 `base_commit` 已在等待集成期间发生漂移：
   1. 当前结果不得自动并入主仓；
   2. 必须进入冲突处理或重建工作副本流程。
3. 若用户请求直接改动 `../workflow_code`：
   1. 普通执行链路不得放行；
   2. 必须转入受控集成或管理员路径。
4. 若运行态根目录损坏或证据文件丢失：
   1. 系统必须外显“证据不完整”；
   2. 不得假装本轮执行已具备可集成条件。
5. 若工作副本已被清理但会话仍尝试继续执行：
   1. 系统必须先重建或重新分配工作副本；
   2. 不得沿用失效路径继续执行。

## 8. 依赖项与开放问题
1. 依赖项：
   1. 依赖 `../workflow_code` 作为稳定可访问的代码根仓。
   2. 依赖稳定的 `agent_id/developer_id -> development workspace path` 映射关系。
   3. 依赖当前任务产物外置链路继续可用，以承接 `agent_runtime_root` 的首期派生。
   4. 依赖 Git 在多数开发工作区可用，以支持拉取、提交、推送到代码根仓远程分支。
   5. 依赖 `角色中心`、`任务中心`、发布评审链可消费新的租约、锁与集成状态字段。
2. 开放问题：
   1. `agent_workspace_root` 与 `agent_runtime_root` 首期是否只做内部派生，还是需要在设置页显式展示为高级配置，仍待确认。
   2. `../workflow_code` 作为代码根仓时，Git 远程采用非 bare 仓安全接收，还是再加一层 bare proxy，仍待实现阶段结合 Git 策略收口。
   3. `.codex` 兼容结构最终采用“运行态外置 + 挂载/映射”还是“工作副本按需注入 + 退出后回收”，仍待实现阶段细化。
   4. 开发工作区采用“长期固定一人一仓”还是“固定人仓 + 临时任务副本混合”，仍待结合使用频率与磁盘占用评估。
3. 能力扩展评估（skill/agent）：
   1. 新增 skill：`agent-workspace-lease-governor`
      1. 触发理由：工作副本租约、锁恢复、陈旧副本清理与冲突可见性会在多条执行链路中重复出现。
      2. 采用成本：中。
      3. 采用决策：`采用`，进入实现阶段时创建。
   2. 新增 agent：`integration-orchestrator`
      1. 触发理由：若后续集成门禁、差异审查与发布前验证继续膨胀，可能需要独立集成编排角色。
      2. 采用成本：中到高。
      3. 采用决策：`暂不采用`，当前先由 `../workflow` 控制平面统一编排，PM 也可兼任代码开发。

## 9. 分阶段落地建议
1. Phase A：双仓边界固定
   1. 固定 `../workflow = PM/控制工作区`。
   2. 固定 `../workflow_code = 代码根仓`。
   3. 停止把 `../workflow` 继续当作正式代码开发仓。
2. Phase B：运行态外置
   1. 先把提示词、结果、日志、`.codex` 运行附属文件、锁与租约从 `../workflow_code` 中迁出。
   2. 当前阶段可暂时保留“固定开发工作区”兼容模式，但必须先完成写入边界收口。
3. Phase C：开发工作区隔离
   1. 为 PM 或其他开发主体准备各自独立开发工作区。
   2. 各开发工作区统一从 `../workflow_code` 拉取代码并向其远程分支推送。
4. Phase D：统一集成门禁
   1. 在写 `../workflow_code` 正式集成分支前增加差异预览、冲突校验、证据门禁与集成锁。
   2. 将 `Gate-D 回写冲突门禁` 从逻辑占位升级为真实基础设施门禁。
5. Phase E：发布与部署收口
   1. 发布评审、测试部署、正式升级候选统一只消费已集成版本。
   2. 补充开发工作区、租约、集成状态的可见性与恢复操作入口。
