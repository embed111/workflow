# 需求详情-缺陷处理任务命名可指定

- 状态：正式需求（当前有效口径）
- 当前边界：本轮仅收口“`需求与缺陷 -> 缺陷` 向 `任务中心` 创建处理/复核任务时的任务名称能力”，不改缺陷状态机、不改顺序建单主规则、不新增并行处理能力。

## 1. 主题目标与价值说明
1. 主题目标：
   1. 当一条问题已被 agent 判定为正式缺陷，或已进入 `有分歧` 复核链后，从 `缺陷` 视图向 `任务中心` 创建任务时，系统必须支持指定任务名称，而不是继续只使用“分析缺陷 / 修复缺陷 / 推送到目标版本 / 复核争议”这类固定通用名。
   2. 缺陷处理链上的多个节点必须围绕同一条“任务名称基名”派生命名，使 `任务中心` 中的任务可一眼识别对应的是哪条 DTS、哪一类问题。
   3. 当用户未手工改名时，系统也必须按缺陷记录自动生成具备业务语义的默认任务名称，不能回退成纯固定名。
2. 业务价值：
   1. 降低 `任务中心` 中大量泛化名称带来的检索、区分和追溯成本。
   2. 让“缺陷记录 -> 处理链 -> 版本回写”三者在名称语义上保持一致，减少串行追 bug 时的认知切换。
   3. 避免后续验收或复盘时，只能通过 `node_id`、`report_id` 或打开详情才能判断某条任务到底在处理什么问题。

## 2. 用户画像与使用场景
1. 用户画像 A：缺陷处理者
   1. 关注点：创建处理任务时希望名称直接体现当前缺陷，而不是进入 `任务中心` 后还要逐条点开确认。
2. 用户画像 B：缺陷治理者
   1. 关注点：在串行解决缺陷时，希望当前主动处理项、任务引用和任务中心节点名称能保持同一语义。
3. 用户画像 C：复盘与验收人员
   1. 关注点：通过任务名称即可快速回溯任务与 DTS 记录的对应关系。
4. 核心场景：
   1. 一条问题被判定为正式缺陷，用户点击 `处理缺陷`。
   2. 一条问题进入 `有分歧`，用户补充证据后点击 `提交复核`。
   3. “按顺序创建任务”总开关开启后，系统自动为下一条正式缺陷创建处理任务。
   4. 用户或系统重入建单动作时，需要保证既有名称与既有处理链保持稳定，不因重复点击出现第二条并存链。

## 3. 用户旅程或关键流程
1. 用户提交问题，系统通过 agent 判断其是否属于正式缺陷。
2. 当记录进入 `未解决` 或 `有分歧` 后，用户在缺陷详情中点击 `处理缺陷` 或 `提交复核`。
3. 系统在正式建单前，必须提供“任务名称”确认入口；该入口可以是弹层、抽屉或详情内联输入区，但必须发生在首次建单动作之前。
4. 系统为该入口预填默认任务名称基名，默认值优先使用：`DTS 展示编号 + 空格 + 缺陷摘要`。
5. 用户可直接使用默认值，也可改成更具体的中文名称。
6. 系统基于最终确认的“任务名称基名”派生处理链节点名称，并在 `任务中心` 创建真实任务。
7. 若为复核链，只创建复核任务，但其名称也必须沿用相同的命名语义，而不是回退成通用固定名。
8. 若“按顺序创建任务”总开关开启而当前建单由系统自动触发，系统必须在无人工输入的前提下按默认规则派生任务名称。
9. 若同一 `report_id + action_kind` 已存在有效处理链，系统必须复用既有链路与既有任务名称，不得因再次点击或自动推进再创建第二条并存有效链。

## 4. 功能需求清单
1. FR-DTN-01 建单前必须具备任务名称确认能力
   1. `处理缺陷` 与 `提交复核` 两条建单链路都必须支持任务名称确认。
   2. 对于首次创建的处理链或复核链，系统必须在真正向 `任务中心` 提交前提供可见、可编辑的任务名称输入能力。
   3. 当前能力面向中文工作流，默认文案、默认命名和校验反馈都必须优先使用中文。
2. FR-DTN-02 默认任务名称基名规则
   1. 默认任务名称基名优先采用：`<DTS展示编号> <缺陷摘要>`。
   2. 若 `缺陷摘要` 为空，系统必须至少保留 `DTS 展示编号`，并追加简短兜底语义，如“缺陷处理”或“争议复核”；不得生成空名称，也不得直接回退成“分析缺陷”这一类与具体缺陷脱钩的纯动作名。
   3. 若名称长度超出任务中心当前可接受上限，系统必须优先保留 `DTS 展示编号`，再对摘要部分做稳定截断。
3. FR-DTN-03 处理链节点名称派生规则
   1. 处理链必须围绕同一条“任务名称基名”派生节点名称。
   2. 对 `process` 链，至少包括以下三类节点语义：
      1. `<任务名称基名> - 分析`
      2. `<任务名称基名> - 修复`
      3. `<任务名称基名> - 推送到目标版本`
   3. 对 `review` 链，节点名称至少应表达为：`<任务名称基名> - 复核`。
   4. 分隔符样式可随现有 UI/命名规范微调，但不得丢失“缺陷语义 + 动作语义”这两层信息。
4. FR-DTN-04 用户可手工覆盖默认名称
   1. 用户在首次建单前可编辑任务名称基名。
   2. 系统必须对名称做去首尾空格处理。
   3. 任务名称不得为空；若用户清空后提交，系统必须给出明确中文提示。
   4. 当前轮不要求支持“建单完成后的在线改名”；本轮只要求支持首次建单前确认或修改。
5. FR-DTN-05 自动顺序建单也必须采用语义化命名
   1. 当“按顺序创建任务”总开关开启，且系统自动为下一条缺陷创建处理任务时，也必须使用语义化默认名称。
   2. 自动顺序建单路径不得因为没有人工参与而回退成固定通用名。
   3. 自动顺序建单产生的名称规则必须与手动建单保持同一真相源。
6. FR-DTN-06 缺陷详情、任务引用与任务中心名称必须一致
   1. 缺陷详情中的任务引用标题必须与 `任务中心` 实际创建出的节点名称保持一致或可直接推导一致。
   2. 缺陷历史记录中的“已在任务中心创建处理任务/复核任务”事件，必须可追溯到当时采用的任务名称基名或派生名称。
   3. 用户在 `缺陷` 视图、顺序建单摘要与 `任务中心` 中看到的当前主动处理项，应能通过名称直接互相对上。
7. FR-DTN-07 幂等与重入约束
   1. 对同一 `report_id + action_kind`，若已经存在有效处理链，重复点击、自动推进、刷新或重试不得因为任务名称参数变化而创建第二条并存有效链。
   2. 若系统返回既有处理链，应同时返回既有任务名称，避免 UI 误以为本次改名已经生效。
   3. 当前轮不要求对历史已存在的泛化名称任务做批量重命名。
8. FR-DTN-08 接口契约要求
   1. 面向 UI 与自动化测试复用的缺陷建单接口，必须允许显式传入任务名称基名或等效字段。
   2. 当调用方未显式传入该字段时，后端必须按统一默认规则自动派生。
   3. `process-task` 与 `review-task` 两条接口链路必须采用一致命名契约，避免一条支持、一条不支持。

## 5. 非功能需求
1. 可追溯性：
   1. 任一缺陷处理链都必须能从任务名称直接回溯到对应 `DTS` 记录。
   2. 默认派生命名与人工覆盖命名都必须留下可追溯依据。
2. 一致性：
   1. 同一缺陷在 `缺陷` 视图、任务引用、顺序建单摘要、`任务中心` 节点名称之间不得出现语义错位。
   2. 手动建单与自动顺序建单必须共享同一命名规则。
3. 可理解性：
   1. 用户进入 `任务中心` 后，应在 3 秒内通过名称判断该任务对应的缺陷对象与当前动作阶段。
4. 兼容性：
   1. 历史已创建的固定通用名任务允许继续保留，不要求本轮一次性回写重命名。
   2. 新创建的缺陷处理/复核任务必须采用新命名规则。

## 6. 验收标准
1. AC-DTN-01 手动创建处理链可指定名称
   1. Given 一条正式缺陷记录已生成 `DTS-00003`，摘要为“创建角色画像执行失败”
   2. When 用户点击 `处理缺陷`
   3. Then 系统在正式建单前展示预填任务名称，默认值为 `DTS-00003 创建角色画像执行失败` 或等效语义
   4. And 用户可将其改为 `DTS-00003 创建角色画像执行失败-首轮修复`
   5. And 创建出的分析/修复/推版节点都基于该名称派生，而不是仅显示“分析缺陷 / 修复缺陷 / 推送到目标版本”
2. AC-DTN-02 自动顺序建单不再回退成固定通用名
   1. Given “按顺序创建任务”总开关已开启，且当前队首缺陷为 `DTS-00004`
   2. When 系统自动为该缺陷创建处理链
   3. Then 新任务名称必须按默认规则带上 `DTS-00004` 与缺陷语义
   4. And 不得只生成“分析缺陷”等固定通用名
3. AC-DTN-03 复核链命名一致
   1. Given 一条缺陷记录处于 `有分歧`
   2. When 用户补充证据后点击 `提交复核`
   3. Then 复核任务名称必须带上当前 `DTS` 编号与缺陷语义
   4. And 不得回退成孤立的“复核争议”通用名
4. AC-DTN-04 有效链已存在时保持幂等
   1. Given 某条缺陷的 `process` 处理链已经存在有效任务
   2. When 用户再次点击 `处理缺陷` 并尝试传入另一份名称
   3. Then 系统不得再创建第二条并存有效链
   4. And 应返回既有任务引用与既有任务名称
5. AC-DTN-05 摘要缺失时仍可生成可用名称
   1. Given 某条正式缺陷缺少可用摘要
   2. When 用户或系统触发建单
   3. Then 任务名称仍必须至少保留 `DTS` 编号与简短兜底语义
   4. And 名称不得为空，也不得退化为无法区分对象的纯动作名

## 7. 边界条件与异常处理
1. 若当前缺陷尚未被判定为正式缺陷，也未进入 `有分歧`，则当前轮不要求开放任务名称输入与建单能力。
2. 若缺陷摘要包含噪声文本、超长文本或纯空白，系统必须在默认命名时做清洗、截断或回退处理。
3. 若调用方未传任务名称字段，后端必须自行派生命名，而不是把命名责任完全推给前端。
4. 若用户输入的名称只包含空白字符，系统必须阻止提交并给出中文校验提示。
5. 若历史数据中同一缺陷已存在固定通用名链路，本轮允许继续展示旧名称；是否补做历史改名迁移不纳入当前范围。

## 8. 依赖项与开放问题
1. 依赖项：
   1. `任务中心` 现有节点名称字段仍为当前任务展示真相源。
   2. 缺陷记录可稳定提供 `display_id` 与 `defect_summary`，供默认命名规则使用。
   3. `process-task` / `review-task` 的幂等约束继续沿用现有 `report_id + action_kind` 主规则。
2. 开放问题：
   1. 建单前名称确认入口采用弹层、抽屉还是详情内联输入区，需在后续原型阶段确认。
   2. 是否允许“已创建但尚未执行”的处理链在 `任务中心` 内二次改名，当前轮暂不纳入。
3. 能力扩展评估：
   1. 当前不需要为该需求新增 skill 或新增长期子角色。
   2. 若后续进入跨工作区交付，建议先补一版与现有 `需求与缺陷` 风格一致的建单前命名确认原型，再进入实现与验收。
