# 需求详情-角色中心与训练优化工作流

- 状态：正式需求（当前有效口径）
- 对应原型：沿用 `docs/workflow/prototypes/创建角色工作流-预稿/创建角色工作流预稿.html` 与 `docs/workflow/prototypes/统一入口与训练优化模块/训练闭环-创建任务参考图.html` 的骨架语法；本轮新增 `docs/workflow/prototypes/角色中心与训练优化工作流/训练优化参考图.html` 用于收口“能力列表 + 效果预览 + 评分 + 右侧任务/能力演进”的正式原型口径

## 1. 主题目标与价值说明
1. 主题目标：
   1. 将 `训练中心` 当前有效命名升级为 `角色中心（原训练中心）`，统一承载角色资产与版本、创建角色、训练优化三类角色运营工作。
   2. 将 `训练优化` 从“训练计划表单 + 队列执行”升级为与 `创建角色` 同类的对话式工作流。
   3. 让用户通过对话说清楚“当前 agent 还缺什么能力、要补什么功能、边界是什么、验收标准是什么”，再由系统生成真实优化任务链。
   4. 让 `训练优化` 的执行结果可持续回收到当前会话，并在用户确认通过后回写当前 agent、自动进入发布评审与推送链路。
   5. 为 `创建角色` 与 `训练优化` 共用一套“能力验收门禁 + 历史能力防退化门禁”框架。
2. 业务价值：
   1. 降低“想优化当前 agent，但不知道该填什么表单”的使用门槛。
   2. 把“目标收集 / 任务执行 / 结果回收 / 生效发布”收口成一条可追溯闭环。
   3. 避免 `角色中心`、`任务中心`、发布治理链分别维护三套不一致的状态。
   4. 让优化成功不再停留在“生成了几份任务结果”，而是落实到“当前 agent 已被真正改进并验证通过”。

## 2. 用户画像与使用场景
1. 用户画像 A：角色运营者
   1. 关注点：我想直接告诉系统“当前 agent 还缺什么”，而不是先自己拆训练计划。
2. 用户画像 B：workflow 操作者
   1. 关注点：优化过程要像创建角色一样自然，能边对话边看任务推进与结果回收。
3. 用户画像 C：平台治理者
   1. 关注点：优化结果回写、发布评审和正式推送必须可追溯，且不能把历史功能改坏。
4. 核心场景：
   1. 用户进入 `角色中心 -> 训练优化`，选择一个或多个目标 agent。
   2. 用户通过对话输入目标能力、预期功能、示例、限制、验收标准。
   3. 系统收敛出结构化 `improvement_spec`，并在 `任务中心` 创建真实优化任务图。
   4. 优化任务执行结果持续回传到当前会话，用户可继续调向、补充、否决或通过。
   5. 当门禁通过后，系统回写当前 agent 的 `AGENTS.md`、画像字段与相关能力产物，并自动进入发布评审与推送。

## 3. 用户旅程或关键流程
1. 用户进入 `角色中心`，默认可见三个子模块：
   1. `角色资产与版本`
   2. `创建角色`
   3. `训练优化`
2. 用户进入 `训练优化`，系统要求先明确优化对象：
   1. 允许单 agent 优化；
   2. 允许跨多个 agent 的并行优化；
   3. 每个优化会话必须显式记录本轮目标 agent 集合。
3. 页面默认展示：
   1. 左侧：优化会话列表；
   2. 中部：单聊天壳，用于优化目标收集、系统任务更新、结果回传与确认问答；
   3. 中部聊天壳内默认直接展示“本轮能力列表”；
   4. 右侧：`任务 / 能力演进` 与 `当前能力基线` 双标签辅助区；
   5. 右侧默认激活 `任务 / 能力演进`，不得默认先落到纯门禁说明页。
4. 用户通过对话逐步提供：
   1. 当前问题或缺口；
   2. 想新增或补强的能力；
   3. 禁止边界；
   4. 影响范围；
   5. 示例输入/输出；
   6. 验收标准；
   7. 是否允许自动发布评审与推送。
5. 系统生成结构化 `improvement_spec`，至少包含：
   1. `target_agents`
   2. `improvement_goal`
   3. `problem_statement`
   4. `expected_capabilities`
   5. `constraints`
   6. `acceptance_criteria`
   7. `evidence_examples`
   8. `auto_publish_enabled`
6. 当 `improvement_spec` 达到最小可执行要求后，系统生成优化任务图并在 `任务中心` 创建真实任务。
7. 任务执行过程中：
   1. 当前会话持续接收任务中心回传的系统消息、产物预览、结果摘要与失败原因；
   2. 用户可继续通过对话要求补方向、撤销某条结果、补任务或追加约束；
   3. 系统可根据用户新输入继续向任务中心追加任务。
8. 当任务结果足以支撑回写时，系统进入共享门禁检查：
   1. 新能力验收门禁；
   2. 历史能力防退化门禁；
   3. 回写冲突门禁；
   4. 自动发布门禁。
9. 所有门禁通过后：
   1. 系统回写目标 agent 的 `AGENTS.md`、画像字段与相关能力产物；
   2. 若 `auto_publish_enabled=true`，系统自动进入发布评审、确认发布与推送；
   3. 若任一门禁不通过，系统保持在优化态，等待用户处理或再次执行任务。

## 4. 功能需求清单（编号）
1. FR-RO-01 `角色中心` 命名与子模块收口
   1. 当前 `训练中心` 的有效产品命名改为 `角色中心（原训练中心）`。
   2. `角色中心` 当前有效子模块至少为：
      1. `角色资产与版本`
      2. `创建角色`
      3. `训练优化`
   3. 旧命名 `训练中心` 可在迁移期作为兼容文案存在，但不得继续作为主标题。
2. FR-RO-02 `训练优化` 采用对话式工作流
   1. `训练优化` 必须以对话式收集为主入口，而不是一次性表单。
   2. 中栏必须保持单聊天壳，承载：
      1. 用户输入；
      2. 系统任务更新；
      3. 结果回传；
      4. 验收确认问答。
   3. 右侧辅助区至少提供：
      1. `优化演进图`
      2. `当前能力基线`
3. FR-RO-03 结构化优化草案 `improvement_spec`
   1. 系统必须把用户输入收口为结构化 `improvement_spec`。
   2. 若关键字段缺失，系统必须明确提示缺失项，而不是直接生成任务。
   3. `improvement_spec` 必须支持按轮次迭代，不得把聊天原文直接当作最终优化指令。
4. FR-RO-04 并行优化支持
   1. 系统允许多个优化会话并行存在。
   2. 单个优化会话允许针对一个或多个 agent 同时优化。
   3. 若多个优化会话同时涉及同一 `agent_id`，系统必须在“回写生效”阶段启用 per-agent 写回锁：
      1. 允许并行分析与执行；
      2. 不允许并发写回同一 agent。
   4. 对存在回写冲突的会话，系统必须明确展示冲突对象、先后顺序与阻塞原因。
5. FR-RO-05 训练优化必须通过任务中心执行
   1. `训练优化` 的具体后台动作必须在 `任务中心` 创建真实任务，不得只停留在会话内口头分析。
   2. 当前会话只展示当前优化工作流关联任务，不展示任务中心全量任务。
   3. 每条结果回传必须可追溯到对应 `ticket_id / node_id / run_id`。
6. FR-RO-06 结果回收与会话联动
   1. 任务执行结果必须持续回收到当前优化会话。
   2. 回收内容至少包括：
      1. 结果摘要；
      2. 关键证据；
      3. 预览截图或等效产物；
      4. 失败原因；
      5. 建议下一步动作。
   3. 用户可基于这些回收结果继续通过对话调向。
7. FR-RO-07 允许回写 `AGENTS.md`
   1. 当优化结果进入生效阶段，系统允许修改目标 agent 工作区中的 `AGENTS.md`。
   2. 回写范围至少允许覆盖：
      1. 角色画像正文；
      2. 能力描述；
      3. 边界与协作方式；
      4. 与当前优化目标直接相关的章节。
   3. 回写前必须产出结构化差异预览，供系统或用户审查。
8. FR-RO-08 允许回写角色画像字段
   1. 系统允许在优化生效阶段更新：
      1. `capability_summary`
      2. `knowledge_scope`
      3. `agent_skills`
      4. `applicable_scenarios`
      5. `version_notes`
   2. 更新后的字段必须与发布报告/当前正式画像保持可追溯关系。
9. FR-RO-09 自动发布评审与推送
   1. 当 `auto_publish_enabled=true` 且所有门禁通过时，系统可自动进入发布评审与推送链路。
   2. 自动链路至少包括：
      1. 进入发布评审；
      2. 生成发布报告；
      3. 确认发布；
      4. 推送到目标版本或正式环境。
   3. 若任一环节失败，系统必须回传失败原因并中止后续自动动作。
10. FR-RO-10 共享能力验收门禁框架
   1. `创建角色` 与 `训练优化` 必须共用一套共享门禁框架。
   2. 当前至少包含以下门禁：
      1. `Gate-A 输入收敛门禁`
      2. `Gate-B 新能力验收门禁`
      3. `Gate-C 历史能力防退化门禁`
      4. `Gate-D 回写冲突门禁`
      5. `Gate-E 自动发布门禁`
11. FR-RO-11 Gate-A 输入收敛门禁
   1. 未收敛 `improvement_spec` 前，不得创建优化任务图。
   2. 至少要求：
      1. 目标 agent 已明确；
      2. 优化目标已明确；
      3. 约束已明确；
      4. 验收标准已明确。
12. FR-RO-12 Gate-B 新能力验收门禁
   1. 优化结果未通过新能力验收前，不得进入回写阶段。
   2. 至少要求：
      1. 新能力有可回看结果；
      2. 结果与目标能力一一对应；
      3. 验收标准存在明确通过结论。
13. FR-RO-13 Gate-C 历史能力防退化门禁
   1. `训练优化` 在生效前必须验证历史能力无明显退化。
   2. 至少要求：
      1. 存在优化前能力基线；
      2. 存在历史关键能力用例或等效 smoke 测试；
      3. 对比结果可追溯到具体证据。
   3. 若历史能力出现明显退化，系统不得自动发布。
14. FR-RO-14 Gate-D 回写冲突门禁
   1. 若同一 agent 存在并行优化会话，系统在写回前必须检查是否存在未解决冲突。
   2. 冲突未解决前，不得自动修改 `AGENTS.md` 或进入发布评审。
15. FR-RO-15 Gate-E 自动发布门禁
   1. 自动发布前必须同时满足：
      1. 新能力验收通过；
      2. 历史能力无明显退化；
      3. 回写冲突已解决；
      4. 本轮回写差异已成功落盘。
   2. 任一条件不满足时，系统必须停留在“待处理”状态，而不是继续自动推送。
16. FR-RO-16 创建角色共享门禁适配
   1. `创建角色` 虽然不存在“历史能力防退化”主诉求，但仍需接入共享门禁框架。
   2. 在 `创建角色` 场景下：
      1. `Gate-A` 适用于角色画像与约束收敛；
      2. `Gate-B` 适用于首批能力验收；
      3. `Gate-C` 可退化为“首发基线完整性校验”；
      4. `Gate-D / Gate-E` 视是否启用首发自动发布而决定是否执行。
17. FR-RO-17 历史能力基线可视化
   1. `训练优化` 右侧必须可查看当前 agent 的能力基线摘要。
   2. 至少展示：
      1. 当前正式版本；
      2. 当前角色画像摘要；
      3. 历史关键能力清单；
      4. 本轮回归验证结果。
18. FR-RO-18 能力对象模型
   1. `训练优化` 中的“能力”必须被建模为独立对象，而不是零散文案片段。
   2. 每个能力对象至少包含：
      1. `capability_id`
      2. `capability_name`
      3. `capability_goal`
      4. `preview_evidence`
      5. `score_current`
      6. `score_target`
      7. `gate_status`
      8. `historical_regression_result`
      9. `impact_scope`
   3. `Gate-B` 与 `Gate-C` 的判定必须能回溯到具体能力对象，不能只给会话级总结果。
19. FR-RO-19 能力列表展示规则
   1. `训练优化` 默认以列表展示本轮目标能力，不以图表、矩阵或折叠摘要替代主入口。
   2. 每条能力列表项至少展示：
      1. 能力名称；
      2. 当前状态；
      3. 对应能力展示效果；
      4. 能力评分；
      5. 关联门禁状态；
      6. 对历史能力的影响结论。
   3. 能力列表应优先出现在中部聊天壳中，作为会话结果回收的一部分。
20. FR-RO-20 能力展示效果与评分绑定
   1. 每条能力项都必须同时展示“效果证据”和“评分”，不得只展示其一。
   2. `preview_evidence` 至少允许以下任一种：
      1. 截图预览；
      2. 结构化输出片段；
      3. 页面块预览；
      4. 结果引用卡。
   3. 评分展示至少包含：
      1. 基线分或当前正式分；
      2. 当前轮分数；
      3. 目标分或阈值；
      4. 分差结论。
21. FR-RO-21 历史能力回归结果展示
   1. 对任一能力对象，若其影响历史能力，系统必须在该能力项内直接展示回归结果。
   2. 当历史能力出现下降时：
      1. 该能力项必须标记为风险或阻塞；
      2. `Gate-C` 必须直接绑定到该能力项；
      3. 自动发布必须保持阻塞态。
   3. 当历史能力未受影响时，也必须明确显示“未影响”或等效安全结论。

## 5. 非功能需求（性能、安全、可用性、合规等）
1. 可理解性：
   1. 用户进入 `训练优化` 后，应在 3 秒内理解：
      1. 当前正在优化哪个 agent；
      2. 当前处于哪个阶段；
      3. 当前结果是否已可回写；
      4. 下一步是继续补任务还是确认生效。
2. 一致性：
   1. `角色中心` 的布局、对话语法与任务图表达需与 `创建角色` 保持同类语义。
   2. 同一优化会话中的状态、任务结果与门禁结论不得出现前后矛盾。
3. 可追溯性：
   1. 任一优化会话必须可追溯到：
      1. 原始目标输入；
      2. `improvement_spec`；
      3. 任务中心任务链；
      4. 结果回收记录；
      5. 回写差异；
      6. 发布评审/推送记录。
4. 安全性：
   1. 自动回写 `AGENTS.md` 时必须保留变更前后差异证据。
   2. 自动推送失败时不得掩盖失败原因。
5. 稳定性：
   1. 同一 agent 的并发优化会话不得导致无锁写回。
   2. 历史能力防退化验证失败时，不得错误放行到自动发布。

## 6. 验收标准（Given/When/Then）
1. AC-RO-01 `角色中心` 命名生效
   1. Given 用户进入原 `训练中心`
   2. When 页面加载完成
   3. Then 主标题显示为 `角色中心`
   4. And 子模块至少为 `角色资产与版本 / 创建角色 / 训练优化`
2. AC-RO-02 训练优化入口为对话式
   1. Given 用户进入 `角色中心 -> 训练优化`
   2. When 页面渲染完成
   3. Then 可见单聊天壳输入区与会话消息流
   4. And 不再只展示训练计划表单
3. AC-RO-03 `improvement_spec` 收敛后才能建任务
   1. Given 用户尚未明确目标 agent、目标能力或验收标准
   2. When 用户尝试启动优化
   3. Then 系统拒绝创建优化任务图
   4. And 明确提示缺失项
4. AC-RO-04 并行优化可见但同 agent 写回受控
   1. Given 同时存在两个优化会话
   2. When 两个会话涉及不同 agent
   3. Then 系统允许并行推进
   4. When 两个会话同时尝试回写同一个 agent
   5. Then 系统拦截后一个会话并提示回写冲突
5. AC-RO-05 优化必须走任务中心
   1. Given 用户已启动训练优化
   2. When 系统生成执行链路
   3. Then `任务中心` 中可看到真实任务节点
   4. And 当前优化会话只引用该工作流关联任务
6. AC-RO-06 结果持续回收到会话
   1. Given 某条优化任务已开始执行
   2. When 任务产生中间结果或最终结果
   3. Then 当前优化会话中可看到系统消息、结果摘要与证据预览
7. AC-RO-07 新能力验收门禁生效
   1. Given 新能力结果尚未达到验收标准
   2. When 系统尝试进入回写阶段
   3. Then 系统拒绝回写
   4. And 明确指出未通过的验收项
8. AC-RO-08 历史能力防退化门禁生效
   1. Given 本轮优化结果已实现新增能力
   2. When 历史关键能力验证失败
   3. Then 系统不得自动发布
   4. And 会话中明确提示“历史能力有明显退化”
9. AC-RO-09 允许回写 `AGENTS.md`
   1. Given 当前优化结果已通过所有共享门禁
   2. When 系统进入生效阶段
   3. Then 系统可回写目标 agent 的 `AGENTS.md`
   4. And 页面可查看回写差异摘要
10. AC-RO-10 自动发布评审与推送可执行
   1. Given `auto_publish_enabled=true` 且门禁全部通过
   2. When 系统执行生效链路
   3. Then 系统自动进入发布评审并继续推送
   4. And 失败时会回传明确失败原因并停止后续动作
11. AC-RO-11 创建角色共享门禁适配
   1. Given 用户处于 `角色中心 -> 创建角色`
   2. When 系统准备完成角色创建
   3. Then 至少完成输入收敛、首批能力验收与首发基线完整性校验
   4. And 未通过门禁前不得直接完成创建
12. AC-RO-12 能力列表为训练优化默认主视图
   1. Given 用户进入 `角色中心 -> 训练优化`
   2. When 页面完成首屏渲染
   3. Then 中部聊天壳中可直接看到本轮能力列表
   4. And 列表项数量与 `expected_capabilities` 数量一致或有明确差异提示
13. AC-RO-13 每项能力都能同时看到效果与评分
   1. Given 某项能力已进入结果回收阶段
   2. When 用户查看该能力项
   3. Then 可同时看到该能力的效果证据与评分信息
   4. And 不会出现“只有评分没有效果”或“只有效果没有评分”
14. AC-RO-14 门禁结果绑定到具体能力项
   1. Given 某项能力未通过新能力验收或触发历史能力退化
   2. When 用户查看该能力项
   3. Then 该能力项内可直接看到对应的 `Gate-B` 或 `Gate-C` 结论
   4. And 会话级总门禁结论可追溯到该能力项
15. AC-RO-15 右侧默认先看任务 / 能力演进
   1. Given 用户首次进入某个训练优化会话
   2. When 右侧辅助区加载完成
   3. Then 默认激活 `任务 / 能力演进`
   4. And `当前能力基线` 作为可切换的第二标签而非默认首屏

## 7. 边界条件与异常处理
1. 当用户只说“优化一下当前 agent”但未说明具体目标：
   1. 系统保持在收集阶段；
   2. 不得创建任务。
2. 当并行优化会话引用同一 agent：
   1. 允许并行分析和执行；
   2. 不允许并发写回。
3. 当历史能力基线缺失：
   1. 系统不得跳过防退化门禁直接自动发布；
   2. 必须提示补齐基线或转人工确认。
4. 当自动发布链路失败：
   1. 优化会话必须保留在可恢复状态；
   2. 已成功回写的工作区差异不得被静默丢失。
5. 当 `AGENTS.md` 回写失败：
   1. 系统不得继续发布；
   2. 必须回传失败原因与回滚建议。

## 8. 依赖项与开放问题
1. 依赖项：
   1. 依赖 `任务中心` 提供真实执行、执行链路可视化与结果回收能力。
   2. 依赖 `角色资产与版本` 提供当前能力基线与发布状态事实源。
   3. 依赖发布评审链已具备可自动进入、确认发布与推送的实现基础。
2. 开放问题：
   1. 跨多个 agent 的同一优化会话在 UI 上采用“单主图 + 多 agent 分组”还是“多主图并列”，仍待实现阶段细化。
   2. 历史能力“明显退化”的默认阈值与判定规则，仍待实现阶段结合现有验收资产细化。
3. 能力扩展评估（skill/agent）：
   1. 新增 skill：`role-optimization-gate`
      1. 触发理由：训练优化与创建角色共享门禁较多，需统一校验。
      2. 采用成本：中。
      3. 采用决策：`采用`（进入实现阶段时创建）。
   2. 新增 agent：专职“历史能力回归验证 agent”
      1. 触发理由：后续若回归验证规模增大，手写 smoke 规则可能不足。
      2. 采用成本：中到高。
      3. 采用决策：`暂不采用`（当前先复用任务中心与现有执行链路）。
