# 需求详情-定时任务模块与日历视图

- 状态：正式需求（当前有效口径）
- 当前边界：本轮收口与 `任务中心` 同级的 `定时任务` 模块；首期覆盖规则配置、列表/详情视角、日历视角、命中后向 `任务中心` 全局唯一任务图追加单任务节点并请求运行，不展开节假日跳过、停机补跑与复杂 cron 表达式。

## 1. 主题目标与价值说明
1. 主题目标：
   1. 在 `../workflow` 左侧导航中新增一个与 `任务中心` 同级的 `定时任务` 模块。
   2. 让用户可以把重复性的后台工作沉淀为可持续复用的定时计划，而不是每天手工去 `任务中心` 建任务。
   3. 计划命中后，系统统一向 `任务中心` 全局唯一任务图追加 1 条新的单任务节点，并请求进入既有运行流程。
   4. 同一计划允许混合启用 `每月 / 每周 / 每日 / 定时` 四类规则，但同一分钟命中时只创建一次任务。
   5. 让用户既能用“左侧列表 + 右侧详情”管理计划，也能通过“日历视角”像飞书日历那样回看计划与执行结果。
2. 业务价值：
   1. 把“周期性工作”的真相源从聊天记忆和人工习惯迁到系统内，减少漏跑和忘跑。
   2. 让“计划规则”与“真实执行”解耦：计划归 `定时任务`，执行归 `任务中心`，避免复制两套执行系统。
   3. 让用户可以直接看见“什么时候应该触发、什么时候真的触发了、结果如何”，而不是只看到抽象规则文本。

## 2. 用户画像与使用场景
1. 用户画像 A：重复工作操作者
   1. 关注点：把每天/每周/每月都会做的工作变成固定计划，到点自动建单并尽快开始执行。
2. 用户画像 B：平台治理者
   1. 关注点：触发是否准时、是否会重复建单、任务中心中的任务引用能否追溯回原始计划。
3. 用户画像 C：结果查看者
   1. 关注点：在日历里快速看某天有哪些计划、本来该跑什么、最后是否成功。
4. 核心场景：
   1. 用户进入 `定时任务` 模块，默认看到左侧计划列表与右侧详情。
   2. 用户创建一个新的定时计划，为其配置发起摘要、执行清单、完成标准、执行 agent、优先级和触发规则。
   3. 用户为同一计划同时开启 `每周一/三/五 09:00` 与 `每月 1 号 09:00` 等混合规则。
   4. 当同一计划的多条规则在同一分钟同时命中时，系统只向全局唯一任务图追加 1 条新的任务节点。
   5. 用户切换到 `日历视角`，查看未来计划和历史执行结果，并能从某一天回到具体计划详情。

## 3. 用户旅程或关键流程
1. 用户进入 `定时任务` 模块。
2. 页面默认进入 `列表视角`，左侧展示计划列表，右侧展示当前选中计划的详情。
3. 用户点击左侧 `定时任务列表` 右上角的 `+` 新建入口，在当前模块内打开创建/编辑面板。
4. 用户填写计划基础信息：
   1. 计划名称；
   2. 执行 agent；
   3. 发起摘要；
   4. 执行清单；
   5. 完成标准；
   6. 优先级；
   7. 预期产物（选填）；
   8. 产物交付方式（沿用任务中心口径）。
5. 用户为当前计划配置 1 条或多条规则组：
   1. `每月`：选择月内日期与时段；
   2. `每周`：选择周几与时段；
   3. `每日`：选择每日固定时段；
   4. `定时`：填写一个或多个明确的日期时间点。
6. 用户保存计划后，系统开始按分钟级扫描该计划的下一次命中时间。
7. 当计划命中时：
   1. 系统先做“同计划同分钟”去重；
   2. 去重通过后，向 `任务中心` 全局唯一任务图追加 1 条新的单任务节点；
   3. 新任务节点写入 `source_schedule_id / planned_trigger_at / trigger_instance_id` 等来源信息；
   4. 系统立即请求 `任务中心` 进入既有调度/运行流程。
8. 若 `任务中心` 当前因依赖、单 agent 单并发或全局并发上限无法立即运行：
   1. 新任务节点仍然创建成功；
   2. 实际运行状态继续由 `任务中心` 判定；
   3. `定时任务` 模块仅同步展示其结果，不绕过门禁强行开跑。
9. 用户切换到 `日历视角`：
   1. 可看到未来计划触发；
   2. 可看到已触发实例的执行结果；
   3. 点击某一天或某条日历事件后，右侧详情同步显示该计划与关联任务引用。

## 4. 功能需求清单
1. FR-ST-01 顶层模块结构
   1. `workflow` 左侧导航当前有效口径至少包含以下 7 个顶层模块：
      1. `对话`
      2. `工作记录与训练`
      3. `训练中心`
      4. `任务中心`
      5. `定时任务`
      6. `需求与缺陷`
      7. `设置`
   2. `定时任务` 必须作为独立顶层模块存在，不挂到 `任务中心` 二级页签下。
2. FR-ST-02 双视角工作台
   1. `定时任务` 模块内容区至少包含两个顶层视角：
      1. `列表视角`
      2. `日历视角`
   2. 两个视角必须通过模块顶部页签切换，而不是跳到两个独立页面。
3. FR-ST-03 列表视角结构
   1. `列表视角` 默认采用“左侧计划列表 + 右侧详情”的双栏结构。
   2. 左侧列表至少展示：
      1. 计划名称；
      2. 计划状态；
      3. 规则摘要；
      4. 下一次触发时间；
      5. 最近一次执行结果。
   3. 右侧详情至少展示：
      1. 计划基础信息；
      2. 发起任务内容预览；
      3. 规则配置；
      4. 未来触发时间；
      5. 最近生成的任务中心任务引用；
      6. 最近执行结果；
      7. 计划管理动作。
   4. 计划管理动作至少包含：
      1. `编辑计划`
      2. `启用/停用计划`
      3. `删除计划`
   5. 用户必须可从右侧详情直接进入编辑态，而不是只能删除后重建。
   6. `新建定时任务` 入口必须放在左侧 `定时任务列表` 面板右上角，采用轻量 `+` 图标按钮，不放在模块页头主标题区。
4. FR-ST-04 日历视角结构
   1. `日历视角` 必须保持与现有 `workflow` 单窗口工作台一致的视觉语言。
   2. 日历主区域至少展示按日期排列的计划触发事件。
   3. 右侧详情区继续保留，用于展示当前选中日期或计划的详细信息。
   4. 日历右侧详情区必须直接提供 `编辑计划` 入口，不要求用户必须先切回 `列表视角` 才能修改计划。
   5. 日历右侧详情区必须能看到本次将发起的任务摘要或关键动作，避免只知道“会触发”但不知道“触发后具体做什么”。
   6. 日历月视图必须支持 `上一月 / 下一月` 切换，用户不需要返回其他页面即可连续查看不同月份。
   7. 当前展示月份必须在日历头部明确可见，并在切月后同步刷新日期格与右侧详情上下文。
   8. 推断/假设（置信度: 中）：首期日历默认采用“月视图 + 选中日详情”的结构，不强制在首期同时做周视图/日视图切换。
5. FR-ST-05 定时计划基础字段
   1. 每个定时计划至少包含：
      1. `schedule_id`
      2. `schedule_name`
      3. `enabled`
      4. `assigned_agent_id`
      5. `launch_summary`
      6. `execution_checklist`
      7. `done_definition`
      8. `priority`
      9. `expected_artifact`（选填）
      10. `delivery_mode`
      11. `delivery_receiver_agent_id`（指定交付人时必填）
      12. `rule_sets`
      13. `next_trigger_at`
      14. `last_trigger_at`
      15. `last_result_status`
   2. `assigned_agent_id` 候选范围必须与 `任务中心` 一致，来自当前角色池。
   3. `priority` 口径必须与 `任务中心` 一致，固定为 `P0 / P1 / P2 / P3`。
   4. `launch_summary / execution_checklist / done_definition` 不得只停留在抽象目标层，必须让执行者在触发后明确知道“现在要做什么、做到什么程度算完成”。
   5. 上述字段名称可按现有 `任务中心` 数据模型做等价映射，首期重点是语义完整，不强制要求后端立即统一改名。
6. FR-ST-06 新建/编辑/删除/启停
   1. 模块必须支持定时计划的新增、编辑、删除、启用与停用。
   2. 停用后的计划不得继续触发新的任务引用。
   3. 删除计划不得抹除其历史触发记录与已创建的任务中心任务引用。
   4. 编辑计划时必须允许修改：
      1. 计划名称；
      2. 执行 agent；
      3. 发起摘要；
      4. 执行清单；
      5. 完成标准；
      6. 优先级；
      7. 规则组；
      8. 交付方式。
   5. 编辑保存后，后续未触发的计划应按最新规则生效；已创建的历史任务中心任务引用不得被回写篡改。
7. FR-ST-07 四类规则组
   1. 首期必须支持以下 4 类规则组，并允许同一计划混合启用：
      1. `每月`
      2. `每周`
      3. `每日`
      4. `定时`
   2. 各规则组的最小配置口径如下：
      1. `每月`：月内日期 + 一个或多个时段；
      2. `每周`：周几 + 一个或多个时段；
      3. `每日`：一个或多个每日固定时段；
      4. `定时`：一个或多个明确的日期时间点。
   3. 至少配置 1 组规则后，计划才允许保存为启用态。
8. FR-ST-08 同分钟合并规则
   1. 当同一计划的多条规则在同一分钟内同时命中时，系统必须合并为一次触发。
   2. 合并后的单次触发仍需保留“命中来源规则摘要”用于审计和详情展示。
   3. 不同计划之间即使命中同一分钟，也不得互相合并。
   4. 上述合并规则属于系统行为与留痕要求，默认界面不要求在模块页头持续展示解释性提示。
9. FR-ST-09 触发到任务中心的对象语义
   1. 每次计划命中后，系统必须向 `任务中心` 全局唯一任务图追加 1 条新的单任务节点。
   2. 首期不得把一次命中直接展开成整张任务图模板。
   3. 新任务节点至少应写入以下来源字段：
      1. `source_schedule_id`
      2. `planned_trigger_at`
      3. `trigger_instance_id`
      4. `trigger_rule_summary`
   4. 每次计划命中追加到全局唯一任务图的新任务节点，必须同时带上计划侧保存的 `launch_summary / execution_checklist / done_definition` 快照，避免执行者只看到抽象任务名。
   5. 定时任务模块不得为每次命中创建新的独立任务图；其输出对象固定为“全局唯一任务图中的 1 个新任务节点”。
10. FR-ST-10 创建后立即请求运行
   1. 成功向全局唯一任务图追加任务节点后，系统必须立即请求进入 `任务中心` 的既有调度/运行流程。
   2. `定时任务` 模块不得绕过 `任务中心` 的依赖门禁、单 agent 单并发与全局并发上限。
   3. 若当前无法立即运行，该任务节点应保留在 `任务中心` 的真实状态中等待后续调度。
11. FR-ST-11 任务中心真相源关系
   1. `定时任务` 模块只负责：
      1. 计划配置；
      2. 命中触发；
      3. 展示来源关联与执行结果摘要。
   2. 一旦任务节点追加成功，其执行状态、执行证据与最终结果真相源必须来自 `任务中心`。
   3. 用户必须可从定时计划详情跳转或联动查看对应的 `任务中心` 任务引用。
12. FR-ST-12 列表视角中的结果可见性
   1. 左侧计划列表必须直接显示最近一次执行结果。
   2. 最近一次执行结果至少区分：
      1. `待触发`
      2. `已建单待调度`
      3. `运行中`
      4. `已成功`
      5. `已失败`
   3. 推断/假设（置信度: 中）：若最近一次触发因多规则命中被合并，可在摘要中提示“本次为合并触发”。
13. FR-ST-13 详情视角中的规则与结果
   1. 右侧详情区至少包含以下区块：
      1. `基础配置`
      2. `发起任务内容预览`
      3. `触发规则`
      4. `未来触发`
      5. `最近执行结果`
      6. `关联任务中心任务引用`
   2. `最近执行结果` 至少展示最近 5 次触发的计划时间、建单结果、运行状态与任务引用。
   3. `发起任务内容预览` 至少包含：
      1. 本次目标摘要；
      2. 执行清单或关键动作；
      3. 完成标准或交付结果要求。
14. FR-ST-14 日历视角中的计划与结果叠加
   1. 日历视角必须同时展示：
      1. 未来计划触发；
      2. 已触发任务引用的实际执行结果。
   2. 不得只展示计划、不展示结果。
   3. 同一天存在多次触发时，应在该日期格中分条显示或聚合后可下钻查看。
15. FR-ST-15 列表与日历一致性
   1. 同一计划在 `列表视角` 与 `日历视角` 中看到的下一次触发时间、最近结果与启停状态必须一致。
   2. 用户从日历点击某一事件后，右侧详情区必须能回到同一条定时计划与对应任务引用。
16. FR-ST-16 审计与留痕
   1. 系统必须记录以下关键动作：
      1. 创建计划；
      2. 编辑计划；
      3. 启用/停用计划；
      4. 删除计划；
      5. 计划命中；
      6. 同分钟合并；
      7. 向任务中心全局唯一任务图追加任务节点；
      8. 请求调度/运行；
      9. 触发失败。
   2. 留痕至少包含：时间、对象、动作、操作者或系统来源、结果与引用。
17. FR-ST-17 首期边界
   1. 首期不强制支持：
      1. 节假日跳过；
      2. 工作日历；
      3. 停机补跑；
      4. 复杂 cron 表达式；
      5. 跨时区共享计划。
   2. 上述能力若未实现，页面不得伪装为已支持。
18. FR-ST-18 时间基准
   1. 首期所有规则命中与日历展示默认使用 `北京时间（Asia/Shanghai）`。
   2. 默认界面当前不要求显式展示时区标签；只需按统一时间基准解释与计算。
   3. 若后续引入显式时区配置，应以统一配置为准，首期不做计划级单独时区。

## 5. 非功能需求
1. 准时性：
   1. 计划命中后，向任务中心全局唯一任务图追加任务节点的延迟 `p95 <= 60 秒`。
2. 去重准确性：
   1. 同一计划因多规则在同一分钟重复命中时，重复建单率必须为 `0`。
3. 一致性：
   1. 列表视角、日历视角与 `任务中心` 中的任务引用来源关联必须保持一致。
4. 可理解性：
   1. 用户进入模块后，应在 3 秒内理解：
      1. 哪些计划启用中；
      2. 下一个要触发什么；
      3. 最近一次跑得怎么样。
5. 可执行性：
   1. 用户在查看计划详情或日历详情时，应在 5 秒内理解“触发后要做的具体动作”和“什么结果才算完成”。
6. 可观测性：
   1. 任一计划触发都必须可追溯到规则命中、去重结果、追加节点结果与任务引用。

## 6. 验收标准
1. AC-ST-01 顶层模块可见
   1. Given 用户打开 `workflow`
   2. When 左侧导航加载完成
   3. Then 可看到 `对话 / 工作记录与训练 / 训练中心 / 任务中心 / 定时任务 / 需求与缺陷 / 设置`
2. AC-ST-02 默认进入列表视角
   1. Given 用户进入 `定时任务`
   2. When 页面首次加载完成
   3. Then 默认看到左侧计划列表与右侧详情
   4. And 顶部可切换到 `日历视角`
   5. And 左侧计划列表右上角可见轻量 `+` 新建入口
   6. And 右侧详情可见 `编辑计划` 等管理入口
3. AC-ST-03 支持混合规则保存
   1. Given 用户正在创建定时计划
   2. When 同时配置 `每周一 09:00` 与 `每月 1 号 09:00`
   3. Then 系统允许保存为同一条启用中的定时计划
4. AC-ST-04 同分钟命中合并
   1. Given 同一计划的两条规则在 `2026-04-01 09:00` 同时命中
   2. When 调度器执行命中处理
   3. Then 系统只向全局唯一任务图追加 1 条新的任务节点
   4. And 当前触发记录可追溯到“多规则合并”
5. AC-ST-05 命中后建单并请求运行
   1. Given 某定时计划已到命中时间
   2. When 触发流程执行
   3. Then 系统向 `任务中心` 全局唯一任务图追加 1 条新的单任务节点
   4. And 随后请求进入既有调度/运行流程
6. AC-ST-06 不绕过任务中心门禁
   1. Given 当前目标 agent 已有运行中任务
   2. When 定时计划命中并追加新任务节点
   3. Then 新任务节点不得绕过 `任务中心` 的并发门禁直接伪运行
   4. And `定时任务` 模块可看到该任务引用处于真实等待/调度状态
7. AC-ST-07 日历同时展示计划与结果
   1. Given 用户切换到 `日历视角`
   2. When 查看某一日期
   3. Then 可同时看到计划中的触发和已经发生的执行结果
   4. And 右侧日历详情区可直接看到 `编辑计划` 入口
   5. And 可看到该次触发会发起的任务摘要或关键动作
8. AC-ST-08 启停生效
   1. Given 某定时计划当前为启用状态
   2. When 用户将其停用
   3. Then 后续命中时间不再向全局唯一任务图追加新的任务节点
9. AC-ST-09 计划可编辑
   1. Given 用户已选中一条已有定时计划
   2. When 用户点击 `编辑计划` 并修改规则或基础字段后保存
   3. Then 计划详情、列表摘要与后续未触发时间按最新配置更新
   4. And 已创建的历史任务引用保持原样可追溯
10. AC-ST-10 删除不抹历史
   1. Given 某定时计划已经产生过历史触发与任务引用
   2. When 用户删除该定时计划
   3. Then 计划不再出现在活动列表
   4. And 历史触发记录与已创建任务引用仍可追溯
11. AC-ST-11 列表与日历信息一致
   1. Given 某定时计划在列表中显示下一次触发为 `明天 09:00`
   2. When 用户切换到日历视角并查看对应日期
   3. Then 日历中应能看到相同的计划事件和一致的状态信息
12. AC-ST-12 发起内容清晰可执行
   1. Given 用户查看某条定时计划详情
   2. When 页面展示该计划的发起任务内容
   3. Then 用户可直接看到本次目标摘要、执行清单和完成标准
   4. And 计划命中后追加到全局唯一任务图中的任务节点也能看到同一份内容快照
13. AC-ST-13 日历支持切月
   1. Given 用户位于 `日历视角`
   2. When 用户点击 `上一月` 或 `下一月`
   3. Then 日历头部的月份标签发生变化
   4. And 日期格中的计划与结果同步切换到对应月份
   5. And 右侧详情区同步刷新为当前月份中的选中日期或默认日期上下文

## 7. 边界条件与异常处理
1. 若定时计划未配置任何规则组：
   1. 不得允许其保存为启用状态。
2. 若同一计划在同一分钟内被多条规则命中：
   1. 必须先合并，再建单。
3. 若向全局唯一任务图追加任务节点成功，但请求调度/运行失败：
   1. 必须记录“已追加任务但请求运行失败”的结果，不得伪装为运行成功。
4. 若 `任务中心` 当前不可用：
   1. 本次触发必须记录失败原因，并可在详情中查看。
5. 若显式 `定时` 规则中的某个具体时点已过期：
   1. 该时点不得重复触发；
   2. 历史记录仍需保留。
6. 若用户删除计划时仍存在已创建但未完成的任务引用：
   1. 删除动作只影响未来触发，不得反向删除既有任务引用。

## 8. 依赖项与开放问题
1. 依赖项：
   1. 依赖 `任务中心` 提供稳定的“向全局唯一任务图追加单任务节点 + 调度 + 状态读取”能力。
   2. 依赖当前角色池提供可选执行 agent。
   3. 依赖统一的时间扫描与事件留痕能力。
2. 开放问题：
   1. 服务停机期间错过的计划，首期是否需要自动补跑。
      1. 推断/假设（置信度: 中）：首期可先不补跑，但必须明确留痕“该次命中未执行”。
   2. 首期是否需要加入节假日跳过、工作日历与法定节假日规则。
      1. 推断/假设（置信度: 中）：建议后续独立扩展，不在本轮混入。
   3. 后续是否需要把“单任务节点”升级为“任务模板/任务图模板”。
      1. 推断/假设（置信度: 中）：首期先保持“向全局唯一任务图追加单任务节点”，避免把 `定时任务` 做成第二个任务编排中心。
