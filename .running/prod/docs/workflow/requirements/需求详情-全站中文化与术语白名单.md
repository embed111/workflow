# 需求详情-全站中文化与术语白名单

## 1. 主题目标与价值说明
1. 主题目标：
   1. 将 `workflow` 的用户可见文案统一为中文为主。
   2. 建立“可保留英文”的术语白名单，避免全量硬替换导致可读性下降。
2. 业务价值：
   1. 降低非技术用户阅读门槛。
   2. 统一产品语义，减少同义词和中英混写带来的理解偏差。
   3. 为后续文案门禁与自动扫描提供稳定规则。

## 2. 用户画像与使用场景
1. 用户画像 A：平台运营与产品负责人
   1. 关注点：界面术语是否统一、是否易懂。
2. 用户画像 B：训练与验收人员
   1. 关注点：验收截图中的文案是否一致可追溯。
3. 核心场景：
   1. 打开工牌与发布管理页面，看到中文术语一致展示。
   2. 全站页面中，除白名单术语外不出现英文 UI 文案。

## 3. 用户旅程或关键流程
1. 建立文案清单（页面路径 -> 文案项）。
2. 建立术语白名单与中文映射表。
3. 执行页面文案替换（仅展示层）。
4. 执行残留英文扫描与人工复核。
5. 回归 UO/AR 并输出证据。

## 4. 功能需求清单（编号）
1. FR-ZH-01 全站中文化范围
   1. 覆盖工牌、版本发布、训练运营、会话入口、设置页等用户可见 UI。
2. FR-ZH-02 术语白名单机制
   1. 允许保留常用简短英文术语（白名单），其余 UI 文案中文化。
3. FR-ZH-03 术语映射表
   1. 维护 `ui_zh_copy_map.md`，记录“旧文案 -> 新文案”。
4. FR-ZH-04 扫描门禁
   1. 输出 `ui_zh_scan_report.txt`，标记残留英文并区分白名单/非白名单。
5. FR-ZH-05 协议兼容
   1. 不修改 API 路径、字段名、错误码语义，仅改展示层文案。

## 5. 非功能需求（性能、安全、可用性、合规等）
1. 一致性：
   1. 同一术语全站统一（如“预发布”“已发布”“发布说明”）。
2. 可维护性：
   1. 新增页面必须复用同一术语映射与白名单规则。
3. 可追溯性：
   1. 每轮中文化改动都必须提交映射表和扫描报告。

## 6. 验收标准（Given/When/Then）
1. AC-ZH-01 工牌中文化
   1. Given 工牌页面
   2. When 打开概览与详情
   3. Then 主标题、状态、动作文案均为中文（白名单除外）
2. AC-ZH-02 发布管理中文化
   1. Given 发布列表与详情页
   2. When 查看版本信息
   3. Then 版本状态、发布时间、发布说明均为中文
3. AC-ZH-03 白名单生效
   1. Given 白名单术语
   2. When 执行扫描
   3. Then 白名单术语不计为违规，非白名单英文被准确标记
4. AC-ZH-04 协议不变
   1. Given 中文化改造完成
   2. When 执行 UO/AR
   3. Then 脚本退出码均为 `0`
5. AC-ZH-05 证据完备
   1. Given 本轮交付
   2. When 审核证据
   3. Then 包含映射表、扫描报告、功能截图、回归输出

## 7. 边界条件与异常处理
1. 品牌与产品名：
   1. `workflow` 允许保留英文品牌写法。
2. 技术缩写：
   1. 白名单术语允许英文保留（如 `API`、`ID`、`URL`）。
3. 若术语存在歧义：
   1. 先入“待确认词条”，不强行替换。

## 8. 依赖项与开放问题
1. 依赖项：
   1. 前端与模板可稳定回归，UO/AR 脚本可复跑。
2. 开放问题：
   1. 白名单是否继续扩大到更多工程术语（当前先纳入常用集合，后续按评审扩展）。
3. 推断/假设（置信度: 中）：
   1. 首批白名单建议：`workflow`、`AGENT`、`project`、`repo`、`branch`、`commit`、`tag`、`PR`、`CI/CD`、`API`、`ID`、`URL`、`UI`、`CLI`、`JSON`、`SQL`、`Git`、`CPU`。
