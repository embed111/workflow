# 详细设计-代码体积治理与无损重构

## 1. 设计目标
1. 将超大文件拆分为“可独立理解、可独立测试、可独立回退”的模块。
2. 保持运行入口、接口行为、验收门禁结果稳定。
3. 提供分阶段重构路线，避免一次性高风险改写。

## 2. 现状基线
1. 后端超大文件（事实）：
   1. `src/workflow_app/workflow_web_server.py` 约 `553 KB`。
2. 前端高复杂文件（事实）：
   1. `src/workflow_app/web_client/03_policy_confirm_and_interactions.js`
   2. `src/workflow_app/web_client/05_training_center_and_bootstrap.js`
   3. `src/workflow_app/web_client/00_app_state_and_utils.js`

## 3. 拆分原则
1. 行为冻结原则：
   1. 先复制行为再重构，不在同一提交混入新功能。
2. 接口稳定原则：
   1. URL、请求体、响应体、错误码保持兼容。
3. 渐进交付原则：
   1. 每阶段可独立验收与回退。

## 4. 后端模块化方案（建议）
1. 目录建议：
   1. `src/workflow_app/server/bootstrap.py`（启动与装配）
   2. `src/workflow_app/server/routes_chat.py`
   3. `src/workflow_app/server/routes_training.py`
   4. `src/workflow_app/server/routes_config.py`
   5. `src/workflow_app/server/views_html.py`
   6. `src/workflow_app/server/services_*.py`（状态、统计、查询）
2. 兼容入口：
   1. 保留 `workflow_web_server.py` 作为兼容门面，内部转发到新模块。
3. 阶段目标：
   1. Phase A：抽路由与服务函数。
   2. Phase B：抽页面模板与配置处理。
   3. Phase C：清理遗留耦合并补映射文档。

## 5. 前端模块化方案（建议）
1. 目录建议：
   1. `web_client/modules/session/`
   2. `web_client/modules/training_center/`
   3. `web_client/modules/settings/`
   4. `web_client/modules/shared/`
2. 保留公共契约：
   1. `state` 结构、`getJSON/postJSON`、关键渲染入口函数名保持兼容。
3. 加载策略：
   1. 先保留现有 bundle 入口，再分批迁移内部实现。

## 6. 回归与门禁策略
1. 基线冻结：
   1. 重构前先保存 UO/AR 验收结果作为对照。
2. 阶段门禁：
   1. 每个阶段结束至少跑一次 UO + AR。
3. 放行条件：
   1. 两类脚本均 `exit code 0`。
   2. 核心页面功能抽样无回退。

## 7. 交付物清单
1. 代码：
   1. 拆分后的模块文件。
   2. 兼容入口文件。
2. 文档：
   1. 旧职责 -> 新路径映射表。
   2. 阶段回归记录。

## 8. 验收映射
1. `FR-RF-01~FR-RF-03` -> 第 2~5 节。
2. `FR-RF-04~FR-RF-05` -> 第 6 节。
3. `FR-RF-06~FR-RF-08` -> 第 7~9 节。

## 9. 验收取证规范
1. 每条 AC 证据最小集合：
   1. 功能截图。
   2. 验收脚本输出摘要。
   3. 关键 API 示例。
   4. 代码路径。
2. 统计/门禁矩阵截图仅辅助，不替代功能截图。

## 10. 推断/假设与确认项
1. 推断/假设（置信度: 中）：
   1. 后端拆分收益最高，应优先于前端大规模重构。
2. 推断/假设（置信度: 中）：
   1. 保留兼容门面可显著降低联动改造风险。
3. 已确认：
   1. 本轮重构前提是“不影响当前工作效果”。
4. 待确认：
   1. 本轮目标是“先后端后前端”还是“后前端并行”。
