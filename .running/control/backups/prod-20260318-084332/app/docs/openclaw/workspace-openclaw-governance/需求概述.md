# 需求概述

## 1. 项目背景与问题定义
当前 `workflow` 工作区已有 `AGENTS.md`、`docs/`、`logs/`、`state/`、`src/`、`scripts/` 等产品与运行资产，但尚未具备 OpenClaw 风格的 `.codex/` 记忆层。与此同时，现有代码和文档已多处默认 `.codex/skills/` 是正式技能入口，实际目录结构与运行口径存在缺口。

## 2. 目标用户与核心场景
1. 目标用户：`workflow` 工作区内的 agent、owner、执行维护者。
2. 核心场景：
   - 启动时恢复身份、用户上下文和近期记忆；
   - 保持现有产品运行态与 agent 记忆层分离；
   - 对齐 `.codex/skills/` 的文档/代码口径。

## 3. 业务目标与成功指标
1. 业务目标：
   - 为 `workflow` 补齐 `.codex/` 能力层；
   - 在不破坏运行链路的前提下接通 OpenClaw 记忆能力。
2. 成功指标：
   - 顶层新增 `.codex/`，且其中包含核心记忆文档与目录；
   - `AGENTS.md` 已具备 OpenClaw 风格读取顺序；
   - 现有运行链路不受损。

## 4. 范围界定
### In Scope
1. `../workflow/.codex/` 结构补齐。
2. `../workflow/AGENTS.md` 的读取顺序与边界说明更新。
3. `state/`、`logs/`、`.codex/` 三层职责分工明确化。

### Out of Scope
1. `src/`、`scripts/`、`docs/workflow/` 的业务重构。
2. UI、接口或数据库实现改造。

## 5. 优先级与阶段规划
1. P0：补齐 `.codex/` 结构。
2. P1：更新 `AGENTS.md`。
3. P2：验证 `.codex/skills/` 口径与最小运行回归。

## 6. 风险、依赖与待确认事项
1. 风险：误把 `.codex` 记忆层与 `state/` 运行态混写。
2. 依赖：执行方具备新增目录与更新 `AGENTS.md` 的权限。
3. 待确认：`.codex/skills/` 当前先采用空目录还是最小说明文件；本轮默认空目录或最小说明均可。
