# 需求详情-workflow命名与去原型化治理

## 1. 主题目标与价值说明
1. 主题目标：
   1. 统一产品对外运行时名称为 `workflow`（或 `Workflow`），去除“训练闭环工作台”等阶段化命名。
   2. 去除代码与文件体系中的原型阶段标识（如 `Phase0`、`01_*`、`02_*`）。
2. 业务价值：
   1. 提升产品信任感与长期品牌一致性。
   2. 降低“原型感命名”对工程维护和跨团队协作的干扰。
   3. 建立长期可持续的命名治理规则。

## 2. 用户画像与使用场景
1. 用户画像 A：最终使用者
   1. 关注点：界面命名稳定、专业，不出现实验阶段词。
2. 用户画像 B：研发与维护者
   1. 关注点：文件命名可读、可定位，不依赖序号文件名。
3. 核心场景：
   1. 打开系统首屏，标题显示 `workflow`/`Workflow`。
   2. 查看代码目录，文件名基于领域职责而非编号。
   3. 迭代时不再把版本阶段词固化在运行时资产中。

## 3. 用户旅程或关键流程
1. 先建立命名字典（运行时名、模块名、文件名规则）。
2. 执行运行时文案替换（UI 标题、菜单、日志可见名称）。
3. 执行文件重命名迁移（去除 `Phase0`、`01_*`、`02_*`）。
4. 回归关键页面与脚本，确认兼容。
5. 输出命名映射与迁移记录文档。

## 4. 功能需求清单（编号）
1. FR-NG-01 运行时主名称统一
   1. 对外主名称统一为 `workflow`（可配置显示 `Workflow`）。
2. FR-NG-02 去阶段化 UI 命名
   1. UI 中不得出现 `Phase0`、`MVP` 等阶段标签作为常驻名称。
3. FR-NG-03 去编号文件命名
   1. 业务代码文件不再使用 `01_*`、`02_*` 作为长期命名。
4. FR-NG-04 领域职责命名
   1. 文件名采用“领域 + 职责”语义，如 `session_interactions`、`training_routes`。
5. FR-NG-05 阶段信息隔离
   1. 阶段词仅允许出现在里程碑文档，不进入运行时和核心模块名。
6. FR-NG-06 命名映射文档
   1. 输出旧名 -> 新名映射清单，便于追溯。
7. FR-NG-07 无损兼容
   1. 重命名后现有脚本、引用路径、构建流程保持可用。
8. FR-NG-08 证据语义门禁（沿用）
   1. 功能验收需功能截图，不接受仅统计截图。

## 5. 非功能需求（性能、安全、可用性、合规等）
1. 可维护性：
   1. 命名规则必须可重复执行，支持后续新增模块复用。
2. 一致性：
   1. 同一概念仅保留一个对外名称（`workflow`）。
3. 可迁移性：
   1. 命名变更需提供兼容策略，减少外部引用断裂风险。

## 6. 验收标准（Given/When/Then）
1. AC-NG-01 主名称统一
   1. Given 用户进入系统
   2. When 查看首屏标题与主导航
   3. Then 显示名称为 `workflow`（或 `Workflow`），不出现“训练闭环工作台”
2. AC-NG-02 UI 去阶段词
   1. Given 用户浏览主要页面
   2. When 检查可见文案
   3. Then 不出现 `Phase0` 作为常驻产品名
3. AC-NG-03 代码去编号命名
   1. Given 核心业务目录
   2. When 检查文件名
   3. Then 不再保留新增的 `01_*`/`02_*` 命名模式
4. AC-NG-04 兼容运行
   1. Given 完成命名迁移
   2. When 启动应用并执行关键路径
   3. Then 功能可用且无命名迁移导致的引用错误
5. AC-NG-05 门禁回归通过
   1. Given 完成命名治理改造
   2. When 执行 UO/AR 脚本
   3. Then 脚本退出码为 `0`
6. AC-NG-06 命名映射可追溯
   1. Given 交付结果
   2. When 查看文档
   3. Then 可检索旧名到新名完整映射
7. AC-NG-07 证据完备性
   1. Given 任一 AC 回传
   2. When 审核证据
   3. Then 至少包含功能截图、脚本输出、代码路径、映射文档路径

## 7. 边界条件与异常处理
1. 若重命名引发导入失败：
   1. 必须补兼容层或回退后重做迁移。
2. 若存在外部脚本硬编码旧名：
   1. 先补映射/别名，再逐步替换，避免一次性断裂。
3. 若当前重构回合冲突：
   1. 先完成结构重构，再执行命名治理回合。

## 8. 依赖项与开放问题
1. 依赖项：
   1. 当前工程化重构分支稳定可回归。
2. 开放问题：
   1. 展示名最终固定 `workflow` 还是 `Workflow`（建议 `Workflow`）。
3. 能力扩展评估（skill/agent）：
   1. 新增 agent 角色：`暂不采用`。
   2. 新增 skill：`暂不采用`（该任务可由命名规则文档+门禁执行）。
