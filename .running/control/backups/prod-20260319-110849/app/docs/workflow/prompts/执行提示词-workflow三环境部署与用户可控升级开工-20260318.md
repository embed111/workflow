请先阅读以下文件，并严格按文档开工，不要从当前实现反推需求，也不要沿用旧的“单机 A/B 在线升级自动切换”口径：

2. `docs/workflow/requirements/需求详情-workflow三环境部署与用户可控升级.md`

5. `docs/workflow/requirements/需求详情-任务中心产物交付与路径管理.md`

本轮目标只有一个：把 `workflow` 落成“开发 / 测试 / 正式”三环境部署与正式环境用户可控升级机制。

## 一、你必须按以下口径实现
1. `../workflow/.running/dev|test|prod` 都是部署副本。
2. `../workflow` 源码工作区负责开发，不直接作为正式运行目录。
3. `../workflow/run_workflow.bat` 默认只运行正式环境 `prod`。
4. 必须提供可指定目标环境的一键部署脚本，至少支持：
   1. `dev`
   2. `test`
   3. `prod`
5. `dev/test` 不允许与 `prod` 使用相同的任务运行路径。
6. `agent路径` 可以在三环境之间相同。
7. 正式发布门禁固定为：
   1. 先部署到 `test`
   2. 自动执行一批不消耗 token 的自动化测试
   3. 全部通过后自动生成正式升级候选
8. 正式环境用户在页面顶部看到悬浮置顶升级提示与升级按钮。
9. 若存在 `running` 任务，正式升级不得放行。
10. 正式升级允许页面短暂刷新和重连，但不得出现半切换状态。

## 二、本轮优先实现范围
1. 环境与部署
   1. 明确 `dev/test/prod` 运行根目录、端口、日志和进程标识。
   2. 提供一键部署脚本与环境启动入口。
   3. `run_workflow.bat` 默认只指向 `prod`。
2. 发布门禁
   1. `test` 环境部署完成后自动跑本地自动化测试。
   2. 这些自动化测试不得调用 token 消耗型执行链路。
   3. 自动化测试通过后自动生成正式升级候选记录。
3. 正式环境升级
   1. 正式页面顶部增加最小升级提示与升级按钮。
   2. 升级前检查 `running` 任务。
   3. 无 `running` 任务时执行正式切换。
   4. 切换失败必须回退到上一正式版本。

## 三、实现边界
1. 不要把这次需求扩成复杂集群、双机热备或微服务发布平台。
2. 不要把正式升级改成“自动强制切换”，升级时机仍由正式用户自己触发。
3. 本轮包含页面改动，但当前未补专门原型；页面落地时必须严格沿用现有视觉风格与信息架构，只做最小必要 UI，不得自行发散。
4. 若涉及任务记录、工作记录、索引层，不得破坏当前“用户数据只在指定路径持久化”的口径。

## 四、最低验收要求
1. 能证明 `run_workflow.bat` 默认启动 `prod`。
2. 能证明 `dev/test` 与 `prod` 任务运行路径冲突时被阻断。
3. 能证明 `test` 部署后自动执行不消耗 token 的自动化测试。
4. 能证明自动化测试通过后生成正式升级候选。
5. 能证明正式环境页面出现顶部悬浮升级入口。
6. 能证明存在 `running` 任务时升级被阻断。
7. 能证明无 `running` 任务时可升级，且页面允许短暂刷新/重连。
8. 能证明升级失败时自动回退到上一正式版本。
