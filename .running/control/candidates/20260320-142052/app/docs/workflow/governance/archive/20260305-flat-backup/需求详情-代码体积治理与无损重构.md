# 需求详情-代码体积治理与无损重构

## 1. 主题目标与价值说明
1. 主题目标：
   1. 对 `../workflow` 的超大文件进行职责拆分，降低阅读、修改和回归成本。
   2. 在“无行为回退”前提下建立可持续演进的代码结构。
2. 业务价值：
   1. 提升迭代速度与评审质量。
   2. 降低高风险文件单点故障概率。
   3. 为后续功能扩展留出稳定边界。

## 2. 用户画像与使用场景
1. 用户画像 A：`workflow` 研发者
   1. 关注点：单文件过大导致定位慢、改动冲突高。
2. 用户画像 B：验收者
   1. 关注点：重构后功能与验收门禁不能退化。
3. 核心场景：
   1. 修改会话接口时不必打开 1 个超大后端文件。
   2. 修改训练中心前端逻辑时只进入对应模块文件。
   3. 重构后仍能通过既有 UO/AR 验收。

## 3. 用户旅程或关键流程
1. 先建立“当前行为基线”（执行现有验收脚本并记录）。
2. 按模块拆分超大文件，保留兼容入口。
3. 每次拆分后执行回归脚本，确认无行为回退。
4. 完成目标拆分后输出映射文档（旧路径 -> 新路径）。

## 4. 功能需求清单（编号）
1. FR-RF-01 拆分范围优先级
   1. 优先拆分后端 `workflow_web_server.py`。
   2. 次优先拆分前端大文件（会话交互/训练中心/状态工具）。
2. FR-RF-02 后端按职责拆分
   1. 路由层、配置层、会话层、训练中心层、页面模板层独立模块化。
   2. 保留兼容启动入口，避免运行命令变更。
3. FR-RF-03 前端按职责拆分
   1. 保持现有加载顺序与公共状态对象契约不变。
   2. 拆分后禁止出现跨模块循环依赖。
4. FR-RF-04 行为无损门禁
   1. 重构前后关键 API 响应字段、错误码、页面交互语义保持一致。
5. FR-RF-05 验收脚本回归
   1. 至少执行并通过现有 `UO` 与 `AR` 验收脚本。
6. FR-RF-06 拆分映射文档
   1. 必须提供“旧文件职责 -> 新文件路径”的映射清单。
7. FR-RF-07 重构节奏控制
   1. 按阶段提交，不允许单次跨域大规模改动。
8. FR-RF-08 证据语义门禁（沿用）
   1. 功能级验收必须有功能截图，统计图仅辅助。

## 5. 非功能需求（性能、安全、可用性、合规等）
1. 可维护性：
   1. 重构后单文件建议控制在 `<= 800` 行（生成文件除外）。
2. 可靠性：
   1. 重构后服务启动命令、默认端口、配置读取行为不变。
3. 可观测性：
   1. 关键日志与错误码不应因重构丢失。
4. 性能：
   1. 本轮不要求性能提升，但不得出现明显性能退化。

## 6. 验收标准（Given/When/Then）
1. AC-RF-01 后端大文件拆分达标
   1. Given 当前后端存在超大文件
   2. When 完成本轮重构
   3. Then `workflow_web_server.py` 仅保留启动与聚合入口，核心逻辑迁移到子模块
2. AC-RF-02 前端大文件拆分达标
   1. Given 当前前端存在多个 700+ 行文件
   2. When 完成本轮重构
   3. Then 关键模块按职责拆分，页面可正常启动并工作
3. AC-RF-03 API 行为一致
   1. Given 重构前后同一请求样例
   2. When 对比响应
   3. Then 关键字段与错误码一致
4. AC-RF-04 UO 回归通过
   1. Given 重构后代码
   2. When 执行 `run_acceptance_training_center_uo.py`
   3. Then 脚本退出码为 `0`
5. AC-RF-05 AR 回归通过
   1. Given 重构后代码
   2. When 执行 `run_acceptance_agent_release_ar.py`
   3. Then 脚本退出码为 `0`
6. AC-RF-06 结构映射可追溯
   1. Given 重构交付
   2. When 查看文档
   3. Then 可定位旧职责到新文件
7. AC-RF-07 证据完备性
   1. Given 任一 AC 回传
   2. When 审核证据
   3. Then 至少包含功能截图、API/日志、代码路径；统计截图不替代功能截图

## 7. 边界条件与异常处理
1. 若某模块拆分后出现回归：
   1. 必须支持回退到上一个稳定阶段，不继续扩散改动。
2. 若拆分导致加载顺序问题：
   1. 优先恢复兼容入口，再逐步分层。
3. 若周期内无法覆盖全部大文件：
   1. 允许分期交付，但必须先落地最高风险文件拆分。

## 8. 依赖项与开放问题
1. 依赖项：
   1. `../workflow` 现有验收脚本稳定可复跑。
   2. 当前运行与部署命令可保持不变。
2. 开放问题：
   1. 前端拆分颗粒度是按“页面域”还是按“能力域”优先（建议先按能力域）。
3. 能力扩展评估（skill/agent）：
   1. 新增 agent 角色：`暂不采用`（当前核心是工程结构治理，不是角色分工问题）。
   2. 新增 skill：`暂不采用`（本轮先用需求/设计+门禁脚本控制风险）。
