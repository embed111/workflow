# 详细设计-全站中文化与术语白名单

## 1. 设计目标
1. 在不改动后端协议的前提下完成全站中文化。
2. 用白名单机制保留必要英文术语，避免过度本地化损伤可读性。
3. 把中文化能力纳入可重复执行的门禁流程。

## 2. 术语治理模型
1. 词典层：
   1. `ui_zh_copy_map.md`：维护“旧文案 -> 新文案 -> 页面路径”。
2. 白名单层：
   1. `ui_zh_whitelist.txt`（或等效配置）：维护允许保留英文术语。
3. 扫描层：
   1. `ui_zh_scan_report.txt`：输出英文残留项与白名单命中情况。

## 3. 改造范围
1. 页面范围：
   1. 工牌概览/详情。
   2. 发布管理与版本切换。
   3. 训练运营关键面板。
   4. 会话入口与设置页。
2. 代码范围：
   1. 前端 JS、模板 HTML、可见提示文案。
3. 不改范围：
   1. API 字段名、错误码、数据库字段、脚本参数名。

## 4. 实施步骤
1. 步骤 A：基线抽取
   1. 抽取当前 UI 文案，生成基线词典。
2. 步骤 B：术语统一
   1. 批量替换非白名单英文 UI 文案为中文。
3. 步骤 C：白名单校验
   1. 扫描残留英文，输出 `ui_zh_scan_report.txt`。
4. 步骤 D：回归与证据
   1. 执行 UO/AR；
   2. 生成功能截图与证据清单。

## 5. 术语建议（首版）
1. 中文优先术语：
   1. `pre-release` -> `预发布`
   2. `release` -> `发布`
   3. `version` -> `版本`
   4. `queue` -> `队列`
   5. `trainer` -> `训练师`
2. 白名单建议（推断/假设，置信度: 中）：
   1. `workflow`
   2. `AGENT`
   3. `project`
   4. `repo`
   5. `branch`
   6. `commit`
   7. `tag`
   8. `PR`
   9. `CI/CD`
   10. `API`
   11. `ID`
   12. `URL`
   13. `UI`
   14. `CLI`
   15. `JSON`
   16. `SQL`
   17. `Git`
   18. `CPU`

## 6. 门禁设计
1. Gate-ZH-01 文案一致性门禁
   1. 同术语跨页面不允许多写法混用。
2. Gate-ZH-02 英文残留门禁
   1. 白名单外英文残留必须为 0。
3. Gate-ZH-03 兼容门禁
   1. UO/AR 均通过，且关键 API 输出字段不变。

## 7. 交付物
1. `ui_zh_copy_map.md`
2. `ui_zh_whitelist.txt`（或等效配置）
3. `ui_zh_scan_report.txt`
4. 中文化前后功能截图（工牌/发布/训练运营）
5. 回归结果摘要（UO/AR）

## 8. 风险与缓解
1. 风险：误改技术字段导致兼容性问题。
   1. 缓解：限定只改展示层，脚本回归校验字段不变。
2. 风险：中文化后语义不统一。
   1. 缓解：先词典后替换，强制扫描门禁。
3. 风险：白名单过大导致英文残留泛化。
   1. 缓解：白名单最小化，逐轮评审扩展。
