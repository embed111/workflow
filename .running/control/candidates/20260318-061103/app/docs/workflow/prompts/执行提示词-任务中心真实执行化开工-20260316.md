## 2. 本次任务性质
1. 本次不是继续补静态原型，而是把 `任务中心` 从“流程流转”升级为“真实执行”。
2. 当前执行工作区必须以 `2026-03-16` 最新真实执行化口径为准，不再按旧的“`running` 只是等待人工回写”实现。
3. 第一版真实执行通道只有 `Codex`，但接口层必须预留后续扩展能力。

## 3. 本次必须完成的范围
1. 后端必须抽象统一的任务执行 provider 接口。
2. 第一版必须实现 `Codex` provider，并确保 `dispatch-next` 会真正触发执行，而不是只改业务状态。
3. `running` 状态必须代表真实存在的执行批次。
4. 每次真实执行至少要有稳定的 `run_id`，并关联：
   1. `ticket_id`
   2. `node_id`
   3. `provider`
   4. `workspace_path`
   5. `prompt_ref`
   6. `stdout_ref`
   7. `stderr_ref`
   8. `result_ref`
5. `设置` 页必须新增最小 `任务执行` 区块，至少包含：
   1. `执行通道`
   2. `Codex 调用路径`
   3. `命令模板`
   4. `全局并发上限`
6. 任务详情必须新增 `执行链路` 卡片，至少展示：
   1. `执行通道`
   2. `运行批次`
   3. `最新事件`
   4. `目标工作区`
   5. `调用命令`
   6. `完整提示词`
   7. `执行过程`
7. 完整提示词必须可见，不能只给摘要。
8. 运行过程必须尽量实时可见：
   1. 优先 `SSE/WebSocket`
   2. 若当前工程不适合，首版允许 `<= 1 秒` 轮询
   3. 但不能等任务结束后才一次性补全
9. 运行输出至少要覆盖：
   1. 调度事件
   2. 命令摘要
   3. stdout
   4. stderr
   5. 最终结果
10. 成功后必须自动回写任务结果与产物引用。
11. 失败后必须自动标记失败并阻塞下游。
12. `重跑任务 / 人工修改执行状态` 继续保留在 `任务管理` 中，作为失败补救入口。
13. 产物交付、`workspace/` 任务记录、删除/清空留痕等既有规则继续保持，不得因真实执行化而回退。

## 4. 明确禁止
1. 不要继续把 `running` 实现成“已派发但未真正执行”。
2. 不要把完整提示词和执行过程塞进 `设置` 页。
3. 不要只显示“执行中/执行成功”摘要，而隐藏 stdout/stderr 与提示词主体。
4. 不要绕过 provider 抽象，直接把 `Codex` 命令散落写死在多个页面或控制器里。
5. 不要因为第一版只有 `Codex`，就把 provider 切换能力做死。
6. 不要破坏现有的产物交付、删除桥接、并发门禁和暂停语义。

## 5. 推荐实现顺序
1. 先补后端 provider 抽象与 `Codex` provider。
2. 再补 `dispatch-next -> 创建 run_id -> 真实启动执行` 主链路。
3. 再补运行批次记录与 stdout/stderr/结果留痕。
4. 再补 `设置` 页最小任务执行配置。
5. 再补任务详情 `执行链路` 卡片。
6. 最后联调：
   1. 成功自动回写
   2. 失败阻塞下游
   3. `pause / resume`
   4. `rerun / override-status`
   5. 产物交付与 `workspace/` 记录不回退

   2. 至少一个真实 `run_id`

3. 代码落点：
   1. provider 抽象位置
   2. `Codex` provider 实现位置
   3. 任务详情执行链路 UI 位置
   4. 设置页任务执行配置位置
4. 行为验证：
   1. 成功后自动回写证据
   2. 失败后阻塞下游证据
   3. `rerun` 产生新的 `run_id`
5. 风险与未完成项：

   2. 若 stdout/stderr 先做折叠或分段加载，也必须明确写出。
