# 需求详情-agent发布管理与版本切换

## 1. 主题目标与价值说明
1. 主题目标：
   1. 让训练中心成为“所有 agent 的统一发布管理入口”。
   2. 提供“已发布版本之间切换”能力，并与训练门禁联动。
   3. 建立“训练 -> 预发布 -> 发布评审 -> 人工审核 -> 确认发布”的标准生命周期。
   4. 让发布报告同时承担“发布评审依据 + 角色详情公开介绍”双重职责。
2. 业务价值：
   1. 统一发布口径，减少多入口造成的版本状态分裂。
   2. 保持主线可用版本稳定，把试错沉淀到预发布阶段。
   3. 通过“可切换、可克隆、可舍弃、可评审”降低训练带来的回滚成本。
   4. 将正式发布前的判断依据沉淀为结构化发布报告，减少口头评估与主观遗漏。
   5. 让角色详情查看者能通过最新正式发布报告快速理解 agent 当前完整能力，而不是只看到版本差异。

## 2. 用户画像与使用场景
1. 用户画像 A：平台治理者
   1. 关注点：所有 agent 的版本状态一致、可控、可恢复。
2. 用户画像 B：训练运营者
   1. 关注点：切换后门禁规则明确，训练动作不误伤生产角色。
3. 用户画像 C：业务使用者
   1. 关注点：训练后的能力先在预发布阶段验证，再决定发布。
4. 用户画像 D：发布审核人
   1. 关注点：发布前必须先看到由对应工作区 agent 生成的发布报告，再做最终人工确认。
5. 用户画像 E：角色详情查看者
   1. 关注点：打开角色详情时，希望直接看到该 agent 当前正式发布版本完整能做什么。
5. 核心场景：
   1. 在训练中心查看全部 agent 的已发布版本时间线。
   2. 将某个 agent 切换到已发布旧版本（不允许中间 commit）。
   3. 切换后原 agent 禁训；如需继续训练，克隆新角色基于当前版本训练。
   4. 训练结束后进入预发布态，用户点击进入发布评审。
   5. 系统委派对应工作区 agent 通过 Codex 生成发布报告，并展示完整分析链路。
   6. 人工审核发布报告后，再决定是否确认发布。
   7. 对单个 agent 舍弃预发布内容，直接回到发布态。

## 3. 用户旅程或关键流程
1. 用户进入训练中心，查看 agent 发布状态总览。
2. 选择 agent，查看已发布版本时间线（`tag + release note`）。
3. 发起版本切换（目标必须是已发布版本），系统直接覆盖该 agent 工作区。
4. 切换完成后原 agent 标记为“禁训”，直到切回最新发布版本才解除禁训。
5. 若业务需要基于当前版本继续训练，用户克隆新角色并在克隆角色上训练。
6. 克隆角色训练后进入预发布态，进入真实使用观察期。
7. 观察期结束，用户点击“进入发布评审”。
8. 系统基于该 agent 对应工作区，委派对应 agent 通过 `codex exec` 生成发布报告。
9. 发布报告正文使用 agent 第一人称视角，同时展示“当前正式发布目标版本的全量能力集 + 本次版本增量 + 风险与证据”。
10. 发布审核人查看报告正文、分析链路、风险与建议结论。
11. 人工审核完成后执行决策：
   1. 通过：系统进入“确认发布”执行阶段，使用 Git + release note 完成正式发布，并校验是否发布成功；
      1. 正式发布成功后，将本次发布报告绑定为该版本的角色详情介绍来源；
   2. 若发布执行失败：自动启动 `../workflow` 工作区的 agent 进行兜底，并输出兜底结果；
   3. 不通过：继续训练或舍弃预发布内容（单 agent）。

## 4. 功能需求清单（编号）
1. FR-AR-01 统一纳管范围
   1. 训练中心必须纳入 `agent_search_root` 下全部 agent 的发布管理。
   2. 后续新增 agent 自动进入纳管范围。
2. FR-AR-02 已发布版本视图
   1. 仅展示已发布版本时间线与发布说明（`tag + release note`）。
   2. 每个已发布版本必须绑定独立版本上下文（至少含 `release_id/version_label/released_at`），用于版本切换与历史报告查看。
   3. 若某已发布版本存在可追溯的发布报告或能力快照，页面必须提供“查看发布报告”入口。
   4. 用户切换不同已发布版本后，“查看发布报告”必须打开该版本对应的发布评审记录/发布报告，不得误展示当前其他版本或当前预发布评审内容。
   5. 若某已发布版本未绑定可展示的发布报告，页面必须明确提示“当前版本未绑定可展示的发布报告文件”，且不得用当前活跃评审内容冒充该历史版本报告。
   6. 不展示中间 commit 明细。
3. FR-AR-03 版本切换能力
   1. 仅允许切换到已发布版本。
   2. 切换执行方式固定为“直接覆盖工作区”。
4. FR-AR-04 切换后训练门禁
   1. 原 agent 执行版本切换后，进入禁训状态。
   2. 仅当原 agent 切回“最新发布版本”时，才恢复可训练状态。
5. FR-AR-05 克隆训练分支
   1. 支持从当前版本克隆新角色（新 `agent_id`）。
   2. 克隆角色纳入统一发布管理。
   3. 克隆角色允许继续训练。
6. FR-AR-06 预发布状态定义
   1. 发布版本之外的最新修改统一定义为“预发布内容”。
   2. 训练完成后默认进入预发布态。
7. FR-AR-07 预发布舍弃
   1. 支持对单个 agent 执行“舍弃预发布内容”。
   2. 舍弃后该 agent 必须回到发布态。
   3. 不支持批量舍弃。
8. FR-AR-08 发布评审入口
   1. 仅当 `agent` 处于预发布态时，页面才显示“进入发布评审”入口。
   2. 点击后必须创建一条与该 `agent_id` 绑定的发布评审记录。
   3. 角色详情首页只展示“当前活跃评审”。
   4. “当前活跃评审”定义为：当前 agent 最近一条尚未因“确认发布成功”或“显式废弃”而结束的发布评审记录。
   5. 已完成并进入发布历史的评审记录，不得继续常驻在首页“发布评审”区块。
9. FR-AR-09 发布报告委派生成
   1. 进入发布评审后，系统必须委派该 agent 对应工作区的 agent 执行发布报告生成任务。
   2. 报告生成主链路固定使用 `Codex`，执行方式参考“会话入口 agent 分析功能”的 `codex exec + 分析链路可见 + 证据落盘` 机制。
   3. 系统不得以本地静态规则替代发布报告生成主链路；Codex 失败时默认 `fail-closed`，不得允许直接确认发布。
   4. 发布报告自然语言正文统一采用 agent 第一人称视角，叙述风格固定为“我是/我能/我已完成/我建议”。
   5. 发布报告必须同时呈现“当前目标版本的全量能力清单”与“相对上一正式发布版本的能力增量”，不得只输出 `capability_delta`。
   6. 发布报告至少包含：
      1. `target_version/current_workspace_ref`
      2. `first_person_summary`
      3. `full_capability_inventory`
      4. `knowledge_scope`
      5. `agent_skills`
      6. `applicable_scenarios`
      7. `change_summary`
      8. `capability_delta`
      9. `risk_list`
      10. `validation_evidence`
      11. `release_recommendation`（`approve/reject/needs_more_validation`）
      12. `next_action_suggestion`
   7. 发布报告必须可被未参与研发的查看者直接用于理解“该 agent 当前正式发布版本完整能做什么”。
   8. 发布报告的主判断边界必须是“当前角色正式发布可理解性 + 相对上一正式发布版本差异 + 是否满足发布门禁”，不得退化为纯仓库卫生巡检报告。
   9. 若不存在上一正式发布版本或 `released_versions` 为空，系统必须按“首发基线评审”处理，仍需产出完整第一人称全量能力报告，不得仅因无历史发布而输出残缺结构。
   10. 生成 `first_person_summary/full_capability_inventory/knowledge_scope/agent_skills/applicable_scenarios` 时，必须优先结合当前工作区 `AGENTS.md`、本地 skills、角色画像已知字段、可读需求/说明文档与版本上下文补齐；信息不足时写入 `warnings`，不得直接省略字段。
   11. 对 Git 状态、diff、未提交改动的风险判断默认限定在目标工作区路径范围内；工作区外噪声只允许作为辅助告警，不得成为当前 agent 发布建议的主依据。
   12. `README/CHANGELOG/release note` 缺失默认记为“可追溯性不足警告”；除非产品已显式将其定义为发布硬门禁，否则不得仅因这些文件缺失直接输出 `reject`。
   13. `release_recommendation` 语义固定为发布建议，不得复用人工审核决策枚举；人工审核仍单独使用 `approve_publish/reject_continue_training/reject_discard_pre_release`。
   14. 若输入上下文中的版本元数据自相矛盾（如 `target_version/current_registry_version/latest_release_version/released_versions` 冲突），系统应优先返回结构化元数据冲突说明，并阻断或降级本次评审，而不是把该矛盾伪装成 agent 能力风险。
10. FR-AR-10 人工审核发布报告
   1. 发布审核人必须先看到发布报告与分析链路，再提交审核结论。
   2. 审核结论至少支持：
      1. `approve_publish`
      2. `reject_continue_training`
      3. `reject_discard_pre_release`
   3. 审核意见必须绑定 `reviewer/review_comment/reviewed_at`。
11. FR-AR-11 确认发布执行
   1. 仅当发布报告已生成且人工审核结论为 `approve_publish` 时，才允许点击“确认发布”。
   2. “确认发布”必须以 Git 与 release note 作为正式发布事实来源，不得只写数据库状态。
   3. 发布执行完成后，系统必须按当前版本识别规则重新读取 Git + release note 元数据，校验是否已形成新的正式发布记录。
   4. 仅当成功校验通过时，系统才可生成正式发布记录，并将该 agent 生命周期状态切回发布态。
   5. 未完成人工审核时，不得直接从预发布态进入正式发布完成。
   6. 发布全过程必须记录日志，至少覆盖：
      1. 发布准备；
      2. Git 执行；
      3. release note 写入/关联；
      4. 发布成功校验；
      5. 失败后的兜底触发与结果。
   7. 若 Git 发布失败、release note 写入失败或发布成功校验失败，系统必须标记“确认发布失败”。
   8. “确认发布失败”后，系统必须自动启动 `../workflow` 工作区的 agent 进行兜底，且兜底过程与结果也必须留痕。
   9. 兜底最小职责固定为：
      1. 输出本次发布失败原因；
      2. 自动重试一次发布；
      3. 输出重试结果与下一步建议。
   10. 正式发布成功后，系统必须将本次发布报告绑定为该版本的 `latest_released_role_profile` 内容来源，供角色详情页优先展示。
12. FR-AR-12 发布评审机制扩展位
   1. 预留评审机制接入点，后续可支持自动训练与自动发布。
13. FR-AR-13 agent-first 基础单元约束
   1. 对话与后续新增功能必须以 `agent_id` 作为基础单元关联键。
   2. 发布、训练、评估、会话状态需可按 agent 单元汇聚展示。

## 5. 非功能需求（性能、安全、可用性、合规等）
1. 性能：
   1. 单 agent 版本切换操作反馈 `p95 <= 5s`（不含大型仓库极端场景）。
   2. 单 agent 预发布舍弃操作反馈 `p95 <= 5s`。
2. 可用性：
   1. 切换失败时不得破坏当前已生效发布态。
   2. 舍弃失败时必须保持原状态不变并返回明确错误信息。
   3. 发布报告生成中，页面必须持续展示当前阶段（排队中/执行中/已完成/失败），不得出现“静默等待”。
   4. 确认发布执行中，页面必须持续展示“发布中/校验中/失败兜底中/完成”状态，不得静默跳转。
   5. 角色详情页读取最新正式发布报告时，若存在绑定报告，应在首屏优先展示第一人称摘要与全量能力清单。
   6. 首页“发布评审”区块与“发布版本 -> 查看发布报告”弹窗必须职责分离：
      1. 首页只承载当前活跃评审；
      2. 历史正式发布版本只通过“查看发布报告”按需下钻。
3. 一致性：
   1. 任一时刻，一个 agent 仅允许一个明确生命周期状态：`发布态|预发布态|禁训态`。
   2. 训练门禁判断与状态展示必须一致，不允许“页面可训但后端拒绝”。
   3. “发布报告状态、人工审核状态、确认发布可点击状态”三者必须口径一致，不允许页面已可确认但后端仍拒绝。
   4. 确认发布是否成功，必须以“重新读取后的 Git + release note 识别结果”为准，不允许仅依赖前端按钮返回成功。
   5. 发布报告失败提示必须优先展示后端真实失败原因；仅当后端明确判定为“结构化报告不完整”时，前端才允许显示“缺字段”文案，且缺字段列表必须与后端门禁保持一致。
4. 可追溯性：
   1. 发布评审必须可追溯到提示词、Codex 执行命令摘要、stdout/stderr、结构化报告、人工审核结论。
   2. 确认发布必须可追溯到 Git 命令摘要、release note 引用、成功校验结果、失败兜底记录。
4. 治理边界：
   1. 当前阶段不要求自动发布。
   2. 当前阶段仍保持“人工审核 + 人工确认发布”为最终门禁。

## 6. 验收标准（Given/When/Then）
1. AC-AR-01 全量纳管
   1. Given `agent_search_root` 下存在多个 agent
   2. When 打开训练中心发布管理视图
   3. Then 可看到全部 agent 的发布管理条目
2. AC-AR-02 已发布版本可见性
   1. Given 某 agent 有多个发布版本
   2. When 查看版本时间线
   3. Then 仅显示已发布版本与 release note，不显示中间 commit
3. AC-AR-03 切换范围限制
   1. Given 用户选择一个非发布 commit
   2. When 尝试版本切换
   3. Then 系统拒绝并提示“仅支持已发布版本”
4. AC-AR-04 切换后禁训
   1. Given 原 agent 已切换到非最新发布版本
   2. When 用户尝试发起训练
   3. Then 系统拒绝并标记该 agent 为禁训态
5. AC-AR-05 切回最新恢复训练
   1. Given 原 agent 当前为禁训态
   2. When 用户将其切回最新发布版本
   3. Then 系统恢复可训练状态
6. AC-AR-06 克隆训练
   1. Given 原 agent 当前禁训
   2. When 用户克隆新角色并发起训练
   3. Then 克隆角色可正常训练且纳入发布管理
7. AC-AR-07 训练后预发布
   1. Given 某 agent 完成训练
   2. When 训练任务回写完成
   3. Then 该 agent 状态变为预发布态
8. AC-AR-08 单 agent 舍弃预发布
   1. Given 某 agent 处于预发布态
   2. When 用户执行舍弃预发布内容
   3. Then 该 agent 回到发布态，且仅影响该单个 agent
9. AC-AR-09 进入发布评审
   1. Given 某 agent 处于预发布态且经过真实使用观察
   2. When 用户点击“进入发布评审”
   3. Then 系统创建该 agent 的发布评审记录，并进入“报告生成中”状态
10. AC-AR-10 委派生成发布报告
   1. Given 某 agent 已进入发布评审
   2. When 系统发起发布报告任务
   3. Then 必须使用与该 agent 对应工作区绑定的 agent 通过 `codex exec` 生成发布报告
   4. And 页面可查看提示词、Codex 执行摘要、stdout/stderr、结构化报告路径
   5. And 报告自然语言正文统一使用第一人称视角
   6. And 报告同时包含“全量能力清单”与 `capability_delta`
11. AC-AR-11 人工审核发布报告
   1. Given 发布报告已生成完成
   2. When 发布审核人提交审核结论
   3. Then 系统记录 `reviewer/review_comment/reviewed_at/review_decision`
   4. And 若结论不是 `approve_publish`，则“确认发布”按钮保持禁用
12. AC-AR-12 确认发布
   1. Given 发布报告已生成且人工审核结论为 `approve_publish`
   2. When 用户点击“确认发布”
   3. Then 系统必须执行 Git 发布动作并关联 release note
   4. And 系统必须按当前“Git + release note”识别规则重新校验发布是否成功
   5. And 仅在校验通过后才生成正式发布记录，并将该 agent 状态切回发布态
   6. And 发布成功后该报告被绑定为角色详情页的最新正式发布介绍来源
13. AC-AR-13 确认发布日志完备
   1. Given 任意一次确认发布执行
   2. When 用户查看发布评审详情
   3. Then 可查看发布准备、Git 执行、release note 处理、成功校验、兜底触发与结果等关键日志
14. AC-AR-14 确认发布失败自动兜底
   1. Given Git 发布失败、release note 写入失败或发布成功校验失败
   2. When 系统判定本次确认发布失败
   3. Then 系统必须自动启动 `../workflow` 工作区的 agent 进行兜底
   4. And 兜底输出必须包含失败原因，并自动重试一次发布
   5. And 发布状态保持未成功，直到再次通过 Git + release note 成功校验
15. AC-AR-15 发布报告失败阻断
   1. Given Codex 执行失败、超时或输出不符合发布报告契约
   2. When 用户查看发布评审详情
   3. Then 系统明确标记为失败并阻断“确认发布”，仅允许重试、继续训练或舍弃预发布
16. AC-AR-16 验收证据完备性
   1. Given 任一 AC 进入验收
   2. When 提交验收材料
   3. Then 每条 AC 至少包含截图、API 请求/响应、关键日志或数据库记录路径、对应代码路径
   4. And `AC-AR-03/04/06/08/10/12/14` 必须提供录屏或等效 GIF
   5. And 门禁统计表或汇总矩阵截图仅作辅助，不计入“该 AC 对应功能截图”
   6. And 验收通过并完成复核后，可执行证据精简，仅保留门禁截图与汇总文件
17. AC-AR-17 首页仅展示当前活跃评审
   1. Given 某 agent 同时存在至少一个正式发布版本，且当前有一条未结束的发布评审记录
   2. When 用户打开该 agent 角色详情页
   3. Then 首页“发布评审”区块只展示当前活跃评审
   4. And 历史已发布版本对应的旧评审记录不得常驻展示在该区块
18. AC-AR-18 无活跃评审时不常驻展示评审内容
   1. Given 某 agent 当前不存在活跃评审记录
   2. When 用户打开该 agent 角色详情页
   3. Then 首页不得常驻展示完整发布评审内容
   4. And 页面仍保留“发布版本”历史列表供查看
19. AC-AR-19 查看指定版本的历史发布报告
   1. Given 某 agent 存在多个已发布版本，且其中至少一个版本已绑定可展示发布报告
   2. When 用户切换到某个已发布版本并点击“查看发布报告”
   3. Then 页面以弹窗展示该版本对应的发布报告/发布评审记录
   4. And 弹窗内容必须与该版本的 `release_id/version_label` 一一对应
   5. And 弹窗不得误展示当前其他版本或当前预发布评审的内容
20. AC-AR-20 历史版本未绑定报告时的提示口径
   1. Given 某已发布版本未绑定可展示的发布报告
   2. When 用户查看该版本所在的“发布版本”列表
   3. Then 页面明确展示“当前版本未绑定可展示的发布报告文件”
   4. And 不展示“查看发布报告”按钮

## 7. 边界条件与异常处理
1. 无可用发布版本：
   1. 禁止版本切换，提示先完成首次发布。
2. 目标版本已是当前版本：
   1. 允许幂等返回，不重复执行覆盖动作。
3. 克隆名称冲突：
   1. 阻断克隆并提示重命名规则。
4. 舍弃预发布时无预发布内容：
   1. 返回幂等成功，不改变发布态。
5. 无法找到对应工作区 agent：
   1. 阻断发布报告生成，返回明确错误并提示修复映射关系。
6. Codex 执行失败或超时：
   1. 发布评审进入失败态，不得直接跳过报告进入人工确认。
7. 结构化报告字段不完整：
   1. 必须显式区分“Codex 未产出报告”“Codex 产出旧结构”“服务端补齐后仍缺关键字段”三类原因。
   2. 失败态仍应保留可查看的结构化失败报告骨架与 `raw_result/warnings`，不得只回传空对象。
8. Git 发布失败或 release note 写入失败：
   1. 本次确认发布必须标记失败，并触发 `../workflow` 工作区 agent 兜底。
9. 发布成功校验未通过：
   1. 不得将状态标记为已发布，必须保持未成功并触发兜底。
10. `../workflow` 工作区 agent 启动失败：
   1. 必须记录兜底启动失败日志，并提示人工介入，不得伪装为已兜底成功。
11. 人工评估未完成：
   1. 预发布态不得直接进入“正式发布完成”。

## 8. 依赖项与开放问题
1. 依赖项：
   1. agent 工作区可读取发布版本元数据（`tag + release note`）。
   2. 训练中心已具备按 agent 维度展示与操作能力。
   3. 训练执行回写可触发状态更新（训练后置为预发布态）。
   4. 系统已具备“agent -> 工作区 -> Codex 执行上下文”的稳定映射能力。
   5. 可复用会话入口 `Codex` 分析链路展示与证据落盘能力。
   6. 目标 agent 工作区具备可执行 Git 发布与写入/关联 release note 的运行权限。
   7. `../workflow` 工作区 agent 可被稳定启动为失败兜底执行器。
   8. 角色详情页可读取最新正式发布版本绑定的发布报告或其结构化快照。
2. 开放问题：
   1. “对应工作区 agent” 的映射规则是否固定为 `agent_id -> workspace_path -> AGENTS.md`，待实现侧最终确认。
   2. 自动评估机制引入时的最低门槛指标（会话样本量、失败率等）待后续专题定义。
   3. 历史正式发布版本若无绑定发布报告，是否需要离线补录第一人称角色述职报告，待后续专题定义。
3. 能力扩展评估（skill/agent）：
   1. 新增 skill：`agent-release-lifecycle-gate`
      1. 触发理由：需长期校验“切换范围、禁训门禁、发布报告生成、人工审核、确认发布”规则一致性。
      2. 采用成本：中。
      3. 采用决策：`采用`（后续实现阶段创建）。
   2. 新增 agent 角色：专职“发布评估 agent”
      1. 触发理由：若未来“对应工作区 agent”无法稳定承担发布报告生成职责，需引入独立评审 agent。
      2. 采用成本：中到高。
      3. 采用决策：`暂不采用`（当前阶段直接复用对应工作区 agent + 人工审核）。
