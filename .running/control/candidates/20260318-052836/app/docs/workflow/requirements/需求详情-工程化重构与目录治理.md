# 需求详情-工程化重构与目录治理

## 1. 主题目标与价值说明
1. 主题目标：
   1. 将当前“单文件聚合服务”升级为工程化分层架构。
   2. 同步完成目录治理，形成可持续扩展的代码组织方式。
2. 业务价值：
   1. 降低高风险单文件改动冲突与回归成本。
   2. 提高开发协作效率与代码可审阅性。
   3. 为后续功能演进提供稳定模块边界。

## 2. 用户画像与使用场景
1. 用户画像 A：`workflow` 工程开发者
   1. 关注点：功能定位快、改动范围可控、回归成本低。
2. 用户画像 B：验收与治理角色
   1. 关注点：重构不破坏现有行为，证据可追溯。
3. 核心场景：
   1. 只改某个 API 域时，不需要打开 1 个超大后端文件。
   2. 只改会话前端域时，不影响训练中心域代码。
   3. 重构后仍可通过既有 AR/UO 门禁。

## 3. 用户旅程或关键流程
1. 冻结行为基线（执行 AR/UO 并留存）。
2. 执行后端工程化拆分（路由、服务、基础设施、展示层）。
3. 执行前端工程化拆分（会话、训练、设置、共享模块）。
4. 阶段结束即执行回归，失败则阶段回滚。
5. 输出工程化目录映射与职责说明文档。

## 4. 功能需求清单（编号）
1. FR-ER-01 后端分层架构落地
   1. 至少落地以下层次：
      1. `api`：路由分发与入参校验；
      2. `services`：业务逻辑；
      3. `infra`：配置、数据库、持久化、审计；
      4. `presentation`：HTML/CSS 模板输出；
      5. `bootstrap`：应用启动与装配。
2. FR-ER-02 路由拆分为域模块
   1. `do_GET/do_POST` 不再承载全量分支。
   2. 至少按域拆分：`config/chat/training/policy/dashboard`。
3. FR-ER-03 模板与样式外置
   1. 内联页面模板（HTML/CSS）从主服务文件迁移到展示层模块。
4. FR-ER-04 DB 与迁移逻辑解耦
   1. 建表/迁移/连接管理从主服务文件迁出到 `infra/db`。
5. FR-ER-05 Agent 发现与策略解析链路拆分
   1. 将 Agent 发现、策略解析等独立到服务模块。
6. FR-ER-06 兼容入口保留
   1. 保留当前启动入口文件作为兼容门面，避免运行命令漂移。
7. FR-ER-07 前端领域化拆分
   1. 至少形成：`session`、`training_center`、`settings`、`shared`。
   2. 公共状态契约保持兼容。
8. FR-ER-08 目录治理文档化
   1. 提供工程化目录树与“文件职责说明”。
9. FR-ER-09 阶段门禁与回滚
   1. 每阶段独立验收与回滚点，禁止一次性大爆炸改写。
10. FR-ER-10 证据语义门禁（沿用）
   1. 统计截图仅辅助，不可替代功能截图。
11. FR-ER-11 行数硬门禁与完成判定
   1. 本轮完成必须同时满足：
      1. `src/workflow_app/workflow_web_server.py <= 3000`；
      2. `src/workflow_app/server/api/legacy.py <= 1000`；
      3. 输出“Top 15 大文件行数（重构前 vs 重构后）”。
12. FR-ER-12 命名去原型化（本轮新增文件）
   1. 本轮新增/改名文件禁止出现 `Phase*`、`01_*`、`02_*` 等原型命名。
   2. 文件名必须体现“领域 + 职责”语义。

## 5. 非功能需求（性能、安全、可用性、合规等）
1. 可维护性：
   1. 重构后核心人工维护文件门禁：
      1. `workflow_web_server.py` 必须 `<= 3000` 行（本期硬门禁）。
      2. `server/api/legacy.py` 必须 `<= 1000` 行（本期硬门禁）。
      3. 后端其余核心业务文件建议 `<= 1500` 行（超出需给拆分计划）。
      4. 前端核心业务文件建议 `<= 1200` 行（超出需给拆分计划）。
      5. 入口门面建议 `<= 500` 行（超出需解释不可避免原因）。
2. 可靠性：
   1. 启动命令、关键 API 路径、错误码保持兼容。
3. 可观测性：
   1. 关键日志、审计链路与错误上下文不丢失。
4. 性能：
   1. 不要求本轮提速，但不得出现明显退化。
5. 完成判定：
   1. 任何一项硬门禁不达标，本轮结论为“未完成（fail_hard_gate）”。
   2. 未完成状态下禁止宣称“重构完成”，仅可提交“阶段进展 + 剩余拆分计划”。

## 6. 验收标准（Given/When/Then）
1. AC-ER-01 路由主干拆分
   1. Given 当前主服务文件含超大 `do_GET/do_POST` 分发
   2. When 完成本轮重构
   3. Then 主文件不再承载全量分发分支，路由由域模块注册
2. AC-ER-02 模板外置
   1. Given 当前存在内联 HTML/CSS
   2. When 完成本轮重构
   3. Then 模板逻辑迁移到展示层模块，主服务文件不再包含大块内联模板
3. AC-ER-03 DB/迁移解耦
   1. Given 当前建表/迁移混在主服务文件
   2. When 完成本轮重构
   3. Then DB 连接与迁移逻辑独立到 `infra/db` 相关模块
4. AC-ER-04 Agent 链路解耦
   1. Given 当前 Agent 发现与策略解析集中在主服务文件
   2. When 完成本轮重构
   3. Then 相关链路迁移到服务层模块并由 API 层调用
5. AC-ER-05 兼容启动不变
   1. Given 现有运行方式
   2. When 使用既有命令启动
   3. Then 服务可正常启动且关键页面可用
6. AC-ER-06 UO 回归通过
   1. Given 重构后代码
   2. When 执行 `run_acceptance_training_center_uo.py`
   3. Then 退出码为 `0`
7. AC-ER-07 AR 回归通过
   1. Given 重构后代码
   2. When 执行 `run_acceptance_agent_release_ar.py`
   3. Then 退出码为 `0`
8. AC-ER-08 目录治理文档完备
   1. Given 重构交付
   2. When 查看文档
   3. Then 可看到“目录树 + 职责映射 + 迁移阶段记录”
9. AC-ER-09 证据完备性
   1. Given 任一 AC 回传
   2. When 审核证据
   3. Then 至少包含功能截图、API/脚本输出、日志/DB 路径、代码路径
10. AC-ER-10 主文件瘦身门禁
   1. Given 完成本期重构
   2. When 统计 `src/workflow_app/workflow_web_server.py` 行数
   3. Then 行数必须 `<= 3000`
11. AC-ER-11 路由遗留瘦身门禁
   1. Given 完成本期重构
   2. When 统计 `src/workflow_app/server/api/legacy.py` 行数
   3. Then 行数必须 `<= 1000`
12. AC-ER-12 大文件清单可解释
   1. Given 完成本期重构
   2. When 输出前 15 大文件行数清单
   3. Then 任一超阈值文件都附“下一阶段拆分计划 + 预计降幅”
13. AC-ER-13 硬门禁失败判定
   1. Given 任一硬门禁超限
   2. When 提交本轮交付
   3. Then 必须标记 `fail_hard_gate`，且不得标记“全部完成”
14. AC-ER-14 新增文件命名合规
   1. Given 本轮新增或重命名文件
   2. When 审核命名
   3. Then 不得出现 `Phase*`/`01_*`/`02_*` 原型命名

## 7. 边界条件与异常处理
1. 若阶段回归失败：
   1. 必须回退到上个稳定点，再进入下一阶段。
2. 若拆分导致循环依赖：
   1. 优先调整模块边界，不允许以全局引用绕过。
3. 若本轮无法完成前后端全覆盖：
   1. 允许先完成后端核心拆分，但必须附下一阶段计划。

## 8. 依赖项与开放问题
1. 依赖项：
   1. `../workflow` 验收脚本可稳定复跑。
   2. 现有运行入口可保留兼容。
2. 开放问题：
   1. 前端是否同步引入构建/打包治理（本轮可先不改构建系统）。
3. 能力扩展评估（skill/agent）：
   1. 新增 agent 角色：`暂不采用`（当前是工程结构治理，不是角色职责缺失）。
   2. 新增 skill：`暂不采用`（本轮通过分阶段门禁与文档约束先落地）。
