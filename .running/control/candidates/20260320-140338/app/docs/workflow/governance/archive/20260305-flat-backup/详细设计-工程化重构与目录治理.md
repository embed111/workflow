# 详细设计-工程化重构与目录治理

## 1. 设计目标
1. 用“分层 + 分域 + 渐进迁移”替代当前单文件聚合形态。
2. 保持运行与验收链路兼容，避免重构期功能退化。
3. 形成可执行的目录治理标准，防止后续再次回流单文件。

## 2. 现状与改造焦点
1. 当前主运行文件仍为 `workflow_web_server.py`（超大单文件）。
2. 四个高风险聚合点：
   1. 内联页面模板（HTML/CSS）；
   2. DB 建表/迁移与流程辅助函数；
   3. Agent 发现与策略解析链路；
   4. `do_GET/do_POST` 全量路由分发。

## 3. 目标目录蓝图（后端）
1. 建议结构：
```text
src/workflow_app/
  workflow_web_server.py                # 兼容门面（薄入口）
  server/
    bootstrap.py                        # app 装配与启动
    api/
      config.py
      chat.py
      training.py
      policy.py
      dashboard.py
    services/
      agent_discovery.py
      policy_analysis.py
      training_orchestration.py
      dashboard_metrics.py
    infra/
      runtime_config.py
      audit_log.py
      db/
        connection.py
        migrations.py
        repositories/
          chat_repo.py
          training_repo.py
    presentation/
      pages.py
      templates/
        index.html
```
2. 兼容策略：
   1. `workflow_web_server.py` 仅保留参数解析、启动、路由注册调用。

2. 兼容策略：
   1. 保留现有全局 `state` 契约与初始化入口，先内部迁移后再收敛接口。

## 5. 迁移分阶段（强门禁）
1. 阶段 A：基线冻结
   1. 记录当前 AR/UO 结果与关键 API 样例。
2. 阶段 B：路由分发解耦
   1. 先拆 `do_GET/do_POST` 到 `server/api/*`。
   2. 主文件改为路由注册分发。
3. 阶段 C：模板与基础设施解耦
   1. 模板迁移到 `presentation`。
   2. DB 连接/迁移迁移到 `infra/db`。
4. 阶段 D：业务服务解耦
   1. Agent 发现、策略分析、训练编排迁移到 `services`。
5. 阶段 E：前端领域化拆分
   1. 拆分 `session/training/settings/shared`。
6. 每阶段结束动作：
   1. 回归 UO+AR；
   2. 失败即回滚，不进入下一阶段。

## 6. 路由注册机制（建议）
1. 采用显式注册：
   1. `register_config_routes(router)`
   2. `register_chat_routes(router)`
   3. `register_training_routes(router)`
   4. `register_policy_routes(router)`
   5. `register_dashboard_routes(router)`
2. Handler 仅做：
   1. 请求解析；
   2. 路由分发；
   3. 统一异常包装。

## 7. 目录治理规则
1. 新增规则：
   1. 不允许新增“跨域混合巨型模块”。
   2. 新增模块必须声明所属层（api/services/infra/presentation）。
2. 行数守门建议：
   1. 本期硬门禁：
      1. `workflow_web_server.py <= 3000`；
      2. `server/api/legacy.py <= 1000`。
   2. 其余建议门禁：
      1. 后端模块 `<= 1500` 行；
      2. 前端模块 `<= 1200` 行；
      3. 入口门面 `<= 500` 行。
3. 命名规则：
   1. 文件名与职责一一对应，避免 `misc/common2/final_new` 风格。

## 9. 硬门禁判定规则（新增）
1. 本轮完成必须同时满足：
   1. `src/workflow_app/workflow_web_server.py <= 3000`；
   2. `src/workflow_app/server/api/legacy.py <= 1000`；
   3. `Top 15` 行数对比报告已提交且可复核。
2. 任一条件不满足：

   1. 禁止 `Phase*`、`01_*`、`02_*` 原型命名；
   2. 必须采用“领域 + 职责”语义命名。
