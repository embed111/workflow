# 需求详情-测试数据展示环境策略与开关下线

## 1. 主题目标与价值说明
1. 主题目标：
   1. 正式下线设置页中的“展示测试数据”用户开关，不再允许正式用户在运行时手动切换测试数据可见性。
   2. 将“测试数据是否可见”的控制权从用户设置收口为环境策略，和 `dev/test/prod` 三环境治理保持一致。
   3. 明确 `prod` 环境默认不展示测试数据，避免正式使用场景被测试样本、示例数据和测试 agent 污染。
2. 业务价值：
   1. 降低正式环境的认知噪声，避免用户把测试数据误判为真实业务数据。
   2. 让“测试数据是否可见”与环境职责一致，减少历史开关在三环境架构下的语义冲突。
   3. 为后续环境隔离、发布验收和正式升级门禁提供更稳定的产品口径。

## 2. 用户画像与使用场景
1. 用户画像 A：正式环境使用者
   1. 关注点：在 `prod` 只看到正式可用数据，不被测试数据打扰。
2. 用户画像 B：开发者与测试验证者
   1. 关注点：`dev/test` 环境可按环境职责决定是否包含测试数据，但不通过页面设置临时改口径。
3. 用户画像 C：平台维护者
   1. 关注点：测试数据可见性由环境策略统一治理，不再依赖用户浏览器状态或页面开关。
4. 核心场景：
   1. 用户进入 `prod` 页面时，设置页不再看到“展示测试数据”开关，所有核心页面默认不纳入测试数据。
   2. `dev/test` 启动后按各自环境策略决定是否纳入测试数据，刷新页面或服务重启后口径不漂移。
   3. 会话入口、Dashboard、训练中心等模块统一读取环境策略，而不是各自拼装独立开关语义。

## 3. 用户旅程或关键流程
1. 系统启动时先识别当前运行环境与对应环境策略。
2. 后端解析当前环境下“测试数据是否可见”的有效值，并将其作为只读运行口径提供给前端。
3. 前端加载设置页时，不再渲染“展示测试数据”开关。
4. 会话入口、Dashboard、会话列表、训练中心等查询统一按环境策略决定是否纳入测试数据。
5. 页面刷新、浏览器重开或服务重启后，系统继续读取同一环境策略，不允许由本地缓存改写。

## 4. 功能需求清单（编号）
1. FR-TD-01 设置页开关下线
   1. `设置` 页不得再展示“展示测试数据”开关或等效可写控件。
   2. 会话入口、其他页面也不得保留任何等效的测试数据可见性用户开关。
2. FR-TD-02 环境策略接管
   1. 测试数据是否可见必须由当前环境策略决定，而不是由用户运行时切换。
   2. 页面侧不得再把浏览器缓存、本地键值或即时 UI 状态作为真实控制源。
3. FR-TD-03 正式环境强约束
   1. `prod` 环境默认不得展示测试数据、示例数据和测试 agent。
   2. `prod` 页面不得提供任何用户可见的运行时覆盖入口。
4. FR-TD-04 开发/测试环境策略化
   1. `dev/test` 环境是否纳入测试数据，由各自部署策略或环境配置决定。
   2. 即使 `dev/test` 可见测试数据，也不得通过页面设置让用户临时改写该策略。
5. FR-TD-05 全模块统一生效
   1. 会话入口 agent 列表（含测试/系统 agent）必须受环境策略统一控制。
   2. Dashboard、会话列表、训练中心相关查询必须受同一环境策略控制。
6. FR-TD-06 启动加载与只读呈现
   1. 前端启动时必须先读取后端返回的当前环境与有效测试数据策略。
   2. 若页面需要展示当前口径，只能以只读说明或调试信息展示，不得提供写入口。
7. FR-TD-07 查询参数统一口径
   1. 页面侧不得继续为不同模块拼装不同测试数据参数语义。
   2. 页面 API 调用必须统一依赖后端环境策略决定是否纳入测试数据。
8. FR-TD-08 向后兼容迁移
   1. 历史前端本地键、旧设置项和旧写接口在迁移后不得继续驱动真实行为。
   2. 旧能力下线后，不得影响核心会话创建、训练查看和正式环境使用。
9. FR-TD-09 证据语义门禁（沿用）
   1. 验收必须提交功能截图，不接受仅门禁统计截图。

## 5. 非功能需求（性能、安全、可用性、合规等）
1. 一致性：
   1. 同一环境下，测试数据可见性在刷新页面、重开浏览器和服务重启后必须保持一致。
2. 安全性：
   1. `prod` 环境必须默认 fail-closed，不因策略读取异常而误展示测试数据。
3. 可维护性：
   1. 环境策略应可由部署侧或环境配置统一管理，避免再次回退到页面级手动开关。
4. 可观测性：
   1. 当前环境、有效策略值、策略加载结果至少保留最小日志，便于排障与验收。

## 6. 验收标准（Given/When/Then）
1. AC-TD-01 设置页不再存在测试数据开关
   1. Given 用户打开任一环境的 `设置` 页
   2. When 检查测试数据相关控制项
   3. Then 页面不存在“展示测试数据”开关或等效写入口
2. AC-TD-02 正式环境不展示测试数据
   1. Given 系统运行在 `prod`
   2. When 用户查看会话入口、Dashboard、训练中心
   3. Then 页面不展示测试数据、示例数据或测试 agent
   4. And 用户无法在页面中手动改写该结果
3. AC-TD-03 开发/测试环境按环境策略生效
   1. Given 系统运行在 `dev` 或 `test`
   2. When 页面启动并加载数据
   3. Then 测试数据是否纳入由当前环境策略决定
   4. And 刷新页面后结果保持一致
4. AC-TD-04 全模块口径一致
   1. Given 当前环境策略已确定
   2. When 查看会话入口、Dashboard、会话列表与训练中心
   3. Then 各模块对测试数据的纳入结果一致
5. AC-TD-05 历史本地键不再生效
   1. Given 浏览器中仍残留历史测试数据开关键
   2. When 完成本次版本升级并重新进入系统
   3. Then 历史本地键不再驱动真实行为
   4. And 系统以当前环境策略为准
6. AC-TD-06 旧写接口下线或失效
   1. Given 外部仍尝试调用旧“写测试数据开关”接口
   2. When 请求发送到后端
   3. Then 接口返回明确的下线/忽略结果或不再提供该接口
   4. And 不得改写当前环境策略
7. AC-TD-07 策略异常时正式环境 fail-closed
   1. Given `prod` 启动时环境策略读取异常
   2. When 页面继续加载
   3. Then 系统默认不展示测试数据
   4. And 记录明确错误或告警
8. AC-TD-08 验收证据完备性
   1. Given 任一 AC 回传
   2. When 审核证据
   3. Then 至少包含功能截图、API 请求/响应、日志/配置路径、代码路径；门禁统计截图仅作辅助

## 7. 边界条件与异常处理
1. 若环境策略元数据缺失：
   1. `prod` 默认按“不展示测试数据”处理。
   2. 系统必须记录错误并允许运维排查。
2. 若部分模块仍沿用旧查询参数：
   1. 必须视为迁移未完成，不得宣称该需求已验收通过。
3. 若 `dev/test` 需要临时验证测试数据展示：
   1. 应通过部署配置或环境参数调整，而不是重新把用户开关放回页面。

## 8. 依赖项与开放问题
1. 依赖项：
   1. `workflow` 已具备可识别 `dev/test/prod` 运行环境的基础能力。
   2. 后端可向前端暴露当前环境与当前有效测试数据策略。
   3. 会话入口、Dashboard、训练中心均可接收统一环境策略。
2. 开放问题：
   1. 推断/假设（置信度: 中）：`dev/test` 的默认策略可由执行工作区按验证需要决定，本轮先不在需求侧强制写死为固定值。
3. 能力扩展评估（skill/agent）：
   1. 新增 agent 角色：`暂不采用`（本需求属于环境策略收口，不涉及角色分工重构）。
   2. 新增 skill：`暂不采用`（当前变更复杂度低，现有文档与门禁模板可覆盖）。
