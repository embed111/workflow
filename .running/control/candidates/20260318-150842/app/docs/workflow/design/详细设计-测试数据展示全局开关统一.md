# 详细设计-测试数据展示环境策略与开关下线

## 1. 设计目标
1. 移除设置页与页面侧所有“展示测试数据”用户可写开关。
2. 将测试数据可见性从“用户可改全局配置”升级为“环境策略驱动的只读运行口径”。
3. 保证会话入口、Dashboard、会话列表、训练中心四类视图都遵循同一环境策略。
4. 在 `prod` 默认 fail-closed，避免策略异常时误展示测试数据。

## 2. 现状基线与问题定位
1. 会话入口存在独立开关：
   1. `showSystemAgentsCheck`（`workflow_web_server.py:2245`）。
2. 全局设置存在另一开关：
   1. `showTestDataCheck`（`workflow_web_server.py:2500`）。
3. 前端当前为“双状态 + 本地缓存”：
   1. `showTestDataCacheKey` 与 `showSystemAgentsCacheKey` 并存（`00_app_state_and_utils.js:7-8`）。
   2. `showSystemAgentsCheck` 与 `showTestDataCheck` 分别绑定独立事件（`05_training_center_and_bootstrap.js:1123-1149`）。
4. 数据查询依赖页面拼装参数：
   1. `withTestDataQuery(...include_test_data=...)`（`00_app_state_and_utils.js:812-815`）。
   2. 后端多路由按 query 参数单次解析（如 `workflow_web_server.py:12569,12601,12684,12735,12771`）。
5. 根因结论：
   1. 历史方案默认“测试数据可见性是一个用户偏好”，但在 `dev/test/prod` 三环境架构下，该能力更接近环境职责而不是用户设置。
   2. 若继续保留页面开关，会让正式环境仍然存在“用户手动把测试数据放回页面”的语义漏洞。

## 3. 设计范围与非范围
1. 设计范围：
   1. 用户开关下线。
   2. 环境策略解析。
   3. 后端统一口径返回。
   4. 前端状态收敛与旧缓存清理。
2. 非范围：
   1. 测试数据判定规则重写（沿用现有识别逻辑）。
   2. 新增复杂权限系统或多租户隔离。
   3. 重新定义 `dev/test` 的最终默认策略矩阵。

## 4. 环境策略模型
1. 策略来源：
   1. 当前运行环境 + 部署配置。
2. 运行时有效字段（建议）：
   1. `environment: dev | test | prod`
   2. `show_test_data: bool`
   3. `show_test_data_source: environment_policy`
3. 强约束：
   1. `prod` 默认 `show_test_data=false`。
   2. 页面不得提供写入口覆盖该值。
4. 真值来源：
   1. 后端环境策略解析结果，前端只读消费。

## 5. 后端设计
1. 解析函数（建议）：
   1. `resolve_environment_name(cfg, state) -> str`
   2. `resolve_show_test_data_by_environment(cfg, state) -> bool`
2. 策略优先级（建议）：
   1. 当前运行环境
   2. 环境级部署配置
   3. 安全兜底默认值（`prod=false`，其他环境建议同样 fail-closed）
3. API 设计（需求级草案）：
   1. `GET /api/agents` 返回：
      1. `environment`
      2. `show_test_data`
      3. `show_test_data_source`
   2. `GET /api/status` 返回：
      1. `environment`
      2. `show_test_data`
   3. 旧 `POST /api/config/show-test-data`：
      1. 下线，或
      2. 返回明确废弃结果，不再写入真实配置
4. 查询口径统一：
   1. 抽象统一解析函数（建议）：`resolve_include_test_data(query, cfg, state)`。
   2. 主路径应优先使用环境策略，不再信任前端传入的即时切换值。
5. 路由改造范围（最小）：
   1. `/api/dashboard`
   2. `/api/chat/sessions`
   3. `/api/training/agents`
   4. `/api/training/queue`
   5. `/api/training/trainers`
6. 失败处理：
   1. 环境策略解析失败时记录错误。
   2. `prod` 必须退回到 `show_test_data=false`。

## 6. 前端设计
1. 状态模型收敛：
   1. 保留 `state.showTestData` 作为只读展示值。
   2. 删除 `state.showSystemAgents` 与其本地缓存键。
   3. 删除“用户切换测试数据”的本地写路径。
2. UI 收敛：
   1. 移除会话入口 `showSystemAgentsCheck` 与 `showSystemAgentsMeta`。
   2. 移除设置页 `showTestDataCheck`。
   3. 若需要展示当前环境口径，只允许在诊断/调试区域做只读展示。
3. 启动加载：
   1. bootstrap 以 `/api/agents` 或 `/api/status` 返回的环境策略初始化。
   2. 本地旧缓存只可用于清理，不再参与真值决策。
4. 查询参数治理：
   1. 页面不再依赖 `withTestDataQuery` 作为真实控制路径。
   2. `visibleAgents()` 改为基于后端返回的只读 `state.showTestData` 过滤。
5. 联动范围：
   1. 会话入口 agent 列表；
   2. sessions/workflows/dashboard；
   3. training center agents/queue。

## 7. 迁移与兼容
1. 本地键迁移：
   1. 废弃 `workflow.p0.settings.showSystemAgents`。
   2. 废弃 `workflow.p0.settings.showTestData` 或等效历史键。
2. 兼容期策略：
   1. 若前端仍收到旧缓存值，应在启动时清理或忽略。
   2. 若外部仍调用旧写接口，不得再改写真实策略。
3. 原型资产治理：
   1. 旧“测试数据开关开启/关闭态”参考图转为历史参考，不再代表当前有效 UI。

## 8. 状态机与时序
1. 启动时序：
   1. 页面启动
   2. 拉取 `/api/agents` 或 `/api/status`
   3. 读取 `environment/show_test_data`
   4. 渲染页面
   5. 继续加载会话与训练数据
2. 运行期：
   1. 不再存在“用户点击测试数据开关 -> 写回 -> 联动刷新”的时序。
   2. 若环境策略改变，应通过环境重启、部署切换或配置重载生效，而不是通过页面设置即时改写。

## 11. 推断/假设与确认项
1. 推断/假设（置信度: 中）：
   1. `dev/test` 是否默认展示测试数据，交由环境部署策略决定，比继续保留页面开关更符合当前阶段目标。
2. 推断/假设（置信度: 高）：
   1. 旧 `include_test_data` 查询参数可保留兼容壳，但不应再成为用户可控的真实入口。
3. 已确认：
   1. 设置页测试数据开关需要正式下线。
   2. `prod` 不应再给用户暴露测试数据切换能力。
