## 2. 本次任务性质
1. 本次是训练闭环后端真实实现，不是继续补 UI 示意。
2. 已有界面骨架、页签结构、演进图位置和滚动分层基本到位；本次重点是把这些区域接上真实后端数据与动作。
3. 若某一部分你判断“前端已实现”，本轮仍需验证其背后是否已有真实接口、真实落库与真实审计；没有则继续补齐。

## 3. 必须完成的后端范围
1. 三轮评测真实回写：
   1. 每轮必须真实产生 `Run1 / Run2 / Run3` 三条独立记录。
   2. `Avg` 必须由后端根据三条记录计算并返回。
   3. 未完成三条 run 时，不允许伪造 `Avg`、`decision`、`next_action`。
2. 状态页四个主视图真实数据驱动：
   1. `当前概览`
   2. `工作集变化`
   3. `三轮评测`
   4. `历史记录`
   5. 推荐最小接口：
      1. `GET /api/training/queue/{queue_task_id}/status-detail`
      2. `GET /api/training/queue/{queue_task_id}/loop`
3. 演进图与节点详情真实绑定：
   1. 演进图节点必须关联真实轮次、真实评测记录和真实决策。
   2. 不允许继续用硬编码节点目录或示意分数驱动页面。
4. 关键动作真实可执行：
   1. `POST /api/training/queue/{queue_task_id}/loop/enter-next-round`
   2. `POST /api/training/queue/{queue_task_id}/loop/rollback-round-increment`
   3. 动作后必须产生真实状态变更、可查询结果和审计日志。
5. 执行主体统一：
   1. 训练执行主体固定为 `workflow_native`。
   2. 页面、接口、数据库、审计日志口径保持一致。
   3. 旧的训练师选择语义只允许停留在兼容层，不得再作为当前主语义。

## 4. 推荐实现落点
1. 数据层：
   1. 为三轮评测结果增加可查询持久化结构。
   2. 为闭环演进图增加可查询持久化结构。
   3. 历史记录页签必须能按轮次回放真实摘要。
2. 接口层：
   1. `status-detail` 至少返回：
      1. `current_overview`
      2. `workset_changes`
      3. `evaluations`
      4. `history_records`
   2. `evaluations` 至少覆盖：
      1. `round_index`
      2. `run_results[]`
      3. `avg_score`
      4. `threshold`
      5. `previous_avg_score`
      6. `decision`
      7. `next_action`
   3. `loop` 至少返回：
      1. `current_node_id`
      2. `nodes`
      3. `edges` 或等效父子关系
      4. `metrics_available`
      5. `metrics_unavailable_reason`
3. 动作与幂等：
   1. `进入下一轮` 与 `回退本轮新增` 必须具备重复请求保护或明确错误返回。
   2. 不允许一次重复点击产生多个相同主状态写入。
4. 执行链路：
   1. `创建任务 -> 保存并启动首轮` 后，应走 `workflow_native` 执行主体。
   2. 后端应能把首轮执行结果回写到当前状态页，而不是只停留在队列初始态。

## 5. 明确禁止
1. 不要只改前端 mock 数据或静态 JSON。
2. 不要只让按钮“看起来可点”，但后端无真实状态变化。
3. 不要在 run 未完成时拼装伪 `Avg`。
4. 不要继续把 `三轮评测` 页签长期留成业务空态，除非后端明确返回真实空态/部分完成态。
5. 不要把“已有截图”当作功能完成；本轮必须以 API/DB/审计证据为主。

## 6. 最低验收要求
1. 功能验收：
   1. 创建一个训练任务后，状态页可看到真实 `Run1 / Run2 / Run3 / Avg` 或真实部分完成态。
   2. `当前概览 / 工作集变化 / 三轮评测 / 历史记录` 四个页签由后端返回真实数据驱动。
   3. `进入下一轮` 与 `回退本轮新增` 均可执行，且执行后页面状态发生真实更新。
   4. 对外执行主体统一显示为 `workflow_native`。
2. 证据验收：
   1. 页面截图：
      1. 状态页四个主页签至少各 1 张。
      2. 进入下一轮前后各 1 张。
      3. 回退本轮新增前后各 1 张。
   2. 接口证据：
      1. `status-detail` 响应 JSON
      2. `loop` 响应 JSON
      3. `enter-next-round` 响应 JSON
      4. `rollback-round-increment` 响应 JSON
   3. 数据证据：
      1. 三轮评测记录落库证据
      2. 闭环节点或闭环状态落库证据
      3. 审计日志落库证据
3. 自动化验收：
   1. 更新或新增验收脚本，至少覆盖：
      1. 创建训练任务
      2. 查询状态详情
      3. 查询演进图
      4. 触发进入下一轮
      5. 触发回退本轮新增
      6. 落盘 API/DB 证据到 `logs/runs/<run-id>/`
