# 需求详情-训练闭环后端回写与三轮评测契约

## 1. 主题目标与价值说明
1. 将训练闭环从“前端结构已收口、后端仍部分空态/占位”推进到“真实数据驱动、动作可执行、状态可追溯”。
2. 补齐当前最容易被执行方误解的后端边界：三轮评测真实回写、状态页数据契约、进入下一轮/回退动作留痕、执行主体统一收口。
3. 让 `任务状态` 页中的 `当前概览 / 工作集变化 / 三轮评测 / 历史记录` 在后端数据到位时可真实承载，而不是继续用业务空态长期占位。

## 2. 用户画像与使用场景
1. 用户画像：
   1. workflow 使用者：需要看到训练任务的真实当前状态，而不是示意结果。
   2. 需求分析师/验收者：需要判断演进图、三轮评测、动作按钮是否已经真正接上后端。
   3. 执行工作区后端实现者：需要明确数据模型、接口、动作回写和验收证据边界。
2. 使用场景：
   1. 用户启动首轮训练后，在 `任务状态` 查看本轮 `Run1 / Run2 / Run3 / Avg` 的真实结果。
   2. 用户在状态页执行 `进入下一轮` 或 `回退本轮新增`，并在同页看到后端状态更新。
   3. 验收者通过接口响应、数据库记录、审计日志与页面截图确认：页面不再依赖静态示意数据。

## 3. 用户旅程或关键流程
1. 用户在 `创建任务` 页提交 `保存并启动首轮`。
2. 后端创建训练计划、训练队列任务，并以 `workflow_native` 作为执行主体。
3. 某一轮训练执行时，系统必须完成 3 次“清空上下文后的独立评测”，并分别落库。
4. 后端计算本轮 `Avg`、与上一轮均值/阈值对比后，生成当前决策与下一步动作。
5. 状态页通过后端接口读取：
   1. 当前概览；
   2. 工作集变化；
   3. 三轮评测结果；
   4. 历史记录；
   5. 演进图与当前节点详情。
6. 用户触发 `进入下一轮` 或 `回退本轮新增` 后，后端更新闭环状态、审计日志与页面可见结果。

## 4. 功能需求清单
1. FR-UO-43 三轮评测结果必须后端真实回写
   1. 每一轮训练必须产生 3 条独立评测记录，分别对应 `Run1 / Run2 / Run3`。
   2. 每条评测记录至少包含：
      1. `queue_task_id`
      2. `round_index`
      3. `run_index`（1~3）
      4. `status`
      5. `score`
      6. `summary`
      7. `started_at`
      8. `finished_at`
      9. `context_reset=true`（或等效事实字段）
      10. `evidence_ref`（日志/文件/结果引用，允许为空但字段需存在）
   3. 本轮平均分 `avg_score` 必须由后端根据 3 条独立评测记录计算，不允许前端自行拼装或计算。
   4. 当 3 条记录未全部完成时，系统不得生成伪 `avg_score` 作为最终判定依据。
2. FR-UO-44 状态页后端查询接口必须覆盖四个主视图
   1. 系统必须提供用于获取 `任务状态` 页主数据的后端接口。
   2. 推荐最小契约：
      1. `GET /api/training/queue/{queue_task_id}/status-detail`
      2. `GET /api/training/queue/{queue_task_id}/loop`
   3. `status-detail` 返回至少包含：
      1. `current_overview`
      2. `workset_changes`
      3. `evaluations`
      4. `history_records`
   4. `evaluations` 至少包含：
      1. `round_index`
      2. `run_results[]`（3 条）
      3. `avg_score`
      4. `threshold`
      5. `previous_avg_score`
      6. `decision`
      7. `next_action`
3. FR-UO-45 演进图与节点详情必须关联真实轮次结果
   1. `loop` 接口返回的当前节点详情，必须能关联到真实轮次与评测结果。
   2. 节点详情至少应可追溯：
      1. `round_index`
      2. `queue_task_id`
      3. `run_ids` 或等效评测记录引用
      4. `decision`
      5. `next_action`
      6. `impact`
   3. 演进图中的“提升但未达标 / 劣化回退 / 当前主线继续”不得仅由前端文案推断，必须与后端真实决策一致。
4. FR-UO-46 关键动作必须改变后端状态并留下审计
   1. `POST /api/training/queue/{queue_task_id}/loop/enter-next-round` 必须可执行，且执行后至少完成：
      1. 闭环状态更新；
      2. 下一轮节点或等效记录创建；
      3. 审计日志写入。
   2. `POST /api/training/queue/{queue_task_id}/loop/rollback-round-increment` 必须可执行，且执行后至少完成：
      1. 当前轮增量工作集回退标记或等效状态变更；
      2. 演进图回退节点或等效回退记录写入；
      3. 审计日志写入。
   3. 以上动作执行后，页面刷新或自动联动时必须可见更新后的真实状态。
5. FR-UO-47 训练执行主体统一收口为 `workflow_native`
   1. 训练计划、训练队列、训练运行、闭环节点、审计日志与接口返回中的执行主体口径必须统一。
   2. 对外主字段固定为 `execution_engine=workflow_native`（或等效中性字段）。
   3. 若实现期需兼容历史字段 `trainer_match`，仅允许保留在后端兼容层，不得继续作为页面与接口主语义。
6. FR-UO-48 空态与部分可用状态必须真实表达
   1. 若三轮评测尚未开始，应返回明确空态，不得回填示例数据。
   2. 若 3 次评测仅完成部分记录，应返回部分完成态：
      1. 已完成 run 显示真实结果；
      2. 未完成 run 显示真实状态；
      3. `avg_score`、`decision`、`next_action` 不得伪造最终值。
   3. 若指标暂不可用，必须通过 `metrics_available=false` 与 `metrics_unavailable_reason`（或等效字段）向前端明确说明。
7. FR-UO-49 历史记录页签必须由后端回写轮次历史
   1. 每一轮闭环结束后，后端必须向历史记录提供可查询的轮次摘要。
   2. 历史记录至少包含：
      1. `round_index`
      2. `decision`
      3. `avg_score`
      4. `threshold`
      5. `workset_delta_summary`
      6. `created_at/updated_at`
      7. `audit_refs`

## 5. 非功能需求
1. 一致性：
   1. 页面、接口、数据库、审计日志中的 `decision / next_action / execution_engine` 口径必须一致。
2. 可追溯性：
   1. 任一页面中的本轮结论都必须可追溯到接口响应、数据库记录与审计日志。
3. 幂等性：
   1. `进入下一轮` 与 `回退本轮新增` 至少应具备“重复请求不产生重复主状态写入”的保护或明确错误返回。
4. 真实性：
   1. 后端返回中不得默认混入示意节点、示意分数、示意阈值。
5. 验收性：
   1. 后端交付回传必须同时包含页面截图、接口证据、日志/数据库证据与代码路径。

## 6. 验收标准
1. AC-UO-43 三轮评测结果真实回写
   1. Given 某训练任务完成一轮训练评测
   2. When 用户查看 `三轮评测` 页签或调用后端接口
   3. Then 可见 `Run1 / Run2 / Run3 / Avg`
   4. And `Avg` 由后端返回
   5. And 数据库中存在 3 条对应 run 记录
2. AC-UO-44 部分完成态不展示伪平均分
   1. Given 某一轮仅完成部分评测 run
   2. When 用户查看 `三轮评测` 页签
   3. Then 页面只显示已完成 run 的真实结果与未完成 run 的真实状态
   4. And 不展示伪 `Avg` 与伪最终决策
3. AC-UO-45 状态页四个主页签由后端真实数据驱动
   1. Given 某训练任务已有真实后端数据
   2. When 用户查看 `当前概览 / 工作集变化 / 三轮评测 / 历史记录`
   3. Then 四个页签内容来自后端接口返回
   4. And 不再长期以业务空态占位
4. AC-UO-46 进入下一轮动作可执行且留痕
   1. Given 当前轮结果高于上一轮但未达到阈值
   2. When 用户执行 `进入下一轮`
   3. Then 后端生成下一轮节点或等效状态记录
   4. And 审计日志记录该动作
   5. And 页面刷新后可见新的主线推进结果
5. AC-UO-47 回退本轮新增动作可执行且留痕
   1. Given 当前轮结果低于上一轮
   2. When 用户执行 `回退本轮新增`
   3. Then 后端生成回退节点或等效回退状态
   4. And 审计日志记录该动作
   5. And 页面刷新后可见回退结果
6. AC-UO-48 执行主体固定为 `workflow_native`
   1. Given 用户查看训练任务的计划、执行、闭环与审计数据
   2. When 查询接口响应与数据库记录
   3. Then 对外执行主体统一为 `workflow_native`
   4. And 页面不再出现训练师选择语义
7. AC-UO-49 历史记录页签可追溯轮次历史
   1. Given 某训练任务已完成多轮训练
   2. When 用户查看 `历史记录` 页签
   3. Then 可见每轮的真实历史摘要
   4. And 任一历史项可追溯到对应轮次的数据库记录与审计日志

## 7. 边界条件与异常处理
1. 若训练执行链路暂不可用：
   1. 允许保留计划与任务；
   2. 禁止伪造运行结果；
   3. 接口应返回明确不可用原因。
2. 若单个 run 执行失败：
   1. 该轮应进入部分完成或失败态；
   2. 不得直接产出最终平均分和最终决策。
3. 若历史字段与新字段并存：
   1. 允许后端兼容读取；
   2. 前端与验收一律以新字段口径为准。
4. 若历史任务没有完整闭环数据：
   1. 可返回空态或部分可用态；
   2. 不允许回填示意演进图伪装为真实历史。

## 8. 依赖项与开放问题
1. 依赖项：
   1. 依赖 `docs/workflow/requirements/需求详情-统一入口与训练优化模块.md` 中 `FR-UO-16/17/32/33/34/35/36` 的主题口径。
   2. 依赖 `docs/workflow/design/详细设计-统一入口与训练优化模块.md` 中第 24、25 节的接口与数据设计方向。
   3. 依赖 `workflow` 内建训练执行链路可稳定产出 3 次独立评测结果。
2. 开放问题：
   1. 单个失败 run 是否允许单独重试，还是必须整轮重跑。
   2. 历史记录页签是否需要支持按轮次展开查看单个 run 明细。
   3. `status-detail` 与 `loop` 是否最终合并为单接口，允许实现侧在不丢信息的前提下调整。
