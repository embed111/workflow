# 需求详情-任务中心产物交付与路径管理

## 1. 主题目标与价值说明
1. 为 `任务中心` 增加统一的任务数据与产物落盘规则，避免任务信息散落在 `workflow` 本地库与多个目录中。
2. 通过统一的 `任务产物路径` 配置，确保任务信息、执行链路文件与最终产物都驻留在用户本地目录。
3. 通过“无交付人 / 指定交付人”两种情况，支撑默认落盘和点对点交付两类场景。
4. 将“任务成功”与“产物已交付”绑定，避免任务状态成功但产物不可追溯。
5. 将目录结构从“按 agent 散落”收口为“按任务聚合”，让任务路径更干净、可浏览、可动态加载。

## 2. 用户画像与使用场景
1. 用户画像 A：workflow 操作者
   1. 关注点：创建任务时就明确产物交付方式，任务结束后能看到实际落盘路径。
2. 用户画像 B：平台治理者
   1. 关注点：所有任务产物是否都受统一根路径约束，交付链路是否可追溯。
3. 用户画像 C：接收产物的目标 agent
   1. 关注点：当任务指定自己为交付对象时，能否在统一约定目录下接收产物。
4. 核心场景：
   1. 管理员在 `设置` 既有内容中新增并维护 `任务产物路径`。
   2. 用户在 `任务中心` 创建任务时，选择产物交付类型。
   3. 若选择 `指定交付人`，则为该任务指定唯一交付目标 agent。
   4. 任务执行完成后，通过统一接口提交产物。
   5. 用户在任务详情中查看产物状态、交付类型、实际落盘路径与任务目录。
   6. 未完成产物交付时，任务不能被人工修改为成功态。

## 3. 用户旅程或关键流程
1. 管理员进入 `设置`，在既有设置内容中配置 `任务产物路径`。
2. 系统校验路径可用后，将其标记为当前生效路径。
3. 系统在 `任务产物路径` 下创建 `tasks/` 目录，并生成根目录说明文件 `TASKS_STRUCTURE.md`。
4. 用户进入 `任务中心`，点击 `+ 创建任务`。
5. 在现有任务表单基础上补充：
   1. `产物交付类型`：`无交付人 / 指定交付人`
   2. `交付给 agent`：仅当选择 `指定交付人` 时显示，且单选
6. 任务保存后，系统在 `<任务产物路径>/tasks/<ticket_id>/` 下写入：
   1. `task.json`
   2. `nodes/<node_id>.json`
   3. `TASK_STRUCTURE.md`
7. 任务执行过程中，通过统一接口提交产物文件。
8. 系统按规则落盘：
   1. 运行链路：`<任务产物路径>/tasks/<ticket_id>/runs/<run_id>/...`
   2. 无交付人：`<任务产物路径>/tasks/<ticket_id>/artifacts/<node_id>/output/...`
   3. 指定交付人：
      1. `<任务产物路径>/tasks/<ticket_id>/artifacts/<node_id>/output/...`
      2. `<任务产物路径>/tasks/<ticket_id>/artifacts/<node_id>/delivery/<receiver_agent_id>/...`
   4. 目录结构说明：
      1. 根目录：`<任务产物路径>/TASKS_STRUCTURE.md`
      2. 单任务目录：`<任务产物路径>/tasks/<ticket_id>/TASK_STRUCTURE.md`
9. 用户在任务详情中查看：
   1. 产物交付状态
   2. 交付类型
   3. 交付对象
   4. 实际落盘路径
10. 若任务尚未完成产物交付，则“人工修改执行状态 -> 成功”动作被阻止。

## 4. 功能需求清单
1. FR-PM-01 任务产物路径配置
   1. `设置` 中必须新增 `任务产物路径` 配置区块。
   2. 该区块必须作为既有设置页中的新增内容出现，不得替代或遮蔽原有设置内容。
   3. 区块至少包含：
      1. 当前生效路径
      2. 选择本地路径入口
      3. 路径校验状态
      4. 根目录说明文件：`<任务产物路径>/TASKS_STRUCTURE.md`
      5. 任务目录根：`<任务产物路径>/tasks/`
   4. 系统必须保证任务信息、执行链路文件与任务产物都受该路径约束。
2. FR-PM-02 不新增独立主模块
   1. 第一版前端不得新增左侧独立 `产物管理` 顶层模块。
   2. 产物能力在界面上仅通过 `设置` 与 `任务中心` 暴露。
3. FR-PM-03 创建任务表单增量字段
   1. `任务中心` 创建任务表单必须新增：
      1. `产物交付类型`
      2. `交付给 agent`
   2. `产物交付类型` 必须支持：
      1. `无交付人`
      2. `指定交付人`
   3. `交付给 agent` 仅在 `指定交付人` 时显示。
4. FR-PM-04 单交付人限制
   1. 第一版 `交付给 agent` 仅允许单选。
   2. 第一版不支持多个接收 agent 协作交付。
5. FR-PM-05 任务详情产物区块
   1. `任务中心` 右侧任务详情中必须新增 `产物` 区块。
   2. 该区块至少展示：
      1. 产物状态
      2. 交付类型
      3. 交付对象
      4. 实际落盘路径
      5. 最近一次交付时间
   3. 至少提供 `查看产物` 入口。
6. FR-PM-06 无交付人路径规则
   1. 当任务处于 `无交付人` 情况时，产物必须落到：
      1. `<任务产物路径>/tasks/<ticket_id>/artifacts/<node_id>/output/...`
7. FR-PM-07 指定交付人路径规则
   1. 当任务指定交付人时，系统必须同时生成两份产物落盘：
      1. `<任务产物路径>/tasks/<ticket_id>/artifacts/<node_id>/output/...`
      2. `<任务产物路径>/tasks/<ticket_id>/artifacts/<node_id>/delivery/<receiver_agent_id>/...`
   2. 第二条路径中的 `交付agent` 必须取自任务创建时指定的接收对象。
8. FR-PM-08 成功门禁绑定产物交付
   1. 若任务未完成产物交付，则不得人工修改为成功态。
   2. 若系统尚未配置有效 `任务产物路径`，则不得允许任务被人工修改为成功态。
9. FR-PM-09 统一交付接口
   1. 所有 agent 任务产物必须通过统一接口交付，不得绕过系统协议直接记成功。
   2. 统一接口至少需要记录：
      1. 任务标识
      2. 工作 agent
      3. 交付类型
      4. 交付对象（若有）
      5. 实际落盘路径
      6. 交付时间
10. FR-PM-10 路径预览
   1. 创建任务表单中应能预览当前任务的目标落盘路径。
   2. 任务详情中应展示最终实际落盘路径，而不是只展示抽象状态。
11. FR-PM-11 任务数据与目录结构规则
   1. 系统必须在 `任务产物路径` 下固定创建 `tasks/` 目录。
   2. 每个任务必须统一落到 `<任务产物路径>/tasks/<ticket_id>/`。
   3. 每个任务目录至少包含：
      1. `task.json`
      2. `nodes/`
      3. `runs/`
      4. `artifacts/`
   4. 每次真实执行完成产物落盘后，系统必须同步刷新：
      1. `<任务产物路径>/TASKS_STRUCTURE.md`
      2. `<任务产物路径>/tasks/<ticket_id>/TASK_STRUCTURE.md`
   5. `workflow` 本地目录不得持久化任务明文；任务信息、执行链路文件与产物文件都必须以 `任务产物路径` 为唯一持久化落点。

## 5. 非功能需求
1. 一致性：
   1. 页面展示的路径规则必须与后端真实落盘规则一致。
2. 可追溯性：
   1. 任一任务产物都应能追溯到任务、工作 agent、交付类型和接收对象。
   2. 任一任务记录都应能追溯到 `<任务产物路径>/tasks/<ticket_id>/` 目录位置。
   3. 用户应能通过根目录与单任务目录中的结构说明文件，快速理解当前目录的用途与分层。
3. 可用性：
   1. 用户在查看任务详情时，应能在 3 秒内判断“是否已交付、交付给谁、文件在哪”。
4. 安全性：
   1. 任务信息与产物落盘路径必须限制在配置的 `任务产物路径` 下，不得任意越界写入。
5. 隐私边界：
   1. `workflow` 仅允许在内存中保留运行中的调度工作集，不得在自身目录中持久化任务明文。

## 6. 验收标准
1. AC-PM-01 设置中配置任务产物路径
   1. Given 用户进入 `设置`
   2. When 查看产物路径配置区
   3. Then 可看到既有设置内容仍保留
   4. And 可看到当前生效路径、选择本地路径入口与路径校验状态
   5. And 可看到任务目录根为 `<根路径>/tasks/`
   6. And 可看到根目录说明文件为 `<根路径>/TASKS_STRUCTURE.md`
2. AC-PM-02 创建任务表单出现产物交付字段
   1. Given 用户点击 `+ 创建任务`
   2. When 表单展开
   3. Then 可看到 `产物交付类型`
   4. And `产物交付类型` 仅包含 `无交付人 / 指定交付人`
   5. And 选择 `指定交付人` 后可看到 `交付给 agent`
3. AC-PM-03 无交付人路径正确
   1. Given 某任务选择 `无交付人`
   2. When 任务提交产物
   3. Then 产物落到 `<根路径>/tasks/<ticket_id>/artifacts/<node_id>/output/...`
4. AC-PM-04 指定交付人路径正确
   1. Given 某任务选择 `指定交付人`
   2. And 交付对象为某单一 agent
   3. When 任务提交产物
   4. Then 任务目录中的 `output/` 保留一份
   5. And 任务目录中的 `delivery/<receiver_agent_id>/` 目录也收到一份
5. AC-PM-05 未交付产物不得改为成功态
   1. Given 某任务尚未完成产物交付
   2. When 用户尝试通过 `人工修改执行状态` 将任务改为成功
   3. Then 系统拒绝并提示需先完成产物交付
6. AC-PM-06 任务详情可见实际路径
   1. Given 某任务已完成产物交付
   2. When 用户打开该任务详情
   3. Then 可看到交付类型、交付对象、实际落盘路径与最近一次交付时间
7. AC-PM-07 第一版不出现独立产物主模块
   1. Given 用户查看左侧导航
   2. When 页面加载完成
   3. Then 不出现独立 `产物管理` 顶层模块
8. AC-PM-08 任务目录固定
   1. Given 用户已配置任务产物路径并创建任务
   2. When 系统写入任务记录
   3. Then 任务记录被统一写入 `<根路径>/tasks/<ticket_id>/` 目录下
9. AC-PM-09 结构说明自动刷新
   1. Given 用户已配置任务产物路径且某任务完成一次真实执行并完成产物落盘
   2. When 用户查看 `<根路径>/`
   3. Then 可看到最新的 `TASKS_STRUCTURE.md`
   4. And 在 `<根路径>/tasks/<ticket_id>/` 下可看到最新的 `TASK_STRUCTURE.md`
10. AC-PM-10 workflow 本地目录不落盘任务明文
   1. Given 用户已配置任务产物路径并完成任务创建与执行
   2. When 检查 `workflow/state/`、`workflow/.runtime/` 与 `workflow/logs/`
   3. Then 不应看到任务明文的持久化副本
   4. And 任务信息、执行链路文件与最终产物都应可在 `任务产物路径` 下找到

## 7. 边界条件与异常处理
1. 若未配置任务产物路径：
   1. 任务仍可创建，但不得人工修改为成功态。
2. 若用户选择 `指定交付人` 但未选择接收 agent：
   1. 系统必须拒绝保存该任务。
3. 若交付接口写入路径超出任务产物路径：
   1. 系统必须拒绝写入，并返回路径越界错误。
4. 若任务名称包含不适合作为目录名的字符：
   1. 系统必须进行安全归一化处理，并在展示层保持可读映射。
5. 若指定交付人不存在于当前角色池：
   1. 系统必须拒绝保存该任务。
6. 若 `tasks/` 目录或单任务目录创建失败：
   1. 系统必须拒绝将任务创建标记为成功，并返回明确错误。
