# 详细设计-角色画像发布格式与预发布判定

## 1. 主题目标与价值说明
1. 目标：把训练中心角色详情区从“先版本操作”调整为“先简历式角色画像”，并建立发布格式与预发布状态的统一判定链路。
2. 价值：确保“发布版本”语义稳定，避免误切换到不合规提交；同时把头像能力收敛为整洁的单头像展示与轻量更换入口。
3. 正式发布成功后，角色详情页优先展示“最新正式发布版本的第一人称角色述职报告”。

## 2. 用户画像与使用场景
1. 训练中心操作者：先看角色能力/知识再决定发布操作。
2. 发布治理人员：希望系统自动拦截不合规发布格式。
3. 维护人员：需要实时识别预发布状态（基于工作区改动）。
4. 角色运营人员：需要保留一个轻量的“更换头像”入口，而不是复杂的头像管理区。

## 3. 用户旅程或关键流程
1. 用户选择角色工牌 -> 触发角色详情加载。
2. 服务端返回角色画像字段 + 最新正式发布报告快照 + 版本元数据 + Git 状态。
3. 前端先执行“角色画像来源决策”：`latest_release_report -> structured_fields_fallback`。
4. 前端先渲染“单栏简历式画像区”，再渲染发布历史与操作区。
5. 发布版本列表由“发布格式校验结果=通过”的记录组成。
6. 预发布标识由 `git status --porcelain --untracked-files=normal` 判定驱动。
7. 头像通过“更换头像”入口触发本地文件选择并更新；失败时回退默认头像或保留旧头像。

## 4. 功能设计清单（与需求 FR 对应）
1. DRP-01 角色画像数据模型
   1. 新增/统一字段：
      1. `capability_summary`（摘要）
      2. `first_person_summary`
      3. `what_i_can_do[]`（3~5 条短句）
      4. `full_capability_inventory[]`
      5. `knowledge_scope`
      6. `skills`（agent skill 清单，当前来源为工作区 `.codex/skills/`）
      7. `applicable_scenarios`
      8. `version_notes`
      9. `avatar_uri`（可空）
      10. `profile_source`（`latest_release_report|structured_fields_fallback`）
      11. `source_release_id`（当前角色详情使用的正式发布版本）
2. DRP-02 角色画像来源决策器
   1. 优先级固定为：`最新正式发布版本绑定的公开发布报告 > 最新正式发布版本结构化字段回退`。
   2. 预发布报告在正式发布成功前不得进入角色详情页主内容。
   3. 若公开发布报告解析失败，则降级到结构化字段回退，并返回 `profile_source=structured_fields_fallback` 与原因。
3. DRP-03 发布格式校验器
   1. 输入：候选版本元数据。
   2. 规则：`what_i_can_do/knowledge_scope/skills/applicable_scenarios/version_notes` 均存在且非空才判定 `release_valid=true`。
   3. 其中 `skills` 的业务语义固定为 `agent skill` 清单，不允许写入泛化后的技能标签。
   3. 输出：
      1. `release_valid`
      2. `invalid_reasons[]`
      3. `classification`（`release` | `normal_commit`）
4. DRP-04 版本列表生成规则
   1. `release` 列表仅包含 `classification=release`。
   2. `normal_commit` 不进入可切换版本候选。
5. DRP-05 预发布状态判定器
   1. 执行命令：`git status --porcelain --untracked-files=normal`。
   2. 判定：输出非空 -> `pre_release=true`；输出空 -> `pre_release=false`。
   3. `.gitignore` 忽略文件天然不进入该输出。
6. DRP-06 前端单栏简历式布局
   1. 详情区顺序（单栏）：`身份头卡（单头像/名称/定位/更换头像入口） -> 我是/我当前能做什么 -> 全量能力清单 -> 知识范围 -> Agent Skills -> 适用场景 -> 版本说明 -> 发布历史与操作`。
   2. 首屏必须可见“我能做什么”信息块，避免首屏被版本操作区挤占。
   3. 不采用双栏摘要卡并列主体区，信息按照阅读顺序自上而下展开。
   4. 头像区域只保留一个头像与一个次级入口，不引入独立上传面板、拖拽框或多头像候选区。
   5. `Agent Skills` 区默认展示当前 agent 工作区 `.codex/skills/` 下的本地技能，不混入系统级全局 skills。
   6. `capability_delta` 与发布亮点若需展示，应位于版本说明或折叠区，不能替代“全量能力清单”。
7. DRP-07 更换头像入口与上传动作
   1. 建议接口：`POST /api/training-center/agents/{agent_id}/avatar`（`multipart/form-data`）。
   2. 入参：`file`。
   3. 出参：`avatar_uri`、`updated_at`、`error_code`（失败时）。
   4. 前端交互：点击“更换头像”入口后再触发本地文件选择。
   5. 能力：单头像展示、替换头像、失败回退默认头像。
8. DRP-08 错误处理与降级
   1. 上传失败（格式/大小/写入失败）：
      1. 保留旧头像或默认头像。
      2. 返回可读错误文案，不阻断其他角色信息展示。
   2. 最新正式发布报告缺失或解析失败：
      1. 降级使用结构化字段渲染。
      2. 向前端返回 `profile_source` 与降级原因。

## 5. 非功能需求（性能、安全、可用性、合规等）
1. 性能：角色切换详情加载 `p95 <= 1.5s`。
2. 稳定性：Git 状态判定失败时返回可读错误码与提示，不得导致页面崩溃。
3. 可观测：记录发布格式校验结果、预发布判定日志、头像更换结果、角色详情来源决策结果（含时间戳、角色 ID）。
4. 兼容：旧版本记录缺少画像字段时默认归类 `normal_commit`。
5. 安全：头像上传需做后缀/内容类型校验，避免危险文件写入。
6. 上传限制（已确认）：`png/jpg/webp`，单文件 `<=2MB`。

## 6. 验收标准（Given/When/Then）
1. AC-RP-01~AC-RP-15 全量映射到 `docs/workflow/requirements/需求详情-角色画像发布格式与预发布判定.md`。
2. 补充设计级验收：
   1. Given 存在缺字段版本
   2. When 校验器执行
   3. Then 返回 `classification=normal_commit` 且 `invalid_reasons` 非空
3. 补充设计级验收：
   1. Given Git 命令执行异常
   2. When 判定器运行
   3. Then 返回 `pre_release=unknown` 与错误原因，不得误判为 `release`
4. 补充设计级验收：
   1. Given 用户进入角色详情页
   2. When 身份头卡完成渲染
   3. Then 页面只展示一个头像与一个“更换头像”入口
5. 补充设计级验收：
   1. Given 用户通过“更换头像”入口选择本地头像文件
   2. When 上传接口返回成功
   3. Then 页面无需刷新即可显示新头像
6. 补充设计级验收：
   1. Given 用户进入角色详情页
   2. When 页面完成首屏渲染
   3. Then 主体画像区按单栏顺序展示，无双栏并列主体区
7. 补充设计级验收：
   1. Given 最新正式发布版本已绑定公开发布报告
   2. When 页面执行角色画像来源决策
   3. Then 返回 `profile_source=latest_release_report`
   4. And 首屏优先展示第一人称摘要与全量能力清单
8. 补充设计级验收：
   1. Given 最新正式发布报告解析失败
   2. When 页面执行角色画像来源决策
   3. Then 返回 `profile_source=structured_fields_fallback`
   4. And 页面仍可完成角色详情展示

## 7. 边界条件与异常处理
1. 非 Git 仓库或 Git 不可用：
   1. 版本切换能力保持可读但禁用发布语义判定，状态展示为“不可判定”。
2. 多工作区并行：
   1. 预发布判定必须以角色对应工作区为单位执行，不得跨角色复用结果。
3. 字段兼容：
   1. `skills` 可支持字符串或数组输入，渲染时统一归一化为列表。
   2. 若 `skills` 为空，则展示“未声明 Agent Skills”空态。
   3. 当启用自动提取时，`skills` 的默认数据源为工作区 `.codex/skills/*/SKILL.md` 的技能元数据。
   4. `first_person_summary/full_capability_inventory` 缺失时，允许从结构化字段回退生成最小展示。
4. 上传并发：
   1. 同一角色并发上传按最后成功写入生效，失败请求不覆盖最新成功版本。

## 8. 依赖项与开放问题
1. 依赖项：
   1. 角色详情 API 支持新增画像字段与校验结果字段。
   2. 发布记录元数据可扩展并可回溯。
   3. 运行环境允许执行 Git 状态命令。
   4. 后端提供头像存储与访问路径回传。
   5. 发布管理模块已在正式发布成功后写入 `active_role_profile_ref/public_profile_ref`。
2. 开放问题：
   1. 头像存储介质与生命周期清理策略（覆盖旧图是否立即删除）待实现侧确认。
   2. 第一人称角色述职报告采用 Markdown 还是结构化富文本作为主展示源，待实现侧最终确认。
