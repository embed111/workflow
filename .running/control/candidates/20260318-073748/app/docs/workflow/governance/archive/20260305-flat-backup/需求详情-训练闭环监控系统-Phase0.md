# 需求详情-训练闭环监控系统-Phase0

## 1. 主题目标与价值说明
1. 主题目标：建立“自动采集 -> 自动建队列 -> 一键判定 -> 一键训练回写”的最小闭环。
2. 业务价值：
   1. 降低人工串联流程成本。
   2. 提升训练样本回流效率。
   3. 保证训练决策与执行全链路可追溯。

## 2. 用户画像与使用场景
1. 用户画像 A：workflow 使用者
   1. 关注点：在页面内直接对话完成工作，不切换工具。
2. 用户画像 B：需求分析师
   1. 关注点：判定入训是否准确、是否可追溯。
3. 用户画像 C：训练师操作者
   1. 关注点：执行训练是否稳定、结果是否可回写。
4. 用户画像 D：workflow 运维者
   1. 关注点：在线升级切换迅速、不中断核心工作。
3. 场景：
   1. 在 workflow 页面与目标 agent 连续对话处理任务。
   2. 早晨打开看板查看新增会话和待处理任务。
   3. 批量点击“判定是否入训”。
   4. 批量点击“执行训练”并查看回写结果。
   5. 发布 workflow 新版本时，系统自动完成切换并持续可用。
   6. 新建会话时自动加载目标 agent 的 `AGENTS.md` 规则，并冻结到该会话上下文。
   7. 多会话并行生成时，用户可明确看到“哪个会话正在运行”，并按需展开调用链路与提示词。
   8. 用户在对话中使用 `$root` 引用全局根路径，减少输入完整绝对路径的成本。

## 3. 用户旅程或关键流程
1. 用户在 workflow 页面选择目标 agent 并发起对话（禁止直连 agent；无可用 agent 时禁止创建会话）。
2. 新建会话时系统读取目标目录 `AGENTS.md`，并写入会话快照（`agents_path/hash/loaded_at/version`）。
3. workflow 网关先写入 `conversation_events`，再转发到目标 agent。
4. 页面展示 agent 回答，并在同一会话持续交互。
5. 对话生成期间，页面提供会话级运行提示（运行中标识、耗时、可中断）。
6. 每轮回复默认折叠显示调用链路入口，按需查看提示词与关键执行事件。
7. 系统自动生成 `analysis_tasks`（状态 `pending`）。
8. 需求分析师点击“判定是否入训”。
9. 若判定 `train`，系统自动创建 `training_tasks`。
10. 训练师点击“执行训练”。
11. 训练完成后写回状态与报告路径，任务进入 `done/failed`。
12. 若发布新版本，系统在健康检查通过后自动切换流量，页面继续可用并保持会话可追溯。

## 4. 功能需求清单（编号）
1. FR-00 统一入口强约束
   1. 所有 agent 请求必须经过 workflow 网关。
   2. 未经网关的直连请求默认拒绝（仅允许受控紧急旁路）。
2. FR-00A 对话工作台（页面内与 agent 对话）
   1. 支持创建/继续会话、发送消息、查看流式回复。
   2. 支持选择目标 agent 与会话标签。
   3. 每轮对话自动产生日志事件并关联 `session_id`。
3. FR-00B 对话体验对齐 CLI（丝滑交互）
   1. 支持流式 token 渲染与中断生成。
   2. 支持快捷发送与换行（`Enter` 发送，`Shift+Enter` 换行）。
   3. 支持一键重试上一轮请求。
   4. 支持会话内快速定位最近上下文（不跳页、不阻塞输入）。
4. FR-CTX-01 会话级 `AGENTS.md` 上下文冻结
   1. 新会话创建时必须加载并注入目标 agent 目录的 `AGENTS.md`。
   2. 会话元数据必须记录 `agents_path/agents_hash/agents_loaded_at/agents_version`。
   3. `AGENTS.md` 更新只影响后续新会话，不影响已创建会话。
   4. 若当前搜索根下无可用 agent（无 `AGENTS.md` 目录），禁止创建会话并返回可读错误提示。
5. FR-OBS-01 会话级运行可见性与 CLI 风格运行提示
   1. “运行中任务”区域必须展示会话级映射（`session_id/agent/started_at/elapsed/status`）。
   2. 对话区在生成阶段必须显示 CLI 风格运行态提示，且支持一键中断当前会话生成。
   3. 每轮助手回复必须提供默认折叠的“调用链路”入口，可查看提示词、命令参数与关键中间事件。
   4. 运行任务与会话消息必须可双向定位（从任务跳会话、从会话看任务）。
6. FR-UX-01 全局 `$root` 路径别名
   1. `$root` 必须解析到全局 `agent_search_root`，不是会话私有根路径。
   2. 支持在输入与任务参数中使用 `$root/...` 引用路径，减少绝对路径输入。
   3. 路径解析必须进行标准化和越界校验，禁止跳出 `$root` 边界。
   4. 切换 `agent_search_root` 前必须关闭全部会话；切换成功后仅对新创建会话生效。
7. FR-01 自动采集
   1. 周期扫描各 agent 日志源，识别新增会话事件。
8. FR-02 自动建分析队列
   1. 每条新增会话自动映射为一条分析任务。
9. FR-03 一键入训判定
   1. 支持对 `analysis_tasks.pending` 任务执行 `train|skip|need_info` 判定。
10. FR-04 一键训练执行
   1. 支持对 `training_tasks.pending` 任务触发训练师流程。
11. FR-05 结果回写
   1. 训练结果回写 `status/result_summary/trainer_run_ref`。
12. FR-06 可视化看板
   1. 展示 5 张卡片与 2 个按钮：
      1. 新增会话数（24h）
      2. 待分析任务数
      3. 待训练任务数
      4. 最近一次判定结果
      5. 最近一次训练结果
      6. 按钮：判定是否入训
      7. 按钮：执行训练
13. FR-07 审计留痕
   1. 所有关键动作写入日志并可追溯到 `task_id`。
14. FR-08 对账与回补
   1. 每日执行“入口请求数 vs 事件入库数”对账。
   2. 有缺口自动回补并产生告警记录。
15. FR-09 单机 A/B 在线升级自动切换（MVP）
   1. workflow 提供同机 A/B 两个运行槽位（一个在线，一个待发布）。
   2. 新版本先部署到待发布槽位并执行健康检查。
   3. 检查通过后自动切换流量到新槽位，无需人工介入。
   4. 切换后保留旧槽位作为快速回滚目标。
   5. 切换过程产生审计日志（含触发人、版本、耗时）。

## 5. 非功能需求（性能、安全、可用性、合规等）
1. 性能：
   1. 采集延迟 <= 60 秒。
   2. 判定结果回写 <= 10 秒。
   3. workflow 相比 CLI 基线的首 token 延迟增量 `p95 <= 200ms`。
   4. 流式渲染刷新间隔 `p95 <= 120ms`（无明显卡顿）。
2. 可靠性：
   1. 训练任务支持失败重试（至少 3 次）。
   2. `running` 超时可回滚为 `pending`。
   3. 事件写入失败时不返回成功响应（fail-closed）。
   4. 在线升级自动切换满足 `RTO <= 30 秒`。
   5. 切换导致的数据丢失窗口 `RPO <= 5 秒`。
   6. `AGENTS.md` 快照缺失或写入失败时，会话创建必须失败（fail-closed）。
3. 可观测性：
   1. 运行中任务状态刷新间隔 `p95 <= 1 秒`。
   2. 每轮回复至少保留一条可展开调用链路记录（默认折叠）。
   3. 调用链路查询失败时返回明确错误信息，不得静默丢失。
4. 可维护性：
   1. Phase 0 使用 SQLite 单文件，避免过早分布式化。
5. 安全：
   1. 高风险任务继续遵守 `prod 外发 + cand 影子` 路由。
   2. 旁路请求默认拒绝并记录审计日志。
   3. `$root` 路径越界请求必须拒绝并记录审计事件。

## 6. 验收标准（Given/When/Then）
1. AC-01 自动采集
   1. Given 有新的 agent 会话日志
   2. When 采集器下一轮扫描完成
   3. Then 60 秒内看板“新增会话数”增加，且数据库新增事件记录
2. AC-02 自动建单
   1. Given 新事件入库
   2. When 建单器执行
   3. Then 自动生成 `analysis_task`，状态为 `pending`
3. AC-03 一键判定
   1. Given 存在 `analysis_task=pending`
   2. When 点击“判定是否入训”
   3. Then 10 秒内回写 `decision/decision_reason`；若为 `train` 自动建 `training_task`
4. AC-04 一键训练
   1. Given 存在 `training_task=pending`
   2. When 点击“执行训练”
   3. Then 任务状态进入 `running` 并最终进入 `done/failed`，且有报告路径
5. AC-05 可追溯
   1. Given 任意训练结果
   2. When 追溯 `task_id`
   3. Then 可定位到判定日志、训练日志、训练报告
6. AC-06 统一入口约束
   1. Given 任意 agent 请求
   2. When 请求未通过 workflow 网关
   3. Then 系统拒绝请求并记录安全审计事件
7. AC-07 先日志后返回
   1. Given workflow 接收用户请求
   2. When 事件写入失败
   3. Then 不向用户返回成功结果，并提示稍后重试
8. AC-08 对账回补
   1. Given 每日对账任务执行
   2. When 发现入口请求数大于事件入库数
   3. Then 自动触发回补并生成缺口告警
9. AC-09 对话工作台可用
   1. Given 用户已登录 workflow 页面
   2. When 用户选择目标 agent 并发送消息
   3. Then 页面可收到回复且生成关联 `session_id` 的日志事件
10. AC-10 在线升级自动切换
   1. Given 有新 workflow 版本发布到待发布槽位
   2. When 健康检查全部通过
   3. Then 系统自动切换并在 `30 秒` 内完成升级切换，且切换事件可审计追溯
11. AC-11 对话丝滑体验
   1. Given 同模型同网络条件下的 CLI 基线压测样本
   2. When 通过 workflow 对话工作台执行同批次请求
   3. Then 首 token 延迟增量 `p95 <= 200ms` 且支持“中断/重试/快捷发送”完整闭环
12. AC-12 `AGENTS.md` 新会话生效
   1. Given `agent-A` 目录中的 `AGENTS.md` 初始版本为 `v1`，并已创建会话 `S1`
   2. When `AGENTS.md` 更新为 `v2` 后再创建会话 `S2`
   3. Then `S1` 继续使用 `v1` 快照，`S2` 使用 `v2` 快照
13. AC-13 会话级运行可见
   1. Given `S1/S2` 两个会话同时处于生成中
   2. When 用户查看“运行中任务”与任一会话窗口
   3. Then 用户可看到每个运行任务对应的会话标识与耗时，并可从任务跳转到会话
14. AC-14 `$root` 全局解析
   1. Given 全局 `agent_search_root=C:\\work\\agents`
   2. When 用户在任意会话输入 `$root/workflow/docs/workflow/需求概述.md`
   3. Then 系统解析并访问 `C:\\work\\agents\\workflow\\docs\\workflow\\需求概述.md`
15. AC-15 根路径切换与越界拦截
   1. Given 当前存在未关闭会话
   2. When 用户尝试切换 `agent_search_root` 或输入 `$root/../Windows`
   3. Then 系统拒绝操作并提示原因；仅在全部会话关闭后允许切换根路径

## 7. 边界条件与异常处理
1. 日志缺字段：
   1. 记录 `need_info` 并进入 `analysis_tasks.blocked`。
2. 训练执行失败：
   1. 自动重试，超过阈值后标记 `failed` 并告警。
3. 重复事件：
   1. 通过 `event_id` 去重，不重复建单。
4. 高风险任务误路由：
   1. 拒绝执行并写安全告警日志。
5. 直连旁路请求：
   1. 默认拒绝；若命中紧急旁路白名单，必须补录事件并标记 `bypass=true`。
6. 在线升级切换失败：
   1. 自动回退到旧槽位或进入只读保护模式，并触发告警。
7. 新会话缺少 `AGENTS.md`：
   1. 阻止会话创建并提示“当前搜索根下无可用 agent”。
8. `AGENTS.md` 文件不可解析：
   1. 记录 `session_init_failed`，并提示修复文件格式后重试。
9. `$root` 路径越界：
   1. 返回 `path_out_of_root` 并记录审计日志，不执行后续任务。
10. 切换根路径时存在活动会话：
   1. 拒绝切换并返回活动会话列表，提示先关闭全部会话。

## 8. 依赖项与开放问题
1. 依赖项：
   1. `trainer/` 现有执行链路可被 workflow 调用。
   2. agent 日志源路径可统一发现。
2. 开放问题：
   1. 采集频率默认值最终取 30 秒还是 60 秒。
   2. “最近一次判定/训练结果”卡片是否替换为其他业务指标。
   3. 紧急旁路白名单审批人和时效如何定义。
   4. 对话工作台第一批 CLI 对齐能力范围（是否纳入文件拖拽、命令别名）。

## 10. 本轮增量需求：对话直连训练内容（2026-02-26）
### 10.1 主题目标与价值说明
1. 目标：
   1. 将“工作记录”统一为“会话对话本体”，减少中间层对象与重复维护。
   2. 用“消息已分析/未分析”管理增量分析，避免重复处理。
2. 价值：
   1. 降低用户理解成本（只看会话，不看双轨数据）。
   2. 提升分析与训练计划的追溯一致性。

### 10.2 增量功能需求（FR）
1. FR-10 对话即训练输入源
   1. 训练入口中的“工作记录”列表必须直接映射到会话对话内容。
   2. 会话详情中必须可查看消息级训练相关标记（状态、价值判定、最近分析批次）。
2. FR-11 消息级分析状态
   1. 每条消息必须存在状态：`未分析` / `已分析`。
   2. 分析完成后，消息可被标记为“已分析但无训练价值”，并保留原因字段（如 `no_training_value`、`missing_previous_context`）。
3. FR-12 会话勾选门禁
   1. 仅允许勾选“包含未分析消息”的会话。
   2. 全部消息均为“已分析”的会话默认不可勾选，并展示不可勾选原因。
4. FR-13 全量上下文分析
   1. 每次分析必须基于该会话的完整对话上下文执行。
   2. 不允许仅对局部消息片段单独触发分析并覆盖会话级结论。
5. FR-14 分析结果到训练计划映射
   1. 每次分析运行必须产出唯一 `analysis_run_id`。
   2. 每个训练计划项必须能追溯到 `analysis_run_id` 与证据消息集合（`message_id[]`）。
   3. 允许出现 `0` 个训练计划项（即“已分析但无训练价值”）。
6. FR-15 失败回滚与缺失上下文处理
   1. 分析失败默认回滚：本次目标消息状态恢复为“未分析”。
   2. 若系统可判定“上文缺失”，允许直接写入“已分析 + 无训练价值 + 缺失原因=上文缺失”。
   3. 若判定“下文缺失”，保持“未分析”，等待后续新增消息后由用户再次触发分析。
7. FR-16 单条消息删除（会话分析状态页）
   1. 支持在会话分析状态页删除单条聊天记录（仅限“未生成训练计划”的会话）。
   2. 若会话已分析且已生成训练计划（`training_plan_items > 0`），必须拒绝删除该会话任意聊天记录。
   3. 删除动作与删除拒绝动作都必须写审计日志（操作者、时间、会话、消息、影响范围、拒绝原因）。
8. FR-17 功能 CRUD 审查门禁（需求治理）
   1. 每个功能需求条目必须显式声明 `Create/Read/Update/Delete` 四类能力：需要/可选/禁用。
   2. 对禁用能力必须给出原因、触发条件、错误码与用户提示语。
   3. 变更评审时若缺少 CRUD 审查，不得进入实现排期。

### 10.3 增量验收标准（AC）
1. AC-16 勾选门禁
   1. Given 会话 `S1` 全部消息为已分析，会话 `S2` 至少一条未分析
   2. When 用户在训练入口勾选会话
   3. Then 仅 `S2` 可勾选，`S1` 显示不可勾选原因
2. AC-17 全量分析
   1. Given 会话 `S` 有历史消息与新增消息
   2. When 用户触发分析
   3. Then 分析输入包含 `S` 的完整对话上下文，并生成 `analysis_run_id`
3. AC-18 映射关系
   1. Given 一次分析运行完成
   2. When 查看训练计划详情
   3. Then 可追溯 `analysis_run_id -> training_plan_items -> message_id[]`；无价值时计划项数量可为 0 且有原因
4. AC-19 失败与缺失上下文
   1. Given 分析任务执行异常
   2. When 判定为普通失败
   3. Then 本次目标消息回滚为“未分析”
   4. When 判定为上文缺失
   5. Then 写入“已分析 + 无价值 + 上文缺失”
   6. When 判定为下文缺失
   7. Then 保持“未分析”，等待新增消息后可再次分析
5. AC-20 单条删除
   1. Given 会话 `S1` 尚未生成训练计划
   2. When 用户在会话分析状态页选择一条消息删除
   3. Then 删除成功，消息从会话视图移除，且审计日志可追溯
6. AC-21 已生成训练计划会话的删除禁用
   1. Given 会话 `S2` 已分析且存在训练计划项（`training_plan_items > 0`）

## 11. 本轮增量需求：AGENTS 角色/目标强约束（2026-02-27）
### 11.1 主题目标与价值说明
1. 目标：
   1. 会话入口不再让用户手工定义角色/目标，改为自动从所选 agent 的 `AGENTS.md` 提取。
   2. 将提取结果冻结为会话级策略，并在每一轮对话和训练决策链路中强制执行。
2. 价值：
   1. 降低配置负担与口径漂移风险。
   2. 提升“会话可控性 + 训练可追溯性”。
   3. 避免“展示已做、行为未变”的假闭环。

### 11.2 增量功能需求（FR）
1. FR-18 `AGENTS.md` 角色/目标自动提取
   1. 创建会话时，系统必须从所选 agent 目录 `AGENTS.md` 提取结构化字段：`role_profile`、`session_goal`、`duty_constraints`。
   2. 提取规则需可配置关键词（例如：`角色`、`角色定位`、`目标`、`职责边界`、`核心职责`）。
   3. 提取结果必须与 `agents_hash/agents_version/agents_path` 一并写入会话快照。
2. FR-19 会话级策略冻结与每轮注入
   1. 会话创建成功后，`role_profile/session_goal/duty_constraints` 必须冻结在该会话上下文中。
   2. 每次任务执行构造 prompt 时，必须注入该会话冻结策略，且不可被普通用户消息覆盖。
   3. 会话内允许“显式重设策略”的操作必须写审计，并默认关闭。
3. FR-20 严格执行门禁（对话 + 训练决策）
   1. 聊天回复必须符合会话冻结策略（角色边界、目标导向、禁行项）。
   2. 分析、计划、训练执行阶段也必须遵循同一策略，不得出现“聊天遵守、训练不遵守”。
   3. 若策略缺失或解析失败，系统必须 fail-closed：拒绝创建会话或拒绝执行任务并返回明确错误码。
4. FR-21 策略可见性与可追溯
   1. 对话区必须展示当前生效的会话策略摘要（默认简版，支持展开详情）。
   2. 每轮任务 trace 必须包含策略来源（`agents_path/hash/version`）与策略摘要快照。
   3. 训练事件流中必须包含 `policy_alignment` 字段（是否对齐、偏离原因）。
5. FR-22 身份/职责类问答的规范化回复
   1. 当用户询问“你是谁/你的职责是什么/你能做什么”及同义表达时，系统必须返回标准自我说明。
   2. 自我说明必须基于当前会话冻结策略生成，至少包含：
      1. `我是一名 <agent_name> agent`
      2. `我的职责/角色是 <role_profile + duty_constraints 摘要>`
      3. `我能 <session_goal 对应能力>`
   3. 若用户请求超出职责边界，自我说明后必须追加边界提示，不得直接越界执行。
   4. 该规范化回复同样适用于训练链路中的决策说明文本（避免出现与会话角色冲突的决策口吻）。
6. FR-23 角色/目标清晰度判定
   1. 系统必须对提取结果执行清晰度评分（`clarity_score`），范围 `0~100`。
   2. 清晰度评分至少覆盖 4 个维度：
      1. 完整性：是否同时存在角色、目标、职责边界。
      2. 具体性：是否包含可执行动词与边界描述（非空泛口号）。
      3. 一致性：角色、目标、职责之间是否冲突。
      4. 可追溯性：是否能定位到 `AGENTS.md` 原文证据行。
   3. 分级规则：
      1. `>=80`：清晰（可自动生效）。
      2. `60~79`：可疑（需人工确认）。
      3. `<60`：不清晰（阻断执行并要求补齐）。
7. FR-24 兜底人工确认弹窗
   1. 当 `parse_status=incomplete` 或 `clarity_score<80` 时，必须弹出确认对话框。
   2. 弹窗必须展示：
      1. 自动提取结果（角色/目标/职责摘要）。
      2. 原文证据片段。
      3. 风险提示（可能职责漂移）。
   3. 用户可执行动作：
      1. `确认并使用本会话`。
      2. `编辑后使用本会话`。
      3. `拒绝并终止会话创建/执行`。
   4. 确认与编辑行为必须写审计事件（操作者、时间、原值、新值、原因）。
8. FR-25 训练补齐闭环
   1. 对“人工确认/人工编辑”通过的兜底结果，系统必须生成“策略补齐训练任务”。
   2. 训练任务目标为推动对应 agent 后续补齐 `AGENTS.md` 标准章节（角色/目标/职责边界）。
   3. 补齐完成后，新会话优先使用 `AGENTS.md` 显式定义，兜底频率应可观测下降。

### 11.3 增量验收标准（AC）

   1. 选择 agent 后，入口状态必须进入 `analyzing_policy`，并显示运行中提示。

      3. 命中有效缓存时可直接进入门禁判定，不重复解析。

   2. 该模块默认折叠，并提供清晰展开入口。
   3. 展开后信息必须完整可复制，不得丢失审计字段。
6. FR-36 评分版本与校准留痕
   1. 系统必须记录评分版本号（如 `score_model=v2`）。
   2. 当评分规则或权重调整时，需写入变更日志并可追溯生效时间。
   3. 人工确认结果应可用于后续评分校准分析（仅用于机制改进，不覆盖原始审计记录）。
7. FR-37 职责边界结构化展示（职责与限制合并）
   1. `职责边界` 必须独立展示，不得将职责与限制散落在多个无结构段落中。
   2. 至少包含三类条目：
      1. 必须项（`must`）
      2. 禁止项（`must_not`）
      3. 前置条件（`preconditions`）
   3. 每条职责边界条目必须可追溯到 `AGENTS.md` 证据片段；无证据条目不得进入“自动生效”判定。
   4. 当职责边界缺失或冲突时，门禁至少进入 `confirm`，不得直接 `auto`。

### 13.3 增量验收标准（AC）
1. AC-36 首屏优先信息顺序
   1. Given 用户打开角色策略详情
   2. When 页面首屏渲染完成
   3. Then 首先看到 `角色/目标`，随后看到 `职责边界` 与 `门禁得分项`，并且 `门禁与来源` 与 `AGENTS.md 原文证据` 默认不展开

8. AC-43 职责边界门禁联动

## 14. 本轮增量需求：角色提取主链路切换为 Codex Agent（2026-03-01）
### 14.1 主题目标与价值说明
1. 目标：
   1. 将角色提取主路径从“本地规则解析 `AGENTS.md`”切换为“Codex Agent 执行提取”。
   2. 把提取执行链路做成可见、可追溯、可复盘的证据闭环。
2. 价值：
   1. 降低对 `AGENTS.md` 固定格式的强耦合依赖。
   2. 提升复杂语义提取的可用性。
   3. 让“提取是否执行、如何执行、结果是什么”对用户透明可核验。

### 14.2 增量功能需求（FR）
1. FR-38 Codex 提取主路径替换
   1. 角色提取必须由 Codex Agent 执行，不再以本地规则解析作为主路径。
   2. 标准命令语义为 `codex exec`（禁止使用错误别名 `codex -exec`）。
   3. 每次提取任务都必须显式传入目标 `AGENTS.md` 路径与输出契约。
2. FR-39 Codex 提示词模板规范化
   1. 系统必须维护可版本化提示词模板（含 `prompt_version`）。
   2. 提示词至少包含：
      1. 提取目标（角色、目标、职责边界）。
      2. 输出 JSON 契约（字段、类型、必填项）。
      3. 失败返回规范（错误码、原因、证据缺口）。
   3. 模板变更必须留痕并可追溯生效时间。
3. FR-40 Codex 输出交付契约
   1. Codex 输出必须可被系统稳定解析为结构化结果，至少字段：
      1. `role_profile`
      2. `session_goal`
      3. `duty_constraints`（数组）
      4. `parse_status`（`ok/incomplete/failed`）
      5. `parse_warnings`（数组）
      6. `evidence_snippets`（`role/goal/duty`）
      7. `clarity_score`
      8. `clarity_gate`（`auto/confirm/block`）
   2. 输出不满足契约时，必须标记 `parse_status=failed` 并阻断自动放行。
4. FR-41 角色与职责确认/兜底中的“分析链路可见性”
   1. 在确认/兜底详情中新增固定模块 `分析链路`，按顺序展示：
      1. 本次使用的提示词（可复制）。
      2. Codex 执行命令摘要（脱敏后）。
      3. Codex 原始输出文件位置（stdout/stderr/ref）。
      4. 结构化分析结果文件位置与内容摘要。
      5. 最终门禁判定（`ready/confirm/block`）及原因。
   2. 上述信息默认折叠，用户可按需展开，不得隐藏关键路径字段。
   3. 展开后必须展示完整链路内容，不得仅保留摘要：
      1. 提示词显示完整正文（支持复制）。
      2. Codex 输出至少提供完整原文查看入口（stdout/stderr），允许长内容滚动但不得静默截断。
      3. 结构化分析结果展示完整 JSON（含字段缺失/校验错误明细）。
5. FR-42 提取证据落盘与复盘
   1. 每次提取必须落盘证据包，最小文件集合：
      1. `prompt.txt`
      2. `codex-command.json`
      3. `codex-stdout.jsonl`
      4. `codex-stderr.log`
      5. `parsed-result.json`
      6. `gate-decision.json`
   2. 证据包必须关联 `session_id/agent_name/agents_md_path/agents_hash/prompt_version`。
   3. 证据包路径必须写入数据库可查询字段，并在 UI 可一键查看。
6. FR-43 失败策略（fail-closed）
   1. Codex 执行失败、超时、输出不符合契约时，默认 `block`。
   2. `block` 场景仅允许：
      1. 人工兜底编辑后继续。
      2. 终止创建会话。
   3. 禁止静默回退到“旧规则解析自动放行”。
7. FR-44 根路径与作用域一致性
   1. 提取执行时必须明确“工作区根路径”和“目标 `AGENTS.md` 可访问路径”。
   2. 配置的工作区根路径下必须包含 `workflow/` 子目录，系统据此构建默认运行上下文。
   3. 若目标工作区不在当前执行根授权范围内，必须返回明确错误码（如 `target_agents_path_out_of_scope`）。
   4. UI 必须提示“当前 root 与目标路径是否匹配”，避免隐式越界或隐式失败。

### 14.3 增量验收标准（AC）
1. AC-44 主路径切换生效
   1. Given 用户进入会话入口并选择 agent
   2. When 系统执行角色提取
   3. Then 提取来源标记为 `codex_exec`，且不再使用规则解析主路径作为放行依据
2. AC-45 提示词可见可复制
   1. Given 打开“角色与职责确认/兜底”详情
   2. When 展开“分析链路”
   3. Then 用户可看到本次提示词全文、`prompt_version`，并可复制，且无静默截断
3. AC-46 Codex 输出可定位
   1. Given 任意一次角色提取执行完成
   2. When 查看“分析链路”
   3. Then 可见 `stdout/stderr/parsed-result/gate-decision` 的文件路径与完整查看入口（默认折叠）
4. AC-47 结果契约校验
   1. Given Codex 返回输出
   2. When 系统执行契约校验
   3. Then 字段齐全时进入 `ok|incomplete`，字段缺失时进入 `failed` 且记录缺失原因
5. AC-48 fail-closed 生效
   1. Given Codex 执行失败或超时
   2. When 用户尝试创建会话
   3. Then 系统阻断自动创建，并仅提供“人工兜底/终止”两种动作
6. AC-49 根路径作用域提示
   1. Given 用户配置执行根路径为“包含 `workflow/` 子目录的工作区根路径”
   2. When 目标 `AGENTS.md` 不在授权作用域
   3. Then 系统返回 `target_agents_path_out_of_scope`，并在界面提示修复路径
7. AC-50 复盘证据完备
   1. Given 任意一条被阻断或人工兜底的会话创建请求
   2. When 事后复盘
   3. Then 可通过证据包完整还原“提示词 -> Codex 输出 -> 解析结果 -> 门禁决策”链路
