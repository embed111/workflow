# 详细设计-测试数据展示全局开关统一

## 1. 设计目标
1. 将“测试数据展示”收敛为唯一全局配置源，消除入口分裂。
2. 从“前端本地缓存驱动”升级为“后端持久化驱动”。
3. 保证会话入口、Dashboard、会话列表、训练中心四类视图口径一致。

## 2. 现状基线与问题定位
1. 会话入口存在独立开关：
   1. `showSystemAgentsCheck`（`workflow_web_server.py:2245`）。
2. 全局设置存在另一开关：
   1. `showTestDataCheck`（`workflow_web_server.py:2500`）。
3. 前端当前为“双状态 + 本地缓存”：
   1. `showTestDataCacheKey` 与 `showSystemAgentsCacheKey` 并存（`00_app_state_and_utils.js:7-8`）。
   2. `showSystemAgentsCheck` 与 `showTestDataCheck` 分别绑定独立事件（`05_training_center_and_bootstrap.js:1123-1149`）。
4. 数据查询依赖页面拼装参数：
   1. `withTestDataQuery(...include_test_data=...)`（`00_app_state_and_utils.js:812-815`）。
   2. 后端多路由按 query 参数单次解析（如 `workflow_web_server.py:12569,12601,12684,12735,12771`）。
5. 根因结论：
   1. 控制源不唯一 + 语义不一致（agent 可见性与数据查询分别控制）导致“看起来是全局，实际不是全局”。

## 3. 设计范围与非范围
1. 设计范围：
   1. 单一全局开关模型。
   2. 后端持久化读写。
   3. 前端状态收敛与入口控件收敛。
   4. 多模块联动刷新。
2. 非范围：
   1. 测试数据判定规则重写（沿用现有识别逻辑）。
   2. 多租户/多环境级配置隔离。
   3. 历史业务数据回写清洗。

## 4. 全局配置模型
1. 配置键：
   1. `show_test_data: bool`。
2. 真值来源：
   1. 后端运行时配置（持久化文件/配置存储），前端仅消费。
3. 默认值：
   1. 首次部署建议 `true`（沿用当前体验，避免突变）。
4. 作用域：
   1. 单实例全局（跨会话、跨浏览器）。

## 5. 后端设计
1. 配置读写函数（建议）：
   1. `current_show_test_data(cfg, state) -> bool`
   2. `set_show_test_data(cfg, state, requested: bool) -> tuple[bool,bool]`
2. 持久化策略：
   1. 复用现有 `load_runtime_config/save_runtime_config` 机制。
   2. 新增持久化键 `show_test_data`。
3. API 设计（需求级草案）：
   1. `POST /api/config/show-test-data`
      1. 入参：`show_test_data` 或 `showTestData`
      2. 出参：`show_test_data`、`previous_show_test_data`
   2. `GET /api/agents` 返回 `show_test_data`（用于前端启动同步）。
   3. `GET /api/status` 返回 `show_test_data`（用于状态校验与调试）。
4. 查询口径统一：
   1. 抽象统一解析函数（建议）：`resolve_include_test_data(query, cfg, state)`。
   2. 页面主路径默认使用全局配置值，不再要求前端在每个请求拼装 `include_test_data`。
5. 路由改造范围（最小）：
   1. `/api/dashboard`
   2. `/api/chat/sessions`
   3. `/api/training/agents`
   4. `/api/training/queue`
   5. `/api/training/trainers`
6. 失败处理：
   1. 配置写入失败返回 `500` + 明确错误码（如 `show_test_data_save_failed`）。

## 6. 前端设计
1. 状态模型收敛：
   1. 保留 `state.showTestData`。
   2. 删除 `state.showSystemAgents` 与其本地缓存键。
2. UI 收敛：
   1. 移除会话入口 `showSystemAgentsCheck` 与 `showSystemAgentsMeta`。
   2. 保留设置页 `showTestDataCheck` 作为唯一操作入口。
3. 事件与联动：
   1. `showTestDataCheck.onchange` 改为先调用 `POST /api/config/show-test-data`。
   2. 写入成功后刷新：
      1. agent 选择列表；
      2. sessions/workflows/dashboard；
      3. training center agents/queue。
   3. 写入失败时回滚勾选状态并提示错误。
4. 启动加载：
   1. bootstrap 不再用本地缓存初始化全局真值。
   2. 以 `/api/agents` 返回的 `show_test_data` 初始化，并覆盖本地旧值。
5. 查询参数治理：
   1. 页面不再依赖 `withTestDataQuery` 作为真实控制路径。
   2. `visibleAgents()` 改为基于 `state.showTestData` 统一过滤。

## 7. 迁移与兼容
1. 本地键迁移：
   1. 废弃 `workflow.p0.settings.showSystemAgents`。
   2. 旧键存在时仅用于一次性迁移参考，不再驱动行为。
2. 兼容期策略：
   1. 若后端暂未返回 `show_test_data`，前端临时按 `true` 兜底并提示“配置未同步”（过渡期可选）。

## 8. 状态机与时序
1. 开关切换时序：
   1. 用户点击开关 -> 前端发起 `POST /api/config/show-test-data` -> 后端持久化成功 -> 前端写状态 -> 前端批量刷新视图。
2. 启动时序：
   1. 页面启动 -> 拉取 `/api/agents` -> 读取 `show_test_data` -> 渲染设置开关与入口列表 -> 继续加载会话与训练数据。

## 11. 推断/假设与确认项
1. 推断/假设（置信度: 中）：
   1. 现阶段采用 runtime config 持久化已满足“全局配置”诉求，无需新增数据库表。
2. 推断/假设（置信度: 中）：
   1. 页面 API 查询参数中的 `include_test_data` 可逐步退场，不需要一次性删除所有兼容代码。
3. 已确认：
   1. 控制模型必须统一。
   2. 全局配置必须后端持久化。
4. 待确认：
   1. 首次部署默认值是否固定为 `true`（建议固定为 `true`）。
