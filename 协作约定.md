# 项目协作约定

更新时间：`2026-04-05`

## 1. 文件定位
- 本文件是当前 `workflow` 项目的顶层协作真相源。
- 用于约束当前单人维护阶段，以及后续扩展到多成员协作时的默认工作方式。
- 若与零散对话表述冲突，以本文件为准；`AGENTS.md` 作为治理入口引用本文件，但不再重复维护完整协作正文。

## 2. 当前阶段
- 当前项目仍处于单人维护阶段。
- 当前默认执行主体为当前 Codex。
- 当前默认开发主体固定为 `pm-main`。
- 当前阶段默认直接跟踪 `main` 开发、验证并推回 `../workflow_code/main`，不再维持 `dev/pm-main` 这类单人阶段分支。
- 当前阶段即使后续补进少量小伙伴，各自工作区的已验证代码也统一直接合入 `../workflow_code/main`，不额外拆长期并行开发分支。
- 后续若新增协作者，再按本文件扩展到各自独立的 `.repository/<developer_id>/` 工作区。

## 3. 仓库边界
- `../workflow_code/`
  - 唯一正式代码根仓。
  - 只放代码与代码历史，不承载 `.codex`、`AGENTS.md`、治理留痕或便捷 wrapper。
  - 默认只作为代码共享真相源与 Git 历史真相源，不作为任何成员的直接手改工作树。
- `../workflow/`
  - 当前 PM/治理壳仓。
  - 承载治理、记忆、留痕、运行态、顶层便捷入口与本地开发工作区。
- `.repository/<developer_id>/`
  - 本地临时代码开发工作区。
  - 实际代码修改、验证、提交、推送都在这里进行。

## 3.1 代码修改铁律
- 所有成员默认不得直接修改 `../workflow_code` 工作树。
- 所有成员默认不得把 `.running/prod`、`.running/test`、`.running/dev` 当成开发面。
- 所有正式代码改动必须先落到 `workflow/.repository/<developer_id>/`。
- 所有成员每次开始开发前，默认先从 `../workflow_code` 拉取最新代码到自己的 `.repository/<developer_id>/`，再开始改动；不得基于过期工作区直接开发。
- 本地开发工作区允许提交并推送到 `../workflow_code`；随后再由 PM 基于代码根仓版本执行部署。
- 若因线上止损不得不直接热修 `prod`，该改动只视为“现场临时补丁”，必须在下一步立即回放到对应 `.repository/<developer_id>/`，并推送回 `../workflow_code`；未完成回放前，不得继续在热修后的 `prod` 真身上叠加开发。
- `workflow` PM 仓根目录中的 `src/`、`scripts/` 若存在，应视为误放代码副本而不是合法长期真相源；同步完毕后应清理，避免与代码根仓双写漂移。

## 3.2 工作区同步第一责任
- `ahead/dirty` 从来不是抽象的“任务包状态”，而是某个具体 `.repository/<developer_id>/` 的发布边界状态。
- 哪个工作区产生了代码改动，哪个工作区就对这批改动的 `commit / push / 根仓同步收口` 负第一责任；这条责任不能仅因为版本计划里某个任务包写了别的负责人就被转移。
- “只允许在当前 `workspace_path` 内写文件”这类任务边界，不应被误解成“可以不把当前工作区的已验证提交 push 回根仓”；它限制的是不要直接去 `../workflow_code` 工作树手改代码，不豁免从当前工作区执行 `git commit / push` 回 `../workflow_code/main` 的收口责任。
- `pm` 负责编排、催收、兜底和跨工作区冲突处理，但默认不以另一个工作区的未推改动为基础继续开发；若要接管，也必须先让源工作区把已验证改动推回 `../workflow_code/main`，或者先导出 patch 再由接管方回放到自己的工作区并推根仓。
- 任一工作区只要出现“已改代码但尚未推回根仓”，后续跨工作区接力默认先处理这条根仓同步动作，再继续下一位成员的正式开发；不得把未推改动长期留在各自 `.repository/` 里口头交接。

## 4.1 生产环境 bug 修复口径
- 后续进入生产环境 bug 修复场景时，默认先由 `workflow_bugmate` 承担主执行。
- `workflow_bugmate` 的默认职责包括：复现问题、收口影响面、定位根因、整理修复建议、推进回归检查。
- 当需要提交真正可审阅的代码修复时，`workflow_bugmate` 默认在自己的 bugfix mirror 中改动，并导出 patch 供 `pm` 审阅、整合与发布。
- 当前 `pm` 保留协调、把关、兜底和复杂场景接管职责，尤其是：
  - 正式升级或回滚
  - 跨多个角色/工作区/部署链路的复杂修复
  - `workflow_bugmate` 未就绪、任务图阻塞或用户明确要求我直接处理的情况
- 若本节与其他默认成员说明冲突，以本节为准。

## 4.2 生产环境测试与缺陷流转口径
- 后续进入生产环境测试、升级后验收或修复后回归场景时，默认先由 `workflow_testmate` 承担主执行。
- `workflow_testmate` 的默认职责包括：`healthz` 检查、核心 API smoke、关键路径巡检、升级后验证、修复后回归、结构化测试结论输出。
- 当测试发现问题时，默认链路固定为：
  1. `workflow_testmate` 在生产环境提交缺陷
  2. `workflow_testmate` 通过生产环境触发修复 bug
  3. 修复任务默认路由给 `workflow_bugmate`
  4. `workflow_testmate` 在修复后继续做回归验证并向 `pm` 回报
- `pm` 负责跨角色协调、优先级把关、正式升级/回滚决策，以及异常场景接管。
- 若本节与其他默认成员说明冲突，以本节为准。

## 4.3 用户可见文案口径
- 面向你的创建角色文案、运行日志、界面标题、状态提示、派发摘要、测试结论和缺陷摘要，默认使用中文。
- `agent_id`、工作区目录名、脚本名、路径 token 等技术标识可以继续保留 ASCII，以兼顾稳定命名和脚本兼容性。
- 如果必须暴露技术字段，默认采用“中文说明 + 原字段/ID”的写法，而不是直接抛出整句英文提示。

## 4.4 质量检查到修复闭环口径
- 后续进入生产环境质量检查场景时，默认先由 `workflow_qualitymate` 承担主执行，负责巡检、smoke、证据采集与正式缺陷提报。
- 默认闭环固定为：
  1. `workflow_qualitymate` 执行质量检查并输出中文质量结论
  2. 命中问题后由 `workflow_qualitymate` 提交正式缺陷
  3. 修复任务默认路由给 `workflow_bugmate`
  4. `workflow_testmate` 或 `workflow_qualitymate` 继续执行回归验证并向 `pm` 回报
- `pm` 负责闭环编排、优先级把关、异常场景接管，以及定时巡检策略是否继续扩展到自动触发。

## 4.5 工程治理协作口径
- 工程治理的第一责任始终在 `pm`，但效率同样属于 `pm` 的职责；默认不追求“所有治理工作都由 `pm` 亲手完成”。
- 只要任务边界清楚、适合领域化沉淀，`pm` 应主动把工程治理/工程质量执行切片派给 `workflow_devmate` 等对应角色，让不同小伙伴在各自领域逐步成长为比 `pm` 更强的专家。
- `pm` 的核心职责是：定方向、定优先级、切批次、定验收标准、看跨泳道影响和做最终收口；不是把所有重构、整改、验证都长期堆在自己身上。
- `workflow_qualitymate` 给出的超大文件、设计冗余、危险实现或稳定性风险告警，默认不仅是观测结果，也应视为后续派给 `workflow_devmate / workflow(pm)` 的工程治理输入。

## 4.6 专业成长与修炼口径
- `pm / workflow_devmate / workflow_bugmate / workflow_testmate / workflow_qualitymate / workflow_ucdmate` 以及后续新增成员，默认都不只是“当前任务执行者”，还要把各自领域当成长线专业方向持续修炼，目标是逐步成长为本领域专家。
- 这条要求同样适用于 `pm` 本人。`pm` 除了负整体责任，也要持续提升自己在版本治理、发布边界、跨角色协作、质量收口与长期演进方面的专业能力，而不是只做任务分发。
- 各角色默认都应保持自主学习与修炼提高意识：结合真实缺陷、真实发布、真实巡检、真实重构和真实验收现场，持续沉淀方法手册、检查清单、经验卡、模板、案例库和判断准则。
- 角色的“学习/修炼”不应脱离实际项目主线；默认优先顺序仍是“先完成当前职责内的交付与收口，再把本轮经验回灌成可复用资产，并在后续任务中持续提纯”。
- `pm` 在编排任务时，默认不仅看“谁能做完”，也要看“这条任务是否适合帮助该角色在自己的领域继续长成专家”；避免长期把所有高价值成长型任务都收回到 `pm` 自己手里。

## 5. 默认协作顺序
1. 每个成员开始开发前，先同步 `../workflow_code` 最新代码到自己的 `.repository/<developer_id>/`。
2. 在 `.repository/<developer_id>/` 中开发、验证并跑门禁。
3. 当前阶段默认将通过验证的代码直接推回并合入 `../workflow_code/main`，让所有小伙伴的生效代码始终汇聚在同一分支。
4. 必要时让其他成员从 `../workflow_code` 再同步到自己的本地工作区，而不是互相复制工作区目录。
5. 生成或刷新 `prod` 升级候选。
6. 仅当用户明确要求“更新生产环境代码”或“部署生产环境”时，才执行正式环境升级。

### 5.1 分批推送规则
- 后续代码改动允许按“可独立验证的小批次”推进，不必等整轮大改全部完成后再统一推送。
- 只要当前这一部分改动已经完成对应测试或门禁验证，就应及时从开发工作区推回 `../workflow_code`。
- 未经过对应测试的本地改动，不得提前推回根仓。
- 默认目标是保持 `../workflow_code` 持续接收“小步、已验证、可回溯”的增量，而不是在 `.repository/<developer_id>/` 长时间积压大批未推改动。

### 5.2 跨工作区接力规则
- 若 `workflow_devmate / workflow_bugmate / workflow_testmate / workflow_qualitymate / workflow(pm)` 需要接力同一条代码主线，前一工作区必须先完成“最小验证 -> commit -> push 到 `../workflow_code/main`”。
- 若前一工作区暂时做不到直接推根仓，允许交付 `patch + 证据 + 阻塞说明` 作为临时中间产物；但接力方必须先把该 patch 回放到自己的 `.repository/<developer_id>/`、完成必要验证并推根仓，之后才算正式接棒。
- 不允许出现“`workflow_devmate` 改了一半留在自己的 `.repository/`，`pm-main` 或其他小伙伴再凭口头状态继续往前改”的常态协作。
- `V1-P2` 这类发布边界任务包的职责是定义规则和收口流程，不等于替所有工作区代持未推代码；真正的代码同步责任始终绑定到产生改动的那个工作区。

## 6. 启动与运行口径
- 用户提到 `./run_workflow.bat` 时，默认指 `../workflow/` 顶层便捷入口。
- 顶层 `run_workflow.bat` 只做便捷代理：
  - 优先启动 PM 顶层 `.running/prod`
  - 必要时回退到默认开发工作区启动脚本
- 正式代码仓 `../workflow_code` 不再保留 `run_workflow.bat`。
- 涉及 `7*24` 连续运行、主线唤醒、保底巡检、断链修复、空窗升级和“为什么没 token / 为什么看起来没在跑”的现场判断时，统一先读 `docs/workflow/governance/7x24连续运行机制.md`。
- 当前 `7*24` 的协作真相源以 `docs/workflow/governance/7x24连续运行机制.md` 为准；版本优先级与任务包推进仍回到 `docs/workflow/governance/PM版本推进计划.md`。

## 7. 推送、部署、生产环境解释
- `推回代码根仓` / `同步到代码根仓`
  - 默认解释为：从 `.repository/<developer_id>` 提交并推送到 `../workflow_code` 对应分支。
  - 不等于直接修改 `../workflow_code` 工作树。
- `推送最新生产环境`
  - 默认解释为“生成或刷新 `prod` 升级候选”。
  - 不等于直接覆盖生产环境。
- `更新生产环境代码` / `部署生产环境`
  - 才解释为允许执行正式环境升级或等效生产代码更新。
- 无明确指令时：
  - 默认允许 `test` 部署与 `prod` candidate 刷新。
  - 默认不允许直接覆盖 `prod`。

## 8. 默认验证与发布约束
- 只要存在实际代码改动，默认先在对应开发工作区执行：
  - `python scripts/quality/check_workspace_line_budget.py --root .`
- 工程质量提升默认同时允许“做加法”和“做减法”，但当命中超大文件、复杂度偏高或设计冗余告警时，优先考虑减法重构：删重复逻辑、压平多余包装层、收口不必要抽象与冗余配置，而不是只靠继续补新层或新模块。
- 验证策略默认分两层：
  - 单项测试优先使用真实链路，验证当前改动直接影响的关键行为。
  - 单项测试已经通过后，全流程串联回归默认允许使用 mock/stub，避免把同一条慢链路重复实跑多次。
- 若全流程验收主要目标是“确认编排、阶段衔接、状态流转与门禁仍成立”，应优先使用 mock 版全流程；只有用户明确要求、或本轮改动直接触达真实外部链路时，才追加真实全流程。
- 改动验证通过且门禁通过后，默认继续执行：
  1. 部署 `test`
  2. 生成或刷新 `prod` 候选
  3. 由用户手动升级 `prod`
- 更详细发布约束见：
  - `docs/workflow/governance/默认发布约束.md`

## 9. 常用命令口径
- 检查开发工作区状态：
  - `git -C .repository/<developer_id> status --short --branch`
- bootstrap/刷新本地开发工作区：
  - `python ../workflow_code/scripts/manage_developer_workspace.py --root .running/control/runtime/prod --workspace-root D:/code/AI/J-Agents bootstrap --developer-id <developer_id>`
- 启动正式入口：
  - 在 `../workflow/` 根目录执行：`.\run_workflow.bat`
- 启动开发工作区：
  - `powershell -NoProfile -ExecutionPolicy Bypass -File .repository/<developer_id>/scripts/launch_workflow.ps1 -OpenBrowser`
- 门禁验收：
  - `python .repository/<developer_id>/scripts/acceptance/run_acceptance_workflow_gate.py --root .repository/<developer_id> --host 127.0.0.1 --port 8098`
- `test` 部署：
  - `powershell -NoProfile -ExecutionPolicy Bypass -File .repository/<developer_id>/scripts/deploy_workflow_env.ps1 -Environment test`
- `prod` 直发：
  - 仅在用户明确要求时使用：
  - `powershell -NoProfile -ExecutionPolicy Bypass -File .repository/<developer_id>/scripts/deploy_workflow_env.ps1 -Environment prod -AllowDirectProdDeploy`

## 9.1 提交日志口径
- 当前项目所有参与开发的小伙伴，默认统一使用中文提交日志；这条规则不只适用于 `pm-main`，也适用于 `workflow_devmate / workflow_bugmate / workflow_testmate / workflow_qualitymate / workflow_ucdmate` 以及后续新增成员。
- 提交信息默认写成“改动对象 + 行为变化 + 目的/原因”的完整中文句子，避免只写过短缩写、只写英文关键词，或把上下文压得过于简略。
- 若出于工具链、统计或检索需要保留 `feat:`、`fix:`、`docs:`、`chore:` 等类型前缀，可以保留；但前缀后的主句仍默认使用中文，并保持完整可读。
- 默认不接受“我用中文、其他小伙伴用英文”的混乱状态；项目级提交口径应统一，便于你快速审阅整体历史、跨角色比对与长期治理。
