# `8090` 页面桌面截图诊断口径 - 2026-04-18

## 1. 定位
- 本文档定义 `workflow_ucdmate` 在只拿到桌面截图时，如何直接指出 `prod` 当前可见的设计问题。
- 这份口径服务于 `V4-R1`，目标不是替代交互验收，而是让“桌面截图 -> 可见设计问题 -> 实现批次建议”成为稳定入口。

## 2. 适用范围
- `8090` live `prod` 的桌面截图。
- 任务中心、版本看板、密集信息工作面、右侧详情栏、长任务反馈面板。
- 用户明确要基于截图判断“现在页面设计哪里有问题”的场景。

## 3. 不适用范围
- 需要点击后才能确认的交互真相。
- 依赖 DOM、日志或 API 才能确认的数据真相问题。
- 纯视觉偏好争论，例如“颜色喜不喜欢”。

## 4. 默认判断顺序
1. 先看首屏是否稳定。
   - 是否存在顶部遮挡、内容被裁切、初始滚动位置异常、首屏关键区块没有完整露出。
2. 再看信息结构是否误导。
   - 是否出现“导航入口长得像正文卡片”“两个同名块竞争注意力”“主线与支线同权重”等问题。
3. 再看扫读成本。
   - 是否存在右侧长文失控、字段堆叠、指标口径打架、卡片层级不清。
4. 最后看实现建议是否能最小切片。
   - 先给 `真 bug / 视觉歧义 / 文案与指标歧义` 三类，再给最小改动方向。

## 5. 默认输出格式
### 5.1 先给可见问题
- `真 bug`
  - 只写截图里已经能直接确认的可见问题，例如首屏被裁切、sticky 遮挡、初始滚动定位错误。
- `结构问题`
  - 只写截图里已经能看出的层级和工作面问题，例如 jump strip 像正文、失败诊断和主线焦点抢位。
- `扫读问题`
  - 只写截图里已经能看出的高密度阅读负担，例如右栏过密、指标缺解释、主次不分。

### 5.2 再给最小实现批次
- `批次 1`
  - 先修首屏稳定性和裁切问题。
- `批次 2`
  - 再修导航入口与正文卡片的视觉歧义。
- `批次 3`
  - 最后收口右栏长文和指标解释。

## 6. 当前桌面截图可直接识别的典型问题
- 首屏顶部内容被裁切或被 sticky 区域压住。
- jump strip 和正文 section 使用近似卡片模型，导致用户误判成“重复渲染”。
- 右侧详情栏长文本没有分段收口，首屏阅读压力超过主工作面。
- `并行agent数`、`活跃小伙伴` 这类指标缺少口径说明，导致“像冲突但不知道为什么”。
- 主线、失败池、定时入口如果同屏同权重，会让用户不知道“现在先看哪里”。

## 7. 当前禁止动作
- 不要因为只有截图，就把交互结论写成已验证事实。
- 不要把“我需要再点一下确认”伪装成“设计一定有 bug”。
- 不要只给抽象评价，例如“有点乱”“层级不太好”；必须落到可见对象和具体原因。

## 8. 与 `V4-R1` 的绑定
- 当前版本要求 `workflow_ucdmate` 在桌面截图输入下，默认先输出 `prod` 可见设计问题，再补结构化优先级和实现批次建议。
- 这条口径补齐前，`V4-R1` 不应被视为已经完成对 `8090` UCD 诊断入口的系统性收口。
