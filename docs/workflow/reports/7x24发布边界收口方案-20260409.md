# `7x24` 发布边界收口方案 - 2026-04-09

- active_version: `V1`
- task_package: `V1-P2`
- lane: `工程质量探测`
- lifecycle_stage: `变更控制`
- runtime_baseline: `prod=20260409-105430`

## 1. 问题定义
1. 当前 `7x24` 已经能持续推进任务，但还没有把“本地工作区长期 `ahead/dirty` 却迟迟不推根仓”视为必须优先收口的异常。
2. 结果就是：
   - 本地 `.repository/pm-main` 持续积压混合改动
   - 根仓 `../workflow_code/main` 长时间没有收到已验证增量
   - `V1` 的退出门槛“`小步已验证推根仓 -> test -> candidate`”无法恢复成常态
3. 这类问题不能只靠人工提醒；必须让 `7x24` 在满足特定条件时自动切换到“发布边界收口模式”。

## 2. 目标
1. 让 `7x24` 不只会继续开发，也会在本地边界失控时优先恢复根仓同步节奏。
2. 让“已验证的小步增量及时推回 `workflow_code/main`”成为常态，而不是例外。
3. 让后续多 agent 协作始终站在同一条根仓 `main` 主线上，而不是各自长期漂移。

## 3. 触发条件
满足以下任一条件时，`7x24` 下一轮默认切到“发布边界收口模式”：
1. `.repository/<developer_id>` 出现 `main...origin/main [ahead N]`，且 `N > 0`
2. `.repository/<developer_id>` 存在已验证但未推送的 dirty tracked 改动
3. 某一轮中途失败、退出或被打断，遗留 dirty/ahead 历史包袱
4. 当前版本计划明确把“恢复小步推根仓节奏”标成下一轮最高优先级

## 4. 进入收口模式后的执行顺序
1. 冻结改动面
   - 先停止继续扩当前本地功能面或体验切片
   - 除非 `prod` 主链断裂、升级止血或高优先事故，否则不新增同工作区实现任务
2. 盘点 release boundary
   - 记录 `git status --short --branch`
   - 记录 `git diff --stat`
   - 输出当前 `root_sync_state`
3. 切推送批次
   - 按“任务包归属 + 文件耦合 + 最小验证面”切成小批次
   - 每个批次必须可独立说明“改了什么、如何验证、推完会影响什么”
4. 逐批验证
   - 只跑命中该批次的最小验证
   - 不用“全量还没测完”作为拖延已验证批次推送的理由
5. 逐批推根仓
   - 在开发工作区提交
   - 立即推回 `../workflow_code/main`
   - 记录已推送批次、剩余未推送批次和阻塞原因
   - 只要本轮代码改动已经验证通过，就不允许把“已验证但未 push”的状态留给下一轮
6. 推送后再进入 `test / candidate`
   - 根仓同步恢复后，再按默认发布约束继续 `test -> candidate`
   - 不把本地工作区长期当成候选缓存区
7. 历史失败轮先收口
   - 若本轮是接手上一轮失败/中断留下的 dirty/ahead，第一优先级先处理这批历史问题
   - 在确认保留、切批、回滚或推根仓之前，不继续基于这批错误现场扩同工作区改动面

## 5. 当前推荐批次切法
1. 批次 A：`workflow gate / acceptance` 边界清理
   - 文件：`run_acceptance_workflow_gate.py`、`workflow_gate_probe_registry.py` 及相关 probe 脚本
   - 目的：先恢复门禁脚本可维护性和最小推送节奏
2. 批次 B：`V1-P1 / V1-P2` 后端真相与 finalize 收口
   - 文件：`graph_model_and_payloads.py`、`task_artifact_store_queries.py`、`task_artifact_store_run_runtime.py`、`dashboard.py` 与对应专项验收
   - 目的：先把后端真相与收尾逻辑收成一批可追溯提交
3. 批次 C：`V1-P14 / V1-P16` 前端体验切片
   - 文件：`index.html`、`index_training_loop_panels.css`、`assignment_center_render_runtime.js`、`assignment_center_events.js`、`app_status_and_icon_utils.js`
   - 目的：把固定工作面与 freshness 信号作为独立 UCD 批次推进
4. 归属不清或无关文件
   - 如 `logs/` 或零散旁路修改，先单独确认归属；未确认前不混进任何推送批次

## 6. 每轮 `7x24` 必须回写的状态
1. `root_sync_state`
   - `clean_synced`
   - `ahead_clean`
   - `ahead_dirty`
   - `diverged_or_unknown`
2. `ahead_count`
3. `dirty_tracked_count`
4. `untracked_count`
5. `push_block_reason`
6. `next_push_batch`

## 7. done definition
1. 当前工作区中，至少存在一个“已验证并已推回 `workflow_code/main`”的小批次
2. 本地 `ahead/dirty` 不再无限膨胀，而是持续下降
3. 版本计划与巡检总结里能明确说明：
   - 哪一批已推
   - 哪一批待推
   - 为什么待推
4. 成功轮不会把“已验证但未推根仓”的状态留给下一轮
5. 新一轮 `7x24` 不再默认继续扩写同一批 dirty 改动

## 8. 例外条件
1. 若 `prod` 主链断裂、ready 堆积假健康、升级卡死或真实运行异常，需要先止血
2. 止血完成后，下一轮立即回到发布边界收口模式，不允许再次长期漂移

## 9. 本轮结论
1. 当前问题不是“没有做事”，而是 `7x24` 还没有把“成功轮当轮推根仓、失败轮先收历史问题”变成和“补链”一样硬的执行动作。
2. 从现在开始，release boundary 失控本身就应被视为 `V1` 的一个现场异常，而不是普通 backlog。
