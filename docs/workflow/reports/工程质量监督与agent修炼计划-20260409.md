# 工程质量监督与 agent 修炼计划 - 2026-04-09

- active_version: `V1`
- task_package: `V1-P4 -> V1-P17`
- lane: `需求分析`
- lifecycle_stage: `形成基线`
- runtime_baseline: `prod=20260409-105430`

## 1. 本轮判断
1. 仅靠我一个人做全维工程质量判断，天然会有盲区；更合理的做法是让不同 agent 在各自领域做到顶尖，再通过协作闭环提升整体质量。
2. `workflow_qualitymate` 不能只停留在 `smoke / 巡检 / 提 bug` 这一层，而应该升级成“多维工程质量监督者”。
3. 当前 `训练优化` 模块还达不到“稳定承载 agent 能力成长”的标准，因此短期不把“修炼提高”能力押在现有训练优化实现上。
4. 短期先用 `7x24 + 每日一次自主学习提升 + 角色资产沉淀` 的方式跑起来；长期再把 `训练优化` 升级为正式的 `修炼提高` 模块。

## 2. `workflow_qualitymate` 的职责边界
1. 架构质量监督
   - 关注模块职责是否清晰、边界是否收口、是否存在跨层耦合和真相源分叉。
   - 输出物应是架构热点清单、重构优先级和架构风险说明，而不是泛泛评价。
2. 超大文件与可维护性监督
   - 持续盯行数门禁命中项、热点大文件、重复逻辑和“能跑但不好维护”的代码块。
   - 输出物应是拆分建议、热点排序和“先拆哪里、为什么”的明确结论。
3. 设计模式与设计合理性监督
   - 关注是否误用模式、过度堆抽象、职责漂移、临时补丁长期化。
   - 输出物应是“当前设计是否合理、该不该重构、推荐模式是什么”的工程判断。
4. 危险函数与高风险写法监督
   - 关注危险 IO、宽泛异常吞噬、破坏性命令、无边界路径写入、容易误判状态的逻辑。
   - 输出物应是高风险函数清单、风险等级和替代建议。
5. 系统稳定性监督
   - 关注 7x24 接力、stale recovery、upgrade gate、schedule/dispatch handoff、异常恢复与 fail-closed 行为。
   - 输出物应是稳定性缺口、现场证据和建议路由给 `workflow_bugmate / workflow_devmate` 的任务。
6. 系统性能监督
   - 关注慢查询、大 IO、trace 体积、页面卡顿、轮询噪音、浏览器端信息密度过高等问题。
   - 输出物应是性能热点、观测证据和优先级建议。
7. 监督结果的闭环职责
   - 不只负责“看见问题”，还要把问题沉淀成 checklist、报告、缺陷、任务包或门禁建议。
   - 与 `workflow_bugmate / workflow_devmate / workflow_testmate / workflow(pm)` 形成“发现 -> 定级 -> 派发 -> 回归 -> 经验沉淀”的闭环。

## 3. 短期方案：每日一次自主学习提升
1. 频率
   - 短期先让每个核心 agent 每天执行 `1` 次自主学习提升：`workflow_qualitymate / workflow_devmate / workflow_bugmate / workflow_testmate`。
2. 输入材料
   - 当日版本计划
   - 当日失败任务与回归结果
   - 热点代码 diff 与 line budget 报告
   - 运行态 incident、性能热点、升级异常、任务中心现场问题
   - 各自领域的历史经验资产
3. 学习输出
   - 至少更新 `1` 条领域经验或方法卡
   - 至少补 `1` 条专项 checklist / 审核清单 / 风险规则
   - 至少给出 `1` 条下一轮可执行的改进建议或任务建议
   - 必须把“今天学到了什么、以后怎么避免再踩”写入角色资产或当日日记
4. 质量小伙伴的专项学习重点
   - 架构热点图
   - 超大文件和职责漂移清单
   - 危险函数与高风险模式观察清单
   - 稳定性/性能周边结论模板
   - 质量结论如何更好路由到 bug/dev/test
5. PM 的编排职责
   - 我负责给每日学习留出版本窗口、限制范围、收口产物路径，并把真正值得进入版本计划的结论提炼回主线。
   - 我不会把每日学习变成空泛复读，而是要求它必须回写到角色资产、经验卡或下一轮任务建议。

## 4. 长期方案：把“训练优化”升级成“修炼提高”
1. 名称与定位
   - 长期不再把它理解成单纯“训练优化”，而是理解成 agent 的“修炼提高”工作台。
2. 目标
   - 让每个 agent 能围绕自己的专业方向持续提高，而不是只追加零散 prompt 或临时修补规则。
3. 修炼方法主链
   - 能力差距诊断：当前短板是什么，为什么影响交付。
   - 领域方向选择：架构、稳定性、性能、测试、缺陷修复、UCD 等不同修炼方向要分轨管理。
   - 任务化修炼：每次修炼都要落成真实任务，而不是纯会话空谈。
   - 定向评测与防退化：既验证新增能力，也检查旧能力没有明显退化。
   - 资产回写：把修炼结果回写到 `AGENTS.md`、角色画像、方法手册、经验卡和能力版本记录。
   - 发布评审：只有通过门禁的修炼成果才进入正式生效。
4. 模块能力要求
   - 支持按角色维护“修炼方向 / 能力树 / 当前等级 / 下一阶段目标”
   - 支持修炼任务、证据、评测、回写、发布评审一体化闭环
   - 支持跨 agent 协作修炼，例如质量小伙伴产出方法，开发/缺陷/测试小伙伴吸收后各自演练
   - 支持可追踪的成长记录，而不是训练完成后只剩一条任务状态

## 5. 版本落点
1. `V1`
   - 先把 `workflow_qualitymate` 的职责从“巡检”扩到“多维工程质量监督”。
   - 先让四个核心 agent 跑起“每日一次自主学习提升”。
2. `V5`
   - 在工程质量基线稳定后，把 `训练优化 -> 修炼提高` 作为正式功能方向推进。
   - 届时不只做界面改名，而是要把“诊断、修炼、评测、回写、发布”设计成完整方法与系统。

## 6. 本轮结论
1. 你的判断是对的：单个 agent 很难面面俱到，但多个专业 agent 可以通过协作接近高质量产出。
2. 当前最务实的动作不是先强推现有训练优化，而是先让 agent 每天有节奏地学习、沉淀、回写。
3. 只要这条短期机制跑顺，长期“修炼提高”模块就不会是空壳，而是对现有有效做法的系统化承接。
