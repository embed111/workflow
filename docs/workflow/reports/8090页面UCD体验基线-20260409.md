## 2. 当前体验问题与代码落点
1. 工作台默认先落到“任务图”，而“任务看板”是隐藏副视图，不利于连续盯盘。
   - `index.html` 已把 `任务看板 / 任务图` 做成二选一 tab，且初始 `aria-selected='true'` 落在 `任务图`，`assignmentWorkboardSection` 默认 `hidden`。
   - `assignment_center_state_helpers.js` 只是在 `graph/workboard` 两个 section 之间切换显隐，没有做“主线优先视图”或“按状态自动切换”的策略。
2. 当前 workboard 主体仍是“摘要卡 + 小伙伴分组卡 + 最近 schedule”串行纵向堆叠，适合巡检，不适合长时间连续工作。
   - `assignment_center_render_runtime.js` 的 `renderAssignmentWorkboard()` 仍按 `summary -> groups -> schedulePreview` 顺序直接拼接 HTML，并把每组 `running / queued / failed` 各自截断到最多 `4` 条。
   - `dashboard.py` 的 `_workboard_payload()` 也只给 workboard 提供分组计数、节点简表和 schedule preview，没有为“主线连续工作”单独准备更强的结构化摘要。
3. 现有样式明确把看板、详情和审计都收口为单列纵向容器，信息密度一高就只能靠长滚动。
   - `index_training_loop_panels.css` 里 `.assignment-workboard-panel` 使用 `overflow:auto`，`.assignment-workboard-body` 是 `flex-direction: column`。
   - 同文件里 `.assignment-detail-grid` 是单列，`.assignment-audit-list` 也是单列纵向列表，这解释了为什么当前看板和详情都容易越看越长，而不是形成固定工作面。
4. 刷新反馈仍然是局部提示，缺少统一的“新鲜度 / 重连 / 后台仍在工作”表达。
   - `assignment_center_state_helpers.js` 当前只把历史区域状态写成“历史状态待刷新 / 正在刷新任务图 / 历史已全部显示”。
   - `app_runtime_state_helpers.js` 只在执行设置元信息里显示“最近更新 / 刷新方式 / 断线兜底”，这更像配置说明，不是任务中心主视图的实时状态提示。
5. 当前 API 已经能给 workboard 提供运行计数兜底，但还没有把“页面需要的刷新感知”做成一等字段。
   - `dashboard.py` 已经把 `running_task_count` 与 workboard fallback 对齐，解决了“upgrade/status 假空闲”的工程问题。
   - 但当前 payload 仍缺统一的 `fetched_at / freshness / connection_state / workboard_focus_card / long_run_stage_summary` 这类 UCD 直接可用字段，所以前端只能自己拼零碎提示。

## 5. 验收口径
1. 不切换 tab 的情况下，用户进入任务中心后应能直接看到“当前主线是谁、接棒是谁、是否需要介入”。
2. 任务看板、任务图、详情区、历史区都必须在固定窗口内完成滚动，不再依赖整页长滚动承载核心信息。
3. 页面头部必须给出统一的“最近更新时间 / 刷新方式 / 连接状态 / 数据是否过期”表达。
4. 对长时间 running 的主线或保底节点，必须提供阶段感或等待建议，而不是只显示一个 `running` 状态词。
5. 当前主线与保底巡检必须在 workboard 里一眼可区分，不能继续与普通 helper 失败历史混在同一视觉层级。
6. 新改造不能破坏现有 `dashboard -> workboard fallback -> runtime-upgrade gate` 的真实运行计数一致性。

## 6. 本轮暂不做

2. 本轮不把 `V1-P13` 扩写成新的长期角色体系；`workflow_ucdmate` 只有在路线稳定后再考虑孵化。
3. 本轮不覆盖 `V1-P1 / V1-P2` 的 release boundary 义务；体验迭代仍需服从当前版本计划与发布约束。
