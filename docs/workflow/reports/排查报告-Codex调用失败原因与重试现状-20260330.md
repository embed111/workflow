# 排查报告-Codex调用失败原因与重试现状-20260330

- 状态：已完成首轮排查，可直接作为整改输入
- 排查对象：`../workflow` 中所有真实发起 `Codex` 请求的功能链路
- 分析方法：
  1. 第一性原理：先只找“真正调用 Codex 的链路”，不把普通页面报错、配置报错混进范围。
  2. 逻辑学分析：分别核对“失败原因是否可见”和“是否存在可执行重试动作”，避免把两者混为一谈。

## 1. 结论先行
1. 已确认 `../workflow` 当前至少有 5 条真实 `Codex` 请求链路：
   1. 策略提取 / `AGENTS.md` 分析
   2. 会话任务执行中心（`/api/tasks/execute`）
   3. 创建角色分析对话
   4. 发布评审 / 确认发布兜底
   5. 任务中心真实执行（`execution_provider=codex`）
2. 当前不是“完全没有失败提示和重试”，而是“各链路能力不一致”：
   1. 有的链路失败原因很强，但重试入口弱或不统一。
   2. 有的链路已经能重试，但失败原因仍停留在原始字符串或泛化提示。
3. 发布评审链路成熟度最高；策略提取、会话任务执行、任务中心真实执行居中；创建角色分析对话当前仍偏“隐式自动重试”。
4. 整改方向应收口为“统一 `Codex` 调用失败合同 + 各链路适配”，而不是每个模块各自补一段提示词或按钮。

## 2. 范围与排除项
1. In Scope：
   1. 由服务端真实发起 `Codex` 请求，并把结果回写到页面、任务、评审或会话中的链路。
   2. 失败原因展示、重试入口、重试幂等、执行证据可见性。
2. Out of Scope：
   1. 与 `Codex` 无关的普通接口失败。
   2. 泛化 UI 文案优化但不改变失败语义与重试能力的改动。
   3. 本工作区直接修改 `../workflow` 实现代码。

## 4. 关键证据
1. 策略提取链路：
   1. `policy_fallback_service.py` 已显式写入 `policy_error`，并把 `prompt/stdout/stderr/result/gate` 路径串到分析链中。
   2. `policy_session_actions.js` 已能展示门禁、解析告警和文件路径。
2. 会话任务执行链路：
   1. `app_navigation_and_event_binding.js` 已存在 `retryBtn`，并调用 `runTask(true)`。
   2. `policy_confirm_and_interactions.js` 已给出“重试上一轮”提示，但很多失败仍停留在泛化文案层。
3. 创建角色链路：
   1. `analyst_bridge.py` 已有多类 `Codex` 失败分支。
   2. `role_creation_message_runtime.py` 与 `training_center_role_creation.js` 目前把重试主要做成“继续发送自动合并重试”。
4. 发布评审链路：
   1. `release_review_execution.py` 已具备较完整的失败分类和中文建议文案。
   2. `release_confirmation.py` 已具备失败诊断后自动重试一次的兜底语义。
5. 任务中心真实执行链路：
   1. `task_artifact_store_run_runtime.py` 已写入 `failure_reason` 与 `stdout/stderr/result` 产物。
   2. `assignment_center_render_runtime.js` 已展示 `失败原因`，并支持 `重跑任务`。

## 5. 共性缺口
1. 失败字段未统一：
   1. 有的叫 `policy_error`；
   2. 有的叫 `error`；
   3. 有的只剩 `failure_reason` 文本；
   4. 前端难以稳定复用统一展示卡。
2. 失败码与用户文案未统一：
   1. 同样是超时，有的显示标准码，有的显示自由文本；
   2. 同样是执行退出，有的保留 `exit`，有的只剩“执行异常”。
3. 重试语义未统一：
   1. 有的是按钮重试；
   2. 有的是重新进入发布评审；
   3. 有的是继续发消息自动重试；
   4. 前端没有统一的 `retry_action` 协议。
4. 证据可见性未统一：
   1. 有的能下钻看到 `prompt/stdout/stderr/result`；
   2. 有的只有失败摘要；
   3. 用户无法稳定知道“这次失败去哪里看证据”。
5. 幂等边界未统一：
   1. 有的链路已有重跑/重试，但未形成统一“本次重试不会生成重复有效链”的产品约束。

## 6. 整改优先级建议
1. P0：统一后端失败合同
   1. 为 5 条链路补齐统一字段：
      1. `failure_code`
      2. `failure_stage`
      3. `failure_message`
      4. `retryable`
      5. `retry_action`
      6. `trace_refs`
      7. `attempt_count`
   2. 允许保留链路原始字段，但前端统一只消费标准合同。
2. P1：统一前端失败卡与重试动作
   1. 所有 `Codex` 链路失败时都显示同类失败卡。
   2. 明确展示：失败原因、失败阶段、是否可重试、重试按钮、证据入口、下一步建议。
3. P1：逐链路补显式重试
   1. 策略提取：显式“重试分析”。
   2. 创建角色：显式“重试本轮分析”，保留“继续发送自动重试”作为补充。
   3. 会话任务执行、任务中心真实执行、发布评审：沿用现有按钮，但改为走统一动作合同。
4. P2：统一幂等与指标
   1. 对重试次数、最近失败码、重试结果做统计。
   2. 补充“重试不生成重复有效链”的回归验收。

## 7. 推荐验收口径
1. 5 条链路都必须满足：
   1. 失败时可见明确失败原因；
   2. 若允许重试，则有显式重试入口；
   3. 若不允许重试，则明确说明原因与下一步动作；
   4. 可下钻查看本次失败对应的执行证据或路径。
2. 页面不得在后端已返回标准失败码时仍显示泛化文案：
   1. 不允许继续只显示“请求未成功”“执行出现异常”。
3. 重试动作必须可追溯：
   1. 至少能看到第几次尝试、最近一次尝试时间、重试后结果。
4. 重试动作必须幂等：
   1. 不得因重复点击或刷新生成第二条并存有效执行链。
