## 1. 当前判断
- 之前完成的 `schedule trigger -> assignment node -> dispatch -> runtime/status` 只是一条高风险主链收口，不等于全仓问题已解决。
- 当前全仓的主要问题不是“文件不够小”，而是业务主链被拆散到 service、`*_parts`、runtime helper、API readback 和 acceptance support 之间，读写真相和状态归属不够集中。
- 代码量偏大也不是单纯行数问题，而是大量兼容、fallback、重复测试支撑、manifest 装配和 wrapper 层一起抬高理解成本。

## 3. 根因
1. 行数门禁解决了“单文件过大”，但很多拆分是技术性横切，不是领域边界收口。
2. `exec_local_parts(...)` 把 part 执行进同一 module globals，短期降低单文件行数，长期放大隐式依赖和排障成本。
3. 同一业务真相在 schedule、assignment、runtime-upgrade、project、dashboard/status 等读面分别投影，导致修一处漏一处。
4. acceptance 资产按单次缺陷持续复制支撑函数，形成大量重复的 `wait_health / call / assert / fixture / Edge capture`。
5. 兼容/fallback 分支缺少明确退出条件，历史补丁层不断堆叠。

## 4. 目标架构
`V13` 的目标不是继续“拆”，而是把系统收成更少但更清楚的核心结构：

| 层 | 责任 | 规则 |
| --- | --- | --- |
| `api/presentation` | 请求解析、响应组装、页面资源 | 不承载业务状态机 |
| `application` | 用户动作对应的 use case / facade | 每条主链只有一个入口 |
| `domain` | 状态机、合同、领域规则 | 显式 import，不依赖 manifest globals |
| `repository` | 文件/SQLite/运行态读写 | 只做存取，不拼业务真相 |
| `projection` | read model 和 UI/API 投影 | 多个读面消费同一投影 |
| `infra` | 进程、路径、部署、外部命令 | 与业务规则隔离 |
| `acceptance_support` | 验收支撑库 | 复用 fixture/call/assert，不复制脚手架 |

核心设计口径：
- 一个业务主链只允许一个 application facade。
- 一个状态真相只允许一个 owner projection。
- 新增 split 必须有 owner domain、公开入口、禁止反向依赖和 probe。
- 优先删除 wrapper、重复 support、过期 compat，而不是继续新增抽象层。

## 5. V13 根治范围
`V13` 作为全仓逻辑边界与冗余实现根治版，退出门槛按硬指标验收：

| 门槛 | 当前基线 | V13 退出目标 |
| --- | ---: | ---: |
| `exec_local_parts` 核心业务模块 | `13` | 核心域清零；仅允许非核心 legacy adapter 临时保留且有删除计划 |
| `assignment_service_parts` | `37` | 收到不超过 `12` 个有明确 owner 的 domain/application/repository/projection 模块 |
| `server/services` 行数 | `66554` | 至少减少 `25%`，且关键主链调用深度下降 |
| `scripts/acceptance` 行数 | `64772` | 至少减少 `20%`，重复支撑函数收口到公共 support |
| 顶部超长核心文件 | 多个 `900+` 行 | 核心业务文件不超过 `650` 行；验收大脚本必须拆成 shared support + case |
| 新增 `*_parts` | 无门禁 | 默认禁止；例外必须 review approve 并写退出条件 |
| 合入 review | 无专职角色 | `workflow_reviewmate` 默认先审，不通过不得合入 |

## 6. 分批策略
1. `R1 架构地图与删除清单`：冻结全仓 domain map、公开入口、重复实现清单、必须删除/合并清单。
2. `R2 reviewmate 合入门禁`：让 review 角色、记录模板和打回条件进入日常合入流程。
3. `R3 assignment/schedule/runtime truth 内核化`：把当前最高风险链路从 part globals 迁到显式 use case、domain contract 和 read projection。
4. `R4 role/project/governance 主链收口`：收口角色创建、项目创建/点火、PM 版本读面的入口和真相归属。
5. `R5 acceptance 去重与 fixture 内核`：把重复验收支撑库合并，删除一次性脚手架和重复 browser/http helper。
6. `R6 frontend surface 减法`：前端不再只靠 manifest 切片保命，把工作面按稳定 view model 和动作边界重组。
7. `R7 删除 legacy/fallback 债务`：对保留的 compat/fallback 写到期条件，逐项删掉已无价值分支。
