# 7x24 连续运行机制

## 1. 文档定位
- 本文件是 `workflow` 当前 `7*24` 连续运行机制的专项真相源。
- 目标是把“系统现在为什么能连续跑、靠什么接力、什么时候算真在跑、什么时候只是服务活着、升级时怎么处理”写清楚。
- 本文件描述的是当前实现事实与协作注意事项，不替代运行态数据；实际现场仍以 `prod` API、任务图文件、`run.json`、`events.log` 为准。

## 2. 当前机制一句话
- 当前 `7*24` 的实现，不是“单个永不退出的大循环”，而是：
  `单例全局任务图 + 主线滚动一次性 schedule + 保底巡检一次性 schedule + 文件真相恢复 + 空闲窗口升级`。

## 3. 当前核心组成

### 3.1 单例全局任务图
- 当前 `7*24` 只认一张任务图，不再拆多张并行主图。
- schedule 命中后，节点统一落到 `任务中心全局主图`。
- 这条约束的目的很直接：所有主线、保底、小伙伴协作节点都在同一张图里观察、排队、接力，避免“看板 A 在跑、真实执行却在看板 B”的漂移。
- 当前代码真相主要在：
  - `.repository/pm-main/src/workflow_app/server/services/schedule_service.py`
  - 关键标识：`SCHEDULE_ASSIGNMENT_GRAPH_NAME = "任务中心全局主图"`
  - 关键标识：`SCHEDULE_ASSIGNMENT_GRAPH_REQUEST_ID = "workflow-ui-global-graph-v1"`

### 3.2 两条核心 schedule
- 主线 schedule：
  - 名称：`[持续迭代] workflow`
  - 作用：推动 `workflow(pm)` 继续做版本推进，不允许空转。
  - 预期产物：`continuous-improvement-report.md`
- 保底 schedule：
  - 名称：`pm持续唤醒 - workflow 主线巡检`
  - 作用：在主线失败、断链、卡住或需要补链时兜底巡检，确保至少还有一个未来入口。
  - 预期产物：`workflow-pm-wake-summary`
- 这两条都不是长期固定 cron，而是“滚动一次性 schedule”。
  - 当前一轮收尾后，再把下一轮一次性时间点写回去。
  - 这样做的好处是：每轮都能带上最新上下文，而不是死板重复旧 prompt。

### 3.3 主线与保底的时间规则
- 主线成功后：
  - 下一轮主线默认 `+15 分钟`
  - 优先级默认 `P2`
- 主线失败后：
  - 下一轮主线默认 `+30 分钟`
  - 优先级默认 `P1`
- 保底巡检：
  - 默认挂在主线下一轮时间之后 `+30 分钟`
- 当前代码真相主要在：
  - `.repository/pm-main/src/workflow_app/server/services/assignment_service_parts/assignment_self_iteration_runtime.py`
  - 关键常量：
    - `ASSIGNMENT_SELF_ITERATION_SUCCESS_DELAY_MINUTES = 15`
    - `ASSIGNMENT_SELF_ITERATION_FAILURE_DELAY_MINUTES = 30`
    - `ASSIGNMENT_PM_WAKE_DELAY_MINUTES = 30`

### 3.4 schedule worker 的工作方式
- schedule worker 每分钟扫描一次命中窗口，负责把“该触发了”快速落盘。
- 命中后先写 `trigger_hit`，然后后台继续做：
  - 建单
  - 建节点
  - 自动派发
  - 状态回写
- 若同一条 schedule 在旧节点还停留在 `pending/ready/queued` 时再次命中，系统默认只保留最新这条 pending trigger；旧的待执行节点会被删除，旧 trigger 会标记成 `superseded`，避免 backlog 永远排不完；已经 `running` 的旧节点则保持不动。
- 同时每 `8` 秒做一次最近 trigger 的恢复扫描，补偿：
  - 进程重启
  - trigger 半断裂
  - 已建单但没回写
  - 已 ready 但未继续处理
- 当前代码真相主要在：
  - `.repository/pm-main/src/workflow_app/server/services/schedule_service.py`
  - 关键常量：
    - `SCHEDULE_WORKER_INTERVAL_SECONDS = 8`
    - `SCHEDULE_TRIGGER_RECOVERY_START_LIMIT_PER_PASS = 1`

### 3.5 真实执行链
- 节点进入 `ready` 后，由 assignment 调度链继续派发。
- 真实执行不是页面状态，而是 run 落盘：
  - `run.json`
  - `events.log`
  - `stdout.txt`
  - `stderr.txt`
  - `result.json`
- 当前执行超时已经从固定短超时改为“无进展超时”：
  - 默认按活动超时 `7200s`
  - 不是简单的墙钟 `1200s`
- 当前也会对瞬时 `Codex` 流断开做一次自动重试，避免把可恢复抖动直接记成真失败。
- 当前代码真相主要在：
  - `.repository/pm-main/src/workflow_app/server/services/assignment_service_parts/task_artifact_store_run_runtime.py`

### 3.6 收尾后的接力机制
- 一轮 run 收尾后，系统会继续做几件事：
  - 把节点终态写回任务图
  - 写结果文件与 audit
  - 追加工作区今日日记
  - 续挂下一轮主线 schedule
  - 续挂保底巡检 schedule
  - 若 `prod` 已空闲且存在更新 candidate，则可尝试触发正式升级
  - 若当前不抑制派发，则继续 `dispatch next ready`
- 这就是为什么 `7*24` 的连续性，不是靠一个节点永远跑，而是靠“每轮收尾都保证下一棒已经排上”。
- 当前还有一个实现细节需要明确：
  - 若本轮收尾是由 `stale recovery / 文件真相补终态` 触发的，系统会抑制这次收尾内部的 follow-up dispatch，避免在外层 `dispatch/reconcile` 已持锁时再次递归派发导致卡死。
  - 这类场景的真实接棒由外层 dispatch 或 schedule worker 继续完成，不再由补终态收尾自己二次递归。

### 3.7 升级链
- 当前 `prod` 升级不是“带着 running task 无感切换”。
- 当前只能做到：
  - `running_task_count = 0`
  - 且 `/api/runtime-upgrade/status.can_upgrade = true`
  - 才安全 `apply`
- 升级方式有两类：
  - 手工或任务内 loopback 调 `/api/runtime-upgrade/apply`
  - 空闲 watcher 等到空窗后自动 apply
- 当前 watcher 脚本真相在：
  - `.repository/pm-main/scripts/apply_prod_candidate_when_idle.py`
- 当前无痛升级限制的专项设计说明在：
  - `docs/workflow/design/详细设计-prod无痛升级与连续运行保障.md`

## 4. 当前 7*24 的真实运行步骤
1. `schedule worker` 到点命中 `[持续迭代] workflow` 或 `pm持续唤醒 - workflow 主线巡检`。
2. 先把 `trigger_hit` 落盘，保证“命中过”这件事可追溯。
3. 后台线程在单例全局任务图里创建或恢复对应节点。
4. 节点进入 `ready`。
5. 调度器在并发槽允许时把 `ready` 节点派发为真实 run。
6. run 在工作区里执行，并持续写 `run.json/events.log/result.json`。
7. 收尾阶段写终态、写 audit、补记今日日记。
8. 收尾阶段继续续挂下一轮主线和保底，必要时申请空闲升级。
9. 若当前还有其他 `ready` 节点，系统继续派发下一条。
10. 这样形成“当前 running + 下一条 ready/future”的接力。

## 5. 怎样判断 7*24 是不是真的在跑

### 5.1 不能只看 token
- token 在涨，不一定代表主线健康，可能只是某个 helper 在跑。
- token 不涨，也不一定代表系统死了，可能只是：
  - 当前没有真实 running
  - 只有 `ready/future`
  - 正处于升级空窗

### 5.2 不能只看“服务活着”
- `healthz` 正常，只说明 Web 还活着。
- 不等于主线已经被派发。

### 5.3 当前推荐判断顺序
1. 看 `/healthz`
   - 确认服务在不在
2. 看 `/api/status`
   - 确认 `running_task_count / queued_task_count / assignment_workboard_summary`
3. 看 `/api/runtime-upgrade/status`
   - 确认当前是否被 running 槽挡住升级，还是已经空闲
4. 看 `/api/schedules`
   - 确认主线和保底是否还有 future/最近结果
5. 直读任务图和 run 文件
   - `nodes/<node>.json`
   - `runs/<run>/run.json`
   - `runs/<run>/events.log`

### 5.4 当前几种典型现场
- `running_task_count=1`，且 `run.json.status=running`、`events.log` 有 `provider_start/turn.started`
  - 这才是真正在执行
- `running_task_count=0`，但有 `ready`
  - 这是“服务在线，任务待派发”，不是正在吃 token
- `running_task_count=0`，有 `future`
  - 这是“下一轮已经排上”，但当前没有执行
- `active_agent_count=1`，workboard 却 `0 running`
  - 这不算主线在跑，只能说明当前还有活跃视图或 future 计划
- `running_task_count=0`，`workflow` 存在已到时的 `ready` 节点，且 recent trigger/message 出现 `assigned agent already has running node`
  - 这是“假健康/接力异常”现场，不能按“还有 ready/future”误报成主线健康，必须继续追派发链与 live run 真相

## 6. 当前协作注意事项

### 6.1 单例任务图不能破坏
- 不要为 `workflow` 主线再平行创建第二张长期主图。
- 观察、补链、恢复、派发都要优先回到单例全局任务图判断。

### 6.2 不要把“未来入口”理解得过于死板
- 连续运行不是要求永远同时存在：
  - `1 running`
  - `1 ready`
  - `1 future`
- 真正要守住的是：系统里始终至少保留“当前执行”或“下一棒出口”。
- 这个出口可能表现为：
  - 当前 `running`
  - 下一条 `ready`
  - 下一次 `future schedule`
  - 保底巡检的 future

### 6.3 不要只看前端表象
- dashboard、status、workboard 都可能短时落后。
- 真相优先级仍然是：
  - `run.json/result.json/events.log`
  - `node.json`
  - `schedule detail / schedules.jsonl`

### 6.4 当前升级仍然必须尊重空窗
- 只要还有真实 running task，就不要把“升级正式环境”理解成可无损热切。
- 当前正确做法仍然是：
  - 先等空窗
  - 再 `apply`
  - 升级后立刻复核 ready 节点能否被新版本自动接回

### 6.5 主线和保底不能一起丢
- `[持续迭代] workflow` 可以失败，但失败后必须续挂新的主线或保底入口。
- `pm持续唤醒 - workflow 主线巡检` 的职责就是防止主线静默断链。
- 任何修改如果会让主线失败路径下不再续挂保底，都视为破坏 7*24 基线。

### 6.6 helper 小伙伴不是主线本身
- `workflow_devmate / workflow_testmate / workflow_qualitymate / workflow_bugmate` 负责推进泳道工作。
- 他们失败了需要修，但不应让主线接力因此消失。
- 主线连续性永远优先于 helper 并行度。

### 6.7 不要把“服务活着”误说成“7*24 正在推进”
- 若现场是 `0 running + ready/future`，应明确说：
  - 服务在运行
  - 但当前没有真实 token 消耗
  - 主线处于待派发或等待下一棒
- 不要把这类空窗误汇报成“已经在稳定执行”。

### 6.8 根仓同步滞后时必须切到发布边界收口模式
- `7x24` 的目标不只是继续开发；只要某一轮已经产生“已验证但未推回根仓”的代码改动，这一轮本身就还不算完整结束；若 `.repository/<developer_id>` 已经 `ahead`、存在已验证但未推送的 dirty tracked 改动，或中途失败后遗留 dirty/ahead，这本身就算当前版本现场异常。
- 一旦命中这类现场，下一轮默认先冻结同工作区新增实现，优先盘点 `git status --short --branch`、`git diff --stat` 和当前批次归属，而不是继续顺手扩功能或 UI 切片。
- 成功轮的默认收尾顺序固定为：切最小推送批次 -> 跑命中该批次的最小验证 -> `commit / push` 回 `../workflow_code/main` -> 再继续 `test / prod candidate`；除非 `prod` 主链断裂、升级止血或高优先事故，否则根仓同步优先级高于继续扩本地功能面。
- 如果某轮是“做到一半失败/退出”而不是“验证并收口完成”，下一轮第一优先级不是继续基于这批改动开发，而是先处理这批历史问题：确认保留、切批、回滚或推根仓，并在收口或明确阻塞原因前停止同工作区扩面。
- 每轮巡检或自迭代总结都要显式回写：`root_sync_state(clean_synced/ahead_clean/ahead_dirty/diverged_or_unknown)`、`ahead_count`、`dirty_tracked_count`、`untracked_count`、`push_block_reason`、`next_push_batch`。
- 不要把“当前还有 running/ready/future”理解成可以无限期积压本地未推改动；对多 agent 协作来说，`../workflow_code/main` 的持续同步本身就是 `7x24` 连续推进的一部分。

## 7. 变更 7*24 机制时的最低检查清单
- 改 schedule 命中/恢复逻辑时，至少复核：
  - 主线命中能否建单
  - trigger 是否回写 `assignment_ticket_id / assignment_node_id`
  - ready 节点能否在 running 槽释放后自动接力
- 改 assignment 收尾/恢复逻辑时，至少复核：
  - 终态是否保真，不再被误写成 `运行句柄缺失`
  - 收尾后是否续挂下一轮主线
  - 收尾后是否续挂保底巡检
  - 工作区今日日记是否被追加
- 改升级逻辑时，至少复核：
  - `/api/runtime-upgrade/status` 的 `running_task_count` 是否与 workboard/file 真相一致
  - 升级后 `ready` 节点是否能被新版本接回
  - 升级成功后 `current_version`、实例文件、last action 三处是否一致
- 改 `7x24` 版本推进/发布边界逻辑时，至少复核：
  - `ahead/dirty` 现场是否会把下一轮切到发布边界收口模式
  - 已验证批次是否能恢复到 `workflow_code/main`
  - `root_sync_state / push_block_reason / next_push_batch` 是否被显式记录
