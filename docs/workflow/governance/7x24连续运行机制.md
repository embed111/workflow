# 7x24 连续运行机制

## 1. 文档定位
- 本文件是 `workflow` 当前 `7*24` 连续运行机制的专项真相源。
- 目标是把“系统现在为什么能连续跑、靠什么接力、什么时候算真在跑、什么时候只是服务活着、升级时怎么处理”写清楚。
- 本文件描述的是当前实现事实与协作注意事项，不替代运行态数据；实际现场仍以 `prod` API、任务图文件、`run.json`、`events.log` 为准。

## 1.1 长期优化目标
- 当前 `7x24` 不只是为了维持“不断线”；它还承担一条更高阶的长期目标：把 `workflow` 的项目推进节奏、协作方式和治理成熟度持续优化到企业级，甚至在关键维度上超过常规企业级团队。
- 因此，后续每一轮 `7x24` 的价值不只看“本轮有没有继续推进版本”，也要看“有没有让项目推进方式本身变得更稳、更快、更清楚、更可复制”。
- 这条长期优化默认至少覆盖：
  - 节奏优化：版本规划、执行、验收、发布、升级、回归的节奏越来越稳定。
  - 协作优化：`workflow(pm)` 与各 helper 的分工、并行、接力、验收、升级与发布边界持续改进。
  - 治理优化：版本切换、阻塞处理、AAR、学习回流、质量证据和发布边界都更可回读、更少靠人工兜底。
  - 工程优化：开发工作区、根仓同步、`test/prod` 发布链、可观测性和恢复链越来越标准化。
- 如果当前协作方式已经成为效率或质量瓶颈，`7x24` 默认不应只沿旧方式继续跑，而要把“改进协作方式”本身纳入正式推进事项。

## 2. 当前机制一句话
- 当前 `7*24` 的实现，不是“单个永不退出的大循环”，而是：
  `单例全局任务图 + 终态 durable handoff outbox + 主线滚动一次性 schedule + 文件真相恢复 + 空闲窗口升级`。

## 3. 当前核心组成

### 3.1 单例全局任务图
- 当前 `7*24` 只认一张任务图，不再拆多张并行主图。
- schedule 命中后，节点统一落到 `任务中心全局主图`。
- 这条约束的目的很直接：所有主线和小伙伴协作节点都在同一张图里观察、排队、接力，避免“看板 A 在跑、真实执行却在看板 B”的漂移。
- 当前代码真相主要在：
  - `.repository/pm-main/src/workflow_app/server/services/schedule_service.py`
  - 关键标识：`SCHEDULE_ASSIGNMENT_GRAPH_NAME = "任务中心全局主图"`
  - 关键标识：`SCHEDULE_ASSIGNMENT_GRAPH_REQUEST_ID = "workflow-ui-global-graph-v1"`

### 3.2 单条核心 schedule
- 主线 schedule：
  - 名称：`[持续迭代] workflow`
  - 作用：推动 `workflow(pm)` 继续做版本推进，不允许空转。
  - 预期产物：`continuous-improvement-report.md`
- 主线仍是“滚动一次性 schedule”。
  - 当前一轮收尾后，再把下一轮一次性时间点写回去。
  - 这样做的好处是：每轮都能带上最新上下文，而不是死板重复旧 prompt。
- 不再创建或续挂 `pm持续唤醒 - workflow 主线巡检`。
  - 主线失败、断链、卡住、烟测守卫阻断或 dispatch 失败时，恢复动作直接写回原 `[持续迭代] workflow` schedule。
  - 这避免把“巡检任务”误展示成正在推进版本的工作项，也避免主线和保底两套 prompt/优先级长期分叉。
- 旧巡检的代码兼容壳也不再保留。
  - 后端不再输出 `is_workflow_patrol / patrol_node / patrol_schedule_preview` 这类历史读面字段。
  - 前端不再渲染 `patrolNode / patrolSchedulePreview` 分支。
  - schedule 管理与任务图序列化不再暴露 `_schedule_is_workflow_patrol_payload / _assignment_is_workflow_patrol_node` 这类 helper。
  - 后续不要为了“兼容旧巡检”恢复这些字段；连续性真相只看 `[持续迭代] workflow`、`assignment_mainline_handoffs`、任务图节点与 run 文件。

### 3.3 主线时间规则
- 主线成功后：
  - 下一轮主线默认 `+15 分钟`
  - 优先级默认 `P1`
- 主线失败后：
  - 下一轮主线默认 `+10 分钟`
  - 优先级默认 `P1`
- 当前口径刻意保持“主线十几分钟级”：规则/治理变更由主线在十几分钟内重新读取；异常恢复不再转给第二条巡检计划，而是直接重排主线。
- 当前代码真相主要在：
  - `.repository/pm-main/src/workflow_app/server/services/assignment_service_parts/assignment_self_iteration_runtime.py`
  - `.repository/pm-main/src/workflow_app/server/services/schedule_self_iter_runtime.py`
  - 关键常量：
    - `ASSIGNMENT_SELF_ITERATION_SUCCESS_DELAY_MINUTES = 15`
    - `ASSIGNMENT_SELF_ITERATION_FAILURE_DELAY_MINUTES = 10`

### 3.4 schedule worker 的工作方式
- schedule worker 每轮都会先尝试 drain `assignment_mainline_handoffs` 中的待接棒记录。
  - 如果上一轮主线已经成功、失败，或被文件真相恢复成终态，handoff worker 会幂等地确保下一轮 `[持续迭代] <agent>` schedule 已经排上。
  - 如果项目或 assignment 被手动暂停，handoff 保持 `paused`，不派发下一轮；恢复启动后下一次 drain 会继续接棒。
  - handoff drain 自身异常只写事件，不阻断正常 schedule scan。
- schedule worker 每分钟扫描一次命中窗口，负责把“该触发了”快速落盘。
- 对 `[持续迭代] ...` 这类自迭代滚动一次性计划，worker 还会补偿“错过命中分钟”的现场：如果 `next_trigger_at` 已经过期但对应 trigger 尚未落盘，下一次 scan 会按原计划时间补一次 `trigger_hit`，而不是把主线当成普通 once 计划退休。
- 命中后先写 `trigger_hit`，然后后台继续做：
  - 建单
  - 建节点
  - 自动派发
  - 状态回写
- 若同一条 schedule 在旧节点还停留在 `pending/ready/queued` 时再次命中，系统默认只保留最新这条 pending trigger；旧的待执行节点会被删除，旧 trigger 会标记成 `superseded`，避免 backlog 永远排不完；已经 `running` 的旧节点则保持不动。
- `workflow` 主线的 queue priority 不能继续依赖历史 `schedule_plans` 行里残留的旧值。当前代码真相要求：
  - schedule create/update 时把 `[持续迭代] workflow` 强制归一到 `P1`
  - 即使存量 schedule 行还没来得及修正，`schedule -> assignment` 桥接层在建节点时也会按 canonical priority 建出 `mainline=P1`
- 同时每 `8` 秒做一次最近 trigger 的恢复扫描，补偿：
  - 进程重启
  - trigger 半断裂
  - 已建单但没回写
  - 已 ready 但未继续处理
- 若单例全局任务图已经出现“`running_task_count=0`、ready 节点年龄超过阈值、却还没继续 dispatch”的假待派发现象，worker 还会主动补一次 `ready dispatch recovery`，不再把这类现场误当成健康空窗。
- 当前代码真相主要在：
  - `.repository/pm-main/src/workflow_app/server/services/schedule_service.py`
  - `.repository/pm-main/src/workflow_app/server/services/assignment_service_parts/assignment_mainline_handoff_runtime.py`
  - 关键常量：
    - `SCHEDULE_WORKER_INTERVAL_SECONDS = 8`
    - `SCHEDULE_TRIGGER_RECOVERY_START_LIMIT_PER_PASS = 1`

### 3.5 真实执行链
- 节点进入 `ready` 后，由 assignment 调度链继续派发。
- 真实执行不是页面状态，而是 run 落盘：
  - `run.json`
  - `events.log`
  - `stdout.txt`
  - `stderr.txt`
  - `result.json`
- 当前执行超时已经从固定短超时改为“无进展超时”：
  - 默认按活动超时 `7200s`
  - 不是简单的墙钟 `1200s`
- 当前也会对瞬时 `Codex` 流断开做一次自动重试，避免把可恢复抖动直接记成真失败。
- 当前代码真相主要在：
  - `.repository/pm-main/src/workflow_app/server/services/assignment_service_parts/task_artifact_store_run_runtime.py`

### 3.6 收尾后的接力机制
- 一轮 run 收尾后，系统会继续做几件事：
  - 把节点终态写回任务图
  - 写结果文件与 audit
  - 追加工作区今日日记
  - 在 `assignment_mainline_handoffs` 里记录“这一轮终态需要下一棒”
  - 由独立 handoff worker 续挂下一轮主线 schedule
  - 若 `prod` 已空闲且存在更新 candidate，则可尝试触发正式升级
  - 若当前不抑制派发，则继续 `dispatch next ready`
- 这就是为什么 `7*24` 的连续性，不是靠一个节点永远跑，而是靠“每轮收尾都保证下一棒已经排上”。
- 主线触发失败、烟测守卫 fail-closed 或 dispatch 失败时，系统不再创建第二条保底计划；恢复函数会把同一个 `[持续迭代] workflow` 改回 `enabled=true / P1 / once=future`。
- 主线 once 时间点如果已经被 durable handoff 排上，但 worker 因重启、阻塞或空窗错过该分钟，读模型不得把它按“已终态且无 future 的普通一次性计划”自动停用；连续性优先级高于通用 once 计划退休规则。
- 当前还有一个实现细节需要明确：
  - 若本轮收尾是由 `stale recovery / 文件真相补终态` 触发的，系统会抑制这次收尾内部的 follow-up dispatch，避免在外层 `dispatch/reconcile` 已持锁时再次递归派发导致卡死。
  - 这类场景的真实接棒由 handoff outbox 与 schedule worker 继续完成，不再由补终态收尾自己二次递归。
  - assignment finalize 现在也不再把 `workspace memory / pm daily governance / prod upgrade / follow-up dispatch` 整包压在全局 ticket 锁里；锁内只保留最小必要终态回写，避免一条慢 finalize 把整张全局主图一起拖成假待派发。
  - 看门狗或 stale recovery 的职责是把假运行收成真实失败并记录 handoff；它不创建第二条用户可见主线。

### 3.7 升级链
- 当前 `prod` 升级不是“带着 running task 无感切换”。
- 当前只能做到：
  - `running_task_count = 0`
  - 且 `/api/runtime-upgrade/status.can_upgrade = true`
  - 才安全 `apply`
- 升级方式有两类：
  - 手工调 `/api/runtime-upgrade/apply`
  - `prod` supervisor 托管的空闲 watcher 周期检查并在空窗时自动 apply
- 当前 watcher 脚本真相在：
  - `.repository/pm-main/scripts/apply_prod_candidate_when_idle.py`
- 当前 `prod` 默认自动升级口径：
  - 由 `start_workflow_env.ps1` 托管的 supervisor 每 `2` 小时做一次单次检查
  - 若 `candidate_is_newer=false`、没有 candidate，或当前仍有 running task，就直接跳过，不扰动现网
  - 只有命中 `running_task_count=0 && can_upgrade=true` 时才发起 `/api/runtime-upgrade/apply`
  - 当前 running 的主线节点不再靠 `exclude_assignment_*` 自己触发升级，避免把自己切成假 running
  - 当前更高 `candidate` 只作为升级建议与 single-check 触发条件，不再单独把新任务派发硬冻结；真正的硬 drain 只保留 `upgrade_request_pending`
- 当前无痛升级限制的专项设计说明在：
  - `docs/workflow/design/详细设计-prod无痛升级与连续运行保障.md`

## 4. 当前 7*24 的真实运行步骤
1. `schedule worker` 到点命中 `[持续迭代] workflow`。
2. 先把 `trigger_hit` 落盘，保证“命中过”这件事可追溯。
3. 后台线程在单例全局任务图里创建或恢复对应节点。
4. 节点进入 `ready`。
5. 调度器在并发槽允许时把 `ready` 节点派发为真实 run。
6. run 在工作区里执行，并持续写 `run.json/events.log/result.json`。
7. 收尾阶段写终态、写 audit、补记今日日记，并落一条 durable handoff。
8. schedule worker drain handoff，续挂下一轮主线；若遇到用户/项目暂停，则把 handoff 保持为 `paused`。
9. 必要时申请空闲升级。
10. 若当前还有其他 `ready` 节点，系统继续派发下一条。
11. 这样形成“当前 running + 下一条 ready/future”的接力。

## 5. 怎样判断 7*24 是不是真的在跑

### 5.1 不能只看 token
- token 在涨，不一定代表主线健康，可能只是某个 helper 在跑。
- token 不涨，也不一定代表系统死了，可能只是：
  - 当前没有真实 running
  - 只有 `ready/future`
  - 正处于升级空窗

### 5.2 不能只看“服务活着”
- `healthz` 正常，只说明 Web 还活着。
- 不等于主线已经被派发。

### 5.3 当前推荐判断顺序
1. 看 `/healthz`
   - 确认服务在不在
2. 看 `/api/status`
   - 确认 `running_task_count / queued_task_count / assignment_workboard_summary`
3. 看 `/api/runtime-upgrade/status`
   - 确认当前是否被 running 槽挡住升级，还是已经空闲
4. 看 `/api/schedules`
   - 确认主线是否还有 future/最近结果
5. 直读任务图和 run 文件
   - `nodes/<node>.json`
   - `runs/<run>/run.json`
   - `runs/<run>/events.log`

### 5.4 当前几种典型现场
- `running_task_count=1`，且 `run.json.status=running`、`events.log` 有 `provider_start/turn.started`
  - 这才是真正在执行
- `running_task_count=0`，但有 `ready`
  - 这是“服务在线，任务待派发”，不是正在吃 token
- `running_task_count=0`，有 `future`
  - 这是“下一轮已经排上”，但当前没有执行
- `active_agent_count=1`，workboard 却 `0 running`
  - 这不算主线在跑，只能说明当前还有活跃视图或 future 计划
- `running_task_count=0`，`workflow` 存在已到时的 `ready` 节点，且 recent trigger/message 出现 `assigned agent already has running node`
  - 这是“假健康/接力异常”现场，不能按“还有 ready/future”误报成主线健康，必须继续追派发链与 live run 真相
- `running_task_count=0`，ready 节点已经老化、最近没有新的 `dispatch/run`，但服务和页面仍可读
  - 这也是不健康；当前 worker/watchdog 应主动补一次 ready dispatch recovery，而不是继续把它汇报成“待派发但正常”

## 6. 当前协作注意事项

### 6.1 单例任务图不能破坏
- 不要为 `workflow` 主线再平行创建第二张长期主图。
- 观察、补链、恢复、派发都要优先回到单例全局任务图判断。

### 6.2 不要把“未来入口”理解得过于死板
- 连续运行不是要求永远同时存在：
  - `1 running`
  - `1 ready`
  - `1 future`
- 真正要守住的是：系统里始终至少保留“当前执行”或“下一棒出口”。
- 这个出口可能表现为：
  - 当前 `running`
  - 下一条 `ready`
  - 下一次 `future schedule`

### 6.3 不要只看前端表象
- dashboard、status、workboard 都可能短时落后。
- 真相优先级仍然是：
  - `run.json/result.json/events.log`
  - `node.json`
  - `schedule detail / schedules.jsonl`

### 6.4 当前升级仍然必须尊重空窗
- 只要还有真实 running task，就不要把“升级正式环境”理解成可无损热切。
- 当前正确做法仍然是：
  - 先等空窗
  - 再 `apply`
  - 升级后立刻复核 ready 节点能否被新版本自动接回

### 6.5 主线失败必须直接重排主线
- `[持续迭代] workflow` 可以失败，但失败后必须续挂新的主线入口。
- 不再用 `pm持续唤醒 - workflow 主线巡检` 防止断链；这条能力已经收敛为主线自身的失败恢复。
- 任何修改如果会让主线失败路径下不再重排 `[持续迭代] workflow`，都视为破坏 7*24 基线。

### 6.5.1 失败重试次数不能成为停止条件
- 对 `7*24` 连续运行项目，失败后的默认策略是继续接力，直到用户手动暂停、项目退役、运行窗口不允许或安全门禁明确阻断。
- `failure_retry_max_attempts=0` 表示无限接力；非零次数只能作为告警、升级或人工复核阈值，不能被解释成“达到次数后停止主线”。
- 代码、界面和验收用例不应再使用“最大重试”这类会暗示硬停止的表达；应统一称为“失败接力”或“失败升级阈值”。
- 现有真实 `7*24` 项目的运行策略应逐步收敛到 `failure_retry_max_attempts=0`，避免旧配置在未来被误读成成本或连续性硬闸。

### 6.5.2 失败必须先分类，非正常失败不能只恢复
- 正常失败是指业务、质量或评审合同本身给出明确 `NO_GO / request_changes / blocked`，且执行链、交付链、状态写回都正常。这类失败可以按原任务继续失败接力、回派修复、重试或重新验收。
- 非正常失败包括但不限于：`provider_start` 卡住、`starting_without_provider_pid`、`running_finalize_stall`、`create_node / dispatch-next` 请求超时但状态不明、`stream disconnected before completion`、artifact delivery 丢失、run/node 终态分叉、ghost running、ready 长期无人派发、生产读面卡死。
- 命中非正常失败时，恢复动作只算止血，不算收口；同轮或下一轮必须安排生产问题定位，至少回答：
  - 根因是什么，属于运行时、调度、provider、文件真相、交付投影、UI 读面还是部署/升级链。
  - 是否影响同类项目或同类 helper。
  - 哪个用例、probe 或 harness 能先红灯复现。
  - 修复后用哪个绿灯证明下一次运行不会再次以同样方式失败。
- 对非正常失败，不允许只写“已 repair / 已 rerun / 已恢复”为最终结论；版本计划、history 或运行日志必须留下后续缺陷/质量债路由，直到补上用例并通过。
- 如果当轮因为生产连续性必须先恢复，也要把“定位生产问题 + 补用例 + 防复发验证”排成下一棒的明确任务，不得被普通功能开发覆盖。

### 6.6 helper 小伙伴不是主线本身
- `workflow_devmate / workflow_testmate / workflow_qualitymate / workflow_bugmate` 负责推进泳道工作。
- 他们失败了需要修，但不应让主线接力因此消失。
- 主线连续性永远优先于 helper 并行度。

### 6.7 不要把“服务活着”误说成“7*24 正在推进”
- 若现场是 `0 running + ready/future`，应明确说：
  - 服务在运行
  - 但当前没有真实 token 消耗
  - 主线处于待派发或等待下一棒
- 不要把这类空窗误汇报成“已经在稳定执行”。

### 6.7.1 不要把“纯观察”误当成推进
- 主线轮次不允许只输出“继续观察 / 风险滚动 / 无新阻塞”这类纯观察结论。
- 每轮至少要完成 `1` 项推进性修改；单纯写 `history / logs / 今日日记 / 观察摘要` 不算推进性修改。
- 一旦识别到 live 风险，本轮第一优先动作默认就是尝试降低该风险；只要仍在受支持动作范围内，就应先做治理、修复、派发、调度调整或发布收口，而不是把风险原样递延到下一轮。
- 只有在明确没有安全可执行动作、或继续动作会引入更大风险时，才允许把当前轮记为 `blocked`；否则“先观察一轮再说”按空转处理。

### 6.8 根仓同步滞后时必须切到发布边界收口模式
- `7x24` 的目标不只是继续开发；只要某一轮已经产生“已验证但未推回根仓”的代码改动，这一轮本身就还不算完整结束；若 `.repository/<developer_id>` 已经 `ahead`、存在已验证但未推送的 dirty tracked 改动，或中途失败后遗留 dirty/ahead，这本身就算当前版本现场异常。
- 一旦命中这类现场，下一轮默认先冻结同工作区新增实现，优先盘点 `git status --short --branch`、`git diff --stat` 和当前批次归属，而不是继续顺手扩功能或 UI 切片。
- `root_sync_state / ahead_count / dirty_tracked_count / push_block_reason / next_push_batch` 这些字段只是异常触发信号，不是交付本身；只要命中 dirty/ahead/阻塞，agent 默认必须立即执行清理、切批、提交、推根仓或明确阻塞收口，而不是只汇报数字然后等待状态自己恢复。
- 成功轮的默认收尾顺序固定为：切最小推送批次 -> 跑命中该批次的最小验证 -> `commit / push` 回 `../workflow_code/main` -> 再继续 `test / prod candidate`；除非 `prod` 主链断裂、升级止血或高优先事故，否则根仓同步优先级高于继续扩本地功能面。
- 如果某轮是“做到一半失败/退出”而不是“验证并收口完成”，下一轮第一优先级不是继续基于这批改动开发，而是先处理这批历史问题：确认保留、切批、回滚或推根仓，并在收口或明确阻塞原因前停止同工作区扩面。
- 每轮主线总结都要显式回写：`root_sync_state(clean_synced/ahead_clean/ahead_dirty/diverged_or_unknown)`、`ahead_count`、`dirty_tracked_count`、`untracked_count`、`push_block_reason`、`next_push_batch`。
- 不要把“当前还有 running/ready/future”理解成可以无限期积压本地未推改动；对多 agent 协作来说，`../workflow_code/main` 的持续同步本身就是 `7x24` 连续推进的一部分。

### 6.8.1 命中异常治理现场时允许 pm 直接收口
- 若 `workflow(pm)` 的主线命中异常治理现场，例如：
  - `../workflow_code` 本地根仓相对当前工作区未收口，或本地根仓自身出现漂移
  - helper 长时间停在 `runtime_status=creating`
  - 旧 supervisor / `request_pending` / stale pid 与真实现场不一致
- 这类动作默认视为 `7x24` 合法治理收口，不应再被“只允许写 workspace_path”机械拦住。
- 允许的动作仅限：
  - 基于本机 `../workflow_code` 执行 non-destructive 本地根仓收口
  - 调用受支持的 developer workspace bootstrap/refresh
  - 调用受支持 API/脚本处理 helper stale lock、schedule 断链、supervisor 恢复、runtime upgrade 切换
- 默认不做的动作：
  - 主动 `fetch/pull origin`
  - 主动拉 GitHub 或其他远端镜像代码
- 仍然禁止：
  - 直接手改 `../workflow_code` 工作树内容
  - 把 `.running/prod` / `.running/test` 当开发面长期叠改
  - destructive git 或绕过验证的手工热修

### 6.9 已退役运营项目不再自动恢复
- `project-comics-smoke` / `Comics Bootstrap Smoke` 只是历史上的第二项目启动 smoke，不再是当前需要持续运行的运营项目。
- 2026-04-27 起，该项目按用户要求停止，原因是避免无意义 token 成本；后续不得把它恢复成 active，也不得为它续挂 schedule、handoff、starter task 或 project-controller 节点。
- 生产运行态应保留一条 archived tombstone，而不是把记录物理删除：
  - `state/project-registry.json`: `project_id=project-comics-smoke` 保持 `lifecycle_state=archived`，并记录 `archived_reason=user requested stop; do not restore; avoid token cost`。
  - `state/project-runtime-policies.json`: 同项目保持 `manual_pause=true`。
- 保留 tombstone 的目的不是让它继续显示为运行项目，而是避免旧 continuity repair 把“记录缺失”误判成“需要按历史证据恢复”。
- 后续任何 gate、release check、runtime smoke、项目读面回归，都不得再把 `project-comics-smoke` 作为必须存在的 quiet project / continuity sample / startup-ready 样本。
- 若未来需要验证多项目能力，应使用中性临时 fixture 或当前真实 active 项目；除非用户明确重新启用，否则不得从历史 evidence、旧日志、旧 run 输出或 `WORKFLOW_PROJECT_CONTINUITY_PROJECT_IDS` 恢复 `project-comics-smoke`。

## 7. 变更 7*24 机制时的最低检查清单
- 改 schedule 命中/恢复逻辑时，至少复核：
  - 主线命中能否建单
  - trigger 是否回写 `assignment_ticket_id / assignment_node_id`
  - ready 节点能否在 running 槽释放后自动接力
- 改 assignment 收尾/恢复逻辑时，至少复核：
  - 终态是否保真，不再被误写成 `运行句柄缺失`
  - 收尾后是否续挂下一轮主线
  - 工作区今日日记是否被追加
- 改升级逻辑时，至少复核：
  - `/api/runtime-upgrade/status` 的 `running_task_count` 是否与 workboard/file 真相一致
  - 升级后 `ready` 节点是否能被新版本接回
  - 升级成功后 `current_version`、实例文件、last action 三处是否一致
- 改 `7x24` 版本推进/发布边界逻辑时，至少复核：
  - `ahead/dirty` 现场是否会把下一轮切到发布边界收口模式
  - 已验证批次是否能恢复到 `workflow_code/main`
  - `root_sync_state / push_block_reason / next_push_batch` 是否被显式记录
