## 1. 定位
- 本文件定义 `workflow` 正式代码合入前的 review 门禁。
- 目标不是增加流程负担，而是防止“逻辑继续散、代码继续胖、测试最后补、坏抽象被合入”成为常态。
- 适用范围：所有进入 `.repository/<developer_id>/` 并准备推回 `../workflow_code/main` 的正式代码改动。
- `workflow_reviewmate` 已通过生产环境创建角色功能建立；后续若角色仍处于 `pre_release`，不影响它先作为合入前 review 小伙伴执行门禁。

## 2. 默认顺序
1. 开发者在自己的 `.repository/<developer_id>/` 同步最新 `../workflow_code/main`。
2. 先补红灯 probe / 最小失败用例，再改代码，再跑绿灯验证。
3. 跑 `python scripts/quality/check_workspace_line_budget.py --root .`。
4. 整理 review 输入，交给 `workflow_reviewmate`。
5. `workflow_reviewmate` 输出 `approve / request_changes / block`。
6. 只有 `approve` 后，当前批次才能 commit / push / 根仓同步。

## 3. Review 输入
- `change_goal`: 本批改动要解决的用户问题或版本需求。
- `changed_files`: 触达文件列表和主要影响域。
- `architecture_impact`: 是否改变业务边界、读写入口、状态机、存储或投影。
- `tests`: 红灯证据、绿灯验证命令、完整 gate 或跳过原因。
- `deletion_delta`: 删除了哪些冗余实现；如果净新增代码，解释为什么不是继续堆层。
- `parts_delta`: 是否新增、删除或触达 `*_parts / manifest.json / exec_local_parts`。
- `compatibility`: API 字段、运行态数据、历史节点/项目/角色读面的兼容判断。
- `rollback`: 如果上线后异常，最小回退方式是什么。

## 4. 打回条件
- 新增业务逻辑直接散落到 API、presentation、runtime helper 或 part 私有函数里，缺少 application use case / facade。
- 同一状态、同一读模型或同一写入口被多个模块重复计算。
- 继续新增 `exec_local_parts` 或无 owner 的 `*_parts`，却没有明确退出计划和边界 probe。
- 为了过行数门禁只横向拆文件，没有缩短调用链、减少依赖或删除重复实现。
- 没有红灯/绿灯证据，或者测试只覆盖“脚本能跑”，没有覆盖本批要防的行为回归。
- 用 fallback / compat / legacy 分支掩盖真实设计债，并且没有明确移除条件。
- 文件、函数、命名或抽象让职责更难读懂；reviewer 无法在 10 分钟内画出主链入口和状态归属。
- 触达发布、运行态、任务派发、角色创建、项目、schedule、runtime truth 等高风险链路，却没有说明 live 兼容和失败恢复。

## 5. 通过标准
- 业务主链可从 `用户动作 -> application use case -> domain service -> repository/projection -> response` 读通。
- 本批改动至少减少一种复杂度：删除重复、收口入口、统一读模型、缩短调用链、去掉隐式全局、或把临时兼容改成显式合同。
- 测试证据能证明本批目标行为从红灯到绿灯，且必要的回归已跑。
- 新增代码的存在理由清楚；如果没有净删除，必须能解释为什么这次加法会降低后续维护成本。
- review 记录有明确结论、阻塞项和允许合入的最小条件。

## 6. 例外
- 生产止损热修可以先由 `pm` 直接推进，但必须满足：
  - 只做最小补丁；
  - 发布后补 retrospective review；
  - 若 review 发现设计债，必须排入当前或下一版本；
  - 不得把热修例外常态化为绕过 review。
