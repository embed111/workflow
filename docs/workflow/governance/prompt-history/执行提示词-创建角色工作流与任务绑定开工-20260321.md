## 2. 本轮目标
1. 本轮目标只有一个：把 `训练中心 -> 创建角色` 落成可运行的正式能力，而不是继续堆静态示意。
2. 当前实现必须严格以最新需求与原型为准，不要从旧页面结构、旧“阶段流 + 任务图”拆分稿，或旧的表单式创建角色实现反推需求。
3. 这次改动只允许发生在 `创建角色` 工作台内部；左侧模块切换栏与 `训练中心` 顶层标题区保持现有语义和主视觉不变。

## 3. 你必须按以下口径实现
1. 中部工作台必须是单一聊天工作面：
   1. 用户消息在右侧气泡；
   2. agent 消息在左侧气泡；
   3. 系统任务更新保持居中轻提示；
   4. 不得再出现“面板里套第二层聊天框”的双层结构。
2. 输入区必须支持“图片 + 文字同草稿、同消息发送”：
   1. 支持粘贴图片；
   2. 支持拖拽图片；
   3. 支持系统选图；
   4. 不要额外做一个抢主视觉的“上传图片”主按钮。
3. 右侧辅助区必须采用标签切换：
   1. `阶段与任务演进`
   2. `角色画像`
4. `阶段与任务演进` 必须收口成一张统一图：
   1. 采用纵向阶段主线或 stage gutter；
   2. 不再额外保留横向阶段概览；
   3. 图面语法尽量继承现有任务中心任务图，而不是重新造一套六列流程卡片。
5. 阶段 1 `初始化工作区` 只是前置条件：
   1. 由脚本完成；
   2. 默认不生成任务中心任务卡；
   3. 其结果必须为后续阶段提供真实可用的角色工作区与记忆骨架。
6. 阶段 2 起所有以任务卡出现的节点，都必须是任务中心真实任务引用：
   1. 至少能看到 `task_id` 或等效引用标识；
   2. 状态、证据、上下游关系都以任务中心真相源为准；
   3. 不允许继续使用前端硬编码伪任务。
7. 当前分析师动作不得伪装成任务中心任务卡：
   1. 分析引导；
   2. 顶层 `role_spec` 整合；
   3. 结果回传；
   4. 验收追问；
   5. 完成角色创建接口调用；
   6. 这些都应用系统消息、阶段动作提示或状态锚点表达。
8. 当用户在对话中明确表达“另起一个任务去…… / 单独起个任务…… / 后台去跑……”时，必须优先在任务中心创建真实后台任务，而不是只让当前分析师口头回复。
9. 已作废任务不得物理消失：
   1. 主线默认按阶段收口到“废案收纳”入口；
   2. 悬浮或点击固定后可查看废案 `task_id`、任务名、关闭原因；
   3. 仍可从详情卡跳到任务中心查看真实记录。
10. 任务详情必须采用“悬停预览 / 点击固定”的浮层卡：
   1. 浮层不能被滚动容器裁剪；
   2. 浮层出现不能挤动任务流布局；
   3. “去任务中心查看”收口在这个浮层里，而不是占用默认卡面。
11. 任务卡命名必须体现具体后台目标，例如“收集行业案例资料”“生成漏斗样例草案”，不要长期停留在“资料整理/能力补强”这种泛化名称。
12. 视觉必须继续采用本轮确认过的扁平化方向：
   1. 纯色底；
   2. 细边框；
   3. 轻状态色；
   4. 减少重投影、强渐变和厚浮层感。
13. 默认界面坚持少字原则：
   1. 对当前操作无帮助的信息不要放在默认视图；
   2. 解释性内容优先放进悬浮详情或文档；
   3. 默认窗口下尽可能完整展示主流程。
14. 高任务量场景不能失真：
   1. 多个已完成任务、待开始任务和废案并存时，当前方案仍需可读；
   2. 任务默认贴齐所属阶段；
   3. 当前任务标题与状态在默认比例下必须可辨认。

## 4. 明确禁止
1. 不要改动左侧全局模块切换栏和 `训练中心` 顶层标题区。
2. 不要把 `创建角色` 再做回表单页、说明页或卡片堆叠页。
3. 不要把“阶段流”和“任务演进图”拆成两张重复图。
4. 不要把分析师动作继续做成带 `task_id` 的伪任务。
5. 不要删除或前端伪隐藏已作废任务的真实历史记录。
6. 不要把“去任务中心查看”做成默认卡面常驻文案。
7. 不要为了省事改回重渐变、重阴影或高噪声配色。
8. 不要只交静态页面，不接后端接口与真实任务引用。

## 5. 推荐实现顺序
1. 先补 `创建角色` 所需的后端事实接口：
   1. 当前阶段；
   2. 当前 `role_spec`；
   3. 关联任务引用；
   4. 创建后台任务；
   5. 阶段切换；
   6. 完成角色创建。
2. 再补阶段 1 的工作区初始化与记忆机制接入。
3. 再补中部单聊天壳、图片+文字同发草稿与系统消息插入。
4. 再补右侧统一阶段与任务演进图、悬浮详情、点击固定、废案收纳与高任务量可读性。
5. 最后联调“显式委派语义 -> 任务中心建任务 -> 右侧图引用更新 -> 聊天系统消息回传”闭环。

## 6. 最低验收回传
1. 页面截图：
   1. `创建角色` 默认视图；
   2. 用户发送“图片 + 文字”同消息后的聊天视图；
   3. 右侧切到 `角色画像` 的视图；
   4. 任务悬停预览浮层；
   5. 任务点击固定浮层；
   6. 存在废案收纳入口的视图；
   7. 高任务量视图。
2. 接口与行为证据：
   1. 当前阶段读取接口；
   2. `role_spec` 读取或刷新接口；
   3. 显式委派语义触发任务中心建任务的请求/响应；
   4. 右侧任务卡与任务中心 `task_id` 一致的证据；
   5. 阶段 1 初始化角色工作区与记忆骨架的证据。
3. 代码落点：
   1. `创建角色` 页面入口；
   2. 聊天气泡与输入草稿实现位置；
   3. 右侧统一阶段与任务演进图实现位置；
   4. 任务详情浮层实现位置；
   5. 后端阶段/任务/角色接口位置。
4. 风险与未完成项：
   1. 若图片同发首版采用本地文件引用而非最终产物链路，必须明确说明；
   2. 若某些任务中心接口仍需补充，也必须逐项列出，不要笼统写“后续优化”。
