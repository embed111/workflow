# 详细设计-统一入口与训练优化模块

## 1. 设计目标
1. 建立一个统一入口，先展示 agent 资产与版本事实，再进入训练运营动作。
2. 保持“展示层”和“训练状态机层”解耦，避免单页混合导致维护复杂度上升。
3. 保障手动主路径可落地，同时预留分析模块自动计划并行接入能力。

## 2. 设计范围与非范围
1. 设计范围：
   1. 入口信息架构与页面交互。
   2. 训练计划入队、优先级调度、任务移除、执行回写。
   3. 发布版本视图只读集成（仅已发布迭代）。
2. 非范围：
   1. 训练算法细节与模型参数调优。
   2. Git 写操作（打 tag/回滚）实现。
   3. 复杂 trainer 智能匹配策略（首期仅字符串匹配）。

## 3. 信息架构（IA）
1. 一级菜单：`训练中心`。
2. 二级 Tab：
   1. `Agent资产与版本`
   2. `训练运营`
3. 默认进入顺序：
   1. 进入训练中心默认落在 `Agent资产与版本`。
   2. 选择 agent 后，才允许进入 `训练运营`。

## 4. 页面详细设计
### 4.1 Agent资产与版本
1. 顶部统计卡：
   1. `agent_total`
   2. `git_available_count`
   3. `latest_release_at`
   4. `training_queue_pending`
2. 左侧角色卡墙（可搜索）：
   1. `vector_icon`（头像式矢量人物图，非抽象几何图标）
   2. `agent_name`
   3. `current_version`
   4. `core_capabilities`（摘要）
   5. `status_tags`（如 `git_unavailable`）
   6. `identity_style`（视觉风格：名片/工牌）
3. 右侧详情区：
   1. 基础信息：路径、版本、最近发布说明。
   2. 已发布迭代时间线：按时间倒序，支持分页；发布判定固定为 `tag + release note`。
   3. 功能迭代变更：每个发布版本对应 `change_summary`。
   4. 不展示中间 Git 提交列表（commit 明细）。
   5. 操作按钮：`进入训练优化`。

### 4.2 训练运营
1. 计划创建区（手动）：
   1. 必填：目标 agent、能力目标、训练任务清单、验收标准、优先级（P0-P3）。
   2. 提交后生成队列任务。
2. 队列列表区：
   1. 字段：`queue_task_id/source/priority/target_agent/status/similar_flag/enqueued_at`。
   2. 操作：`移除任务`（无二次确认，显示风险提示）。
3. trainer 指定区：
   1. 来源：`../trainer`。
   2. 匹配方式：字符串匹配结果列表。
4. 执行结果区：
   1. `run_ref`
   2. `status`
   3. `result_summary`
   4. `updated_at`

## 5. 数据模型设计（逻辑模型）
1. `agent_registry`

   3. `workspace_path`
   4. `current_version`
   5. `core_capabilities`
   6. `vector_icon`

2. `agent_release_history`

   2. `agent_id`（FK）
   3. `version_label`
   4. `released_at`
   5. `change_summary`
   6. `release_source_ref`（内部追溯字段，不在视图展示）
3. `training_plan`

   3. `target_agent_id`
   4. `capability_goal`

## 6. 状态机设计
1. `training_queue.status`：
   1. `queued -> running -> done|failed`
   2. `queued|running -> removed`（人工移除）
2. 人工移除规则：
   1. 允许直接移除，不弹二次确认。
   2. UI 先显示风险提示，再执行移除动作。
   3. 审计日志必写。
3. 相似计划处理：
   1. 仅写 `similar_flag=true`。
   2. 不改变状态机流转，不自动阻断。

## 7. 调度与并行策略
1. 队列排序键：
   1. `priority`（P0 > P1 > P2 > P3）
   2. `enqueued_at`（同优先级 FIFO）
2. 来源并行：
   1. `manual` 与 `auto_analysis` 共享同一训练队列。
   2. 不按来源分队列，避免调度分裂。
3. 调度冲突处理：
   1. 同一 agent 可存在多个任务，默认允许顺序执行。
   2. 相似任务由人工判断是否移除。

## 8. 接口草案（需求级）
1. `GET /api/training/agents`
   1. 返回 agent 角色卡列表（含头像式矢量图标识字段）。
2. `GET /api/training/agents/{agent_id}/releases`
   1. 返回已发布迭代时间线（不返回中间 commit 明细）。
3. `POST /api/training/plans/manual`
   1. 创建手动训练计划并入队。
4. `GET /api/training/queue`
   1. 返回训练队列。
5. `POST /api/training/queue/{queue_task_id}/remove`
   1. 人工移除任务（无二次确认）。
6. `POST /api/training/queue/{queue_task_id}/execute`
   1. 启动训练执行。
7. `GET /api/training/runs/{run_id}`
   1. 查询执行结果。
8. `GET /api/training/trainers?query={text}`
   1. 从 `../trainer` 字符串匹配训练师。

## 9. 安全、审计与风险控制
1. 安全边界：
   1. Git 读取路径限制在 agent 根目录作用域内。
2. 审计要求：
   1. 所有写动作必须审计：创建计划、入队、移除、执行、回写。
3. 风险提示：
   1. 移除任务前显示风险提示文本（非阻断）。
   2. 相似计划标注展示“可能重复训练”提示。

## 10. 迭代计划建议
1. Phase A（当前）：
   1. 统一入口 + Agent 角色卡概览/详情 + 已发布迭代视图 + 手动计划 + 队列 + trainer 字符串匹配。
2. Phase B（下一步）：
   1. 分析模块自动计划接入（`auto_analysis`）。
   2. 自动计划人工审核门禁（若启用）。
3. Phase C（增强）：
   1. trainer 匹配策略升级（标签/能力画像）。
   2. Git 写操作能力（受控权限）。

## 11. 与需求映射关系
1. `FR-UO-01~FR-UO-04` -> 第 3~4 节（入口与版本层）。
2. `FR-UO-05~FR-UO-11` -> 第 4~7 节（训练运营层）。
3. `FR-UO-12~FR-UO-13` -> 第 8~9 节（回写、审计、门禁）。

## 12. 推断/假设与确认项
1. 推断/假设（置信度: 中）：
   1. `core_capabilities` 可先由 `AGENTS.md` 自动提取并允许人工修正。
2. 推断/假设（置信度: 中）：
   1. 自动计划首期默认进入队列前需人工可见（是否强制审核待确认）。
3. 已确认：
   1. 发布迭代的来源口径固定为 `tag + release note`。
4. 待确认：
   1. 是否在 Phase A 就展示“训练计划模板库”。
   2. 是否需要将 `removed` 任务支持恢复为 `queued`。

## 13. 验收取证规范
1. 证据最小集合（每条 AC）：
   1. 至少 1 张界面截图（可定位页面状态）。
   2. 接口请求/响应示例（或等效抓包/日志片段）。
   3. 关键日志或数据库记录路径（可复查）。
4. 对应代码文件路径。
5. 门禁统计表/汇总矩阵截图仅可作为辅助，不可替代功能页面截图。
6. 证据生命周期：
   1. 验收进行中保留完整证据。
   2. 验收通过后允许执行精简，保留门禁截图与汇总文件，清理过程型截图/GIF/API/DB细项证据。
2. 关键交互录屏（至少）：
   1. 统一入口首屏加载（`AC-UO-01`）。
   2. Agent 详情版本史查看（`AC-UO-02`）。
   3. 手动计划入队（`AC-UO-05`）。
   4. 人工移除任务（`AC-UO-09`）。
3. 证据目录建议（实现侧可调整）：
   1. `.evidence/screenshots/`
   2. `.evidence/recordings/`
   3. `.evidence/api/`
   4. `.evidence/db/`
4. 视觉验收补充：
   1. 角色卡整体布局需体现“名片/工牌”风格。
   2. 头像元素需为“人物语义矢量头像”，不接受仅用抽象形状占位。
