# PM版本推进计划

## 1. 文档定位
- 本文件是 `workflow` 当前由我维护的滚动版本计划真相源。
- 目标不是给出固定日期，而是把“当前活跃版本做什么、下一版本准备什么、哪些内容明确不在本版承诺内”持续写清楚。
- 任务中心、定时唤醒、自迭代总结和阶段性汇报，默认都应回到本文件对齐版本口径。

## 2. 维护规则
1. 版本节奏
   1. 始终只允许 `1` 个 `active` 版本。
   2. 始终保留 `1` 个 `planned` 下一版本。
   3. 其余版本进入 `queued/backlog`，不提前抢占当前版本资源。
2. 每个版本都必须写清楚
   1. 本版目标
   2. 必修复特性
   3. 质量提升项
   4. 性能提升项
   5. 本版不承诺的新特性
   6. 退出门槛
3. 任务衔接
   1. 我每次完成一轮任务后，先更新本文件，再决定下一条任务。
   2. 若当前没有可立即执行的下一条任务，必须确保 `任务中心` 或 `[持续迭代] workflow` 定时唤醒里至少保留一个未来可执行入口。
   3. 若需要多人协作，任务包应优先映射到 `workflow_devmate / workflow_testmate / workflow_qualitymate / workflow_bugmate / workflow(pm)`。
4. 版本切换原则
   1. 当前版本的 `退出门槛` 未满足前，不进入下一版本执行。
   2. 下一版本允许预研，但不得稀释当前活跃版本的 P0/P1 收口。

## 4. 当前活跃版本：`V0.1` 工程质量基线与运行稳态

### 4.1 本版目标
- 把 `workflow` 从“能跑”推进到“连续跑、可恢复、可验证、可继续派下一轮”。
- 先解决工程质量和系统稳定性问题，再放大新特性增量。
- 让我作为 PM 的自迭代推进不再依赖临场记忆，而是依赖计划真相源 + 任务接力 + 定时唤醒。

### 4.2 必修复特性
1. `prod` 入口与 supervisor 稳定恢复，避免停机后 7x24 链直接断。
2. `任务中心` 对 stale running / 假 running / 运行句柄缺失的状态收口必须稳定。
3. `[持续迭代] workflow` 的 schedule 和任务中心接力必须不断链。
4. 开发工作区 -> `workflow_code` -> `test` -> `prod candidate` 的发布链必须稳定，不允许再被错误 remote 或脏工作区漂移破坏。

### 4.3 质量提升项
1. 建立 PM 版本计划真相源，并要求自迭代任务先对照版本计划推进。
2. 固化工程边界：
   1. 双仓边界
   2. 开发工作区边界
   3. 运行态与记忆态边界
3. 补齐与当前主线直接相关的定向验收：
   1. stale run recovery
   2. dashboard runtime fallback
   3. task center/view tabs
   4. self-iteration prompt / schedule contract
4. 收口 `origin` / runtime-config / test gate` 这类易漂移配置的守护策略。

### 4.4 性能提升项
1. 继续压 `dashboard` / `workboard` / `assignments` 读路径的慢查询与大 IO。
2. 继续控制任务运行 trace 与详情回传体积，避免页面和 token 消耗失控。
3. 对所有高频“真相读取”链路，优先做“轻量快路径 + 文件真相兜底”。

### 4.5 本版不承诺的新特性
1. 不优先扩张新的复杂业务模块。
2. 不优先追求大 UI 改版。
3. 不把多 agent 完全自治当成 `V0.1` 交付目标。
4. 不在本版承诺“无人监管长期自治”；本版只承诺“连续推进机制更稳”。

### 4.6 本版任务包
| 任务包 | 默认负责人 | 类型 | 状态 | 说明 |
| --- | --- | --- | --- | --- |
| `V0.1-P0` 连续运行链止血与真相收口 | `workflow(pm)` | 稳定性 | `in_progress` | prod 入口、stale run、schedule 接力、计划真相源 |
| `V0.1-P1` 发布链与工作区边界守护 | `workflow_devmate` | 工程质量 | `planned` | remote 漂移、防脏工作区、候选发布安全 |
| `V0.1-P2` 7x24 巡检与质量结论模板 | `workflow_qualitymate` | 质量 | `planned` | prod 巡检、任务中心/定时链质量清单 |
| `V0.1-P3` 自迭代链 smoke 与回归组 | `workflow_testmate` | 测试 | `planned` | self-iteration / schedule / dispatch / upgrade smoke |
| `V0.1-P4` 运行失败高频根因收敛 | `workflow_bugmate` | 缺陷治理 | `planned` | 运行句柄缺失、路径漂移、编码/重试类残留问题 |

### 4.7 退出门槛
1. `prod` 连续运行链恢复到“停机可拉回、假 running 可收口、下一轮可续排”。
2. 当前主线相关验收具备稳定证据，而不是只靠人工口头判断。
3. 版本计划、任务中心和定时唤醒至少有一条稳定接力链。
4. 发布链恢复到“小步已验证推根仓 -> test -> candidate”的稳定节奏。

## 5. 下一版本：`V0.2` 项目规范与设计规范固化

### 5.1 本版目标
- 把当前已经反复踩过的工程问题，收成明确规范和设计约束，而不是继续靠日记与口头记忆兜底。

### 5.2 计划覆盖
1. 项目规范：
   1. 文档落点
   2. 命名规则
   3. 目录边界
   4. 验收证据要求
2. 设计规范：
   1. 前后端模块职责边界
   2. 超大文件拆分节奏
   3. 运行态 / 真相源读取优先级
3. 协作规范：
   1. PM / dev / test / quality / bug 闭环
   2. 任务包与版本计划的映射

### 5.3 性能与质量重点
1. 把高频热点文件继续拆小到可维护区间。
2. 把 `workflow gate` 与专项验收的职责边界写清楚。

### 5.4 本版不承诺
1. 不优先上新业务功能。
2. 不为短期速度牺牲目录和设计约束。

## 6. 后续版本预告

### `V0.3` 可观测性与运行真实性
- 目标：把 `dashboard / workboard / run.json / events.log / schedule detail` 的真相源关系统一，减少“页面看起来像 A，文件真相其实是 B”。

### `V0.4` 持续唤醒与任务编排闭环
- 目标：让 PM 的版本推进形成“计划 -> 任务 -> 验证 -> 下一轮唤醒”的稳定闭环。
- 重点：自派单策略、下一轮自动接力、失败后的恢复建议。

### `V0.5` 功能扩展批次
- 目标：在工程质量基线稳定后，再系统推进更明显的新特性与体验增长。

## 7. PM 执行口径
1. 我每次正式推进前，先回看本文件的 `active` 版本和任务包。
2. 我每次正式收尾时，至少完成以下动作：
   1. 更新本文件中当前版本状态或任务包状态
   2. 给出验证证据
   3. 确保下一轮任务或下一次唤醒已排上
3. 若运行态与版本计划冲突，以“先修连续运行和真相收口，再推进新任务”为优先级。
