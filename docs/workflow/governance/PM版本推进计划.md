# PM版本推进计划

## 1. 文档定位
- 本文件是 `workflow` 当前由我维护的滚动版本计划真相源。
- 目标不是给出固定日期，而是把“当前活跃版本做什么、下一版本准备什么、哪些内容明确不在本版承诺内”持续写清楚。
- 任务中心、定时唤醒、自迭代总结和阶段性汇报，默认都应回到本文件对齐版本口径。

## 2. 维护规则
1. 版本节奏
   1. 始终只允许 `1` 个 `active` 版本。
   2. 始终保留 `1` 个 `planned` 下一版本。
   3. 其余版本进入 `queued/backlog`，不提前抢占当前版本资源。
2. 每个版本都必须写清楚
   1. 本版目标
   2. 必修复特性
   3. 质量提升项
   4. 性能提升项
   5. 本版不承诺的新特性
   6. 退出门槛
3. 任务衔接
   1. 我每次完成一轮任务后，先更新本文件，再决定下一条任务。
   2. 若当前没有可立即执行的下一条任务，必须确保 `任务中心` 或 `[持续迭代] workflow` 定时唤醒里至少保留一个未来可执行入口。
   3. 若需要多人协作，任务包应优先映射到 `workflow_devmate / workflow_testmate / workflow_qualitymate / workflow_bugmate / workflow(pm)`。
4. 版本切换原则
   1. 当前版本的 `退出门槛` 未满足前，不进入下一版本执行。
   2. 下一版本允许预研，但不得稀释当前活跃版本的 P0/P1 收口。

## 4. 当前活跃版本：`V1` 工程质量基线与运行稳态

### 4.0 7x24 版本推进职责
- 7x24 运行不是为了“继续活着”，而是把版本推进做成一个有节奏、可验证的业务主线；定时唤醒、任务中心和自迭代都必须服务于活跃版本而非空转。每次保底巡检都需要确认当前版本的 `任务包`、候选版本和下一个可执行入口都在同一条版本推进轨道上。
- PM 的 7x24 工作需要在一个循环中覆盖几个周期性泳道：
  - `UCD/设计优化`：定期复盘用户路径与交互体验，调整版本目标。
  - `测试探测`：交付后持续用烟雾和门禁探测版本脆弱点。
  - `工程质量探测`：用 line budget、记忆 bootstrap、运行真相核对等手段验证工程基础。
  - `需求分析`：梳理新需求、优先级和版本依赖，确保每轮推进之前有清晰假设。
  - `架构优化`：在版本间隙收口运行态、角色记忆、监控等平台能力。
  - `功能开发`：落实版本承诺的必修功能，保持小步推进。
  - `高价值功能探索`：为下一版本识别可放大的新方向，把发现写成可验证的任务。
- 这几个泳道不是孤立的，而是按照标准迭代流程串联：
  `需求提出 -> 澄清/评审 -> 形成基线 -> 变更控制 -> 开发实现 -> 基于基线测试 -> 验收 -> 归档回溯`。
  每次醒来，先定位本轮所在生命周期阶段，再在流水线中捕捉 lane、验证点和证据，确保 7x24 巡检的产出都能落在当前版本基线上。

### 4.1 本版目标
- 把 `workflow` 从“能跑”推进到“连续跑、可恢复、可验证、可继续派下一轮”。
- 先解决工程质量和系统稳定性问题，再放大新特性增量。
- 让我作为 PM 的自迭代推进不再依赖临场记忆，而是依赖计划真相源 + 任务接力 + 定时唤醒。

### 4.2 必修复特性
1. `prod` 入口与 supervisor 稳定恢复，避免停机后 7x24 链直接断。
2. `任务中心` 对 stale running / 假 running / 运行句柄缺失的状态收口必须稳定。
3. `[持续迭代] workflow` 的 schedule 和任务中心接力必须不断链。
4. 开发工作区 -> `workflow_code` -> `test` -> `prod candidate` 的发布链必须稳定，不允许再被错误 remote 或脏工作区漂移破坏。

### 4.3 质量提升项
1. 建立 PM 版本计划真相源，并要求自迭代任务先对照版本计划推进。
2. 固化工程边界：
   1. 双仓边界
   2. 开发工作区边界
   3. 运行态与记忆态边界
3. 补齐与当前主线直接相关的定向验收：
   1. stale run recovery
   2. dashboard runtime fallback
   3. task center/view tabs
   4. self-iteration prompt / schedule contract
4. 收口 `origin` / runtime-config / test gate` 这类易漂移配置的守护策略。
5. 把 `8090` 生产页面体验问题纳入正式版本计划与迭代评审，不再只停留在现场抱怨或临时微调。
6. 把 `workflow_qualitymate` 从“巡检执行者”升级为“多维工程质量监督者”，并让核心 agent 形成每日学习提升与经验沉淀节奏。

### 4.4 性能提升项
1. 继续压 `dashboard` / `workboard` / `assignments` 读路径的慢查询与大 IO。
2. 继续控制任务运行 trace 与详情回传体积，避免页面和 token 消耗失控。
3. 对所有高频“真相读取”链路，优先做“轻量快路径 + 文件真相兜底”。

### 4.5 本版不承诺的新特性
1. 不优先扩张新的复杂业务模块。
2. 不优先追求大 UI 改版。
3. 不把多 agent 完全自治当成 `V0.1` 交付目标。
4. 不在本版承诺“无人监管长期自治”；本版只承诺“连续推进机制更稳”。

### 4.6.2 当前 `8090` 页面体验问题分析（2026-04-09）
1. 固定窗口连续工作体验不足
   1. 当前任务看板与详情虽然局部已有滚动容器，但 `workboard` 仍以单列长卡片堆叠为主，`summary -> group list -> schedule preview` 全部串在一个纵向流里，长时间使用时容易变成“信息都在，但不好连续工作”。
   2. 现有 `V1-P11` 只覆盖“滚动条要出现”，范围太窄，不足以解决 `8090` 页面整体的固定窗口工作面问题。
2. 图形化表达不足
   1. 当前任务图本身已有节点/连线，但工作状态看板主体仍主要依赖数字、文字卡片和按钮列表。
   2. 用户难以快速判断“当前主线是谁、保底是谁、谁在接棒、谁只是排队、哪条失败需要优先处理”，这对 `7x24` 连续运行尤其不够。
3. 刷新状态感知不足
   1. 目前只有局部位置会提示“正在刷新任务图”或“刷新方式 event_stream/兜底轮询”，页面缺少统一的“刷新中 / 已刷新 / 已过期 / 重连中 / 数据时间戳”表达。
   2. 用户很容易分不清“页面没更新”“页面在后台刷新”“数据已过期但还没重连成功”这几种状态。
4. 长耗时连续工作反馈不足
   1. 对长时间运行的连续性工作，当前更多是暴露原始事件流与 run 细节，缺少阶段感、预计等待感和后台仍在工作的明确反馈。
   2. 页面虽然能看到 `running`，但用户仍可能体感为“卡住了”或“不知道现在该等、该刷新还是该切走”。
5. 当前版本计划缺少 UCD 驱动的持续迭代机制
   1. 现有任务包以稳定性、质量、升级连续性为主，缺少一条专门把“体验问题 -> UCD 方案 -> 分批交互优化 -> 验收”串起来的节奏。
   2. 因此本轮把 `V1-P13 ~ V1-P16` 明确补进来，作为后续每轮迭代都要持续推进的体验主线，而不是一次性装修。
6. 当前 `UCD` 已经不再只是 `workflow(pm)` 的临时兼职泳道
   1. `V1-P13` 的基线与首批切片虽然最初由 `workflow(pm)` 直接起头，但这条线已经具备长期专业化价值，不应长期停留在“以后再考虑孵化角色”。
   2. 因此从当前版本计划起，`workflow_ucdmate` 正式进入协作口径：短期由 `workflow(pm)` 继续持有真相源与接棒判断，合适的 release boundary 窗口后由 `workflow_ucdmate` 承接长期体验治理与迭代执行。

### 4.6.5 `V1-P2` 发布边界收口模式（2026-04-09）
1. `7x24` 不再把“本地 `ahead/dirty` 但主线继续扩写”当成普通 backlog；只要某一轮已经产生已验证但未推送的代码改动，这一轮本身就还不算完整结束；若 `.repository/<developer_id>` 出现 `ahead > 0`、存在已验证但未推送的 dirty tracked 改动，或中途失败后遗留 dirty/ahead，下一轮就默认切到“发布边界收口模式”。
2. 该模式下的第一责任按工作区归属，而不是按任务包名称归属：哪个 `.repository/<developer_id>` 产生了 dirty/ahead，哪个工作区就先负责在本工作区完成 `commit / push / 根仓同步`；`workflow_devmate` 这类任务负责人可以牵头工程治理，但不能替代所有工作区统一代持未推代码。
3. 进入该模式后，执行顺序固定为：冻结命中工作区的新增实现 -> 盘点该工作区的 `git status --short --branch` / `git diff --stat` -> 按最小验证面切推送批次 -> 跑命中该批次的最小验证 -> 在该工作区 `commit / push` 回 `../workflow_code/main` -> 再恢复 `test -> candidate`。
4. `V1-P14 / V1-P16` 这类体验切片在 release boundary 未收口前，不再顺手继续扩面；除非现场出现 `prod` 主链断裂、升级止血或更高优先事故，否则 helper 也不在同一批 dirty 文件上继续加码。
5. 若需要跨工作区接力同一条代码主线，前一工作区必须先把已验证改动推回根仓；若只能先交 patch，则接力方必须先回放到自己的工作区并推根仓，再继续后续开发。
6. 每轮 `7x24` 必须按工作区显式回写 `developer_id / root_sync_state / ahead_count / dirty_tracked_count / untracked_count / push_block_reason / next_push_batch`，不能只写“继续收口 Git 边界”这种口头结论。
7. 对成功轮来说，“已验证但未 commit/push”的状态不再允许跨轮滞留；对失败轮来说，下一轮第一优先级必须先处理这批历史 dirty/ahead，而不是基于错误现场继续扩写。
8. 当前专项方案真相源已落到 `docs/workflow/reports/7x24发布边界收口方案-20260409.md`；后续主线和保底巡检都按这份方案判断是否要先恢复小步推根仓节奏。
