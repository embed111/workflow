# 默认发布约束

更新时间：`2026-04-03`

## 1. 目的
- 把“代码改完后的默认发布动作”固化为强约束，避免后续执行依赖口头记忆。
- 该文件用于约束 workflow 仓库内后续默认发布行为；若与零散对话表述冲突，以本文件为准。

## 2. 适用范围
- 适用于当前工作区内所有实际代码改动。
- 不适用于纯讨论、纯排查、纯文档改动或用户明确要求“先不要发”的场景。

## 2.1 当前协作口径
- 顶层协作真相源：`../workflow/协作约定.md`
- 当前项目仍处于单人维护阶段，默认由 Codex 承担代码同步、开发验证、`test` 部署与 `prod` 候选刷新。
- 当前默认开发工作区为 `../workflow/.repository/pm-main/`；后续新增协作者时，再切分到各自的 `.repository/<developer_id>/`。
- 用户提到 `./run_workflow.bat` 时，默认指 `../workflow/` PM 仓顶层便捷入口；该入口代理到默认开发工作区的启动脚本，并统一落到 PM 顶层 `.running/prod`。
- 若用户只要求“推送最新生产环境”，默认动作是生成或刷新 `prod` 升级候选；只有用户明确要求“更新生产环境代码”或“部署生产环境”时，才进入正式环境升级。

## 3. 默认发布链路
当满足以下前置条件时，默认必须继续执行发布动作，不再额外等待“是否推送”的确认：
- 已完成改动；
- 已完成必要验证；
- 已跑完门禁，且结果通过。

默认发布顺序固定为：
1. 先把 `../workflow_code` 最新代码同步到对应开发工作区。
2. 在开发/临时环境完成验证。
3. 执行门禁并确认通过。
4. 执行 `deploy_workflow_env.ps1 -Environment test` 部署 `test`。
5. 同步生成或刷新 `prod` 升级候选。
6. 由用户在生产环境页面中手动执行升级；若用户明确要求，也可由 Codex 执行正式环境更新。

### 3.1 分批推根仓
- 后续允许按“完成一部分改动 -> 做完对应测试/门禁 -> 推回 `../workflow_code`”的小批次节奏执行。
- 不要求必须等一个大主题全部结束后才推根仓。
- 但每次推回根仓之前，当前这部分改动必须已经完成最小必要验证，并能说明对应证据。

### 3.2 单项真实验证与全流程 mock
- 默认不把“真实单项验证”和“真实全流程回归”叠加成重复成本。
- 若某条关键能力已经通过对应单项真实测试，则后续同批次的全流程回归默认允许改用 mock/stub。
- mock 全流程的主要目标是验证：
  - 阶段编排是否正确
  - 状态流转是否闭合
  - 删除/完成/门禁等关键分支是否仍可走通
- 只有在以下场景才继续要求真实全流程：
  - 用户明确要求跑真实全流程
  - 本轮改动直接触达真实外部依赖、真实长任务调度或 mock 无法覆盖的运行时差异
  - 历史证据表明 mock 与真实链路存在显著偏差，需要专项回归
- 交付说明中应明确标注：哪些 case 是真实验证，哪些 case 是 mock 全流程回归。

### 3.3 行数门禁自检
- 每次存在实际代码改动时，默认应在验证阶段自检：
  - `python scripts/quality/check_workspace_line_budget.py --root .`
- 判定口径分两层：
  - `Hard Gate`：必须通过。
  - `Refactor Trigger Gate / Guideline Gate`：用于判断本轮是否需要顺手做职责拆分或设计模式重构。
- 默认执行原则：
  - 若本轮触达的代码文件命中 `> 1000` 行重构触发门槛，或命中对应 advisory 上限，则应优先在本轮做设计模式重构/职责拆分，再继续发布。
  - 若命中的是历史未触达文件，只记录结果，不阻塞本轮无关发布。
- 推荐重构方向：
  - 先做“职责拆分 + 兼容门面保留”，避免一次性大改。
  - 参考：`docs/workflow/design/详细设计-代码体积治理与无损重构.md`

## 4. 关键定义
- `发布到测试环境`：
  - 指部署 `test` 环境，使最新通过门禁的版本在 `test` 可用。
- `推送到生产环境`：
  - 在本仓库中固定解释为“生成/刷新 `prod` 升级候选”，让生产环境可看到最新候选版本。
  - 不等于直接覆盖 `prod`，不等于自动执行正式升级。
- `生产环境升级`：
  - 固定由用户手动触发。
  - 默认不自动执行 `prod` 切换，不自动调用正式升级应用动作。

## 5. 执行边界
- 默认允许：
  - 部署 `test`
  - 生成或刷新 `prod` candidate
  - 回传 candidate 版本号、证据路径、门禁结果
- 默认禁止：
  - 未经用户明确要求，直接执行 `scripts/deploy_workflow_env.ps1 -Environment prod -AllowDirectProdDeploy`
  - 未经用户明确要求，自动应用生产升级
  - 为了发布 `test` 而停止或干扰 `.running/prod` 正在服务的生产进程

## 6. 推荐命令口径
- 同步当前开发工作区：
  - `git -C .repository/<developer_id> fetch --all --prune`
  - `git -C .repository/<developer_id> pull --ff-only`
- 本地启动正式入口：
  - 在 `../workflow/` 根目录执行：`.\run_workflow.bat`
- `test` 部署：
  - `powershell -NoProfile -ExecutionPolicy Bypass -File .repository/<developer_id>/scripts/deploy_workflow_env.ps1 -Environment test`
- `prod` 直发：
  - 仅当用户明确要求“部署到生产环境”或明确允许维护窗口时才可使用。
  - 命令示例：
    - `powershell -NoProfile -ExecutionPolicy Bypass -File .repository/<developer_id>/scripts/deploy_workflow_env.ps1 -Environment prod -AllowDirectProdDeploy`

## 7. 交付时必须说明
- 本轮是否已完成 `test` 部署。
- 本轮是否已生成/刷新 `prod` 候选。
- 用户下一步是否只需在生产页面手动升级。
- 若未执行默认发布链路，必须明确说明阻塞原因。

## 8. 优先级
- 本文件是“默认发布动作”的治理约束。
- `AGENTS.md` 需引用本文件。
- `state/user-preferences.md` 作为偏好事实源，应与本文件保持一致。
