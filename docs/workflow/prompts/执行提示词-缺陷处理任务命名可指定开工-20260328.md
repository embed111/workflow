## 2. 当前已核实事实
1. `任务中心` 手工创建任务已经支持自定义任务名称，当前前端有任务名称输入框，提交时会带 `node_name`。
2. `需求与缺陷 -> 缺陷` 当前“创建处理任务”链路还没有把这种能力接过去。
3. 以当前实现为准，缺陷中心建单链路至少存在以下偏差：
   1. 缺陷前端当前发起 `/process-task` 时只传了 `operator`，没有传任务名称；
   2. 后端 `create_defect_process_task(...)` 当前只消费 `operator / auto_queue / assigned_agent_id`；
   3. 处理链节点名当前固定写成“分析缺陷 / 修复缺陷 / 推送到目标版本”；
   4. 复核链节点名当前固定写成“复核争议”。
4. 用户当前明确要的是：
   1. 缺陷被确认后，去任务中心提任务时可以指定任务名称；
   2. 即使没有手工输入，也必须自动带出具备缺陷语义的默认任务名称；
   3. 该能力不能破坏既有“串行顺序建单 + 单主动处理位 + 幂等防重复建链”约束。

## 3. 本轮目标
1. 补齐 `缺陷 -> 任务中心` 的任务命名能力，而不是继续让任务中心里只出现一组通用动作名。
2. 让以下三处名称语义保持一致：
   1. `缺陷` 视图中的任务引用；
   2. 顺序建单摘要中的当前主动处理项；
   3. `任务中心全局主图` 中的真实节点名称。
3. 保证手动建单和自动顺序建单都采用同一命名真相源。

## 4. 你必须按以下口径实现
1. 建单前名称确认能力：
   1. `处理缺陷` 与 `提交复核` 两条链路都必须支持任务名称确认；
   2. 首次建单前必须提供一个可见、可编辑的任务名称输入入口；
   3. 交互形态可按现有界面风格实现为弹层、抽屉或详情内联输入，但必须发生在真正建单之前。
2. 默认命名规则：
   1. 默认任务名称基名优先使用：`DTS 展示编号 + 空格 + 缺陷摘要`；
   2. 若缺陷摘要为空，至少保留 `DTS 展示编号`，再追加简短兜底语义；
   3. 不允许默认值退化成“分析缺陷”这种与具体缺陷脱钩的纯动作名。
3. 处理链与复核链命名规则：
   1. `process` 链节点名称必须围绕同一任务名称基名派生；
   2. 至少保证三类节点语义清晰可辨：
      1. `<任务名称基名> - 分析`
      2. `<任务名称基名> - 修复`
      3. `<任务名称基名> - 推送到目标版本`
   3. `review` 链节点名称至少应表达为：`<任务名称基名> - 复核`；
   4. 分隔符样式可微调，但不得丢失“缺陷对象 + 动作阶段”两层信息。
4. 自动顺序建单不得回退：
   1. 当“按顺序创建任务”总开关开启，系统自动为下一条缺陷创建处理链时，也必须使用同一默认命名规则；
   2. 自动路径不得因为没有人工输入而回退成固定通用名；
   3. 自动路径与手动路径必须共享同一套后端命名真相源。
5. 幂等与重入约束不能被破坏：
   1. 对同一 `report_id + action_kind`，重复点击、刷新、自动推进、重试都不得因为名称变化而创建第二条并存有效链；
   2. 若既有链路已存在，应返回既有任务引用与既有名称，不要假装本次改名已生效；
   3. 当前轮不要求对历史已存在的固定通用名任务做批量回写改名。
6. 接口契约要求：
   1. 面向 UI 和自动化测试复用的 `process-task` / `review-task` 接口都必须支持显式传入任务名称基名或等效字段；
   2. 若调用方未传该字段，后端必须按统一默认规则自行派生命名；
   3. 两条链路必须采用一致契约，不允许一条支持、一条缺失。

## 5. 推荐排查与实现落点
1. 先补前端入口：
   1. 缺陷详情中的 `处理缺陷` 和 `提交复核` 触发前，增加任务名称确认能力；
   2. 默认预填 `DTS 展示编号 + 缺陷摘要`。
2. 再补接口载荷：
   1. 检查 `/api/defects/{report_id}/process-task`；
   2. 检查 `/api/defects/{report_id}/review-task`；
   3. 补齐任务名称字段透传。
3. 最后补后端命名真相源：
   1. 统一处理“前端显式传入”与“后端默认派生”两种路径；
   2. 修正当前固定写死的节点名称构造；
   3. 确保缺陷详情任务引用标题与任务中心节点名称一致或可直接推导一致。

## 6. 明确禁止
1. 不要只改前端显示文案，而后端仍继续创建“分析缺陷 / 修复缺陷 / 推送到目标版本 / 复核争议”。
2. 不要只让手动点击支持命名，而自动顺序建单仍回退成固定通用名。
3. 不要为了支持命名而放松既有幂等约束，导致同一缺陷动作重复挂链。
4. 不要要求用户每次都必须手工输入名称；必须存在稳定可用的默认派生命名。
5. 不要把这轮实现扩大成“历史任务批量重命名工程”。

## 7. 最低验收回传
1. 页面证据：
   1. `处理缺陷` 前出现任务名称确认入口的截图；
   2. `提交复核` 前出现任务名称确认入口的截图；
   3. 自动顺序建单后，缺陷详情任务引用标题已带语义化名称的截图。
2. 接口证据：
   1. 至少 1 次手动 `process-task` 请求与响应；
   2. 至少 1 次手动 `review-task` 请求与响应；
   3. 至少 1 次自动顺序建单后 `/api/defects` 或等效详情接口返回的任务引用；
   4. 证明请求未显式传名称时，后端仍生成了语义化名称。
3. 任务中心证据：
   1. 对应 `任务中心全局主图` 中的分析/修复/推版节点名称截图或接口返回；
   2. 至少 1 条复核任务名称截图或接口返回；
   3. 证明名称已从“固定通用动作名”升级为“缺陷语义 + 动作阶段”。
4. 幂等证据：
   1. 同一条缺陷重复触发 `process-task` 不会生成第二条并存有效链；
   2. 同一条缺陷重复触发 `review-task` 不会生成第二条并存有效链；
   3. 若既有链已存在，返回的仍是既有任务引用与既有任务名称。
5. 代码落点：
   1. 缺陷页建单前名称确认入口位置；
   2. `process-task` / `review-task` 请求载荷位置；
   3. 后端默认派生命名与固定名称替换位置；
   4. 任务引用标题与节点名称同步位置；
   5. 幂等防重复建链位置。
