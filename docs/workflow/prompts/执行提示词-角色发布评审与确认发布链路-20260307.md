# 执行提示词-角色发布评审与确认发布链路-20260307

请按照以下要求，在 `../workflow` 工作区实现“角色发布评审与确认发布链路”。

## 1. 先阅读的文档
1. `docs/workflow/requirements/需求详情-agent发布管理与版本切换.md`
2. `docs/workflow/design/详细设计-agent发布管理与版本切换.md`
3. `docs/workflow/overview/需求概述.md`
4. `docs/workflow/requirements/需求详情-角色画像发布格式与预发布判定.md`
5. `docs/workflow/design/详细设计-角色画像发布格式与预发布判定.md`

重点关注以下口径：
1. 发布流程固定为四步：
   1. 进入发布评审
   2. 委派对应工作区 agent 生成发布报告
   3. 人工审核发布报告
   4. 确认发布
2. 发布报告生成主链路固定复用“会话入口 agent 分析”现有的 `Codex` 执行机制：
   1. `codex exec`
   2. 提示词版本化
   3. 分析链路可见
   4. 证据落盘
3. “确认发布”不是单纯写业务状态，而是必须执行真实发布动作：
   1. 使用 `Git + release note`
   2. 发布后再次按当前版本识别规则校验是否成功
   3. 未校验通过前不得标记为正式发布成功
4. 若确认发布失败，必须自动启动 `../workflow` 工作区 agent 兜底：
   1. 先输出失败原因
   2. 自动重试一次发布
   3. 输出重试结果与下一步建议
5. 发布报告同时承担“发布评审依据 + 角色详情公开介绍”双重职责：
   1. 自然语言正文统一使用第一人称视角
   2. 必须展示最新正式发布目标版本的全量能力
   3. 正式发布成功后，角色详情页优先展示该报告

## 2. 本次实现目标
请在 `../workflow` 中实现以下能力：

1. 发布评审入口
   1. 对 `pre_release` 状态的 agent 提供“进入发布评审”入口。
   2. 点击后创建发布评审记录。

2. 发布报告生成
   1. 系统自动委派对应工作区 agent 通过 `codex exec` 生成发布报告。
   2. 报告同时承担两个职责：
      1. 发布评审页展示“功能差异报告”，重点说明相对上一正式发布版本的变化、风险与证据。
      2. 正式发布成功后绑定为角色详情页的最新正式发布第一人称全量能力介绍。
   3. 报告必须至少包含：
      1. `target_version/current_workspace_ref`
      2. `first_person_summary`
      3. `full_capability_inventory`
      4. `knowledge_scope`
      5. `agent_skills`
      6. `applicable_scenarios`
      7. `change_summary`
      8. `capability_delta`
      9. `risk_list`
      10. `validation_evidence`
      11. `release_recommendation`
      12. `next_action_suggestion`
   4. 将 Codex 生成的报告理解为 agent 面向发布评审环节提交的“述职报告”，自然语言字段统一使用第一人称视角描述：
      1. 例如：`我是…`、`我当前能做…`、`我本次补充了…`、`我识别到…`、`我建议下一步…`
      2. 适用于：`first_person_summary`、`change_summary`、`risk_list`、`validation_evidence`、`next_action_suggestion`
      3. `capability_delta` 只表达变化，不替代全量能力介绍
   5. 页面需展示分析链路：
      1. 提示词
      2. Codex 执行摘要
      3. stdout/stderr 路径或查看入口
      4. 结构化报告路径或查看入口

3. 人工审核
   1. 仅当发布报告生成完成后，允许人工提交审核结论。
   2. 审核结论至少支持：
      1. `approve_publish`
      2. `reject_continue_training`
      3. `reject_discard_pre_release`
   3. 必须记录：
      1. `reviewer`
      2. `review_comment`
      3. `reviewed_at`

4. 确认发布
   1. 仅当人工审核结论为 `approve_publish` 时，允许点击“确认发布”。
   2. 点击后进入真实发布执行阶段，不允许只写数据库状态。
   3. 发布动作必须使用 `Git + release note`。
   4. 发布后必须再次按当前版本识别规则读取 `Git + release note`，确认发布是否真正成功。
   5. 只有成功校验通过后，才允许：
      1. 写入正式发布记录
      2. 将 agent 生命周期状态切回发布态
      3. 将本次发布报告绑定为角色详情页的最新正式发布介绍来源

6. 角色详情联动
   1. 角色详情页优先展示最新正式发布版本绑定的发布报告。
   2. 首屏需优先看到：
      1. 第一人称角色述职摘要
      2. 全量能力清单
      3. 知识范围
      4. Agent Skills
      5. 适用场景
   3. 预发布报告在正式发布成功前不得覆盖当前正式发布角色详情。
   4. 历史正式发布版本若缺少绑定报告，允许回退到结构化字段渲染，但需标记来源。

5. 发布失败兜底
   1. 若 Git 发布失败、release note 写入失败或发布成功校验失败：
      1. 将状态标记为发布失败
      2. 自动启动 `../workflow` 工作区 agent 兜底
   2. 兜底动作固定为：
      1. 分析并输出失败原因
      2. 自动重试一次发布
      3. 再次执行发布成功校验
      4. 输出重试结果与下一步建议
   3. 若自动重试一次后仍失败：
      1. 保持失败态
      2. 明确提示人工介入
      3. 不得伪装为发布成功

## 4. 界面要求
1. 发布评审详情页显示四步状态条：
   1. 进入发布评审
   2. 生成发布报告
   3. 人工审核
   4. 确认发布
2. 确认发布后需继续显示子阶段：
   1. Git 发布中
   2. release note 处理中
   3. 成功校验中
   4. 完成 / 失败
3. 若进入兜底，还需显示：
   1. 兜底中
   2. 兜底完成
   3. 兜底失败
4. 任何执行中阶段都不得静默等待。

## 5. 失败策略
1. 发布报告生成失败：`fail-closed`
   1. 不允许跳过报告直接确认发布。
2. 人工审核未通过：
   1. “确认发布”按钮必须禁用。
3. Git 发布失败 / release note 失败 / 成功校验失败：
   1. 必须进入 `publish_failed`
   2. 必须触发兜底
4. 兜底启动失败：
   1. 必须记录日志
   2. 必须提示人工介入
