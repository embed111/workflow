## 2. 本轮要解决的核心问题
1. 总开关位置要改：
   1. 不再放在详情正文里。
   2. 需要放到左侧 `缺陷记录` 标题栏右侧，作为全局控制入口。
2. 开关效果要改清楚：
   1. 用户要的不是“点一下只启动当前一条”。
   2. 用户要的是总开关开启后，系统会持续开始解决 bug。
3. 持续解决 bug 的准确含义：
   1. 所有缺陷共用同一个总开关；
   2. 同一时刻只允许 1 条缺陷占用主动处理位；
   3. 上一条缺陷进入 `已解决` 或 `已关闭` 后，系统必须继续自动推进下一条缺陷进入真实处理链；
   4. 只要总开关未关闭，就要持续重复这个过程，直到队列为空。

## 3. 你必须按以下口径实现
1. 总开关展示口径：
   1. `缺陷记录` 标题栏右侧展示独立总开关；
   2. 该开关是全局控制，不属于某一条缺陷详情；
   3. 不要再把总开关埋进右侧详情正文。
2. 详情区展示口径：
   1. 右侧当前缺陷详情区只保留 `顺序建单摘要`；
   2. 摘要至少展示：
      1. 当前主动处理缺陷；
      2. 下一条待建单；
      3. 串行规则说明。
3. 开关行为口径：
   1. 总开关关闭时：
      1. 系统不得自动为任何缺陷创建新的 `任务中心` 任务；
      2. 仍允许用户手动点击 `处理缺陷` / `提交复核`。
   2. 总开关开启时：
      1. 若当前没有占用主动处理位的缺陷，系统立即按排序规则自动推进首条可处理缺陷；
      2. 若当前已有主动处理缺陷，系统等待其进入 `已解决` 或 `已关闭`；
      3. 一旦上一条退出主动处理位，系统立即继续推进下一条；
      4. 该过程必须持续进行，直到队列为空或用户关闭总开关。
4. 不得回退的既有约束：
   1. 缺陷列表仍按 `task_priority` 高到低、同优先级 `reported_at` 升序；
   2. 显式优先级仍是真相源；
   3. 所有缺陷仍共用单主动处理位；
   4. 所有缺陷任务仍统一收口到 `任务中心全局主图`；
   5. 同一缺陷动作仍不得重复挂链或重复建图。

## 4. 明确禁止
1. 不要只改标题栏位置，而不保证开启后持续推进下一条缺陷。
2. 不要只做前端按钮切换，而后端自动推进逻辑仍停留在“一次性推进一条”。
3. 不要把总开关做成仅对当前选中缺陷生效的局部控制。
4. 不要让总开关开启后出现多条缺陷同时进入主动处理位。
5. 不要把“连续解决 bug”实现成前端假状态；必须以真实建单、真实任务引用、真实任务中心链路为准。

## 5. 推荐实现顺序
1. 先改展示结构：
   1. 标题栏放总开关；
   2. 详情区保留摘要。
2. 再核对全局控制语义：
   1. 标题栏总开关是否真正驱动全局队列模式；
   2. 是否仍有局部状态残留在单条缺陷详情里。
3. 最后核对持续推进效果：
   1. 开启后是否会自动推进首条可处理缺陷；
   2. 上一条退出主动处理位后是否会自动推进下一条；
   3. 是否会持续重复到队列为空；
   4. 中间是否始终保持单主动处理位。

## 6. 最低验收回传
1. 页面证据：
   1. `缺陷记录` 标题栏可见独立总开关的截图；
   2. 当前缺陷详情区展示 `顺序建单摘要` 的截图；
   3. 总开关开启态与关闭态各 1 张截图。
2. 行为证据：
   1. 证明开启总开关后，系统会自动推进首条可处理缺陷；
   2. 证明上一条缺陷进入 `已解决` 或 `已关闭` 后，系统会继续自动推进下一条；
   3. 证明该过程会持续进行，直到队列为空或手动关闭总开关；
   4. 证明整个过程中始终只有 1 条缺陷占用主动处理位。
3. 接口证据：
   1. `/api/defects` 返回中的队列摘要；
   2. 对应缺陷详情中的 `task_ref`；
   3. 对应 `任务中心全局主图` 中的节点链；
   4. 若连续推进多条缺陷，至少回传前后两条缺陷的衔接证据。
4. 代码落点：
   1. 标题栏总开关渲染位置；
   2. 全局开关状态读写位置；
   3. “上一条退出主动处理位后自动推进下一条”的实现位置；
   4. 单主动处理位门禁位置。
