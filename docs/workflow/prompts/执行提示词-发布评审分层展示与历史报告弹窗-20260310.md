## 2. 本次实现目标
请在保持当前 `workflow` 视觉风格、信息架构和已有训练中心交互语言不变的前提下，实现以下调整：

1. 首页只展示当前正在进行的发布评审。
   1. `当前进行中的发布评审` 区域只承载当前工作区尚未完成的评审链路。
   2. 已正式发布版本的历史评审记录不得继续常驻展示在首页。

2. 已发布版本的历史评审记录改为通过“查看发布报告”弹窗查看。
   1. 在 `发布版本` 列表中，对存在可展示来源的版本显示“查看发布报告”按钮。
   2. 点击后以 `dialog/modal` 打开该版本对应的发布报告/发布评审记录。
   3. 弹窗内容优先复用现有 `renderTrainingCenterReleaseReport` 的模块结构，不要重新发明一套展示体系。

3. 版本切换与报告查看必须严格绑定。
   1. 用户切换到不同已发布版本后，点击“查看发布报告”必须打开该版本对应的历史记录。
   2. 不得误打开当前预发布评审内容。
   3. 不得误打开其他版本的历史报告。

4. 对未绑定可展示报告的已发布版本，仅显示不可查看原因。
   1. 保留只读提示。
   2. 不展示“查看发布报告”按钮。
   3. 不弹出空弹窗。

5. 角色详情区域保持现有样式与结构。
   1. 保持左侧主导航位置不变。
   2. 保持角色详情的头像、标题、角色介绍、版本切换、单列分段信息结构不变。
   3. 不要因为本次改动引入新的视觉风格或新的页面布局语言。

6. 长内容滚动语义必须保留。
   1. 右侧详情区仍需支持长内容滚动。
   2. 当前进行中的发布评审区在长内容下需可滚动。
   3. 发布报告弹窗正文需可滚动。
   4. 不要把长内容强行压缩成一屏静态摘要。

## 3. 当前评审区要求
请保留并对齐当前现有发布评审链路，不要把它简化成只剩几条摘要文案。

至少应继续承载以下内容：
1. 状态提示（如目标版本、工作区基线、当前阶段）。
2. 运行进度区。
3. 四步阶段条：
   1. 进入发布评审
   2. 生成发布报告
   3. 人工审核
   4. 确认发布
4. 发布后子阶段 / 兜底子阶段：
   1. Git
   2. release note
   3. verify
   4. fallback
5. 结构化发布报告。
6. 分析链路：
   1. 提示词版本/路径
   2. 报告路径
   3. stdout/stderr
   4. trace 目录
   5. Codex 摘要
7. 人工审核结论：
   1. 审核结论
   2. 审核人
   3. 审核时间
   4. 审核意见
8. 确认发布执行日志。
9. 失败兜底结果。

## 4. 发布报告弹窗要求
发布报告弹窗请尽量复用现有模块渲染，不要用“一个摘要卡片 + 若干标签”替代。

弹窗至少应支持以下模块：
1. 核心信息
   1. 目标版本
   2. 工作区基线
   3. 上一正式版本
   4. 发布建议
   5. 下一步建议
2. 正式发布角色介绍预览
3. 功能差异报告
4. 风险清单
5. 验证证据
6. 补充提示
