# 执行提示词-创建workflow_tester角色并验证创建链路-20260327

请按照文件《docs/workflow/prompts/执行提示词-创建workflow_tester角色并验证创建链路-20260327.md》中的提示词执行。

## 1. 先读文件
1. `docs/workflow/overview/需求概述.md`
2. `docs/workflow/requirements/需求详情-创建角色工作流与任务绑定.md`
3. `docs/workflow/requirements/需求详情-训练中心创建角色链路修正.md`
4. `docs/workflow/prototypes/创建角色工作流-预稿/创建角色工作流预稿.html`

## 2. 本轮核心目标
1. 使用 `http://127.0.0.1:8090` 当前真实可用的 `训练中心 -> 创建角色` 功能，尝试创建一个新角色：`workflow_tester`。
2. 目标不是手工伪造一条角色记录，而是验证“真实创建角色链路现在能不能成功完成这件事”。
3. 若创建成功，该角色应可用于后续 `workflow` 生产环境低风险功能巡检、问题复现、证据采集与缺陷提报。

## 3. 角色目标口径
1. `role_name`：`workflow_tester`
2. `role_goal`：
   1. 在 `workflow` 生产环境中执行低风险功能测试与回归验证；
   2. 复现问题、收集截图与接口证据；
   3. 对确认问题通过 `需求与缺陷 -> 缺陷` 模块提交缺陷记录；
   4. 为后续验收与缺陷闭环提供稳定测试角色。
3. `core_capabilities` 至少覆盖：
   1. 功能巡检与基础 smoke test；
   2. 缺陷复现与步骤整理；
   3. 截图、接口返回与关键状态取证；
   4. 按约定优先级提交缺陷记录。
4. `boundaries` 至少覆盖：
   1. 不直接改代码；
   2. 不直接改数据库伪造角色创建成功；
   3. 不绕过“创建角色”真实链路手工落角色；
   4. 不做高风险生产配置改动。
5. `applicable_scenarios` 至少覆盖：
   1. 新功能 smoke test；
   2. 回归验收；
   3. 缺陷复现；
   4. 上线后稳定性巡检。
6. `collaboration_style` 至少覆盖：
   1. 先执行；
   2. 再给出证据；
   3. 发现问题优先沉淀为缺陷记录；
   4. 不把模糊猜测当成缺陷结论。

## 4. 必须遵守的执行约束
1. 必须优先使用真实 `创建角色` UI/API 链路完成本次创建。
2. 不允许直接写库、补文件、造假角色记录来冒充“创建成功”。
3. 如果链路中途失败，必须按“真实失败”回传证据，而不是跳过失败点继续伪造后续状态。
4. 若当前链路支持字段级编辑，应优先在真实页面中把字段修正到目标口径；不要把脏数据留到最终角色上。
5. 若 `role_spec` 出现聊天污染、状态假完成、任务未真实生成等问题，应直接作为本轮失败或缺陷证据返回。

## 5. 你要验证的关键点
1. `workflow_tester` 名称是否能真实写入 `role_spec.role_name`，而不是停留在历史草稿名。
2. `role_goal / boundaries / applicable_scenarios / collaboration_style` 是否干净，没有混入寒暄语、抱怨语或元对话。
3. `可开始` 与 `已真正开始创建` 是否被正确区分：
   1. 若只是可开始，不能伪装成已创建中；
   2. 若已开始创建，必须有真实任务引用、真实开始时间或等效证据。
4. 若创建成功，角色是否真正可进入后续可用状态，而不是只停留在一条孤立草稿记录。
5. 若角色创建成功，是否能用该角色发起至少 1 条最小工作验证：
   1. 创建一个该角色会话；
   2. 给它下达 1 条低风险测试任务；
   3. 证明该角色已可承担后续测试/提缺陷工作。

## 6. 推荐执行顺序
1. 进入 `训练中心 -> 创建角色`。
2. 按本提示词中的目标口径填写或对齐 `workflow_tester` 角色草稿。
3. 核对字段污染问题：
   1. 名称是否正确；
   2. 目标是否正确；
   3. 边界/场景/协作方式是否被脏文本污染。
4. 若链路允许，推进到真实 `开始创建`。
5. 若成功创建，再用新角色做 1 条最小 smoke task，证明该角色不是“只创建成功但无法工作”。
6. 若任一步失败，立即保留现场并回传失败证据。

## 7. 失败时必须回传的证据
1. 页面截图：
   1. 创建角色会话页面；
   2. 当前 `role_spec` 展示；
   3. 报错、卡住或超时位置。
2. 接口证据：
   1. 当前创建会话对应的 `/api/training/role-creation/sessions/<session_id>` 返回；
   2. 若有任务引用，回传对应任务接口结果；
   3. 若有角色记录，回传对应角色查询结果。
3. 运行证据：
   1. 相关 trace 路径；
   2. 相关日志路径；
   3. 若有工作区初始化结果，回传对应路径。
4. 结论必须明确写成三选一：
   1. `创建成功，且角色可工作`
   2. `创建成功，但角色仍不可工作`
   3. `创建失败`

## 8. 成功时最低回传
1. `workflow_tester` 创建成功的页面截图。
2. 干净 `role_spec` 的截图或结构化返回。
3. 真实开始创建/真实创建完成的证据：
   1. `started_at` 或等效开始证据；
   2. 真实任务引用；
   3. 角色记录或角色详情。
4. 新角色的最小可用性证据：
   1. 已创建会话；
   2. 已下达 1 条低风险测试任务；
   3. 该角色已给出结果或至少进入可执行状态。

## 9. 额外提醒
1. 这轮真正要验证的是“创建角色链路能不能把 `workflow_tester` 真正创建出来并投入工作”，不是只验证静态 UI。
2. 如果你发现当前 `创建角色` 链路依旧存在“字段污染 / 假完成 / 任务未真实生成 / 角色不可用”等问题，不要硬往后推，直接按缺陷返回。
