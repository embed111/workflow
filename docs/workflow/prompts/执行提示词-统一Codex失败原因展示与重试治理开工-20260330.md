# 执行提示词-统一Codex失败原因展示与重试治理开工-20260330

请按照文件《docs/workflow/prompts/执行提示词-统一Codex失败原因展示与重试治理开工-20260330.md》中的提示词执行。

## 1. 先读文件
1. `docs/workflow/overview/需求概述.md`
2. `docs/workflow/reports/排查报告-Codex调用失败原因与重试现状-20260330.md`
3. `docs/workflow/requirements/需求详情-统一Codex失败原因展示与重试治理.md`
4. `docs/workflow/requirements/需求详情-任务中心真实执行化与执行链路可视化.md`
5. `docs/workflow/requirements/需求详情-训练中心创建角色链路修正.md`
6. `docs/workflow/requirements/需求详情-agent发布管理与版本切换.md`
7. `docs/workflow/requirements/需求详情-创建角色工作流与任务绑定.md`

## 2. 本轮目标
1. 把 `../workflow` 中所有真实 `Codex` 请求链路统一治理为：
   1. 失败原因可见；
   2. 失败阶段可见；
   3. 显式可重试或明确禁止原因；
   4. 可下钻查看本次失败证据；
   5. 可看到尝试次数与最近一次尝试结果。
2. 不接受只修某一条链路或只补几句文案。

## 3. 本轮必须覆盖的 5 条链路
1. 策略提取 / `AGENTS.md` 分析
2. 会话任务执行中心
3. 创建角色分析对话
4. 发布评审 / 确认发布兜底
5. 任务中心真实执行

## 4. 你必须落地的统一合同
1. 后端统一失败对象至少包含：
   1. `feature_key`
   2. `attempt_id`
   3. `attempt_count`
   4. `failure_code`
   5. `failure_detail_code`
   6. `failure_stage`
   7. `failure_message`
   8. `retryable`
   9. `retry_action`
   10. `trace_refs`
   11. `failed_at`
2. 前端统一失败卡至少展示：
   1. 失败原因
   2. 失败阶段
   3. 最近尝试时间
   4. 已尝试次数
   5. 重试按钮或禁止原因
   6. 证据入口
   7. 下一步建议
3. `retry_action` 至少支持以下动作语义：
   1. `retry_policy_analysis`
   2. `retry_session_round`
   3. `retry_role_creation_analysis`
   4. `retry_release_review`
   5. `retry_publish`
   6. `rerun_assignment_node`

## 5. 明确禁止
1. 不要只在前端把原始 stderr 摘要换成中文文案，但后端仍没有标准失败码。
2. 不要只在少数链路做统一合同，剩下链路继续沿用 `policy_error / error / failure_reason` 各说各话。
3. 不要继续让页面在后端已给出标准失败原因时仍显示“请求未成功”“执行出现异常”这类泛化提示。
4. 不要把创建角色链路继续停留在“继续发送会自动重试”，必须补显式“重试本轮分析”。
5. 不要让重试动作制造重复有效链路；重复点击、刷新、并发点击都必须有幂等保护。
6. 不要只改文档或原型，不改真实后端/前端链路。

## 6. 推荐实现顺序
1. 先补统一失败对象和失败码映射层。
2. 再补统一前端失败卡组件或等效统一渲染层。
3. 再逐条适配 5 条链路：
   1. 先发布评审，抽出最佳实践；
   2. 再策略提取；
   3. 再会话任务执行；
   4. 再创建角色；
   5. 最后任务中心真实执行。
4. 最后补幂等、重试状态回写和验收取证。

## 7. 最低验收要求
1. 策略提取失败时：
   1. 页面能明确显示失败原因；
   2. 页面能直接点“重试分析”；
   3. 页面能下钻到 `prompt/stdout/stderr/result/gate` 或等效证据。
2. 会话任务执行失败时：
   1. 页面不再只显示“请求未成功 / 执行出现异常”；
   2. 页面仍保留“重试上一轮”；
   3. 重试后有新的尝试状态回写。
3. 创建角色分析失败时：
   1. 页面明确展示失败原因；
   2. 页面新增显式“重试本轮分析”；
   3. “继续发送自动重试”可以保留，但不能再是唯一重试方式。
4. 发布评审 / 确认发布失败时：
   1. 页面继续支持“重新进入发布评审 / 重试发布”；
   2. 两类动作都由统一 `retry_action` 协议驱动；
   3. 自动兜底重试链路也要回写标准失败对象。
5. 任务中心真实执行失败时：
   1. `失败原因` 优先展示标准失败码映射后的中文文案；
   2. 页面保留“重跑任务”；
   3. 重跑不会生成第二条并存有效执行链。

## 8. 最低回传内容
1. 代码落点：
   1. 统一失败对象定义位置；
   2. 统一失败码映射位置；
   3. 统一前端失败卡位置；
   4. 5 条链路各自接入位置。
2. 页面证据：
   1. 5 条链路至少各 1 张失败态截图；
   2. 至少 3 条链路的重试成功或重试失败回写截图。
3. 接口/日志证据：
   1. 标准失败对象示例；
   2. `retry_action` 示例；
   3. 至少 1 个重试幂等证明。
4. 验收说明：
   1. 哪些旧字段被保留兼容；
   2. 哪些页面已不再显示泛化错误提示；
   3. 若仍有暂未统一的链路，必须逐条说明阻塞原因，不得含糊写“后续再补”。

## 9. 交付边界
1. 本轮主目标是完成 `Codex` 失败治理统一合同和 5 条链路适配。
2. 不要借机重构无关模块，也不要把主题发散成“所有错误统一治理”。
3. 若你发现某条链路目前没有真实 `Codex` 调用，请拿代码证据说明，而不是主观排除。
