## 2. 当前已收敛口径
1. `训练中心` 当前有效主命名已升级为 `角色中心（原训练中心）`。
2. `角色中心` 当前至少包含：
   1. `角色资产与版本`
   2. `创建角色`
   3. `训练优化`
3. `训练优化` 当前不是“训练计划表单页”，而是与 `创建角色` 同类的对话式工作流。
4. 用户本轮新增的刚性要求是：
   1. 当前阶段先用“能力列表”作为训练优化的主展示；
   2. 每条能力必须同时展示“能力展示效果 + 能力评分”；
   3. 右侧默认先看 `任务 / 能力演进`，不是先看门禁说明；
   4. 验收门禁要绑定到具体能力项，而不是只给会话级总状态。
5. 当前需求文档已明确新增：
   1. `FR-RO-18 能力对象模型`
   2. `FR-RO-19 能力列表展示规则`
   3. `FR-RO-20 能力展示效果与评分绑定`
   4. `FR-RO-21 历史能力回归结果展示`
   5. `AC-RO-12 ~ AC-RO-15`

## 3. 本轮目标
1. 把 `角色中心 -> 训练优化` 的当前主视图收口为：
   1. 左侧优化会话列表；
   2. 中部单聊天壳；
   3. 中部聊天壳内直接回收“本轮能力列表”；
   4. 右侧默认 `任务 / 能力演进`，可切换到 `当前能力基线`。
2. 把“能力”从零散说明升级为独立对象，让每条能力都可被追踪、评分、验收和回归。
3. 确保 `Gate-B / Gate-C` 能直接落到具体能力项，而不是只显示在会话汇总里。

## 4. 你必须按以下口径实现
1. 训练优化主布局：
   1. 保持单聊天壳，不要回退成多列表拼表单；
   2. 中部聊天壳中必须能直接看到本轮能力列表；
   3. 右侧默认激活 `任务 / 能力演进`；
   4. `当前能力基线` 作为第二标签，不要默认抢首屏。
2. 能力对象模型：
   1. 每个能力对象至少具备：
      1. `capability_id`
      2. `capability_name`
      3. `capability_goal`
      4. `preview_evidence`
      5. `score_current`
      6. `score_target`
      7. `gate_status`
      8. `historical_regression_result`
      9. `impact_scope`
   2. 不要继续把能力结果混在纯文本消息里，导致前端无法稳定渲染。
3. 能力列表展示：
   1. 每条能力项至少展示：
      1. 能力名称
      2. 当前状态
      3. 对应能力展示效果
      4. 能力评分
      5. 关联门禁状态
      6. 对历史能力的影响结论
   2. 能力列表当前应优先出现在中部聊天壳内，而不是只放在右侧。
4. 效果展示与评分绑定：
   1. 每条能力项都必须同时展示“效果证据”和“评分”；
   2. 不允许出现“只有评分没有效果”；
   3. 不允许出现“只有效果没有评分”；
   4. 评分至少应包含：
      1. 基线分或当前正式分
      2. 当前轮分数
      3. 目标分或阈值
      4. 分差结论
5. 门禁绑定规则：
   1. `Gate-B` 必须能直接绑定到具体能力项；
   2. `Gate-C` 必须能直接绑定到触发历史能力退化的能力项；
   3. 若某能力导致历史能力下降：
      1. 该能力项必须标记为风险或阻塞；
      2. 自动发布必须保持阻塞；
      3. 右侧 `任务 / 能力演进` 里必须能看到对应补救任务或回补阶段。
6. 右侧任务 / 能力演进：
   1. 默认展示当前阶段、阻塞项、待追加节点、自动发布状态；
   2. 展示和本轮会话绑定的任务列表；
   3. 展示能力演进图，明确：
      1. 基线
      2. 新能力主线
      3. 历史能力退化分支
      4. 回补后再进入发布
7. 右侧当前能力基线：
   1. 至少展示：
      1. 当前正式版本
      2. 当前角色画像摘要
      3. 历史关键能力清单
      4. 本轮回归验证结果
   2. 不要把基线信息和默认右侧主视图混成一个首屏。

## 5. 推荐实现顺序
1. 先补训练优化页的数据模型和假数据/真实接口契约，让前端能拿到稳定的能力对象数组。
2. 再补中部聊天壳中的能力列表渲染。
3. 再补右侧默认标签切换和能力演进图。
4. 最后把 `Gate-B / Gate-C` 的结果真正绑定到能力项，而不是只停在 UI 假标签。

## 6. 明确禁止
1. 不要把这轮实现扩大成整个训练系统重写。
2. 不要只做静态原型对齐，而没有真实数据结构或接口契约落点。
3. 不要把能力列表塞到右侧，导致中部聊天壳仍然只有普通消息流。
4. 不要继续保留“总门禁通过/失败”但看不到是哪条能力触发的问题。
5. 不要只补新能力展示，不补历史能力回归结果展示。

## 7. 最低验收回传
1. 页面证据：
   1. `角色中心 -> 训练优化` 首屏截图；
   2. 中部聊天壳内出现能力列表的截图；
   3. 至少 1 条能力项同时展示“效果 + 评分”的截图；
   4. 至少 1 条能力项展示 `Gate-B` 或 `Gate-C` 绑定结果的截图；
   5. 右侧默认落在 `任务 / 能力演进` 的截图；
   6. 切换到 `当前能力基线` 的截图。
2. 数据/接口证据：
   1. 训练优化页能力对象的前端数据结构或接口返回；
   2. 至少 1 条能力对象完整字段示例；
   3. `Gate-B / Gate-C` 如何映射到能力对象的代码落点。
3. 行为证据：
   1. 历史能力退化时，自动发布保持阻塞的证据；
   2. 历史能力未退化时，对应能力项展示“未影响”或等效结论的证据。
4. 代码落点：
   1. 训练优化页主布局位置；
   2. 中部能力列表渲染位置；
   3. 能力对象模型定义位置；
   4. 右侧默认标签与演进图位置；
   5. `Gate-B / Gate-C` 绑定到能力项的位置。
