## 3. 本次必须修复的点
1. 报告契约统一
   1. `Codex` 生成的结构化报告必须稳定对齐当前新契约：
      1. `target_version`
      2. `current_workspace_ref`
      3. `previous_release_version`
      4. `first_person_summary`
      5. `full_capability_inventory`
      6. `knowledge_scope`
      7. `agent_skills`
      8. `applicable_scenarios`
      9. `change_summary`
      10. `capability_delta`
      11. `risk_list`
      12. `validation_evidence`
      13. `release_recommendation`
      14. `next_action_suggestion`
   2. `release_recommendation` 只允许：
      1. `approve`
      2. `reject`
      3. `needs_more_validation`
   3. 不允许再输出人工审核决策枚举到该字段。

2. 报告内容焦点修正
   1. 发布评审报告应优先回答：
      1. 当前角色完整能做什么
      2. 相对上一正式发布版本变了什么
      3. 当前是否适合确认发布
   2. 不要把报告写成“纯仓库卫生巡检单”。
   3. 对 `README/CHANGELOG/release note` 缺失：
      1. 默认写入 `warnings`
      2. 除非规则显式要求，否则不要单独把它当成 `reject` 主因
   4. 若无上一正式发布版本：
      1. 按“首发基线评审”处理
      2. 仍必须输出完整角色全量能力
      3. 不得只输出拒绝理由

3. 事实来源与边界修正
   1. 生成 `first_person_summary/full_capability_inventory/knowledge_scope/agent_skills/applicable_scenarios` 时，优先结合：
      1. 当前工作区 `AGENTS.md`
      2. 当前角色画像字段
      3. 本地 skills
      4. 工作区说明文档与版本上下文
   2. Git 风险判断默认只限目标工作区路径范围。
   3. 工作区外或兄弟目录的脏文件，只能作为辅助告警，不得主导 `release_recommendation`。
   4. 若版本元数据自相矛盾（如 `target_version/current_registry_version/released_versions` 冲突），优先返回 `metadata_conflict` 类失败原因，而不是把它伪装成 agent 能力风险。

4. 失败态与前端展示修正
   1. `report_failed` 时，后端必须在 `review.report` 中保留失败报告骨架：
      1. 至少补齐 `target_version/current_workspace_ref/change_summary/release_recommendation/next_action_suggestion`
      2. 保留 `raw_result/warnings`
   2. 前端错误展示优先级改为：
      1. 先显示后端真实 `report_error`
      2. 仅当后端明确返回“结构化报告不完整”时，再显示“缺字段”
   3. 前端缺字段列表必须和后端 `RELEASE_REVIEW_REQUIRED_FIELDS` 保持一致。
