# 发布边界专项用例编号

## 使用说明
- 本文件用于维护 `workflow(pm)` 侧围绕候选刷新、发布边界和 `prod` 升级助手的正式专项用例编号。
- 当发布边界逻辑需要独立编号、独立验收和独立证据引用时，优先在这里登记，而不是继续混在通用 smoke 或历史候选记录里。
- 当前只收录已经落到代码与 acceptance 的编号；纯口头规划不进入本表。

## 用例列表

### `TC-REL-001`
- 绑定需求：`V2-R3` 候选刷新与发布边界助手
- 目标：发布边界默认以本机 `workflow/.repository/<developer_id>` 与 `../workflow_code` 的同步真相为准，不把 `origin/main ahead` 误判成当前 release boundary 阻塞。
- 主要覆盖：
  - `docs/workflow/reports/7x24发布边界收口方案-20260409.md`
  - `.repository/pm-main/src/workflow_app/server/services/release_boundary_service.py`
  - `.repository/pm-main/scripts/acceptance/verify_release_boundary_local_root_sync_policy.py`
  - `.repository/pm-main/scripts/acceptance/verify_release_boundary_tc_rel_001.py`
- 通过标准：
  - 发布边界快照会把 `workspace_head/code_root_head` 与本机同步状态识别为主判断。
  - `origin/main ahead` 只作为上游镜像参考，不会把当前轮误打成 release boundary blocked。
  - `workflow gate` 已能通过 `TC-REL-001` 直接回归这条口径。

### `TC-REL-002`
- 绑定需求：`V2-R3` 候选刷新与发布边界助手
- 目标：`prod` idle watcher 与单次检查助手只在空窗发起 `/api/runtime-upgrade/apply`，并把切版后续交给 supervisor 收口。
- 主要覆盖：
  - `.repository/pm-main/scripts/apply_prod_candidate_when_idle.py`
  - `.repository/pm-main/scripts/workflow_env_common.ps1`
  - `.repository/pm-main/scripts/acceptance/verify_apply_prod_candidate_when_idle.py`
  - `.repository/pm-main/scripts/acceptance/verify_prod_auto_upgrade_single_check_helper.py`
  - `.repository/pm-main/scripts/acceptance/verify_release_boundary_tc_rel_002.py`
- 通过标准：
  - long-poll watcher 只在 `running_task_count=0 && can_upgrade=true` 时发起 apply。
  - single-check helper 命中空窗时会发起 apply，未命中时只记录跳过，不会误等待或重复请求。
  - `workflow gate` 已能通过 `TC-REL-002` 直接回归 watcher 与 helper 的一致语义。

### `TC-REL-003`
- 绑定需求：`V2-R3` 候选刷新与发布边界助手
- 目标：candidate drain 与 pending upgrade fallback 能冻结错误派发，并在恢复窗口里继续回到一致的发布边界语义。
- 主要覆盖：
  - `.repository/pm-main/src/workflow_app/server/services/runtime_upgrade_service.py`
  - `.repository/pm-main/scripts/acceptance/verify_runtime_upgrade_dispatch_drain.py`
  - `.repository/pm-main/scripts/acceptance/verify_prod_watchdog_pending_upgrade_fallback.py`
  - `.repository/pm-main/scripts/acceptance/verify_release_boundary_tc_rel_003.py`
- 通过标准：
  - 当存在更高 `prod candidate` 时，新派发会命中 `upgrade_drain_active`，而不是继续错误放量。
  - idle watcher/pending request 恢复后，可重新识别 `request_ready / request_candidate_mismatch / candidate_not_newer` 等语义，不遗留卡死态。
  - `workflow gate` 已能通过 `TC-REL-003` 直接回归 drain 与 fallback 的恢复链。
