# 持续唤醒专项用例编号

## 使用说明
- 本文件用于维护 `workflow` 在 `自迭代 / 持续唤醒` 方向的正式专项用例编号。
- 当前重点是把 `TC-AWAKE-*` 从 `workflow_testmate` 的项目级目录，收口到 PM 工作区可引用、可回归、可挂版本矩阵的正式文档。
- 仅收录已经存在稳定 acceptance / probe / live evidence 的编号；纯口头规划不进入本表。

## 用例列表

### `TC-AWAKE-001`
- 绑定需求：`V2-R4` 7x24 主线治理动作轻量自动化
- 联动覆盖：`V2-R7` 需求点 -> 用例 -> 验收证据映射矩阵
- 目标：`[持续迭代] workflow` 命中后能自动建出 `workflow` 节点，并把 `source_schedule_id / trigger_instance_id` 留成可追溯真相。
- 主要覆盖：
  - `.repository/pm-main/scripts/acceptance/run_acceptance_assignment_self_iteration_schedule.py`
  - `.repository/pm-main/scripts/acceptance/verify_assignment_self_iteration_schedule_alignment.py`
  - `.repository/pm-main/scripts/acceptance/verify_pm_awake_tc_pack.py`
- 通过标准：
  - 自迭代命中后会自动生成 `workflow` 节点，而不是只停在 schedule 命中或人工补链。
  - 主线 schedule 的优先级、触发时间和 `trigger -> node` 关联保持一致，且不再自动生成巡检 schedule。
  - PM 侧 gate 已能直接引用这条编号，而不是只靠 helper 口头挂接。

### `TC-AWAKE-002`
- 绑定需求：`V2-R4` 7x24 主线治理动作轻量自动化
- 联动覆盖：`V2-R7` 需求点 -> 用例 -> 验收证据映射矩阵
- 目标：自迭代计划命中后，节点能够自动派发起跑，并在结束后续挂下一轮 mainline 出口。
- 主要覆盖：
  - `.repository/pm-main/scripts/acceptance/run_acceptance_assignment_self_iteration_schedule.py`
  - `.repository/pm-main/scripts/acceptance/verify_assignment_self_iteration_plan_reference.py`
  - `.repository/pm-main/scripts/acceptance/verify_schedule_prompt_contract_repair.py`
  - `.repository/pm-main/scripts/acceptance/verify_pm_awake_tc_pack.py`
- 通过标准：
  - 节点在 schedule -> assignment 后能够进入真实执行，不会长期停在“已建单未起跑”的灰区。
  - 自迭代 prompt 读链、版本读链和升级边界要求保持在最新治理口径内。
  - 即使 schedule row 里仍保留旧文案，读链也会把 `version_transition_decision` 与 `next_activation_candidate / next_activation_ready` 的合同补回当前模板。
  - 任务结束后会自动续挂新的 mainline，而不是把 7x24 连续性留给人工补洞。

### `TC-AWAKE-003`
- 绑定需求：`V2-R4` 7x24 主线治理动作轻量自动化
- 联动覆盖：`V2-R7` 需求点 -> 用例 -> 验收证据映射矩阵
- 目标：当 smoke / guard 未满足时，系统会 fail-closed 并直接重排 mainline，不会误把高风险状态继续放量。
- 主要覆盖：
  - `.repository/pm-main/scripts/acceptance/verify_self_iteration_mainline_retry_on_smoke_block.py`
  - `.repository/pm-main/scripts/acceptance/verify_pm_awake_tc_pack.py`
- 通过标准：
  - smoke guard 命中阻塞时，自迭代主线不会继续无约束放量。
  - 原 `[持续迭代] workflow` schedule 会被重新启用并写入未来 retry，且不会创建 `pm持续唤醒 - workflow 主线巡检`。
  - PM 侧已经有正式编号与 gate 引用，不再只有 testmate 目录里的项目级描述。

### `TC-AWAKE-004`
- 绑定需求：`V2-R4` 7x24 主线治理动作轻量自动化
- 联动覆盖：`V2-R7` 需求点 -> 用例 -> 验收证据映射矩阵
- 目标：24h 连续运行目标在任务中心、工作面和 dashboard 中都可见，并优先展示 mainline 的 live 真相。
- 主要覆盖：
  - `.repository/pm-main/scripts/acceptance/verify_assignment_mainline_visibility.py`
  - `.repository/pm-main/scripts/acceptance/verify_assignment_center_mainline_visibility.js`
  - `.repository/pm-main/scripts/acceptance/verify_dashboard_schedule_preview.py`
  - `.repository/pm-main/scripts/acceptance/verify_pm_awake_tc_pack.py`
- 通过标准：
  - backend 默认节点选择会优先回到当前真实 `running` 节点；若 workflow 当前没有 running，也必须保留下一棒 `workflow` mainline 的 handoff 真相，而不是把 ready 主线误当 live run。
  - 前端任务中心能稳定展示“当前运行 + 主线接棒 + handoff 提示”，不把 7x24 主线淹没在杂讯里。
  - dashboard/schedule preview 对 mainline 的状态展示保持紧凑且不泄漏冗长治理正文。
