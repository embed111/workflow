# 详细设计-统一入口与训练优化模块

## 1. 设计目标
1. 建立一个统一入口，先展示 agent 资产与版本事实，再进入训练闭环动作。
2. 保持“展示层”和“训练状态机层”解耦，避免单页混合导致维护复杂度上升。
3. 保障手动主路径可落地，同时预留分析模块自动计划并行接入能力。

## 2. 设计范围与非范围
1. 设计范围：
   1. 入口信息架构与页面交互。
   2. 训练计划入队、优先级调度、任务移除、执行回写。
   3. 发布版本视图只读集成（仅已发布迭代）。
2. 非范围：
   1. 训练算法细节与模型参数调优。
   2. Git 写操作（打 tag/回滚）实现。
   3. 外部训练工作区协同与复杂匹配策略。

## 3. 信息架构（IA）
1. 一级菜单：`训练中心`。
2. 二级 Tab：
   1. `Agent资产与版本`
   2. `训练闭环`
3. 默认进入顺序：
   1. 进入训练中心默认落在 `Agent资产与版本`。
   2. 选择 agent 后，才允许进入 `训练闭环`。

## 4. 页面详细设计
### 4.1 Agent资产与版本
1. 顶部统计卡：
   1. `agent_total`
   2. `git_available_count`
   3. `latest_release_at`
   4. `training_queue_pending`
2. 左侧角色卡墙（可搜索）：
   1. `vector_icon`（头像式矢量人物图，非抽象几何图标）
   2. `agent_name`
   3. `current_version`
   4. `core_capabilities`（摘要）
   5. `status_tags`（如 `git_unavailable`）
   6. `identity_style`（视觉风格：名片/工牌）
3. 右侧详情区：
   1. 基础信息：路径、版本、最近发布说明。
   2. 已发布迭代时间线：按时间倒序，支持分页；发布判定固定为 `tag + release note`。
   3. 功能迭代变更：每个发布版本对应 `change_summary`。
   4. 不展示中间 Git 提交列表（commit 明细）。
   5. 操作按钮：`进入训练优化`。

### 4.2 训练闭环
1. 计划创建区（手动）：
   1. 必填：目标 agent、能力目标、训练任务清单、验收标准、优先级（P0-P3）。
   2. 提交后生成队列任务。
2. 队列列表区：
   1. 字段：`queue_task_id/source/priority/target_agent/status/execution_engine/similar_flag/enqueued_at`。
   2. 操作：`移除任务`（无二次确认，显示风险提示）。
3. 训练执行触发区：
   1. 执行主体固定为 `workflow` 内建训练能力。
   2. 页面不提供训练师搜索、匹配或选择区。
   3. 队列任务可直接触发执行，并在执行中显示状态与结果回写。
4. 执行结果区：
   1. `run_ref`
   2. `status`
   3. `result_summary`
   4. `updated_at`

## 6. 状态机设计
1. `training_queue.status`：
   1. `queued -> running -> done|failed`
   2. `queued|running -> removed`（人工移除）
2. 人工移除规则：
   1. 允许直接移除，不弹二次确认。
   2. UI 先显示风险提示，再执行移除动作。
   3. 审计日志必写。
3. 相似计划处理：
   1. 仅写 `similar_flag=true`。
   2. 不改变状态机流转，不自动阻断。

## 7. 调度与并行策略
1. 队列排序键：
   1. `priority`（P0 > P1 > P2 > P3）
   2. `enqueued_at`（同优先级 FIFO）
2. 来源并行：
   1. `manual` 与 `auto_analysis` 共享同一训练队列。
   2. 不按来源分队列，避免调度分裂。
3. 调度冲突处理：
   1. 同一 agent 可存在多个任务，默认允许顺序执行。
   2. 相似任务由人工判断是否移除。

## 8. 接口草案（需求级）
1. `GET /api/training/agents`
   1. 返回 agent 角色卡列表（含头像式矢量图标识字段）。
2. `GET /api/training/agents/{agent_id}/releases`
   1. 返回已发布迭代时间线（不返回中间 commit 明细）。
3. `POST /api/training/plans/manual`
   1. 创建手动训练计划并入队。
4. `GET /api/training/queue`
   1. 返回训练队列。
5. `POST /api/training/queue/{queue_task_id}/remove`
   1. 人工移除任务（无二次确认）。
6. `POST /api/training/queue/{queue_task_id}/execute`
   1. 启动训练执行。
7. `GET /api/training/runs/{run_id}`
   1. 查询执行结果。

## 9. 安全、审计与风险控制
1. 安全边界：
   1. Git 读取路径限制在 agent 根目录作用域内。
2. 审计要求：
   1. 所有写动作必须审计：创建计划、入队、移除、执行、回写。
3. 风险提示：
   1. 移除任务前显示风险提示文本（非阻断）。
   2. 相似计划标注展示“可能重复训练”提示。

## 10. 迭代计划建议
1. Phase A（当前）：
   1. 统一入口 + Agent 角色卡概览/详情 + 已发布迭代视图 + 手动计划 + 队列 + `workflow` 内建训练执行。
2. Phase B（下一步）：
   1. 分析模块自动计划接入（`auto_analysis`）。
   2. 自动计划人工审核门禁（若启用）。
3. Phase C（增强）：
   1. 训练执行引擎的资源隔离、限流与失败恢复治理。
   2. Git 写操作能力（受控权限）。

## 11. 与需求映射关系
1. `FR-UO-01~FR-UO-04` -> 第 3~4 节（入口与版本层）。
2. `FR-UO-05~FR-UO-11` -> 第 4~7 节（训练闭环层）。
3. `FR-UO-12~FR-UO-13` -> 第 8~9 节（回写、审计、门禁）。

## 14. 本轮增量设计（2026-03-11）- 训练闭环页的单窗口客户端式布局
1. 设计目标补充：
   1. 保持 `训练中心` 的单窗口客户端视图，不把训练闭环拆成多个分散页面容器。
   2. 在同一主窗口内承载 `创建任务` 与 `任务状态` 两种工作模式，并避免两类信息混在同一可视层级。
   3. 长内容优先通过区内滚动解决，而不是让主页面无限变长。
2. 页面结构补充：
   1. 主窗口标题区沿用当前 `训练中心` 页面口径：
      1. 左上显示 `训练中心` 标题。
      2. 右上显示“训练对象是 agent 工作区能力，独立于会话流程”说明。
      3. 标题位于主窗口内部，不单独悬浮在窗口外。
   2. 训练中心第二子模块统一命名为 `训练闭环`，并以闭环主视图承载训练流程。
   3. 窗口顶部新增 `工作模式` 切换条：`创建任务`、`任务状态`。
   4. 结构分为：
      1. 窗口顶部：模块切换、角色说明、风险提示、摘要条、工作模式切换条。
      2. 左侧常驻：`训练任务列表 + 创建任务入口`。
      3. `任务状态` 模式：
         1. 中部顶部：`当前任务总览`，明确当前任务、当前轮次、当前结论与下一步动作。
         2. 中部下方：`总览 / 工作集 / 评测结果 / 迭代记录` 二级标签，默认落在 `总览`。
         3. 右侧：`当前判定 / 关键分数 / 下一步动作`。
      3. `创建任务` 模式：
         1. 中部主区集中展示计划定义、阈值、初始用例、初始参考资料 / Skill 与“启动首轮”。
         2. 右侧展示“创建后将生成的任务摘要 / 校验结果 / 首轮执行方式”。
         3. 不把评分结果、迭代时间线作为主内容同屏展开。
3. 滚动策略补充：
   1. `参考资料与 Skill 工作集` 为独立滚动区，固定在主窗口右上区域。
   2. `训练用例与三轮评测` 在用例数量过多时使用区内滚动。
   3. `迭代记录时间线` 在轮次过多时使用区内滚动。
   4. 主窗口本身在桌面基线下不应依赖整页长滚动承载核心内容。
4. 交互语义补充：
   1. `创建任务` 与 `任务状态` 通过同一主窗口内的模式切换完成，不做整页跳转。
   2. 有活跃任务时默认落在 `任务状态`。
   3. “创建并启动首轮”成为 `创建任务` 模式主动作。
   4. 左侧任务列表支持直接切换任务，不要求返回上级列表页。
   5. `任务状态` 模式默认进入 `总览` 子标签，先解决“我现在在看什么、结果如何、下一步干什么”。
   6. “执行三轮评测”替代原“执行队列任务”成为 `任务状态` 模式主动作。
   7. “回退本轮新增参考资料与 Skill”替代原“移除任务”成为失败态主动作。
   8. “进入下一轮”替代原“调度下一个”成为提升未达标时的主动作。
5. 视觉风格补充：
   1. 继续沿用现有 `workflow` 的左侧导航、浅色面板、轻边框、局部滚动与业务卡片密度。
   2. 原型应给人“模拟客户端截图”而不是“说明海报”或“多页拼贴图”的感觉。
6. 推断/假设（置信度: 中）：
   1. 桌面单窗口验收基线暂按 `1440x900` 视图设计。
   2. 如后续用户指定更低高度，应优先压缩区块密度与内部滚动区高度，而不是放弃单窗口布局原则。

## 15. 本轮增量设计（2026-03-11）- 任务状态页的优先级展开与默认折叠
1. 设计目标：
   1. 保留状态页的信息完整性，不用“删信息”来换整洁。
   2. 通过展示优先级控制首屏注意力，把“当前任务 / 当前结论 / 下一步动作”固定为主阅读线。
   3. 让低优先级信息保留在同页，但默认折叠，避免与一级判断信息争抢视觉层级。
2. 布局原则：
   1. 一级信息常显：
      1. 中部顶部固定展示 `任务名称 / 目标能力 / 当前轮次`。
      2. 右侧固定展示 `当前结论 / 下一步动作 / 核心原因`。
      3. 一级信息在首屏不依赖任何展开动作。
   2. 二级信息默认展开：
      1. `本轮评分摘要`
      2. `本轮工作集变化`
      3. `当前判定原因`
      4. 二级信息优先通过 `标签 / 等效切换区` 承载，默认激活最关键的一项。
      5. 不强制三块二级信息在同一时刻同时完整展开，否则会反向压缩主阅读区。
   3. 三级信息默认折叠：
      1. `Run1 / Run2 / Run3` 完整明细
      2. `候选资料与未采纳原因`
      3. `完整迭代记录`
      4. `阅读说明/辅助提示`
      5. 以上内容统一采用可展开面板承载，默认不占据首屏主视区。
3. 去重规则：
   1. `当前结论` 与 `下一步动作` 只允许在右侧 `当前判定` 区作为主表达。
   2. 中部 `当前任务总览` 只负责回答“当前看的是谁、现在到哪一轮”，不再重复展开判定结论全文。
   3. 原先用于解释页面阅读方式的说明文案降级到三级折叠信息，不再占用首屏高优先级位置。
4. 交互语义：
   1. 二级和三级内容统一使用折叠面板语义，但二级默认 `open`、三级默认 `closed`。
   2. 用户展开三级信息后，原有信息不跳页、不跳转，仍停留在同一状态页上下文中。
   3. 若后续增加新的状态字段，默认先判定其展示优先级，再决定落入一级、二级还是三级区域。
   4. 若二级信息项过多，优先转为“二级标签切换 + 三级折叠明细”的结构，而不是继续纵向堆叠多个默认展开块。

## 16. 本轮增量设计（2026-03-11）- 任务状态页的视口内收纳
1. 设计目标：
   1. 在保持当前信息分层结构不变的前提下，把 `任务状态` 页压回到固定桌面视口内。
   2. 不通过删掉核心信息达成“看起来不超屏”，而是通过压缩冗余占高与区内滚动完成收纳。
2. 视口约束：
   1. 主窗口继续以 `100vh` 为外层基线。
   2. 顶部区域仅保留必要的标题、模式切换与风险提示，不重复放置大块说明性组件。
   3. `workbench` 区域占用剩余高度，并作为三栏主工作区的唯一高度容器。
3. 区域收纳策略：
   1. 左侧：
      1. 任务列表保持独立滚动。
      2. 任务卡只保留必要摘要，不继续叠加高占位说明。
   2. 中部：
      1. 顶部 `当前任务总览` 保留，但高度受控，不再重复承载判定全文。
      2. 详情区使用纵向折叠栈；超出视口时仅详情区内部滚动。
   3. 右侧：
      1. 仅保留 `当前结论 / 下一步动作` 两组核心卡片。
      2. 右侧辅助说明卡降级或移除，避免把右栏撑高到超出视口。
4. 压缩规则：
   1. 重复表达优先删除，而不是继续缩字体硬塞。
   2. 展示优先级说明不再作为独立大区块常驻顶部。
   3. 低优先级辅助文案应并入折叠区或页脚短说明，不得继续占用首屏主体高度。

## 17. 本轮增量设计（2026-03-11）- 显示框底边对齐
1. 设计目标：
   1. 在固定视口内，不仅控制页面不超屏，还要保证同层级显示框底边对齐。
   2. 让页面在视觉上呈现稳定的主工作区边界，避免出现左中右三栏或同排卡片高低错落。
2. 对齐规则：
   1. 三栏主工作区中的左侧主面板、中部主面板、右侧主面板共享同一可用高度基线。
   2. 同一行并排的卡片组件采用统一高度或拉伸对齐策略，保证底边一致。
   3. 某一列内容较多时，不允许直接把该列整体撑高破坏底边对齐。
3. 收纳方式：
   1. 高度差优先通过区内滚动解决。
   2. 其次通过默认折叠和摘要化降低展开高度。
   3. 不通过增加外层页面高度来换取内容完整展示。
4. 实现提示（需求级设计）：
   1. 三栏容器应采用同一行拉伸对齐语义。
   2. 面板内部内容区采用 `min-height: 0` 与内部滚动策略，避免内容把外层面板顶出对齐基线。
   3. 滚动条应保持在显示框内部，形成“固定窗口 + 局部滚动区”的客户端式交互感。
   4. 优先让左侧列表、中部详情栈、右侧判定列成为独立滚动容器，而不是让整个页面滚动。

## 18. 本轮增量设计（2026-03-11）- 底部防裁切与显式滚动条
1. 设计目标：
   1. 消除“内容在显示框底部被裁切，但页面没有给出明确滚动通道”的状态。
   2. 让滚动条成为当前固定窗口原型中的正式收纳结构，而不是偶发出现的浏览器默认行为。
2. 结构约束：
   1. `app -> workspace -> window -> workbench -> column-shell/panel` 整条容器链都应具备 `min-height: 0` 或等效可收缩语义。
   2. 左侧任务列、中部详情列、右侧判定列都应保持 `overflow: hidden` 的外层边界，并把滚动集中到列内专用容器。
   3. 页脚说明区与主操作区属于固定窗口内容的一部分，不得被上方溢出内容挤出可见范围。
3. 滚动条策略：
   1. `任务列表`、`详情折叠栈`、`创建表单主体`、`右侧校验/判定列` 使用各自独立的纵向滚动容器。
   2. 滚动容器优先采用稳定滚动条轨道或等效占位，避免因“刚好溢出时才出现”导致视觉误判为被裁切。
   3. 不允许把应进入区内滚动的内容继续放在外层 `window` 或页面根节点上滚动。
4. 原型呈现提示：
   1. 若桌面浏览器真实可用视口低于名义分辨率，应优先保证滚动条可见，而不是依赖继续压缩字体和间距硬塞。
   2. `任务状态` 与 `创建任务` 两张参考图应使用同一套固定窗口 + 区内滚动条约束，避免目录内原型口径分叉。

## 19. 本轮增量设计（2026-03-11）- 把空间优先还给任务状态详情
1. 设计目标：
   1. 在保留固定窗口、底边平齐和区内滚动的前提下，恢复 `任务状态详情` 的自然阅读密度。
   2. 避免为满足视口约束而继续压缩详情区本身，导致信息虽然“装下了”，但阅读体验变差。
2. 调整原则：
   1. 顶部 `当前任务总览` 只保留任务身份与关键状态，不再承载重复动作按钮。
   2. `进入下一轮`、回退等主动作继续由右侧 `当前判定 / 下一步动作` 区主承载，不在中部顶部重复出现。
   3. 中部 `任务状态详情` 的折叠面板、卡片与表格恢复到常规阅读间距，不再继续压缩内边距来换高度。
   4. 二级默认展开面板只承载“当前轮摘要”，完整评分表和长清单优先下沉到三级折叠面板，避免摘要区反向膨胀。
   5. 当二级摘要项并存会明显压缩详情区时，二级层改为“单主摘要区 + 标签切换”，保持中心阅读区稳定。
   6. `任务状态详情` 区本身使用独立纵向滚动容器，摘要区、三级折叠区和后续详情内容统一在该滚动容器内流动，不再靠外层压缩收纳。
3. 空间重分配策略：
   1. 优先压缩：
      1. 顶部重复摘要与重复动作；
      2. 说明性文案占高；
      3. 左右侧非主阅读区的横向占位。
   2. 优先保留：
      1. 中部详情区纵向可视面积；
      2. 详情卡片内部留白；
      3. 评分摘要与工作集变化的正常表格/卡片节奏。
4. 原型呈现提示：
   1. 若仍需进一步收纳，优先缩减左侧与右侧列宽，把横向空间让给中部详情区。
   2. `任务状态详情` 标题说明应收成短句，不再占用过高的标题区。
   3. 中部详情区的滚动条应直接挂在 `任务状态详情` 容器上，而不是仅挂在其子块上，避免“摘要区固定、下半区单独挤压”的视觉错觉。

## 20. 本轮增量设计（2026-03-11）- 任务状态详情采用外框化滚动区
1. 设计目标：
   1. 在保留独立滚动的前提下，进一步明确 `任务状态详情` 的边界，让用户直观看出“这是一个固定框内滚动的详情区”。
   2. 消除“内容像是被压进面板里”与“滚动归属不清”之间的视觉歧义。
2. 结构口径：
   1. 中部 `任务状态详情` 标题保持在详情框外，负责定义语义分区。

   2. 三级折叠区不再额外拆出第二套主滚动区，避免“框内还有框内滚动”的阅读负担。
   3. 当详情内容继续增长时，优先延续“外框内单滚动通道”原则，不回退为外层整页滚动或局部内容压缩。

## 21. 本轮增量设计（2026-03-11）- 训练工作流采用阶段流展示
1. 设计目标：
   1. 让训练闭环从“结果页集合”收口为“流程中的当前一步”，首屏先表达阶段进度，再表达当前阶段细节。
   2. 让 `创建任务` 与 `任务状态` 共用同一套训练流程骨架，降低模式切换时的语义跳变。
2. 阶段定义：
   1. 统一采用五阶段语义：
      1. `创建任务`
      2. `配置本轮工作集`
      3. `执行三轮评测`
      4. `生成结果判定`
      5. `进入下一轮 / 完成训练`
   2. `创建任务` 视图默认激活阶段 1。
   3. `任务状态` 视图按当前轮次进展激活阶段 3 或 4；当已产出下一步动作但未进入新一轮时，优先视为阶段 4。
3. 结构口径：
   1. 中部顶部主卡改为 `流程总览卡`，内部包含：
      1. 当前任务/计划身份；
      2. 当前阶段状态条；
      3. 横向阶段流；
      4. 少量关键指标摘要。
   2. 原先分散在标题、副标题和摘要卡中的“当前轮次/当前阶段/下一步”优先并入流程总览卡，减少顶部碎片信息。
   3. `任务状态详情` 继续承载当前阶段细节与低优先历史明细，但其阅读前置由“先看详情”改为“先看流程总览卡，再看详情”。
4. 视觉与交互口径：
   1. 阶段流中的已完成、当前、待进入阶段应有明确视觉差异，但保持现有浅色工作台风格。
   2. 阶段卡不使用大面积装饰性配图，重点保留步骤编号、阶段名称与极短状态提示。
   3. 当前阶段状态条需直接回答：
      1. 当前在做什么；
      2. 当前轮次/阶段上下文；
      3. 下一步是什么。
   4. 若视口收纳压力上升，优先压缩阶段卡描述文案，而不是取消阶段流本身。

## 22. 本轮增量设计（2026-03-11）- 训练演进图表达分叉与收敛
1. 设计目标：
   1. 在线性阶段流之外，再补一层“路径演进视角”，让用户看清训练并非单线推进，而是会经历分叉、回退与主线保留。
   2. 解决“为什么这一轮要回退”“为什么分数提升了但还要继续下一轮”这类无法仅靠阶段流回答的问题。
2. 结构口径：
   1. 中部顶部的 `流程总览卡` 在当前阶段状态条下新增 `训练演进图`。
   2. `阶段流` 负责回答“当前在闭环哪一步”；
   3. `训练演进图` 负责回答“当前沿哪条训练路径走到这里”。
   4. 演进图优先采用从左到右的图结构，接近 `git graph` 的阅读心智：
      1. 起点在左；
      2. 分叉向上或向下展开；
      3. 当前保留主线继续向右推进；
      4. 未来收敛目标位于最右侧。
3. 节点设计：
   1. 节点类型至少包括：
      1. 基线节点；
      2. 劣化分支节点；
      3. 回退节点；
      4. 保留继续节点；
      5. 当前活动节点；
      6. 未来收敛节点。
   2. 节点文案以“轮次/得分/动作/原因短句”为上限，不把完整解释塞进图里。
   3. 劣化分支使用失败或回退语义颜色，保留主线使用当前主题主色，未来收敛用弱化虚线或弱化节点表达。
4. 交互与收纳口径：
   1. 演进图本身保持首屏可读，但不展开所有历史长说明。
   2. 详细原因继续落在下方 `任务状态详情` 中，通过摘要卡或折叠区承接。
   3. 若轮次进一步增多，优先让演进图展示“关键分叉节点 + 当前主线”，更早历史通过折叠或横向缩略下钻，不把图无限拉长。

## 24. 本轮增量设计（2026-03-12）- 演进图数据对接与关键动作落库（最小可用）
1. 设计目标：

2. 推荐接口契约（实现可调整，但必须满足 FR-UO-32/33/34 的信息要求）：
   1. 获取演进图（建议）：
      1. `GET /api/training/queue/{queue_task_id}/loop`
      2. 返回字段建议包含：
         1. `loop_id`
         2. `queue_task_id`
         3. `target_agent_id/agent_name/capability_goal`

         5. `nodes[]`（至少含 `node_id, title, round_index/round_label, node_type, decision, next_action, impact, metrics`）

         8. `is_test_data`
   2. 动作接口（建议）：
      1. `POST /api/training/queue/{queue_task_id}/loop/enter-next-round`
      2. `POST /api/training/queue/{queue_task_id}/loop/rollback-round-increment`
      3. 请求体建议包含：`operator`、`reason`（可选）。
      4. 返回字段建议包含：`ok`、`action_id/audit_id`、`loop_id`、`current_node_id`、以及可选的 `created_queue_task_id`（若进入下一轮会新建队列任务）。
3. 数据落库建议（两种实现任选其一，优先选择更易验收与追溯的方案）：
   1. 方案 A（推荐，显式图结构，便于渲染与审计）：
      1. 新增表 `training_loop`：描述一个训练闭环（按 agent+目标聚合），记录 `loop_id`、`target_agent_id`、`capability_goal`、`created_at/updated_at`、`is_test_data`。
      2. 新增表 `training_loop_node`：每个演进节点一行，至少包含：
         1. `node_id`、`loop_id`、`parent_node_id`

         3. `round_index`、`branch_key`
         4. 关联对象：`plan_id`、`queue_task_id`、`run_id`（按可用性写入）
         5. 指标：`metrics_json`（可为空）+ `metrics_available`（或等效字段）
         6. 展示字段：`title/decision/next_action/impact`
         7. `status`（active/rolled_back/archived）
         8. `created_at/updated_at`
      3. `training_audit_log` 继续记录关键动作（enter-next-round / rollback-round-increment），以便统一审计入口。
   2. 方案 B（最小改动，图状态存 JSON）：
      1. 新增表 `training_loop_state`，用 `loop_id` 作为主键，存储 `graph_json/current_node_id/updated_at/is_test_data`。
      2. 每次动作更新 `graph_json`，同时在 `training_audit_log` 写入动作记录，确保可追溯。
4. 前端对接口径（与现有 UI 约束保持一致）：
   1. 前端不得再以内置节点目录作为默认数据源；演进图与右侧面板以 `GET /loop` 返回为准。
   2. 指标不可用时：UI 显示 `暂无评分/暂无阈值`（或等效空态），禁止展示固定示例数值。
   3. 动作按钮点击后：触发对应 `POST`，成功后刷新演进图与右侧节点面板；失败需提示明确错误码/原因，并保持幂等语义。

   3. `detail-view-head` 与 `form-view-head` 不承担滚动，不显示用于承载正文的滚动条。
   4. 页签栏在桌面基线视口下应完整展示；若后续页签数量增长，优先调整分组或合并低频项，不先给页签栏加滚动。
3. 默认页签与内容分层：
   1. `任务状态` 默认激活 `当前概览`。
   2. `创建任务` 默认激活 `基础信息`。
   3. 其他页签内容通过切换进入，不在首屏正文中同层完整展开。
4. 窄桌面降级策略：
   1. 当桌面有效宽度接近 `1280px`（例如 `1920x1080` 且系统缩放 `150%`）时，`创建任务` 的 `基础信息 / 首轮工作集 / 启动确认` 切换为三列等宽紧凑页签。
   2. 同类规则适用于 `任务状态` 的 `当前概览 / 工作集变化 / 三轮评测 / 历史记录`，切换为四列等宽紧凑页签。
   3. 紧凑态优先压缩页签内边距、间距与字号，不压缩到不可辨认，也不引入页签滚动条。
   4. 若未来页签数量继续增长，优先重构为 `当前页签 + 更多` 或等效分组，不依赖“继续挤压”处理。
   5. 页签上方的五阶段步骤条在窄桌面下不得继续维持高占用卡片形态，应同步收口为紧凑步骤条：保留阶段顺序与当前态，减少描述文案与卡片高度，优先把首屏高度让给主页签与正文内容。
   6. 若紧凑页签与紧凑步骤条后，中部仍无法在首屏完整容纳主页签与正文入口，则中间主栏升级为独立纵向滚动容器：
      1. `创建任务`：`form-pane` 作为主滚动容器，`form-scroll` 在该场景下降级为普通内容流；
      2. `任务状态`：中部 `column-shell` 作为主滚动容器，`detail-scroll` 在该场景下降级为普通内容流。
   7. 该 fallback 只在窄桌面/高缩放场景下启用；常规桌面仍优先采用“页签固定 + 下方内容区滚动”的主结构。
5. 文案边界口径：
   1. 页面正文不得常驻展示“页签固定在顶部、本身不滚动、只展开一个主标签、不再同屏摊开”等布局说明。
   2. 页面文案应只表达业务事实、阶段状态、训练结果、下一步动作与可查看的信息范围。
   3. 布局规则统一保留在需求文档、设计文档和实现注释中，避免页面被误读为产品正式说明。

## 27. 本轮增量设计（2026-03-13）- 左侧任务列表收口为唯一主入口
1. 设计目标：
   1. 消除“左侧任务列表 + 顶部模式切换 + 右侧重复校验”三套并行控制。
   2. 让训练闭环形成稳定入口心智：左侧选对象，中部看当前对象主阅读区，右侧仅在状态态承载当前节点判定。
2. 结构口径：
   1. 左侧任务列表在创建页与状态页统一采用同一骨架：
      1. `训练任务` 标题；
      2. 搜索框；
      3. 筛选按钮；
      4. 状态筛选行；
      5. `+ 新建训练任务`；
      6. 任务卡列表。
   2. `+ 新建训练任务` 作为进入创建态的唯一主入口，置于左侧任务列表顶部。
   3. 选择某个已有任务作为进入对应任务状态页的唯一主入口。
   4. 页面顶部保留模块级 `训练闭环` 选中态，但移除 `创建任务 / 任务状态` 主模式切换，避免与左侧列表重复。
3. 创建态与状态态的差异控制：
   1. 创建态：
      1. 左侧列表中 `+ 新建训练任务` 高亮；
      2. 中部展示创建表单与提交区；
      3. 不再保留右侧 `创建结果与校验` 独立栏。
   2. 状态态：
      1. 左侧列表中当前任务卡高亮；
      2. 中部展示流程卡、演进图与状态详情；
      3. 右侧保留 `当前选中节点 / 节点判定 / 下一步动作` 面板。
4. 创建页提交口径：
   1. “进入创建态”与“真正提交创建”必须分层：
      1. 左侧 `+ 新建训练任务` 表示进入创建态；
      2. 中部底部 `保存并启动首轮 / 保存草稿` 表示提交动作。
   2. 创建校验摘要并入中部底部提交区，不再单独开第三列重复展示。

## 28. 本轮增量设计（2026-03-13）- 最宽中栏作为统一主滚动容器
1. 设计目标：
   1. 解决“中栏信息过多无法首屏展示完整”与“虽然有滚动条但内容仍被压缩/裁切”的问题。
   2. 将最宽主阅读列收口为稳定的单滚动通道，减少多层纵向滚动抢占滚轮。
2. 滚动责任划分：
   1. 中栏默认承担主阅读滚动：
      1. 创建页：中部主表单列；
      2. 状态页：中部流程卡 + 演进图 + 状态详情列。
   2. 左侧任务列表与右侧节点判定列继续保留各自独立滚动。
   3. 中栏内部的内容子块原则上不再承担第二套主纵向滚动，除非后续明确存在长表格/代码块等局部阅读例外。
3. 布局实现口径：
   1. 中栏滚动容器需显式开启纵向滚动，并保留底部留白，避免最后一块内容贴底被遮挡。
   2. 中栏直系子块（如流程卡、详情外框、提交区）不得允许 `flex-shrink` 导致高度被压扁；应完整进入滚动流。
   3. 若某内层容器已经降级为普通内容流，应移除其自身纵向滚动与裁切语义，避免出现“外层可滚、内层仍裁切”的假滚动现象。
4. 验收要点：
   1. 用户看到滚动条时，必须能通过中栏滚动查看到完整底部内容。
   2. 不允许出现“滚动条已存在，但底部模块仍被压缩、裁切或只露出半截”的视觉 bug。
