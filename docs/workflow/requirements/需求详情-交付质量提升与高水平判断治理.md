## 1. 主题目标与价值说明
1. 问题定义（用户明示 + 当前治理复盘）：
   1. 当前交付仍经常出现“半成品交付”。
   2. 这类问题不能继续只归因于“某一轮没做好”，而要拆成三个稳定问题：
      1. 单个 agent 专业性不足，仍偏向凭感觉输出。
      2. 工程化不足，交付质量还没有被流程、门禁和试错闭环稳定托住。
      3. 对“什么叫高质量”的判断仍不准确，验收与放行还带明显感觉化。
2. 高水平判断工作定义（用户认可口径）：
   1. `高水平判断 = 大量样本 + 反复比较 + 真实后果 + 高手校准 + 原则抽象 + 价值排序`
   2. 后续 agent 的专业化修炼、工程化改造和质量判断体系，都默认围绕这 6 个要素展开。
3. 主题目标：
   1. 让“高质量交付”从感觉判断，升级为有明确问题拆分、阶段排期和验收标准的正式治理能力。
   2. 让专业化、工程化和质量判断不再各自散落，而是被同一份需求与版本切片统一编排。
   3. 让 `7x24` 不只推进功能，也持续反思工程化和交付质量本身的不足，并通过小步试错持续改进。
4. 业务价值：
   1. 降低半成品交付率。
   2. 提高首版产物可用度，减少 PM 或用户回收返工。
   3. 让小伙伴真正沿各自专业方向成长，而不是继续停在“会做一点，但不够稳”的中间态。

## 2. 问题拆解
1. P1. 专业化不足
   1. 当前角色职责有了，但系统方法论还没真正前置进接单、执行和交付链。
   2. 具体表现：
      1. 缺少强制方法卡绑定；
      2. 缺少角色专属检查单；
      3. 缺少明确反模式约束；
      4. 首版交付仍常带“凭感觉补一段”的原始性。
2. P2. 工程化不足
   1. 当前已开始补流程、门禁和结构治理，但工程系统还没有稳定产出“高质量默认结果”。
   2. 具体表现：
      1. 半成品缺少硬性 blocker；
      2. 完成定义和质量门不够硬；
      3. `7x24` 还没有固定的工程化反思与试错泳道；
      4. 工程质量改进的效果缺少连续观测。
3. P3. 质量判断不足
   1. 当前对“高质量”的判断还偏主观。
   2. 具体表现：
      1. 缺少样本库；
      2. 缺少多轮对比与 pairwise compare；
      3. 缺少真实后果回写；
      4. 缺少高手校准与价值排序机制；
      5. 缺少“半成品/可交付/高质量/超高质量”的分层判断标准。

## 4. 功能需求清单
1. FR-QLT-01 三角色专业合同前置
   1. `workflow_devmate / workflow_testmate / workflow_qualitymate` 的真实任务默认必须带专业合同。
   2. 合同最少包含：
      1. `problem_type`
      2. `method_card_id`
      3. `done_definition`
      4. `forbidden_anti_patterns`
2. FR-QLT-02 角色检查单与反模式前置
   1. 三角色必须各自拥有专属检查单。
   2. 每个角色还必须有一组“本轮禁止出现的反模式”。
3. FR-QLT-03 半成品 blocker 清单
   1. 必须形成一份正式的“半成品 blocker”清单。
   2. 至少覆盖：
      1. 真相源不清；
      2. 关键验证缺失；
      3. 主要边界未收口；
      4. 结论与证据不匹配；
      5. 交付仍需 PM 大幅二次加工。
4. FR-QLT-04 交付分层定义
   1. 必须正式定义至少四档：
      1. `半成品`
      2. `可交付`
      3. `高质量`
      4. `超高质量`
   2. 每一档都要有可回读的判断条件。
5. FR-QLT-05 `7x24` 工程化反思泳道
   1. 每轮 `7x24` 不只推进业务/工程任务，也要显式判断工程化不足。
   2. 至少记录：
      1. 本轮发现的不足
      2. 当前假设
      3. 本轮小步试错
      4. 结果与后续动作
6. FR-QLT-06 质量判断 rubric v1
   1. 必须形成一份交付质量 rubric v1。
   2. 首期至少覆盖：
      1. 真相正确性
      2. 方法使用正确性
      3. 完成定义符合度
      4. 证据充分度
      5. 价值排序是否合理
7. FR-QLT-07 比较评审机制
   1. 高质量判断不能只看单件产物，必须支持 pairwise compare。
   2. 同类交付至少要能回答：
      1. 哪一份更好
      2. 为什么更好
      3. 差距属于方法、工程还是判断
8. FR-QLT-08 样本库与后果回写
   1. 需要正式沉淀真实样本，而不是只留最佳实践。
   2. 必须同时保留：
      1. 好样本
      2. 差样本
      3. 返工样本
      4. 线上/回归后暴露问题样本
9. FR-QLT-09 高手校准入口
   1. 后续要预留“高手校准/用户校准”能力。
   2. 校准结果不能覆盖原始记录，而是作为额外判断层回写。
10. FR-QLT-10 价值排序规则
   1. 高质量判断不能只看技术整洁度。
   2. 至少要显式排序：
      1. 真相正确
      2. 风险控制
      3. 用户价值
      4. 长期可维护性
      5. 表达与可读性

## 5. 验收标准
1. AC-QLT-01 专业合同前置成立
   1. Given 三角色收到真实任务
   2. When 查看任务合同
   3. Then 能看到 `problem_type / method_card_id / done_definition / forbidden_anti_patterns`
2. AC-QLT-02 半成品 blocker 生效
   1. Given 某份交付缺关键真相、关键验证或主要边界
   2. When 进入验收
   3. Then 系统必须能明确判定为 `半成品`
   4. And 不得直接放行为 `高质量`
3. AC-QLT-03 `7x24` 工程化反思可回读
   1. Given 连续若干轮 `7x24`
   2. When 回读留痕
   3. Then 能看到稳定的工程化不足、试错动作和结果回看
4. AC-QLT-04 rubric v1 可用于比较
   1. Given 至少两份同类交付
   2. When 执行 compare review
   3. Then 能输出结构化评分、优劣判断和差异原因
5. AC-QLT-05 样本与后果建立链接
   1. Given 一批真实交付已进入样本库
   2. When 回读样本
   3. Then 至少一部分样本能追到后续返工、失败或成功后果

## 6. 边界条件与异常处理
1. 若当前缺少足够样本：
   1. 不得伪称“判断体系已成熟”。
   2. 应先以 `rubric v1 + pairwise compare` 作为过渡。
2. 若角色当前没有可用方法卡：
   1. 不得直接进入正式专业派单。
   2. 应先补最小方法入口。
3. 若工程化反思只留下观察、没有试错动作：
   1. 不得视为完成 `QLT-S3`。
4. 若高质量判断与真实后果长期冲突：
   1. 必须优先修判断体系，而不是只调整文案或补报告。

## 7. 依赖项与开放问题
1. 依赖项：
   1. `pm/expertise/` 下的专业能力、学习方法和问题-方法映射持续维护。
   2. `V9-R4 / V9-R5` 具备正式版本目录、阶段看板和 history 留痕。
   3. `7x24` 主线能稳定把治理动作写回版本现场。
2. 开放问题：
   1. 首期的 compare review 是先走文档治理，还是直接进入任务中心字段合同。
   2. 高手校准在首期默认由谁承担：用户、PM，还是 `workflow_qualitymate` 协助。
   3. 样本库首期是只收失败/返工样本，还是同时收“高质量正例”。
