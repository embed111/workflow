# 需求详情-第二个项目最小启动能力

- 状态：正式需求（当前有效口径）
- 当前边界：本轮目标不是把完整多项目体系一次做完，而是尽快让“第二个项目”具备最小可启动能力；项目模板市场、复杂项目看板、跨项目高级编排后续再补。
- 关联主题：
  1. `docs/workflow/requirements/需求详情-运营项目与项目主控角色模型.md`
  2. `pm/versions/V5/版本计划.md`

## 1. 主题目标与价值说明
1. 问题定义（用户明示）：
   1. 当前推进速度偏慢，用户不希望继续等待完整多项目体系成熟后，才开始第二个项目。
   2. 用户当前更在意“尽快开始创建第二个项目”，而不是先把 `V5-R4` 的全部理想形态一次补齐。
2. 主题目标：
   1. 把 `V5-R4` 的首批目标压缩为“第二个项目最小启动能力”。
   2. 让系统尽快具备：
      1. 定义第二个项目对象；
      2. 绑定该项目的项目主控角色；
      3. 绑定首批领域角色；
      4. 为该项目保留独立节奏和最小运行口径；
      5. 能区分“当前基础项目 `workflow`”与“第二个项目”。
3. 业务价值：
   1. 尽快验证“多项目底座”不是纸面概念，而是真能起第二个项目。
   2. 避免因为追求完整项目模型而长期推迟第二项目启动。
   3. 用真实第二项目反过来暴露项目模型和项目看板真正缺什么，而不是继续在抽象层设计。

## 2. 首期最小范围
1. 首期必须包含：
   1. 第二个项目的最小对象模型
   2. 项目主控角色绑定
   3. 项目成员列表绑定
   4. 项目目标字段
   5. 最小 `runtime_policy_ref`
   6. 当前项目与第二项目的明确区分入口
2. 首期可以暂缓：
   1. 完整项目看板视觉深化
   2. 丰富项目模板市场
   3. 跨项目高级编排与冲突治理
   4. 子项目体系
   5. 项目级复杂统计分析

## 3. 核心流程
1. 用户在当前 `workflow` 底座下创建第二个项目。
2. 用户为第二个项目指定：
   1. `project_name`
   2. `project_type`
   3. `controller_role_id`
   4. `member_role_ids`
   5. `project_goal`
3. 系统为第二个项目生成最小运行配置：
   1. `runtime_policy_ref`
   2. `default_handoff_interval_minutes`
   3. `next_handoff_interval_minutes`
4. 系统能够区分：
   1. 基础项目 `workflow`
   2. 新建的第二个项目
5. 第二个项目创建完成后，用户至少能确认：
   1. 项目已存在
   2. 项目主控角色已绑定
   3. 项目成员已绑定
   4. 下一棒节奏已有独立默认值

## 4. 功能需求清单
1. FR-P2-01 第二个项目最小对象创建
   1. 系统必须支持创建第二个项目的最小对象。
   2. 首期至少包含：
      1. `project_id`
      2. `project_name`
      3. `project_type`
      4. `controller_role_id`
      5. `member_role_ids`
      6. `project_goal`
      7. `runtime_policy_ref`
2. FR-P2-02 第二个项目主控角色绑定
   1. 第二个项目必须能显式绑定项目主控角色。
   2. 未绑定主控角色时，不得伪装成已具备 7x24 运行条件。
3. FR-P2-03 第二个项目最小运行口径
   1. 第二个项目必须拥有独立的最小运行口径。
   2. 首期至少支持：
      1. `default_handoff_interval_minutes`
      2. `next_handoff_interval_minutes`
4. FR-P2-04 `workflow` 与第二项目区分
   1. 系统必须明确区分：
      1. `workflow` 作为底座/基础项目
      2. 第二个项目作为新运营项目
   2. 不得继续把第二项目语义混回 `workflow` 自己的单项目口径。
5. FR-P2-05 首期不等待完整项目看板
   1. 第二项目启动能力不得被“完整项目看板 UI 设计未定”长期阻塞。
   2. 只要最小对象、角色绑定和最小运行口径已经成立，就应允许进入“可启动”状态。

## 5. 验收标准
1. AC-P2-01 第二项目可被正式创建
   1. Given 当前系统已经存在基础项目 `workflow`
   2. When 用户创建第二个项目
   3. Then 系统能返回一个独立项目对象
2. AC-P2-02 项目主控角色已绑定
   1. Given 用户完成第二项目最小创建
   2. When 查看项目对象
   3. Then 能明确看到 `controller_role_id`
3. AC-P2-03 第二项目有独立节奏
   1. Given 第二个项目创建成功
   2. When 查看其最小运行配置
   3. Then 能看到独立 `runtime_policy_ref`
   4. And 至少包含默认接力间隔字段
4. AC-P2-04 不再只有单项目口径
   1. Given 系统里已有 `workflow` 和第二个项目
   2. When 对照对象或入口
   3. Then 能明确区分两者，不再把全部运行语义压回单一项目

## 6. 边界条件与异常处理
1. 若第二个项目尚未绑定主控角色：
   1. 系统必须明确提示“项目已建，但尚未具备完整运行条件”。
2. 若项目模板或项目看板未就绪：
   1. 不得阻止第二个项目最小创建。
3. 若后续补项目看板：
   1. 应在现有项目对象基础上增强，而不是推翻首期最小项目对象。

## 7. 依赖项与开放问题
1. 依赖项：
   1. `需求详情-运营项目与项目主控角色模型.md`
   2. 现有角色、任务中心与运行策略基础设施
2. 开放问题：
   1. 第二个项目首期是通过专门创建入口、配置文件还是最小 UI 面板创建，后续实现切片再定。
   2. 首期第二个项目默认项目类型是否直接由用户输入，还是先提供少量预设选项，后续再定。
