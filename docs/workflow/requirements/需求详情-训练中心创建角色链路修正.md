## 1. 主题目标与价值说明
1. 问题定义（事实）：
   1. 当前 `训练中心 -> 创建角色` 链路中，结构化草稿 `role_spec` 已被聊天原文污染，而不是被维护为一个干净、可编辑、可追溯的角色草案对象。
   2. 取证会话 `rcs-20260323-102238-f565cb` 中，当前存储结果存在明显错位：
      1. `role_name` 仍是 `专门用于测试 workflow的`，未收口到用户后续确认的 `workflow_tester`；
      2. `role_goal` 混入了用户对当前画像不满意的抱怨语句；
      3. `boundaries` 被写入 `不能单独先改个名称吗`、`不能做好质量把关这件事` 这类对链路能力的质疑文本；
      4. `applicable_scenarios` 被写入 `你好` 等寒暄文本；
      5. `collaboration_style` 被写成用户对系统能力的要求原话，而不是可操作的协作规则。
   3. 当前链路把“消息已处理”误等同于“草稿已对齐”：
      1. 当前会话 `unhandled_user_message_count=0`；
      2. 但用户已经明确表达“角色目标写得不对”“和想象中还有差距”；
      3. 系统仍未进入结构化差异对齐回路。
   4. 当前链路把“最小字段已存在”误等同于“可放心进入下一步”：
      1. 当前 `can_start=true`、`missing_fields=[]`；
      2. 但会话仍是 `status=draft`、`current_stage=persona_collection`、`started_at=''`、`assignment_ticket_id=''`、`task_refs=0`；
      3. 用户侧容易感知为“系统说准备好了，但其实什么也没真正推进”。
   5. 当前分析链路对附件的注入方式过重：
      1. 最近一次 analyst prompt 文件约 `321333` bytes；
      2. trace `result.json` 返回 `codex_exec_timeout`；
      3. 当前存储中的 `example_assets` 仍内嵌 `data_url`，说明附件二进制被直接拼进了上游上下文。
2. 主题目标：
   1. 把 `创建角色` 阶段的 `role_spec` 升级为真正的一等草稿对象，而不是聊天抽取残渣。
   2. 让 `角色名称 / 角色目标 / 核心能力 / 边界 / 适用场景 / 协作方式` 都支持字段级独立修正。
   3. 把“消息已处理”“草稿已对齐”“允许开始创建”“已真正开始创建”四类状态拆开，避免状态语义混写。
   4. 当用户明确表达“不对 / 不满意 / 帮我调整”时，系统必须进入差异对齐回路，而不是继续泛化追问或直接超时。
   5. 收口附件注入策略，避免因为图片 base64 直接拼接导致 prompt 膨胀和 analyst 超时。
3. 业务价值：
   1. 让用户相信“当前看到的画像草稿”就是系统后续真正使用的对象，而不是聊天内容的偶然投影。
   2. 降低用户在早期验证阶段的交互挫败感，支持“先改名、再补目标、再调协作方式”的小步收敛。
   3. 让 `创建角色` 链路从“消息驱动”升级为“草稿驱动”，为后续任务创建、工作区初始化和验收阶段提供稳定输入。
   4. 降低分析超时与脏数据回写概率，提升训练中心创建链路的可用性与可追溯性。

## 2. 用户画像与使用场景
1. 用户画像 A：角色需求提出者
   1. 关注点：先用自然语言快速说出一个雏形，后续按字段逐步修，不想每次都重填整份角色画像。
2. 用户画像 B：workflow 验证者
   1. 关注点：当前轮要验证的是“创建角色链路是否真的可改、可收口、可开始”，而不是只看 agent 有没有回复。
3. 用户画像 C：训练中心治理者
   1. 关注点：草稿字段必须可信、干净、可回放，避免后续工作区初始化或任务生成建立在脏数据之上。
4. 核心场景：
   1. 用户先用一句模糊描述创建一个测试角色，后续再单独修改角色名称。
   2. 用户发现 `角色目标` 写错后，希望只改目标，不破坏已确认的能力信息。
   3. 用户对当前画像不满意，希望系统显式指出“当前值是什么、建议改成什么、还差什么确认”。
   4. 用户在带图片的场景下继续调角色草稿，但不希望因为附件过重导致系统超时或状态混乱。

## 3. 用户旅程或关键流程
1. 用户进入 `训练中心 -> 创建角色`，系统创建一个新的角色草稿会话。
2. 用户发出第一轮自然语言描述后，系统做两层处理：
   1. 原始消息按对话证据保留；
   2. 结构化草稿 `role_spec` 只写入经过字段归类与清洗后的候选值。
3. 系统展示当前草稿时，至少按字段区分：
   1. 已确认；
   2. 推断待确认；
   3. 空缺；
   4. 存在冲突或待修正。
4. 当用户说“改成 `workflow_tester`”“角色目标写得不对”“你能单独先改个名称吗”这类字段编辑语句时，系统应把它识别为“字段级修正动作”，而不是新的自由描述轮次。
5. 系统执行字段级修正后，应同步更新：
   1. 当前草稿对应字段；
   2. 草稿主标题或当前显示名称；
   3. 该字段的确认状态与最近一次修改来源。
6. 当用户表达“不满意 / 不对 / 帮我调整”时，系统进入“画像差异对齐”态：
   1. 明确指出当前不一致字段；
   2. 给出修正候选或最小澄清问题；
   3. 在这些字段确认前，不得把草稿视为“已对齐完成”。
7. 当最小字段满足后，系统可以进入“可开始创建但尚未开始”态：
   1. 允许用户点击 `开始创建`；
   2. 但不能伪装成已经进入 `创建中`。
8. 仅当用户明确开始创建，且后台真实完成工作区初始化与任务生成后，系统才把会话推进到 `creating`，并回写真实开始时间、任务引用和状态证据。

## 4. 功能需求清单（编号）
1. FR-RCF-01 结构化角色草稿升级为一等对象
   1. `role_spec` 必须作为独立草稿对象维护，不得仅由当前聊天消息即时拼装后再丢弃。
   2. 草稿字段至少包含：
      1. `role_name`
      2. `role_goal`
      3. `core_capabilities`
      4. `boundaries`
      5. `applicable_scenarios`
      6. `collaboration_style`
      7. `example_assets`
   3. 每个字段至少应具备以下业务语义：
      1. 当前值；
      2. 最近一次修改来源；
      3. 当前确认状态。
   4. 聊天原文是证据，不是最终字段值本身；任何字段都不得简单等同于“最近一句用户原话”。
2. FR-RCF-02 支持字段级独立编辑
   1. 系统必须支持对以下字段进行独立修正：
      1. `角色名称`
      2. `角色目标`
      3. `核心能力`
      4. `边界`
      5. `适用场景`
      6. `协作方式`
   2. 用户发起某一字段修正时，不得强制重填全部字段。
   3. 仅改某个字段时，系统只更新受影响字段及其依赖展示，不得把其他已确认字段回退成未确认。
   4. 用户确认名称后，当前草稿主标题必须同步体现最新名称，不得长期停留在历史初始文案。
3. FR-RCF-03 聊天抽取必须具备脏数据拦截与字段清洗
   1. 寒暄语、系统能力质疑、链路抱怨、元对话说明不得直接写入结构化角色字段。
   2. `role_goal` 只能承载角色目标与价值，不得混入“你能不能帮我改一下”这类编辑请求。
   3. `boundaries` 只能承载角色不做什么或被禁止做什么，不得混入对系统当前功能缺失的抱怨。
   4. `applicable_scenarios` 只能承载真实使用场景，不得写入 `你好`、`你知道 workflow 是什么吧` 这类无效文本。
   5. 当系统无法确定某句属于哪个字段时，应保留为待确认线索，而不是硬写入某个字段。
4. FR-RCF-04 用户不满意时必须进入差异对齐回路
   1. 当用户表达“当前画像不对 / 目标写错了 / 和预期有差距 / 帮我调整”时，系统必须进入差异对齐态。
   2. 差异对齐态至少需要输出：
      1. 当前字段值；
      2. 建议修正值或修正方向；
      3. 还待确认的最小问题集。
   3. 在差异对齐完成前，系统不得把本轮仅标记为“已处理完成”。
   4. 差异对齐完成的判断依据必须是“用户确认字段已收口”，而不是“消息队列已清空”。
5. FR-RCF-05 启动门槛与对齐状态分离
   1. `ready_to_start/can_start` 仅表示最小启动字段已具备，不等同于草稿已经完全对齐。
   2. 系统必须引入独立的“草稿对齐状态”语义，用于表达：
      1. 收集中；
      2. 可开始但未开始；
      3. 待对齐；
      4. 已真正创建中；
      5. 已完成。
   3. 若用户明确表示当前画像仍有问题，会话应进入“待对齐”而不是继续显示为正常收口完成。
6. FR-RCF-06 开始创建后的状态推进必须真实同步
   1. 只有在用户明确触发 `开始创建` 且后台成功执行后，系统才允许把会话状态推进到 `creating`。
   2. 进入 `creating` 时必须同步具备以下事实：
      1. `started_at` 已写入；
      2. `workspace_init_status` 不再停留在纯 `pending`；
      3. 已存在真实 `assignment_ticket_id` 或等效任务引用；
      4. `task_refs` 或等效任务投影可回查。
   3. 当仅达到“可开始”但尚未真正开始时，页面必须明确显示“已准备好开始”而不是让用户误判为“系统已经推进”。
7. FR-RCF-07 附件传递改为轻量引用策略
   1. 角色草稿中的附件必须按“元数据 + 可回取引用”保存，而不是在分析 prompt 中直接拼接大段二进制 `data_url`。
   2. analyst 调用所需的附件信息至少包括：
      1. `attachment_id`
      2. `file_name`
      3. `content_type`
      4. `size_bytes`
      5. 可读取引用或稳定本地路径
   3. 当附件不可读取或引用失效时，系统应明确提示该轮只能按文本继续对齐，而不是静默失败或整体超时。
8. FR-RCF-08 消息已处理、草稿已对齐、会话已开始必须分开留痕
   1. `unhandled_user_message_count=0` 只表示消息队列已清空，不得被解释为草稿已经对齐。
   2. 系统必须提供独立指标或状态，用于表达当前还有多少字段待对齐或待确认。
   3. 每次字段修正都应保留：
      1. 变更前值；
      2. 变更后值；
      3. 触发消息；
      4. 处理结果；
      5. 失败原因（若有）。
9. FR-RCF-09 本轮修正不重写六阶段主链路
   1. 本主题不改动现有 `创建角色` 的六阶段主流程语义：
      1. 初始化工作区
      2. 角色画像收集
      3. 能力后台生成
      4. 结果回看与调向
      5. 确认验收
      6. 完成角色创建
   2. 本主题只修正“进入这些阶段前的草稿质量、状态真实性与对齐机制”。
   3. 当前阶段图、任务引用和工作区初始化链路应继续复用既有主题，不另起一套平行流程。

## 5. 非功能需求（性能、安全、可用性、合规等）
1. 数据质量：
   1. 草稿字段不得出现明显与字段语义无关的寒暄语、抱怨语或元对话文本。
   2. 对脏数据应优先拦截和标记，而不是“先写进去、以后再说”。
2. 性能：
   1. 在“单图 + 单轮文本修正”的常见对齐场景下，analyst prompt 文本体积 `p95 <= 80KB`。
   2. 图片二进制不得以内联 base64 方式直接进入 analyst prompt。
   3. 对于仅修改 `名称 / 目标 / 协作方式` 的轻量对齐请求，系统返回 `p95 <= 10s`。
3. 一致性：
   1. 用户当前看到的草稿字段、后端存储值与后续开始创建时使用的值必须一致。
   2. `草稿主标题 / 角色名称 / 当前会话主显示名称` 的口径必须一致或明确标注差异来源。
4. 可恢复性：
   1. 页面刷新或重新进入后，草稿字段值、待对齐字段和最近一次修正状态必须可恢复。
   2. analyst 超时或失败时，不得把会话伪装成“已完成处理”；应保留待重试或待继续对齐状态。
5. 可追溯性：
   1. 每次字段编辑必须可追溯到原始消息与处理 trace。
   2. 附件引用链必须可回放到原始上传记录，不得只留下内联文本快照。

## 6. 验收标准（Given/When/Then）
1. AC-RCF-01 单独改名应真实生效
   1. Given 当前角色草稿已存在初始名称
   2. When 用户说“改成 `workflow_tester`”
   3. Then `role_spec.role_name` 更新为 `workflow_tester`
   4. And 当前草稿主标题同步展示 `workflow_tester`
   5. And 系统不得要求用户重新填写全部画像字段
2. AC-RCF-02 单独改目标不应污染字段
   1. Given 当前 `role_goal` 与用户预期不一致
   2. When 用户说“当前角色画像中角色目标写得不对，你能帮我调整一下吗”
   3. Then 系统进入目标字段对齐态
   4. And 输出新的目标候选或最小澄清问题
   5. And `role_goal` 中不得混入该句抱怨文本本身
3. AC-RCF-03 无效聊天文本不得进入边界或场景
   1. Given 用户消息中包含 `你好`、`你能单独先改个名称吗`
   2. When 系统抽取结构化草稿
   3. Then `applicable_scenarios` 不得写入 `你好`
   4. And `boundaries` 不得写入“你能单独先改个名称吗”这类编辑请求
4. AC-RCF-04 不满意必须进入差异对齐而非假完成
   1. Given 当前 `can_start=true`
   2. And 用户明确表示“当前画像不对”或“和想象中的还有差距”
   3. When 系统处理该轮消息
   4. Then 会话状态进入“待对齐”或等效语义
   5. And 页面可见待对齐字段
   6. And 不得仅因 `unhandled_user_message_count=0` 就显示为已收口完成
5. AC-RCF-05 可开始与已开始必须分开
   1. Given 角色草稿最小字段已齐备
   2. When 用户尚未点击 `开始创建`
   3. Then 会话显示为“可开始但未开始”
   4. And `started_at` 仍为空
   5. And 不得伪装出已创建任务图或已进入 `creating`
   6. When 用户明确点击 `开始创建` 且后台成功创建任务
   7. Then 会话状态切为 `creating`
   8. And `started_at`、`assignment_ticket_id`、任务引用同步可见
6. AC-RCF-06 附件不得以内联 base64 注入 analyst prompt
   1. Given 用户在当前草稿会话中上传一张图片并继续发起对齐
   2. When 系统组装 analyst prompt
   3. Then prompt 中只包含附件元数据与引用
   4. And 不包含大段 `data:image/...;base64,...` 文本
   5. And 单图对齐场景下 prompt 文本体积满足性能阈值
7. AC-RCF-07 analyst 超时后状态不得假成功
   1. Given 某轮 analyst 执行超时或失败
   2. When 系统回写该轮处理结果
   3. Then 会话保留“待继续对齐 / 待重试”状态
   4. And 用户能看到失败原因或可读提示
   5. And 不得把该轮标记成草稿已对齐
8. AC-RCF-08 字段变更可追溯
   1. Given 用户完成一次字段修正
   2. When 用户或系统回看该字段历史
   3. Then 至少可看到“变更前 / 变更后 / 触发消息 / 处理 trace”
9. AC-RCF-09 当前案例的污染数据被纠正
   1. Given 会话 `rcs-20260323-102238-f565cb` 的现状取证结果
   2. When 按本需求完成修正后重新跑同类链路
   3. Then `role_name` 应收口为用户确认名称
   4. And `role_goal` 只保留角色目标本身
   5. And `boundaries`、`applicable_scenarios`、`collaboration_style` 不再混入抱怨语、寒暄语和元对话

## 7. 边界条件与异常处理
1. 当用户连续两条消息分别修改不同字段：
   1. 系统应逐字段处理；
   2. 不得因为后一条覆盖前一条而丢失已确认字段。
2. 当用户在“可开始但未开始”状态继续改字段：
   1. 系统应允许继续修；
   2. 并重新评估对齐状态；
   3. 不得强行推进到 `creating`。
3. 当用户上传图片但图片引用暂不可读：
   1. 系统应提示“本轮按文本继续”或引导重新上传；
   2. 不得因为附件处理失败而直接把整轮对齐标记成功。
4. 当 analyst 超时：
   1. 会话应保留原草稿；
   2. 待对齐字段与失败原因需继续可见；
   3. 不得清空用户刚刚发出的修正意图。
5. 当用户输入与字段语义冲突：
   1. 系统应标记冲突并发起最小澄清；
   2. 不得硬写为某个字段的新最终值。

## 8. 依赖项与开放问题
1. 依赖项：
   1. 依赖 `role_creation_sessions` 能承载结构化草稿与字段状态扩展。
   2. 依赖 `创建角色` 前端详情区支持字段级展示、待对齐提示和主标题同步。
   3. 依赖 analyst 调用链路支持“附件引用注入”而不是只支持文本内联。
   4. 依赖 trace 与运行日志可继续作为字段修正与失败回放证据。
2. 开放问题：
   1. `草稿主标题` 是否严格同步 `session_title`，还是允许只同步页面主显示名称，仍待确认。
   2. 差异对齐态的页面呈现方式，是中栏内联 diff 还是右侧辅助区承载，仍待原型确认。
   3. 当前是否需要补一组“仅改此字段”的快捷动作按钮，仍待交互确认。
3. 本轮边界（事实）：
   1. 本主题只修正创建角色草稿语义、对齐回路、状态语义和附件策略。
   2. 本主题不展开新的任务图结构、阶段重排或角色发布逻辑。
   3. 若后续要跨工作区交付实现，仍应先补一版与现有风格一致的修正原型预览。

## 9. 能力扩展评估（skill/agent）
1. 新增 skill：`暂不采用`
   1. 理由：当前问题核心是 `../workflow` 产品链路与状态语义设计缺失，不是当前分析工作区缺少新的通用分析技能。

3. 当前兜底方案：
   1. 先以本修正文档收口产品语义；
   2. 后续若进入跨工作区实现，再由执行工作区按本主题补接口、状态字段和交互原型。
