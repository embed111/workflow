## 0. 所有项目创建共识
1. 后续所有运营项目创建，都不能只理解成“登记一个 project 对象”。
2. 一个项目只有同时具备以下三份合同，才允许被表述为“已完成创建并具备启动条件”：
   1. `API 能力合同`
      - 项目创建时必须能回答：当前平台有哪些受支持 API 可调用、哪些 API 与该项目启动/运行/回读直接相关。
      - 默认真相源为 `GET /api/platform/interfaces` 与 `GET /api/platform/interfaces/{interface_id}`。
      - 若相关接口仍是 `metadata_status=incomplete` 或 `metrics_status=unavailable`，项目创建结果必须显式标注为“接口目录待补齐 / 指标待接入”，不得包装成完整能力。
   2. `7x24 运行合同`
      - 项目创建时必须能回答：项目主控角色是谁、下一棒由什么 schedule/handoff 触发、什么情况下会把主控角色派发到任务中心执行。
      - 最小字段包括：`controller_role_id`、`member_role_ids`、`default_handoff_interval_minutes`、`next_handoff_interval_minutes`、启动触发来源、失败恢复策略和人工暂停状态。
      - 必须明确派发前置条件：主控角色可用、成员满足最低数量、项目未暂停、当前不处于升级 drain、并发槽允许、任务图存在可派发节点或未来 schedule。
   3. `启动准入合同`
      - 项目创建后必须明确区分 `已登记`、`可启动`、`已点火`、`ready 待派发`、`running 真执行`、`blocked`、`failed`、`false-running/ghost-running`。
      - `startup_readiness=ready` 只表示具备启动条件，不等于项目已经 7x24 运行。
      - 若没有 enabled schedule、ready 节点或明确下一次 handoff，不能把项目说成“已经开始持续运行”。
3. “项目创建成功”默认只代表项目对象、runtime policy 和最小现场已经建立；“项目 7x24 已启动”必须另有可回读证据。
4. 这条共识适用于所有后续运营项目。`AI小说盈利项目` 只是一条真实样本，不应把它的业务内容写成 `workflow` 平台需求本体；但从中踩出的 API 目录、项目点火、7x24 派发和运行真相问题，必须回灌到平台通用创建能力。

## 1. 当前真实流程
1. 现在的“项目创建”本质上是 `project bootstrap API`，不是完整项目创建工作流。
2. 当前实际链路：
   1. 通过 `POST /api/projects/bootstrap` 创建或更新项目对象。
   2. 写入项目注册表与 runtime policy。
   3. 项目页通过 `dashboard.project_bootstrap_summary` 读到项目摘要。
   4. 若要真正运行，还要继续在任务中心创建 assignment/node，并把节点绑定到该项目后再 dispatch。
3. 当前已覆盖的核心字段：
   1. `project_name`
   2. `project_type`
   3. `controller_role_id`
   4. `member_role_ids`
   5. `project_goal`
   6. `project_board_ref`
   7. `runtime_policy_ref`
   8. `default_handoff_interval_minutes`
   9. `next_handoff_interval_minutes`
   10. `next_handoff_interval_override_source`
   11. `next_handoff_interval_effective_after_run`

## 2. 当前为什么还不算完整真实流程
1. 创建门不够硬：
   1. 当前强制必填的只有 `project_name` 和 `controller_role_id`。
   2. `project_goal / member_role_ids / project_type` 还不是硬门。
2. 角色真实性校验不足：
   1. 还没有在创建时强校验主控角色和成员是否真实存在、是否 workspace ready、是否可承担该项目职责。
3. “项目已创建”和“项目可运行”没有严格区分：
   1. 现在 `runtime_ready` 主要还是字段级判断，不是完整启动准入判断。
4. 创建后项目现场不完整：
   1. `project_board_ref` 现在更像一个引用字符串，不是完整项目板、首批计划和项目现场已建立的证明。
5. 页面没有正式创建向导：
   1. 现在项目页主要消费摘要，不是“填表 -> 校验 -> 确认 -> 创建完成”的完整创建流程。
6. 项目级运维参数过薄：
   1. 现在项目级运维参数几乎只覆盖“接力节奏”。
   2. 失败恢复、启动准入、人工控制这三类参数还没有正式进入项目模型或项目页。
7. 项目产出表达还停在文字摘要：
   1. 现在“项目产出”更像任务摘要入口，不足以把 `workflow` 这种平台项目讲清楚。
   2. 对这类项目来说，架构图、运行拓扑图和关键能力图比单纯任务摘要更适合作为主表达。

## 3. 目标真实流程
1. 用户进入 `项目运营`，点击创建运营项目。
2. 创建向导至少分四步：
   1. 基础信息
      1. `project_name`
      2. `project_type`
      3. `project_goal`
   2. 主控角色
      1. `controller_role_id`
      2. 主控角色可用性校验
   3. 成员与边界
      1. `member_role_ids`
      2. 成员可用性校验
      3. 成员重复/冲突提示
   4. 运行策略
      1. `default_handoff_interval_minutes`
      2. 首轮 `next_handoff_interval_minutes`
      3. 失败恢复策略
      4. 启动准入判断
      5. 人工控制项
3. 创建成功后，系统至少要同时产出：
   1. 正式项目对象
   2. 最小 runtime policy
   3. 最小项目现场引用
   4. 启动状态：`可启动 / 已创建待补齐 / 创建失败`
4. 项目详情里的“项目产出”默认不再只展示任务摘要，而要采用“架构表达为主、任务摘要为辅”的结构：
   1. 架构总览图
   2. 运行拓扑图
   3. 关键能力图
   4. 任务摘要入口
5. 若项目达到 `可启动`：
   1. 后续任务可明确绑定到该项目
   2. 项目主控角色可继续接 7x24 / self-iteration cadence
6. 若项目只是 `已创建待补齐`：
   1. 必须明确告诉用户缺什么，而不是让页面看起来像已经建好了

## 4. 最小字段与门禁口径
1. 后续创建运营项目的最小强制字段建议升级为：
   1. `project_name`
   2. `project_type`
   3. `project_goal`
   4. `controller_role_id`
   5. `member_role_ids`
   6. `default_handoff_interval_minutes`
2. 后续必须补的校验：
   1. 主控角色必须存在
   2. 主控角色必须可用
   3. 成员角色必须存在
   4. 成员列表不能为空
   5. 项目目标不能为空
   6. 项目类型不能为空
3. 后续必须补的结果字段：
   1. `creation_status`
   2. `startup_readiness`
   3. `blocking_items`
   4. `created_artifacts`

## 4.1 项目级运维参数扩展建议
1. 节奏参数
   1. `default_handoff_interval_minutes`
   2. `next_handoff_interval_minutes`
   3. `next_handoff_interval_override_source`
   4. `next_handoff_interval_effective_after_run`
2. 失败恢复参数
   1. `failure_retry_cooldown_minutes`
   2. `failure_retry_max_attempts`
   3. `failure_escalation_threshold`
   4. `failure_route_kind`
3. 启动准入参数
   1. `require_controller_ready`
   2. `require_member_roles_ready`
   3. `minimum_member_count`
4. 人工控制参数
   1. `manual_pause`
   2. `auto_resume_at`
   3. `allowed_run_window`
5. 分层原则
   1. 项目级参数只承接“策略级”控制，不直接暴露引擎内部退避秒数、provider 重试秒数或 polling 频率。
   2. 引擎级重试仍保留在调度/执行运行时，不直接下沉到项目页。

## 4.2 项目产出表达方案
1. 适用判断
   1. 对 `workflow` 这种平台型项目，“程序本体 + 结构 + 运行方式 + 已交付能力”才是项目产出的主体。
   2. 因此项目产出页不应只剩任务摘要或产物文件列表。
2. 首期默认结构
   1. `系统架构`
      1. 用架构总览图表达“这个项目是什么、由哪些核心部分组成”。
   2. `运行拓扑`
      1. 用运行拓扑图表达“它怎么跑、和哪些环境/仓/运行副本连接”。
   3. `关键能力`
      1. 用能力图表达“当前已经稳定交付了哪些能力面”。
   4. `任务摘要`
      1. 保留为辅助入口，用来追最新运行和最近留痕，不再承担主表达职责。
3. 首期表达原则
   1. 架构图做主表达，任务摘要退到辅助位。
   2. 图必须配最小文字说明，不做纯装饰图。
   3. 项目产出页优先表达“项目本体”，不是继续做第二个任务中心。

## 7. 当前结论
1. 当前接口已经够“最小可运行项目创建”。
2. 当前还不够“完整真实项目创建流程”。
3. 后续最关键的不是继续加字段，而是把：
   1. 创建合同
   2. 角色可用性
   3. 项目级运维参数
   4. 启动准入
   5. 页面向导
   6. 创建后联动
   这六块收成正式产品链路。
