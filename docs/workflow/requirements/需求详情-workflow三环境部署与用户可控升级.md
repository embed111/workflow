# 需求详情-workflow三环境部署与用户可控升级

## 1. 主题目标与价值说明
1. 主题目标：
   1. 将 `workflow` 的开发、测试、正式运行彻底分离，避免开发中代码直接影响正式用户。
   2. 建立 `dev -> test -> prod` 的单机三环境发布链路，并沉淀为可重复执行的一键部署机制。
   3. 让正式环境用户在不受开发/测试波动影响的前提下，自主选择升级时机。
   4. 将正式发布门禁收口为“不消耗 token 的自动化测试通过后自动形成正式升级候选”。
2. 业务价值：
   1. 降低每次升级对正式用户的干扰，避免“边开发边影响生产”。
   2. 让环境切换和发布行为标准化，减少人工漏操作与灰色状态。
   3. 让正式环境升级具备明确门禁、可追溯证据与可恢复路径。

## 2. 用户画像与使用场景
1. 用户画像 A：`workflow` 开发者
   1. 关注点：在不碰正式环境的前提下持续开发、部署和验证。
2. 用户画像 B：测试验证者
   1. 关注点：测试环境可独立部署、独立验收，结果可直接决定是否进入正式候选。
3. 用户画像 C：正式环境使用者
   1. 关注点：系统存在新版本时可见、可控，但不会被强制打断当前使用。
4. 用户画像 D：平台运维者
   1. 关注点：部署链路清晰、门禁自动执行、失败可回退、环境互不污染。
5. 核心场景：
   1. 开发者先将 `../workflow_code` 同步到 `../workflow/.repository/<developer_id>/` 开发工作区，并在该工作区完成修改后，一键部署到 `.running/dev` 自测。
   2. 开发完成后，一键部署到 `.running/test` 并执行一批不消耗 token 的自动化测试。
   3. 自动化测试全部通过后，系统自动形成“正式升级候选”。
   4. 正式环境用户打开页面时，在顶部看到悬浮置顶的升级提示与升级按钮。
   5. 用户在确认当前无运行中任务后，主动触发升级；或在明确授权维护窗口时，由 Codex 执行正式部署；页面允许短暂刷新和重连。

## 3. 用户旅程或关键流程
1. 开发者先将 `../workflow_code` 最新代码同步到 `../workflow/.repository/<developer_id>/`，并在该开发工作区完成本轮开发。
2. 开发者在对应开发工作区执行一键部署脚本，目标环境选择 `dev`，系统将当前版本部署到 `../workflow/.running/dev/`。
3. 开发者自测通过后，再在对应开发工作区执行一键部署脚本，目标环境选择 `test`，系统将当前版本部署到 `../workflow/.running/test/`。
4. 测试环境部署完成后，系统自动执行一批本地自动化测试用例；这些用例不得调用需要消耗 token 的执行链路。
5. 若测试全部通过，系统自动将该版本标记为“正式升级候选”。
6. 正式环境 `../workflow/.running/prod/` 持续提供用户使用；用户手动启动当前正式入口时，应在 `../workflow/` 根目录执行 `.\run_workflow.bat`。
7. 当正式环境检测到存在新“正式升级候选”且版本高于当前运行版本时，页面顶部显示悬浮置顶升级提示。
8. 用户可随时点击升级按钮触发升级前检查：
   1. 若仍存在 `running` 任务，则本次升级不得放行，并明确提示阻塞原因；
   2. 若不存在 `running` 任务，则允许进入正式切换。
9. 正式切换执行时，页面允许短暂刷新和重连；切换完成后用户继续在正式环境使用新版本。
10. 若正式切换失败，系统必须保留或恢复上一正式版本，并记录完整失败留痕。

## 4. 功能需求清单（编号）
1. FR-RU-01 三环境部署副本
   1. `workflow` 必须在 `../workflow/.running/` 下固定提供 3 个部署副本目录：
      1. `dev`
      2. `test`
      3. `prod`
   2. 这 3 个目录都属于运行副本，不是源码主工作区。
2. FR-RU-02 源码工作区与运行副本分离
   1. `../workflow_code` 作为正式代码根仓，`../workflow/.repository/<developer_id>/` 作为本地开发工作区；二者均不直接作为正式运行目录。
   2. 正式用户使用的程序必须来自 `../workflow/.running/prod/`。
3. FR-RU-03 一键部署脚本
   1. 系统必须提供可指定目标环境的一键部署脚本。
   2. 脚本至少支持 `dev/test/prod` 三种目标环境。
   3. 部署脚本必须支持重复执行，重复执行时不得破坏未选中环境。
4. FR-RU-04 正式启动入口固定
   1. `../workflow/run_workflow.bat` 默认只启动正式环境，但其本质是 PM 顶层便捷代理，不是正式代码副本。
   2. 该入口不得默认切到 `dev` 或 `test`。
5. FR-RU-05 环境运行态隔离
   1. `dev/test/prod` 必须拥有各自独立的运行态目录、端口、日志与进程标识。
   2. 任一环境部署失败或运行异常，不得直接污染其他环境运行态。
6. FR-RU-06 任务运行路径隔离
   1. `dev` 和 `test` 环境不得配置为与 `prod` 相同的任务运行路径。
   2. 若检测到 `dev/test` 与 `prod` 任务运行路径相同，系统必须阻断部署或启动，并明确提示原因。
7. FR-RU-07 agent 路径可共享
   1. `dev/test/prod` 可配置相同的 `agent路径`。
   2. 系统不得因为 `agent路径` 相同而误判为环境冲突。
8. FR-RU-08 测试环境自动发布门禁
   1. `test` 环境部署完成后，系统必须自动执行一批固定的自动化测试用例。
   2. 该批测试用例不得消耗 token。
   3. 自动化测试全部通过前，不得生成正式升级候选。
9. FR-RU-09 正式升级候选自动生成
   1. 当 `test` 环境自动化测试全部通过后，系统必须自动生成一条正式升级候选记录。
   2. 该候选记录至少包含：
      1. 候选版本标识
      2. 来源环境
      3. 自动化测试批次标识
      4. 通过时间
      5. 可追溯的测试证据路径
10. FR-RU-10 正式环境升级提示
   1. 当存在高于当前正式运行版本的正式升级候选时，正式环境页面顶部必须显示悬浮置顶升级提示。
   2. 提示至少展示：
      1. 当前版本
      2. 待升级版本
      3. 升级按钮
      4. 阻塞原因或当前可升级状态
11. FR-RU-11 用户可控升级
   1. 正式升级不得在用户无感知的情况下自动切换。
   2. 用户必须可在页面中主动选择升级时机。
12. FR-RU-12 升级前运行任务门禁
   1. 正式升级前，系统必须检查当前是否存在 `running` 任务。
   2. 若存在 `running` 任务，则升级按钮不得放行。
   3. 页面必须明确提示“存在运行中任务，暂不可升级”或等效原因。
13. FR-RU-13 正式切换体验
   1. 正式切换期间允许页面短暂刷新和重连。
   2. 切换完成后，页面必须恢复到正式环境新版本并保持可继续使用。
14. FR-RU-14 正式切换失败回退
   1. 若正式切换失败，系统必须回退或保留上一正式版本。
   2. 失败后不得让正式环境停留在“版本不明”或“半切换”状态。
15. FR-RU-15 发布留痕
   1. 每次 `dev/test/prod` 部署、自动测试、候选生成、正式升级、失败回退都必须留痕。
   2. 留痕内容至少包括：
      1. 环境
      2. 版本
      3. 动作类型
      4. 开始/结束时间
      5. 结果
      6. 证据路径

## 5. 非功能需求（性能、安全、可用性、合规等）
1. 可用性：
   1. 正式环境升级切换目标继续保持 `RTO <= 30 秒`。
   2. 正式环境升级切换目标继续保持 `RPO <= 5 秒`。
2. 隔离性：
   1. `dev/test` 不得污染 `prod` 的任务运行路径。
   2. 任一环境的异常进程、异常日志或失败部署不得导致其他环境失效。
3. 自动化门禁：
   1. 正式发布门禁固定为本地自动化测试通过，不依赖 token 消耗型执行链路。
   2. 自动化测试结果必须可复查、可重跑、可留痕。
4. 可追溯性：
   1. 正式环境当前版本、待升级版本与最近一次升级结果必须可追溯。
5. 可维护性：
   1. 部署脚本必须支持按环境重复执行，避免手工改路径和命令。
6. 安全性：
   1. 环境切换不得绕过“无运行中任务”门禁直接升级正式环境。

## 6. 验收标准（Given/When/Then）
1. AC-RU-01 可部署到开发环境
   1. Given 开发者已将 `../workflow_code` 同步到 `../workflow/.repository/<developer_id>/` 并完成修改
   2. When 执行一键部署脚本并指定 `dev`
   3. Then 系统成功将版本部署到 `../workflow/.running/dev/`
   4. And 不影响 `test/prod`
2. AC-RU-02 可部署到测试环境
   1. Given 当前版本已可部署
   2. When 执行一键部署脚本并指定 `test`
   3. Then 系统成功将版本部署到 `../workflow/.running/test/`
   4. And 不影响 `prod`
3. AC-RU-03 正式入口固定为 prod
   1. Given 用户在 `../workflow/` 根目录执行 `.\run_workflow.bat`
   2. When 服务启动
   3. Then 实际运行环境为 `../workflow/.running/prod/`
4. AC-RU-04 dev/test 与 prod 任务路径冲突时阻断
   1. Given `dev` 或 `test` 被配置为与 `prod` 相同的任务运行路径
   2. When 执行对应环境部署或启动
   3. Then 系统阻断本次动作
   4. And 明确提示冲突原因
5. AC-RU-05 agent 路径允许共享
   1. Given `dev/test/prod` 使用相同 `agent路径`
   2. When 各环境启动
   3. Then 系统允许启动
   4. And 不把该配置判定为冲突
6. AC-RU-06 测试通过后自动生成正式升级候选
   1. Given `test` 环境部署完成
   2. When 固定自动化测试用例全部通过
   3. Then 系统自动生成正式升级候选
   4. And 该过程不消耗 token
7. AC-RU-07 正式环境显示升级提示
   1. Given 存在高于当前正式版本的正式升级候选
   2. When 用户打开正式环境页面
   3. Then 页面顶部显示悬浮置顶升级提示与升级按钮
8. AC-RU-08 存在 running 任务时禁止升级
   1. Given 正式环境仍存在 `running` 任务
   2. When 用户点击升级按钮
   3. Then 系统不得放行升级
   4. And 页面明确提示阻塞原因
9. AC-RU-09 无 running 任务时可升级
   1. Given 正式环境不存在 `running` 任务
   2. And 存在正式升级候选
   3. When 用户点击升级按钮
   4. Then 系统执行正式切换
   5. And 页面允许短暂刷新和重连
   6. And 切换后正式环境版本变为候选版本
10. AC-RU-10 正式切换失败自动回退
   1. Given 用户已触发正式升级
   2. When 切换过程失败
   3. Then 系统回退或保留上一正式版本
   4. And 正式环境仍可继续使用
   5. And 有完整失败留痕

## 7. 边界条件与异常处理
1. 若 `dev` 部署失败：
   1. 仅标记 `dev` 本次部署失败，不得影响 `test/prod`。
2. 若 `test` 自动化测试失败：
   1. 不得生成正式升级候选。
   2. 页面不得向正式环境用户显示该版本可升级。
3. 若正式环境当前无升级候选：
   1. 页面不得展示可点击升级按钮，或需展示禁用态并说明“暂无可升级版本”。
4. 若用户点击升级时恰好出现新的 `running` 任务：
   1. 系统以最终门禁检查结果为准，不得放行存在竞争条件的升级。
5. 若正式候选记录存在但证据路径缺失：
   1. 不得允许进入正式升级。
   2. 系统需提示候选不完整并要求重新生成。
6. 若正式切换完成后新版本健康检查失败：
   1. 系统必须自动回退到上一正式版本并留痕。

## 8. 依赖项与开放问题
1. 依赖项：
   1. `../workflow` 支持按环境区分运行根目录、端口与进程，并由 `../workflow_code` 或开发工作区副本提供启动与部署脚本。
   2. 存在一批本地可执行且不消耗 token 的自动化测试用例。
   3. 正式环境页面可读取当前版本与候选版本元数据。
   4. 任务运行状态可被稳定识别，用于升级前门禁判断。
2. 开放问题：
   1. 推断/假设（置信度: 中）：正式升级按钮是否需要附带“查看本次升级候选测试结果/变更摘要”的二级入口，后续原型阶段可再收口。
