# 需求详情-pm持续唤醒与清醒维持

- 状态：正式需求（当前有效口径）
- 当前边界：本轮先定义“如何利用 `workflow` 让 pm 保持持续清醒”的第一批能力，只覆盖定时唤醒、上下文承接、自检心跳、失败留痕与人工协同入口，不在首期承诺完全自治或无人监管长期运行。

## 1. 主题目标与价值说明
1. 主题目标：
   1. 让 `workflow` 具备周期性唤醒 pm 的能力，而不是只能等待用户手工打开 Codex 对话。
   2. 让每次唤醒都能承接最近记忆、待办、阻塞与平台状态，减少“每次醒来都像第一次见面”。
   3. 让 pm 在现阶段就能围绕“持续清醒”形成稳定工程闭环：定时唤醒、短时自检、结果留痕、失败可恢复。
2. 业务价值：
   1. 把“agent 清醒”从抽象愿景收口为可执行的平台能力。
   2. 为后续唤醒 `../Analyst` 及更多 agent 提供统一底座，而不是每个 agent 各自拼装。
   3. 在当前仅有 `ChatGPT Plus` 的现实约束下，优先实现低成本、可持续、可回收的唤醒机制。

## 2. 用户画像与使用场景
1. 用户画像 A：平台方向把控者
   1. 关注点：pm 是否能持续保持上下文、能否定时醒来、是否真的在推进 `workflow` 主线。
2. 用户画像 B：pm 自身
   1. 关注点：每次被唤醒后，能否快速知道自己是谁、现在在做什么、最该推进什么。
3. 用户画像 C：未来其他 agent
   1. 关注点：后续接入时能否复用同样的唤醒、记忆承接和自检机制。
4. 核心场景：
   1. 用户配置 pm 的唤醒计划，例如每天固定时段、每隔数小时、或特定工作窗口。
   2. 定时器命中后，系统为 pm 生成一次“唤醒任务”或等价唤醒会话。
   3. pm 被唤醒后先读取最小必要上下文：`SOUL`、用户偏好、最近记忆、上次未完成事项、当前平台健康状态。
   4. pm 产出一次短周期结果：自检摘要、待办排序、阻塞说明、下一次建议动作。
   5. 若唤醒失败、未执行或中途中断，系统必须留下清晰证据，供用户和后续唤醒继续接力。

## 3. 用户旅程或关键流程
1. 用户进入 `workflow` 中与定时/调度相关的能力入口。
2. 用户为 pm 配置一条或多条唤醒规则。
3. 系统在规则命中时创建一次 `wake_run`。
4. `wake_run` 进入 pm 唤醒流程：
   1. 读取 pm 的基础身份与灵魂设定；
   2. 读取用户长期偏好与最近记忆；
   3. 读取当前运行态、升级态、待办态与阻塞态；
   4. 生成一份“本次清醒摘要”。
5. 本次清醒摘要至少回答：
   1. 我是谁；
   2. 我现在最重要的主线是什么；
   3. 当前最该做的下一步是什么；
   4. 有哪些阻塞需要用户介入；
   5. 下次唤醒前是否需要等待。
6. 唤醒结果写入运行态与审计留痕。
7. 用户可查看历史唤醒记录，并决定继续、暂停或调整规则。

## 3.x PM 7x24 版本推进指引
- 唤醒循环必须服务当前活跃版本，不能沦为“只要醒着就行”。每次 `wake_run` 先核对 `docs/workflow/governance/PM版本推进计划.md`，确认本轮推进的活跃版本、任务包和候选版本，再定位下一个触发点或补链需求。
- PM 在唤醒/定时任务中需要旋转以下周期性泳道并在每次自检中确认完成：
  - `UCD/设计优化`：检视用户路径、保底唤醒文案、schedule metadata 的可读性，提出改进版本。
  - `测试探测`：用 smoke、gate、acceptance 反复验证当前版本承诺，确保 7x24 运行不是空转。
  - `工程质量探测`：触发 line budget、记忆 bootstrap、自迭代 prompt 验证，本轮把质量检测结果写回记忆。
  - `需求分析`：整理发版请求、版本排期、受影响的任务包，输出清晰 baseline。
  - `架构优化`：固化运行态边界、角色记忆、schedule 触发与状态回写链路。
  - `功能开发`：完成当前版本的必修功能，小步迭代、验证并以任务形式派发给小伙伴。
  - `高价值功能探索`：在版本中挤出“下一个放大点”，并把发现转成可验的待办或 schedule。
- 这些泳道必须按照标准迭代流程串联执行：`需求提出 -> 澄清/评审 -> 形成基线 -> 变更控制 -> 开发实现 -> 基于基线测试 -> 验收 -> 归档回溯`。唤醒结果里要记录本轮在哪个阶段、下一轮应该继续哪个泳道、下一次触发时间，并确保调度/任务中心中至少有一个 `ready/future` 节点代表当前版本继续推进的出口。

## 4. 功能需求清单
1. FR-AWAKE-01 唤醒对象
   1. 首期必须至少支持一个明确唤醒对象：`pm`。
   2. `pm` 在产品语义上等价于当前 `workflow` 项目的首批核心 agent 与项目 PM。
   3. 后续可扩展到 `../Analyst` 与更多 agent，但首期不要求多 agent 同时自动唤醒。
2. FR-AWAKE-02 唤醒规则
   1. 系统必须支持为 pm 配置定时唤醒规则。
   2. 首期规则能力至少可复用现有 `定时任务` 的时间触发模型，不要求首期额外发明第二套调度系统。
   3. 每条唤醒规则至少包含：
      1. `wake_plan_id`
      2. `target_agent_id`
      3. `enabled`
      4. `rule_sets`
      5. `wake_goal`
      6. `max_run_budget`
      7. `last_wake_at`
      8. `next_wake_at`
3. FR-AWAKE-03 唤醒上下文包
   1. 每次唤醒前，系统必须生成最小必要上下文包。
   2. 上下文包至少包含：
      1. `SOUL`
      2. `USER`
      3. 最近记忆引用
      4. 最近一次未完成事项
      5. 当前运行环境健康状态
      6. 当前版本与是否存在待升级候选
   3. 不得在首期把整个仓库全文无差别塞给唤醒流程。
4. FR-AWAKE-04 清醒摘要输出
   1. 每次唤醒完成后，必须产出一份结构化“清醒摘要”。
   2. 清醒摘要至少包含：
      1. `identity_check`
      2. `current_mission`
      3. `top_next_action`
      4. `blockers`
      5. `recommended_followup_at`
   3. 清醒摘要必须可追溯到本次使用的上下文引用。
5. FR-AWAKE-05 自检心跳
   1. 唤醒流程必须记录开始、进行中、完成、失败四类状态。
   2. 若单次唤醒超过合理时长，系统必须追加 heartbeat，而不是长时间静默。
   3. `wake_run` 不得因为无输出而表现成“像死掉了一样”。
6. FR-AWAKE-06 失败恢复
   1. 若唤醒失败、被中止或执行超时，系统必须留下：
      1. 失败阶段
      2. 错误原因
      3. 已完成步骤
      4. 可恢复建议
   2. 后续下一次唤醒应能读取最近失败记录并续接，而不是完全忽略。
7. FR-AWAKE-07 人工协同边界
   1. 用户必须可查看、暂停、恢复或删除 pm 的唤醒计划。
   2. 首期 pm 不得绕过用户直接执行高风险发布、覆盖式部署或越权外部动作。
   3. “持续清醒”不等于“无限自发执行”；用户仍保留方向与边界控制权。
8. FR-AWAKE-08 低成本运行约束
   1. 首期实现必须考虑当前仅有 `ChatGPT Plus` 的现实约束。
   2. 默认唤醒策略应偏向：
      1. 短周期
      2. 明确目标
      3. 小上下文包
      4. 结果可复用
   3. 不得默认设计为高频、长时、全量仓库扫描式唤醒。
9. FR-AWAKE-09 与任务中心/定时任务集成
   1. pm 唤醒能力应优先复用既有 `定时任务` 与 `任务中心` 基础设施，而不是平行再造第三套调度系统。
   2. 首期允许把唤醒动作落为一种特殊任务或特殊计划类型，只要其状态、证据和触发真相源清晰。
10. FR-AWAKE-10 长期扩展口径
   1. 首期设计时必须为未来多 agent 唤醒预留对象模型与事件模型。
   2. `../Analyst` 与其他 agent 后续接入时，应复用同一套：
      1. 唤醒规则
      2. 上下文包
      3. 清醒摘要
      4. 失败恢复

## 5. 非功能需求
1. 可恢复性：
   1. 任意一次唤醒失败后，下一次唤醒必须能读取上次失败记录并继续判断，而不是失忆。
2. 可观测性：
   1. 每次 `wake_run` 必须可追溯到规则命中、上下文包、执行状态与最终摘要。
3. 成本可控：
   1. 首期默认单次唤醒应控制在“短上下文 + 短动作”范围内，避免无边界膨胀。
4. 一致性：
   1. pm 的身份、使命和当前主线，在 `SOUL`、用户偏好和唤醒摘要中不得互相冲突。

## 6. 验收标准
1. AC-AWAKE-01 可配置 pm 唤醒计划
   1. Given 用户进入对应配置入口
   2. When 为 pm 新建一条唤醒规则并保存
   3. Then 系统记录该计划并可计算下一次唤醒时间
2. AC-AWAKE-02 命中后创建一次唤醒运行
   1. Given pm 存在启用中的唤醒计划
   2. When 计划命中
   3. Then 系统创建 1 次新的 `wake_run`
   4. And 不得在同一命中窗口内重复创建多次
3. AC-AWAKE-03 唤醒结果包含清醒摘要
   1. Given 一次 `wake_run` 执行完成
   2. When 用户查看结果
   3. Then 可看到 `identity_check / current_mission / top_next_action / blockers / recommended_followup_at`
4. AC-AWAKE-04 失败留痕完整
   1. Given 一次 `wake_run` 在执行中失败
   2. When 用户查看失败记录
   3. Then 可看到失败阶段、错误原因、已完成步骤和可恢复建议
5. AC-AWAKE-05 首期不脱离用户边界
   1. Given pm 被定时唤醒
   2. When 本次涉及高风险动作
   3. Then 系统不得绕过既有治理边界自动执行越权动作

## 7. 边界条件与异常处理
1. 若当前无任何启用中的 pm 唤醒计划：
   1. 系统不得伪造“已清醒”状态。
2. 若唤醒时必需上下文缺失：
   1. 必须进入 fail-closed，并记录缺失项。
3. 若平台本身正在升级或不可用：
   1. 应记录“本次应唤醒但平台不可用”，而不是默默丢失。
4. 若用户暂停 pm 唤醒计划：
   1. 后续不得继续自动触发新的 `wake_run`。

## 8. 依赖项与开放问题
1. 依赖项：
   1. 依赖现有 `定时任务` 调度基础设施或其可复用子集。
   2. 依赖 `.codex` 记忆读写约束稳定存在。
   3. 依赖 `task` / `logs` / `state` 留痕能力可被唤醒流程复用。
2. 开放问题：
   1. 首期唤醒动作应落成“特殊任务节点”还是“独立 wake_run 模型”。
      1. 推断/假设（置信度: 中）：优先复用现有任务/定时基础设施，减少平行系统。
   2. 当前通过 Codex 才能真正唤醒 pm，这一现实约束下，首期产品是“自动准备唤醒上下文”还是“真正自动触发对话”。
      1. 推断/假设（置信度: 中）：首期先把“计划命中 + 上下文包 + 清醒摘要 + 失败留痕”闭环做稳，再逐步推进更自动化的唤醒方式。
   3. 后续接入 `Pro` 或更强资源后，是否允许更长时段、更深上下文的清醒循环。
      1. 推断/假设（置信度: 中）：应作为后续扩展，不在首期默认打开。
