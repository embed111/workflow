# 需求详情-任务中心后端接口与调度规则

## 1. 主题目标与价值说明
1. 为 `任务中心` 建立独立后端业务域，避免把任务编排语义误塞进训练队列或通用任务执行表。
2. 为后续其他 workflow 向同一张全局唯一任务图追加任务、调度任务预留稳定接口，而不是等页面实现完成后再补后端模型。
3. 让依赖门禁、优先级调度、并发门禁、失败阻塞成为后端强规则，而不是仅靠前端提示。
4. 让当前“创建任务”抽屉表单与后端字段契约一一对应，避免前后端口径漂移。
5. 将任务明文持久化边界从 `workflow` 自身目录迁出，收口到用户本地 `任务产物路径`。

## 2. 用户画像与使用场景
1. 用户画像 A：workflow 平台治理者
   1. 关注点：调度规则是否一致、审计是否完整、并发是否可控。
2. 用户画像 B：页面操作者
   1. 关注点：页面看到的任务图与后端状态一致，不出现“页面能点但后端拒绝”的漂移。
3. 用户画像 C：外部 workflow 调用方
   1. 关注点：能否通过接口向全局唯一任务图追加任务节点、配置依赖并读取图状态。
4. 核心场景：
   1. 页面内确保并读取全局唯一任务图状态。
   2. 页面内通过 `+ 创建任务` 提交单个任务节点。
   3. 外部 workflow 通过接口向全局唯一任务图追加任务。
   4. 调度器根据依赖、优先级和并发规则把任务推进到 `running`。
   5. 人工通过 `任务管理` 修改任务状态。
   6. 失败任务支持重跑，并允许继续人工修改执行状态。

## 3. 用户旅程或关键流程
1. 客户端或外部 workflow 调用 `确保全局任务图` 接口，或读取全局唯一任务图元信息。
2. 页面内点击 `+ 创建任务` 后，调用单节点创建接口提交：
   1. 任务名称；
   2. 执行 agent；
   3. 优先级；
   4. 确认任务目标；
   5. 预期产物（选填）；
   6. 上游任务（可选）。
3. 后端落盘：
   1. `<任务产物路径>/tasks/<ticket_id>/task.json`；
   2. `<任务产物路径>/tasks/<ticket_id>/nodes/<node_id>.json`；
   3. `<任务产物路径>/tasks/<ticket_id>/TASK_STRUCTURE.md`；
   4. 任务相关审计与引用文件；
   5. 运行中的内存调度工作集。
4. 客户端读取 `graph/status-detail` 与 `scheduler-state` 接口时，后端从 `任务产物路径` 动态加载任务图、节点详情与执行链路。
5. 用户调用 `dispatch-next`，后端按依赖门禁、优先级、单 agent 单并发和全局并发上限选出可运行任务。
6. 对每个被派发节点，后端按当前执行通道真正启动运行批次，并写入执行链路记录。
7. 用户调用 `pause` 时，后端进入 `pause_pending`，等待当前运行任务结束后不再派发新任务。
8. 真实执行过程中，系统持续回写运行事件、结果与引用；用户仍可手动回写成功、失败、重跑或人工改状态。
9. 用户可对单个任务执行删除，或对全局唯一任务图执行清空。
10. 后端更新任务状态、写审计、重算后继任务是否进入 `ready` 或 `blocked`，并通过图读取接口返回反向下游关系。

## 4. 功能需求清单
1. FR-AB-01 独立业务域
   1. 任务指派必须拥有独立的数据模型和接口命名空间。
   2. 不得直接复用训练队列表作为任务指派主存储。
   3. 不得将任务明文持久化到 `workflow` 自身的 SQLite、`.runtime/` 或 `logs/` 目录。
2. FR-AB-02 全局唯一任务图接口
   1. 后端至少提供：
      1. `POST /api/assignments`
      2. `GET /api/assignments`
      3. `GET /api/assignments/{ticket_id}`
   2. 首期任务中心只允许存在 1 张全局唯一任务图。
   3. 若保留 `POST /api/assignments`，其语义必须是“确保全局唯一任务图存在并返回稳定 `ticket_id`”，不得每次调用都创建新图。
   4. `GET /api/assignments` 首期应返回当前全局唯一任务图概览，或只返回 1 条活动任务图记录，不得暴露多张活动任务图列表。
   5. 全局任务图至少支持字段：
      1. `graph_name`
      2. `source_workflow`
      3. `summary`
      4. `review_mode`
      5. `global_concurrency_limit`
      6. `nodes`
      7. `edges`
      8. `external_request_id`（建议）
3. FR-AB-03 单节点创建接口
   1. 后端至少提供：
      1. `POST /api/assignments/{ticket_id}/nodes`
   2. 页面内创建任务的首期请求体至少支持：
      1. `node_name`
      2. `assigned_agent_id`
      3. `node_goal`
      4. `priority`
      5. `expected_artifact`（选填）
      6. `upstream_node_ids`（选填）
      7. `delivery_mode`
      8. `delivery_receiver_agent_id`（指定交付人时必填）
   3. 当前页面表单不直接提交 `downstream_node_ids`。
   4. `POST /api/assignments/{ticket_id}/nodes` 只能向这张稳定的全局唯一任务图追加节点，不得隐式创建第二张图。
   5. 外部 workflow 或后续编辑接口允许继续使用依赖边或上下游数组方式追加任务关系。
4. FR-AB-04 review_mode 预留
   1. 接口层必须预留 `review_mode` 字段。
   2. 枚举口径保留为：`none / partial / full`。
   3. 首期仅允许 `none` 成功确保全局图。
   4. 若收到 `partial/full`，必须明确返回 `not_supported_in_phase1` 类错误，不得假装支持。
5. FR-AB-05 节点字段契约
   1. 每个任务节点至少包含：
      1. `node_id`
      2. `node_name`
      3. `assigned_agent_id`
      4. `node_goal`
      5. `expected_artifact`（选填）
      6. `status`
      7. `priority`
      8. `completed_at`（选填）
      9. `success_reason`
      10. `result_ref`
      11. `failure_reason`
      12. `delivery_mode`
      13. `delivery_receiver_agent_id`
      14. `artifact_delivery_status`
   2. 后端 `priority` 首期使用 `0/1/2/3` 归一化表示，且 `0` 代表最高优先级。
   3. 页面下拉值 `P0/P1/P2/P3` 必须稳定映射为后端 `0/1/2/3`。
   4. `assigned_agent_id` 必须来自训练中心角色池。
6. FR-AB-06 依赖边与反向下游契约
   1. 每条依赖边至少包含：
      1. `from_node_id`
      2. `to_node_id`
      3. `edge_kind`
   2. 创建或编辑任务时，后端至少支持：
      1. `upstream_node_ids`
      2. `downstream_node_ids`（供外部 workflow 或后续编辑使用）
   3. 后端应将上下游输入统一归一化为依赖边。
   4. 当前页面表单仅传 `upstream_node_ids` 时，后端仍必须能够在读取接口中反向返回下游关系。
   5. 保存时必须校验：
      1. 节点存在；
      2. 不允许自依赖；
      3. 不允许循环依赖。
7. FR-AB-07 图读取接口
   1. 后端至少提供：
      1. `GET /api/assignments/{ticket_id}/graph`
      2. `GET /api/assignments/{ticket_id}/status-detail`
      3. `GET /api/assignments/{ticket_id}/scheduler-state`
   2. `graph` 返回内容至少包含：
      1. `nodes`
      2. `edges`
      3. `metrics_summary`
      4. `priority_rules`
   3. `status-detail` 返回内容至少包含：
      1. `graph_overview`
      2. `selected_node`
      3. `blocking_reasons`
      4. `available_actions`
      5. `audit_refs`
   4. `scheduler-state` 返回内容至少包含：
      1. `state`
      2. `running_agent_count`
      3. `global_concurrency_limit`
      4. `pause_note`
   5. 图读取接口返回的数据应来自 `任务产物路径` 的动态加载结果，并与当前内存调度状态合并。
   6. 当前口径下，图读取接口中的 `ticket_id` 必须稳定指向全局唯一任务图。
8. FR-AB-08 调度与控制接口
   1. 后端至少提供：
      1. `POST /api/assignments/{ticket_id}/dispatch-next`
      2. `POST /api/assignments/{ticket_id}/pause`
      3. `POST /api/assignments/{ticket_id}/resume`
      4. `GET /api/assignments/settings/concurrency`
      5. `POST /api/assignments/settings/concurrency`
   2. `dispatch-next` 可一次推进多个节点，但必须受并发门禁控制。
   3. `pause` 的语义必须是：
      1. 当前运行中的节点继续完成；
      2. 不再派发新的节点；
      3. 状态先进入 `pause_pending`，待运行节点清空后转为 `paused`。
   4. `resume` 必须允许从 `paused / pause_pending / idle` 恢复到 `running`。
9. FR-AB-09 单 agent 单并发
   1. 同一个 `assigned_agent_id` 在任何时刻只能存在 1 个 `running` 节点。
   2. 即使该节点来自不同来源模块，也必须遵守此规则。
10. FR-AB-10 全局并发上限
   1. 系统必须保存一个可配置的全局并发上限。
   2. 当 `running` 节点总数达到上限时，新的就绪节点不得被派发。
11. FR-AB-11 依赖与就绪判定
   1. 节点只有在所有上游节点均为 `succeeded` 时，才可从 `pending` 进入 `ready`。
   2. 若节点不存在上游，则保存后直接进入优先级调度队列。
   3. 若任一上游节点为 `failed`，则该节点必须进入 `blocked`。
12. FR-AB-12 优先级调度规则
   1. 无上游节点的任务必须按 `priority` 进入调度队列。
   2. 数字越小优先级越高，`0` 为最高。
   3. 推断/假设（置信度: 中）：若优先级相同，后端应采用稳定次序继续派发，建议按 `created_at` 升序处理。
13. FR-AB-13 人工修改执行状态接口
   1. 后端至少提供：
      1. `POST /api/assignments/{ticket_id}/nodes/{node_id}/override-status`
   2. 请求至少包含：
      1. `target_status`
      2. `reason`
      3. `result_ref`（选填）
   3. 当 `target_status=succeeded` 时：
      1. `reason` 必填，并写入 `success_reason`；
      2. `result_ref` 选填；
      3. 必须先满足 `artifact_delivery_status=delivered`；
      4. 必须写审计日志。
   4. 当 `target_status=failed` 时：
      1. `reason` 必填，并写入 `failure_reason`；
      2. 必须触发下游节点阻塞重算；
      3. 必须写审计日志。
14. FR-AB-14 重跑任务接口
   1. 后端至少提供：
      1. `POST /api/assignments/{ticket_id}/nodes/{node_id}/rerun`
   2. `rerun` 仅允许对失败节点执行。
   3. 重跑后节点重新进入调度或等待依赖校验。
   4. 必须写审计日志。
15. FR-AB-15 任务管理动作回传
   1. 图读取接口或节点详情接口必须返回当前节点可用的 `management_actions`。
   2. 至少包含：
      1. `override_status`
      2. `rerun`
      3. `delete`
   3. 前端不应依赖额外拆分的动作区块判断动作可用性，统一以 `management_actions` 为准。
16. FR-AB-16 审计日志
   1. 后端必须记录：
      1. 确保全局图；
      2. 创建任务；
      3. 编辑上游/下游；
      4. 编辑优先级；
      5. 派发；
      6. 暂停；
      7. 继续运行；
      8. 人工修改为成功；
      9. 人工修改为失败；
      10. 重跑；
      11. 删除 / 清空；
      12. 并发配置变更。
17. FR-AB-17 任务产物路径与目录结构
   1. 后端必须在 `任务产物路径` 下固定创建 `tasks/` 目录。
   2. 每个任务必须统一写入 `<任务产物路径>/tasks/<ticket_id>/`。
   3. 若保留 `ticket_id`，其首期必须稳定指向全局唯一任务图，而不是每次建单重新生成新的图级标识。
   4. 每个任务目录至少包含：
      1. `task.json`
      2. `nodes/`
      3. `runs/`
      4. `artifacts/`
   5. 每次真实执行完成产物落盘后，后端必须同步刷新：
      1. `<任务产物路径>/TASKS_STRUCTURE.md`
      2. `<任务产物路径>/tasks/<ticket_id>/TASK_STRUCTURE.md`
   6. 任务目录结构应采用“按任务聚合”方式，避免旧口径中的 `workspace/assignments` 与按 agent 散落目录继续并存。
18. FR-AB-18 来源可追溯
   1. 任务图必须记录来源 workflow。
   2. 后续其他 workflow 向全局图追加任务时，应能追溯是哪个 workflow 发起。
19. FR-AB-19 真实执行化升级说明
   1. 从 2026-03-16 起，后端目标已升级为真正触发 agent 自动执行。
   2. `running` 必须表示“已创建运行批次并处于真实执行中”。
   3. provider 抽象、完整提示词可见、执行过程可见与运行批次记录的详细契约，以 `docs/workflow/requirements/需求详情-任务中心真实执行化与执行链路可视化.md` 为准。
20. FR-AB-20 未来修复模式预留
   1. 数据模型需为后续失败修复策略预留扩展位。
   2. 但首期不实现：
      1. 人工指定修复方向 agent；
      2. agent 自动解决。
21. FR-AB-21 清空全局任务图接口
   1. 后端至少提供：
      1. `POST /api/assignments/{ticket_id}/clear`
   2. 该接口作用于全局唯一任务图中的全部活动节点与依赖边。
   3. 若全局唯一任务图中仍存在 `running` 节点，必须拒绝清空。
   4. 清空成功后，必须写入批量清空审计。
22. FR-AB-22 单任务删除接口
   1. 后端至少提供：
      1. `DELETE /api/assignments/{ticket_id}/nodes/{node_id}`
   2. 若目标节点为 `running`，必须拒绝删除。
   3. 删除成功后，必须同步摘除与该节点相关的依赖边。
   4. 若该节点同时存在上游和下游，后端必须按“摘除中间节点”的语义自动桥接其原有上游与下游。
   5. 推断/假设（置信度: 中）：当删除节点存在多个上游与多个下游时，后端应补齐有效桥接组合，并去重、拒绝形成循环。
   6. 删除成功后，必须重算受影响节点的上下游关系与调度状态。
23. FR-AB-23 删除留痕与任务记录处理
   1. `清空任务` 与 `删除单个任务` 不得静默物理抹除审计。
   2. `任务产物路径/tasks/<ticket_id>/` 中的对应任务记录必须保留删除标记或等效留痕。
   3. 推断/假设（置信度: 中）：首期可采用逻辑删除或状态标记，而不是物理删除任务记录文件。
24. FR-AB-24 workflow 本地无任务明文持久化
   1. `workflow` 仅允许在内存中保留运行中的调度工作集、并发锁与临时状态。
   2. 任务名、任务目标、依赖图、完整提示词、stdout/stderr、结果正文与任务审计明文，均不得持久化到 `workflow/state/`、`workflow/.runtime/`、`workflow/logs/`。
   3. 服务重启后，如需恢复任务页面展示，应通过 `任务产物路径` 重新扫描与动态加载，而不是依赖 `workflow` 本地持久化副本。

## 5. 非功能需求
1. 一致性：
   1. 所有依赖门禁、优先级门禁和并发门禁必须以后端判定为准。
   2. 页面、缺陷模块、定时任务模块与外部 workflow 都必须读写同一张全局唯一任务图。
   3. 页面 `P0/P1/P2/P3` 与后端 `0/1/2/3` 的映射必须稳定且可测试。
2. 可靠性：
   1. 同一节点不得因重复点击或重复请求重复进入 `running`。
3. 可追溯性：
   1. 任一节点状态变化必须可追溯到操作接口、操作者和理由。
   2. 任一任务记录都必须可追溯到 `任务产物路径/tasks/<ticket_id>/` 中的稳定位置。
4. 幂等性：
   1. 外部 workflow 确保全局图或向全局图追加任务时，建议支持 `external_request_id` 幂等保护。
5. 安全性：
   1. 外部 workflow 向全局图追加任务的范围仅限 `agent_search_root` 管控范围内的角色池。
6. 隐私边界：
   1. `workflow` 页面可展示任务明文，但其持久化落点只能是用户本地 `任务产物路径`。

## 6. 验收标准
1. AC-AB-01 API 确保全局图成功
   1. Given 调用 `POST /api/assignments`
   2. When 首次初始化或重复调用该接口
   3. Then 后端返回稳定的全局图 `ticket_id`
   4. And 不会因此创建第二张活动任务图
2. AC-AB-02 review_mode 首期限制
   1. Given 调用全局图接口并传入 `review_mode=partial` 或 `review_mode=full`
   2. When 请求提交
   3. Then 后端明确返回当前阶段不支持，而不是静默降级
3. AC-AB-03 单节点创建字段映射
   1. Given 页面调用 `POST /api/assignments/{ticket_id}/nodes`
   2. When 请求中传入 `priority=P1` 等页面语义值
   3. Then 后端必须稳定归一化为内部优先级值
   4. And `expected_artifact` 为空时不构成拒绝理由
4. AC-AB-04 当前页面只传上游
   1. Given 页面创建任务时仅传入 `upstream_node_ids`
   2. When 任务保存成功
   3. Then 后端成功生成依赖边
5. AC-AB-05 任务目录落盘
   1. Given 系统已配置任务产物路径
   2. When 确保全局图或创建任务成功
   3. Then 后端在 `<任务产物路径>/tasks/<ticket_id>/` 下写入对应任务记录
   4. And 后续图读取接口可反向返回该节点的下游关系
6. AC-AB-06 循环依赖校验
   1. Given 节点依赖形成环
   2. When 提交追加节点或编辑依赖
   3. Then 后端拒绝保存并返回循环依赖错误
7. AC-AB-07 无上游任务优先级调度
   1. Given 两个无上游任务分别为 `priority=0` 和 `priority=1`
   2. When 调用 `dispatch-next`
   3. Then 后端必须优先派发 `priority=0` 的任务
8. AC-AB-08 单 agent 单并发
   1. Given 同一 agent 已有一个 `running` 节点
   2. When 再次派发分配给该 agent 的就绪节点
   3. Then 后端拒绝派发第二个节点
9. AC-AB-09 全局并发上限
   1. Given 当前 `running` 节点数已达到全局并发上限
   2. When 调用 `dispatch-next`
   3. Then 后端不得继续推进新的节点进入 `running`
10. AC-AB-10 暂停语义正确
   1. Given 系统当前存在运行中的节点
   2. When 调用 `pause`
   3. Then 调度状态先进入 `pause_pending`
   4. And 当前运行节点完成前不被中断
   5. And 当前运行节点完成后状态转为 `paused`
11. AC-AB-11 人工修改成功态理由必填
   1. Given 节点通过 `override-status` 改为成功
   2. When 请求中缺少 `reason`
   3. Then 后端拒绝提交
   4. And `result_ref` 为空时不构成拒绝理由
12. AC-AB-12 失败阻塞下游
   1. Given 某节点存在下游节点
   2. When 当前节点通过 `override-status` 被改为失败
   3. Then 下游节点被重算为 `blocked`
13. AC-AB-13 任务管理动作可追溯
   1. Given 某节点在 `任务管理` 中触发 `rerun` 或 `override-status`
   2. When 请求提交
   3. Then 后端写入动作审计
   4. And 图读取接口返回更新后的状态与可用 `management_actions`
14. AC-AB-14 审计可追溯
   1. Given 节点状态发生变化
   2. When 查询该节点的审计信息
   3. Then 可看到操作者、动作、理由、时间和引用
15. AC-AB-15 清空全局任务图受控
   1. Given 全局唯一任务图仍存在 `running` 节点
   2. When 调用 `POST /api/assignments/{ticket_id}/clear`
   3. Then 后端拒绝清空并返回明确错误
   4. And 当无 `running` 节点时，后端成功清空活动节点与依赖边并写入批量清空审计
16. AC-AB-16 单任务删除受控
   1. Given 全局唯一任务图中存在一个非运行中的节点
   2. When 调用 `DELETE /api/assignments/{ticket_id}/nodes/{node_id}`
   3. Then 后端成功删除该节点并同步摘除相关依赖边
   4. And 若该节点位于依赖链中间，则后端自动桥接其原有上游与下游
   5. And 图读取接口返回更新后的上下游关系
17. AC-AB-17 删除留痕保留
   1. Given 任务被单删或任务图被清空
   2. When 查询任务审计与对应任务目录
   3. Then 可以看到删除动作与删除状态留痕
18. AC-AB-18 真实执行后刷新目录结构说明
   1. Given 系统已配置任务产物路径且某任务完成一次真实执行并完成产物落盘
   2. When 查询 `<任务产物路径>/` 与 `<任务产物路径>/tasks/<ticket_id>/`
   3. Then 后端已生成或刷新最新目录结构说明
   4. And 说明中能区分 `task.json`、`runs/` 与 `artifacts/` 的用途
19. AC-AB-19 workflow 本地目录无任务明文
   1. Given 已完成全局图初始化、执行、回写与调度展示
   2. When 检查 `workflow/state/`、`workflow/.runtime/` 与 `workflow/logs/`
   3. Then 不应看到任务明文的持久化副本
   4. And 当前任务页面仍可通过 `任务产物路径` 动态加载完整任务信息

## 7. 边界条件与异常处理
1. 若 `assigned_agent_id` 不在当前角色池：
   1. 拒绝向全局图追加任务或更新任务。
2. 若页面传入未识别的优先级标签：
   1. 必须返回明确映射错误，不得静默兜底。
3. 若多个无上游任务优先级相同：
   1. 推断/假设（置信度: 中）：后端应按稳定次序继续派发，建议按 `created_at` 升序处理。
4. 若某节点被重复改为成功态：
   1. 应返回幂等成功或显式拒绝重复变更，不得产生双重审计污染。
5. 若外部 workflow 使用非契约字段调用：
   1. 必须返回明确字段错误，不得静默丢弃关键字段。
6. 若并发配置被更新：
   1. 已在 `running` 的节点继续运行；
   2. 新派发受新配置约束。
7. 若 `override-status` 试图把存在未完成上游的节点直接改为 `running`：
   1. 首期建议拒绝，并返回依赖门禁错误。
8. 若删除节点后的桥接将产生重复边或循环边：
   1. 后端必须跳过非法桥接，并返回或记录明确的桥接处理结果。
9. 若 `workflow` 服务重启导致内存调度工作集丢失：
   1. 系统应通过 `任务产物路径` 重新扫描并恢复页面可见的任务图与执行链路信息。

## 8. 依赖项与开放问题
1. 依赖项：
   1. 依赖训练中心角色注册表作为执行 agent 候选来源。
   2. 依赖 `任务产物路径` 具备稳定的本地文件读写能力，支持按任务目录动态加载。
2. 开放问题：
   1. 首期是否提供任务改派接口。
      1. 推断/假设（置信度: 中）：建议仅允许未运行任务改派。
   2. 外部 workflow 是否要求批量追加节点。
      1. 推断/假设（置信度: 中）：首期可先不做批量，逐节点追加到全局唯一任务图即可。
