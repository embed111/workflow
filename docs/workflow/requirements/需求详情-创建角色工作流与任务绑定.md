# 需求详情-创建角色工作流与任务绑定

- 状态：正式需求（当前有效口径）
- 对应原型：`docs/workflow/prototypes/创建角色工作流-预稿/创建角色工作流预稿.html`

## 1. 主题目标与价值说明
1. 主题目标：
   1. 在 `训练中心` 内新增 `创建角色` 子模块，用分析引导式对话取代单次表单式录入。
   2. 允许用户通过类似聊天的消息草稿，同时携带文本描述与图片示例来表达“想要什么角色”。
   3. 在用户完成描述后，将角色草案结构化为 `role_spec`，并基于该草案自动生成“创建角色工作流”关联任务。
   4. `创建角色` 页面只展示与当前角色创建工作流相关的任务引用，不展示任务中心的全部任务。
   5. 为角色新增与版本生命周期并行的运行状态轴 `runtime_status`，明确创建期、训练期、执行期与空闲期的任务门禁。
2. 业务价值：
   1. 把“角色需求表达 -> 多轮画像调整 -> 创建任务生成 -> 创建过程追踪 -> 能力验收 -> 决定进入下一轮或收敛”收口为一条连续业务流。
   2. 避免 `训练中心`、`创建角色`、`任务中心` 分别维护多套任务真相源。
   3. 让角色创建过程可追踪、可留痕、可恢复，而不是只靠一次性表单或口头描述。
   4. 让新角色从初始化开始就具备与当前工作区同类的连续记忆能力，而不是后续再补记忆机制。
   5. 为后续把 `训练功能` 改造成同类“业务页引用任务底座”的模式建立模板。

## 2. 用户画像与使用场景
1. 用户画像 A：角色需求提出者
   1. 关注点：能像对话一样逐步说清楚角色目标、能力、边界与风格，不想一次性填满复杂表单。
2. 用户画像 B：workflow 操作者
   1. 关注点：创建角色后能直接看到当前工作流推进到了哪一步、卡在哪个任务，不需要切到全局任务中心才能理解进度。
3. 用户画像 C：平台治理者
   1. 关注点：角色的“版本状态”和“运行占用状态”必须分离，避免状态机混乱。
4. 核心场景：
   1. 用户进入 `训练中心 -> 创建角色`，通过对话输入角色目标、能力、边界、协作方式和示例图片。
   2. 系统将这些输入收口为结构化 `role_spec`。
   3. 用户启动角色创建工作流后，系统在任务中心生成一组关联任务。
   4. `创建角色` 页仅展示当前角色的创建任务流图、阶段、当前卡点和必要动作。
   5. 所有创建任务完成后，角色自动从 `创建中` 回到 `空闲`。

## 3. 用户旅程或关键流程
1. 用户进入 `训练中心`，切换到 `创建角色` 子模块。
2. 页面默认展示：
   1. 左侧：更接近工作聊天产品的角色创建会话列表，展开态优先采用“头像 / 名称 / 最近消息预览 / 时间 / 当前阶段标签”的列表语法，并支持类似 `<< / >>` 的横向折叠与展开；
   2. 中部：外层 `增长分析师` 面板本身就是单一聊天工作面，直接承载对话历史、系统任务更新消息与底部固定输入区（支持图片+文字同发），不再内嵌第二层聊天卡；
   3. 右侧：随聊天与 `role_spec` 调整联动的辅助区，默认只保留一张统一阶段与任务演进图，并在图内直接表达当前阶段、当前任务、状态真相源与回退路径；
   4. 默认将中部与右侧按约 `6 / 4` 分配主工作台宽度，中部作为主阅读区。
3. 用户通过对话逐步提供：
   1. 角色名称；
   2. 角色目标；
   3. 核心能力；
   4. 禁止边界；
   5. 适用场景；
   6. 协作方式/输出风格；
   7. 可选图片示例。
4. 系统识别描述内容并实时生成结构化角色草案 `role_spec`。
5. `role_spec` 不视为单轮冻结产物，而应支持按轮次迭代调整。
6. 当系统判断当前轮信息已足够时，页面进入 `启动创建` 阶段，展示：
   1. 当前结构化角色草案；
   2. 预计生成的创建任务摘要；
   3. 当前仍缺失或不清晰的信息（若有）。
7. 默认采用 6 阶段模型：
   1. 阶段 1：初始化工作区
      1. 作为前置条件存在；
      2. 由脚本一次性完成当前角色工作区结构、记忆骨架与接口接入点；
      3. 默认不挂任务中心任务。
   2. 阶段 2：角色画像收集
      1. 当前对话 agent 通过分析引导而不是模板式提问，帮助用户说出真正想要的角色；
      2. 该阶段可创建后台任务，例如资料收集整理、画像沉淀入工作区。
   3. 阶段 3：能力后台生成
      1. 当前对话 agent 引导被创建角色开始生成具体能力；
      2. 该阶段更偏后台生产与样例产出，用户可继续通过对话随时控制方向，但不以用户逐项回看为主。
   4. 阶段 4：结果回看与调向
      1. 当某个能力已形成可交付结果时，当前对话 agent 将截图或等效预览发到会话窗口；
      2. 该阶段以用户回看结果、确认方向、要求补强或重做为主，而不是继续承载主要生成动作。
   5. 阶段 5：确认验收
      1. 当当前对话 agent 判断所有关键能力都已满足要求时，进入确认验收阶段；
      2. 当前对话 agent 明确询问用户是否满足验收标准。
   6. 阶段 6：完成角色创建
      1. 仅当用户明确给出肯定验收结论后，当前对话 agent 才调用后端接口完成角色创建。
8. 用户点击 `开始创建` 后：
   1. 系统创建临时角色记录；
   2. 角色 `runtime_status` 变为 `创建中`；
   3. 系统完成阶段 1 的工作区初始化前置动作；
   4. 系统在任务中心生成从阶段 2 开始所需的任务与依赖关系；
   5. `创建角色` 页开始按“仅当前角色”范围引用并展示关联任务。
9. 创建过程中：
   1. `创建角色` 页只展示当前角色创建工作流关联任务；
   2. 关键任务变化可作为系统消息插入当前聊天流，例如创建后台资料任务、切换当前阶段、回传能力截图、进入确认验收；
   3. 用户可在右侧一体图与任务浮层详情卡中查看当前阶段、失败原因与执行证据摘要；
   4. 当前对话 agent 可通过后端接口读取当前阶段、查看角色画像、创建后台任务、调整当前工作阶段；
   5. 若用户在对话中明确表达“另起一个任务去…… / 单独起个任务…… / 后台去跑…… / 去整理……并回传”等独立委派语义，系统应优先在 `任务中心` 创建对应后台任务，而不是只把它当作当前分析师的即时动作；
   6. 若需要查看全局任务图，可跳转到 `任务中心`，并自动带上当前角色/当前任务图过滤条件。
10. 当某一轮能力验收完成：
   1. 若验收结论为“能力仍不足，需要继续补能力”，角色继续保持 `创建中`，并回到阶段 2 或阶段 3 继续画像收集/能力生成；
   2. 若验收结论为“已达到当前阶段可用门槛，且用户明确确认通过验收”，系统调用完成接口并将该角色 `runtime_status` 自动切为 `空闲`；
   3. 角色的 `lifecycle_status` 仍按既有发布治理规则独立维护。

## 4. 功能需求清单（编号）
1. FR-RC-01 `训练中心` 新增 `创建角色` 子模块
   1. `训练中心` 顶部子模块至少调整为：
      1. `角色资产与版本`
      2. `创建角色`
      3. `训练闭环`
   2. `创建角色` 当前作为 `训练中心` 内业务子模块存在，不新增左侧独立顶层模块。
2. FR-RC-02 对话式角色需求收集
   1. `创建角色` 页必须以对话式收集为主入口，而不是一次性静态表单。
   2. 页面至少支持两类输入：
      1. 文本描述；
      2. 图片示例。
   3. 页面必须按轮次保留用户已输入的关键信息与系统追问结果。
   4. 页面必须显式展示真实可操作的对话输入入口，而不是只展示历史消息示意。
   5. 对话输入入口必须在当前主工作台首屏显式可见，视觉上与历史消息区分，不得藏在历史消息块尾部。
   6. 当角色已进入 `创建中` 后，固定输入条仍需保留，允许继续补充文本与图片示例，而不是在启动后直接消失。
   7. 默认采用“聊天草稿”模型：
      1. 图片先进入当前消息草稿；
      2. 用户可继续输入文字；
      3. 发送时图片与文字作为同一条消息一起发出。
   8. 默认支持三种加图方式：
      1. 粘贴图片；
      2. 拖拽图片到输入区；
      3. 从系统选图。
   9. 默认输入入口至少包含：
      1. 文本输入框；
      2. 待发送图片预览区；
      3. `发送` 动作；
      4. `开始创建` 动作。
   10. 页面不要求提供单独的“上传图片”主按钮；图片能力应尽量收口在聊天输入区内部。
   11. 默认采用“单聊天壳”布局：
      1. 对话历史；
      2. 系统任务更新消息；
      3. 底部固定输入区；
      4. 均属于同一会话容器，不再拆成上下两个独立卡片；
      5. 外层角色对象面板即聊天工作面本体，不再在其内部额外嵌套第二层聊天框。
   12. 对话中出现的关键任务变化应允许以系统消息形式插入聊天流，而不是只在右侧辅助区静默变化。
   13. 终态中栏应保持高纯度聊天：
      1. 默认不堆叠解释性辅助文案；
      2. 默认不重复展示阶段/状态说明；
      3. 结构化说明、流程解释与门禁信息优先放到右侧辅助区或按需展开。
   14. 当前对话 agent 默认应由当前工作区的分析 agent 承担；若当前工作区不存在该 agent，则回退到 `workflow` 的默认 agent。
   15. 当前对话 agent 的职责是“分析引导 + 顶层角色信息整合”，而不是把用户输入机械塞进模板。
3. FR-RC-03 结构化角色草案 `role_spec`
   1. 系统必须将用户输入收口为结构化角色草案。
   2. 当前草案至少包含：
      1. `role_name`
      2. `role_goal`
      3. `core_capabilities`
      4. `boundaries`
      5. `applicable_scenarios`
      6. `collaboration_style`
      7. `example_assets`
   3. `role_spec` 必须支持按轮次迭代，不得假设第一轮画像即可完成全部能力设计。
   4. 当关键信息不足时，系统必须明确指出缺失项，而不是直接生成创建任务。
4. FR-RC-04 启动创建门禁
   1. 当前方案默认保留显式 `开始创建` 动作。
   2. 仅当 `role_spec` 达到最小可执行要求时，才允许触发角色创建工作流。
   3. 推断/假设（置信度：中）：
      1. 当前阶段用显式启动动作比“系统自动开始创建”更稳，避免对未收口角色描述误生成任务。
5. FR-RC-05 创建角色工作流一体图生成
   1. 用户启动创建后，系统必须在任务中心生成当前角色专属的创建任务图。
   2. 创建任务图至少支持：
      1. 阶段层；
      2. 阶段下挂任务节点；
      3. 上下游依赖；
      4. 当前阶段；
      5. 任务状态；
      6. 图内当前任务提示与状态锚点。
   3. 当前阶段建议至少覆盖：
      1. 初始化工作区
      2. 角色画像收集
      3. 能力后台生成
      4. 结果回看与调向
      5. 确认验收
      6. 完成角色创建
   4. 当前角色右侧的流程主组件应采用“统一阶段与任务演进图”，而不是“独立阶段流 + 独立任务图”重复表达。
   5. 当前角色关联任务图的展示形态应直接继承训练中心现有“训练演进图”表达，而不是普通列表或直线卡片流。
   6. 图中至少需要同时表达四类语义：
      1. 当前共有哪 6 个阶段；
      2. 当前主线已推进到哪个阶段；
      3. 每个阶段下当前挂着什么任务；
      4. 当能力验收结论为“仍需补能力”时，应从哪里分叉回到下一轮角色画像调整。
   7. 阶段 1 `初始化工作区` 默认作为前置条件存在：
      1. 由脚本完成；
      2. 默认不挂任务中心任务；
      3. 其结果应作为后续阶段可运行的前置基础。
   8. 图中以“任务卡”形式展示的节点必须全部来自任务中心任务引用，不得混入本地伪任务。
   8.1. 由当前分析师直接完成的动作，例如分析引导、顶层 `role_spec` 整合、结果回传、确认验收问答、完成接口调用，不得伪装成任务中心任务卡：
      1. 应以阶段动作提示、系统消息或状态锚点表达；
      2. 不参与 `task_id` 引用与任务状态映射。
   9. 图中任务节点状态必须同步映射任务中心中的真实运行状态，至少包含：
      1. 已完成；
      2. 运行中；
      3. 待开始；
      4. 阻塞。
   10. 页面需明确提示“当前图仅为当前角色创建工作流的投影视图，其状态真相源来自任务中心”。
   11. 右侧辅助区默认只保留一张统一阶段与任务演进图；当前任务提示、状态真相源与回退路径作为图内信息随聊天消息与 `role_spec` 变化增量刷新。
   12. 默认将该组件收口为更接近既有“任务展示图”的画布语法：
      1. 左侧使用 stage gutter 表达 6 个阶段；
      2. 右侧在同一连续画布中展示任务节点、依赖线与回退路径；
      3. 不再采用“6 列均匀悬挂任务卡片”的主视觉。
   13. 右侧辅助区默认采用标签页切换：
      1. `阶段与任务演进`
      2. `角色画像`
   14. 用户在聊天过程中可随时切换查看当前阶段任务或当前角色画像，不要求离开当前创建会话。
   15. 任务流程默认应更强调阶段推进感，而不是散点分布：
      1. 阶段顺序需一眼可读；
      2. 每个任务默认贴齐所属阶段；
      3. 当前任务文字在默认视图内必须清晰可读，不依赖缩放后辨认。
      4. 若纵向阶段主线已承担阶段定位语义，则不再额外保留一条横向阶段概览，避免重复表达。
      5. 当前分析师动作与 `任务中心` 任务在视觉语义上必须可区分，不得都画成同一类任务卡。
      6. 任务卡默认采用两档固定尺寸，而不是随内容长度明显波动：
         - 当前阶段任务卡一档；
         - 非当前阶段任务卡一档。
      7. 当前分析师动作应保持更强互动感，至少可见动作标题、动作说明与下一步动作提示，不应退化成纯静态说明段。
      8. 若某阶段累积了多条已作废/已归档任务，默认不在主线平铺全部废案，而是收口为该阶段下的“废案收纳”入口，只展示数量与阶段归属。
      9. “废案收纳”入口的详情默认通过悬浮/固定详情卡查看，至少可见废案 `task_id`、任务名、关闭原因与去任务中心查看入口。
6. FR-RC-06 `创建角色` 页只引用关联任务
   1. `创建角色` 页不得直接展示任务中心全量任务。
   2. 页面默认只展示当前角色创建工作流关联任务。
   3. 若用户需要全局视角，页面必须提供跳转到 `任务中心` 的入口，并自动附带当前角色/当前任务图过滤条件。
7. FR-RC-07 任务中心作为唯一任务真相源
   1. 当前角色创建工作流的任务状态、执行证据、运行结果与失败原因，必须以任务中心及其关联任务产物路径为准。
   2. `创建角色` 页面只做引用展示，不得私自维护第二套任务状态真相源。
   3. 当前方案中的具体任务卡必须显式表达“该任务已由任务中心创建，本页仅为引用投影”：
      1. 至少展示 `task_id` 或等效任务引用标识；
      2. 至少展示“任务中心引用”来源语义；
      3. 点击动作语义应指向任务中心中的同一任务，而不是本页局部副本。
   4. 阶段 1 的脚本前置动作应明确标注为“非任务中心任务”，与阶段 2 及之后的真实任务引用区分开。
   5. 当前方案可通过悬浮详情卡承接更完整的任务信息：
      1. 来源；
      2. 上下游关系；
      3. 证据/产物摘要；
      4. 跳转任务中心提示；
      5. 详情卡必须在当前辅助区内直接可见，不得因为滚动容器裁剪而不可读；
      6. 详情卡应采用不挤动任务轨道布局的浮层样式，允许用户悬停预览、点击固定后稳定查看。
   6. 是否创建 `任务中心` 任务，默认按“后台执行真相源”判定，而不是按“这件事是否重要”判定：
      1. 需要后台执行、可独立指派、需要重试/排队、需要保留状态与产物证据的，创建 `任务中心` 任务；
      2. 仅由当前分析师在当前会话内直接完成的引导、判断、整合与放行动作，不创建 `任务中心` 任务。
   7. 当前分析师默认直接承担以下动作，不进入 `任务中心`：
      1. 分析引导与追问；
      2. 顶层 `role_spec` 整合；
      3. 判断画像是否足够清晰；
      4. 判断是否需要回退或进入下一阶段；
      5. 把后台产物回传给用户并引导回看；
      6. 发起确认验收问答；
      7. 用户确认后调用完成角色创建接口。
   8. 以下类型默认进入 `任务中心`：
      1. 资料收集整理；
      2. 用户明确要求“另起一个任务去……”的资料整理、资料沉淀、补能力或样例生成任务；
      3. 单项能力生成或补强任务；
      4. 需要其他 agent 在后台运行并回传产物的任务；
      5. 为结果回看服务的产物整理、截图生成或等效产物打包任务。
   9. 若用户在对话中使用“另起一个任务去…… / 单独起个任务…… / 后台去跑……”这类明确委派语义，系统应优先创建 `任务中心` 任务，并由当前分析师继续承担协调、回传与验收动作。
   10. 阶段 5 `确认验收` 与阶段 6 `完成角色创建` 默认属于当前分析师动作，而不是 `任务中心` 任务；只有当用户明确要求把其中某个子工作拆成独立后台任务时，才允许进入 `任务中心`。
   11. `任务中心` 任务名默认应直接反映当前后台目标，不得长期使用“资料整理 / 能力补强 / 结果回传”这类过泛名称：
      1. 更优先使用“收集行业案例资料 / 生成问题树与漏斗样例 / 补行业实验排期资料”这类具体命名；
      2. 允许不同轮次由运行中的分析师生成不同任务名，不要求固定为同一组模板任务。
   12. 默认卡面不再单独占位展示“去任务中心查看”动作；该跳转应收口在悬浮详情卡中，避免默认态信息噪音。
   13. 已作废/已归档的后台任务仍应保留在 `任务中心` 作为真实历史记录，不得因为当前页面要降噪而直接删除或前端伪装消失。
   14. `创建角色` 页默认按阶段把这类任务收口到“废案收纳”入口，而不是继续占用主线任务位；只有用户主动悬浮、点击固定或跳转任务中心时，才展开查看详细废案列表。
   15. 当用户明确表达“这个先不用了 / 作废这个方向 / 别继续这个任务”且对应后台任务已经创建时，系统应把该任务更新为已作废或已归档，并移动到所属阶段的废案收纳入口，而不是只在聊天里口头忽略。
8. FR-RC-08 新增并行 `runtime_status` 状态轴
   1. 在现有 `lifecycle_status`（`发布态 / 预发布态 / 禁训态`）之外，新增并行 `runtime_status`。
   2. `runtime_status` 固定包含：
      1. `空闲`
      2. `创建中`
      3. `训练中`
      4. `执行中`
   3. 两条状态轴必须并行展示，不得互相替换。
9. FR-RC-09 `runtime_status` 运行门禁
   1. 当角色处于 `执行中` 时，不得再被指派新普通任务。
   2. 当角色处于 `训练中` 时，不得再被指派新普通执行任务。
   3. 当角色处于 `创建中` 时：
      1. 不得被普通任务中心继续指派无关任务；
      2. 仅允许 `创建角色` 模块继续向该角色追加创建相关任务。
   4. 当角色处于 `空闲` 时，才允许恢复普通任务接单能力。
10. FR-RC-10 能力验收后的状态跳转
   1. 当当前轮结果回看与调向或阶段验收结论为“仍需继续补能力”时，角色必须继续保持 `创建中`，并回到阶段 2 或阶段 3 继续画像收集/后台生成。
   2. 当某个单项结果回看通过时，只代表该能力可继续推进，不得直接视为该角色已完成创建。
   3. 仅当所有关键能力均达到要求，且用户在阶段 5 明确确认满足验收标准后，系统才允许调用完成接口并将该角色 `runtime_status` 切回 `空闲`。
   4. 上述动作不得自动改写该角色的 `lifecycle_status`。
11. FR-RC-11 创建失败与未完成状态处理
   1. 当任一创建任务失败或未完成时，角色必须继续保持 `创建中`。
   2. 页面必须明确展示：
      1. 当前失败/阻塞任务；
      2. 失败原因或阻塞来源；
      3. 下一步动作（重试/补充描述/跳转任务中心）。
12. FR-RC-12 创建中角色的可见性
   1. 角色一旦进入创建工作流，必须可在 `角色资产与版本` 或等效列表中被识别为“创建中”。
   2. 页面需明确区分：
      1. 当前角色是否已可用；
      2. 当前角色是否仍在创建流程中；
      3. 当前角色的版本生命周期状态。
13. FR-RC-13 图片示例展示
   1. 用户加入到消息草稿中的图片示例，必须在发送前以待发送预览的方式可见。
   2. 图片与文字发送后，必须以同一条消息内容的一部分出现在当前对话区中。
   3. 图片示例在当前阶段仅承担“示例参考”语义，不直接等同于最终资产。
14. FR-RC-14 创建角色页的流程展示形态
   1. `创建角色` 页必须采用与训练闭环一致的“统一阶段与任务演进图”表达方式，并把当前任务提示收口到图内。
   2. 阶段信息应直接整合到任务演进图中，不再独立重复展开一块完整阶段流。
   3. 不得退化为纯长表单，也不得只给出普通列表式任务引用。
   4. 任务图主视觉应优先继承既有任务中心任务图的读图方式，而不是重新发明独立的六列流程卡片语法。
   5. 当前方案中，阶段信息应优先通过左侧 gutter / 阶段标记进入任务图画布，而不是作为并列大卡片占据主视觉。
   6. 中栏当前会话对象名在主阅读区内应只保留一层主标题，不得同时以“面板名 + 会话名”两层重复指向同一角色对象。
15. FR-RC-15 证据与留痕
   1. 创建角色工作流至少可追溯到：
      1. 用户原始输入
      2. 图片示例引用
      3. 结构化 `role_spec`
      4. 创建任务图
      5. 当前任务执行结果与证据摘要
      6. 能力验收结论
      7. 角色记忆机制初始化证据
16. FR-RC-16 新角色完整沿用记忆机制
   1. 新初始化的角色工作区必须完整沿用当前工作区的 `.codex` 记忆机制，而不是仅保留占位文件。
   2. 角色工作区初始化结果至少包含：
      1. `.codex/MEMORY.md`
      2. `.codex/memory/全局记忆总览.md`
      3. `.codex/memory/YYYY-MM/记忆总览.md`
      4. `.codex/memory/YYYY-MM/YYYY-MM-DD-每日记忆.md`
   3. 角色工作区必须同时具备与当前工作区同类的记忆维护能力，包括但不限于：
      1. 日切/月切维护脚本；
      2. 当轮记忆追加脚本；
      3. 启动恢复时的固定读取顺序。
   4. 角色初始化完成判定不得只检查“文件存在”，还应检查该记忆链路在新角色工作区内可被实际使用。
17. FR-RC-17 当前对话 agent 的后端接口能力
   1. 当前对话 agent 必须可通过后端接口：
      1. 查看当前角色所处阶段；
      2. 查看当前角色画像；
      3. 查看当前阶段下挂任务与状态；
      4. 创建后台任务；
      5. 调整当前工作阶段。
   2. 当前对话 agent 只负责顶层角色信息整合与阶段推进判断，不直接承担所有后台执行。
18. FR-RC-18 结果回看与用户方向控制
   1. 当某个能力达到可回看标准时，当前对话 agent 必须将截图或等效预览回传到当前会话窗口。
   2. 用户在阶段 3 与阶段 4 可随时继续与当前对话 agent 对话，要求改方向、补能力、重做结果或补资料。
   3. 当前对话 agent 可基于用户反馈调用后端接口，把流程切回阶段 2 或阶段 3。
19. FR-RC-19 用户确认后完成角色创建
   1. 当当前对话 agent 判断所有关键能力均达到要求时，必须进入阶段 5 发起确认验收问答。
   2. 仅当用户明确回答满足验收标准时，当前对话 agent 才允许调用后端接口完成角色创建。
   3. 未收到用户明确肯定答案前，不得提前完成角色创建。
20. FR-RC-20 左侧角色列表横向折叠展开
   1. 左侧 `角色创建草稿 / 进行中角色` 列表必须支持类似 `<< / >>` 的横向折叠与展开。
   2. 折叠后左栏应缩成窄栏，仅保留可一眼识别的入口、数量或当前进行态提示；展开后恢复完整角色列表与新建入口。
   3. 展开态列表默认应更接近聊天客户端会话列表，而不是信息卡片墙：
      1. 每项优先展示头像、角色名、最近消息预览、最近时间与当前阶段/状态标签；
      2. 不默认堆叠大段说明、重复标题或卡片化说明区。
   4. 折叠与展开不应影响当前选中角色、中部会话状态与右侧进度状态。
   5. 默认可采用“展开态作为主查看态、折叠态作为让位动作”的策略，以保证聊天区在需要时获得更多横向空间。
   6. 折叠控件应贴附在左栏与主工作台的分界边缘，并采用矢量 chevron 图标语义，不再使用位于栏内顶部的普通文本按钮。
   7. 折叠后的最小态应保持稳定可读：
      1. 不得出现仅剩细长标签或视觉重心失衡的临时态；
      2. 应优先保留当前会话指示、会话数量或最小头像栈等可识别信息。

## 5. 非功能需求（性能、安全、可用性、合规等）
1. 可理解性：
   1. 用户进入 `创建角色` 页后，应在 3 秒内理解：
      1. 当前是否仍在收集描述；
      2. 当前角色创建推进到哪一步；
      3. 当前卡在哪个任务；
      4. 下一步应该做什么。
   2. 默认视口适配：
      1. 在当前默认窗口尺寸下，主阅读区与右侧当前阶段任务应尽可能完整进入首屏；
      2. 不得依赖大段解释文字挤占首屏高度。
2. 一致性：
   1. `创建角色` 页的布局、色彩、滚动方式与分栏逻辑必须继承现有 `workflow` 工作台视觉语言。
   1.1. 当前方案对聊天体验的调整应优先收敛在 `创建角色` 工作台内部，不改写左侧顶层模块切换栏与 `训练中心` 大标题区的既有语义和主视觉。
   1.2. 默认遵循“少字原则”：
      1. 对当前使用无帮助的信息不在主界面默认展示；
      2. 解释性文字优先放到鼠标悬浮详情或需求文档；
      3. 默认卡面只保留当前阶段、当前任务、任务来源、关键状态与必要动作。
   2. 中部必须保持为单一聊天壳；对话历史、系统任务更新和底部输入区不得拆成多个割裂卡片。
   2.1. 外层角色对象面板与聊天内容需融为同一连续工作面，不再出现“面板里再套一个独立聊天面板”的双层框感。
   3. 右侧辅助区需默认收口为单一统一阶段与任务演进图，不再拆出第二块任务详情卡或门禁卡；必要状态说明应直接内嵌在图内，并与中部聊天保持同一主工作台语义。
   3.1. 当前方案允许在同一右侧辅助区内，通过标签页切换“阶段与任务演进”与“角色画像”两类事实视图；但不得再次拆成并列多块主卡片。
   4. 配色应优先采用扁平化层级，优先使用纯色底、细边框与轻状态色，减少大面积渐变、重投影和高饱和底色，避免主工作台出现过强的浮层感。
   5. 输入模型应更接近聊天产品，而不是带独立上传按钮的表单控件区。
   6. 中栏信息密度应接近飞书聊天：默认只保留消息列表、少量系统消息和底部输入区，不得把需求说明、流程解释和重复状态堆进聊天区。
      1. 普通消息优先采用工作聊天气泡结构：
         - agent / assistant 消息位于左侧；
         - 用户消息位于右侧；
         - 系统任务更新维持居中轻提示样式。
      2. 系统任务更新优先采用分割线式或轻提示式系统消息，而不是与普通正文消息同权重的大卡片。
      3. 底部输入区优先采用“单输入框 + 附件预览 + 发送动作”的聊天输入条，而不是能力标签区、状态说明区和表单控件混排。
      4. 中栏顶部优先采用轻量聊天头，只保留角色对象本身与必要状态，不再保留大段页面说明或第二层对象标题。
      5. 输入区优先采用编辑器式输入盒 + 工具条 + 发送动作的组合，不再追加大段解释性辅助文案。
   7. 同一角色对象在中栏主区域的命名层级需避免重复，优先只保留一层主对象标题；
      1. 主标题默认保留角色名本身；
      2. agent 身份仅作为次级状态说明出现，不得再次形成第二个标题。
   8. 默认应让中部 `创建角色会话` 成为主阅读区；中部与右侧建议按约 `6 / 4` 分栏，而不是让右侧占据过重比重。
   9. 任务流程视图中的当前任务标题、状态和下一步提示在默认比例下必须清晰可读，不得依赖极小字号或 SVG 缩放后文字。
   10. 阶段语义默认只保留一套主表达；若纵向主线已可读，不再额外增加横向阶段条。
3. 可追溯性：
   1. 从 `创建角色` 页点击任务后，必须可追溯到任务中心中的同一任务对象，而不是页面本地副本。
   2. 当前方案允许在任务卡悬浮时展示详情卡，以降低默认信息密度，同时保留对任务中心事实的可见性。
   3. 浮层详情不得通过任务卡高度变化或任务流重排来表达，否则会破坏当前流程阅读稳定性。
4. 状态一致性：
   1. `runtime_status` 门禁与后端指派结果必须一致，不允许页面显示可派发、后端却拒绝。
5. 恢复能力：
   1. 页面刷新后，当前角色创建任务引用视图应能基于任务中心相关数据恢复。
6. 连续性能力：
   1. 新角色工作区在初始化完成后，应具备与当前工作区同类的连续记忆能力，而不是依赖后续人工补建。

## 6. 验收标准（Given/When/Then）
1. AC-RC-01 创建角色入口可见
   1. Given 用户进入 `训练中心`
   2. When 页面加载完成
   3. Then 可看到 `角色资产与版本 / 创建角色 / 训练闭环` 三个子模块
   4. And `创建角色` 可正常切换进入
2. AC-RC-02 对话与图片输入可用
   1. Given 用户进入 `训练中心 -> 创建角色`
   2. When 用户在输入区粘贴/拖拽/选图，并继续输入文本
   3. Then 页面在发送前可展示待发送图片预览
   4. And 发送后页面可同时展示文本轮次与同条消息内的图片示例
   5. And 页面可见真实可操作的聊天输入框与发送动作，而不强制依赖单独“上传图片”主按钮
   6. And 对话输入入口在首屏即可直接看到，且视觉上与历史消息区分
   7. And 当角色已进入 `创建中` 时，输入条仍保留在当前草案页内可继续补充描述
   8. And 对话历史、系统任务更新与输入区处于同一聊天壳中，而不是拆分成多个独立卡片
   9. And 消息区整体读感更接近飞书聊天，而不是说明卡片流
   10. And 用户消息展示在右侧气泡，agent 消息展示在左侧气泡
   11. And 外层 `增长分析师` 面板与聊天内容构成同一连续工作面，不再额外套一层聊天框
3. AC-RC-03 结构化角色草案生成
   1. Given 用户已完成角色目标、能力、边界与示例的基础描述
   2. When 系统完成当前轮收口
   3. Then 页面可展示结构化 `role_spec`
   4. And 若仍缺关键字段，页面明确提示缺失项
4. AC-RC-04 仅生成关联任务
   1. Given 用户点击 `开始创建`
   2. When 系统生成创建任务图
   3. Then `创建角色` 页只展示当前角色创建工作流的关联任务
   4. And 不显示任务中心其他无关任务
5. AC-RC-05 统一阶段与任务图同步状态
   1. Given 当前角色已生成创建任务图
   2. When 用户查看 `创建角色` 页中的任务图
   3. Then 右侧主组件以训练中心现有“训练演进图”同类方式展示统一阶段与任务演进图
   4. And 页面可一眼区分 6 个阶段、当前阶段、每个阶段下挂任务、待开始节点与回补分支
   5. And 阶段 1 仅表现为前置条件，不展示任务中心任务
   6. And 以任务卡形式呈现的节点全部来自任务中心任务引用
   7. And 当前分析师动作以阶段动作提示、系统消息或状态锚点表达，而不是伪装成任务中心任务卡
   8. And 节点状态与任务中心中的真实运行状态一致
   9. And 关键任务变化可在当前聊天流中看到对应的系统消息
   10. And 右侧一体图会随聊天补充与 `role_spec` 变化同步刷新
   11. And 图面整体读法更接近既有任务中心任务图：左侧先读阶段 gutter，右侧再读连续任务节点与依赖线
   12. And 当前图不再以“6 列悬挂任务卡片”作为主表达
   13. And 右侧默认不再拆出第二块任务详情或门禁卡，当前阶段、当前任务与回退路径在同一图内即可读
   14. And 任务默认贴齐所属阶段展示，而不是漂浮在与阶段弱关联的位置
   15. And 当前任务文字在默认视图中清晰可读
   16. And 当纵向阶段主线已成立时，右侧不再额外展示一条横向阶段概览
6. AC-RC-05A 右侧辅助区标签切换可用
   1. Given 用户处于 `创建角色` 会话中
   2. When 用户切换右侧辅助区标签
   3. Then 可在 `阶段与任务演进` 与 `角色画像` 两个视图之间切换
   4. And 切换过程中中部聊天上下文不丢失
   5. And 两个视图都基于当前同一角色事实持续刷新
6. AC-RC-06 任务真相源唯一
   1. Given 当前角色已有创建任务与执行结果
   2. When 用户在 `创建角色` 页查看某个任务
   3. Then 所见状态与任务中心中的同一任务一致
   4. And 页面不维护第二套独立任务状态
   5. And 任务卡可见 `task_id` 或等效任务引用标识
   6. And 阶段 2 及之后的具体任务卡可见“任务中心引用”来源语义
   7. And 当用户悬浮任务卡时，可查看该任务的来源、上下游、证据摘要与跳转任务中心提示
   8. And 悬浮详情在当前辅助区内直接可见，不因容器裁剪而消失
   9. And 点击任务卡后，可将当前详情以浮层卡形式固定住继续查看
   10. And 详情浮层的出现不会挤动任务流或改变任务卡高度
   11. And 默认卡面不展示与当前使用无关的大段解释文字
   12. And 当前分析师动作不会被伪装成带 `task_id` 的任务中心任务卡
   13. And “去任务中心查看”默认收口在悬浮详情卡中，而不占据任务卡默认展示位
   14. And 任务名优先采用具体后台目标命名，而不是长期停留在泛化占位名
7. AC-RC-06A 明确委派语义触发任务中心建任务
   1. Given 用户在当前会话中明确说出“另起一个任务去…… / 单独起个任务…… / 后台去跑……”
   2. When 系统解析该轮意图
   3. Then 应优先在 `任务中心` 创建对应后台任务
   4. And 当前分析师继续承担协调、回传与验收动作
   5. And 该动作不得只表现为当前分析师的即时聊天回复
8. AC-RC-06B 任务卡尺寸与分析师动作交互感
   1. Given 用户查看右侧统一阶段与任务演进图
   2. When 当前阶段任务卡与非当前阶段任务卡同时出现
   3. Then 两者分别落在稳定的两档固定尺寸上，而不是随文案长度明显抖动
   4. And 当前分析师动作卡可见明确的动作标题、说明和下一步提示
9. AC-RC-06C 废案任务按阶段收纳
   1. Given 某阶段已经存在 2 条及以上已作废或已归档的后台任务
   2. When 用户查看右侧统一阶段与任务演进图
   3. Then 主线默认不平铺这些废案任务，而是显示一个按阶段归属的“废案收纳”入口
   4. And 默认卡面只展示废案数量与阶段归属，不重复铺开全部废案标题
   5. And 当用户悬浮或点击固定该入口时，可看到每条废案的 `task_id`、任务名、关闭原因与去任务中心查看入口
   6. And 这些废案在任务中心中仍保持可回查的真实历史记录
10. AC-RC-07 运行状态门禁生效
   1. Given 角色当前 `runtime_status=创建中`
   2. When 普通任务中心尝试为其指派无关任务
   3. Then 系统拒绝该指派
   4. And 仅允许 `创建角色` 模块继续追加创建相关任务
11. AC-RC-08 能力验收后的状态跳转
   1. Given 当前角色已完成当前轮能力验收
   2. When 验收结论为“仍需继续补能力”
   3. Then 该角色继续保持 `创建中`
   4. And 页面与任务图回到阶段 2 或阶段 3 的下一轮调整路径
   5. When 所有关键能力均达到要求，且用户在阶段 5 明确确认通过验收
   6. Then 系统调用完成接口，并将该角色 `runtime_status` 切为 `空闲`
   7. And 若未收到用户明确肯定答案，则流程继续保持 `创建中`
12. AC-RC-09 并行状态轴可区分
   1. Given 当前角色同时存在版本生命周期状态与运行状态
   2. When 用户查看角色草案或角色详情摘要
   3. Then 页面可同时区分 `lifecycle_status` 与 `runtime_status`
   4. And 两者不会互相覆盖
13. AC-RC-10 记忆机制完整初始化
   1. Given 当前角色已完成初始化工作区与基础资产生成
   2. When 用户或系统检查该角色工作区
   3. Then 可见与当前工作区同类的 `.codex` 四层记忆结构
   4. And 记忆维护脚本与启动恢复读取链可在该角色工作区内实际使用
14. AC-RC-11 分析引导与阶段切换可用
   1. Given 用户正在 `创建角色` 页持续对话
   2. When 当前对话 agent 判断画像已足够清晰
   3. Then 可通过后端接口把当前阶段从“角色画像收集”切换到“能力后台生成”
   4. And 右侧当前角色画像可被随时查看
15. AC-RC-12 用户确认后才完成角色创建
   1. Given 当前对话 agent 已判断所有关键能力满足要求
   2. When 当前对话 agent 发起确认验收问答
   3. Then 仅在用户明确回答满足验收标准后，系统才调用完成接口创建角色
   4. And 若用户未确认通过，则流程保持在 `创建中`
16. AC-RC-13 左侧列表可横向折叠展开
   1. Given 用户查看左侧角色创建列表
   2. When 用户点击 `<<` 折叠或 `>>` 展开
   3. Then 左侧列表宽度发生对应变化
   4. And 折叠态仍保留可见的展开入口与最小状态提示
   5. And 当前选中角色与中右主工作台状态保持不变

## 7. 边界条件与异常处理
1. 当用户描述不足：
   1. 页面保持在收集阶段；
   2. 不得提前生成创建任务图。
2. 当图片示例上传失败：
   1. 页面需提示失败原因；
   2. 已进入当前消息草稿的其他图片与文字不得丢失。
3. 当创建任务图生成失败：
   1. 角色不得误切到 `空闲`；
   2. 页面应明确提示当前仍未进入正式创建执行阶段。
4. 当创建中任务失败：
   1. 角色继续保持 `创建中`；
   2. 页面需暴露失败任务与补救动作。
5. 当能力验收结论为“不通过但可继续补能力”：
   1. 角色不得误切到 `空闲`；
   2. 页面需明确指向下一轮角色画像调整入口。
6. 当用户刷新页面或重新进入：
   1. 当前角色草案与关联任务引用视图应能恢复；
   2. 当前高亮任务与当前阶段允许回退为默认项，但不得丢失整体进度。

## 8. 依赖项与开放问题
1. 依赖项：
   1. 依赖现有 `训练中心` 继续作为角色业务域主入口。
   2. 依赖现有 `任务中心` 作为唯一任务真相源与执行底座。
   3. 依赖角色详情/角色资产列表支持新增 `runtime_status` 展示。
2. 开放问题：
   1. 当前方案默认保留显式 `开始创建` 动作；是否最终改为“信息足够即自动生成任务”，仍待确认。
   2. 创建角色工作流的标准任务模板是否固定为 5~6 步，还是允许按角色类型动态扩展，仍待确认。
   3. 图片示例的格式、大小、张数限制当前未收口，仍待确认。
   4. 角色工作区的自动命名规则与目录落位规则当前未展开，仍待确认。
