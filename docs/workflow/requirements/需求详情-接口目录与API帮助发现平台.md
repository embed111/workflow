# 需求详情-接口目录与API帮助发现平台

- 状态：正式需求（当前有效口径）
- 当前边界：本轮先把“workflow 作为系统基座需要提供接口目录 / 帮助 / 指标读面”正式收进后续版本规划；当前不在 `V5` 直接实现完整接口目录 API、页面和性能看板。
- 关联主题：
  1. `pm/versions/V6/版本计划.md`
  2. `docs/workflow/requirements/需求详情-运营项目与项目主控角色模型.md`
  3. `docs/workflow/requirements/需求详情-第二个项目最小启动能力.md`

## 1. 主题目标与价值说明
1. 问题定义（用户明示 + 当前系统事实）：
   1. `workflow` 作为系统基座，后续会承接多个运营项目，不应该只让使用者靠读源码或靠记忆去猜有哪些接口可用。
   2. 当前系统虽然已有大量 `api/*` 路由和若干运行/验收报告，但还缺一个统一的“接口帮助 / 接口目录 / 指标读面”。
   3. 用户明确希望它具备类似 `--help` 的能力：
      1. 能查询当前所有接口；
      2. 页面上能展示已有接口；
      3. 支持接口搜索；
      4. 每个接口能展示性能与压测指标；
      5. 便于后续持续优化。
2. 主题目标：
   1. 让 `workflow` 具备一套系统级接口发现能力，而不是把接口知识分散在代码、浏览器面板和零散报告里。
   2. 让后续运营项目、项目主控角色和开发/测试/质量角色都能用统一方式查看接口能力、搜索接口和判断稳定性。
   3. 让接口级性能与压测证据进入同一工作面，作为后续优化与回归的长期真相源。
4. 当前事实澄清（2026-04-22）：
   1. 当前现网接口目录能力已经具备“统一列表 API + 首批 canonical detail + 搜索/详情工作面”的第一阶段闭环。
   2. 但它还远未达到“全站接口目录已经补齐完成”。
   3. 截至 `2026-04-22 08:00 +08:00` 的现网统计：
      1. 接口总数约 `132`
      2. `metadata_status=complete` 的接口约 `10`
      3. 仍为 `incomplete / 待补目录元数据` 的接口约 `122`
      4. `metrics_status=partial` 的接口约 `2`
      5. 其余大多数接口的指标仍是 `unavailable / missing`
   4. 因此当前只能说“首批接口目录基础闭环已完成”，不能把它表述成“全站接口目录已经完成”。
3. 业务价值：
   1. 降低后续运营项目接入 `workflow` 底座的理解成本。
   2. 让平台能力更可发现，不再依赖少数人脑内知识。
   3. 为接口治理、性能优化、压测回归和版本比较提供统一入口。

## 2. 用户画像与核心场景
1. 用户画像 A：`workflow` 平台维护者
   1. 关注点：当前有哪些接口、哪些接口已经稳定、哪些接口性能需要优化。
2. 用户画像 B：运营项目创建者 / 项目主控角色
   1. 关注点：新项目要用哪些接口、怎么快速确认可用能力，不想翻源码找入口。
3. 用户画像 C：测试 / 质量 / 性能治理角色
   1. 关注点：某个接口最近的延迟、错误率、压测基线和最近一次证据路径。
4. 核心场景：
   1. 用户通过统一接口目录 API 获取当前所有可发现接口，像看 `--help` 一样快速扫盘。
   2. 用户在页面里搜索关键字，例如 `assignments / runtime-upgrade / projects`，快速定位接口。
   3. 用户点开某个接口，查看：
      1. 方法、路径、摘要、所属模块
      2. 请求/响应要点
      3. 适用场景
      4. 最近性能指标
      5. 最近压测/回归证据
   4. 运营项目在接入 `workflow` 基座时，先用这套目录确认“能用什么接口、哪些接口已经有稳定指标、哪些接口还只是实验态”。

## 3. 核心能力模型
1. 接口目录查询 API
   1. 面向程序与页面，提供统一的接口清单真相。
   2. 首期建议路由：
      1. `GET /api/platform/interfaces`
      2. `GET /api/platform/interfaces/<interface_id>`
2. 接口目录页面
   1. 作为 `workflow` 系统基座的工作面之一，承载接口列表、搜索、筛选和详情。
3. 指标聚合层
   1. 负责把接口级性能指标、压测结果、最近回归证据和缺失状态聚合到接口详情。
4. 可发现性元数据
   1. 每个接口至少要能回答：
      1. `method`
      2. `path`
      3. `summary`
      4. `module/source`
      5. `stability`
      6. `visibility`
      7. `metrics_status`

## 4. 功能需求清单
1. FR-API-HELP-01 统一接口目录查询 API
   1. 系统必须提供一个统一接口，用于查询当前所有可发现接口。
   2. 这条接口应承担类似 `--help` 的系统帮助能力，不要求用户先知道具体模块位置。
2. FR-API-HELP-02 接口列表最小字段
   1. 接口目录列表首期至少返回：
      1. `interface_id`
      2. `method`
      3. `path`
      4. `summary`
      5. `module/source_ref`
      6. `stability`
      7. `visibility`
      8. `metrics_status`
3. FR-API-HELP-03 搜索与筛选
   1. 页面和目录 API 都应支持按以下维度搜索或筛选：
      1. 路径关键字
      2. 方法（GET/POST/...）
      3. 模块
      4. 标签/能力域
      5. 稳定性状态
4. FR-API-HELP-04 接口详情页
   1. 每个接口都应有可展开的详情读面。
   2. 首期详情至少展示：
      1. 接口摘要
      2. 路径与方法
      3. 适用场景
      4. 请求/响应要点
      5. 关联代码或来源说明
      6. 最近性能指标
      7. 最近压测/回归证据入口
5. FR-API-HELP-05 性能指标展示
   1. 每个接口在有数据时，首期至少展示：
      1. 请求量
      2. 错误率
      3. `p50 / p95 / p99`
      4. 最近采样时间
   2. 若当前没有指标，必须显式展示“未采集 / 未接入 / 暂无数据”，不能留空白。
6. FR-API-HELP-06 压测指标与证据
   1. 每个接口在有历史压测或回归时，首期至少展示：
      1. 最近一次压测/回归名称
      2. 时间
      3. 关键结果摘要
      4. 证据路径或报告引用
   2. 若当前没有压测基线，也必须显式标注“未建立基线”。
7. FR-API-HELP-07 面向后续运营项目复用
   1. 这套接口目录能力不能只服务 `workflow` 自己的开发调试。
   2. 必须默认按“其他运营项目接入系统基座时也能直接使用”来设计。
8. FR-API-HELP-08 优化导向
   1. 接口目录不只是静态清单。
   2. 首期就应为后续性能优化、接口治理和压测补基线提供入口与字段，不把这部分延后成完全无关的另一个系统。
9. FR-API-HELP-09 扁平化视觉风格
   1. 当前项目的接口目录页面及其相关工作面，默认采用与现有 `workflow` 工作台一致的扁平化配色。
   2. 首期界面优先使用纯色底、细边框、轻状态色和清晰分层，不使用大面积渐变背景、重投影或高饱和大色块抢占主阅读区。
   3. 若后续需要强调状态，优先通过文字层级、边框、浅色状态底和少量点状/条状信号表达，而不是通过大块高对比背景制造噪声。
10. FR-API-HELP-10 全量目录覆盖补齐
   1. 系统后续必须继续把当前 `incomplete / 待补目录元数据` 的接口逐步补齐，不能长期只维护首批 canonical 接口。
   2. 对“已进入目录但仍待补”的接口，至少要逐步补齐：
      1. `summary`
      2. `module/source_ref`
      3. `stability`
      4. `visibility`
      5. `metrics_status`
   3. 当某条接口仍处于待补状态时，页面应显式告诉用户“当前只具备占位元数据”，不得把它包装成已完成能力。
11. FR-API-HELP-11 完成口径分层
   1. 后续版本和汇报中，必须明确区分两种完成：
      1. `首批基础闭环完成`
      2. `全量目录覆盖完成`
   2. 不允许再把第一种完成误写成第二种完成。

## 5. 验收标准
1. AC-API-HELP-01 可通过统一接口拿到清单
   1. Given 用户不知道内部模块分布
   2. When 调用接口目录查询 API
   3. Then 能拿到当前系统可发现接口列表
2. AC-API-HELP-02 页面可搜索接口
   1. Given 用户进入接口目录页面
   2. When 输入关键字或按方法/模块筛选
   3. Then 列表结果会按条件收缩
3. AC-API-HELP-03 接口详情可看指标
   1. Given 某接口存在性能或压测数据
   2. When 用户展开接口详情
   3. Then 可以看到性能和压测指标
4. AC-API-HELP-04 无指标状态不留空
   1. Given 某接口尚未接入性能采样或压测基线
   2. When 用户查看该接口详情
   3. Then 系统明确显示“未采集 / 未建立基线 / 暂无数据”之类状态
5. AC-API-HELP-05 可服务后续运营项目
   1. Given 后续运营项目需要接入 `workflow` 系统基座
   2. When 项目主控角色或维护者查询接口
   3. Then 无需读源码即可先看到当前可用接口与基础质量状态
6. AC-API-HELP-06 页面风格保持扁平化
   1. Given 用户进入接口目录页面
   2. When 查看列表、详情和质量读面
   3. Then 页面整体以纯色底、细边框和轻状态色为主
   4. And 不出现大面积渐变背景、重阴影或高饱和底色抢占主阅读区
7. AC-API-HELP-07 全量目录覆盖状态真实
   1. Given 接口目录中仍存在大量 `待补目录元数据`
   2. When 用户查看接口列表
   3. Then 系统必须显式展示这些接口仍处于待补状态
   4. And 版本汇报与完成定义不得把这部分接口误算为“已完成”

## 6. 边界条件与异常处理
1. 首期不要求一次性做完整 OpenAPI 平台、在线调试台或 SDK 生成器。
2. 若某些接口当前无法安全暴露完整细节：
   1. 也应至少在目录中给出受控摘要，而不是直接从清单里消失。
3. 若性能或压测数据来源不完整：
   1. 必须优先保证状态可见，而不是伪造“已稳定”。
4. 若页面和 API 清单出现不一致：
   1. 以后端目录查询 API 作为首期真相源。
5. 若 UCD 或前端实现尝试引入更强视觉层次：
   1. 默认先检查是否破坏当前项目的扁平化配色约定。
   2. 若只是为了“看起来更有设计感”而引入渐变、重阴影或高噪声背景，默认不接受。

## 7. 依赖项与开放问题
1. 依赖项：
   1. 当前路由注册真相、接口元数据抽取能力
   2. 现有性能指标、运行态指标或压测报告存储位置
   3. 后续版本中对“接口稳定性 / 标签 / 所属模块”字段的统一口径
2. 开放问题：
   1. 首期接口目录是完全动态抽取、半自动注册，还是人工注册 + 自动补指标，后续实现切片再定。
   2. 页面工作面是作为独立一级模块、系统设置子页，还是开发/治理页签，后续在实现阶段再定。
   3. 接口详情首期是否支持“复制调用示例 / curl 示例”，后续再决定。
