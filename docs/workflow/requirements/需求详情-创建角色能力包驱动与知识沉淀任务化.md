# 需求详情-创建角色能力包驱动与知识沉淀任务化

- 状态：正式需求（当前有效口径）
- 对应原型：沿用 `docs/workflow/prototypes/创建角色工作流-预稿/创建角色工作流预稿.html` 的主工作台骨架；本轮先收口需求与执行提示词，后续原型需补齐 `角色画像 / 能力包 / 知识沉淀 / 首批任务` 的可视结构
- 关联主题：
  1. `docs/workflow/requirements/需求详情-创建角色工作流与任务绑定.md`
  2. `docs/workflow/requirements/需求详情-训练中心创建角色链路修正.md`
  3. `docs/workflow/requirements/需求详情-角色中心与训练优化工作流.md`
  4. `docs/workflow/requirements/需求详情-统一Codex失败原因展示与重试治理.md`

## 1. 主题目标与价值说明
1. 问题定义（2026-03-31 live 实测事实）：
   1. 当前 `角色中心 -> 创建角色` 的对话分析师已经能在聊天层面理解“能力包型 agent”诉求：
      1. 能回答“如何拆能力模块”；
      2. 能回答“要沉淀哪些知识文件或示例文件”；
      3. 能继续追问“当前最关键还缺什么”。
   2. 但底层结构仍主要停留在 `六要素角色画像`：
      1. `角色名`
      2. `角色目标`
      3. `核心能力`
      4. `边界`
      5. `适用场景`
      6. `协作方式`
   3. 当前高密度输入中的复杂信息会被错位压缩进不合适字段：
      1. 适用场景与默认交付策略会被混入 `协作方式`；
      2. `SVG 为主、Mermaid / PlantUML 为辅` 这类格式边界会被混入 `边界`；
      3. `能力模块 / 知识文件 / 反例 / 模板 / 首批工作集` 没有独立落点。
   4. 当前页面会给出“草稿信息已齐，可直接开始创建”，但下方仍同时展示与当前草稿无关的 `训练优化` 空态工作台，且 `能力对象 = 0`：
      1. 用户能感知到“前台说已经收口了”；
      2. 但系统并未把本轮收口结果真正落成 `能力对象`、`知识沉淀计划` 或 `首批任务`。
   5. 本轮实测还暴露出多项体验问题：
      1. 新建草稿后输入区是否可用不够直观，未进入激活态前输入框被禁用但缺少明确原因；
      2. 分析执行中长期只显示 `分析中...` 与待处理数，缺少“当前在抽什么”的进度语义；
      3. 当前 `创建角色` 页底部混入 `训练优化` 空壳，会稀释“当前正在创建什么”的注意力；
      4. 用户看不到“哪些内容被识别成角色画像、哪些内容被识别成能力包、哪些内容仍待确认”的结构化差异。
2. 主题目标：
   1. 让 `创建角色` 从“只收口角色画像的人设创建器”升级为“角色画像 + 能力包 + 知识沉淀 + 首批任务”的一体化创建器。
   2. 在创建阶段引入独立的能力包真相源，明确承载：
      1. 能力模块；
      2. 决策规则；
      3. 默认交付策略；
      4. 知识沉淀资产；
      5. 首批能力对象与任务建议。
   3. 让系统在对话收口完成后，不仅能继续问问题，还能真正把结果落成：
      1. `能力包结构`
      2. `知识资产计划`
      3. `首批能力对象`
      4. `首批后台任务`
   4. 把“输入可用性、分析进度、页面聚焦度、结构化差异可见性”一起纳入整改范围，而不是只改字段抽取。
3. 业务价值：
   1. 让用户创建的不是一张抽象角色画像，而是一个可复用、可迭代、可继续训练的能力包型角色。
   2. 避免“前台已经聊明白了，但后台还得靠人工重新拆任务、重新想知识目录”的重复劳动。
   3. 为方法型、模板型、知识沉淀型角色建立标准创建路径，例如：
      1. 图示建模方法师；
      2. 接口梳理方法师；
      3. 文档规约沉淀型 agent。
   4. 降低用户在创建阶段的认知断裂：不再出现“看起来已收口，但没有真实能力对象落地”的伪完成感。

## 2. 用户画像与使用场景
1. 用户画像 A：能力包型角色提出者
   1. 关注点：我要创建的不是“会干活的人设”，而是“会拆方法、会沉淀模板、会留下示例”的能力包角色。
2. 用户画像 B：workflow live 验收者
   1. 关注点：我要看的是系统有没有把收口结果变成结构化对象，而不是只看聊天回答是否更聪明。
3. 用户画像 C：角色平台治理者
   1. 关注点：后续任务、训练优化、发布评审都需要建立在可信的能力包真相源上，而不是散落在聊天文本里。
4. 核心场景：
   1. 用户创建“图示建模方法师”这类角色，希望系统自动拆出：
      1. 图种判断；
      2. 信息抽取；
      3. SVG 技法；
      4. 模板与反例；
      5. 出图验收规则。
   2. 用户明确要求创建阶段必须同时完成：
      1. 任务分解；
      2. 知识沉淀。
   3. 用户补充“默认交付粒度”“格式边界”“首批优先场景”后，希望系统立即把这些信息落到结构化计划，而不是继续只停留在聊天摘要。
   4. 用户需要看到：当前草稿里哪些是角色画像，哪些是能力包，哪些会变成后续首批任务。

## 3. 用户旅程或关键流程
1. 用户进入 `角色中心 -> 创建角色`，点击 `新建` 创建草稿。
2. 草稿创建后，页面至少维护四类结构化对象：
   1. `role_profile_spec`：六要素角色画像；
   2. `capability_package_spec`：能力包模块、决策规则与默认交付策略；
   3. `knowledge_asset_plan`：方法文档、技法文档、模板、示例、反例等沉淀资产；
   4. `seed_delivery_plan`：首批能力对象与首批后台任务建议。
3. 用户输入第一轮高密度描述后，系统同步完成两类动作：
   1. 继续按聊天方式回答用户；
   2. 把可识别内容分别写入上述四类结构化对象，而不是只写进六要素画像。
4. 页面默认至少提供以下辅助视图或等效结构区域：
   1. `角色画像`
   2. `能力包`
   3. `知识沉淀`
   4. `首批任务`
5. 当系统无法确定某段内容属于哪一层时：
   1. 应标记为待确认；
   2. 给出最小澄清问题；
   3. 不得硬塞进任意一个已有字段。
6. 当用户补充“适用场景 / 默认交付粒度 / 图源码边界 / 首批优先级”后，系统应同步更新：
   1. `applicable_scenarios`
   2. `capability_package_spec.default_delivery_policy`
   3. `capability_package_spec.format_strategy`
   4. `seed_delivery_plan.priority_order`
7. 只有当 `角色画像最小字段` 与 `能力包最小字段` 都达标时，页面才允许进入“可开始创建”。
8. 用户点击 `开始创建` 后，系统至少应产生两类真实落点：
   1. 首批能力对象；
   2. 与这些能力对象或知识沉淀资产对应的真实后台任务引用。
9. 进入 `创建中` 后，页面不应继续默认露出与当前角色无关的 `训练优化` 空壳；应优先展示与当前创建链路直接相关的能力对象、任务推进与产物回传。

## 4. 功能需求清单（编号）
1. FR-RCP-01 创建角色引入“四层真相源”
   1. 在现有 `role_spec` 语义基础上，系统必须显式维护以下四层真相源：
      1. `role_profile_spec`
      2. `capability_package_spec`
      3. `knowledge_asset_plan`
      4. `seed_delivery_plan`
   2. 这四层对象的具体命名可按实现调整，但业务语义不得缺失。
   3. 每层对象至少应具备：
      1. 当前值；
      2. 最近一次修改来源；
      3. 当前确认状态；
      4. 待确认项。
2. FR-RCP-02 复杂输入不得继续被压扁到六要素画像
   1. 下列内容不得再被简单压入 `边界 / 协作方式 / 适用场景` 等字段：
      1. 能力模块
      2. 默认交付策略
      3. 格式边界
      4. 知识文件类型
      5. 模板与反例要求
      6. 首批任务或优先顺序
   2. 当用户输入明显属于能力包或知识沉淀层时，系统必须写入对应结构，而不是继续“找一个最像的画像字段硬塞进去”。
3. FR-RCP-03 `capability_package_spec` 必须支持能力模块、规则与默认策略
   1. `capability_package_spec` 至少包含：
      1. `capability_modules`
      2. `decision_rules`
      3. `default_delivery_policy`
      4. `format_strategy`
      5. `priority_scenarios`
   2. `capability_modules` 至少支持记录：
      1. 模块名称；
      2. 模块目标；
      3. 模块输出；
      4. 依赖或前置条件。
4. FR-RCP-04 `knowledge_asset_plan` 必须支持沉淀资产清单
   1. `knowledge_asset_plan` 至少支持以下资产类型：
      1. 方法说明
      2. 技法说明
      3. 模板
      4. 最小示例
      5. 中等复杂度示例
      6. 反例 / 失败示例
      7. 验收清单
   2. 每条资产计划至少记录：
      1. 资产类型；
      2. 资产主题；
      3. 推荐落盘目录；
      4. 当前状态；
      5. 优先级。
5. FR-RCP-05 `seed_delivery_plan` 必须能生成首批能力对象与首批任务建议
   1. 当创建阶段输入已足够时，系统必须生成首批能力对象，而不是只显示 `能力对象 0 项` 的空态。
   2. 每条首批能力对象至少包含：
      1. `capability_name`
      2. `capability_goal`
      3. `source_module`
      4. `acceptance_hint`
      5. `current_status`
   3. 每条首批任务建议至少包含：
      1. 任务名称；
      2. 对应能力对象或知识资产；
      3. 任务类型；
      4. 是否应进入 `任务中心`。
6. FR-RCP-06 创建角色页必须提供“角色画像 / 能力包 / 知识沉淀 / 首批任务”可视落点
   1. 当前创建页不得继续只保留 `角色画像` 一块结构化视图。
   2. 页面至少应新增以下视图标签或等效可见区域：
      1. `能力包`
      2. `知识沉淀`
      3. `首批任务`
   3. 用户必须能直接看见：
      1. 哪些内容已经落入能力模块；
      2. 哪些内容会沉淀成知识资产；
      3. 哪些内容会变成首批后台任务。
7. FR-RCP-07 分析执行需要更细粒度进度语义
   1. 当分析时长超过 3 秒，系统不得长期只显示 `分析中...`。
   2. 页面至少应按子阶段显示当前进度，例如：
      1. `抽取角色画像`
      2. `拆解能力模块`
      3. `整理知识资产`
      4. `生成首批任务建议`
   3. 若某一子阶段失败，系统必须明确标识失败阶段与失败原因。
8. FR-RCP-08 草稿激活与输入可用性必须可理解
   1. 新建草稿后，若输入框暂不可用，页面必须明确说明原因，而不是只表现为禁用态。
   2. 当草稿进入可输入态后，页面需显式反馈当前已激活。
   3. 不得要求用户通过试错才知道“是否已经成功进入当前草稿”。
9. FR-RCP-09 启动门禁升级为“画像 + 能力包”双收敛
   1. `可开始创建` 不再只由六要素最小字段决定。
   2. 至少同时满足以下条件，才允许开始创建：
      1. 角色画像最小字段收口；
      2. 能力模块已形成最小集合；
      3. 知识沉淀资产已形成最小计划；
      4. 首批任务建议已生成或明确说明为何暂不生成。
   3. 页面必须区分：
      1. `画像已齐`
      2. `能力包已齐`
      3. `可开始创建`
10. FR-RCP-10 创建页不得默认混入无上下文的训练优化空壳
   1. 在 `创建角色` 会话中，页面不应默认显示与当前角色无关的 `训练优化` 空态字段块，例如：
      1. `目标角色`
      2. `能力目标`
      3. `验收标准`
      4. `本轮能力`
   2. 若需要复用同类工作台骨架，必须替换成当前创建链路对应的结构：
      1. `首批能力对象`
      2. `知识沉淀资产`
      3. `首批任务建议`
11. FR-RCP-11 创建角色分析失败必须接入统一失败原因与显式重试
   1. 创建角色分析链路必须遵循 `统一Codex失败原因展示与重试治理` 的当前有效口径。
   2. 当任一分析子阶段失败时，用户至少可见：
      1. 标准失败码；
      2. 失败阶段；
      3. 中文失败原因；
      4. 当前尝试次数；
      5. 显式重试动作。
   3. 失败后必须保留当前四层真相源与待确认项，不得把草稿回退成空白。
12. FR-RCP-12 结构化差异必须可见
   1. 当用户补充或修正信息后，页面至少应显示本轮影响到了哪一层、哪些字段或模块。
   2. 用户应能直接看见：
      1. 新增了哪些能力模块；
      2. 新增了哪些知识资产；
      3. 哪些字段仍待确认；
      4. 哪些内容因为归类不明被挂起。
13. FR-RCP-13 图示建模类角色必须可作为标准验收案例
   1. 系统必须支持像“图示建模方法师”这类方法型角色的创建输入。
   2. 至少要能正确收口以下类型内容：
      1. `4+1 视图 / 流程图 / 时序图 / 类图`
      2. `箭头怎么画`
      3. `多条连接线怎么走`
      4. `marker / path / text / tspan / foreignObject`
      5. `模板 / 示例 / 反例`
   3. 这些内容不得再退化为“泛化的核心能力一句话”。

## 5. 非功能需求（性能、安全、可用性、合规等）
1. 结构准确性：
   1. 高密度输入中的模块、规则、资产、任务建议不得被大面积错位映射。
   2. 若无法判断归类，优先标记待确认，而不是错误写入。
2. 可理解性：
   1. 用户应在首屏直接理解“当前草稿已收口到哪一层、还差哪一层”。
   2. 不得继续出现“前台说准备好了，底层却没有能力对象”的强冲突信号。
3. 响应体验：
   1. 纯文本能力包收口场景下，分析返回 `p95 <= 15s`。
   2. 当超过 3 秒仍未完成，必须显示子阶段进度。
4. 可恢复性：
   1. 页面刷新后，四层真相源、待确认项、最近一次失败状态都必须可恢复。
   2. 失败或超时后不得丢失当前草稿。
5. 可追溯性：
   1. 任意一条能力模块、知识资产或首批任务建议，都必须可追溯到触发它的用户消息与分析 trace。
6. 视觉聚焦：
   1. `创建角色` 页主工作台默认只承载与当前创建链路直接相关的信息。
   2. 不得默认混入其他子模块的空壳、无关表单或误导性说明块。

## 6. 验收标准（Given/When/Then）
1. AC-RCP-01 高密度能力包输入能够落成多层结构
   1. Given 用户在 `创建角色` 中输入一段高密度能力包描述
   2. When 首轮分析完成
   3. Then 页面除 `角色画像` 外，还能看到 `能力包 / 知识沉淀 / 首批任务` 的结构化落点
   4. And 系统明确展示至少 1 条能力模块与至少 1 类知识资产
2. AC-RCP-02 图示建模案例不再被压扁到六要素
   1. Given 用户输入“图示建模方法师”案例，并明确提出：
      1. 图种判断
      2. SVG 技法
      3. 模板 / 示例 / 反例
   2. When 系统完成抽取
   3. Then 这些内容分别进入 `能力包` 与 `知识沉淀` 层
   4. And 不得仅被压缩成 `核心能力` 的一句话摘要
3. AC-RCP-03 补充默认策略后可见结构化差异
   1. Given 用户第二轮补充：
      1. 适用场景
      2. 默认交付粒度
      3. 格式边界
      4. 首批优先顺序
   2. When 系统处理该轮消息
   3. Then 页面可见本轮新增或更新的结构化差异
   4. And 这些内容分别落到对应层，而不是继续混入 `协作方式` 或 `边界`
4. AC-RCP-04 `可开始创建` 必须以双收敛为前提
   1. Given 角色画像字段已齐
   2. And 能力包、知识沉淀或首批任务计划仍未形成
   3. When 页面计算是否允许开始创建
   4. Then 不得直接显示“草稿信息已齐，可直接开始创建”
   5. And 必须明确指出当前缺少的是哪一层
5. AC-RCP-05 首批能力对象不得继续为空
   1. Given 用户已经完成能力包最小收口
   2. When 页面展示当前创建草稿状态
   3. Then 首批能力对象区域至少出现 1 条真实对象
   4. And 不得继续只显示 `能力对象 0 项`
6. AC-RCP-06 创建页不再混入无关训练优化空壳
   1. Given 用户当前停留在 `创建角色` 草稿会话
   2. When 页面渲染底部或右侧结构区
   3. Then 不得默认出现与当前角色无关的 `目标角色 / 能力目标 / 本轮能力` 空态表单
   4. And 必须替换为当前创建链路相关结构
7. AC-RCP-07 分析进度不再只有 `分析中...`
   1. Given 某轮分析执行时间超过 3 秒
   2. When 页面展示处理中状态
   3. Then 页面至少显示当前子阶段
   4. And 若失败，页面可见失败阶段与重试动作
8. AC-RCP-08 输入区激活状态清晰
   1. Given 用户点击 `新建`
   2. When 草稿尚未完全激活
   3. Then 输入区若不可用，必须明确提示原因
   4. And 草稿可输入时必须有清晰激活反馈
9. AC-RCP-09 失败后保留当前草稿
   1. Given 创建角色分析链路超时或失败
   2. When 页面回写失败结果
   3. Then 当前草稿中的四层真相源与待确认项仍可见
   4. And 用户可直接执行显式重试

## 7. 边界条件与异常处理
1. 当用户只补充“首批优先级”而不补其他信息：
   1. 系统只更新 `seed_delivery_plan.priority_order`；
   2. 不得重置已确认的能力模块或知识资产。
2. 当用户输入的内容既像规则又像边界：
   1. 系统应显式标记待确认；
   2. 不得直接一刀切写入 `边界`。
3. 当系统暂时无法生成首批任务建议：
   1. 页面必须明确说明原因；
   2. 不得继续显示“草稿信息已齐，可开始创建”。
4. 当页面骨架复用了 `训练优化` 组件：
   1. 必须替换文案、字段与对象语义；
   2. 不得把与当前创建链路无关的空态直接暴露给用户。
5. 当创建角色分析失败：
   1. 当前草稿状态保持；
   2. 待确认项保持；
   3. 失败原因与重试入口保持可见。

## 8. 依赖项与开放问题
1. 依赖项：
   1. 依赖 `创建角色` 现有 `role_spec` 草稿对象能力继续存在，但需向上扩展为多层真相源。
   2. 依赖 `任务中心` 提供首批任务建议与真实任务引用的承接能力。
   3. 依赖 `统一Codex失败原因展示与重试治理` 为创建角色分析子阶段提供统一失败合同。
2. 开放问题：
   1. `能力包 / 知识沉淀 / 首批任务` 最终采用标签页、分栏卡片还是折叠分组呈现，后续需结合原型进一步收口。
   2. 首批能力对象是在 `开始创建` 前作为预览对象出现，还是点击后才正式生成，需要在实现设计阶段最终定稿；当前需求只要求“至少有真实对象，不再是空壳”。
3. 能力扩展评估：
   1. 本轮暂不新增独立 `skill` 或独立 `agent` 角色来解决该问题。
   2. 理由：当前核心缺口不是“缺少一个更会聊的 agent”，而是 `workflow` 创建角色链路缺少多层真相源、结构化落点与任务化承接。
