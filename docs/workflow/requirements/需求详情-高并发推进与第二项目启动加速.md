# 需求详情-高并发推进与第二项目启动加速

- 状态：正式需求（当前有效口径）
- 当前边界：本轮先把“并发优先提效”正式收成治理与版本需求；首期不承诺一次性做完整的多项目高级编排中心，而是优先提升 helper 并发承载与第二项目启动速度。
- 关联主题：
  1. `pm/PM版本推进计划.md`
  2. `docs/workflow/requirements/需求详情-第二个项目最小启动能力.md`
  3. `docs/workflow/requirements/需求详情-运营项目与项目主控角色模型.md`

## 1. 主题目标与价值说明
1. 问题定义（用户明示 + 当前治理现场）：
   1. 用户明确要求“尽量提高并发能力来提高推进效率”。
   2. 当前仓内虽然已有“并行提效”治理规则，但实际执行仍容易退回低并发串行收口。
   3. 当完整方案推进偏慢时，第二项目启动会被不必要地拖延。
2. 主题目标：
   1. 把“并发优先提效”从执行偏好提升为正式需求与治理目标。
   2. 让 `V5-R1` 明确承担：
      1. helper 并发承载能力提升
      2. 并发峰值与并发时长提升
      3. 第二项目启动场景下的推进加速
3. 业务价值：
   1. 缩短从“需求确认”到“真实推进”的等待时间。
   2. 降低 PM 因串行回收而成为速度瓶颈的概率。
   3. 让第二项目启动不再被完整多项目体系拖慢。

## 2. 首期最小范围
1. 首期治理范围：
   1. helper 并发派发策略
   2. 并发时长与并发峰值度量
   3. 面向第二项目启动的并行切片优先级
2. 首期不必一次完成：
   1. 高级跨项目冲突自动协调
   2. 全自动任务图优化
   3. 完整项目级资源调度器

## 3. 核心要求
1. 当主线连续性、发布边界、helper workspace 条件成立时：
   1. 系统与 PM 治理都应优先并行切片。
   2. 不得长期以“串行更稳”作为默认口径。
2. 当当前高优先目标是“第二项目最小启动能力”时：
   1. 并发目标应优先服务第二项目加速。
   2. 默认优先并起：
      1. 最小对象/模型切片
      2. 测试/回归切片
      3. 质量/证据冻结切片
3. 并发评估不能只看派发数量：
   1. 还必须看 `parallel_peak_count`
   2. `parallel_peak_duration`
   3. `parallel_total_active_duration`

## 4. 功能需求清单
1. FR-CONC-01 并发优先治理
   1. 当并发前提成立时，系统默认应优先选择并发切片，而不是串行收口。
2. FR-CONC-02 第二项目启动加速
   1. 当当前版本目标包含“第二项目最小启动能力”时，并发策略必须优先为该目标服务。
3. FR-CONC-03 并发质量不低于数量
   1. 并发提效不得只追求节点数量。
   2. 必须同时关注并发时长、有效交付和返工情况。
4. FR-CONC-04 低并发轮次纠偏
   1. 若连续多轮存在明确可并发切片却未形成有效派发，必须视为治理问题，而不是正常保守策略。

## 5. 验收标准
1. AC-CONC-01 并发优先进入正式版本目标
   1. Given 查看 `V5-R1`
   2. When 阅读版本计划与需求映射
   3. Then 能明确看到“高并发推进与第二项目启动加速”已成为正式目标
2. AC-CONC-02 第二项目启动不再被完整体系长期阻塞
   1. Given 当前目标是第二项目最小启动
   2. When 判断推进顺序
   3. Then 能明确看到并发策略优先服务该目标
3. AC-CONC-03 并发指标进入正式治理
   1. Given 查看 PM 治理规则
   2. When 阅读并行提效章节
   3. Then 能看到并发峰值、时长、低并发纠偏和第二项目加速口径已被正式固化
