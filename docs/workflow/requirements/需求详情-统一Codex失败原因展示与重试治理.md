# 需求详情-统一Codex失败原因展示与重试治理

- 状态：需求收口版
- 对应排查报告：`docs/workflow/reports/排查报告-Codex调用失败原因与重试现状-20260330.md`
- 关联主题：
  1. `docs/workflow/requirements/需求详情-任务中心真实执行化与执行链路可视化.md`
  2. `docs/workflow/requirements/需求详情-训练中心创建角色链路修正.md`
  3. `docs/workflow/requirements/需求详情-agent发布管理与版本切换.md`
  4. `docs/workflow/requirements/需求详情-创建角色工作流与任务绑定.md`

## 1. 主题目标与价值说明
1. 问题定义（用户明示）：
   1. `Codex` 运行极不稳定。
   2. `../workflow` 中所有调用 `Codex` 请求的功能，都必须明确标明失败原因。
   3. 所有这类功能都必须提供可执行的重试能力。
2. 现状判断（事实）：
   1. 当前各链路并非完全没有失败提示和重试。
   2. 真正问题是：失败原因展示、失败码标准、重试入口和证据下钻能力分散在不同主题中，成熟度明显不一致。
3. 主题目标：
   1. 为所有真实 `Codex` 请求链路建立统一失败合同。
   2. 让前端统一展示“失败原因 + 失败阶段 + 重试动作 + 证据入口”。
   3. 让不同链路保留各自业务动作，但由统一 `retry_action` 协议驱动。
   4. 把“继续发送自动重试”“重新进入发布评审”“重跑任务”“重试上一轮”等分散语义，收口到一致的产品语言和状态机。
4. 业务价值：
   1. 降低用户面对 `Codex` 不稳定时的黑盒感。
   2. 缩短“失败后重新恢复”的路径，减少人工二次排查成本。
   3. 提高跨链路的一致性，避免用户在不同模块学习不同的失败处理规则。
   4. 为后续失败聚类、可观测性和稳定性治理提供统一数据面。

## 2. 用户画像与使用场景
1. 用户画像 A：workflow 日常使用者
   1. 关注点：这次为什么失败、能不能立刻重试、重试后会不会重复执行。
2. 用户画像 B：平台治理者
   1. 关注点：`Codex` 失败是否可追溯、不同模块是否遵守同一合同、失败聚类是否可统计。
3. 用户画像 C：需求分析 / 角色运营者
   1. 关注点：创建角色、发布评审等关键链路失败时，页面不要只说“出错了”，而要明确下一步。
4. 核心场景：
   1. 策略提取超时后，用户直接看到超时原因和“重试分析”入口。
   2. 会话任务执行失败后，用户直接看到失败阶段和“重试上一轮”。
   3. 创建角色分析失败后，用户既能继续发消息，也能主动点击“重试本轮分析”。
   4. 发布评审失败后，用户能区分“重进评审”和“直接重试发布”。
   5. 任务中心真实执行失败后，用户能看到真实失败码、证据路径和“重跑任务”。

## 3. 用户旅程或关键流程
1. 任一 `Codex` 调用链路发起执行前，系统创建一次可追溯的尝试记录。
2. 若执行成功：
   1. 页面按原业务主题展示成功结果；
   2. 同时保留最近一次尝试的证据引用。
3. 若执行失败：
   1. 后端必须返回统一失败对象；
   2. 前端必须展示统一失败卡；
   3. 失败卡至少包含：
      1. 失败原因
      2. 失败阶段
      3. 是否可重试
      4. 重试按钮或禁止原因
      5. 证据入口
      6. 下一步建议
4. 用户点击重试后：
   1. 系统创建新的尝试记录；
   2. 明确这次重试对应的动作类型；
   3. 不得生成重复的有效业务链或假性成功状态。
5. 用户回看历史时：
   1. 能看到最近一次失败原因；
   2. 能知道已尝试多少次；
   3. 能看到最近一次重试的结果。

## 4. 功能需求清单（编号）
1. FR-CDX-01 统一纳入范围
   1. 下列 5 条链路必须全部纳入本主题：
      1. 策略提取 / `AGENTS.md` 分析
      2. 会话任务执行中心
      3. 创建角色分析对话
      4. 发布评审 / 确认发布兜底
      5. 任务中心真实执行
   2. 不允许只修其中一条链路就宣称“已完成统一治理”。
2. FR-CDX-02 统一失败对象
   1. 每次真实 `Codex` 调用失败时，后端都必须返回统一失败对象，至少包含：
      1. `feature_key`
      2. `attempt_id`
      3. `attempt_count`
      4. `failure_code`
      5. `failure_detail_code`
      6. `failure_stage`
      7. `failure_message`
      8. `retryable`
      9. `retry_action`
      10. `trace_refs`
      11. `failed_at`
   2. 允许链路保留历史字段，如 `policy_error`、`error`、`failure_reason`，但页面统一消费上述标准字段。
3. FR-CDX-03 统一失败码
   1. 标准失败码至少覆盖：
      1. `workspace_not_found`
      2. `workspace_out_of_scope`
      3. `required_input_missing`
      4. `codex_command_not_found`
      5. `codex_exec_timeout`
      6. `codex_exec_failed_exit`
      7. `codex_exec_exception`
      8. `codex_stream_disconnected`
      9. `codex_output_invalid_json`
      10. `codex_result_missing`
      11. `artifact_missing_or_unreadable`
      12. `unknown`
   2. `failure_detail_code` 必须保留原始细节：
      1. 例如 `codex_exec_failed_exit_7`
      2. 例如 `codex_exec_exception:<detail>`
      3. 例如模块私有码如 `analyst_workspace_not_found`
   3. 不允许只有原始自由文本，没有稳定失败码。
4. FR-CDX-04 统一失败阶段
   1. 每次失败都必须归类到稳定阶段：
      1. `precheck`
      2. `prompt_build`
      3. `invoke`
      4. `stream`
      5. `parse`
      6. `postprocess`
   2. 前端必须展示该阶段，帮助用户区分“调用前就失败”和“调用后解析失败”。
5. FR-CDX-05 统一中文失败文案
   1. `failure_message` 必须是面向用户可读的中文解释。
   2. 当后端已提供 `failure_message` 时，前端不得继续退化为“请求未成功”“执行出现异常”“分析失败”等泛化文案。
   3. 只有在后端无法给出标准文案时，前端才允许回退到兜底文案。
6. FR-CDX-06 统一失败卡
   1. 所有纳入范围的前端页面都必须提供同类失败卡。
   2. 失败卡至少展示：
      1. 失败原因
      2. 失败阶段
      3. 最近尝试时间
      4. 已尝试次数
      5. 重试按钮或禁止原因
      6. 证据入口
      7. 下一步建议
   3. 失败卡可按页面风格调整布局，但语义字段不得缺失。
7. FR-CDX-07 统一重试动作协议
   1. 后端必须返回 `retry_action` 对象，而不是只让前端自行猜测该调哪个接口。
   2. `retry_action` 至少包含：
      1. `kind`
      2. `label`
      3. `enabled`
      4. `blocked_reason`
      5. `request_ref`
   3. `kind` 至少覆盖：
      1. `retry_policy_analysis`
      2. `retry_session_round`
      3. `retry_role_creation_analysis`
      4. `retry_release_review`
      5. `retry_publish`
      6. `rerun_assignment_node`
8. FR-CDX-08 各链路保留业务差异，但不破坏统一合同
   1. 策略提取链路：
      1. 必须显式提供“重试分析”；
      2. 不再只依赖“重新生成缓存”作为用户侧唯一理解路径。
   2. 会话任务执行链路：
      1. 保留“重试上一轮”；
      2. 但由统一失败对象驱动文案和按钮状态。
   3. 创建角色链路：
      1. 保留“继续发送自动重试未处理消息”；
      2. 同时必须新增显式“重试本轮分析”。
   4. 发布评审链路：
      1. 继续区分“重新进入发布评审”和“重试发布”；
      2. 但二者都必须通过统一 `retry_action` 协议暴露。
   5. 任务中心真实执行链路：
      1. 继续使用“重跑任务”；
      2. 但失败原因必须优先使用标准失败码和中文文案，而不是 stderr 摘要兜底。
9. FR-CDX-09 重试幂等
   1. 重试不得生成重复有效链路。
   2. 同一失败态下的重复点击、页面刷新或短时间内并发点击，必须满足以下约束：
      1. 会话任务执行不得重复开启两条“上一轮”并行执行；
      2. 创建角色不得重复合并同一批未处理消息；
      3. 发布评审不得并发生成两条有效评审链；
      4. 任务中心真实执行不得为同一节点生成两条并存活动执行链；
      5. 策略提取不得在同一对象上写出彼此冲突的当前态。
10. FR-CDX-10 证据入口统一
   1. 每次失败都必须可下钻到本次尝试对应的证据。
   2. `trace_refs` 至少应支持以下路径或等效引用：
      1. `prompt`
      2. `stdout`
      3. `stderr`
      4. `result`
      5. `gate` 或等效决策文件
   3. 页面允许按需折叠，但不得隐藏证据入口。
11. FR-CDX-11 重试结果必须回写
   1. 用户点击重试后，页面必须能区分：
      1. 重试中
      2. 重试成功
      3. 重试失败
      4. 重试被阻止
   2. 不允许出现“点击了重试但页面没有任何状态变化”的黑洞体验。
12. FR-CDX-12 指标与统计
   1. 系统至少应记录：
      1. `feature_key`
      2. `failure_code`
      3. `failure_stage`
      4. `attempt_count`
      5. `retry_result`
   2. 后续稳定性治理应能按“功能 × 失败码 × 是否重试成功”聚类。

## 5. 非功能需求（性能、安全、可用性、合规等）
1. 一致性：
   1. 5 条链路必须使用同一失败对象语义。
   2. 同一失败码在不同页面的中文解释不得明显冲突。
2. 可用性：
   1. 对于 `retryable=true` 的失败，页面首屏必须能直接触达重试动作，不要求用户先去日志页。
3. 可观测性：
   1. 每次失败都必须能追溯到尝试记录与证据路径。
4. 可靠性：
   1. 重试必须具备幂等约束，不得因连续点击制造新的状态分裂。
5. 兼容性：
   1. 允许在实现期保留历史字段与历史接口结构；
   2. 但新的页面渲染与验收必须以统一失败合同为准。
6. 安全性：
   1. 证据入口可展示路径和必要摘要；
   2. 若包含敏感参数，应对命令摘要做脱敏；
   3. 不得因为脱敏而掩盖失败原因本身。

## 6. 验收标准（Given/When/Then）
1. AC-CDX-01 策略提取失败可见且可重试
   1. Given 策略提取链路发生 `codex_exec_timeout`
   2. When 用户查看当前策略分析结果
   3. Then 页面展示标准失败卡
   4. And 失败原因明确为超时
   5. And 页面存在“重试分析”入口
2. AC-CDX-02 会话任务执行失败文案不再泛化
   1. Given 会话任务执行链路失败且后端已返回标准失败对象
   2. When 用户查看运行态
   3. Then 页面优先展示标准失败原因
   4. And 不再只显示“请求未成功”或“执行出现异常”
   5. And 页面仍可点击“重试上一轮”
3. AC-CDX-03 创建角色分析失败有显式重试
   1. Given 创建角色 analyst 调用超时
   2. When 用户留在当前创建角色会话
   3. Then 页面展示失败原因
   4. And 页面存在“重试本轮分析”入口
   5. And 继续发送消息仍可作为辅助重试路径
4. AC-CDX-04 发布评审沿用统一合同
   1. Given 发布评审链路发生 `codex_result_missing`
   2. When 用户查看评审页
   3. Then 页面展示标准失败卡
   4. And 可看到“重新进入发布评审”或等效动作
   5. And 若处于确认发布失败态，则可看到“重试发布”或其禁止原因
5. AC-CDX-05 任务中心真实执行失败原因标准化
   1. Given 任务中心真实执行因 `Codex` 超时失败
   2. When 用户查看任务详情
   3. Then `失败原因` 优先展示标准中文文案
   4. And 页面保留“重跑任务”动作
   5. And 可查看本次执行对应的证据引用
6. AC-CDX-06 证据入口一致可见
   1. Given 任一链路发生失败
   2. When 用户展开失败卡的证据区
   3. Then 至少可看到 `prompt/stdout/stderr/result` 中的已有项或其路径
7. AC-CDX-07 重试动作幂等
   1. Given 用户对同一失败态连续点击两次重试
   2. When 后端处理该请求
   3. Then 不得创建两条并存有效执行链
   4. And 页面能看到最近一次尝试状态
8. AC-CDX-08 标准失败码可统计
   1. Given 任一 `Codex` 链路产生失败
   2. When 检查日志、状态或统计输出
   3. Then 至少可看到 `feature_key + failure_code + failure_stage + retry_result`

## 7. 边界条件与异常处理
1. 若失败不可重试：
   1. `retryable=false`
   2. 页面必须显示禁止原因；
   3. 不得渲染可点击但无效的按钮。
2. 若后端只有历史字段、尚未完成统一适配：
   1. 该链路不得视为验收通过；
   2. 页面可临时兼容读取旧字段，但必须补齐标准失败对象。
3. 若证据文件缺失或不可读：
   1. 页面必须显式提示“证据缺失/不可读”；
   2. 不得伪装为“无失败细节”。
4. 若重试仍然失败：
   1. 页面必须显示本次重试的新失败原因；
   2. 并更新尝试次数和最近尝试时间。
5. 若同一功能存在多级动作：
   1. 例如发布评审与确认发布；
   2. 系统必须区分动作级失败，而不是只给出模块级失败。

## 8. 依赖项与开放问题
1. 依赖项：
   1. 依赖现有 5 条链路都能返回原始失败信息与尝试上下文。
   2. 依赖前端可接入统一失败卡组件或等效统一渲染层。
   3. 依赖执行证据路径在各链路中可回查。
2. 开放问题：
   1. 统一失败对象是直接写入现有接口响应，还是通过适配层包装后返回，仍待实现阶段定稿。
   2. 是否需要把“失败原因聚类统计”直接做成页面模块，当前可后置到第二阶段。
3. 当前边界（事实）：
   1. 本主题只收口失败原因展示与重试治理。
   2. 本主题不重写各业务链路的主体流程，不改发布策略、不改创建角色阶段流、不改任务中心主结构。
