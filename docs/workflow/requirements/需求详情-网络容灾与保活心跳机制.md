# 需求详情-网络容灾与保活心跳机制

- 状态：后续版本需求（已进入排期）
- 当前边界：本轮先定义“Codex API / 本地网络 / 长任务执行”相关的网络容灾、保活心跳与自动恢复目标，不在当前版本承诺一次性实现跨地域高可用。

## 1. 问题定义
1. 当前 `workflow` 的 7x24 连续运行已不只受代码正确性影响，还会被以下外部连续性问题打断：
   1. `Codex API` 连接中断或响应长时间无结果。
   2. 本地网络波动导致长任务执行被卡死或错误回收。
   3. Web / 调度 / 任务执行链无法区分“临时网络抖动”与“真实失败退出”。
2. 这些问题如果没有专门的容灾与保活设计，会直接让：
   1. 自迭代任务被误判失败；
   2. 运行中任务被收成“运行句柄缺失”；
   3. 7x24 闭环只能靠人工再次点火。

## 2. 目标
1. 提升 `Codex API` 和本地网络波动下的连续运行韧性。
2. 让系统能持续知道“任务是真的死了，还是只是网络短暂抖动”。
3. 让网络层问题不再直接把 7x24 连续推进链打断。

## 3. 核心需求
1. FR-NET-01 任务心跳
   1. 长任务执行期间必须定期落 heartbeat。
   2. heartbeat 需要同时覆盖：
      1. provider 进程仍存活
      2. 最近一次 stdout/stderr/event 时间
      3. 最近一次 API 往返状态
2. FR-NET-02 网络抖动宽限期
   1. 在 heartbeat 连续缺失但 provider 仍可能存活时，不得立刻把任务收成失败。
   2. 需要引入网络抖动宽限窗口与重试策略。
3. FR-NET-03 API 重试与退避
   1. 对可重试的网络错误，必须支持指数退避和有限次重试。
   2. 不得把短时网络异常直接映射成业务失败。
   3. 若外部网络或 `Codex API` 在宽限窗口内恢复，系统应优先自动续跑当前 7x24 主线，而不是等待人工再次点火。
4. FR-NET-04 任务保活状态
   1. 任务状态需要区分：
      1. 业务失败
      2. 网络不稳待恢复
      3. 心跳丢失待确认
      4. 真正不可恢复失败
5. FR-NET-05 容灾恢复
   1. 重启后应能读取最近 heartbeat/事件/结果文件，优先恢复成真实终态。
   2. 若无法恢复终态，也要进入“待人工接管”而不是直接静默丢失。
6. FR-NET-06 连续运行保护
   1. 7x24 自迭代主线需要有一条独立于单次网络故障的保底唤醒机制。
   2. 即使某一轮因网络故障失败，下一轮唤醒也不能断。
   3. 当系统判断“本轮只是网络/API 暂时不可用”时，应进入 `待网络恢复自动续跑` 之类的中间态；一旦心跳恢复或 API 健康恢复，系统要自动续接，而不是只留下失败记录。

## 4. 验收方向
1. 本地网络短暂中断时，长任务不会立刻被误收成失败。
2. `Codex API` 短时超时后，系统能在可控窗口内继续重试或进入待恢复状态。
3. 重启后若已有 `result.json / final_result / heartbeat`，系统能优先恢复真实终态。
4. 7x24 自迭代链在一轮网络故障后仍能继续排上下一轮。
5. 若网络/API 在宽限窗口内恢复，7x24 主线会自动重新进入运行，而不是停在 `failed/ready` 等待人工重派。

## 5. 版本排期建议
1. `V3`：
   1. 先补心跳、状态细分、恢复真相源。
2. `V4`：
   1. 再把网络容灾策略接入自迭代、调度和运行态看板。
3. `V5`：
   1. 再考虑更强的多路径容灾与更长期的保活机制。

## 6. 当前结论
1. 这不是“可选优化”，而是 7x24 连续运行真正稳定之前必须收口的后续主题。
2. 当前先排进后续版本，不抢占 `V1` 的工程质量主线；但后续版本必须明确承接。
