# 需求详情-角色专业能力前置合同与输出质量治理

- 状态：正式需求（当前有效口径）
- 当前边界：本轮先把“角色专业能力治理”从复盘后置推进到执行前置；首期不追求一次性做成完整功能中心，只先收口最小治理合同与试运行范围。
- 关联主题：
  1. `pm/expertise/专业能力总表.md`
  2. `pm/expertise/角色理论库建设清单.md`
  3. `pm/expertise/问题-方法映射总表.md`
  4. `pm/expertise/自我学习提升方法.md`

## 1. 主题目标与价值说明
1. 问题定义（2026-04-18 治理复盘结论）：
   1. 当前仓内已经具备 `专业能力总表`、`角色理论库建设清单`、`问题-方法映射总表`、每日学习报告与 role-assets 脚手架。
   2. 但这些流程规范当前主要作用在“做完以后怎么复盘、怎么留痕、怎么回写学习报告”。
   3. 角色在接单前、执行中、交付时仍缺少前置约束：
      1. 没有强制绑定方法卡；
      2. 没有角色专属执行检查单；
      3. 没有统一输出评分 rubric；
      4. 方法卡与案例卡没有稳定进入真实任务执行链。
   4. 结果是：
      1. 流程文件增多了；
      2. 留痕更完整了；
      3. 但小伙伴输出质量并没有被流程前置显著抬升，PM 仍经常回收高价值判断或只让 helper 补 smoke、证据与总结。
2. 主题目标：
   1. 把角色专业能力治理从“复盘后置”升级为“执行前置”。
   2. 让方法卡、案例卡、反例卡不再只是 role-assets 资料，而是成为正式接单合同的一部分。
   3. 让小伙伴在执行前就明确：
      1. 当前问题模式；
      2. 应绑定哪张方法卡；
      3. 哪些反模式本轮禁止出现；
      4. 交付什么才算完成。
   4. 让 PM 的验收不再只看“有没有产出”，而是能按角色专属 rubric 评价“专业度是否达标”。
3. 业务价值：
   1. 让流程规范真正参与输出质量治理，而不是只堆文件。
   2. 降低 PM 对首版产物的大量回收和二次重做。
   3. 让角色成长不再依赖“今天学了什么”，而依赖“真实任务里默认会先用什么方法”。

   3. 独立 UI 或产品中心页面

## 3. 核心流程
1. 接单前
   1. 角色必须明确当前 `problem_type`。
   2. 角色必须绑定 `method_card_id` 或等价方法入口。
   3. 角色必须声明本轮 `done_definition`。
   4. 角色必须声明 `forbidden_anti_patterns`。
2. 执行中
   1. 角色必须按各自专属检查单推进，而不是只凭经验即兴输出。
   2. 若执行中发现当前方法卡不适用，必须显式回报“方法失配”，而不是静默偏航。
3. 交付后
   1. PM 或等价验收方必须按角色专属 rubric 评分。
   2. 若低于阈值，本轮不能只补学习报告后算通过。
   3. 若方法卡被证明有效或失效，要回灌角色理论库与全局映射。

## 4. 功能需求清单
1. FR-PRO-01 角色接单合同前置
   1. 每个进入首期试运行范围的角色任务，都必须显式包含：
      1. `problem_type`
      2. `method_card_id`
      3. `done_definition`
      4. `forbidden_anti_patterns`
   2. 若缺少上述字段，不得视为正式专业派单。
2. FR-PRO-02 方法卡绑定进入执行链
   1. 首期试运行角色的真实任务，必须引用各自 role-assets 中的最小方法卡入口。
   2. 方法卡不得继续只停留在 `role-assets` 资料层。
3. FR-PRO-03 角色专属执行检查单
   1. `workflow_devmate` 必须有工程治理检查单。
   2. `workflow_testmate` 必须有测试策略检查单。
   3. `workflow_qualitymate` 必须有质量冻结检查单。
   4. 检查单不得写成所有角色通用的大而空模板。
4. FR-PRO-04 输出评分 rubric
   1. 每个首期试运行角色都必须具备自己的输出评分 rubric。
   2. rubric 首期至少覆盖：
      1. 真相是否正确
      2. 方法是否使用正确
      3. 边界是否清晰
      4. 证据是否充分
      5. 建议是否可执行
5. FR-PRO-05 失败不得只回流为学习报告
   1. 若输出质量未达标，不得只通过补一份学习报告视作闭环。
   2. 必须明确是：
      1. 方法卡不适用
      2. 执行检查单缺失
      3. 接单合同不完整
      4. 角色专业边界不清
6. FR-PRO-06 首期效果评估
   1. 首期试运行必须明确评估以下结果：
      1. 首版产物是否更接近可用
      2. PM 回收返工是否下降
      3. 方法卡是否真的进入真实任务引用

## 5. 首期角色专属最小合同
1. `workflow_devmate`
   1. 默认合同至少包含：
      1. truth-source 边界
      2. 投影/排序/截断是否混写
      3. 职责拆分与删冗余判断
      4. 回归与发布风险
2. `workflow_testmate`

      1. 影响面分析
      2. 真实 smoke 与 mock 回归的分层选择
      3. live 合同是否独立于 code-side gate
      4. 漏测盲区
3. `workflow_qualitymate`
   1. 默认合同至少包含：
      1. 证据冻结是否充分
      2. 风险分层是否清楚
      3. 是否达到正式缺陷路由门槛
      4. 回归要求是否明确

## 6. 验收标准
1. AC-PRO-01 接单合同前置成立
   1. Given 首期试运行角色收到真实任务
   2. When 查看任务合同
   3. Then 能看到 `problem_type / method_card_id / done_definition / forbidden_anti_patterns`
2. AC-PRO-02 方法卡进入真实任务
   1. Given 首期试运行角色完成真实任务
   2. When 查看交付与回写
   3. Then 能确认该任务引用了对应方法卡或等价入口
3. AC-PRO-03 角色检查单不是通用空壳
   1. Given `workflow_devmate / workflow_testmate / workflow_qualitymate` 执行任务
   2. When 对照各自检查单
   3. Then 三者的检查点明显不同，且与各自专业方向一致
4. AC-PRO-04 输出质量有评分门槛
   1. Given 角色提交首版产物
   2. When 进入验收
   3. Then 至少按统一维度评分
   4. And 低于阈值不会因为补一份学习报告直接过关
5. AC-PRO-05 首期效果可评估
   1. Given 首期试运行完成一批真实任务
   2. When 回顾效果
   3. Then 至少能回答：
      1. 首版产物是否更接近可用
      2. PM 回收返工是否减少
      3. 方法卡是否进入真实任务

## 7. 边界条件与异常处理
1. 若角色当前还没有可用方法卡：
   1. 不得伪造“已完成方法绑定”。
   2. 应先补最小方法卡，再进入正式专业派单。
2. 若任务强耦合且不适合 helper 先做：
   1. 允许 PM 临时接管。
   2. 但接管后仍要回灌“为什么这条合同暂时无法交给 helper”。
3. 若方法卡与现场问题不匹配：
   1. 应显式记录为“方法失配”。
   2. 后续进入 `纳入 / 合并 / 替换 / 淘汰` 评审。
