# 需求详情-运营项目与项目主控角色模型

- 状态：正式需求（当前有效口径）
- 当前边界：本轮先定义“`workflow` 作为长期运行底座、运营项目作为真实持续运行实体”的产品模型；首期不承诺一次性做完多项目 live 运行、复杂模板市场或子项目体系。

## 1. 主题目标与价值说明
1. 主题目标：
   1. 让 `workflow` 从“当前这个项目”上提为“为多个运营项目提供基础运行能力的底座”。
   2. 让“运营项目”成为产品中的一等对象，而不是继续把所有长期运行语义都挤进单一任务看板。
   3. 让 `项目运营` 成为与 `任务中心` 同级的一级工作模块，而不是继续埋成任务中心里的附属区块。
   4. 统一后续 7x24 唤醒口径：默认唤醒的是“项目主控角色”，不再把 `PM` 写死成唯一合法主控名。
   5. 让任务看板回到通用执行面，只承载任务、状态、运行和证据；项目特有信息转入项目看板。
   6. 让不同运营项目的消耗可控：项目下一棒接力间隔应能在界面调整，而不是只能写死在后台规则里。
2. 业务价值：
   1. 为后续新增小说、漫画、短视频、软件等不同类型团队提供统一底座，而不是每种团队都重新拼装运行机制。
   2. 把“角色是长期协作主体”这一口径落实到项目模型中：项目主控角色负责判断与编排，领域角色负责专业执行与回传。
   3. 让 `workflow` 做得越稳，后续所有项目的运行质量越高，形成明确的平台复用收益。

## 2. 用户画像与使用场景
1. 用户画像 A：`workflow` 平台维护者
   1. 关注点：底座是否足够稳定，能否持续支撑多个运营项目 7x24 运行。
2. 用户画像 B：运营项目创建者
   1. 关注点：能否快速创建一个项目、指定项目主控角色、配置项目成员与默认运行口径。
3. 用户画像 C：项目主控角色
   1. 关注点：自己是否能持续被唤醒、掌握项目目标、节奏、风险和下一步动作，并按项目消耗情况调整下一棒接力间隔。
4. 用户画像 D：领域角色
   1. 关注点：自己是否被视为项目的一部分，拥有明确职责、上下文和交付边界，而不是通用脚本。
5. 核心场景：
   1. 用户创建一个新的运营项目，指定项目类型、项目主控角色和首批领域角色。
   2. 系统围绕该项目主控角色配置和执行 7x24 唤醒。
   3. 领域角色通过任务与路由机制持续参与项目推进，但不与主控角色争夺主线。
   4. 用户先在 `项目运营` 模块看到极简项目列表，首屏只需快速扫到 `项目名称 / 运行状态 / 少量高信号状态`。
   5. 用户进入单个项目后，通过多个页签查看项目不同维度信息，例如：组织架构、版本信息、项目产出、运维参数等。
   6. 用户在项目内只看任务摘要；若要追任务执行细节，则跳转到通用任务中心查看。
   7. 用户在项目看板中直接调整“下一棒接力间隔”，控制该项目后续一轮的唤醒节奏与消耗。
   8. 用户在通用任务看板中查看所有项目共享的任务执行真相，而不要求任务看板继续承载全部项目语义。

## 3. 核心概念模型
1. 平台层：`workflow`
   1. `workflow` 是长期持续运行的底座，不是唯一项目本体。
   2. `workflow` 负责提供运行、调度、留痕、升级、治理、任务和角色基础设施。
2. 项目层：运营项目
   1. 运营项目是长期持续运行的真实实体。
   2. 每个运营项目至少包含：
      1. `project_id`
      2. `project_name`
      3. `project_type`
      4. `controller_role_id`
      5. `member_role_ids`
      6. `project_goal`
      7. `project_board_ref`
      8. `runtime_policy_ref`
   3. `runtime_policy_ref` 首期至少应能承载：
      1. `default_handoff_interval_minutes`
      2. `next_handoff_interval_minutes`
      3. `next_handoff_interval_override_source`
      4. `next_handoff_interval_effective_after_run`
   4. 后续项目级运维参数至少要扩到四组：
      1. 节奏参数
      2. 失败恢复参数
      3. 启动准入参数
      4. 人工控制参数
3. 角色层：项目主控角色与领域角色
   1. 项目主控角色是默认 7x24 唤醒对象，负责判断方向、编排任务、维持连续性。
   2. 领域角色是项目内的专业执行体，负责在各自领域完成任务、诊断问题和回传结果。
   3. 当前系统语义允许把角色视为长期协作主体，而不是一次性脚本。
4. 看板层：项目看板与通用任务看板分层
   1. `项目运营` 是与 `任务中心` 同级的一级模块，负责承载项目层信息与团队协作视角。
   2. 项目列表首屏保持极简，只展示项目名称、项目运行状态和少量高信号摘要。
   3. 单个项目内部通过多页签承载不同维度信息，而不是把全部信息堆进一个长页面。
   4. 通用任务看板承载跨项目共用的任务执行信息。

## 4. 功能需求清单
1. FR-OPS-01 平台与项目分层
   1. 系统必须显式支持“`workflow` 底座”与“运营项目实体”分层。
   2. 后续任何项目级运行语义不得再默认等同于 `workflow` 自身。
2. FR-OPS-02 运营项目对象模型
   1. 系统必须支持创建和维护运营项目对象。
   2. 运营项目至少包含：`project_id / project_name / project_type / controller_role_id / member_role_ids / project_goal / project_board_ref / runtime_policy_ref`。
3. FR-OPS-03 项目主控角色
   1. 系统必须统一使用“项目主控角色”作为 7x24 默认唤醒对象口径。
   2. `PM` 只能作为一种实例名，不再是唯一标准名。
4. FR-OPS-04 领域角色
   1. 领域角色必须被建模为项目成员，而不是与项目无关的外部 worker。
   2. 领域角色可以保有各自领域判断与专业边界。
5. FR-OPS-05 项目模板不写死为软件团队
   1. 系统不得把项目模板固定为软件研发范式。
   2. 首期至少要为后续自定义项目类型预留扩展口径，允许项目类型覆盖小说、漫画、短视频、软件等不同团队形态。
6. FR-OPS-06 项目看板
   1. 每个运营项目都应有独立项目看板。
   2. 项目看板前必须先存在一个与任务中心同级的 `项目运营` 一级模块入口。
   3. 项目列表首期保持极简，至少展示：
      1. 项目名称
      2. 项目运行状态
      3. 少量高信号状态摘要
   4. 项目看板首期至少需要承载：
      1. 项目目标
      2. 项目主控角色
      3. 项目成员结构
      4. 运行状态
      5. 风险与阻塞
      6. 当前节奏与下一步
      7. 下一棒接力间隔
      8. 关键证据入口
      9. 项目本体表达入口（优先采用图，而不是只堆文字摘要）
   5. 项目看板首期应至少支持以下页签方向：
      1. 组织架构
      2. 版本信息
      3. 项目产出
      4. 运维参数
   6. 后续允许继续增加页签和信息维度，但首期不要求一次性铺满所有未来能力。
7. FR-OPS-07 通用任务看板收口
   1. 当前任务看板后续应收口为通用执行面。
   2. 通用任务看板只保留跨项目共用信息，例如任务、状态、运行、证据、路由。
   3. 项目特有目标、成员、节奏和项目内风险不应继续长期堆在通用任务看板里。
   4. 项目内部只展示任务摘要；任务详情默认回跳任务中心查看，不要求项目页重复承载完整任务细节。
8. FR-OPS-08 子项目首期不开放
   1. 首期不引入子项目模型。
   2. 所有运营项目默认平铺在 `workflow` 底座之下。
9. FR-OPS-09 跨项目角色共享首期不限制
   1. 首期不对角色是否能跨项目共享做强限制。
   2. 但系统应保留项目归属与项目内职责语义，不得因为共享可能性而弱化项目边界。
10. FR-OPS-10 `workflow` 自身的长期定位
   1. `workflow` 自身既是底座，也是需要长期持续运行的基础项目。
   2. `workflow` 的持续质量提升应被视为其他运营项目运行质量的上游基础。
11. FR-OPS-11 项目下一棒接力间隔可在界面调整
   1. 系统必须支持在项目看板或等价项目级运行面中调整该项目的“下一棒接力间隔”。
   2. 这条调整应服务于不同项目的消耗控制，不要求用户通过改后台固定规则才能生效。
   3. 首期至少支持：
      1. 查看当前默认接力间隔
      2. 查看下一棒当前生效间隔
      3. 手工覆盖下一棒接力间隔
      4. 明确本次覆盖来自谁、从哪一轮开始生效
   4. 该调整默认作用于当前项目，不应误改其他项目的运行节奏。
12. FR-OPS-12 运维参数页签
   1. 项目级配置不应继续散落在多个零碎入口。
   2. 首期至少应提供一个 `运维参数` 页签或等价工作面，用于承载项目的运行参数、节奏参数和后续可扩展的项目级控制项。
   3. 当前已知的项目下一棒接力间隔，应优先落在这一页签中。
   4. 项目级运维参数后续至少扩展到：
      1. 节奏参数：`default_handoff_interval_minutes / next_handoff_interval_minutes / next_handoff_interval_override_source / next_handoff_interval_effective_after_run`
      2. 失败恢复参数：`failure_retry_cooldown_minutes / failure_retry_max_attempts / failure_escalation_threshold / failure_route_kind`
      3. 启动准入参数：`require_controller_ready / require_member_roles_ready / minimum_member_count`
      4. 人工控制参数：`manual_pause / auto_resume_at / allowed_run_window`
   5. 失败恢复参数只承接项目级策略，不直接暴露引擎内部退避秒数、provider 启动重试秒数或 polling 频率。
13. FR-OPS-13 项目产出优先用图表达项目本体
   1. `项目产出` 页签不应只作为任务摘要或产物文件列表入口。
   2. 对 `workflow` 这类平台项目，首期应优先采用：
      1. 架构总览图
      2. 运行拓扑图
      3. 关键能力图
      4. 任务摘要入口
   3. 图表达负责讲清“这是什么、怎么组成、怎么运行、已经交付了什么”；任务摘要退到辅助位置。
   4. 项目产出页不得退化成第二个任务中心。

## 5. 非功能需求
1. 一致性：
   1. 产品文案、角色模型、看板字段和调度语义中，“项目主控角色”必须保持统一命名。
2. 可扩展性：
   1. 首期虽然不开放子项目，但模型不得把未来多项目扩展堵死。
3. 可读性：
   1. 项目看板与通用任务看板的边界必须清晰，避免同一信息在两个工作面长期重复堆叠。
4. 连续性：
   1. 项目主控角色与领域角色都应继续服从现有记忆、治理和运行连续性约束。
5. 可控性：
   1. 不同项目的接力节奏必须可单独控制，避免所有项目共用同一固定消耗曲线。

## 6. 验收标准
1. AC-OPS-01 项目主控角色命名冻结
   1. Given 系统需要表达项目 7x24 默认唤醒对象
   2. When 产品文案与模型落盘
   3. Then 统一使用“项目主控角色”
   4. And 不再把 `PM` 写死为唯一标准名
2. AC-OPS-02 运营项目是独立对象
   1. Given 用户查看项目模型定义
   2. When 查看字段与边界
   3. Then 可明确区分 `workflow` 底座与运营项目实体
3. AC-OPS-03 项目模板不限定软件团队
   1. Given 用户创建项目
   2. When 选择或定义项目类型
   3. Then 系统语义允许非软件型团队存在
4. AC-OPS-04 看板分层成立
   1. Given 用户查看项目级信息与任务级信息
   2. When 对照项目看板与通用任务看板
   3. Then 可以清楚区分项目特有信息与跨项目通用执行信息
5. AC-OPS-05 下一棒接力间隔可在界面调整
   1. Given 某运营项目已经存在默认运行口径
   2. When 用户在项目看板调整该项目下一棒接力间隔
   3. Then 系统记录新的下一棒生效间隔
   4. And 该变更只影响当前项目
   5. And 用户可以看到这次覆盖的来源与生效轮次信息
6. AC-OPS-06 `项目运营` 与 `任务中心` 同级
   1. Given 用户查看一级导航
   2. When 进入项目层工作面
   3. Then 能看到与 `任务中心` 同级的 `项目运营` 模块
   4. And `项目运营` 不再只是任务中心内部的一个区块
7. AC-OPS-07 项目列表极简
   1. Given 用户进入 `项目运营` 模块
   2. When 查看项目列表首屏
   3. Then 默认只看到项目名称、项目运行状态和少量高信号摘要
   4. And 不要求首屏堆叠项目全部细节
8. AC-OPS-08 项目页签分层成立
   1. Given 用户进入某个项目
   2. When 查看项目内部信息架构
   3. Then 能看到组织架构、版本信息、项目产出、运维参数等页签方向
   4. And 任务详情不在项目页重复展开，而是默认回到任务中心查看
9. AC-OPS-09 项目产出图表达成立
   1. Given 用户进入 `项目产出` 页签
   2. When 查看当前项目说明
   3. Then 默认先看到项目本体的图表达
   4. And 任务摘要只作为辅助入口，不再承担主表达职责

## 7. 边界条件与异常处理
1. 若项目尚未定义项目主控角色：
   1. 系统不得伪造已具备完整 7x24 主控口径。
2. 若项目类型不是软件团队：
   1. 系统不得因默认研发模板而拒绝项目成立。
3. 若任务看板与项目看板信息重复：
   1. 以后续看板分层整改为准，不得继续把重复堆叠视为稳定设计。
4. 若用户调整下一棒接力间隔：
   1. 系统不得把该变更误广播到其他项目。
   2. 若当前项目不存在可接续主线，应明确提示“已记录，将从下一次有效接棒开始按新间隔生效”或等价状态，而不是静默丢失。

## 8. 依赖项与开放问题
1. 依赖项：
   1. 依赖现有角色、任务中心、定时唤醒和记忆治理基础设施。
   2. 依赖后续项目看板与项目模板能力落地。
2. 开放问题：
   1. `项目运营` 模块与 `任务中心` 的一级导航文案、排序和默认落点，后续仍需结合 UCD 原型进一步细化。
   2. 项目模板是否预置若干行业模板，还是先完全自定义，留待后续版本切片决定。
   3. 跨项目共享角色的调度、上下文隔离和归属展示，当前先不限制，但后续可能需要单独治理。
   4. 下一棒接力间隔的 UI 交互首期是直接录入分钟数、录入预设档位，还是提供常用按钮组，后续在实现切片中再定。
   5. 项目页签首期是否固定四个页签，还是允许按项目类型动态增减，后续在原型阶段再定。
