# 需求详情-任务中心单agent串行与失败任务归档治理

- 状态：正式需求（当前有效口径）
- 提出时间：`2026-04-08`
- 目标：把任务中心里“同 agent 同时只跑一个任务、保底巡检不再和 PM 主线抢运行槽、失败任务支持标准归档处置、任务看板支持区内滚动观察”收成明确的版本拆解与交付边界。

## 1. 问题定义
1. 当前理论口径已经要求“同一个 agent 同时只能运行 1 个任务”，但在 `7*24` 主线与保底巡检场景里，还没有把这条约束彻底变成稳定的工程事实。
2. 当前保底巡检仍可能与 `workflow(pm)` 主线共用同一个执行主体，导致：
   1. PM 主线和保底巡检争抢同一个运行槽；
   2. 页面看起来有任务，但实际只是自己把自己堵住；
   3. `7*24` 的连续推进与保底兜底职责混在同一个 agent 身上。
3. 当前失败任务虽然支持 `rerun` 或人工改状态，但还缺少“PM 已经处理完成后，把失败节点标准归档”的正式处置模型。
4. 当前任务看板在任务堆积时，观察体验仍容易退化成整页拉长，不利于长期巡检和失败归档后的历史对照。

## 2. 本轮新增需求

### 2.1 FR-TC-NEW-01 单 agent 单任务从“理论约束”升级为“7*24 强执行约束”
1. 同一个 `assigned_agent_id` 在任何时刻只允许存在 `1` 个真实 `running` 任务。
2. 这条约束不只适用于普通任务，也适用于：
   1. `[持续迭代] workflow`
   2. `pm持续唤醒 - workflow 主线巡检`
   3. 其他由 schedule 自动创建的 helper 任务
3. 后端调度、schedule 恢复、ready -> dispatch、upgrade 前 drain 复核都必须共用这条规则，不允许某个入口绕过。

### 2.2 FR-TC-NEW-02 保底巡检从 PM 主线分流
1. `pm持续唤醒 - workflow 主线巡检` 不再默认由 `workflow(pm)` 自己执行。
2. 需要新增一个专门承接保底巡检的小伙伴。
3. 暂定命名建议：`workflow_watchmate`
4. 该小伙伴的职责只负责：
   1. 核对主线是否仍保留 `running / ready / future`
   2. 判断当前是否存在可升级空窗
   3. 在主链断裂时补链或补单
   4. 输出结构化巡检结论
5. 该小伙伴不承担 PM 主线版本推进，不替代 `workflow(pm)` 做长期决策。

### 2.3 FR-TC-NEW-03 失败任务标准处置与归档
1. 当前失败任务在 PM 处理后，必须支持从“活动失败”视图转入“已归档失败”。
2. 首期至少支持以下标准处置方式：
   1. `rerun_replaced`
      - 含义：已重启任务，旧失败节点归档，新节点继续承担主责。
   2. `direction_changed`
      - 含义：迭代方向已变更，该失败任务不再需要继续执行。
   3. `duplicate_or_invalid`
      - 含义：重复建单、错误建单或已确认无效。
   4. `manual_closed`
      - 含义：PM 已人工收口，不再继续推进。
3. 归档后必须保留：
   1. 原失败节点
   2. 原失败摘要
   3. 处置人
   4. 处置理由
   5. 处置时间
   6. 若有替代节点，则记录 replacement `node_id`
4. 归档后不再计入当前活动失败总数，但必须可追溯、可检索、可回看。

### 2.4 FR-TC-NEW-04 任务看板区内滚动
1. 任务看板允许使用滚动条承载超出内容，而不是继续拉长整页。
2. 首期至少收口以下区域：
   1. 中部任务图区域
   2. 右侧任务详情区域
   3. 失败归档列表或归档抽屉
3. 当前有效目标是“固定窗口 + 区内滚动 + 头部信息稳定可见”。

## 3. 版本拆解

### 3.1 `V1-P9` 单 agent 串行硬约束与保底巡检分流
- 版本归属：`V1` 工程质量基线与运行稳态
- 默认负责人：`workflow_devmate`
- 类型：稳定性 / 7*24 运行基线
- 目标：
  - 把“同 agent 同时只跑 1 个任务”变成任务中心、schedule、主线和保底共同遵守的硬规则。
  - 把保底巡检从 `workflow(pm)` 分流到专用小伙伴，避免 PM 主线和保底互相堵塞。
- 范围：
  1. 后端强约束同 agent 单 `running`
  2. `pm持续唤醒` 默认改派给专用巡检小伙伴
  3. schedule 文案、done_definition、memory 读链和角色脚手架适配新小伙伴
  4. 验收覆盖“主线 running 时，保底不会再抢占 workflow 自己的运行槽”
- 不含：
  1. 失败任务归档视图
  2. 大规模任务中心 UI 重构
- 预估工期：
  - 开发：`1.5 ~ 2` 个工作日
  - 联调 / gate / test / prod rollout：`0.5 ~ 1` 个工作日
  - 小计：`2 ~ 3` 个工作日

### 3.2 `V1-P10` 失败任务标准处置与归档最小闭环
- 版本归属：`V1` 工程质量基线与运行稳态
- 默认负责人：`workflow(pm)`
- 类型：流程治理 / 任务中心闭环
- 目标：
  - 把“失败后只有 rerun 或人工改状态”升级为“PM 可按标准方式处置并归档”。
  - 让任务图中的活动失败与历史已处理失败分开。
- 范围：
  1. 节点新增 `archived` 或等效关系态
  2. 失败任务新增归档动作
  3. 标准处置原因码
  4. 归档审计、replacement node 绑定
  5. 失败总数与归档总数拆分
  6. PM 处置后下游影响重算规则
- 首期收口建议：
  1. `重跑并归档旧失败`
  2. `方向变更后归档`
  3. `重复/无效任务归档`
  4. `人工关闭归档`
- 不含：
  1. 自动批量归档策略
  2. 复杂统计报表
- 预估工期：
  - 设计与后端实现：`1.5 ~ 2` 个工作日
  - 前端动作与状态展示：`0.5 ~ 1` 个工作日
  - 联调 / 回归：`0.5 ~ 1` 个工作日
  - 小计：`2.5 ~ 4` 个工作日

### 3.3 `V1-P11` 任务看板滚动与归档观察体验收口
- 版本归属：`V1` 工程质量基线与运行稳态
- 默认负责人：`workflow_qualitymate`
- 类型：观察体验 / 低风险 UI
- 目标：
  - 让任务看板在任务堆积和归档后仍然稳定可观察。
- 范围：
  1. 任务图区域区内滚动
  2. 任务详情区区内滚动
  3. 归档列表区滚动
  4. 头部与核心状态信息固定可见
- 不含：
  1. 全量视觉改版
  2. 看板信息架构重做
- 预估工期：
  - 开发：`0.5` 个工作日
  - 验收与截图：`0.5` 个工作日
  - 小计：`1` 个工作日

### 3.4 `V2-P0` 归档规范与协作口径固化
- 版本归属：`V2` 项目规范与设计规范固化
- 默认负责人：`workflow(pm)`
- 类型：规范固化
- 目标：
  - 把失败任务归档、replacement 绑定、PM 标准处置原因码、看板计数口径和巡检要求固化到规范文档与验收矩阵。
- 范围：
  1. 归档理由白名单
  2. 归档后计数与看板口径
  3. 失败处置 SOP
  4. 与小伙伴角色分工的协作规范引用
- 预估工期：
  - `1 ~ 1.5` 个工作日

## 4. 推荐排期
1. 第一批先做 `V1-P9`
   - 原因：这是 7*24 主线基线问题，不先拆开 PM 主线与保底巡检，后面的归档和看板优化都还会被运行槽冲突污染。
2. 第二批做 `V1-P10`
   - 原因：失败任务归档属于当前任务中心的管理闭环，优先级高于纯 UI 优化。
3. 第三批做 `V1-P11`
   - 原因：滚动条是低风险体验增强，适合在状态模型和归档入口基本稳定后一起收口。
4. 最后在 `V2-P0` 固化规范
   - 原因：等 V1 的实现口径稳定后，再把 SOP 和协作规范写死，更不容易反复返工。

## 5. 当前建议的总工期
1. 若按“最小可用闭环”推进，不插入更高优先级止血项：
   1. `V1-P9`：`2 ~ 3` 个工作日
   2. `V1-P10`：`2.5 ~ 4` 个工作日
   3. `V1-P11`：`1` 个工作日
   4. 合计：`5.5 ~ 8` 个工作日
2. 若中途再次出现 `prod` 断链、升级门禁或主线接力故障，需要给 `V1-P0 / V1-P1` 让路，则整体顺延 `1 ~ 3` 个工作日。

## 6. 当前最小交付定义
1. `workflow(pm)` 主线和保底巡检不再由同一个 agent 争抢运行槽。
2. 同一个 agent 无法再同时真实运行两个任务。
3. 失败任务经 PM 处理后，可以明确归档并写明原因。
4. 任务看板在任务堆积时可通过滚动条继续观察，不再整页失控拉长。

## 7. 当前建议
1. 我建议把这 3 条需求全部纳入 `V1` 的尾段，而不是推迟到 `V2` 才开始。
2. 理由：
   1. 第一条和第三条直接影响当前 `7*24` 连续推进的可靠性；
   2. 第二条虽然偏 UI，但对你持续观察主图和归档结果是低成本高收益增强。
