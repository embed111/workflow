# 需求详情-任务中心真实执行化与执行链路可视化

## 1. 主题目标与价值说明
1. 将 `任务中心` 从“只做流程流转”升级为“真正触发 agent 执行”的任务编排中心。
2. 让任务执行过程不再是黑盒，用户可直接看到完整提示词、执行过程与最终结果证据。
3. 让 `设置` 页只承载全局可配置能力，让任务详情承载单任务执行可见性，避免信息层级混乱。
4. 复用 `workflow` 既有的 `Codex` 执行链路与证据留痕能力，减少重复造轮子。

## 2. 用户画像与使用场景
1. 用户画像 A：workflow 操作者
   1. 关注点：点了运行后任务是否真的执行、当前执行到了哪里、失败原因是什么。
2. 用户画像 B：平台治理者
   1. 关注点：执行链路是否可追溯、是否存在黑盒、全局并发是否可控。
3. 用户画像 C：后续外部 workflow 调用方
   1. 关注点：通过接口创建任务图后，是否能直接复用统一的 agent 执行能力，而不是再补一套私有执行链路。
4. 核心场景：
   1. 用户在 `设置` 中查看并维护当前任务执行通道与并发配置。
   2. 调度器将就绪任务推进到 `running` 后，后端真实启动目标 agent 执行任务。
   3. 用户点击运行中的任务，在右侧 `执行链路` 中查看完整提示词、调用命令和实时过程。
   4. 任务执行成功后，系统自动回写执行结果与产物引用。
   5. 任务执行失败后，系统自动标记失败并阻塞下游，用户可选择 `重跑任务` 或 `人工修改执行状态`。

## 3. 用户旅程或关键流程
1. 管理员进入 `设置`，查看 `任务执行` 区块。
2. 当前页面至少可见：
   1. `执行通道`
   2. `Codex 调用路径`
   3. `命令模板`
   4. `全局并发上限`
3. 页面或调度器调用 `dispatch-next`。
4. 后端按依赖门禁、单 agent 单并发与全局并发上限选出可运行节点。
5. 对每个被选中的节点：
   1. 创建运行批次记录；
   2. 解析 `assigned_agent_id -> workspace_path`；
   3. 生成完整提示词；
   4. 调用当前执行通道启动真实执行。
6. 运行过程中，后端持续写入：
   1. 运行事件；
   2. stdout/stderr；
   3. 中间结果摘要；
   4. 最终结果与引用路径。
7. 前端在任务详情的 `执行链路` 中展示：
   1. 执行通道；
   2. 运行批次；
   3. 最新事件；
   4. 目标工作区；
   5. 调用命令；
   6. 完整提示词；
   7. 执行过程。
8. 若执行成功：
   1. 节点进入成功态；
   2. 系统回写结果与产物引用；
   3. 满足条件时释放下游任务。
9. 若执行失败：
   1. 节点进入失败态；
   2. 系统写入失败原因；
   3. 下游节点进入阻塞态；
   4. 用户仍可在 `任务管理` 中选择 `重跑任务` 或 `人工修改执行状态`。

## 4. 功能需求清单
1. FR-EX-01 执行通道抽象
   1. 后端必须抽象统一的任务执行通道接口。
   2. 调度器不得直接把 `Codex` 命令散落写死在业务控制器中。
   3. 第一版仅实现 `Codex`，但接口层必须支持后续扩展更多 provider。
2. FR-EX-02 设置页最小任务执行配置
   1. `设置` 页必须新增 `任务执行` 区块。
   2. 当前区块至少展示：
      1. `执行通道`
      2. `Codex 调用路径`
      3. `命令模板`
      4. `全局并发上限`
   3. 该区块属于既有设置页中的新增内容，不得遮蔽原有设置项。
3. FR-EX-03 第一版执行通道范围
   1. 第一版 `执行通道` 枚举仅允许 `Codex`。
   2. 若收到其他 provider 配置，系统必须明确返回“不支持当前通道”，不得静默降级。
4. FR-EX-04 `dispatch-next` 真实执行
   1. `dispatch-next` 不得只修改节点业务状态。
   2. 当节点进入 `running` 时，后端必须真实创建执行批次并启动目标 agent 执行。
5. FR-EX-05 运行批次记录
   1. 每次真实执行至少生成一个稳定的 `run_id`。
   2. 每个 `run_id` 至少关联：
      1. `ticket_id`
      2. `node_id`
      3. `provider`
      4. `workspace_path`
      5. `started_at`
      6. `finished_at`
      7. `status`
      8. `prompt_ref`
      9. `stdout_ref`
      10. `stderr_ref`
      11. `result_ref`
6. FR-EX-06 agent 到工作区映射
   1. 后端在真实执行前必须能将 `assigned_agent_id` 解析为稳定的工作区路径。
   2. 若映射失败，系统必须 fail-closed，不得假装已启动执行。
7. FR-EX-07 完整提示词可见
   1. 对于每个真实执行批次，页面必须可查看完整提示词。
   2. 完整提示词默认可见于任务详情的 `执行链路` 卡片中，不要求再跳转到其他页面。
8. FR-EX-08 执行过程可见
   1. 页面必须可查看执行过程中的关键事件与输出。
   2. 至少包括：
      1. 调度派发事件；
      2. 调用命令摘要；
      3. stdout；
      4. stderr；
      5. 最终结果。
9. FR-EX-09 实时可见性
   1. 运行中的任务详情应尽量实时刷新。
   2. 推断/假设（置信度: 中）：优先采用 `SSE/WebSocket`，若当前工程不适合，首版可退化为 `<= 1 秒` 轮询，但不得只在任务结束后一次性补全。
10. FR-EX-10 任务详情执行链路卡片
   1. 右侧任务详情必须新增 `执行链路` 卡片。
   2. 该卡片至少展示：
      1. `执行通道`
      2. `运行批次`
      3. `最新事件`
      4. `目标工作区`
      5. `调用命令`
      6. `完整提示词`
      7. `执行过程`
11. FR-EX-11 设置页与任务详情分层边界
   1. `设置` 页只承载全局配置，不承载单任务运行过程。
   2. `完整提示词 / 实时执行过程 / 执行结果证据` 固定放在任务详情中展示。
12. FR-EX-12 `running` 状态语义升级
   1. 从 2026-03-16 起，`running` 不再表示“已派发等待人工回写”。
   2. `running` 必须表示“已创建运行批次并处于真实执行中”。
13. FR-EX-13 失败与阻塞联动保持不变
   1. 若真实执行失败，节点进入失败态。
   2. 任一上游失败时，下游节点必须继续进入阻塞态。
14. FR-EX-14 补救动作保持不变
   1. `重跑任务` 与 `人工修改执行状态` 继续保留在 `任务管理` 中。
   2. 真实执行失败后，这两个入口仍作为当前阶段的补救方式。
15. FR-EX-15 暂停语义保持不变
   1. 点击 `暂停调度` 后，当前运行中的任务继续执行到结束。
   2. 后端不得再派发新任务。
   3. 当前运行任务全部结束后，调度状态转为 `已暂停`。
16. FR-EX-16 产物交付规则继续生效
   1. 真实执行成功后，产物仍必须通过统一交付接口落盘。
   2. 任务成功门禁仍受 `产物已交付` 约束。
17. FR-EX-17 证据留痕
   1. 每次真实执行至少应可追溯到：
      1. 完整提示词
      2. 调用命令摘要
      3. stdout/stderr
      4. 最终结果
      5. 产物路径或结果引用
18. FR-EX-18 安全与脱敏
   1. 页面展示执行链路时，不得泄露不应暴露的敏感配置。
   2. 推断/假设（置信度: 中）：若命令行包含敏感参数，可按字段脱敏；但不得因此隐藏完整提示词主体。

## 5. 非功能需求
1. 实时性：
   1. 运行中任务的执行链路刷新间隔 `p95 <= 1 秒`。
2. 可观测性：
   1. 任一真实执行批次都必须可回放“提示词 -> 执行过程 -> 最终结果”链路。
3. 一致性：
   1. 设置页展示的执行通道配置必须与后端实际使用的 provider 一致。
   2. 任务详情显示的运行状态必须与后端执行状态一致。
4. 可靠性：
   1. 同一节点不得因重复派发而产生两个活动中的真实执行批次。
5. 可维护性：
   1. provider 选择、运行批次记录与执行流输出采集应模块化，不得与页面逻辑紧耦合。

## 6. 验收标准
1. AC-EX-01 设置页任务执行配置可见
   1. Given 用户进入 `设置`
   2. When 页面加载完成
   3. Then 可看到 `任务执行` 区块
   4. And 可看到 `执行通道 / Codex 调用路径 / 命令模板 / 全局并发上限`
2. AC-EX-02 第一版仅允许 Codex
   1. Given 系统当前处于第一版
   2. When 用户或接口尝试设置非 `Codex` 的执行通道
   3. Then 系统明确拒绝并提示当前阶段不支持
3. AC-EX-03 派发即真实执行
   1. Given 某任务满足依赖门禁并进入可运行状态
   2. When 调用 `dispatch-next`
   3. Then 后端创建真实执行批次并启动目标 agent 执行
   4. And 节点状态进入 `running`
4. AC-EX-04 完整提示词可见
   1. Given 某任务已产生真实执行批次
   2. When 用户打开该任务详情
   3. Then 可查看完整提示词
5. AC-EX-05 执行过程实时可见
   1. Given 某任务正在执行
   2. When 用户停留在该任务详情
   3. Then 可看到持续刷新的执行事件或输出
   4. And 不要求等任务结束后才一次性展示
6. AC-EX-06 成功后自动回写
   1. Given 某任务真实执行成功
   2. When 运行批次结束
   3. Then 系统自动回写成功态、结果引用与产物信息
7. AC-EX-07 失败后阻塞下游
   1. Given 某任务真实执行失败且存在下游
   2. When 失败结果被确认
   3. Then 当前任务进入失败态
   4. And 下游进入阻塞态
8. AC-EX-08 补救动作仍可用
   1. Given 某任务真实执行失败
   2. When 用户打开 `任务管理`
   3. Then 仍可看到 `重跑任务` 与 `人工修改执行状态`
9. AC-EX-09 `running` 语义正确
   1. Given 某任务显示为 `running`
   2. When 用户查看运行批次记录
   3. Then 能看到真实存在的 `run_id` 与执行过程
10. AC-EX-10 分层边界正确
   1. Given 用户查看 `设置` 页与任务详情页
   2. When 分别检查两处内容
   3. Then `设置` 页仅展示全局任务执行配置
   4. And 完整提示词与执行过程只在任务详情的 `执行链路` 中展示

## 7. 边界条件与异常处理
1. 若 `assigned_agent_id -> workspace_path` 映射失败：
   1. 节点不得进入伪 `running`，必须直接返回错误并保留审计。
2. 若真实执行启动失败：
   1. 节点进入失败态或等效错误态，不得假装已启动成功。
3. 若实时流通道不可用：
   1. 首版可退化为短轮询，但任务结束后必须保留完整结果可查。
4. 若 stdout/stderr 内容过长：
   1. 页面可折叠或分段加载，但不得静默截断到不可追溯。
5. 若任务被暂停调度时已有运行批次：
   1. 当前运行批次继续完成；仅阻止新任务派发。
6. 若 `Codex` 返回非零退出码：
   1. 系统必须记录失败原因并进入失败处理，而不是停留在 `running`。

## 8. 依赖项与开放问题
1. 依赖项：
   1. 依赖 `workflow` 工作区具备可稳定调用的 `Codex` 本地执行环境。
   2. 依赖 `assigned_agent_id -> workspace_path` 的稳定映射关系。
   3. 依赖现有证据落盘与执行链路展示模式可被复用。
2. 开放问题：
   1. 第一版实时链路最终采用 `SSE/WebSocket` 还是 `<= 1 秒` 轮询。
      1. 推断/假设（置信度: 中）：若当前工程已有稳定流式基础设施，优先 `SSE/WebSocket`；否则首版允许轮询落地。
   2. 第一版是否直接提供“打开 stdout/stderr 原文件”入口。
      1. 推断/假设（置信度: 中）：可作为增强项，首版先保证页面内可见与路径可追溯。
